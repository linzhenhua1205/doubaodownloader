## 1. 导论：分区的第一性原理

### 1.1 为什么需要分区？

从第一性原理出发：**任何一个系统，如果所有组件处于同一"空间"且共享同一套规则，必然导致冲突、耦合与不可控。**

证明方向有三：

| 冲突类型 | 表现 | 物理对应 |
|:---------|:-----|:---------|
| **资源冲突** | 多任务争用同一存储/算力/带宽，无优先级保证 | 进程地址空间隔离 |
| **安全冲突** | 不信任方访问同一区域可窃取/篡改数据 | 用户态/内核态分离 |
| **变更冲突** | 未完成的操作混在已完成的操作中，导致不可回滚 | Git 暂存区隔离 |

分区的本质是**将系统的不同生命周期阶段、不同安全等级、不同操作频率的组件/数据，通过显式边界分隔到不同的"区域"中，在每个区域内制定自治规则，跨区域时通过协议交互。**

### 1.2 五区驱动的根本原因

从五大领域的共性归纳，分区的驱动力来自五个维度：

```
┌─────────────────────────────────────────────────────┐
│          分区五维驱动力模型                           │
│                                                     │
│  稳定性梯度 ──────────────── 可变性梯度              │
│  (越稳定的区域越靠近用户/生产)  (越不确定的区域      │
│   越在开发面)                                        │
│                                                     │
│  可见性梯度 ──────────────── 安全梯度                │
│  (越内部的区可见性越低)       (越核心的区安全约束    │
│                               越严格)               │
│                                                     │
│  时效性梯度 ──────────────── 持久性梯度              │
│  (延迟要求越高的区越靠近计算)  (需要持久保存的越往  │
│                                 下沉固化)           │
└─────────────────────────────────────────────────────┘
```

下表列出五个驱动维度在五大领域的具体体现（后文每域展开）：

| 驱动维度 | 分布式 | Git | 虚拟化 | Docker | 存储层级 |
|:---------|:-------|:----|:-------|:-------|:---------|
| **稳定性梯度** | 控制面需7×24，数据面可弹性 | 远程仓库不可变，工作目录可变 | Host OS 稳定，VM 可崩溃 | Registry 只增，Container 可写 | 寄存器无状态，磁盘持久化 |
| **可变性梯度** | 管理调度策略高频变，设备配置低频变 | 工作目录改最频繁，远程最少改 | Guest OS 自由改，Host 只装基础 | 容器层叠只增不删，容器层可写 | 缓存常改，磁盘少改 |
| **可见性梯度** | 管理节点看全局，工作节点只看自己 | 远程库全局可见，工作目录仅本地 | Host 可见 Guest 内存，Guest 不可见 Host | Registry 全局，Container 仅本机 | 寄存器线程可见，磁盘全局可见 |
| **安全梯度** | API Server 认证，kubelet 免密 | 远程库需鉴权，本地免鉴权 | Host 在 Root Zone，Guest 在 User Zone | Registry 需登录，本地不鉴权 | 缓存无保护，磁盘加密可选 |
| **时效性/持久性** | 调度决策 μs 级，日志存储天级 | commit 永久，工作目录可丢弃 | VM 快照持久，内存不持久 | Image 持久，Container 临时 | 寄存器 ns 级，磁盘 ms 级 |

## 2. 域一：分布式系统——管理面与数据面

### 2.1 区域定位与职责

以 Kubernetes 为例，分区的第一道边界——**管理节点(Master) vs 工作节点(Node)**：

```
┌─────────────────────────────────────────────────────────┐
│                   分布式管理面                            │
│  ┌─────────┐  ┌─────────┐  ┌──────────┐  ┌─────────┐  │
│  │ API     │  │ Scheduler│  │CM/CCM   │  │ etcd    │  │
│  │ Server  │  │          │  │          │  │ (状态)  │  │
│  └────┬────┘  └─────────┘  └──────────┘  └─────────┘  │
│       │                 管理面（大脑）                   │
├───────┼─────────────────────────────────────────────────┤
│       │                 边界：kubelet <-> API Server     │
├───────┼─────────────────────────────────────────────────┤
│  ┌────┴──────────────────────────────────────────┐      │
│  │         数据面（肌肉）                         │      │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐      │      │
│  │  │ Node 1   │ │ Node 2   │ │ Node N   │      │      │
│  │  │ (kubelet,│ │ (kubelet,│ │ (kubelet,│      │      │
│  │  │  Pods)   │ │  Pods)   │ │  Pods)   │      │      │
│  │  └──────────┘ └──────────┘ └──────────┘      │      │
│  └───────────────────────────────────────────────┘      │
└─────────────────────────────────────────────────────────┘
```

### 2.2 各区规则

#### 管理面（控制平面）

| 规则维度 | 具体约束 |
|:---------|:---------|
| **身份** | 全局状态持有者，唯一权威来源（etcd） |
| **区内通信** | API Server 内部组件通过 Watch 机制订阅变更 |
| **状态持久性** | 所有状态写入 etcd，崩溃后重建上下文 |
| **写权限** | 仅 API Server 写入 etcd，其他组件只读+提交 |
| **水平扩展** | 控制面可多副本，但 Leader 仅一个（etcd/选举） |
| **安全边界** | RBAC + TLS 双向认证，所有请求经审计 |

#### 工作节点（数据面）

| 规则维度 | 具体约束 |
|:---------|:---------|
| **身份** | 无状态执行者，不存持久数据 |
| **区内通信** | Pod↔Pod 直通网络（CNI），kubelet ↔ Pod |
| **状态持久性** | 无，连 Pod IP 也是可变的 |
| **写权限** | 只能写本地临时数据，不能改集群状态 |
| **水平扩展** | 可任意增加，对控制面透明 |
| **安全边界** | kubelet 对 API Server 单向注册，Node 间无隐含信任 |

### 2.3 跨区交互协议

| 协议 | 发起方→接收方 | 触发条件 | 频度 | 关键特征 |
|:-----|:-------------|:---------|:-----|:---------|
| **Watch** | Node→Master | 常驻长连接 | 持续 | 增量事件推送，非轮询 |
| **Report** | kubelet→API Server | 周期上报 | 10-60s | Node/Pod 状态心跳 |
| **Schedule** | Scheduler→API Server | 有未调度 Pod | 按需 | Bind 操作写 Pod→Node 关系 |
| **Command** | API Server→kubelet | 调度/运维事件 | 事件驱动 | Pod Create/Delete/Update |
| **Proxy** | kubelet+API Server | 用户请求 | 按需 | kubectl exec/logs 经过 |

### 2.4 关键学理

**管理面与数据面分离的根本原因不是"更好管理"，而是"故障域隔离"**：
- 数据面节点崩溃 → 不影响控制面决策
- 控制面崩溃 → 数据面继续运行已调度 Pod（已有状态）
- 但如果这时数据面 Node 也崩溃 → Pod 无法重调度（因为控制面不可用）

这就是**分区边界设计的关键权衡**：边界越厚（越独立），故障隔离越好，但跨区操作代价越大。

## 3. 域二：Git——四区流转与状态机

### 3.1 四区模型

Git 可能是"位置化分区"最干净的例子：

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  工作目录     │ →   │  暂存区      │ →   │  本地仓库    │ →   │  远程仓库    │
│  Working Dir │     │  Index       │     │  Local Repo  │     │  Remote Repo │
│              │     │  (Stage)     │     │  (.git/obj)  │     │  (Server)    │
├──────────────┤     ├──────────────┤     ├──────────────┤     ├──────────────┤
│ ✏️ 最高可变性│     │ 📋 待定缓冲  │     │ 📦 不可变    │     │ 🔒 最高稳定  │
│ 😱 可丢弃    │     │ 💾 可不提交  │     │  💎 版本化    │     │ 📡 可协作   │
│ 🔍 纯本地     │     │ 🏠 纯本地    │     │  🏠 可本地    │     │ 🌐 需鉴权   │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
```

### 3.2 各区定位与规则

#### 工作目录 (Working Directory)

| 属性 | 值 |
|:-----|:----|
| **稳定性** | 🔴 最低 — 文件随时可编辑/删除/创建 |
| **可见性** | 👤 仅本用户 — 不通知任何人 |
| **持久性** | ❌ 不保证 — `git checkout` 可覆盖未提交修改 |
| **区内操作** | `vim`, `touch`, `rm`, `mv` — 任意操作 |
| **意义** | 隔离「正在做的事」和「已经完成的事」 |

#### 暂存区 (Index/Staging Area)

| 属性 | 值 |
|:-----|:----|
| **稳定性** | 🟡 中 — 可暂存取消（`git restore --staged`） |
| **可见性** | 👤 仅本地 — 但在 `git status` 中可见 |
| **持久性** | ❌ 不保证 — 暂存区可清空 |
| **区内操作** | `git add`, `git reset`, `git restore --staged` |
| **意义** | 隔离「我选了哪些变更去提交」— 可分多次 add 最后一次 commit |

#### 本地仓库 (Local Repository)

| 属性 | 值 |
|:-----|:----|
| **稳定性** | 🟢 高 — commit 不可变（可 amend/rebase 但需显式操作） |
| **可见性** | 👤 仅本地 — 但可推送到远程 |
| **持久性** | ✅ 保证 — 除非显式 gc 删除 |
| **区内操作** | `git commit`, `git log`, `git branch` — 纯本地 |
| **意义** | 隔离「已定稿的变更」和「还在犹豫的变更」 |

#### 远程仓库 (Remote Repository)

| 属性 | 值 |
|:-----|:----|
| **稳定性** | 🟢🟢 最高 — force push 需显式且受保护 |
| **可见性** | 🌐 团队/公开 — 每个人都能看到 |
| **持久性** | ✅✅ 最高保证 — 备份/容灾 |
| **准入** | 🔑 需认证 — push/pull 需权限 |
| **区内操作** | `git push`, `git pull`, `git fetch`, PR/MR |
| **意义** | 隔离「我的提交」和「团队的提交」 |

### 3.3 跨区交互协议

Git 的精妙之处在于：**四个区之间的跨区操作全部需要通过显式的 Git 指令**，没有隐含的自动流转。

```
工作目录 ──git add──→ 暂存区
   ↑                      │
   └──git restore──←──────┘
   ────────────── 不可直接转仓库 ──────────────

暂存区 ──git commit──→ 本地仓库
   ↑                      │
   └──git reset --soft───←┘

本地仓库 ──git push────→ 远程仓库
   ↑                      │
   └──git pull/fetch──←───┘
```

| 操作 | 涉及的区 | 方向 | 允许失败？ | 备注 |
|:-----|:---------|:-----|:----------|:-----|
| `git add` | WD → Index | 前向 | ✅ 可任意反复 | 将文件当前快照放入暂存区 |
| `git restore` | Index → WD | 后向 | ✅ | 用暂存区覆盖工作目录 |
| `git commit` | Index → Local | 前向 | ✅ 可多次提交 | 创建不可变快照 |
| `git reset --soft` | Local → Index | 后向 | ✅ | 撤销 commit 但保留内容在暂存区 |
| `git push` | Local → Remote | 前向 | ❌ 网络问题可能失败 | 需要远程认证+网络 |
| `git pull/fetch` | Remote → Local | 后向 | ❌ 网络问题可能失败 | 从远端拉取最新变更 |

**为什么 Git 要设计四个区而不是一个？**

这是关键洞察：四个区对应了**四条隔离线**，每一条解决一个实际问题：

| 隔离线 | 解决的问题 | 不隔离的后果 |
|:-------|:-----------|:-------------|
| WD vs Index | 只提交我**选择**的变更，不是所有变更 | 每次 commit 伴随所有已改但未完的文件 |
| Index vs Local | 只有**显式冻结**的变更才生成版本 | 无法区分"正在改"和"已定稿" |
| Local vs Remote | 只在**准备好发布**时才推向团队 | 破坏主干的频率急剧升高 |

**这是分区设计的关键模式：每一条边界 = 一个决策点**。过少的分区→一次性做太多决策→质量下降。过多的分区→决策成本太高→流程瘫痪。

Git 选择恰好四条边界，与分布式系统管理/数据面的两条边界、Docker 的三层体系形成鲜明对比——见 [第7章](#7-跨域模式抽象通用分区三轴模型) 的定量分析。

## 4. 域三：虚拟化——Host↔Guest 隔离体系

### 4.1 分区模型

虚拟化将一台物理机器（同构硬件的"一个区域"）划分为多个独立执行环境：

```
┌───────────────────────────────────────────────────────┐
│  Host (Hypervisor / VMM)                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐   │
│  │  Guest VM 1 │  │  Guest VM 2 │  │  Guest VM N │   │
│  │  ┌─────────┐│  │  ┌─────────┐│  │  ┌─────────┐│   │
│  │  │ App     ││  │  │ App     ││  │  │ App     ││   │
│  │  ├─────────┤│  │  ├─────────┤│  │  ├─────────┤│   │
│  │  │ Guest   ││  │  │ Guest   ││  │  │ Guest   ││   │
│  │  │ Kernel  ││  │  │ Kernel  ││  │  │ Kernel  ││   │
│  │  ├─────────┤│  │  ├─────────┤│  │  ├─────────┤│   │
│  │  │ vCPU    ││  │  │ vCPU    ││  │  │ vCPU    ││   │
│  │  │ vMem    ││  │  │ vMem    ││  │  │ vMem    ││   │
│  │  │ vDev    ││  │  │ vDev    ││  │  │ vDev    ││   │
│  │  └─────────┘│  │  └─────────┘│  │  └─────────┘│   │
│  └─────────────┘  └─────────────┘  └─────────────┘   │
│                                                       │
│  ┌─────────────────────────────────────────────┐      │
│  │  Hardware (CPU/MMU/IOMMU/PIC/HPET/... )     │      │
│  └─────────────────────────────────────────────┘      │
└───────────────────────────────────────────────────────┘
```

### 4.2 各区域定位与规则

#### Host 区（Hypervisor / VMM / Root Mode）

| 属性 | 值 |
|:-----|:----|
| **权限等级** | **Ring -1**（硬件最高优先级），可执行所有指令 |
| **职责** | 资源调度、隔离、虚拟设备仿真、VM 生命周期管理 |
| **内部操作** | 直接管理物理硬件、分配 vCPU slice、映射 vMem→pMem |
| **稳定性** | 🟢 最高 — Host 崩溃 = 所有 VM 崩溃 |
| **安全模型** | 从硬件 Root of Trust 开始，经过 Secure Boot→Measured Boot |

#### Guest 区（VM / Non-Root Mode / Ring 0-3）

| 属性 | 值 |
|:-----|:----|
| **权限等级** | **Ring 0-3** — Ring 0 看似有全部权限实则受限（由 hypervisor 捕获敏感指令） |
| **职责** | 运行用户操作系统与应用 |
| **内部操作** | 正常执行指令，不知道自己被虚拟化（半虚拟化除外） |
| **稳定性** | 🟡 中 — Guest OS 崩溃不会影响 Host 和其他 VM |
| **安全模型** | Guest 以为自己在独占硬件 |

### 4.3 跨区交互协议

虚拟化的跨区交互是**最昂贵的**跨区操作之一：

| 协议 | 方向 | 触发条件 | 代价 | 优化 |
|:-----|:-----|:---------|:-----|:-----|
| **VMExit** | Guest→Host | Guest 执行敏感指令/中断/IO/页故障 | **1000-10000 cycles** | 硬件辅助(Intel VT-x/AMD-V)减少→300-1000 cycles |
| **VMEntry** | Host→Guest | 事件处理完返回 | 300-1000 cycles | — |
| **Hypercall** | Guest→Host | 主动请求服务（半虚拟化） | 1000-5000 cycles | PV drivers 替代 MMIO 陷阱 |
| **EPT Violation** | Guest→Host | 访存涉及未映射的 GPA→HPA | **10000-100000 cycles** | 大页(2MB/1GB)、PPN pre-population |
| **DMA Remap** | Device→Guest | 设备直通 DMA 操作 | 硬件级 | IOMMU + VT-d 硬件隔离 |
| **Posted Interrupt** | Host→Guest | 中断注入 | 200-500 cycles | 硬件加速中断投递 |

**关键工程数据**：

```
VMExit 代价拆解（Intel Skylake, ~cycle count）:
  ├─ 退出原因捕获:     50-100
  ├─ VMCS 状态保存:    200-400
  ├─ Host 上下文恢复:  300-500
  ├─ 事件处理:         100-5000 (取决于复杂度)
  └─ Total:           1000-10000

频繁 VMExit 场景（必须避免）:
  - 每个 IO 操作 → 网络流量大时每秒数十万次 VMExit
  - 解决方案: virtio 批量投递 / VFIO 设备直通
```

### 4.4 虚拟化分区的核心学理

与 Git 或 Docker 不同，虚拟化的分区**不是逻辑分区，而是硬件强制的物理分区**：

1. **硬件级强制**：Intel VT-x/AMD-V 在 CPU 内部实现 VMX non-root mode，Guest 根本逃不出
2. **内存隔离**：EPT (Extended Page Tables) 在 MMU 层面做二层地址转换，Guest 物理地址→Host 物理地址映射由 Host 独占控制
3. **IO 隔离**：IOMMU (VT-d) 实现 DMA 重映射，Device→Memory 访问通过 IOMMU 做地址翻译和权限检查

**分区深度对比**：

| 维度 | Git | K8s | Docker | 虚拟化 |
|:-----|:----|:----|:-------|:-------|
| 分区粒度 | 文件修改 | 进程/网络 | 文件系统+进程 | CPU+内存+IO+整机 |
| 隔离强度 | 逻辑层 | 逻辑层+网络 | 逻辑命名空间 | **硬件层** |
| 跨区代价 | 0.01ms | 1-10ms(网络) | 1ms(namespace切换) | 1-100μs(VMExit) |
| 安全保证 | 无 | RBAC | seccomp+Capability | **硬件隔离** |

## 5. 域四：Docker——Build→Ship→Run 三级分区

### 5.1 三层模型

Docker 是 Git 分区思想在应用部署领域的映射：

```
┌──────────────────────┐    ┌──────────────────────┐
│     Registry          │    │     Image             │
│    （远程仓库）       │    │   （可运输镜像）      │
│   docker.io/hub       │◄──►│   docker images      │
│   Harbor, ECR...      │    │   ┌──────────┐       │
│                        │    │   │ Layer 1  │       │
│   🔒 需认证            │    │   │ Layer 2  │       │
│   📡 网络可达          │    │   │ Layer N  │       │
│   💾 持久/备份         │    │   └──────────┘       │
│                        │    │                      │
│   🟢 不可变            │    │   🟢 不可变(只增)    │
└──────────────────────┘    └──────────────────────┘
         │ push/pull                │ run
         ▼                          ▼
┌─────────────────────────────────────────────────────┐
│                    Container                         │
│   ┌─────────────────────────────────────────────┐   │
│   │  Container Layer (可写层)                    │   │
│   │   临时文件、运行时状态、修改                  │   │
│   │   🔴 可变  |  👤 仅本机  |  ❌ 不持久       │   │
│   ├─────────────────────────────────────────────┤   │
│   │  Image Layers (只读层)                      │   │
│   │   🟢 不可变 |  ♻️ 共享 |  📦 可复用        │   │
│   └─────────────────────────────────────────────┘   │
│                                                      │
│   🔴 可变  |  👤 本机  |  ❌ 默认不持久             │
└─────────────────────────────────────────────────────┘
```

### 5.2 各区定位与规则

#### Registry（镜像仓库）

| 属性 | 值 |
|:-----|:----|
| **角色** | 镜像持久化存储+分发中心 |
| **稳定性** | 🟢🟢 最高 — 镜像上传后不可变更（tag 可覆盖） |
| **可见性** | 🌐 公/私 — 取决于 Registry 配置 |
| **持久性** | ✅✅ — 长期存储（配合 GC 策略） |
| **准入** | 🔑 — 需 `docker login` |
| **区内操作** | 管理台查看/删除/GC，Webhook 触发 |

#### Image（本地镜像层）

| 属性 | 值 |
|:-----|:----|
| **角色** | 应用快照——只读层栈 |
| **稳定性** | 🟢 高 — 每层 SHA256 标识，内容不可变 |
| **可见性** | 👤 本机 — 其他节点需 pull |
| **持久性** | ✅ — 存于 `/var/lib/docker/overlay2/` |
| **区内操作** | `docker images`, `docker rmi`, 强制删除 |

#### Container（容器运行时）

| 属性 | 值 |
|:-----|:----|
| **角色** | 应用的运行实例 |
| **稳定性** | 🔴 最低 — 可任意写 |
| **可见性** | 👤 本机 — 端口映射后可达 |
| **持久性** | ❌ — `docker rm` 后所有可写数据丢失 |
| **区内操作** | `docker exec`, 文件写入, `docker stop/start` |

### 5.3 跨区交互协议

| 操作 | 涉及区域 | 方向 | 频度 | 代价 |
|:-----|:---------|:-----|:-----|:-----|
| `docker pull` | Registry→Local Image | 前向 | 低（部署时） | 网络传输大小，慢 |
| `docker push` | Local Image→Registry | 前向 | 低（CI/CD） | 同上 |
| `docker run` | Image→Container | 前向 | 中 | 秒级（创建可写层） |
| `docker commit` | Container→Image | 后向 | 极低（不建议） | 生成新镜像 |
| Volume mount | Host→Container | 旁路 | — | 共享文件系统 |

### 5.4 Docker 分区的核心学理

Docker 最大创新不是容器技术本身（LXC 早在 2008 年就存在），而是**三级分区模型**解决了 Git 的一级问题——"在本地跑起来了怎么部署到生产"：

```
Git → 
  Build 环境: 工作目录 → 本地仓库
  Deploy 环境: 本地仓库 → 远程仓库
  ⚠️ 问题: 代码能部署，环境不能

Docker →
  Build: Dockerfile → Image (打包环境+代码)
  Ship:  Image → Registry (传输环境+代码)
  Run:   Image → Container (运行环境+代码)
  ✅ 解决了: "在我的机器上能跑" → "在任何机器上都能跑"
```

这也是**分区设计的跨域模式**：当系统和外部世界接触时，需要引入新的分区来解决从前不存在的隔离需求。

## 6. 域五：存储层级——速度/容量/成本五层权衡

### 6.1 五层模型

计算机体系结构中最经典的分区——存储金字塔：

```
                   速度         容量/成本        功能
         ┌──────┐ 最快(0.3ns)  最小(几百B)    CPU寄存器
         │ Reg  │  🔴 可变      ❌ 不持久     线程私有
         ├──────┤
         │ L1   │  1ns          32-64KB       核私有
         │      │  🟡 透明      ❌ 不持久     HW管理
         ├──────┤
         │ L2   │  3-5ns        256-512KB     核私有/共享
         │      │  🟡 透明      ❌ 不持久     HW管理
         ├──────┤
         │ L3   │  10-20ns      8-64MB       所有核共享
         │      │  🟡 透明      ❌ 不持久     HW管理
         ├──────┤
  ───────┤ RAM  ├────────────────────────────────────────
         │      │  50-100ns     8GB-2TB      进程共享
         │ RAM  │  🟡 直接      ❌ 不持久(断电丢)
         │      │  字节寻址     🔑 OS管理
         ├──────┤
         │ SSD  │  10-100μs    256GB-30TB    文件系统
         │      │  🟢 持久      ✅ 长期
         │ NAND │  块寻址       🔑 FS管理
         ├──────┤
         │ HDD  │  5-20ms      1-24TB        归档
         │      │  🟢 持久      ✅ 长期
         │      │  块寻址       💰 最便宜(¥0.1/GB)
         ├──────┤
         │ Tape │  秒级         PB级          备份归档
         │      │  🟢 持久      ✅ 数十年
         │      │  顺序访问     💰 ¥0.01/GB
         └──────┘
```

### 6.2 各层定位与规则

#### L1-L3 Cache（CPU Cache）

| 属性 | 值 |
|:-----|:----|
| **角色** | CPU 与 RAM 之间的速度缓冲 |
| **管理主体** | **纯硬件**（Cache Controller + MESI/MESIF协议） |
| **区内操作** | CPU Load/Store，Cache Line 填充/驱逐 |
| **透明性** | 🟡 对软件完全透明 — 写代码时看不见 |
| **一致性协议** | MESI/MOESI/MESIF（跨核/跨Socket） |
| **持久性** | ❌ 断电丢失 |

**规则**：软件无法直接控制 cache line 的预取/驱逐（除极少数 prefetch 指令和非临时访存 NT 指令）。

#### RAM（主存/DRAM）

| 属性 | 值 |
|:-----|:----|
| **角色** | 进程的虚拟地址空间载体 |
| **管理主体** | **OS + MMU** — 通过页表管理虚拟→物理映射 |
| **区内操作** | 进程 Load/Store，malloc/free，缺页异常 |
| **透明性** | 不透明 — 进程间地址空间隔离（❗ 又一个分区） |
| **持久性** | ❌ 断电丢失 |

#### SSD/HDD（持久存储）

| 属性 | 值 |
|:-----|:----|
| **角色** | 文件系统/数据库的持久化载体 |
| **管理主体** | **OS + 文件系统/DB + 存储控制器** |
| **区内操作** | open/read/write/close, 文件锁, r/w offset |
| **透明性** | 对用户程序不透明 — 需通过 syscall |
| **持久性** | ✅ 断电保留 |

### 6.3 跨区交互协议

存储层级间跨区交互的代价差异是**多个数量级**：

```
     延迟(μs)          带宽(GB/s)         跨区操作
Reg    0.0003 ↑↑↑       ---               CPU内部
L1     0.001  ↑↑↑       1000+             Load/Store
L2     0.004  ↑↑        500+              Load/Store
L3     0.015  ↑↑        300+              Load/Store
RAM    0.1    ↑         50-200            Load/Store + TLB Miss
↓                                              ↓
SSD    20-100 ↓         0.5-7             page cache miss → page fault
HDD    5000   ↓↓        0.1-0.3           fs read/write
Tape   10^7   ↓↓↓       0.001             backup/restore (全量)
```

**关键交互协议**：

| 协议 | 涉及的层 | 方向 | 代价 | 触发条件 |
|:-----|:---------|:-----|:-----|:---------|
| **Cache Line Fill** | L3←RAM | 后向 | ~100ns | Cache Miss（Load/Store） |
| **Cache Writeback** | RAM←L3 | 前向 | ~50ns | 脏行被驱逐 / 一致性协议 |
| **Page Fault** | RAM←SSD | 后向 | 10-100μs | 缺页/匿名页首次访问 |
| **Swap In/Out** | RAM↔SSD | 双向 | 50μs-5ms | 内存压力/系统调用 |
| **mmap** | RAM↔SSD | 按页映射 | μs级(建立映射) | mmap() 系统调用 |
| **Page Cache Read** | RAM←SSD | 后向 | 10-100μs | read() 触发缺页 |
| **DMA** | Device↔RAM | 直接 | 硬件异步 | NVMe/网卡介入 |

### 6.4 存储分区的根本动力：速度-容量-成本的不可能三角

```
         速度（快）
           🔺
          / \
         /   \
        /     \
       /       \
容量大 ──────── 成本低
```

没有一种存储技术能同时做到：**极快 + 极大 + 极便宜**。

这就是存储层级必须分区的根本原因——它不是一个设计选择，而是**物理定律强制**的：

| 技术 | 最快 x 器件(单个) | 最大 x 器件(单个) | 最便宜 x 器件 |
|:-----|:-----------------|:-----------------|:--------------|
| SRAM (Cache) | **~0.3ns** ✅ | 144KB per bank | ~$5000/GB ❌ |
| DRAM (RAM) | ~50ns | 128GB per DIMM | ~$5/GB |
| NAND (SSD) | 10μs | 128TB per U.2 | ~$0.05/GB ✅ |
| HDD | 5ms | 24TB per 3.5" | ~$0.01/GB ✅✅ |
| Tape | 秒级 | 60TB per cartridge | ~$0.001/GB ✅✅✅ |

**跨区操作的根本代价来自这里**：每次跨层访问，代价 = 上一层的高速+低带宽与下一层的低速+高带宽之间的**延迟差**。这就是为什么 Cache Line Fill 才 100ns 而磁盘缺页 10ms——差了 100,000 倍。

## 7. 跨域模式抽象：通用分区三轴模型

### 7.1 抽象问题

五个领域的内部分区数量、隔离维度和跨区代价各不相同，但都遵循同一个底层结构。现在提取通用模型。

### 7.2 Zoning Triad：分区的三元组

任何一个有效分区，由三个要素构成：

```
Zoning Triad = (Boundary, Internal-Rules, Cross-Protocol)
     ↑              ↑           ↑               ↑
     分区的三个维度  边界定义    区内自治规则   跨区交互协议
```

#### 要素一：Boundary（边界）

边界定义了**什么在区内、什么在区外**。边界质量由以下属性衡量：

| 属性 | 含义 | 强边界示例 | 弱边界示例 |
|:-----|:-----|:-----------|:-----------|
| **清晰度** | 边界是否精确可判定 | Git 4个区，`git status` 精确显示 | 微服务间的"团队归属"（模糊） |
| **强制力** | 能否绕过边界 | VMware VT-x 硬件隔离（不可绕过） | 权限系统（可被提权绕过） |
| **粒度** | 边界内的对象大小 | 存储 Layer：Cache Line(64B) | 分布式：Node（物理机器） |
| **动态性** | 边界能否运行时改变 | K8s Node 加入/离开（动态） | CPU 核数（固定） |

#### 要素二：Internal-Rules（区内规则）

区内规则定义了该区域内对象的**生命周期操作**：

| 规则类型 | 含义 | 示例 |
|:---------|:-----|:-----|
| **创建规则** | 对象如何在区内创建 | `vm.create()`、`docker run`、`git commit` |
| **修改规则** | 区内对象可变吗 | Container 层可写，Image 层不可变 |
| **销毁规则** | 如何移除对象 | `docker rm`、`git branch -D`、`kubectl delete pod` |
| **可见规则** | 谁能看到区内对象 | 工作目录仅本地，远程仓库全局 |
| **持久规则** | 区内对象是否持久 | etcd 持久，Pod 不持久 |
| **权限规则** | 谁能操作区内对象 | RBAC、file permissions、kernel/user mode |

#### 要素三：Cross-Protocol（跨区协议）

跨区协议定义了**对象如何从一个区迁移到另一个区**：

| 属性 | 含义 | 同步/异步 | 示例 |
|:-----|:-----|:----------|:-----|
| **方向性** | 单向还是双向 | 双向 | git push/pull |
| **原子性** | 操作是否全部成功或不成功 | 要求原子 | git commit（全或全不） |
| **一致性** | 跨区后的状态一致性保证 | 最终/强 | K8s Reconcile Loop（最终） |
| **代价** | 跨区操作的资源开销 | CPU/时间/网络 | VMExit 1000 cycles |

### 7.3 Stability-Visibility-Mutability 三轴坐标系

将五个域的所有分区绘制在同一个三维坐标系中——**三轴足以区分所有的区**：

```
M = Mutability (可变性)
S = Stability   (稳定性/持久性)
V = Visibility  (可见性/共享度)

              M(高)
               │
          Container ● Git WD
               │
      ● Guest VM│
               │
               ├────── V(高)
              /│
             / │
            /  │
   S(高)──●──  │
   Remote Repo  │
              ● │
           Image│
               │
          ● L3 Cache
               │
          ● Reg
              M(低)
```

以[M, V, S]三元组量化各区的坐标（1=低，10=高）：

| 域 | 区 | M(可变性) | V(可见性) | S(稳定性/持久性) | 语义 |
|:---|:---|:---------:|:---------:|:---------------:|:-----|
| **分布式** | 管理面(etcd) | 1 | 10 | 9 | 全局状态，稳定不可轻易改 |
| | 数据面(Node) | 8 | 2 | 3 | 频繁变，仅自己看，不持久 |
| **Git** | 工作目录 | 10 | 1 | 1 | 随便改，自己看，不保证 |
| | 暂存区 | 7 | 3 | 2 | 可改可取消，本地缓冲 |
| | 本地仓库 | 1 | 4 | 8 | 不可变，本地备份 |
| | 远程仓库 | 1 | 10 | 10 | 不可变，全网可见，全量备份 |
| **虚拟化** | Host | 2 | 9 | 10 | 极少变，可见全部，持久 |
| | Guest | 7 | 3 | 5 | 高频变，仅自己可见，快照持久 |
| **Docker** | Registry | 1 | 8 | 10 | 不可变，组织可见，持久 |
| | Image | 2 | 4 | 9 | 只增，本地，持久 |
| | Container | 10 | 2 | 1 | 可写，仅本机，不持久 |
| **存储** | Cache(L1-3) | 8 | 3 | 1 | 高频写，仅本核(局部)，不持久 |
| | RAM | 7 | 6 | 2 | 程序可写，进程可见，不持久 |
| | SSD | 3 | 7 | 8 | 低频写，全局可见，持久 |
| | HDD | 1 | 8 | 10 | 极少写，全局可见，高持久 |

**三轴坐标系的核心洞察**：

1. **没有任何两个不同的区在[M,V,S]上完全相同** → 三轴足以唯一标识任何分区系统中的任何区域
2. **同域内相邻区之间至少有两个轴的变化**（如 Git WD→Index：M下降3，V上升2，S上升1）
3. **跨域的"相同角色"有相似的[M,V,S]模式**（如"最稳定的区"总是S≥9,V≥8,M≤1）

### 7.4 分区数量与复杂度的关系

不同域的区数不同（Git=4, Docker=3, K8s=2+etc, 存储=6+），选择多少分区是核心工程决策：

```
分区数与系统复杂度的关系:
                                                      复杂度过高
                                                      (维护成本爆炸)
                                                      ●
                                                    ↗
                                                  ●
                                                ↗
                    ┌── 甜点区 ──┐              ●
                  ●           ●               ↗
                ↗           ↗                ●
    Git ●───●───●───●     ●               ↗
        (2区)     (4区)  ●              ●
                      K8s etc.       复杂度过低
                      (3个主区)        (功能区混在一起)
                                  
   2       3       4       5       6       7+      分区数
```

**经验法则（来自五域的归纳）**：

| 系统类型 | 最佳分区数 | 依据 |
|:---------|:---------|:-----|
| **个人开发工具** | 3-4 | Git、Docker 验证 |
| **分布式系统** | 2-3(主分区) | K8s(2)、etcd(1)、ZooKeeper(2) |
| **操作系统** | 3-5 | 用户态/内核态(2) + 进程空间(n) |
| **存储系统** | 4-6 | 缓存→主存→持久→存档 |

更多分区不是更好——每多一个分区多一条边界(Cross-Protocol)的维护成本，参见 [第8章](#8-跨区交互协议的分类与设计)。

## 8. 跨区交互协议的分类与设计

### 8.1 四类跨区协议

从五个领域归纳，所有跨区交互可抽象为四种基本类型：

#### 类型A：同步单向推送（Push）

```
Sender ────→ Receiver
```

一方主动推送数据/状态到另一方，发送方等待响应。

| 域 | 示例 | 同步/异步 | 容错 |
|:---|:-----|:---------|:-----|
| 分布式 | kubelet Report→API Server | 同步 | 失败重试+心跳超时 |
| Git | git push Local→Remote | 同步 | 网络失败报错，手动重试 |
| Docker | docker push Image→Registry | 异步(层并行) | 断点续传 |

**设计要点**：Push 协议需要在发送失败时有明确的退避策略，避免"惊群效应"（所有 Node 同时重试压垮 API Server）。

#### 类型B：同步单向拉取（Pull）

```
Sender ←─── Receiver
```

一方主动从另一方拉取数据。

| 域 | 示例 | 同步/异步 | 容错 |
|:---|:-----|:---------|:-----|
| 分布式 | GitOps Controller Pull | 周期轮询 | 指数退避 |
| Git | git pull Remote→Local | 同步 | 网络失败报错 |
| 存储 | Page Cache Read SSD→RAM | 同步阻塞 | 内核重试 |

**设计要点**：Pull 的频率是核心参数——拉的太勤浪费资源，拉的太少延迟增加。分布式系统通常用 Watch 替代 Pull（见下）。

#### 类型C：异步事件订阅（Event/Stream）

```
Subscriber ← EventBus ← Publisher
```

发布方推送事件，订阅方异步消费。

| 域 | 示例 | 同步/异步 | 一致性 |
|:---|:-----|:---------|:-------|
| 分布式 | K8s Watch API | 异步 | 最终一致（Relist on reconnect） |
| 分布式 | etcd Watch | 异步 | 线性一致（Revision 可溯源） |
| 分布式 | Kafka 消息队列 | 异步 | ACK 级别可选 |

**设计要点**：Event 订阅是 Push 的异步升级版——解决了"重试风暴"问题，但引入了事件顺序/重放/丢失的问题。

#### 类型D：共享状态/内存映射（Shared）

```
┌──────────────────────┐
│  Shared State        │
│  (内存/文件/协调服务) │
└──────┬─────────┬─────┘
       │         │
  Reader1    Writer2
```

多个区域通过共享的状态空间间接交互。

| 域 | 示例 | 同步/异步 | 锁机制 |
|:---|:-----|:---------|:-------|
| 分布式 | etcd — 多个 Watch 共享状态 | 异步 | Lease + Revision |
| 虚拟化 | Shared Memory (ivshmem) | 直接 | Gate/Semaphore |
| 存储 | mmap — 用户态↔内核态共享 | 直接 | OS 页保护 |
| Docker | Volume — Host 与 Container 共享 | 直接 | 文件锁 |

### 8.2 跨区操作的通用代价模型

```
Cross-Zone Cost = Setup + Transfer + Transform + Verify
                  建立连接  传输数据  格式转换  校验/同步
```

| 域 | Setup | Transfer | Transform | Verify | 总代价量级 |
|:---|:------|:---------|:----------|:-------|:----------|
| **Git push** | SSH/TLS握手(1-10ms) | 传输diff(1ms-10s) | 对象打包(ms级) | SHA256校验 | **10ms-10s** |
| **K8s Watch** | 长连接建立(1-3ms) | WS/HTTP2流 | 序列/反序列(μs) | ResourceVersion | **1-5ms/事件** |
| **VMExit** | CPU 模式切换 | 状态保存/恢复 | — | VMCS 校验 | **1-10μs** |
| **Docker pull** | Registry认证(10ms) | 层下载(1s-10min) | 解压(tar+gzip) | SHA256校验 | **1s-15min** |
| **Page Fault** | 缺页中断 | 从SSD读页(4KB) | — | 无 | **10-100μs** |
| **Cache Fill** | Miss 检测 | RAM→Cache | — | 无 | **~100ns** |

**核心规律**：跨区代价与**分区间距离**（物理+逻辑）成正比。存储层级每次跨区代价相差约 10 倍——这就是为什么 L1 Miss→L2 的代价远小于 Page Fault。

### 8.3 跨区协议的六大设计原则

从五域共性提取：

| # | 原则 | 含义 | 违反的后果 | 示例 |
|:-:|:-----|:-----|:-----------|:-----|
| **P1** | **可重试** | 跨区操作必须幂等或可重试 | 数据不一致/重复提交 | K8s Reconcile Loop |
| **P2** | **有超时** | 每个跨区操作设超时 | 死锁/级联等待 | gRPC Deadline |
| **P3** | **有回退** | 失败后必须明确的行为(重试/报错/阻塞) | 沉默失败→错误扩散 | Docker pull 失败报错 |
| **P4** | **原子性** | 跨区操作要么全成功要么无效 | 部分更新→状态分裂 | Git commit 全/不 |
| **P5** | **可见性** | 跨区操作的效果何时对其他区可见 | 可见性违反→读取旧数据 | etcd Revision |
| **P6** | **带宽匹配** | 跨区操作的频率必须匹配两端处理能力 | 接收端过载/发送端饥饿 | K8s Watch 合并事件 |

## 9. 分区设计原则（方法论固化）

### 9.1 八条设计原则

从五个领域的交叉分析中提炼，这是本文最核心的方法论贡献：

#### 原则 1：边界的强度由冲突的强度决定

**陈述**：分区边界的强制力（是否可绕过、代价多高），应该与分区隔离要解决的冲突强度正相关。

**推导**（第一性原理）：如果两个区之间的冲突很弱（如不同开发者的代码风格差异），一个软边界（如 CI 检查）就足够。如果冲突很强（如恶意 VM 攻击 Host），则需要硬件级隔离。

**五域验证**：

| 冲突强度 | 边界类型 | 域 |
|:---------|:---------|:---|
| 🟢 低（编码规范差异） | 逻辑规则(CI lint) | Git PR流程 |
| 🟡 中（资源争用） | 操作系统调度 | K8s QoS |
| 🔴 高（安全隔离） | 命名空间/Seccomp | Docker |
| 🔴🔴 极高（恶意代码） | 硬件隔离(VMX) | 虚拟化 |
| 🔴🔴🔴 极极高（信息泄露） | 物理隔离(气隙) | 涉密系统 |

#### 原则 2：区内自治，跨区协议

**陈述**：每个区内部的规则由该区独立决定（区内自治），跨区的规则通过显式协议约定（不可隐含）。

**推导**：区内自治降低耦合度——如果 Git 每次文件修改都要通知远程仓库，那就不是版本控制而是同步网盘了。

**违反案例**：早期的分布式锁通过"假设网络可靠"隐含约定 → 脑裂。解决方法：隐式约定显式化（Lease + 重协商）。

#### 原则 3：显式边界，隐式实现

**陈述**：边界必须被显式定义（API、接口、地址空间），但边界内部的实现机制对区外不透明。

**推导**：这是信息隐藏原则在分区维度的推广。

**验证**：

| 域 | 显式边界 | 隐式实现 |
|:---|:---------|:---------|
| Git | 工作目录↔暂存区（`git add` 命令） | 暂存区用 `.git/index` 实现 |
| Docker | Image↔Container（`docker run` 命令） | OverlayFS 用 lower/upper dir 实现 |
| 虚拟化 | Host↔Guest（VMEntry/VMExit 指令） | VT-x 在 CPU 微码实现 |
| 存储 | RAM↔SSD（缺页异常） | OS 页面置换算法 LRU/Clock |

#### 原则 4：稳定性向外递增

**陈述**：在一个分层系统中，从核心向外的每个区域，其内容/状态的稳定性应该递增。

**推导**：内部区域面向频繁的试验/修改（高可变性），外部区域面向稳定的发布/存储（低可变性）。

**五域验证**：

```
域     内部(高可变)                   外部(高稳定)
─     ──────────                    ──────────
Git:  工作目录 → 暂存区 → 本地仓库 → 远程仓库   (稳定性递增)
Docker:  Container → Image → Registry          (稳定性递增)
存储:  Cache → RAM → SSD → HDD                  (稳定性递增)
K8s:   Pod(频繁启停) → 控制面(稳定)             (稳定性递增)
```

📌 **违反该原则的后果**：如果把稳定性高的数据放在可变区（如正在调试的代码直接 commit 到 master），或者把可变区的东西固化到稳定区（如把临时文件提交到 Image），都会导致系统混乱。

#### 原则 5：跨区操作的频率决定了分区粒度

**陈述**：如果两个实体需要频繁交互，它们应该在同一区域内；如果交互频率低，适合分成两个区。

**推导**：跨区操作有代价（[第8.2节](#82-跨区操作的通用代价模型)），如果交互频率 * 跨区代价 > 区内协作代价，分区就是次优的。

**示例**：

| 场景 | 交互频率 | 最佳分区方案 | 依据 |
|:-----|:---------|:-------------|:-----|
| CPU 与 L1 Cache | 每个指令 | **同一核内** | ns 级延迟要求 |
| K8s Scheduler 与 API Server | 每个 Pod 一次 | **分开**（控制面 vs 数据面） | Scheduler 失败不影响已运行 Pod |
| 数据库主从复制 | 每秒次 | **分开**（主库分片） | 写入延迟可接受 |
| 微服务 A 与 B | 每个请求 | 同一集群 | 网络延迟可接受 |

#### 原则 6：每一个分区都是一个决策点

**陈述**：从一个区到另一个区的每一次跨越，都是一个必须被显式做出的决策。分区数量 = 决策机会数量。

**推导**：如果系统没有分区，所有决策隐式在同一个上下文完成，容易遗漏或推迟关键判断。

**验证**：

```
Git 中:
  工作目录 → 暂存区: 决策"哪些变更我准备好了？"
  暂存区 → 本地仓库: 决策"这个快照值得保存吗？"
  本地仓库 → 远程仓库: 决策"这个版本可以给团队看了吗？"

如果 Git 只有一个区（像 Dropbox 同步）:
  → 每次文件保存都自动同步到团队
  → 无法区分"正在改的"和"准备发布的"
  → 冲突频率激增
```

#### 原则 7：分区=加速，分区=减速

**陈述**：分区在内部加速（并行操作不冲突），在边界减速（跨区协议的开销）。

**推导**：

```
系统总吞吐 = ∑ 各区内部吞吐 - ∑ 跨区开销
```

当分区增加时，内部吞吐增加（并行度提高），但跨区开销也增加。最优分区数 = 边际内部增益 = 边际跨区损失的点。

**工程含义**：不要**过度分区**。每个微服务都是一个独立分区，如果两个服务之间交互频率极高，它们其实应该分在同一个区。

#### 原则 8：一致性模型由分区类型决定

**陈述**：跨区的数据一致性保证，取决于边界的强制力。

| 边界类型 | 可达的一致性 | 示例 |
|:---------|:------------|:-----|
| 硬件强制（VT-x） | 无条件——根本跨不过去 | VM 逃逸不可能 |
| 硬件辅助（Cache MESI） | 强一致性 | 缓存一致性协议保证 |
| 操作系统（进程地址空间） | 进程隔离+IPC | fork 写时复制 |
| 网络（分布式系统） | 最终一致性 | K8s Reconcile |
| 逻辑（Git 远程仓库） | 乐观一致性 | 冲突时 merge |

## 10. 反模式与常见陷阱

### 10.1 自毁分区

**症状**：系统内部有清晰的边界，但提供了绕过边界的"后门"。

**示例**：
- `git commit --no-verify` 跳过 CI — 绕过了暂存区→本地仓库的验证决策点
- Docker `--privileged` 容器 — 绕过了 Container 区与 Host 区的安全隔离
- `kubectl delete pod --force --grace-period=0` — 绕过了正常的 Pod 生命周期

**修复**：后门必须有审计日志，且只能由高级管理员使用。

### 10.2 分区漂移

**症状**：分区边界随时间的推移发生无意识偏移，导致"这个应该属于哪个区"的歧义。

**示例**：
- Git monorepo 中，本应放在不同仓库的模块混在同一个仓库（分区坍缩）
- K8s 中本应属于不同 Namespace（软分区）的资源混在一起
- 微服务中两个本应独立的服务开始直接调用对方的数据库（分区穿透）

**修复**：定期检查边界是否被侵蚀，建立架构治理。

### 10.3 隔间隧道（Silo Tunnel）

**症状**：分区之间的 Cross-Protocol 过于复杂，开发者发明"捷径"来绕过协议。

**示例**：
- Git: 开发者在远程仓库直接修改文件（绕过工作目录/暂存区/本地仓库）
- Docker: 在运行中的 Container 里安装软件然后 commit 成新镜像（绕过 Dockerfile 构建流程）
- 分布式: 运维人员直接修改 etcd 数据（绕过 API Server）

**修复**：如果跨区协议太复杂，说明分区过多或协议设计有问题——简化协议比增加捷径更有效。

### 10.4 边界黑洞

**症状**：分区边界没有清晰的失败处理，跨区操作失败后"无声无息"或被忽略。

**示例**：
- Git push 失败后没有提醒，开发者以为已推送
- K8s Reconcile 循环静默失败，期望状态与实际状态长期不一致
- Docker pull 超时后没有回退

**修复**：[第8.3节](#83-跨区协议的六大设计原则)的 P1-P6 原则必须严格遵守。

## 11. 工程应用：如何设计一个分区系统

### 11.1 分区构建五步法

**步骤①：识别冲突维度**

列出系统中所有潜在的冲突（资源争用、安全级别差异、变更频率差异）。使用[第1.2节](#12-五区驱动的根本原因)的五维驱动力模型做指导。

**步骤②：确定分区数量**

用[第7.4节](#74-分区数量与复杂度的关系)的甜点区指导——分区太少冲突加剧，分区太多交互熵增。

**步骤③：定义 Zoning Triad**

对每个潜在分区，明确其 Boundary、Internal-Rules、Cross-Protocol（[第7.2节](#72-zoning-triad分区的三元组)）。

**步骤④：验证八条设计原则**

对照[第9章](#9-分区设计原则方法论固化)的八条原则检查每个分区的设计。

**步骤⑤：衡量跨区代价**

用[第8.2节](#82-跨区操作的通用代价模型)的模型估算跨区操作的代价，验证交互频率×跨区代价 < 不分区的冲突代价。

### 11.2 分区重构：从坏分区到好分区

#### 案例：从单体到微服务的分区演变

```
阶段 1：单体（没有分区）
  ┌──────────────────────────┐
  │  所有功能在一个进程内     │
  │  - 无跨区代价            │
  │  - 所有冲突都在一起      │
  │  - 部署一个功能 = 部署全部 │
  └──────────────────────────┘
  → 冲突加剧

阶段 2：按功能域分区（SOA/Microservices）
  ┌──────┐ ┌──────┐ ┌──────┐
  │  Order │ │  Pay   │ │  User  │
  │  Svc   │ │  Svc   │ │  Svc   │
  └───┬───┘ └───┬───┘ └───┬───┘
      │          │          │
      └──────────┼──────────┘
            API Gateway
  → 跨区代价增加（网络通信）
  → 但冲突大幅减少

阶段 3：过度分区
  ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐
  │O │ │O │ │P1│ │P2│ │U1│ ...
  │r1│ │r2│ │ay│ │ay│ │s1│
  ├──┤ ├──┤ ├──┤ ├──┤ ├──┤
  │微│ │微│ │微│ │微│ │微│
  └──┘ └──┘ └──┘ └──┘ └──┘
  → 跨区代价 > 冲突改善 → 过度分区

阶段 4：最优分区
  ┌───────────┐  ┌───────────┐  ┌───────────┐
  │ Order     │  │ Payment   │  │ User      │
  │ (含Order1 │  │ (含P1+P2) │  │ (含U1+U2) │
  │  +Order2) │  │           │  │  +Auth)   │
  └───────────┘  └───────────┘  └───────────┘
  → 保留高交互的子功能在同一区
  → 分区间交互频率可控
```

**核心度量**：使用[第7.3节](#73-stability-visibility-mutability-三轴坐标系)的三轴坐标系，将每个服务标在[M,V,S]空间，相邻区的距离应该 ≥ 2 轴变化。

## 12. 结论

### 12.1 核心贡献

本文从五大领域（分布式系统、Git、虚拟化、Docker、存储层级）出发，提取了**位置化分区与隔离**的通用模型：

1. **Zoning Triad** (Boundary, Internal-Rules, Cross-Protocol) — 任何一个有效分区的三要素
2. **三轴坐标系** (Mutability, Visibility, Stability) — 用一个[M,V,S]三元组可以唯一标识任何系统中的任何一个区
3. **八条设计原则** — 从跨域模式中提取的可复用方法论
4. **四类跨区协议** — 所有跨区交互都可归入 Push/Pull/Event/Shared 四型

### 12.2 一句话总结

> **位置即规则**。在一个设计良好的分区系统中，对象"在哪个区"本身就已经决定了它能做什么、不能做什么、以及怎么换区——这就是"位置化"的力量。
