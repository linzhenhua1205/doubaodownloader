## 1. 定位：AI 是工具，不是目的

> 先理解问题，再选择工具，而不是拿起工具再找问题。

**核心原则**：

- **问题优先** — 先明确要解决的问题是什么，预期的解是什么样子（输出格式、质量要求、验收标准）
- **工具服从问题** — AI 只是手段，不要为了用 AI 而用 AI。如果手动 5 分钟能搞定，不需要硬套 AI
- **结果导向** — 评估 AI 产出时，只问"是否解决了问题"，不迷信"AI 做的必然正确"

**常见误区**：

| 误区 | 表现 | 纠正 |
|:-----|:------|:------|
| 工具驱动 | "这个 AI 很强大，我们用它做什么？" | 从业务痛点反向推导 |
| 过拟合 | 把简单问题复杂化，非得用 AI pipeline | 最简可行方案优先 |
| 黑箱依赖 | 不问"为什么是这个结果" | 关键决策必须可追溯 |

### 2.1 表达层：提示词工程（Prompt Engineering）

> 提示词（Prompt）是人机交互的接口语言。好的提示词不是"祈祷式请求"——它是精确的工程规范，包含目标、上下文、约束和格式四要素。
> 提示词质量遵循"输入决定输出"的铁律：模糊的要求产生模糊的结果，这不代表模型"不配合"，而是没有给模型明确的优化目标。

提示词工程不是"学几个魔法词汇"，而是建立一套**可复用的结构化编写规范**。以下 15 个维度覆盖从目标设定到迭代优化的完整生命周期：

---

#### 一、可衡量的目标（Measurable Goals）

> 原则：没有量化指标的要求，等价于"随便怎么做"。

**1.1 系统属性定义**

明确提示词需要覆盖的质量属性，包括但不限于：

| 质量属性 | 含义 | 示例 |
|:---------|:------|:------|
| **性能** | 响应速度、吞吐量 | "3 秒内给出初步结论" |
| **安全性** | 输入校验、权限控制 | "不要使用 SQL 拼接，必须参数化" |
| **易维护性** | 模块化、版本管理、可读性 | "每段代码加注释说明职责边界" |
| **时效性** | 数据的时间敏感性 | "使用 2025-2026 年的最新数据" |

**1.2 细化目标：抽象 → 量化**

将"提升代码质量"这类抽象需求转化为可衡量的具体要求：

| 模糊需求 ❌ | 量化目标 ✅ |
|:------------|:------------|
| "性能要好" | "响应时间 < 200ms，吞吐量 > 1000 QPS" |
| "代码要规范" | "通过 pylint 评分 9.0+，含类型注解和异常处理" |
| "文档要清晰" | "含使用示例、API 参数说明、常见错误排查" |
| "分析要全面" | "至少 5 个维度对比，每个维度附量化数据和来源" |

**自检**：如果读完提示词无法立即知道"什么样算完成"，目标还不够清晰。

---

#### 二、参考的样例（Reference Examples）

> 一个好例子胜过一千句规则描述。

**2.1 通用提示词模板**

推荐使用 **ICIO 框架**（四要素结构化模板）：

```
Instruction（指令）：明确任务是什么
Context（上下文）：背景信息、已知条件
Input Data（输入数据）：需要处理的具体内容
Output Indicator（输出格式）：期望的交付物样式
```

**示例 — 代码评审任务**：

```
Instruction：请审查以下代码的安全性和性能问题
Context：本项目采用 TypeScript + Node.js，运行在容器化环境中，
         用户数据涉及 PII（个人身份信息），需要符合 GDPR 合规要求
Input Data：```typescript
[待评审代码]
~~~
Output Indicator：以表格输出，含"问题行号、问题类型、严重程度（P0-P3）、修复建议"
```

**2.2 场景化模板**

| 场景 | 模板结构 |
|:-----|:---------|
| **代码生成** | "用[语言/库]实现[功能]，包含注释、异常处理和单元测试，遵循[编码规范]" |
| **代码修复** | "问题代码 → 错误信息 → 原因分析 → 修复方案 → 预防建议（避免同类问题再发生）" |
| **文档生成** | "目标读者：[角色]；核心信息：[3-5 个要点]；输出格式：[教程/API文档/设计文档]" |
| **方案对比** | "对比维度：[列明维度]，对比标准：[量化指标]，输出格式：[表格+结论]" |

**2.3 外部规范引用**

当有行业标准或团队规范时，直接引用而非模糊描述：

| 规范类型 | 写法示例 |
|:---------|:---------|
| 编码规范 | "遵循 Google TypeScript Style Guide v3.0" |
| 架构规范 | "符合 12-Factor App 原则" |
| 安全规范 | "参照 OWASP Top 10 (2025) 进行安全审查" |
| 数据规范 | "使用 OpenAPI 3.1 规范描述接口" |

#### 三、定义角色（Role Definition）

> 角色不是"加个标签"，而是划定 AI 的知识边界和响应风格。

**基本三要素**：

| 要素 | 含义 | 示例 |
|:-----|:------|:------|
| **专业身份** | AI 被赋予的领域角色 | "你是一名资深服务器硬件架构师" |
| **能力边界** | AI 的知识范围和职责范围 | "你的专长在 x86 服务器体系结构、散热设计和信号完整性，不涉及软件开发" |
| **风格特征** | 输出的语言风格和详尽程度 | "用专业但通俗的语言解释，避免过多公式" |

**角色引导技巧**（激发 AI 潜能的正向修饰语）：

- **角色增强**：单维 → 多维。例："你不仅是架构师，还是 TCO 分析师，要从技术可行性和商业可行性两个维度评估"
- **能力放大**：使用"精通""十年经验""主导过万卡集群"等强化描述
- **责任明确**：明确 AI 应对哪些方面负责。例："你的输出将直接用于高层决策，请附上每个结论的可信度评分"
- **用户角度设定**：定义 AI 面向的用户角色。例："你的受众是非技术背景的产品经理，请用通俗语言解释"

**反例对比**：

| 弱角色定义 ❌ | 强角色定义 ✅ |
|:--------------|:--------------|
| "帮我分析一下这个方案" | "你是一名有 10 年经验的服务器散热架构师，曾主导过 5 个液冷项目。请评审以下散热方案，关注 PUE 控制、成本和可维护性三个维度。输出给 CTO 做决策参考。" |

#### 四、定义场景（Scenario Definition）

> 同一个任务在不同场景下，最优做法完全不同。明确场景就是给 AI 提供"坐标系统"。

**4.1 代码生成场景**

```
场景：新功能开发
框架/语言：React + TypeScript + Tailwind
数据源：REST API (端点列表见附录)
约束：需适配移动端和桌面端，加载时间 < 2s
输出：功能组件 + 单元测试 + Storybook 文档
```

**4.2 代码优化场景**

```
场景：性能优化
现状：页面首屏加载 5s，Lighthouse 评分 45/100
目标：首屏加载 < 1.5s，Lighthouse 评分 85+
约束：不能改动后端 API 结构
策略：优先优化渲染性能、静态资源处理、代码分割
```

**4.3 文档生成场景**

```
场景：API 文档生成
目标读者：第三方开发者（英语母语者）
输出格式：OpenAPI 3.1 + Markdown
核心要求：每个端点含请求示例、响应示例、错误码说明
```

**4.4 复杂任务 — 提示链（Prompt Chaining）**

复杂任务无法一次性完成，应拆分为有序的提示链：

```
步骤 1（分析）→ "分析用户需求文档，提取核心功能点和约束条件"
步骤 2（设计）→ "基于分析结果，设计系统架构，输出架构图（Mermaid）"
步骤 3（实现）→ "依据架构设计，实现核心模块代码"
步骤 4（验证）→ "编写测试用例验证上述实现，覆盖正常和异常路径"
```

> **关键规则**：每个链节的输出格式必须兼容下一链节的输入格式。链与链之间的"契约"（接口格式）要提前定义。

#### 五、明确当前现状（Current Context）

> AI 不知道你不知道的信息。你不说我告诉它的背景，它就假设"空"或"默认"。

**5.1 现有资源**

| 类型 | 写法示例 |
|:-----|:---------|
| 已有代码 | "已有代码库位于 `/src/`，采用 MVC 架构，详见目录结构如下：..." |
| 现有数据 | "现有用户数 10 万，日活 2 万，数据存储在 PostgreSQL 16" |
| 基础设施 | "服务部署在 AWS EKS，使用 8 台 t3.large 实例" |

**5.2 约束条件**

约束必须**数量化、具体化**，而非笼统描述：

| 约束类型 | 笼统 ❌ | 具体 ✅ |
|:---------|:--------|:--------|
| 时间 | "尽快" | "3 个工作日内完成" |
| 成本 | "便宜的方案" | "月预算不超过 500 美元" |
| 技术 | "兼容性好" | "必须支持 Chrome/Firefox/Safari 过去 2 个版本" |
| 合规 | "要合规" | "需符合 SOC 2 Type II 和 GDPR 要求" |

**5.3 技术时效性**

> 来源：import/结合AI项目落地逻辑

写入提示词时明确数据的引用时效：

```
请优先使用 2025-2026 年的最新数据，若引用更早数据，请标注年份和来源。
对过时的技术方案，请明确指出"该技术在 xx 年已被替代"。
```

#### 六、可执行性：分步拆解（Step-by-Step Decomposition）

> 复杂任务直接给 AI — 像是让一个人同时做三件不同的事。拆解后每个子任务只需要专注一件事。

**通用拆解范式**：

```markdown
请按以下步骤执行：
1. **[分析]** 理解需求，列出关键功能和约束
2. **[设计]** 设计架构方案，至少提供 2 种选项
3. **[实现]** 选择最优方案，实现核心代码
4. **[验证]** 编写测试用例，验证正确性
5. **[总结]** 输出完整文档和使用说明
```

**服务器研发场景示例**：

| 步骤 | 动作 | 说明 |
|:-----|:------|:------|
| 1. 架构设计 | "先画出系统架构图（Mermaid），说明各模块职责" | 可视化整体方案 |
| 2. 核心实现 | "实现协议解析层和数据持久化层代码" | 关键逻辑先行 |
| 3. 测试验证 | "编写单元测试和集成测试，重点覆盖边界条件" | 质量保障 |
| 4. 部署说明 | "提供 Dockerfile 和 k8s 部署 yaml 模板" | 可落地 |

**注意**：分步拆解不是"写很长"，而是"分成明确阶段，每个阶段有清晰的完成标准"。每一阶段结束后可人工确认再进入下一步。

#### 七、允许创造性：多路径与优化（Enabling Creativity）

> 如果只需要一个答案，用"最佳方案"；如果需要做决策，用"多方案对比"。

**7.1 多方案对比**

```markdown
请提供至少 3 种实现方案，并按以下维度对比：
- 实现复杂度（人天）
- 性能预期（QPS/延迟）
- 维护成本（长期）
- 可扩展性
推荐最优方案并说明理由。
```

**7.2 修饰词引导**

通过正向描述激发 AI 的深层能力：

| 引导词 | 效果 |
|:-------|:------|
| "深入分析" | 激活多重推理路径 |
| "从第一性原理出发" | 避免表面套模板，回到基本原理推导 |
| "考虑极端场景" | 激活边界条件想象力 |
| "假设你是质疑者" | 激发批判性思维，反向寻找漏洞 |
| "创新一点，不限于常规做法" | 放松思维约束，探索非常规方案 |
| "用最简洁的方式" | 激活精炼压缩能力 |

**7.3 迭代优化法**

一次提示往往达不到最佳结果，通过多轮补充逼近目标：

```
Round 1（初始）→ "设计一个日志监控系统"
Round 2（补充要求）→ "增加告警阈值配置功能"
Round 3（深化细节）→ "告警支持 Webhook 和邮件两种通道"
Round 4（结构重组）→ "请将整个方案按模块整理为设计文档格式"
```

> **技巧**：不要指望一轮就把所有细节说清楚。观察到输出偏离预期时，分析"少了什么约束"而非重新写一遍。

---

#### 八、格式化处理（Format Specification）

> 输出格式决定了输出的**复用成本**——格式规范的内容可直接使用，格式混乱的需要人工整理。

**8.1 结构化输出**

| 格式 | 适用场景 | 示例 |
|:-----|:---------|:------|
| **表格** | 对比、汇总 | "用 Markdown 表格输出，列：方案名称 · 优势 · 劣势 · 成本" |
| **代码块** | 代码 / JSON / YAML | "用 ```typescript 包裹，含类型注解" |
| **Mermaid** | 架构图 / 流程图 / 时序图 | "用 Mermaid 语法画架构图，标注模块间接口" |
| **层次列表** | 分类信息 | "用缩进列表展示，三级深度：模块 → 功能 → 接口" |

**8.2 代码规范**

```markdown
输出要求：
- 语言：TypeScript
- 范式：函数式，避免 class
- 注释：每个函数含 JSDoc（参数说明、返回值、异常）
- 测试：附 vitest 单元测试，覆盖率 > 90%
- 命名：camelCase 变量、PascalCase 类型、UPPER_CASE 常量
```

**8.3 文档格式**

```markdown
输出格式：Markdown
结构要求：
- 标题层级：H1 文档标题 → H2 章节 → H3 小节
- 每个函数/模块前加 `##` 标题
- 代码示例用 ` ``` ` 包裹
- 关键术语用 `**加粗**` 标出
```

---

#### 九、明确输入与输出（Input/Output Specification）

> 定义输入输出，相当于定义函数的签名——告诉 AI 它拿什么、还回来什么。

**输出规格模板**：

```markdown
输出格式：
- 主体：Markdown 报告
- 长度：不少于 500 字，不多于 1000 字
- 结构：包含问题分析 → 解决方案 → 实施建议三段
- 必含元素：一个对比表格、一个 Mermaid 流程图
- 排除：不需要背景介绍，直接从分析开始
```

**输入规范模板**：

```markdown
输入材料：
- 位置：见下方文件列表
- 格式：部分为 PDF（已转换为文本摘要），部分为 Markdown
- 范围：仅处理与"散热方案"相关的文档，忽略其他
- 限制：如果有冲突信息，优先采信 [来源 A]
```

#### 十、约束的正向引导（Positive Constraints）

> 原则：用"需/应"替代"不要"——正向约束给出行为标准，负面约束只说了什么不能做。

| 负面约束 ❌ | 正向引导 ✅ |
|:------------|:------------|
| "不要用复杂的技术" | "优先使用标准库，避免引入外部依赖" |
| "不要写太长" | "输出控制在 300 字以内，只包含核心结论" |
| "不要有安全隐患" | "所有用户输入必须做参数化查询和 XSS 过滤" |
| "不要忽略错误处理" | "每个 API 调用需包含 try-catch 和错误日志" |

**负面约束的问题**：
- 模型可能执行了"不做什么"但做了其他同样糟糕的事
- 没有明确的替代方案指示

**正向约束的范式**：

```
错误处理：每个 try-catch 块必须：
  - 在 catch 中记录日志（含堆栈信息）
  - 返回明确的 HTTP 状态码和错误码
  - 不泄漏内部实现细节
```

#### 十一、迭代优化与反馈机制（Iterative Refinement）

> 提示词不是一次性写好的，而是"生成-评估-改进"循环打磨出来的。
> 来源：import/结合AI项目落地逻辑 · import/平衡AI辅助与独立思考的策略

**11.1 自我反思（Reflexion）**

引导模型对自身输出进行自评和优化：

```markdown
请完成以下任务：
1. 生成初版方案
2. 然后替换角色为"评审专家"——检查方案中的遗漏、矛盾、风险点
3. 根据评审结果，输出修订版方案
4. 最后总结：修订前后相比，改进了哪些地方
```

这种"自我博弈"模式可显著提升输出质量，尤其适用于方案设计、架构评审等场景。

**11.2 多轮调试流程**

```
初始 Prompt → 观察输出 → 分析偏差 → 补充约束 → 再次输出 → ...
```

**常见偏差及对应调整**：

| 观察到的偏差 | 可能原因 | 调整方法 |
|:-------------|:---------|:---------|
| 输出太泛泛 | 缺少具体约束 | 增加"请具体到 xxx 级别""附量化数据" |
| 输出结构混乱 | 没有指定格式 | 增加明确的输出结构要求 |
| 未覆盖关键方面 | 任务描述遗漏 | 补充"请务必包含 A、B、C 三个方面" |
| 跑题发散 | 角色/场景边界不清晰 | 收紧角色定义 + 排除范围说明 |
| 过于保守 | 缺乏创造性引导 | 增加"请考虑非常规方案""创新一点" |

**11.3 用户反馈整合**

将人工验证结果转化为提示词约束：

```
流程：
① 用户审核 AI 输出 → ② 发现错误或不足 →
③ 分析根因（是缺少知识？约束不全？格式问题？） →
④ 将修复写入提示词约束 →
⑤ 再次运行验证修复效果
```

> **关键**：不要每次都在回复中"现场说"，而是将规律性的修正固化到提示词模板中。例如"每个结论必须标注置信度"可以写进全局角色设定。

#### 十二、上下文结构化与分隔符使用（Context Structuring）

> 格式本身就是信息。好的排版让 AI 更容易解析你的意图。

**12.1 分隔符划分模块**

使用分隔符明确区分提示词的不同区域：

```markdown
### 角色设定
...
### 任务描述
...
### 输入数据
...
### 约束条件
...
### 输出格式
...
### 示例
...
```

**推荐分隔符体系**：

| 分隔符 | 用途 | 示例 |
|:-------|:------|:------|
| `###` 标题 | 模块分区 | `### 任务描述` |
| ` ``` ` 代码块 | 代码 / 数据 / 配置 | ` ```python ` |
| `---` 分割线 | 大幅内容分隔 | 分离指令与数据 |
| `**加粗**` | 关键信息强调 | "**注意**：必须使用参数化查询" |
| `> 引用` | 背景信息 / 引用材料 | `> 以下为需求文档原文` |
| XML 标签 | 精确区域标记 | `<input>...</input>` |

**12.2 上下文优先级标注**

长上下文场景下，关键信息应放开头和结尾（U 形曲线效应）：

```
[最重要的约束放在开头] → [详细背景说明] → [再次强调关键要求放在结尾]
```

**12.3 外部知识注入**

当需要引用实时或外部文档时，通过 MCP/Context 机制嵌入：

```markdown
以下为嵌入的外部文档摘要，请基于此回答问题：
<context>
[文档内容]
</context>
```

#### 十三、示例引导策略（Example Guidance）

> 根据任务复杂度选择示例数量。不是越多越好——关键是示例的质量和代表性。

**13.1 零样本（Zero-Shot）**

适用于**规则清晰、模型已熟悉**的简单任务：

```markdown
请将以下会议记录翻译为英文：
[输入内容]
```

**适用条件**：模型在训练数据中已有足够的相关模式。不需要额外引导。

**13.2 少样本（Few-Shot）**

适用于**需指定输出格式或风格**的任务。提供 1-3 个示例：

```markdown
示例1：
输入："服务器 A 的 CPU 温度过高"
输出：{"alert_level": "high", "component": "CPU", "value": "85°C", "threshold": "75°C", "recommendation": "检查散热风扇转速"}

示例2：
输入："磁盘使用率已达 95%"
输出：{"alert_level": "critical", "component": "disk", "value": "95%", "threshold": "90%", "recommendation": "立即清理日志或扩容"}

--- 以下为实际任务，请按上述格式输出 ---
输入："内存使用率异常波动，最高达 92%"
```

**示例选择原则**：
- **代表性**：覆盖典型场景和边界场景
- **多样性**：示例之间的差异应覆盖不同分支
- **精确性**：示例必须准确无误——模型会"学习"示例中的模式，如果示例有错，输出也会错

**13.3 反例警示（Negative Examples）**

通过"错误示例 + 正确定提供"预防常见错误：

```markdown
错误示例（命名不规范）：
  def get_data():
    pass

正确示例（命名规范）：
  def fetch_user_preferences(user_id: int) -> dict:
    pass

请按正确示例的风格实现以下功能：
...
```

#### 十四、提示词评估与测试方法（Evaluation & Testing）

> 提示词必须经过系统化测试才能投入生产环境。靠"感觉"评估提示词是最大的陷阱。
> 来源：import/结合AI项目落地逻辑迭代思想

**14.1 多版本对比（Ensembling）**

用不同版本的提示词测试同一任务，对比输出质量：

| 版本 | 策略 | 预期特点 |
|:-----|:------|:---------|
| v1 — 简洁版 | 只有核心指令 + 输出格式 | 灵活但可能发散 |
| v2 — 详细版 | 含角色定义 + 约束 + 示例 | 稳定但可能僵硬 |
| v3 — 链式版 | 分步骤提示链 | 适合复杂多步任务 |

**14.2 边界测试**

输入极端情况验证鲁棒性：

- **空输入**：什么都不给，看模型如何反应
- **超大输入**：超出上下文窗口的内容如何处理
- **矛盾输入**：给出互相冲突的需求，看模型是否发现并提示
- **模糊输入**：故意模糊表达，看模型是追问还是猜
- **恶意输入**：测试 prompt injection 防护

**14.3 效果量化**

设定评估指标，将主观感受转化为可衡量的客观标准：

```markdown
评估维度：
- 完整度：是否覆盖所有需求的 100%（是/否）
- 准确度：事实性错误数量
- 格式合规：是否完全按照指定格式输出（是/部分/否）
- 可操作性：人工可直接使用的比例（0-100%）
- 冗余度：不必要的内容占比（越小越好）
```

**推荐流程**：对关键提示词，每次修改后测试 3-5 个不同的输入样例，对比输出质量后再上线。

#### 十五、角色设定的深度与个性化（Deep Role Design）

> 来源：import/结合AI项目落地逻辑 · import/平衡AI辅助与独立思考的策略

超越基础的角色定义，通过三维框架让角色"鲜活"：

**15.1 三维角色框架**

| 维度 | 说明 | 示例 |
|:-----|:------|:------|
| **专业属性** | 知识领域、技能栈、经验年限 | "精通 x86 体系结构，10 年服务器研发经验" |
| **个性特征** | 沟通风格、决策偏好、思维习惯 | "偏好数据驱动的论证，说话简洁直接" |
| **关系定位** | AI 与用户之间的关系定位 | "你是我的技术顾问，负责挑战我的方案" |

**15.2 组合示例**

```markdown
你的角色是：
- **专业属性**：资深分布式系统架构师，专精于存储系统设计，主导过 PB 级存储系统
- **个性特征**：极度重视数据支撑，每个结论必须附带基准测试数据或学术文献引用
- **关系定位**：你是我的"魔鬼代言人"——专门挑战我的方案漏洞，确保方案无死角
```

**15.3 行业术语适配**

在角色中注入行业术语，提升专业相关性：

| 行业 | 角色设定示例 |
|:-----|:-------------|
| 金融 | "熟悉 Basel III 资本充足率、巴塞尔协议对风险计量的要求" |
| 医疗 | "熟悉 HIPAA、FDA 510(k) 审批流程、DICOM 标准" |
| 服务器 | "精通 IPD 研发流程、TR1-TR6 各节点交付物要求、PKI 模块化设计" |
| 云原生 | "精通 K8s operator 模式、CNCF 生态、12-Factor App" |

#### §2.1 总结：提示词质量速查表

每次编写提示词后，对照以下清单自检：

| # | 检查项 | 标准 |
|:-:|:-------|:-----|
| 1 | 目标可衡量 | 有量化指标或明确的完成标准 |
| 2 | 角色清晰 | 明确专业身份 + 能力边界 + 风格 |
| 3 | 场景具体 | 技术栈、环境、约束都写清楚了 |
| 4 | 现状明确 | 现有资源、约束条件、技术时效性已知 |
| 5 | 步骤可执行 | 复杂任务已拆分为有序子步骤 |
| 6 | 格式指定 | 输出格式和结构有明确要求 |
| 7 | 输入输出清晰 | 输入范围和输出规格已定义 |
| 8 | 约束正向 | 用"需/应"替代"不要" |
| 9 | 示例引导 | 复杂格式有参考示例 |
| 10 | 边界包含 | 错误处理、极端情况有涉及 |

### 2.2 编排层：Skills / Agent / MCP

> 提示词是说明书，Skills 是可复用的能力模块，MCP 是远程服务的访问通道。

| 能力 | 类比 | 特点 | 适用场景 |
|:-----|:------|:------|:---------|
| **Skills** | 可复用的技能包 | 本地封装，含脚本/知识/工作流 | 频繁执行的标准任务（归档、分析、生成文档） |
| **Agent** | 自主执行的智能体 | 多步推理+工具调用 | 复杂流程（市场调研、故障排查） |
| **MCP** | 远程 API 适配器 | 访问外部服务，需明确路径和权限 | 查询数据库、调用第三方 API、访问企业内部系统 |

**Skills 的优势**：将经验固化为可复用的"能力单元"。一次编写，反复使用。可通过组合多个 Skill 完成更复杂任务。

**MCP 的注意事项**：
- 需要明确的服务路径（URL/endpoint）
- 必须有对应访问权限（API Key / Token / 认证）
- 网络可达性和延迟需评估
- 服务变更可能导致 MCP 失效，需维护

### 2.3 模型层：Foundation Model

模型能力是天花板，编排和提示都是在这个天花板下发挥。

**关键能力维度**：

| 维度 | 重要性 | 为什么 |
|:------|:--------|:-------|
| **强推理** | ⭐⭐⭐⭐⭐ | 复杂逻辑推理、多步推导、数学/代码问题，弱推理模型会"绕弯"或"幻觉" |
| **大上下文** | ⭐⭐⭐⭐⭐ | 处理长文档、完整代码库、多轮对话历史。128K+ 是基本门槛 |
| **指令遵循** | ⭐⭐⭐⭐ | 严格按照格式/约束输出的能力，直接决定能否自动化 |
| **知识广度** | ⭐⭐⭐ | 领域知识覆盖度，但可通过 RAG/工具弥补 |
| **响应速度** | ⭐⭐⭐ | 对交互式场景重要，批处理场景次之 |

**实践建议**：
- 复杂推理场景（架构评审、方案对比、代码审查）→ 用强推理模型
- 长文档处理（合同分析、技术规范提取）→ 用大上下文模型
- 简单任务（格式化、翻译、摘要）→ 可用轻量模型，降低成本
- **不要把所有任务交给同一个模型**，不同任务匹配不同模型

## 3. 规则：对话纪律与方法论基石

> 没有规则的 AI 像没有红绿灯的路口——谁都能走，但谁也走不通。
> 好的规则体系比好模型更关键：它决定了产出的方向和质量下限。

### 3.1 三层规则体系

```
┌─────────────────────────────────────────┐
│     第一性规则 (绝对不可违反)              │
│  安全 · 隐私 · 合规 · 伦理 · 底线         │
├─────────────────────────────────────────┤
│     沟通纪律 (对话行为准则)                 │
│  锚定前提 · 聚焦主要矛盾 · 收束发散        │
├─────────────────────────────────────────┤
│     方法论规则 (思考与产出规范)             │
│  MECE · 量化 · 验证 · 第一性原理 · 深度研究 │
└─────────────────────────────────────────┘
```

### 3.2 第一性规则（不可违反）

| 规则 | 说明 |
|:-----|:------|
| **安全第一** | 不执行未确认的破坏性命令，不泄露密钥/凭证 |
| **隐私保护** | 不在回复中明文展示令牌、密码等敏感信息（必须脱敏） |
| **合规底线** | 不提供违法/违规的建议，不生成误导性内容 |
| **伦理边界** | 对涉及人身安全、重大财务决策的内容需保留人工确认环节 |
| **责任归属** | AI 提供分析建议，但最终签字决策的是人 |

### 3.3 沟通纪律（对话行为准则）

> 以下规则源于实战经验沉淀，本质是"让对话聚焦、高效、不跑偏"。

#### 3.3.1 锚定前提，不随意跳出

每次讨论开始前，明确以下三个前提：
1. **讨论什么** — 当前话题的边界是什么（不在范围内的内容明确排除）
2. **为什么讨论** — 这次对话的目的是什么（决策？调研？学习？）
3. **输出是什么** — 期待的交付物形式（报告？列表？结论？）

**执行方式**：任意一方发现对话已偏离前提时，应主动拉回，而非顺着新话题继续。

#### 3.3.2 抓主要矛盾，摒弃次要争议

- **不要反向抬杠** — 对用户的小细节表达异议、对举例客体展开对错辩论，都是无效消耗。举例仅用于类比说理，不应对举例本身做纠正或科普
- **不要纠缠边缘** — 当核心问题已有明确方向，不要在次要细节上过度争论
- **区分定性 vs 讨论** — 法定/历史底线性内容（如"法律规定了…"）只需确认引用，不展开辩论；可讨论的仅限于限定场景下的具体细节

#### 3.3.3 收束发散，主动归位

- 当一方提出偏离主线的宏大定性内容（如从技术问题跳到"行业趋势百年大计"），应主动收束：**记录为一个单独话题建议，然后回到原定内容**
- 当对话明显偏离主题时，**主动提醒**："这条线索建议另开对话深入讨论，当前先回到[原主题]"
- 不把所有好想法塞进同一段对话——发散思路值得记录，但不值得在同一对话中切换轨道

#### 3.3.4 对比分析的纪律

对比分析是最高频出错的环节。必须执行以下三步：

1. **维度对齐** — 先问：这两个东西能比吗？它们在同一维度吗？对比的是能力还是方案？
2. **信息确认** — 确保双方对对比对象有相同的了解基础，不基于假设做比较
3. **底层穿透** — 不满足于表面对比，深入到架构哲学/技术权衡/商业模式层面的本质差异

**自检**：如果我这个对比结论放到对方立场看，对方会接受吗？

#### 3.3.5 记录的命名规范

对话记录按统一的命名格式归档：

```
【年份-月份】研究主题 + 用途
```

例如：`【2026-06】AI Agent Skill 体系调研 + 归档笔记`


#### 3.3.6 强制坦诚：让 AI 标注自己的不确定性

> 来源：import/平衡AI辅助与独立思考的策略

AI 的最大风险不是出错，而是"错了还看起来很有道理"。要求 AI 对每个主张标注**置信度等级**：

**1-10 分置信度评分标准**：

| 分数 | 含义 | 判断标准 |
|:----:|:-----|:---------|
| 9-10 | 明确事实 | 基于权威来源的明确事实（政府文件、学术论文、官方数据） |
| 7-8 | 较可靠 | 基于合理推断的信息（需说明推断依据） |
| 5-6 | 有限证据 | 基于有限证据的猜测（需明确标注"不确定"） |
| 1-4 | 高度不确定 | 信息不可靠，应避免直接回答（标注"存疑"） |

**提示词模板**：

```markdown
请为每个主张提供：
1. 置信度评分（1-10分）
2. 评分依据（来源类型、推断逻辑）
3. 若不确定，需说明"我对这一点不确定"并解释原因
```

**执行原则**：置信度 < 7 分的结论，在报告中要标注为"待验证"或"仅供参考"，决策层谨慎使用。

### 3.4 方法论规则（思考与产出规范）

| 规则 | 含义 | 自检 |
|:-----|:------|:------|
| **深度研究五步法** | 信息聚合→逻辑拆解→推理论证→方案沉淀→迭代复盘 | 我是直接转述，还是真正推演过？ |
| **第一性原理** | 不盲从"大家都这么说"，回到物理/经济/信息论的基本规律 | 我的结论能否用基本原理推导出来？ |
| **客观事实优先** | 不简单复用用户输入的信息，从客观可验证的事实出发做判断 | 用户说的"事实"我独立验证过吗？ |
| **MECE 拆解** | 至上而下分层，同层互斥且穷尽 | 有重叠→违ME；有遗漏→违CE |
| **量化指标** | 每条数据含数值+单位+基线+条件 | `吞吐提升28.9%（vs BF16, batch=32, A100）` |
| **来源验证** | 关键结论至少 2 个独立来源 | 换一个渠道查同样的数据，结果一致吗？ |
| **逻辑谬误预防** | 避免 11 类常见谬误（声明-实践不一致、先验分类、术语漂移、类比当定理等） | 见 RULE.md 方法论速查 §E |

## 4. 沉淀：从对话到 Skill 的提炼

> 每一次深度对话都是一次"能力挖掘"。不要让经验只停留在对话记录中——要把可复用的工作流固化为 Skill，实现一次投入、反复受益。

### 4.1 为什么要沉淀 Skill

| 原因 | 说明 |
|:-----|:------|
| **减少 Token 消耗** | Skills 以结构化指令包替代每次重复的冗长提示词，每次调用节省 50-90% 的输入 Token |
| **保证一致性** | 人工写提示词每次都有偏差，Skill 固化的工作流精确可控 |
| **叠加复杂度** | 单个 Skill 完成单一任务，多个 Skill 通过 MetaSKILL DAG 编排完成复杂流程 |
| **经验资产化** | 团队中一个人写好的 Skill 可被所有人复用，经验不流失 |
| **持续进化** | Skill 可基于反馈迭代优化，形成不断改进的能力单元 |

### 4.2 从对话到 Skill 的提炼流程

```
[深度对话]
    ↓ 识别可复用的工作流模式
[识别模式]
    ↓ 提取→步骤化→参数化→测试
[编写 SKILL.md]
    ↓ 注册到 Agent 系统
[投入使用]
    ↓ 收集使用反馈
[迭代优化]
```

**识别信号**：当你在对话中重复做以下事情时，就该沉淀为 Skill 了：

| 信号 | 示例 | 应对 |
|:-----|:------|:------|
| 同一类任务做了 ≥3 次 | 每周做竞品分析、每次访网页都要归档 | 立即创建 Skill |
| 步骤固定但操作繁琐 | 搜索→提取要点→归档→更新索引 | 封装为 Pipeline \\
| 需要特定知识/脚本 | 调用 Python 脚本做链接检查 | 嵌入 Skill 的 resources 目录 |
| 需要多人复用 | 自动日报生成、格式检查 | 团队共享 Skill |

### 4.3 已沉淀的 Skill 示例

当前工作区中已有以下从实战中提炼的 Skill：

| Skill | 来源 | 解决的问题 |
|:------|:------|:-----------|
| **web-archive** | 归档网页文章（非豆包） | 标准化归档流程：抓取→理解→slug→写MD→更新索引→验证链接 |
| **doubao-share** | 归档豆包分享链接 | 三级访问策略（web_fetch→curl|grep→browser），SSR/CSR 判断，归档到知识库 |
| **knowledge-wiki** | 管理知识库 | 创建/更新/索引/日志标准化操作 |
| **skill-creator** | 创建和安装新 Skill | 从零创建 Skill 或从 URL 安装社区 Skill |
| **fault-diagnosis** | 故障排查 | 事件墙法、鱼骨图+5Why、五层诊断架构 |

**关键经验**：每个 Skill 都应该有一个明确的"触发器"（什么场景下触发）和一个"质量检查清单"（完成标准），确保可被自动化调度而非仅靠人工调用。

### 4.4 从 Skill 到 MetaSKILL 的演进路径

```
Ad-hoc (手工操作)
  → 模板 (每次复制的提示词模板)
    → Skill (标准化的 SKILL.md + 资源)
      → MetaSKILL (多 Skill DAG 编排)
        → 自进化 Skill (EvoSkills 模式)
```

不需要一步到位。大多数任务停留在"Skill"层级就足够实用。只有满足以下条件才考虑 MetaSKILL：
- 涉及 3 个以上步骤
- 需要人工介入的检查点
- 步骤间有明确的依赖关系
- 需要有完整的审计轨迹

### 4.5 从使用到沉淀的闭环：AI 使用日志 → 任务分类 → Skill 沉淀

> 来源：import/平衡AI辅助与独立思考的策略 · import/结合AI项目落地逻辑

沉淀不是"写完 Skill 就结束"，它是一个持续运行的闭环：

```
┌─────────────────────────────────────────────────┐
│  ① 每次 AI 交互 → 记录 使用日志                  │
│     (目的/输入/输出/个人贡献/验证方法)            │
│                       ↓                          │
│  ② 定期复盘 → 识别 高频模式                      │
│     (哪些任务重复出现？哪些步骤标准化？)          │
│                       ↓                          │
│  ③ 提炼 → 创建/更新 Skill                        │
│     (步骤化 → 参数化 → 触发器 → 质量检查)        │
│                       ↓                          │
│  ④ 使用 → 收集 反馈                              │
│     (效果如何？哪些地方需要优化？)                │
│                       ↓                          │
│  ⑤ 迭代 → 回到 ①                                 │
└─────────────────────────────────────────────────┘
```

**复盘频率建议**：
- **每周**快速扫描日志，标记可沉淀的模式
- **每月**系统化整理，形成 1-2 个新 Skill
- **每季度**评估已有 Skill 的使用率和效果，淘汰或合并低效的

## 5. 机理：理解 AI 的工作方式才能用好它

> 知其然，更要知其所以然。理解 AI 能做什么、不能做什么、在什么条件下表现最好，才能设计出有效的使用策略。

### 5.1 语言模型的基本工作机理

当前大语言模型（LLM）的核心是**自回归下一个 Token 预测**：

```
输入文本 → Tokenization（分词） → Embedding（向量化）
  → Transformer 层（注意力机制计算上下文关系）
    → 输出概率分布 → 采样 → 下一个 Token
      → 重复直到终止
```

这个机制决定了 LLM 的**先天优势**和**固有限制**。

**先天优势**：
- ✅ **模式匹配** — 见过足够多的训练数据，能识别和复用各种文本模式
- ✅ **上下文关联** — 注意力机制让模型能关联长跨度文本中的相关信息
- ✅ **知识记忆** — 训练数据中高频出现的事实可以被"记住"并召回
- ✅ **语言生成** — 流畅的自然语言生成，风格、语气可控

**固有限制**（不是因为"不够智能"，而是架构决定）：
- ❌ **没有真正的理解** — 模型不"理解"含义，只是在数学上找到最可能的下一个词
- ❌ **无法区分"真实"与"看似合理"** — 输出看似合理但不一定正确（幻觉的根本原因）
- ❌ **训练数据截止** — 不知道训练截止后发生的事情（除非通过工具获取）
- ❌ **需要精确指令** — 模糊的要求产生模糊的结果，这不是"不配合"而是"没有明确的优化目标"

### 5.2 能力天花板与适配策略

#### 5.2.1 知识库（RAG）能做到的与不能做到的

| 维度 | 能做到 ✅ | 不能做到 ❌ |
|:-----|:---------|:-----------|
| **事实检索** | 精准匹配已存入知识库的内容 | **无法推理出隐含事实** — 如果知识库中没有，就是没有 |
| **多文档综合** | 跨多个文档的显式信息聚合 | **跨文档的深度逻辑链** — A→B→C 的推理，如果 B 是隐含的，常常断裂 |
| **时效性** | 存入时的内容是最新的 | **不能自动更新** — 知识库中过时的内容会给出过时的回答 |
| **格式多样** | 文本、表格、代码 | **跨模态整合** — 图片里的数据、音频里的内容，如果以二进制存储而非文字描述，RAG 无法处理 |
| **长文档** | 拆分成块后检索 | **全局结构性理解** — 分块后丢失了文档的整体层次结构 |

**RAG 的黄金法则**：RAG 擅长"找东西"而不是"想东西"。对需要深度推理的问题，RAG 提供素材，模型推理能力负责加工。

#### 5.2.2 大上下文能做到的与不能做到的

| 能力 | 表现 |
|:-----|:------|
| ✅ **128K-1M Token** | 可一次性处理整本书、完整代码库、数小时对话历史 |
| ⚠️ **"大海捞针"效应** | 在超长上下文中，模型对中间部分的信息召回率下降（U 形曲线：开头和结尾最好，中间最差） |
| ⚠️ **注意力稀释** | 上下文越长，模型对每条信息的"注意力"越分散，细节可能被忽略 |
| ❌ **无限上下文** | 不存在。超过上下文窗口的内容会被截断，不可见 |
| ❌ **等精度处理** | 1M Token 的最后一页和第一页的精度不一致——长上下文的"尾部"信息召回率显著下降 |

**应对策略**：
- 关键信息放在 prompt 的开头或末尾
- 长文档优先检索而非全文填充
- 对超长内容，分层处理：先摘要→再选重点→再深入

#### 5.2.3 意图理解的深度与局限

| 意图层次 | LLM 能做到 | LLM 容易漏掉的 |
|:---------|:-----------|:---------------|
| **L1 — 显式指令** | ✅ "对比这三家供应商" → 准确执行 | — |
| **L2 — 隐式意图** | ⚠️ "最近服务器总宕机" → 推测可能需要排查方案 | 可能给出宽泛回答而非针对性诊断 |
| **L3 — 深层需求** | ❌ 用户说"A 方案贵"但真正想要的是"说服老板选 A" | 需要大量的上下文 + 经验才能识别 |
| **L4 — 潜台词** | ❌ "你再看看"可能意味着"我不同意，你改一下" | 需要对话历史 + 语气分析 |

**关键洞察**：LLM 对"说了什么"反应很好，对"没说什么但暗示了什么"表现不稳定。当需求复杂时，**说出来比让 AI 猜更有效**。

### 5.3 策略选择：对症下药

不同问题匹配不同能力层级：

| 问题形态 | 推荐策略 | 依赖的能力 |
|:---------|:---------|:-----------|
| 简单明确的单步任务 | 一次性求解（One-shot） | 指令遵循 |
| 多步骤可分解的复杂问题 | Pipeline / Composition | 编排 + 推理 |
| 需要大量外部知识 | RAG + 搜索 + 知识库 | 检索 + 长上下文 |
| 需要实时数据 | 工具调用 + API | 编排层 |
| 跨模态信息整合 | - 图片: Vision API | 多模态模型 |
| | - 音频: 转文字再处理 | 音频→文本管线 |
| | - 视频: 帧提取 + 时序分析 | 多步编排 |
| 深度行业分析 | 人类判断 + AI 辅助 | 专业经验 + AI 效率 |
| 需要全局一致性 | 人工审核 | 人的判断 |

### 5.4 人机分工：什么该交给 AI，什么必须自己做

> 来源：import/平衡AI辅助与独立思考的策略 · import/结合AI项目落地逻辑

不区分任务类型就盲目使用 AI，是效率最大杀手。以下是基于实践验证的分类框架：

#### 5.4.1 任务分类四象限模型

```
                   高人类介入
                       │
        委托任务       │   只属于我的任务
   (AI执行,人类审核)   │   (人类主导)
        ╱              │              ╲
   AI可完成大部分      │       战略决策、创意发想
   需人类把关          │       价值判断、情感交流
                       │
  ─────────────────────┼─────────────────────→ 可编程性
                       │
        自动化任务     │      融合任务
   (AI主导)           │   (人机协作)
        ╲              │              ╱
   规则清晰、流程固定  │   需要人类创意+AI效率
   结果可预测          │   方案迭代、数据分析辅助
                       │
                    低人类介入
```

| 象限 | 特征 | 判断标准 | 示例 |
|:-----|:------|:---------|:------|
| **自动化任务** | 规则清晰、流程固定 | 能否用明确的 if-then 规则描述整个流程 | 数据清洗、格式转换、文档分类 |
| **委托任务** | AI 可完成大部分，需人类把关 | AI 输出错误的后果是否可控，检查成本是否低于直接完成 | 初稿生成、数据整理、信息摘要、翻译 |
| **融合任务** | 需要人类创意 + AI 效率互补 | 任务需要人类判断力与 AI 效率的互补 | 方案迭代、内容润色、数据分析辅助 |
| **只属于我的任务** | 高度依赖个人经验、情感、价值观 | 任务结果必须体现个人独特性与责任感 | 战略决策、创意发想、价值判断、情感交流 |

#### 5.4.2 三问评估法（判断 AI 输出的可用性）

面对 AI 的输出，先问三个问题：

1. **准确性** — 信息真实可靠吗？有无事实错误或逻辑漏洞？
2. **相关性** — 输出是否精准解决我的问题？有无偏离主题？
3. **价值** — 此输出能否直接交付？差距在哪里？需要多少人工修改？

如果三个答案都是"是"→ 直接使用；如果有任何一个"否"→ 先修正再使用。

#### 5.4.3 人类能动性等级（HAS）

| 等级 | 描述 | 适用场景 |
|:----:|:-----|:---------|
| H1 | AI 完全独立完成 | 自动回复邮件、定时报告生成 |
| H2 | AI 完成大部分，人类偶尔介入 | AI 生成报告，人类审核 |
| H3 | **人机平等协作（推荐）** | 共同迭代方案，这是最理想的状态 |
| H4 | AI 必须依赖人类 | AI 提供数据，人类做决策 |
| H5 | 人类全程主导 | 战略规划、创意发想 |

> **经验法则**：当前阶段，H3（人机平等协作）是最理想的平衡点——既不放任 AI 做决定，也不拒绝 AI 的贡献。

### 5.5 AI 缓冲带：保护思维独立性

> 来源：import/平衡AI辅助与独立思考的策略

工具的目的是"扩展能力"，不是"替代思考"。用以下策略保持对 AI 产出的独立判断：

#### 5.5.1 "先想后用"原则

遇到问题时的标准流程：

```
① 自己先思考 20 分钟（记录初步观点 + 问题框架）
② 用 AI 验证/补充/挑战自己的思路
③ 对比自己的方案 vs AI 的方案
④ 做最终判断
```

**目的**：确保思考不被 AI 带偏。先有骨架，再填充细节。

#### 5.5.2 "质疑前置"机制

在向 AI 提要求之前，先明确"我需要验证什么"：

- 这条信息我已有来源吗？→ 若无，要求 AI 提供来源
- 这个结论是否接受？→ 若不确定，要求 AI 提供反方意见
- 这个方案是否最优？→ 要求 AI 给出 3 种替代方案并对比

#### 5.5.3 AI 使用日志

每次调用 AI 后，记录以下信息（供复盘用）：

| 字段 | 内容 | 示例 |
|:-----|:------|:------|
| 目的 | 我需要 AI 解决什么问题 | 分析三种液冷方案的 TCO |
| 输入 | 我提供了什么信息 | 三种方案的规格表（附件） |
| 输出 | AI 给出的关键结论 | 推荐冷板式液冷，3 年 TCO 最低 |
| 个人贡献 | 我做了什么 | 补充了数据中心 PUE 约束条件 |
| 验证 | 我验证了哪些点 | 交叉核对了供应商报价数据 |

> **价值**：形成可追溯的学习轨迹，帮助判断"哪些场景 AI 真正靠谱、哪些场景需要谨慎"。

### 5.6 终极建议：知道自己不知道

> AI 最大的风险不是它错了，而是它错了还看起来很有道理。

**不同场景的不同可信度**：

| 场景 | AI 可信度 | 建议 |
|:-----|:---------|:------|
| 代码生成/正则/格式化 | ⭐⭐⭐⭐⭐ | 自动化。AI 在这里远超人类速度 |
| 知识问答（常见问题） | ⭐⭐⭐⭐ | 可信任，但关键事实建议验证 |
| 结构化的方案分析 | ⭐⭐⭐ | 作为初稿/思路参考，需要人工审视 |
| 行业趋势判断/预测 | ⭐⭐ | 仅供参考，必须交叉验证 |
| 财务/法律/安全关键决策 | ⭐ | 仅做建议，不能作为依据 |
| 创造性的战略方向 | ⭐ | AI 可以"给选项"，但"选哪个"是人的事 |

## 6. 总结：AI 使用的五个层次

| 层次 | 核心问题 | 关键能力 |
|:-----|:---------|:---------|
| L1 — 知道 | 知道 AI 是什么、能做什么 | 基础认知 |
| L2 — 会用 | 能写提示词、获取有效输出 | 提示词工程 |
| L3 — 善用 | 能用 Skills/Agent/MCP 构建工作流 | 编排能力 |
| L4 — 驾驭 | 能制定规则、评估代价、选择策略、沉淀 Skill | 方法论体系 + 迭代意识 |
| L5 — 判断 | 知道何时用 AI、何时不用；知道 AI 的边界在哪里 | 边界意识 + 决策判断 |

**最终目标不是成为"AI 提示词大师"，而是成为"用 AI 高效解决问题的人"。**
