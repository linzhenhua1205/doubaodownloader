## 1. 意图理解的本质：从"说了什么"到"要什么"

> 意图理解是 AI 交互中最关键的认知瓶颈——它介于"听懂语言"和"解决问题"之间，决定了 AI 是被动执行者还是主动协作者。

### 1.1 意图理解的层级模型

基于对 LLM 工作机制和实践经验的深入分析，可以将意图理解分为四个层次：

```
L4 — 潜台词
    知道用户没说但隐含的意思
    "你再看看" = "我不满意，你改一下"
    ↑
L3 — 深层需求
    识别用户真正想要解决的问题
    "A方案太贵了" = "想要一个说服老板选A的理由"
    ↑
L2 — 隐式意图
    推测用户没直说的目的
    "服务器最近总停机" = "需要排查方案"
    ↑
L1 — 显式指令
    直接理解用户说了什么
    "对比这三家供应商" → 准确执行
```

| 层次 | 定义 | LLM 能力 | 稳定性 |
|:----:|:-----|:---------|:-------|
| **L1** | 字面理解，精确执行显式指令 | ✅ 几乎完美 | ⭐⭐⭐⭐⭐ |
| **L2** | 根据上下文推测隐含目的 | ⚠️ 部分可行，但不稳定 | ⭐⭐⭐ |
| **L3** | 穿透表象识别真实需求 | ❌ 需要大量上下文 + 经验 | ⭐⭐ |
| **L4** | 理解潜台词、语气、社交暗示 | ❌ 极不稳定，容易误判 | ⭐ |

**关键洞察**：
- LLM 对"说了什么"（L1）反应极好，对"没说什么但暗示了什么"（L2+）表现不稳定
- 这本质上是自回归架构决定的——模型擅长模式匹配，但缺乏"心智理论"（Theory of Mind）
- 当需求复杂时，**把意图说出来比让 AI 猜更有效**

### 1.2 理解为什么会"猜错"——LLM 意图理解的固有限制

> 来源：import/千问AI助手的工作方式探讨 · import/平衡AI辅助与独立思考的策略

**模式匹配而非真正理解**：
LLM 理解意图的方式与人类不同——它不是在"思考你的意思"，而是在训练数据中找到与当前输入最匹配的"应答模式"。当意图模糊时，模型会选择统计上最可能的响应路径，但这不一定是正确的路径。

**具体表现**：

| 场景 | 用户期望 | LLM 实际行为 | 根因 |
|:-----|:---------|:-------------|:------|
| 模糊提问 | 追问澄清 | 给出宽泛回答 | 无明确的"不确定性识别"机制 |
| 隐含相反 | 知道用户想要"否" | 顺着字面回答"是" | 缺乏反向推理能力 |
| 多义词 | 根据上下文判断 | 选最常用的含义 | 上下文关联的"稀释效应" |
| 情绪信号 | 理解抱怨背后的需求 | 只处理事实部分 | 缺乏情感计算能力 |
| 否定提问 | 理解"不是A"的意思 | 可能过度反应 | 否定学习的固有限制 |
| 跨文化差异 | 理解特定语境 | 用通用模式回应 | 训练数据的文化偏差 |

### 1.3 知道"猜不准"比"猜得准"更重要

> **核心原则**：在设计 AI 交互系统时，假设 L2+ 意图理解不可靠，用显式机制弥补。

这意味着：
1. **不要依赖 AI 的"直觉"** — 把模糊意图降低为清晰指令是人的责任
2. **显式确认优于隐式推断** — 当不确定时，要求 AI 主动反问而不是自行猜测
3. **结构化输入优于自然语言** — 模板/表单/选择题比纯自然语言更容易被准确理解

## 2. 人类沟通过程中的意图理解

> 来源：import/低成本获取外部反馈的方法 · import/平衡AI辅助与独立思考的策略

人类之间的意图理解尚且存在大量误解，何况人机之间？这一节从人类沟通中的意图理解经验出发，提炼可用于人机交互的通用原则。

### 2.1 人类沟通中意图理解的"冰山模型"

```
        水面以上（显式语言）
        ┌──────────────────────────┐
        │ "我想要更便宜的车"        │ ← 用户说出的话
        └──────────────────────────┘
        ┌──────────────────────────┐
        │ 原因：预算不确定感        │
        │ 场景：刚毕业、第一辆车    │
        │ 情绪：担心买贵了          │
        │ 真实需求：决策安全感      │
        └──────────────────────────┘
        水面以下（隐性意图）
```

**解决人类沟通中意图模糊的经典方法——SPIN 访谈法**：

| 步骤 | 问题类型 | 目的 | 示例 |
|:-----|:---------|:-----|:------|
| S — 情境 | 收集事实 | 建立背景 | "你现在用什么工具管理客户？" |
| P — 难点 | 发现痛点 | 定位问题 | "有没有遇到过客户信息遗漏的情况？" |
| I — 暗示 | 扩大影响 | 让痛点显性 | "这种遗漏会导致客户投诉增加吗？" |
| N — 需求回报 | 定义标准 | 明确需求 | "如果有一套系统能自动提醒，你觉得价值多大？" |

**五问法（5 Whys）**：
连续追问五次"为什么"，从表面需求穿透到本质需求：

```
用户: "我想要更便宜的车"
↓ 为什么？
因为: "现在买车预算紧张"
↓ 为什么预算紧张？
因为: "不清楚总费用，怕超出预算"
↓ 为什么不清楚？
因为: "没有直观的估算工具"
↓ 最终需求: "需要购车预算透明化的决策支持工具"
```

### 2.2 区分"显性需求"与"隐性需求"

| 维度 | 显性需求 | 隐性需求 |
|:-----|:---------|:---------|
| 表达形式 | 清晰、具体、可行动 | 模糊、情绪化、抽象 |
| 例子 | "我想修改头像" | "每次上传头像都很麻烦" |
| 识别难度 | 低，直接执行 | 高，需要推断+验证 |
| AI 处理能力 | ✅ L1 可完美处理 | ❌ 需要 L3/L4，不稳定 |
| 验证方法 | 直接满足 | 通过行为数据验证 |

**隐性需求的识别信号**：

- **情绪词汇** — "烦"、"焦虑"、"困惑"→ 往往指向未被满足的期望
- **否定句式** — "找不到"、"不行"、"不想要"→ 指向某一类的缺失
- **比较句式** — "别人家都能"、"为什么不能"→ 指向参照系差异
- **假设句式** — "如果能…就好了" → 指向理想的替代方案

**应对策略**：当识别到上述信号时，不要直接解决表面问题，而是主动追问"你具体想解决什么？"来澄清意图。

### 2.3 从"猜意图"到"引导意图"的范式转换

人类沟通中有两种策略：

| 策略 | 做法 | 风险 |
|:-----|:-----|:------|
| **猜测策略**（默认） | AI 猜用户意图，基于模式匹配自行补充 | 猜错成本高，用户需要纠正 |
| **引导策略**（推荐） | AI 主动追问、提供选项、结构化分解 | 交互步骤增加，但准确率大幅提升 |

> **选择原则**：简单任务用猜测（快），复杂任务用引导（准）。

## 3. 知识库场景下的意图理解

> 来源：import/Dify知识库系统构建与优化 · import/千问AI助手的工作方式探讨 · import/平衡AI辅助与独立思考的策略

### 3.1 RAG 系统中的意图理解链路

在 RAG（检索增强生成）系统中，意图理解直接影响检索质量和最终回答的准确性：

```
用户输入 → 意图识别 → 查询重写/扩展 → 多路检索 → 结果重排序 → 生成回答
              │            │              │            │
              ↓            ↓              ↓            ↓
       分类问题类型    补充同义词     不同策略并行  按相关性排序
       确定检索策略    语义扩展      召回最相关片段 提升 Top-K 质量
```

### 3.2 意图分类驱动的检索策略

基于实际数据分析，用户在知识库中的提问可按意图分为以下类别（数据仅供参考）：

| 意图类别 | 占比（约） | 检索策略 | 提示词模板取向 |
|:---------|:---------:|:---------|:--------------|
| 查询基础信息 | 40% | 精准匹配+关键词 | 简洁的事实性回答 |
| 技术实现与配置 | 28% | 混合检索（向量+关键词） | 结构化分步+代码/参数示例 |
| 分析判断与决策 | 9% | 多文档综合+对比 | 多维对比+优劣分析 |
| 业务流程与步骤 | 8% | 流程检索+步骤拆分 | 步骤化指南 |
| 文档内容汇总 | 3% | 全文档检索+摘要 | 结构化摘要+关键点 |
| 职责与分工 | 3% | 角色匹配+权限检索 | 责任矩阵+接口说明 |
| 跨文档信息总结 | 1% | 多文档联合检索 | 关联分析+差异对比 |
| 其他 | 35% | 兜底策略：通用检索 | 综合性回答 |

**注意**：部分问题会交叉分类组合——一个"技术实现"问题可能同时包含"分析判断"成分。建议使用多标签分类而非单标签分类。

### 3.3 知识库中意图理解的三大难题

#### 难题一：跨文档综合分析

**问题**：用户问"A 方案和 B 方案哪个好？"——需要跨多个文档提取信息并进行对比分析。

**LLM 的表现**：
- ✅ 如果两个方案的显式参数都在知识库中 → 能准确对比
- ⚠️ 如果需要从不同文档中提取隐含的对比维度 → 可能遗漏
- ❌ 如果需要综合文档外部的业务判断 → 能力不足

**应对策略**：将对比分析拆分为"检索维度→逐维提取→综合对比"的三步流程，每步单独验证。

#### 难题二：隐含条件的推理

**问题**：用户问"用 X 技术实现 Y 功能需要什么？"——但知识库中只有 X 技术的说明和 Y 功能的规格，没有直接的"X→Y 映射"。

**LLM 的表现**：
- 如果 X 和 Y 的关联在训练数据中出现过 → 可能推理正确
- 如果关联在训练数据中罕见 → 容易产生"看起来合理但实际错误"的推理（幻觉）

**应对策略**：使用"plan-then-retrieve"模式——先拆解用户问题为子问题，对每个子问题分别检索，最后综合。对无法在知识库中找到依据的推理步骤，必须标注"基于推理，建议验证"。

#### 难题三：模糊查询的意图消歧

**问题**：用户说"帮我看看存储"——具体含义可能是"查看存储使用量"、"配置存储参数"、"诊断存储故障"、"扩容存储"等。

**LLM 的表现**：
- 通常会选择最常见的一种解释
- 如果用户是存储运维人员，解释可能偏向"诊断"
- 如果用户是开发人员，解释可能偏向"配置"

**应对策略**：
```
第一步：识别模糊点
  "你提到的'看看存储'，具体是想查看容量、配置参数，还是排查问题？"
第二步：提供选项
  "我可以用以下方式帮你：
   a) 查看当前存储使用量和性能
   b) 检查存储配置参数
   c) 分析存储异常日志
   d) 其它（请描述）"
第三步：执行确认后的意图
```

### 3.4 知识库意图理解的最佳实践

| 实践 | 描述 | 价值 |
|:-----|:------|:------|
| **查询重写** | 将用户原始问题改写为更适合检索的形式 | 检索召回率提升 20-40% |
| **多查询扩展** | 自动生成 3 个语义相关的子问题，分别检索后合并结果 | 覆盖度提升明显 |
| **意图检测前置** | 先判断问题类型，再选择检索策略和提示词模板 | 回答准度提升 30%+ |
| **混合检索** | 向量检索（语义） + 关键词检索（精确）权重组合 | 兼顾语义理解和术语精确匹配 |
| **不确定性标注** | 对置信度不足的结果标注"未在知识库中找到直接依据" | 避免幻觉 |
| **反问澄清** | 当意图置信度 < 0.7 时，主动反问而不是猜测 | 减少 80% 的"答非所问" |

## 4. AI 编程场景下的意图理解

> 来源：import/AI_Native_IDE：重塑软件开发的未来 · import/千问AI助手的工作方式探讨 · import/平衡AI辅助与独立思考的策略

### 4.1 编程场景中意图理解的独特挑战

AI 编程不同于知识库问答——用户通常有一个"脑海中的正确方案"，而 AI 需要在不完美的自然语言描述下，还原出用户想要的代码。这里的意图偏差成本极高：几行代码差异可能导致整个功能的正确性。

| 挑战 | 表现 | 根因 |
|:-----|:------|:------|
| **隐式假设** | 用户没说但默认"应该这样做" | 领域知识差异 |
| **二义性描述** | "处理数据" → 清洗？转换？分析？ | 自然语言的固有模糊性 |
| **上下文缺失** | 用户只描述增量需求，不提现有架构 | 注意力局限 |
| **反悔式修正** | "不是这样的，换个方式" | 用户边思考边调整 |
| **技术栈偏好** | 用户没说"用 Python 写"但期望如此 | 隐性约束 |

### 4.2 从"代码生成"到"意图对齐"的范式演进

```
传统代码生成：
  用户输入 → AI 直接生成代码 → 用户审查 → 修改 → 完成
  优点：快      缺点：意图偏差大，改来改去

现代意图对齐（推荐）：
  用户输入 → AI 生成计划/大纲 → 用户确认 → 逐模块生成 → 测试验证 → 完成
  优点：意图一致性强    缺点：交互步骤多

理想模式：
  用户输入 → AI 追问澄清（主动补全意图） → 生成计划 → 分步确认 → 生成 → 测试 → 完成
```

### 4.3 结构化 Prompt：减少意图偏差的核心方法

在 AI 编程场景，**结构化 Prompt** 是减少意图偏差最有效的手段：

```markdown
【角色】资深 Python 后端工程师
【任务】实现一个异步日志监控器
【背景】现有系统使用 FastAPI，Redis 做消息队列
【输入】日志文件路径（JSON 格式，每行一个日志对象）
【输出】检测到异常日志时通过 Redis 发布消息
【约束】使用 asyncio，不要引入额外依赖
【格式】包含完整代码 + 单元测试示例
```

**结构化 Prompt 的要素**：

| 要素 | 作用 | 反例（模糊） | 正例（清晰） |
|:-----|:------|:------------|:------------|
| **角色** | 限定知识范围和回答风格 | "帮忙写个函数" | "你是有 10 年经验的 Go 后端工程师" |
| **任务** | 定义核心能力 | "处理数据" | "实现一个从 CSV 读取销售数据并按月份聚合统计的函数" |
| **背景** | 提供上下文约束 | — | "现有项目使用 Django 4.2 + PostgreSQL" |
| **输入** | 定义数据范围 | "处理用户信息" | "输入是一个包含 name, age, email 字段的 JSON 列表" |
| **输出** | 明确期望格式 | "生成代码" | "输出一个 Python 文件，包含完整函数 + 类型注解 + 文档字符串 + 3 个测试用例" |
| **约束** | 限定边界条件 | "要高效" | "时间复杂度 O(n), 不允许使用第三方库, 支持 Python 3.10+" |

### 4.4 AI 编程中的"意图验证循环"

在关键节点加入意图验证，减少返工：

```
用户描述需求
    ↓
AI 输出实现方案（PRD/设计文档/伪代码）
    ←── ① 验证：用户确认方案方向
    ↓
AI 生成代码框架（函数签名/类结构/接口定义）
    ←── ② 验证：用户确认架构设计
    ↓
AI 逐模块生成实现代码
    ←── ③ 验证：用户确认每个模块的功能
    ↓
AI 生成测试用例
    ←── ④ 验证：测试覆盖用户预期的场景
    ↓
集成测试
    ←── ⑤ 验证：整体功能是否符合预期
```

**关键原则**：越早验证意图偏差，返工成本越低。在"写代码"之前的"方案设计"阶段对齐意图，成本仅为改代码的 1/10。

### 4.5 AI Native IDE 中的多轮意图融合

> 来源：import/AI_Native_IDE：重塑软件开发的未来

在 AI Native IDE（如 Cursor、Trae、Claude Code）中，意图理解是一个**持续融合**的过程：

```
第一轮：初始描述 → AI 生成代码
  ↓ （用户修改代码）
第二轮：修改说明 → AI 理解"用户想要这样改"的意图
  ↓ （用户添加注释）
第三轮：注释引导 → AI 理解"用户为什么要这样写"
  ↓ （用户选择特定代码段）
第四轮：选段提问 → AI 理解"用户对这个部分有疑问"
```

**关键能力**：
- **上下文感知** — AI 需要理解：当前文件、打开的文件、项目结构、用户操作轨迹
- **增量意图** — 用户不是一次性说清所有需求，而是边写边调整，AI 需要追踪"意图演化"
- **隐性反馈** — 用户删除了 AI 生成的代码 → 意图是"这个不对"；用户在某个函数停留时间长 → 意图是"这里有问题"

## 5. 显式引导与约束：让 AI 更容易做对

> 来源：import/平衡AI辅助与独立思考的策略 · import/Dify知识库系统构建与优化 · import/千问AI助手的工作方式探讨

意图理解的短板在 AI 侧，但弥补可以在人侧。通过显式引导和约束设计，大幅降低意图偏差。

### 5.1 意图引导的基本策略

#### 策略一：预设框架（先给结构，再填内容）

不要让 AI 猜你要的格式，先给出模板：

```markdown
❌ "帮我分析一下竞争对手"
→ AI 可能给：一段文字描述、一个列表、一个表格...猜错

✅ "请用表格对比以下竞争对手，列包括：公司名、核心产品、市场占有率、优缺点"
→ AI 准确输出表格
```

#### 策略二：锚定前提（先定边界，再讨论）

每次交互前明确三个前提：
1. **讨论什么** — 话题边界（不在范围内的明确排除）
2. **为什么讨论** — 目的（决策？调研？学习？）
3. **输出是什么** — 交付物形式（报告？列表？代码？）

#### 策略三：约束条件法（减少 AI 的猜测空间）

四维约束框架：

| 维度 | 定义 | 示例 |
|:-----|:------|:------|
| **内容范围** | 限定时间、地域、领域 | "仅分析 2025-2026 年长三角地区" |
| **格式要求** | 指定结构和长度 | "用表格对比，控制在 300 字内" |
| **排除项** | 明确不要什么 | "不要讨论电池回收问题" |
| **风格指向** | 定义语气和深度 | "用入门级语言，避免专业术语" |

#### 策略四：示例驱动（Show, don't just Tell）

提供 1-2 个理想输出的样例，比任何约束描述都有效：

```
输出格式参考：
[
  { "供应商": "XX", "单价": "$X", "交付周期": "X周", "风险": "低" },
  ...
]
```

### 5.2 反问式澄清：最可靠的意图确认手段

当 AI 不确定用户意图时，主动反问的最有效形式是**提供选项**而非开放提问：

```markdown
❌ "请详细说明您的需求？"
→ 用户可能需要思考很久，且表述仍然可能模糊

✅ "我理解您的需求有几种可能，请确认：
   a) 查看当前配置参数
   b) 修改配置参数
   c) 对比不同配置的性能差异
   d) 其他（请描述）"
→ 用户的确认成本最低，意图最准确
```

**反问的触发条件**（基于多来源实践总结）：

| 条件 | 判断标准 | 行动 |
|:-----|:---------|:------|
| 置信度 < 70% | 对用户意图的推断把握不足 | 反问澄清 |
| 关键假设 | 后续输出依赖某个未确认的前提 | 先确认再继续 |
| 二义性关键词 | "处理"、"优化"、"调整"等意义不明确的词 | 明确具体含义 |
| 可选方案 | 有多个合理方向但用户没说选哪个 | 提供选项供选择 |
| 信息缺失 | 完成任务需要的信息尚未获取 | 追问缺失信息 |

### 5.3 全局显式引导体系

将显式引导固化为可复用的**交互规范**，系统化降低意图偏差：

```markdown
【交互规范】
1. 每次讨论开始前，明确话题边界、目的、输出格式
2. 对模糊输入，AI 必须先反问确认再执行
3. 所有对比分析必须包含维度对齐声明
4. 置信度 < 7/10 的结论标注"待验证"
5. 当发现用户描述可能偏离主题时，主动提醒
```

**执行检查清单**：

- [ ] 是否设定了明确的角色？
- [ ] 是否界定了话题边界和排除项？
- [ ] 是否指定了输出格式和约束？
- [ ] 关键假设是否已确认？
- [ ] 是否存在二义性表述需要澄清？
- [ ] 是否需要提供参考示例？

## 6. 未知/不明确意图下的行为与可预测性提升

> 来源：import/平衡AI辅助与独立思考的策略 · import/千问AI助手的工作方式探讨 · import/Dify知识库系统构建与优化

### 6.1 未知意图下的四种行为模式

当 AI 面对不明确的意图时，通常表现为四种行为模式：

| 模式 | AI 行为 | 用户观感 | 风险评估 |
|:-----|:--------|:---------|:---------|
| **安全兜底** | 给出最宽泛/最常见/最安全的回答 | "说了等于没说" | ⭐ 低 |
| **过度猜测** | 按统计上最可能的路径执行，不问用户 | "AI 自以为是" | ⭐⭐⭐ 高 |
| **反问澄清** | 主动提问确认意图后再回答 | "AI 更谨慎了" | ✅ 最佳 |
| **输出不确定声明** | 标注"我不确定，请补充信息" | "AI 很坦诚" | ⭐ 低，但诚实 |

**推荐行为模式**：优先采用"反问澄清"+ "输出不确定声明"，避免"过度猜测"。

### 6.2 提升意图可预测性的核心方法

#### 方法一：意图模板化

为高频任务预定义意图模板，减少每次的猜测成本：

**知识库查询模板**：
```
意图类型：[信息查询 | 方案对比 | 故障诊断 | 流程指南 | 总结归纳]
关键实体：[用逗号分隔]
查询范围：[时间范围/领域范围]
输出偏好：[表格/列表/段落/代码]
```

**编程任务模板**：
```
任务类型：[新增功能 | 修复Bug | 代码重构 | 性能优化 | 测试生成]
目标语言：[Python/Go/Java/...]
关键约束：[性能要求/兼容性/安全性/...]
涉及文件：[文件名列表]
```

#### 方法二：行为约定（Interaction Contract）

在交互开始前与 AI 约定"当我不明确时应如何行为"：

```
默认约定：
1. 当我给出的需求不完整时，请先列出缺失的信息再反问
2. 当你有多个合理理解方向时，请提供选项供我选择
3. 当你对某个重要前提不确定时，请先确认再继续
4. 当你觉得我的需求可能有更好的解决方案时，请先建议再执行
```

#### 方法三：反馈驱动的意图校准

通过持续反馈训练 AI 更准确地理解你的意图偏好：

```
用户: "这个不对"
AI: "我理解错了。请问具体是：
  a) 方向完全错了
  b) 细节需要调整
  c) 格式不符合预期
  d) 其他"

下次类似场景 → AI 会回忆上次的校准结果 → 意图理解更准确
```

#### 方法四：分层不确定性处理

建立"确定性感知"——不仅知道"不知道"，还要知道"不知道的程度"：

```
┌──────────────────────────────────────────┐
│  确定（置信度 > 90%） → 直接执行          │
│  ╱                                        │
│  ├ 较确定（70-90%） → 执行 + 标注"基于推断" │
│  ├ 不确定（50-70%） → 反问澄清后再执行    │
│  ├ 很不确定（30-50%） → 提供选项供选择    │
│  └ 完全不确定（< 30%） → 坦诚说明+建议方向 │
│                                            │
│ 核心原则：未知不是问题的失败，                │
│ 而是对话的自然组成部分。                     │
└──────────────────────────────────────────┘
```

### 6.3 实战：从模糊需求到精确求解的完整流程

以"帮我优化服务器性能"这个高频模糊需求为例：

```
Step 1 — 意图识别
  用户说："帮我优化服务器性能"
  识别为：L2 隐式意图（说了"优化性能"，没说"优化什么、如何衡量"）

Step 2 — 反问澄清（主动）
  AI："收到"优化服务器性能"需求。为了精确定位，请问：
   1) 当前哪些指标表现不佳？（CPU/内存/IO/网络延迟？）
   2) 具体的性能指标和当前值是多少？（例如 CPU 使用率 90%+？）
   3) 优化的目标是什么？（降低延迟？提高吞吐量？降低资源消耗？）
   4) 运行的负载类型是什么？（AI 训练/数据库/Web 服务？）
   请选择或补充信息。"

Step 3 — 意图确认
  用户："CPU 使用率经常到 95%，运行的是 AI 推理服务，希望降到 70% 以下"
  意图现在已从 L2 降级为 L1（清晰指令）

Step 4 — 精确求解
  AI 基于明确的意图进行针对性分析和建议
```

## 7. 总结：意图理解的五个层次

| 层次 | 核心问题 | 关键能力 | 实现方式 |
|:-----|:---------|:---------|:---------|
| L1 — 听懂 | 能准确理解显式指令 | 语言理解 | 结构化 Prompt + 角色设定 |
| L2 — 推测 | 能从上下文推断隐含目的 | 上下文关联 | RAG + 多轮对话记忆 |
| L3 — 引导 | 能通过反问澄清模糊意图 | 主动交互 | 反问式澄清 + 选项提供 |
| L4 — 校准 | 能从反馈中学习意图偏好 | 记忆与适应 | 反馈闭环 + 意图模板 |
| L5 — 预测 | 能在已知行为模式上预判意图 | 用户模型 | 行为分析 + 个性化模型 |

**最终原则**：
1. **显式优于隐式** — 能说清楚的就说清楚，不要指望 AI 猜
2. **引导优于猜测** — 不确定时反问确认，不要过度推测
3. **确认的代价远低于纠正的代价** — 多花 30 秒确认意图，节省 30 分钟返工
4. **意图不是一次完成，而是逐步精确** — 从模糊到精确是正常路径，接受渐进式对齐
5. **系统设计比单次优化更重要** — 把显式引导固化为交互规范，一劳永逸
