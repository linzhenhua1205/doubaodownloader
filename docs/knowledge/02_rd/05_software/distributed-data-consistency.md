## 分布式系统数据一致性维护方案

### 一、前置理论
#### 1. CAP 定理
分布式系统无法同时满足 **一致性(C)、可用性(A)、分区容错(P)**，网络分区(P)必然存在，只能二选一：
- **CP**：牺牲可用性，保证强一致（金融、交易）
- **AP**：牺牲强一致，保证服务可用（社交、内容）

#### 2. BASE 理论
现代互联网主流思路：**基本可用、软状态、最终一致性**，不追求实时强一致，允许短时间数据不一致，最终自动收敛。

### 二、一致性分级（从强到弱）
#### 1. 强一致性（线性一致性）
所有节点读写全局统一，任意时刻所有副本数据完全一致，读写如同单节点。适用：转账、订单、库存、分布式锁。

#### 2. 顺序一致性
保证操作时序全局有序，但不保证实时可见，弱于线性一致。

#### 3. 因果一致性
仅保证有因果关系的操作顺序一致，无关联操作可乱序。

#### 4. 最终一致性（主流）
短暂不一致，经过一段时间/同步后所有副本数据统一。

### 三、核心分布式一致性算法
#### 1. Paxos
经典强一致算法，解决分布式节点选主、数据同步问题，理论完备但实现复杂。
- 角色：提案者、接受者、学习者
- 核心：多数派（Quorum）机制，超过半数节点同意则提案生效

#### 2. Raft（Paxos 简化版，最常用）
易理解、易落地，当前主流强一致算法，拆分三大模块：
1. **领导者选举**：选唯一主节点，所有写请求走主节点
2. **日志复制**：主节点同步日志到从节点，多数派落盘才返回成功
3. **安全性**：保证日志不会被覆盖、丢失
- 落地组件：etcd、Redis Cluster、TiDB、K8s 集群

#### 3. ZAB（ZooKeeper 专用）
专为 ZK 设计，基于 Paxos 变种，侧重主从同步+崩溃恢复，提供顺序一致性。

### 四、主流工程化解决方案

#### （一）强一致性方案（CP 架构）
##### 1. 主从同步 + 半同步复制
- 架构：1主多从，所有写操作仅走主节点
- 半同步：主写入后，至少1个从节点同步完成才返回客户端成功
- 代表：MySQL 半同步、PostgreSQL 流复制

##### 2. 分布式共识集群（Raft/ZAB）
- 代表组件：etcd(Raft)、ZooKeeper(ZAB)
- 用途：服务注册发现、配置中心、分布式锁、集群元数据

##### 3. 两阶段提交 2PC
- 准备阶段：协调者询问所有参与者，是否可提交
- 提交阶段：全部准备成功 → 统一提交；任意失败 → 全局回滚
- 问题：阻塞风险高、性能差，极少用于高并发业务

#### （二）最终一致性方案（AP 架构，互联网主流）
##### 1. 异步复制（主从异步）
- 主节点写完立即返回，后台异步同步从节点
- 代表：MySQL 异步复制、普通 Redis 主从

##### 2. 消息队列 + 本地事务（可靠消息最终一致）
1. 本地业务事务 + 消息表落库（保证消息不丢）
2. 投递消息到 MQ
3. 消费方执行业务，失败则重试
4. 定时任务补偿死信消息
- 代表：RocketMQ 事务消息、RabbitMQ 本地消息表

##### 3. TCC（Try-Confirm-Cancel）
1. **Try**：资源检查、资源预留（锁定数据）
2. **Confirm**：确认执行业务（正常流程）
3. **Cancel**：取消并释放资源（异常回滚）
- 特点：强侵入业务代码，一致性高，适合核心短事务

##### 4. SAGA 模式
长事务拆分，按串行步骤执行，逆向补偿回滚，适合长流程分布式事务（订单、履约）。

#### （三）分布式数据共享三范式选型
| 共享模式 | 数据分布特征 | 一致性难度 | 并发上限 | 适用业务 |
|---------|-------------|-----------|---------|---------|
| 本地无共享 | 数据全在当前服务节点，不跨节点存取 | 最低（单机ACID） | 极高 | 绝大部分业务：订单、商品、用户 |
| 分片共享 | 数据按规则拆分到多节点，单条数据只存在一个主分片 | 中等 | 高 | 海量流水、用户数据、分库分表 |
| 全局共享 | 一份业务数据全集群多节点存储 | 最高 | 低 | 配置中心、分布式锁元数据（小数据量） |

**落地决策优先级**：优先本地无共享 → 其次分片共享 → 迫不得已才全局共享

### 五、如何在保证可用性的前提下提高一致性
#### 核心设计原则
1. **坚守 AP 基底**：优先保证服务不中断、请求可响应
2. **分级一致**：按数据重要性拆分，核心数据做强约束
3. **就近同步+异步补强**：同步链路控制延迟，异步链路做数据收敛
4. **动态降级而非静态切换**：负载、网络、故障状态下自动调整策略

#### 分层落地方案
1. **差异化复制**：自适应半同步，网络正常调高同步副本数，网络抖动降级为异步复制
2. **就近多副本**：同机房区域内强一致，跨区域最终一致
3. **读写路由优化**：强一致性读走主节点，普通查询走从节点；延迟感知路由，延迟超阈值自动切主库
4. **柔性事务**：放弃重型2PC，改用可靠消息事务、TCC、SAGA
5. **缓存与数据库一致**：先更新数据库再删除缓存；设置短TTL兜底；Binlog异步刷新缓存
6. **版本号/乐观锁**：防写覆盖，零可用性损耗
7. **定时校验+自动修复**：离线巡检补齐一致性

#### 策略分级落地
| 数据等级 | 一致性增强方案 |
|---------|---------------|
| 核心数据（资金、库存） | 读走主库 + 自适应半同步 + 乐观锁 + 分钟级数据巡检 + TCC/可靠消息 |
| 重要业务（订单、流水） | 读写分离+延迟路由 + Binlog 异步同步 + 短TTL缓存 |
| 普通数据（资讯、首页） | 全异步复制 + 定时全量同步 + 长TTL缓存 |

### 六、Raft 网络分区与脑裂防护机制

#### 1. etcd-raft 网络分区场景
假设5节点集群`s1-s5`，初始`s1`为Leader，网络分裂为两个分区：
- 分区A（s1、s2）：少数派分区
- 分区B（s3、s4、s5）：多数派分区，选出新Leader

#### 2. CheckQuorum（法定人数校验）机制
- Leader 每次心跳周期，记录每个 Follower 的 `RecentActive`（是否近期有响应）
- 当选举超时计时器到期，Leader 主动执行 `QuorumActive()`，统计活跃节点数量
- 若活跃节点不足半数，主动降级为 Follower，消除脑裂

#### 3. PreVote（预投票）机制
**设计目的**：阻止分区内节点无效递增 Term，从根源避免网络恢复后集群频繁选举。
- 节点超时后，先进入 **PreCandidate（预候选）** 状态
- 发起预投票：**此阶段不递增 Term**，仅询问其他节点"是否允许我参选？"
- 预投票拿到多数选票 → 才转为正式 Candidate，递增 Term，发起正式选举

#### 4. Lease（租期）机制
- Leader 当选后拥有一段租期，租期内默认自己合法
- 租期未超时：即使收到外部更高 Term 投票请求，也直接忽略，拒绝切换主
- 核心作用：进一步抑制不必要选举，提升集群稳定性

#### 5. 三者分工总结
- **CheckQuorum**：让少数派旧 Leader 主动退位，消灭脑裂
- **PreVote**：防止分区节点无效拉高 Term，避免网络恢复后集群震荡
- **Lease**：租期保护合法 Leader，减少频繁选举切换

### 七、场景选型速览
| 业务场景 | 一致性要求 | 推荐方案 |
|---------|-----------|---------|
| 支付、转账、库存 | 强一致 | Raft集群、MySQL半同步、TCC |
| 配置中心、服务注册 | 强一致 | etcd / ZooKeeper |
| 电商订单、履约长流程 | 最终一致 | SAGA、可靠消息事务 |
| 高并发读业务（资讯/首页） | 最终一致 | 主从异步复制、读写分离 |
| 缓存+数据库组合 | 最终一致 | 先更库后删缓存 + Canal 兜底 |

### 八、Raft 脑裂深度问答

#### 1. 原生 Raft 为什么会出现脑裂？
答：网络分区后，少数派分区中的旧 Leader 不会自动退位，仍可能处理客户端写请求。多数派分区会选举出新 Leader，此时出现双 Leader，即脑裂。

#### 2. CheckQuorum、PreVote、Lease 三者为什么缺一不可？
答：
- **CheckQuorum** 解决"旧 Leader 不退位"的问题，但无法阻止分区节点频繁选举拉高 Term；
- **PreVote** 阻止无效 Term 增长，但无法保证合法 Leader 的稳定性；
- **Lease** 保护合法 Leader 不被频繁切换，但无法处理脑裂场景；
- 三者组合：CheckQuorum 灭脑裂 + PreVote 稳 Term + Lease 减选举 = etcd 稳定运行核心三件套。

#### 3. Raft 脑裂和分布式锁脑裂是一回事吗？
答：本质同源（网络分区），但层级不同：
- **Raft 脑裂**：集群多 Leader；
- **分布式锁脑裂**：多客户端同时拿到锁；
- etcd 通过整套 Raft 增强机制，从底层规避上层锁脑裂。

### 九、Raft 面试高频题详解（含标准答案+踩分点）

#### 一、基础概念题（入门必问）

##### 1. 简述 Raft 三种角色、任期 Term、选举超时、心跳作用

**标准回答**
1. 角色：Leader（处理读写、同步日志）、Follower（被动接收日志/心跳）、Candidate（发起选举）。
2. Term（任期）：全局单调递增，每次选举成功+1，用于区分新旧主，**大 Term 优先**。
3. 选举超时：Follower 长时间收不到 Leader 心跳，随机超时后转为 Candidate 发起选举。
4. 心跳：Leader 定时发送，阻止 Follower 发起选举，维持主身份。

**踩分点**：Term 全局递增、随机选举超时（避免同时选举）。

##### 2. Raft 为什么要求多数派（Quorum）？

**标准回答**
多数派机制保证：**任意两个多数派集合必有交集**。
1. 选举：只有拿到半数以上选票才能成为 Leader，集群同一任期最多一个 Leader；
2. 日志复制：日志被多数节点落盘才视为提交，保证已提交日志不会丢失、不会被回滚；
3. 容错：可容忍 `(N-1)/2` 个节点故障。

**延伸追问**：5节点集群最多挂几台？答：2台。

#### 二、网络分区 & 脑裂核心题（重点）

##### 1. 什么是 Raft 脑裂？如何产生？

**标准回答**
脑裂 = 集群出现**两个及以上合法 Leader**。
产生场景：集群发生**网络分区**，节点被分割为多个孤岛。
举例（5节点 s1-s5，原 Leader s1）：
- 分区1（少数派：s1、s2）：原 Leader 收不到多数节点心跳响应，**但初期仍认为自己是 Leader**；
- 分区2（多数派：s3、s4、s5）：Follower 超时，选出**新 Leader**；
- 整个集群同时存在两个 Leader，即脑裂。

**踩分点**：必须点明「网络分区」+「少数派旧Leader未主动退位」是根因。

##### 2. 纯原生 Raft（无 CheckQuorum），分区后旧 Leader 会不会主动下台？为什么？

**标准回答**
**不会主动下台**。
原生 Raft 规则：
- Leader 只负责发心跳、同步日志，**不会主动校验自己是否还拥有多数派**；
- 只要少数派分区内节点持续回复心跳，旧 Leader 就会一直维持 Leader 身份；
- 最终形成长期双 Leader 脑裂，读写混乱。

**考点**：区分「原生Raft」和「etcd增强版Raft」，这是面试区分深浅的关键。

##### 3. etcd 如何解决「少数派分区旧Leader 不肯退位」的问题？（CheckQuorum 必考）

**标准回答**
etcd 引入 **CheckQuorum（法定人数校验）** 机制，让 Leader **周期性自检**：
1. Leader 每次心跳周期，记录每个 Follower 的 `RecentActive`（是否近期有响应）；
2. 当选举超时计时器到期，Leader 主动执行 `QuorumActive()`，统计**活跃节点数量**；
3. 若**活跃节点不足半数**，说明自己已不在多数派分区，**主动降级为 Follower**；
4. 降级后不再处理写请求、不再发送心跳，彻底解决脑裂。

**精简答题版（面试口述）**
etcd 通过 CheckQuorum 让 Leader 定期检查在线活跃节点是否达到多数派，少数派分区内的旧 Leader 自检不通过，主动降为 Follower，消除双 Leader。

#### 三、PreVote 预投票机制（最高频考点）

##### 1. 没有 PreVote 时，网络分区恢复后会出现什么严重问题？

**标准回答**
1. 分区期间，少数派节点（原Leader）超时后会变成 Candidate，**不断递增 Term**（选举失败也会涨任期）；
2. 网络恢复后，少数派节点带着**更大的 Term** 发起正式选举请求；
3. 根据 Raft 规则：节点收到更大 Term 请求，会立刻放弃当前 Leader、投票给对方；
4. 合法新 Leader 被迫下台，集群反复切换主节点，**服务抖动、不可用、日志混乱**。

**核心结论**：无 PreVote → 分区节点 Term 无效暴涨，网络恢复后引发集群震荡。

##### 2. PreVote 预投票的设计目的 & 完整流程？

**标准回答**
**设计目的**：阻止分区内节点**无效递增 Term**，从根源避免网络恢复后集群频繁选举、主节点乱切换。

**完整流程**
1. 节点超时后，先进入 **PreCandidate（预候选）** 状态；
2. 发起 **预投票（PreVote）**：**此阶段不递增 Term**，仅询问其他节点："是否允许我参选？"；
3. 预投票拿到多数选票 → 才转为正式 Candidate，**递增 Term**，发起正式选举；
4. 预投票失败（如少数派分区）→ 不进入正式选举，Term 保持不变。

**踩分核心**：预投票**不涨Term**，正式选举才涨Term。

##### 3. PreVote 能彻底解决分区问题吗？搭配什么机制效果最好？

**标准回答**
不能单独彻底解决。
- PreVote：解决**Term 疯涨、恢复后乱选举**问题；
- CheckQuorum：解决**少数派旧Leader 不下台、脑裂**问题；
生产环境 etcd 标准组合：**PreVote + CheckQuorum + Lease（租期）** 三重防护。

#### 四、Lease 租期机制（etcd 进阶考点）

##### 1. Raft Lease 租期是什么？作用？

**标准回答**
Lease 是 Leader 的**租约有效期**：
1. Leader 当选后拥有一段租期，租期内默认自己合法；
2. 租期未超时：即使收到外部更高 Term 投票请求，节点会直接忽略，拒绝切换主；
3. 租期超时：才正常参与选举、切换主。

**核心作用**：进一步**抑制不必要选举**，提升集群稳定性，配合 PreVote 大幅减少主切换次数。

##### 2. PreVote + CheckQuorum + Lease 三者分工总结（必背综合句）

- **CheckQuorum**：让少数派旧 Leader 主动退位，消灭脑裂；
- **PreVote**：阻止分区节点无效拉高 Term，避免网络恢复后集群震荡；
- **Lease**：租期保护合法 Leader，减少频繁选主；
- **组合效果**：etcd 稳定运行核心三件套，缺一不可。

### 十一、一致性方案选型决策树

```
业务需求
    ↓
是否需要实时强一致？
    ├─ 是 → 是否高并发？
    │       ├─ 是 → Raft集群（etcd/TiDB）
    │       └─ 否 → MySQL半同步 / 2PC
    │
    └─ 否 → 是否长流程事务？
            ├─ 是 → SAGA模式
            └─ 否 → 是否跨服务？
                    ├─ 是 → 可靠消息队列（本地消息表）/ TCC
                    └─ 否 → 主从异步复制 + 读写分离
```

### 十二、落地风险提示

| 风险点 | 描述 | 规避策略 |
|-------|------|---------|
| **同步降级** | 半同步复制超时降级为异步，可能丢失数据 | 设置合理超时阈值，降级时记录日志告警 |
| **消息堆积** | MQ 消息积压导致最终一致性延迟过大 | 设置消息过期时间，监控堆积水位，及时扩容 |
| **补偿失败** | TCC/SAGA 补偿操作失败 | 设置最大重试次数，超过阈值转入人工处理 |
| **缓存击穿** | 缓存与数据库不一致导致脏读 | 先更库后删缓存，设置短TTL兜底，Binlog异步刷新 |
| **脑裂数据冲突** | 网络恢复后双Leader数据冲突 | Raft/ZAB 自动选主+日志回滚，人工介入极端场景 |
