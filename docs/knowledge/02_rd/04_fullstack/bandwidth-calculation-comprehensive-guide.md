# 🧮 超节点场景带宽计算全面指南

> **版本**: v1.0 | **创建**: 2026-06-24
> **目标**: 系统性澄清超节点架构中各类带宽单位的混用问题，建立可复用的计算框架
> **适用**: 服务器架构师、系统设计、性能评估

---

## 一、引言：为什么带宽计算这么容易乱？

### 1.1 混乱的根源

超节点场景中带宽计算处处是陷阱，原因来自**多个维度的单位混用**：

| 维度 | 常见混用 | 实际差异 |
|:-----|:---------|:---------|
| **bit vs byte** | Gbps 与 GB/s 不区分 | 8倍差异 |
| **单向 vs 双向** | "带宽400G"有时指单向，有时总和 | 2倍差异 |
| **物理层 vs 有效负载** | 码流加了编码/FEC后标称400G vs 用户实际数据 | ~3-15%差异 |
| **1000 vs 1024** | 存储用 GiB/Si 前缀混用在网络场景 | ~7%差异 |
| **GT/s vs Gbps** | PCIe GT/s 常被当 Gbps | PCIe 5.0 32 GT/s ≠ 32 Gbps | 
| **端口 vs 链路** | 1个400G端口 = 1条还是多条lane/通道 | 取决于技术 |

### 1.2 本指南的核心方法

```
每层问三个问题：
① 这个数字标注的是「哪一层」的速度？（物理/编码后/MAC/有效负载）
② 这个是「单向」还是「双向」？
③ 这个是「峰值」还是「有效/持续」？
```

## 二、基础单位体系

### 2.1 单位对照表

| 缩写 | 全称 | 基准 | 说明 |
|:-----|:-----|:-----|:------|
| **b** | bit | 2进制位 | 网络/通信的基本单位 |
| **B** | Byte = 8 bits | 8 bit | 存储的基本单位 |
| **GT/s** | Giga Transfers per second | 每次传输1-bit | PCIe 用，不等同于 Gbps（有编码） |
| **Gbaud** | Giga Baud (symbols/s) | 每位号承载多bit | PAM-4 每个 baud = 2 bits |
| **Gbps** | Giga bits per second | bits/s | 数据速率标准单位 |
| **GB/s** | Giga bytes per second | 8×Gbps | 数据中心常用 |
| **GiB/s** | Gibi bytes per second | 2^30 = 1,073,741,824 B/s | 存储/OS 常用（唯1024） |

### 2.2 十进制 vs 二进制前缀

这是**最大的混淆源之一**，尤其在混合讨论网络和存储时：

| 前缀 | 值 | 进制 | 常用场景 |
|:-----|:----|:-----|:---------|
| 1 Kbps / KHz | 1,000 | 十进制 (10³) | **所有网络速率**、时钟频率 |
| 1 Mbps | 1,000,000 | 十进制 (10⁶) | 网络 |
| 1 Gbps | 1,000,000,000 | 十进制 (10⁹) | **所有网络速率** |
| 1 Tbps | 1,000,000,000,000 | 十进制 (10¹²) | 网络 |
| 1 KB (SI) | 1,000 B | 十进制 (10³) | 硬盘厂商标称、部分网络吞吐 |
| **1 KiB** | **1,024 B** | **二进制 (2¹⁰)** | **OS/内存/文件系统报告** |
| 1 MB (SI) | 1,000,000 B | 十进制 (10⁶) | 硬盘标称 |
| **1 MiB** | **1,048,576 B** | **二进制 (2²⁰)** | **OS/DDR/内存** |
| **1 GB (SI)** | **1,000,000,000 B** | **十进制 (10⁹)** | 硬盘/NVMe SSD 厂商标称 |
| **1 GiB** | **1,073,741,824 B** | **二进制 (2³⁰)** | **OS/Linux `free`/内存 DDR/HBM** |

> **铁律 1**: 网络带宽永远用十进制（1 Gbps = 1,000,000,000 bps）
> **铁律 2**: 内存/HBM 带宽永远用二进制（1 GB/s 在 DDR/HBM 语境下实际是 GiB/s，但行业习惯混用，需看上下文：1 GB/s ≈ 0.931 GiB/s）
> **铁律 3**: 存储厂商标称用十进制，OS 报告用二进制（512 GB SSD ≈ 477 GiB）

**快速换算（单向）**：

| 类型 | 公式 | 示例 |
|:-----|:------|:------|
| bit → byte | 除以 8 | 400 Gbps = 50 GB/s |
| GT/s → Gbps (PCIe 5.0) | GT/s × 128/130 | 32 GT/s = 31.51 Gbps/lane |
| Gbps → GT/s (PCIe 5.0) | Gbps × 130/128 | 31.51 Gbps = 32 GT/s |
| 十进制 GB → 二进制 GiB | GB × 0.931 | 50 GB/s ≈ 46.55 GiB/s |
| Gbps → GiB/s | Gbps ÷ 8 ÷ 1.024 ÷ 1.024 ÷ 1.024 | 400 Gbps = 46.55 GiB/s |

### 2.3 单向 vs 双向

| 场景 | 单向 = | 双向 = | 常见标注差异 |
|:-----|:-------|:-------|:------------|
| **NVLink** | 1 link × 每方向带宽 | 单向 × 2 | NVIDIA 经常标**双向**总和（如 NVLink 1.0 "160 GB/s" = 2×80 GB/s） |
| **PCIe** | 单方向 | 单向 × 2 | 行业标准标**单向每方向**（如 PCIe 5.0 ×16 = "~64 GB/s" = 单向） |
| **以太网** | 单方向 | 单向 × 2 | "400GbE" = **单向满双工** 400 Gbps |
| **InfiniBand** | 单方向 | 单向 × 2 | "NDR 400G" = 单向 400 Gbps |
| **HBM** | 单方向 | 单向 × 2 | 标称带宽通常为**单向** × 2（读+写同时发生） |

> **铁律 4**: 看到"400G"先说清是单向还是双向。但绝大多数情况下：
> - 网络端口标称 = **单向满双工**（400GbE = 单向 400 Gbps）
> - 总带宽标称 = 单向 × 2（400GbE 全双工 = 800 Gbps 总带宽）

## 三、PCIe 带宽详解

### 3.1 编码开销

PCIe 各代使用了不同的编码方案，这是 GT/s ≠ Gbps 的原因：

| 代 | 每 Lane 编码 | 编码效率 | 说明 |
|:---|:------------|:--------:|:------|
| PCIe 1.0 | 8b/10b | 80% | 每传输10b只有8b是数据，2b是编码 |
| PCIe 2.0 | 8b/10b | 80% | 同上 |
| PCIe 3.0 | **128b/130b** | **98.46%** | 每130b只有2b开销，大幅提升 |
| PCIe 4.0 | 128b/130b | 98.46% | 同上 |
| PCIe 5.0 | 128b/130b | 98.46% | 同上 |
| PCIe 6.0 | 128b/130b + FLIT | ~98.1% | FLIT 增加少量开销 |

### 3.2 各代 PCIe Lane 带宽计算

#### 核心公式

```
Gbps_per_lane = GT/s × 编码效率
GB/s_per_lane = Gbps_per_lane ÷ 8
```

#### 逐代计算

| 代 | GT/s | 编码 | Gbps/lane | GB/s/lane | ×16 单向 | ×16 双向 |
|:---|:----:|:----:|:---------:|:---------:|:---------:|:---------:|
| **PCIe 3.0** | 8 GT/s | 128b/130b | **7.88** | **0.985** | **~15.75 GB/s** | **~31.5 GB/s** |
| **PCIe 4.0** | 16 GT/s | 128b/130b | **15.76** | **1.969** | **~31.5 GB/s** | **~63 GB/s** |
| **PCIe 5.0** | 32 GT/s | 128b/130b | **31.51** | **3.939** | **~63 GB/s** | **~126 GB/s** |
| **PCIe 6.0** | 64 GT/s | 128b/130b + FLIT | **~62.8** | **~7.85** | **~125.7 GB/s** | **~251 GB/s** |

> **关键洞察**: PCIe 5.0 ×16 的 "64 GB/s" 是**单向**值，**不是双向**。常见说法 "PCIe 5.0 ×16 = 64 GB/s" 已经约等于已包含128b/130b编码效率（32×16×0.9846/8 = 63 GB/s）。

#### 详细计算示例

以 **PCIe 5.0 ×16** 为例：

```
原始速度:        32 GT/s/lane × 16 lanes = 512 GT/s
去掉130b编码:    512 GT/s × 128/130 = 503.75 Gbps
÷8 得 GB/s:     503.75 Gbps ÷ 8 = 62.97 GB/s
常用取整值:      ~63 GB/s（单向），~126 GB/s（双向）

厂商经常写:      "64 GB/s"（向上取整，容易记忆）
严格算:          63 GB/s（更精确）
```

#### 编码效率敏感度分析

| 编码 | 效率 | 32 GT/s ×16 单向 | vs 理想无编码 |
|:-----|:----:|:----------------:|:-------------:|
| 理想（无编码） | 100% | 64.00 GB/s | 基准 |
| 128b/130b | 98.46% | 63.02 GB/s | -1.54% |
| 8b/10b | 80% | 51.20 GB/s | -20% |

> PCIe 3.0 从 8b/10b 切换到 128b/130b 是巨大的效率提升（+23% 有效带宽），而 3.0→5.0/6.0 的编码开销变化很小。

### 3.3 PCIe 带宽对照速查表

| Lane 数 | PCIe 3.0 | PCIe 4.0 | PCIe 5.0 | PCIe 6.0 |
|:--------|:--------:|:--------:|:--------:|:--------:|
| ×1 (单向) | ~0.99 GB/s | ~1.97 GB/s | ~3.94 GB/s | ~7.85 GB/s |
| ×4 (单向) | ~3.94 GB/s | ~7.88 GB/s | ~15.75 GB/s | ~31.4 GB/s |
| ×8 (单向) | ~7.88 GB/s | ~15.75 GB/s | ~31.5 GB/s | ~62.9 GB/s |
| ×16 (单向) | ~15.75 GB/s | ~31.5 GB/s | **~63 GB/s** | **~125.7 GB/s** |
| ×16 (双向) | ~31.5 GB/s | ~63 GB/s | ~126 GB/s | ~251 GB/s |

### 3.4 PCIe 带宽在网络场景中的应用

判断网卡能否在 PCIe 上跑满速：

| 网卡 | 端口 | 单向需 PCIe 带宽 | PCIe 配置 | PCIe 单向 | 是否够？ |
|:-----|:-----|:----------------:|:----------:|:---------:|:--------:|
| CX7 | 1×400GbE | 400 Gbps = 50 GB/s | Gen5 ×16 | ~63 GB/s | ✅ 充足 |
| CX7 | 2×400GbE | 800 Gbps = 100 GB/s | Gen5 ×16 | ~63 GB/s | ❌ 不饱和（2口无法同时满速） |
| BF3 | 1×400GbE | 400 Gbps = 50 GB/s | Gen5 ×16 | ~63 GB/s | ✅ 充足 |
| BF3 | 2×400GbE | 800 Gbps = 100 GB/s | Gen5 ×16 | ~63 GB/s | ❌ 不饱和（上限~63%利用率） |
| CX9 SuperNIC | 1×800GbE | 800 Gbps = 100 GB/s | Gen6 ×16 | ~125.7 GB/s | ✅ 充足 |
| CX9 SuperNIC | 2×800GbE | 1,600 Gbps = 200 GB/s | Gen6 ×16 | ~125.7 GB/s | ❌ 不饱和 |

> **工程要点**: BF3×1 在 PCIe Gen5 ×16 上接 1×400G 足够；如果要同时跑满 2×400G，PCIe 会是瓶颈（实际需要 ~100 GB/s 单向传输，PCIe 5.0 ×16 只提供 ~63 GB/s）。这也是为什么需要 BF4（PCIe Gen6 ×16 = ~125.7 GB/s 单向）来饱和多端口场景。

## 四、以太网带宽详解

### 4.1 以太网速率的三层定义

以太网标称速率（如 "400GbE"）定义在 **MAC/PLS 服务接口层**，而非物理层。这三层之间的速率差异是最大的混淆来源：

```
┌─────────────────────────────────────────┐
│  应用层数据                              │
│  (TCP/UDP/RDMA payload)                 │
├─────────────────────────────────────────┤
│  ▲ 协议包头开销 (Ethernet/IP/UDP/IB BTH) │
├─────────────────────────────────────────┤
│  MAC/PLS 层 ← "400G" 标称速率定义位置    │
│  (64b/66b 编码后的数据)                  │
├─────────────────────────────────────────┤
│  ▲ FEC (RS-FEC 544/514 = ~5.8% 开销)    │
├─────────────────────────────────────────┤
│  ▲ PAM-4 编码                           │
├─────────────────────────────────────────┤
│  物理层 (Lane/PMA)                      │
└─────────────────────────────────────────┘
```

### 4.2 编码层解析 (64b/66b)

| 项目 | 值 | 说明 |
|:-----|:----|:------|
| 原始数据 | 64 bits | 用户数据 |
| 编码后 | 66 bits | 每66b中2b用于同步头 |
| 编码效率 | 64/66 = **96.97%** | |
| 开销比 | 3.13% | 即每传输66b只有64b有效 |

> 100GbE 及以上速率均使用 64b/66b 编码（100GbE 以下使用 8b/10b 或 64b/65b）

### 4.3 FEC (前向纠错)

400GbE 使用 **RS-FEC(544,514)**，也称为 KP-FEC：

| 项目 | 值 | 说明 |
|:-----|:----|:------|
| 原始用户数据 | 514 bytes | 在 FEC 块中 |
| 添加纠错码后 | 544 bytes | 额外 30 bytes RS 码 |
| FEC 效率 | 514/544 = **94.49%** | |
| FEC 开销 | 5.84% | |

### 4.4 三层综合效率

| 层级 | 效率 | 累计效率（乘法） |
|:-----|:----:|:----------------:|
| MAC/PLS 层（64b/66b 后） | 96.97% | 96.97% |
| FEC （RS 544/514）后 | 94.49% | 91.63% |
| **物理层真实负载 vs 标称"400G"** | **91.63%** | |

> **重要**: 但注意——"400G" 标称值本身定义在 **MAC/PLS 层**，也就是说 **64b/66b 编码效率已经体现在"400G"这个数字中**了。所以：

```
标称 400GbE = MAC/PLS 层的 400 Gbps（纯数据，64b/66b已包含）
FEC 开销在标称之外: 400 × 544/514 = ~423.3 Gbaud（实际物理线速率）
PAM-4: 423.3 / 2 = 211.65 Gbaud（每baud 2 bits，即8×26.46 Gbaud）
```

#### 正确理解方式

| 你听到的 | 实际含义 |
|:---------|:---------|
| "400GbE" | MAC/PLS 服务接口 400 Gbps，**64b/66b 已考虑在内** |
| 物理线速率 | ~425 Gbps（400 × 514/544 × 66/64） |
| 有效负载（去掉L1/L2包头） | 取决于包大小（见下文） |

### 4.5 包结构开销

以太网帧在传输时有额外的包结构开销：

```
┌──────────────────────────────────────────────┐
│  Preamble  (7B) + SFD (1B)                   │ 8B (L1)
├──────────────────────────────────────────────┤
│  目标MAC (6B) + 源MAC (6B)                   │
│  EtherType / VLAN Tag (可选 4B)              │
│  Payload (46-1500B 标准，或 ~9000B 巨帧)     │
│  CRC-32 (4B)                                 │
├──────────────────────────────────────────────┤
│  Interframe Gap (12B)                        │ 12B (L1)
└──────────────────────────────────────────────┘
```

#### 包开销对有效带宽的影响

对于 **1500B MTU（标准）**：

```
每包总开销: 8 (Preamble+SFD) + 14 (MAC header) + 4 (CRC) + 12 (IFG) = 38B
每包总传输: 1500 + 38 = 1538B
有效载荷比: 1500/1538 = 97.53%
综合有效带宽（含FEC）: 400 Gbps × 97.53% = ~390.1 Gbps（仅L1-L2）
加上TCP/IP开销（40-60B）: 1460/1538 = 94.93%
最终有效（含TCP/IP）: 400 × 94.93% = ~379.7 Gbps
```

对于 **9000B MTU（巨帧）**：

```
每包总开销: 38B
每包总传输: 9000 + 38 = 9038B
有效载荷比: 9000/9038 = 99.58%
综合有效带宽（含FEC）: 400 Gbps × 99.58% = ~398.3 Gbps
加上TCP/IP开销（40-60B）: 8960/9038 = 99.14%
最终有效（含TCP/IP）: 400 × 99.14% = ~396.6 Gbps
```

#### RDMA 场景的包开销

RDMA（RoCEv2）使用 **InfiniBand BTH（12-16B）+ UDP（8B）+ IP（20B）+ Ethernet（14B）+ 额外**，精简后：

| MTU | 有效载荷比 | 近似有效带宽（从400G中） |
|:----|:----------:|:------------------------:|
| 1500 B | ~97% | **~388 Gbps** |
| 4096 B（RDMA推荐Min） | ~98.5% | **~394 Gbps** |
| 9000 B（巨帧，推荐） | ~99.3% | **~397 Gbps** |

> **总结**: 生产环境 **9000 MTU + RoCEv2** 场景，400GbE 的有效负载带宽约为 **~395-397 Gbps**（扣除所有开销后），相比标称 400G 损失约不到 1.5%。FEC 开销虽然在物理层发生，但严格来说它不减少"标称"值——标称 400G 已经是在 64b/66b 编码后了。

| 标称 | 单向 (Gbps) | 单向 (GB/s, /8) | 双向 (GB/s) | 物理线速率(约) | 
|:-----|:-----------:|:---------------:|:-----------:|:--------------:|
| 1GbE | 1 | 0.125 | 0.25 | 1.25 Gbps (8b/10b) |

## 五、InfiniBand 带宽详解

### 5.1 InfiniBand 速率定义

InfiniBand 的速率标称**定义在物理链路层**（即编码后有效速率），与以太网的 MAC 层对标方式有所不同：

| 代数 | 全称 | 单向 4x (1 端口) | 编码 | 说明 |
|:-----|:------|:----------------:|:----:|:------|
| **FDR** | Fourteen Data Rate | 54.54 Gbps | 8b/10b | 实际每lane 14.0625 Gbps |
| **EDR** | Enhanced Data Rate | 100 Gbps | 64b/66b | 实际每lane 25.78125 Gbaud |
| **HDR** | High Data Rate | **200 Gbps** | 64b/66b (PAM-4) | 4 lanes × 50 Gb/s/lane |
| **NDR** | Next Data Rate | **400 Gbps** | 64b/66b (PAM-4) | 4 lanes × 100 Gb/s/lane |
| **XDR** | eXtreme Data Rate | **800 Gbps** | 64b/66b (PAM-4) | 4 lanes × 200 Gb/s/lane |
| **GDR** | Giga Data Rate | **1.6 Tbps** | (在研) | - |

> **关键**: HDR = 200 Gbps @ 4x（即 InfiniBand 单端口 4 条 lane 总和），NDR = 400 Gbps @ 4x

### 5.2 InfiniBand HDR/NDR 分拆速查

| 名称 | Lane 数 | 有效速率 (Gbps) | 编码 | 备注 |
|:-----|:-------:|:---------------:|:----:|:-----|
| HDR100 (2x) | 2 | 100 | 64b/66b | ConnectX-6 常见配置 |
| HDR (4x) | 4 | **200** | 64b/66b | ConnectX-6 标准配置 |
| NDR200 (2x) | 2 | 200 | 64b/66b | ConnectX-7/8 常见配置 |
| NDR (4x) | 4 | **400** | 64b/66b | ConnectX-7/8 标准配置 |
| XDR (4x) | 4 | **800** | 64b/66b (在研) | ConnectX-9 预期 |

### 5.3 IB vs Eth 的编码差异

| 技术 | 编码 | 标称值定义层 | 标称包含编码？ |
|:-----|:----:|:------------:|:--------------:|
| FDR IB | 8b/10b | 物理层 | ✅ 是，标称 54.54 Gbps 已扣除 8b/10b |
| EDR IB | 64b/66b | 物理层 | ✅ 是 |
| HDR/NDR IB | 64b/66b + PAM-4 | 物理层 | ✅ 是 |
| 100G/200G/400G Eth | 64b/66b + RS-FEC | MAC/PLS 层 | 64b/66b ✅，FEC ❌ |
| PCIe 各代 | 8b/10b 或 128b/130b | 物理层(GT/s) | GT/s 不含编码，Gbps 含编码 |

> **实际上 IB 和 Eth 在 400G 级别的有效数据速率非常接近**（FEC 差异 ~5%），但 InfiniBand 的标称值"更诚实"（不包含上层协议包头开销，但已经扣除物理层编码）。

## 六、NVLink 带宽详解

### 6.1 各代 NVLink 规格

| 代 | 每 SubLink 带宽 | SubLinks/GPU | 总带宽/GPU(双向) | 每信号速率 | 编码 | 硬件 |
|:---|:--------------:|:------------:|:----------------:|:----------:|:----:|:-----|
| **NVLink 1.0** | 20 GB/s/dir | 4 | **160 GB/s** | 20 Gbps/lane | NRZ | P100 |
| **NVLink 2.0** | 25 GB/s/dir | 6 | **300 GB/s** | 25 Gbps/lane | NRZ | V100 |
| **NVLink 3.0** | 50 GB/s/dir | 6 | **600 GB/s** | 25 Gbps/lane? | NRZ? | A100 |
| **NVLink 4.0** | 50 GB/s/dir | 18 | **1.8 TB/s** | >25 Gbps? | PAM-4? | H100 |
| **NVLink 5.0** | 50 GB/s/dir | 18 | **1.8 TB/s** | 同4.0? | PAM-4? | B200 |
| **NVLink 6.0** | 100 GB/s/dir | 18 | **3.6 TB/s** | ... | ... | Vera Rubin |

### 6.2 NVLink 带宽标注的陷阱

> **⚠️ NVIDIA 标注 NVLink 带宽时几乎总是使用双向总和值**

| 产品 | NVIDIA 宣称 | 实际含义 |
|:-----|:-----------|:---------|
| P100 NVLink | "160 GB/s" | 4 SubLinks × 20 GB/s/dir × 2 = **160 GB/s 双向** |
| V100 NVLink | "300 GB/s" | 6 × 25 GB/s/dir × 2 = 300 GB/s 双向 |
| H100 NVLink | "900 GB/s" | 18 × 50 GB/s/dir → 注意 H100 的 18 个 SubLink 是 9 个 Link（每 Link 1 个 SubLink 出 + 1 个入），每个 SubLink 是双向 50 GB/s？|

等等，NVLink 的术语比较特殊。让我详细说明。

### 6.3 NVLink 术语澄清

**NVLink SubLink**: 一个 SubLink 是**单向**的（一个方向一条 TX pair + RX pair）

**NVLink Link**: 1 个 Link = 1 对 SubLink（TX + RX）= **双向**

所以：
- NVLink 4.0 (H100): 每 SubLink = 50 GB/s（单向），每对 SubLink = 100 GB/s（双向）
- H100 有 18 个 SubLink = 9 个 Link
- 总带宽: 18 × 50 GB/s（单向）= 900 GB/s（从GPU发出）
- **NVIDIA 标称**: "900 GB/s" = **双向总和**（9 Links × 2 × 50 GB/s）

> **对照 PCIe**: PCIe 标称的是单向（如 ×16 Gen5 = 63 GB/s），NVLink 标称的是双向总和。

#### 双向总和混淆举例

有 8 个 H100 通过 NVSwitch 互连，每个 GPU 有 900 GB/s NVLink 带宽：

```
每个 H100 NVLink: 18×50 GB/s SubLinks = 900 GB/s 双向（NVIDIA 说法）
                  = 9×100 GB/s Links = 900 GB/s 双向
                  实际每方向: 900/2 = 450 GB/s 单向（从GPU发出）
                  
相当于 PCIe 语境: "PCIe 5.0 ×16 = 63 GB/s 单向 ≈ 126 GB/s 双向"
                  "NVLink 4.0 on H100 = 450 GB/s 单向 ≈ 900 GB/s 双向"
```

### 6.4 NVLink vs NVSwitch vs PCIe 带宽对比

| 互连 | GPU-×16 PCIe Gen5 | H100 NVLink 4.0 | V100 NVLink 2.0 |
|:-----|:-----------------:|:----------------:|:----------------:|
| 单向（从GPU出发） | ~63 GB/s | **450 GB/s** | 150 GB/s |
| 双向总和 | ~126 GB/s | **900 GB/s** | 300 GB/s |
| 相比 PCIe 加速比 | 基准 | **7.1x** | 2.4x |

### 6.5 NVLink 在超节点中的带宽拓扑

以 H100 NVL 平台为例（来自官方规格）：

| 场景 | NVLink 带宽 | 对应 PCIe Gen5 ×16  |
|:-----|:-----------:|:-------------------:|
| 1 GPU → NVSwitch | 450 GB/s 单向 | 63 GB/s 单向（~7x） |
| 1 GPU → 4 NVSwitch | 共享 900 GB/s 总双向 | 126 GB/s 总双向 |
| NVL36 机架总聚合 | 36 GPU × 900 GB/s = **32.4 TB/s 双向** | - |

### 6.6 关于线速率

NVIDIA 公开 NVLink 的"信号速率"信息较少。根据已知信息：

- NVLink 1.0: 每 lane 20 Gbps (NRZ), 8 lanes/SubLink = 160 Gbps/SubLink = 20 GB/s/SubLink (单向)
  - 编码: 通常认为是 128b/130b 或类似高效编码（NVIDIA 私有实现）
- NVLink 2.0: 每 lane 25 Gbps (NRZ), 8 lanes/SubLink = 200 Gbps/SubLink = 25 GB/s/SubLink
- NVLink 3.0-4.0: 每 SubLink 50 GB/s，可能是 PAM-4
- NVLink 6.0 (Vera Rubin 的 NVL72): 每 SubLink 100 GB/s

> **关键**: NVLink 使用**私有低开销编码**（非以太网的 64b/66b 或 PCIe 的 128b/130b），编码效率很高，带宽数字基本就是有效数据速率。

## 七、内存带宽 (HBM/DDR)

### 7.1 DDR 带宽计算

DDR 的标称值也存在混淆——"DDR5-4800"中的"4800"是指 **MT/s**（Mega Transfers per second），即数据传输速率，而非时钟频率或单个 bit 速率。

#### 基本公式

```
DDR 带宽 (GB/s) = 数据传输速率 (MT/s) × 位宽 (bits) ÷ 8 × 通道数
```

| 类型 | 传输速率 | 内部时钟 | 位宽 | 单 Channel 带宽 | 双 Channel 带宽 |
|:-----|:--------:|:--------:|:----:|:---------------:|:---------------:|
| DDR4-3200 | 3,200 MT/s | 1,600 MHz | 64 bit | 25.6 GB/s | 51.2 GB/s |
| DDR5-4800 | 4,800 MT/s | 2,400 MHz | 64 bit | **38.4 GB/s** | 76.8 GB/s |
| DDR5-5600 | 5,600 MT/s | 2,800 MHz | 64 bit | 44.8 GB/s | 89.6 GB/s |
| DDR5-6400 | 6,400 MT/s | 3,200 MHz | 64 bit | 51.2 GB/s | 102.4 GB/s |
| DDR5-7200 | 7,200 MT/s | 3,600 MHz | 64 bit | 57.6 GB/s | 115.2 GB/s |
| DDR5-8000 | 8,000 MT/s | 4,000 MHz | 64 bit | 64.0 GB/s | 128.0 GB/s |

> **⚠️ 混淆点**: "DDR5-4800"中的 "4800" 既不是 MHz 也不是 MHz × 2，它是 **MT/s**（百万次数据传输/秒）。DDR 在每个时钟周期的上升沿和下降沿都传输数据，所以等效于 2×时钟频率。但如果说 "4800MHz" 其实是不准确的，正确说法是 "4800 MT/s"。

#### 带宽计算示例

```
DDR5-5600, 双通道:
  = 5,600 MT/s × 64 bit/channel × 2 channels ÷ 8
  = 5,600 × 128 ÷ 8
  = 89,600 MB/s
  = 89.6 GB/s
```

### 7.2 HBM 带宽计算

HBM 使用**宽位宽 × 相对较低的频率**的方案来达到高带宽：

| 代 | 每 Stack 位宽 | 速率(MT/s) | 每 Stack 带宽 | 典型配置 |
|:---|:-------------:|:----------:|:-------------:|:---------|
| **HBM2** | 1,024 bit | 2.0 GT/s | **256 GB/s** | V100: 2 stacks = 512 GB/s |
| **HBM2e** | 1,024 bit | 3.2 GT/s | **410 GB/s** | A100: 3 stacks = 2 |
| **HBM3** | 1,024 bit | 6.4 GT/s | **819 GB/s** | H100: 3 stacks = 3 |
| **HBM3e** | 1,024 bit | 8.0 GT/s | **1,024 GB/s** | H200: 3 stacks = 3 |
| **HBM4** | 2,048 bit | 6.4 GT/s | **1,638 GB/s** | ? |
| **HBM4** 理论 | 2,048 bit | 10.0 GT/s | **2,560 GB/s** | ? |

#### 计算示例

```
H100 HBM3 (80GB):
  3 stacks × 1,024 bit/stack × 6.4 GT/s ÷ 8
  = 3 × 128 GB/s
  = 3,072 GB/s ≈ 3 TB/s (约3,000 GB/s = 2.79 TiB/s)

H200 HBM3e (141GB):
  3 stacks × 1,024 bit/stack × 8.0 GT/s ÷ 8
  = 3 × 128 GB/s × 1.25 (速率提升)
  ≈ 3,072 GB/s × 1.25
  ≈ 3.84 TB/s ≈ 4 TB/s（约3,840 GB/s）
```

### 7.3 DDR vs HBM 带宽对比

| 对比项 | DDR5 双通道 | HBM3 (H100) 3-stack | 倍数 |
|:-------|:-----------:|:-------------------:|:----:|
| 典型带宽 | ~90 GB/s | ~3,000 GB/s | **~33x** |
| 每 bit 成本 | ~$0.01/GB | ~$5-8/GB | ~500x |
| 位宽 | 128 bit | 3,072 bit（3×1024） | **24x** |
| 延迟 | ~80-100ns | ~150-200ns | 略高 |

> HBM 通过**极端宽位宽**（3,072 bit vs DDR5 128 bit）来达到远超 DDR 的带宽，付出的代价是物理封装复杂度和成本。

### 7.4 内存带宽中的 1000 vs 1024

**内存/HBM 带宽标注是 1000 进制还是 1024 进制？**

说"GB/s"时，行业内**混用严重**。以下是实际情况：

| 场景 | 实际含义 | 差异 |
|:-----|:---------|:----:|
| **DDR/HBM 厂商标称** | 用 **1000 进制 (SI)** | 1 GB/s = 1,000,000,000 B/s |
| **`dmidecode` / OS 报告** | 用 **1000 进制 (SI)** | 同上 |
| **Linux `free` 命令** | 用 **1024 进制 (GiB)** | 显示 "GB" 实际是 "GiB" |
| **`nvidia-smi` 显存容量** | 用 **1024 进制 (GiB)** | 80 GB H100 = 80 GiB |
| **GPU 内存带宽计算** | 混用，多数手册**默认 1000 进制** | H100 标 "3.35 TB/s" 实际是 3.35 × 10^12 B/s |

> **经验法则**: 除非特别说明硬件计算中的"GB"都按 **1000 进制**（SI 十进制）处理。网络带宽则一定是 1000 进制。

## 八、超节点场景综合带宽计算框架

### 8.1 统一计算模板

对超节点中任意部件的带宽计算，使用以下**五步模板**：

```
步骤 ①: 找到源头速率 + 单位     (如 "400G"、"32 GT/s"、"6.4 GT/s")
步骤 ②: 确定该速率的定义层      (物理/MAC/有效负载)
步骤 ③: 确定单向还是双向        (网络=单向满双工，NVLink=双向总和)
步骤 ④: 确定是否有协议包头开销  (Ethernet/DDR/NVLink 各有不同)
步骤 ⑤: 统一转换到目标单位      (GB/s 或 GiB/s，注明进制)
```

### 8.2 常见超节点连线速查（统一到 GB/s 单向）

| 互连链路 | 标称值 | 实际含义 | 单向有效 (GB/s) | 双向有效 (GB/s) |
|:---------|:------|:---------|:---------------:|:---------------:|
| **PCIe 5.0 ×16** | "64 GB/s" | 单向，128b/130b已含 | **63.0** | 126.0 |
| **PCIe 6.0 ×16** | "~128 GB/s" | 单向，128b/130b+FLIT已含 | **~125.7** | ~251.4 |
| **400GbE (RoCE, 9K MTU)** | "400 Gbps" | MAC/PLS单向满双工 | **~49.6** | ~99.2 |
| **800GbE (RoCE, 9K MTU)** | "800 Gbps" | MAC/PLS单向满双工 | **~99.2** | ~198.4 |
| **NDR IB (4x) 400G** | "400 Gbps" | 物理层单向（编码已含） | **~49.6** | ~99.2 |
| **NVLink 4.0 (H100)** | "900 GB/s" | **双向总和**，编码已含 | **450** | 900 |
| **NVLink 6.0 (Vera)** | "3.6 TB/s" | **双向总和** | **1,800** | **3,600** |

### 8.3 超节点全链路带宽图谱 (以 Vera Rubin NVL72 为例)

```
GPU (Vera Rubin) 
│
├── HBM4 (G1): ~3.6 TB/s 单向  ← GPU读HBM带宽
│
├── NVLink 6.0 → NVSwitch
│   ├── 每 GPU: 1,800 GB/s 单向 → NVSwitch
│   ├── 每 GPU: 3,600 GB/s 双向 ← NVIDIA 宣称的 "3.6 TB/s"
│   └── 72 GPU 全互联: ~259 TB/s 总双向 bisection
│
├── PCIe Gen6 ×16 → CX9 SuperNIC
│   ├── 每 GPU PCIe: ~125 GB/s 单向
│   ├── CX9 (2×800G): ~200 GB/s 单向以太网出口
│   └── ⚠️ PCIe 125 < 网卡 200 → PCIe为瓶颈(cap ~63%)
│
├── BF4 DPU → CMX 存储 (G3.5)
│   ├── PCIe Gen6 ×16: ~125 GB/s 单向
│   ├── BF4 存储口: 待 NVIDIA 披露
│   └── CMX 方案: KV Cache SSD 池
│
└── Spectrum-6 (CPO 102.4 Tb/s)
    └── 全 POD 网络吞吐
```

### 8.4 从 GPU 到存储网的完整带宽链

这是一个最复杂的场景——GPU 数据如何到达 CMX 存储：

```
GPU HBM 
  → (NVLink) → NVSwitch → (NVLink) → 另一GPU或主CPU
  → (PCIe Gen6 ×16, ~125 GB/s) → BF4 DPU
    → BF4 内部 RDMA 引擎 → NVMe-oF over RDMA
      → (RoCEv2 over 800GbE) → Spectrum-X 交换机
        → (RoCEv2) → CMX Storage Tray (BF4 + NVMe SSD)

每段带宽漏斗分析:
  ① GPU HBM 读:      ~3.6 TB/s 单向（HBM范围，稍后用于KV Cache）
  ② GPU → Host PCIe: ~125 GB/s 单向（PCIe Gen6 ×16）
  ③ BF4 → 网络:      800 Gbps ≈ 100 GB/s 单向（2口各800G时需确认PCIe瓶颈）
  ④ 网络 → 存储:     存储侧带宽取决于 oversubscription 比例
```

> **结论**: GPU 内 HBM 带宽（~TB/s 级）到存储网络出口带宽（~100 GB/s 级）之间有 **~30x 的差距**。这就是为什么 CMX 需要"Prestaging"机制——在 GPU 需要前就把 KV Cache 提前拉到靠近 GPU 的内存层级。

### 8.5 常见超节点配置带宽速查

| 配置场景 | 计算公式 | 总带宽 |
|:---------|:---------|:-------|
| **512 节点 × 单口 400G 存储网** | 512 × 400 Gbps 单向 | **204.8 Tbps 单向** |
| **512 节点 × 双口 400G 存储网** | 512 × 800 Gbps 单向 | **409.6 Tbps 单向** |
| **64 GPU NVLink all-to-all** | 64 × 450 GB/s × 2 / 2 | ~14.4 TB/s (全双工 bisection) |
| **NVL72 (72×NVLink 3.6TB/s 双向)** | 72×1,800 GB/s 单向 | 129.6 TB/s 单向总出口 |
| **NVL72 到存储网 (假设2×800G)** | 36×200 GB/s 单向 (每托盘2GPU) | ~7.2 TB/s 单向（依赖拓扑） |

### 8.6 ⚠️ 存储层级范围与带宽规划的关系

> **核心洞察**: CMX 存储是**超节点级共享**，不是机架级独立。这对带宽规划有直接影响。

| 维度 | 机架级存储 (错误假设) | 超节点级共享 (正确) | 对带宽的影响 |
|:-----|:---------------------|:-------------------|:-------------|
| **KV共享范围** | 仅限本机架72 GPU | 全超节点512+ GPU均可访问 | 带宽必须覆盖全超节点 |
| **带宽复用比** | 每个机架独立计算 | 全局池化，峰值平摊 | 可以接受更低的重叠比 |
| **跨机架流量** | 不需要 | 常态（序列可能跨机架调度） | 交换机不能过度 oversubscribe |
| **Grove优化** | 大部分命中本机架 | 尽力优化 locality，不保证 | 必须为最坏情况预留余量 |
| **规划单位** | 以单个 NVL72 机架为单位 | 以整个超节点为单位 | 总带宽 = Σ(所有存储网口) |

**带宽计算时的修正**:

```
如果误以为 CMX 是"机架级":
  每机架 72 GPU × 各自的 CMX Tray
  → 每个 CMX Tray 只服务 72 GPU
  → 带宽规划独立，但闲置率可能高

正确理解: 超节点级共享
  全超节点 512/576/1152 GPU 共享 CMX 池
  → 带宽需求按"任意 GPU → 任意 CMX Tray"的最坏路径算
  → 但存储节点可以水平扩展，总带宽线性增长
  → 更优的利用率（全局负载均衡）
```

## 十、总结: 超节点带宽计算的实操建议

### 单次评估检查清单

每次做超节点带宽计算时，按此清单逐步确认：

```
□ ① 明确标注值在哪个层定义 (物理/MAC/有效负载)?
□ ② 明确是单向还是双向?
□ ③ 编码开销是否已包含在标注值中?
□ ④ FEC/ECC 开销是否需要考虑?
□ ⑤ 协议包头开销占多大（包大小决定）?
□ ⑥ 使用 1000 还是 1024 进制?
□ ⑦ 所有值统一到同一单位后再比较!
□ ⑧ 对整个链路的"最窄段"做瓶颈标注
□ ⑨ 考虑真实场景的 over-subscription 比
□ ⑩ 区分"峰值带宽"和"有效/可用带宽"
```

### 常见场景速查

| 场景 | 推荐做法 |
|:-----|:---------|
| 比较 GPU NVLink vs PCIe | 先统一到"单向 GB/s"再比 |
| 评估网卡能否跑满 | 比较网卡 Gbps/8 与 PCIe 单向 GB/s |
| 评估存储网络总吞吐 | 用节点数 × 每节点端口速率 × 有效比（~98%） |
| 评估 PCIe 是否能饱和某种互连 | 总是查 GT/s × 编码效率这个关键步骤 |
| 写技术文档 | 每次出现带宽数字就加注：单向/双向，1000/1024 |
| 交叉验证 | 用 2 个独立来源（官方规格 + 第三方测试）验证每个数字 |
