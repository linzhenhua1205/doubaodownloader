## 第1章：导论——DMA 作为"数据搬运"的底层范式

### 1.1 为什么要 DMA？

计算机系统中，数据需要在不同组件间移动：

```
         ┌──────────┐
         │   CPU    │ ←── 最擅长"计算"，不应浪费时间"搬数据"
         └────┬─────┘
              │ 系统总线 (PCIe/Memory Bus)
    ┌─────────┼──────────┐
    │         │          │
┌───▼───┐ ┌──▼──┐ ┌───▼───┐
│ Memory│ │ GPU │ │  NIC  │  ←── 各设备都需要"搬数据"
└───────┘ └─────┘ └───────┘
```

没有 DMA 的世界——**PIO (Programmed I/O)**：

```
CPU 执行:  LOAD  R1, [NIC_DATA_REG]   // 从 NIC 寄存器读 4字节
           STORE [MEM_ADDR], R1        // 写入内存
           LOAD  R1, [NIC_STATUS_REG]  // 检查状态
           ...                          // 重复数十万次
```

- CPU 被"搬运"占满，无法做计算
- 每 4 字节需要多条指令 → 吞吐极低
- 对于高速设备（25/100/400Gbps 网卡），PIO 根本无法跟上

DMA 的核心思想：**让设备自己搬数据，CPU 只下命令和收完成通知**。

### 1.2 DMA 的本质定义

> **DMA (Direct Memory Access)**: 允许 I/O 设备直接读写系统内存而无需 CPU 介入的硬件机制。

三个关键特征：
1. **CPU 仅参与设置和完成确认** — 中间的每一次数据传输都不经 CPU
2. **Bus Mastering** — 设备作为总线主控主动发起数据传输（而非 CPU 作为代理）
3. **内存驻留描述符** — 传输参数（源/目的地址、长度）存放在内存中，设备自行读取

### 1.3 从 DMA 到 RDMA——一脉相承

```
   DMA (节点内)                 RDMA (节点间)
   ───────────                 ───────────
   设备读/写本地内存            设备读/写远端内存
   CPU 仅参与设置               CPU 仅参与设置
   Bus Mastering (PCIe)         总线主控 + 网络协议
   本地一致性 (CPU→Device)      远端一致性 + 传输可靠性
```

RDMA 不是独立于 DMA 的技术——它是 DMA "远程化" 的产物，继承了 DMA 的 CPU Offload、Kernel Bypass、Zero Copy 三大特性，并额外增加了传输层可靠性和远端地址保护。

## 第2章：DMA Engine 微架构——谁在搬运数据

### 2.1 DMA Engine 的硬件组成

一个典型的 DMA Engine 包含以下核心单元：

```
┌──────────────────────────────────────────────────────┐
│                  DMA Controller (Engine)               │
│                                                        │
│  ┌─────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ Descriptor  │  │ Address      │  │ Transfer     │  │
│  │ Fetch Unit  │  │ Translation  │  │ Execution    │  │
│  │             │  │ (IOMMU)      │  │ Engine       │  │
│  └──────┬──────┘  └──────┬───────┘  └──────┬───────┘  │
│         │                │                  │          │
│  ┌──────▼────────────────▼──────────────────▼───────┐  │
│  │              Internal Queues / FIFOs              │  │
│  └──────────────────────┬───────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────▼───────────────────────────┐  │
│  │           PCIe Transaction Layer Interface        │  │
│  │     (生成 Memory Read/Write TLPs + Completions)   │  │
│  └───────────────────────────────────────────────────┘  │
│                                                        │
│  ┌─────────────┐  ┌──────────────┐                     │
│  │ Register    │  │ Interrupt    │                     │
│  │ File        │  │ Controller   │                     │
│  │ (MMIO映射)  │  │ (MSI-X)     │                     │
│  └─────────────┘  └──────────────┘                     │
└──────────────────────────────────────────────────────────┘
```

| 功能单元 | 职责 | 实现方式 |
|:---------|:-----|:---------|
| **Descriptor Fetch Unit** | 从系统内存读取 DMA 描述符 | DMA 引擎自主发起 PCIe Memory Read TLP |
| **Address Translation Unit** | 将设备虚拟地址(DVA)转换为物理地址 | 与 IOMMU 交互（ATC/PASID） |
| **Transfer Execution Engine** | 执行实际的数据搬运（读源→写目的） | 生成 Read TLP → 接收 Completion → 生成 Write TLP |
| **Internal Queues** | 缓冲传输中的数据 | 片上 SRAM（典型 4-64KB） |
| **PCIe TLP Interface** | 封装/解析 PCIe 事务层包 | PCIe 标准 TLP 格式 |
| **Register File** | 软件可编程的控制/状态寄存器 | MMIO 映射，被 CPU 访问 |
| **Interrupt Controller** | 传输完成/错误通知 | MSI-X 向量中断，无需 INTx 引脚 |

### 2.2 DMA 描述符的微架构设计

DMA 描述符是**软件与硬件之间的传输合约**——描述一次搬运的全部参数。

#### 2.2.1 典型描述符格式（以 Intel 82801 ICH 为例）

```
┌─────────┬─────────┬─────────┬─────────┐
│ Byte 0  │ Byte 1  │ Byte 2  │ Byte 3  │
├─────────┼─────────┼─────────┼─────────┤
│     Source Address (低32位)         │ ← 数据源物理地址
├─────────┼─────────┼─────────┼─────────┤
│     Source Address (高32位)         │
├─────────┼─────────┼─────────┼─────────┤
│     Destination Address (低32位)    │ ← 数据目的物理地址
├─────────┼─────────┼─────────┼─────────┤
│     Destination Address (高32位)    │
├─────────┼─────────┼─────────┼─────────┤
│     Transfer Length                  │ ← 传输字节数 (典型 Max=4MB)
├─────────┼─────────┼─────────┼─────────┤
│ Control │ Status  │ Next Desc (低)  │ ← 链式指针
├─────────┼─────────┼─────────┼─────────┤
│    Next Descriptor (高32位)         │
└─────────┴─────────┴─────────┴─────────┘
```

| 字段 | 位宽 | 含义 |
|:-----|:----:|:-----|
| Source Address | 64b | 数据源物理地址（内存或设备 FIFO 地址） |
| Destination Address | 64b | 数据目的物理地址 |
| Transfer Length | 32b | 传输字节数（典型 1B - 4MB） |
| Control | 16b | 传输方向、中断使能、链结束等控制位 |
| Status | 16b | 传输完成状态、错误码 |
| Next Descriptor | 64b | 链式描述符指针（NULL=链尾） |

#### 2.2.2 Scatter-Gather 列表（SG List）

实际场景中，数据在内存中通常不连续（跨不同物理页）。Scatter-Gather 通过描述符链解决：

```
┌─── 描述符链 ────────────────────────────────────────┐
│                                                      │
│  SG[0]: Src=0x1000, Dst=0xA000, Len=4096 ──────────→ │ 物理页 0xA000-0xAFFF
│        │ next = &SG[1]                               │
│  SG[1]: Src=0x2000, Dst=0xB000, Len=4096 ──────────→ │ 物理页 0xB000-0xBFFF
│        │ next = &SG[2]                               │
│  SG[2]: Src=0x3000, Dst=0xC000, Len=2048 ──────────→ │ 物理页 0xC000-0xC7FF
│        │ next = NULL (链尾)                           │
└──────────────────────────────────────────────────────┘
   ↑                        ↑
 应用程序虚拟地址         物理地址分散
 (连续)                   (不连续)
```

**SG 的核心价值**: 让 DMA 引擎直接处理分页内存，避免"先拷贝到连续缓冲区"这个额外开销。

### 2.3 DMA Engine 的实现变体

| 实现方案 | 典型产品 | 架构特点 | 适用场景 |
|:---------|:---------|:---------|:---------|
| **CPU 内置 DMA** | x86 I/OAT (Intel QuickData) | 集成在 CPU 内部，作为不可见单元 | 内存拷贝加速（memcpy offload） |
| **芯片组/桥片 DMA** | 传统南桥 DMA 控制器 | 独立的 DMA 控制器，多通道 | 磁盘、低速设备 |
| **外设内嵌 DMA** | NVMe Controller、GPU DMA Engine | 设备内部集成 DMA，专为自己服务 | 设备自身的数据传输 |
| **智能 NIC DMA** | Mellanox ConnectX、BlueField | 多重描述符队列 + RDMA 支持 | 网络、存储、GPU 通信 |

**关键趋势**: 现代高性能设备（NVMe SSD、GPU、智能网卡）均**内置自己的 DMA Engine**，而不是依赖中央 DMA 控制器。这是因为：
1. 设备最了解自己的数据格式和传输时机
2. 避免中央 DMA 控制器的瓶颈
3. 支持更复杂的传输语义（如 RDMA Read/Write）

## 第3章：DMA 操作流程——描述符链与 Bus Mastering

### 3.1 一次典型 DMA 传输的完整流程

以 NIC 接收数据包后写入内存为例：

```
时间轴 →
                     CPU                          DMA Engine (NIC)
                     │                                │
  1. 驱动程序准备     │ alloc rx_buffer[]              │
     接收缓冲区       │ fill SG descriptor[]           │
                     │ write DMA_DESC_ADDR → 寄存器    │
                     │ write DMA_START → 寄存器        │
                     │                                │
                     │    [CPU 返回做其他事]            │
                     │                                │
  2. 描述符获取       │                                │ ← DMA Engine 通过 PCIe
  (Descriptor Fetch) │                                │    Memory Read 读取描述符
                     │                                │
  3. 数据搬移         │                                │ ← 网卡接收数据包
                     │                                │ ← DMA Engine 生成 PCIe
                     │                                │    Memory Write TLP
                     │     [内存被 DMA 写入]           │    将数据写入系统内存
                     │                                │
  4. 完成通知         │ ← MSI-X 中断                    │ → 触发 MSI-X 中断
                     │    或 Polling 检查完成标志       │    或更新完成队列
                     │                                │
  5. CPU 处理         │ 处理收到的数据包                 │
                     │ re-fill descriptor (归还)       │
                     │                                │
```

**关键点**: 从步骤 2 开始到步骤 4 之前，CPU 完全不需要参与——这正是 DMA 的价值所在。

### 3.2 Bus Mastering：设备如何成为总线主控

传统的 I/O 操作中，CPU 是唯一的总线主控：

```
PIO 模型:    CPU ──(Master)──→ 总线 ──(Slave)──→ 设备
                 CPU 读写设备的 MMIO 寄存器
```

DMA Bus Mastering 模型：

```
DMA 模型:    CPU ──(Master)──→ 总线 ──(Master)──→ 设备
                 └─ CPU 初始化后，设备自行成为总线主控
                          │
                    ┌─────┴─────┐
                    │  PCIe Bus │
                    └─────┬─────┘
                          │
              ┌───────────┴───────────┐
              │                       │
        Memory (Slave)           NIC/DMA (Master)
```

**PCIe 层面**：设备作为 Requester 发起 Memory Read/Write TLP，目标设备（内存控制器）作为 Completer 返回数据。Requester 和 Completer 的概念取代了传统总线的主/从区分。

### 3.3 DMA 完成通知机制

| 通知方式 | 延迟 | CPU 开销 | 适用场景 |
|:---------|:----:|:--------:|:---------|
| **MSI-X 中断** | ~1-5μs | 高（中断处理开销） | 延迟敏感、频率低 |
| **中断合并** | ~5-50μs | 中 | 吞吐优先场景 |
| **轮询（Polling）** | < 1μs | 极高（占用 CPU） | 高频小包（如 RDMA） |
| **事件索引** | ~1-10μs | 中 | 混合模式 |

**RDMA 的特殊性**: RDMA 操作通常使用**轮询完成队列（Poll CQ）**，因为：
1. 延迟敏感（GPU 同步需 μs 级）
2. 操作高频（每训练步数千次 RDMA 操作）
3. 中断引入的延迟抖动不可接受

## 第4章：DMA 的三种形态——PIO / DMA / RDMA 的本质差异

### 4.1 三者的核心区别

| 维度 | PIO (Programmed I/O) | DMA (Direct Memory Access) | RDMA (Remote DMA) |
|:-----|:--------------------|:---------------------------|:-------------------|
| **搬运主体** | CPU（每字节都经 CPU 寄存器和指令） | DMA Engine（设备自主搬运） | RDMA 网卡（远端的 DMA Engine） |
| **CPU 参与方式** | 每条数据都执行 LOAD/STORE 指令 | 仅设置描述符 + 收完成通知 | 仅 Post Send/Recv Work Request |
| **地址空间** | CPU 虚拟地址 → 物理地址 | 物理地址 / 设备虚拟地址(DVA) | 远端虚拟地址（受保护域认证） |
| **数据传输** | PCIe Memory Read/Write TLP（CPU 发起） | PCIe Memory Read/Write TLP（设备发起） | 网络包（RNIC 封装和解封） |
| **Kernel 参与** | 每次操作都可能（通过系统调用访问 MMIO） | 设置阶段经过 kernel | **完全 Kernel Bypass**（用户态 Verbs 直通网卡） |
| **Zero Copy** | ❌ 数据经 CPU 寄存器中转 | ✅ 设备直接在内存间搬运 | ✅ 直接从应用缓冲区到远端应用缓冲区 |
| **延迟典型值** | ~0.1-1μs（但吞吐极低） | ~0.1-10μs | ~1-3μs（同机房） |

### 4.2 DMA 与 RDMA 的"额外代价"

RDMA 在 DMA 基础上增加了三层机制：

```
DMA:   设备 ├── PCIe TLP ──→ 本地内存
            (简单的读/写请求)

RDMA:  设备 ├── PCIe TLP ──→ 本地内存 (软件层: Verbs API)
            ├── 封装网络包头 (运输层: BTH + 可靠性)
            ├── 跨越物理链路 (网络层: 路由 + 拥塞控制)
            ├── 远端 PCIe TLP ──→ 远端内存
            └── 远端确认 (ACK/NACK)
```

增加的代价：
1. **协议封装开销** — 每个 RDMA 包需添加 BTH/GRH/ICRC 等头部（~60-80 字节）
2. **传输层可靠性** — PSN 跟踪、超时重传、ACK 处理
3. **远端地址保护** — 保护域（PD）/ 内存区域（MR）的权限检查
4. **流控机制** — 基于信用的流控（IB）或拥塞控制（RoCEv2 DCQCN）

### 5.2 NVMe SSD 的 DMA 机制

NVMe 是最典型的现代 DMA 应用：

**提交队列（Submission Queue）与完成队列（Completion Queue）**：

```
CPU 端                                    NVMe SSD 端
──────────                                ────────────
                               ┌────────┐
                               │ Admin  │
                               │ SQ/CQ  │ ← 管理命令
                               ├────────┤
                               │ IO SQs │
应用程序 ──(syscall)──→ 驱动 ───│ (1-64K)│ ← 读/写命令描述符
                               ├────────┤
                               │ IO CQs │
                               │        │ ← 完成状态
                               └────────┘
```

**DMA 操作流程**（以 NVMe 读为例）：

```
1. 驱动在内存中构建 Submission Queue Entry (SQE)
   - 命令类型: READ
   - 起始 LBA: 0x1000
   - 数据长度: 64KB
   - PRP (Physical Region Page) 列表: [0xA000, 0xB000, ...]

2. 驱动写 Doorbell 寄存器 (MMIO): "SQ 尾指针更新"

3. NVMe 控制器 (DMA Engine):
   a. PCIe Memory Read → 读取 SQ Entry
   b. 解析命令（哪个 LBA, 数据放到哪里）
   c. 从 NAND 读取数据到内部 SRAM
   d. PCIe Memory Write → 将数据写入 PRP 指定的内存地址
   e. 构建 Completion Queue Entry (CQE) → PCIe Memory Write
   f. 触发 MSI-X 中断

4. 驱动收到中断 → 处理 CQE → 通知应用程序数据就绪
```

**关键优化**: NVMe 使用 PRP 列表（即 Scatter-Gather 的 SSD 术语）直接描述数据在内存中的分散位置，无需额外拷贝。这是 NVMe 比 SATA/SAS 快 10-100× 的根本原因之一。

### 5.3 NIC 的 DMA 机制

NIC 的 DMA 是最复杂的节点内 DMA 场景，因为它需要同时处理多个流和协议：

**接收侧 DMA 流程**：

```
网络链路 ──→ NIC PHY → MAC → 包缓冲 SRAM
                                  │
                            ┌─────▼──────┐
                            │ RX DMA     │
                            │ Engine     │
                            └─────┬──────┘
                                  │
             ┌────────────────────┼────────────────────┐
             │                    │                    │
         ┌───▼───┐          ┌───▼───┐          ┌───▼───┐
         │  Ring │          │  Ring │          │  Ring │
         │ 0     │          │ 1     │   ...    │ N-1   │
         │ (Core0│          │ (Core1)│         │       │
         └───┬───┘          └───┬───┘          └───┬───┘
             │                    │                    │
         ┌───▼────────────────────▼────────────────────▼───┐
         │                RSS Hash                         │
         │   (Toeplitz 哈希: 5-tuple → 转发到哪个 Ring)     │
         └─────────────────────────────────────────────────┘
```

**RSS（Receive Side Scaling）** 使用 Toeplitz 哈希将数据包均匀分布到多个 DMA Ring，每个 Ring 绑定一个 CPU 核心：

```c
// RSS 哈希计算 (5-tuple)
hash = ToeplitzHash(src_ip, dst_ip, src_port, dst_port, protocol)
ring_index = hash % num_rings
```

**DMA 描述符环（Ring Buffer）** 是 NIC DMA 的核心数据结构：

```
               尾指针 (DDD, NIC 写入)       头指针 (驱动读取)
                   │                            │
    ┌────┬────┬────┬────┬────┬────┬────┬────┐
    │ B0 │ B1 │ B2 │ B3 │ B4 │ B5 │ B6 │ B7 │
    └────┴────┴────┴────┴────┴────┴────┴────┘
    ↑                             ↑
    已 DMA (待驱动处理)           空闲 (待 NIC 写入)
```

每个描述符指向一个内存缓冲区。NIC 收到数据包后，从 Ring 中取下一个空闲描述符，DMA 写入对应缓冲区，然后更新尾指针。

### 5.4 GPU 节点内 DMA (GPU Direct P2P)

GPU 之间的 P2P DMA 是最快速的同节点 GPU 通信方式：

```
                    ┌──────────────────┐
                    │   CPU Memory     │
                    └────────┬─────────┘
                             │ PCIe
                    ┌────────┴─────────┐
                    │   Root Complex   │
                    └──┬──────────┬────┘
                       │          │
          ┌────────────▼──┐  ┌───▼────────────┐
          │   GPU 0       │  │   GPU 1         │
          │   DMA Engine  │  │   DMA Engine    │
          │   (Bar1)      │  │   (Bar1)        │
          └───────────────┘  └────────────────┘
                │                    │
                └──────────┬─────────┘
                           │ PCIe Switch (可选 P2P 路径)
                           │
                     ┌─────▼─────┐
                     │  GPU 2    │
                     │  (NVLink) │
                     └───────────┘
```

**P2P 通信路径比较**：

| 路径 | 方式 | 带宽 | 延迟 | CPU 参与 |
|:-----|:-----|:----:|:----:|:--------:|
| **CPU Bounce** | GPU0→CPU→GPU1 | PCIe 带宽的一半 | ~10μs+ | ✅ 需要 |
| **PCIe P2P** | GPU0→PCIe Switch→GPU1 | PCIe 全带宽 | ~3-5μs | ❌ 不需要 |
| **NVLink P2P** | GPU0→NVLink→GPU1 | 7× PCIe 4.0+ | ~0.5-1μs | ❌ 不需要 |

**NVLink P2P 的 DMA 特殊性**：
- NVLink 有自己的 DMA Engine 和地址翻译单元
- 每个 GPU 可以直接读取另一 GPU 的 HBM 显存
- 硬件保证端到端数据一致性（通过 NVLink 一致性协议）
- 对应用层表现为 Load/Store 语义（与传统 DMA 的"显式搬移"不同）

### 5.5 FPGA / ASIC 加速器的 DMA

FPGA 作为加速器时，其 DMA 设计通常更加灵活：

```
FPGA DDR ──→ FPGA DMA Engine ──(PCIe)──→ Host Memory
                │
           ┌────┴────┐
           │ User    │
           │ Logic   │
           │ (自定义) │
           └─────────┘
```

**FPGA DMA 的典型挑战**：
1. **非对齐传输**：FPGA 的数据宽度可能与 PCIe 要求不匹配
2. **背压处理**：FPGA 内部逻辑产生数据的速率不稳定
3. **多通道**：需要同时处理多个数据流的 DMA

## 第6章：IOMMU / SMMU 与 DMA 重映射

### 6.1 为什么需要 IOMMU？

没有 IOMMU 的世界：

```
问题 1: 设备只能访问物理地址
        应用程序使用虚拟地址 → 物理页可能被换出/迁移
        DMA 写入已释放的物理页 → 安全漏洞

问题 2: DMA 写入错误的物理地址
        恶意/有缺陷的设备可读写任意物理内存
        无保护 → 直接破坏 OS Kernel

问题 3: 大页(>4KB)的支持
        设备 DMA 描述符通常限制最大传输单元
        跨页传输需要 SG 列表
```

### 6.2 IOMMU 的工作原理

IOMMU 位于设备与内存之间，对 DMA 请求进行地址翻译和权限检查：

```
                         IOMMU
                        ┌──────────┐
  设备(DMA Master) ───→ │ 翻译 +   │ ───→ 系统内存
                        │ 权限检查  │
                        └────┬─────┘
                             │
                    ┌────────▼────────┐
                    │   IOMMU Page    │
                    │   Tables        │
                    │ (设备专用页表)   │
                    └─────────────────┘
```

**地址翻译流程**：

```
设备发起的 DMA 请求:
  1. 设备 DMA Engine 生成: Source=0x1000 (设备虚拟地址 DVA)
  2. IOMMU 拦截 PCIe TLP
  3. IOMMU 查询设备页表: DVA 0x1000 → 物理地址 0x7F000000
  4. IOMMU 检查权限: 该设备是否有该区域的读写权限？
  5. 通过 → 转发 TLP 到目标（内存控制器）
     拒绝 → 返回 PCIe Completer Abort (CA) / Unsupported Request (UR)
```

### 6.3 ATC（Address Translation Cache）与性能

IOMMU 每次翻译的延迟约 **40-100ns**——这对高速 DMA 是不可接受的。ATC 解决了这个问题：

```
第一次 DMA (IOMMU 翻译)                  后续 DMA (ATC 缓存)
┌──────────┐    ┌──────────┐           ┌──────────┐    ┌──────────┐
│  Device  │    │  IOMMU   │           │  Device  │    │  IOMMU   │
│(ATC未命中)│    │          │           │(ATC命中) │    │          │
└────┬─────┘    └────┬─────┘           └────┬─────┘    └────┬─────┘
     │ PCIe TLP(翻译请求)  │                  │              │
     │────────────────→    │                  │  直接 DMA    │
     │                     │                  │────────────→│
     │  回 Completion      │                  │              │
     │←───────────────────│                  │              │
     │                     │                  │              │
     │ 缓存 DVA→PA 映射    │                  │              │
     │ 在 ATC 中          │                  │              │
     │                     │                  │              │
     │ 发起实际 DMA        │                  │              │
     │────────────────→    │                  │              │
```

| 配置 | 每次 DMA 延迟 | 吞吐影响 |
|:-----|:------------:|:--------:|
| **无 IOMMU** | 无翻译开销 | 最高（但无保护） |
| **IOMMU + 无 ATC** | +40-100ns | -5-20%（小包场景严重） |
| **IOMMU + ATC** | 命中时无额外开销 | 接近无 IOMMU 性能 |
| **IOMMU + ATC + PASID** | +开销（多进程隔离） | 小包 -3-10% |

### 6.4 PASID（Process Address Space ID）

PASID 让设备可以访问进程的**虚拟地址空间**，而不是物理地址空间：

```
传统 DMA:  设备 → IOMMU(物理地址) → 内存
PASID DMA: 设备 → IOMMU(进程虚拟地址 + PASID) → 内存
```

PASID 的核心意义：
- **应用可以直接将用户态缓冲区地址告诉设备**（无需 pin 物理页）
- **设备出入该进程的完整虚拟地址空间**（不是固定的物理内存区域）
- **多进程共享设备时，通过 PASID 区分不同的地址空间**

### 6.5 IOMMU 的代价：TLB 刷新与设备暂停

```
DMA 场景: 设备正在使用 ATC 缓存地址
         ↓
OS 解除映射某物理页 (unmap)
         ↓
IOMMU 必须让该设备的 ATC 失效
         ↓
发送 PCIe ATC Invalidate Request
         ↓
设备暂停 DMA → 刷新 ATC → 回复 Invalidate Completion
         ↓
设备恢复 DMA (需重新翻译)
```

这个 **ATC 刷新过程** 会增加额外延迟（~5-100μs），是 IOMMU 高性能场景的主要代价。

## 第7章：RDMA——DMA 向远端的扩展

### 7.1 RDMA 的核心理念

RDMA 不是"远程版 DMA"这么简单——它在 DMA 的基础上引入了三个关键变化：

```
DMA:     [设备] ──(PCIe)──→ [本地内存]
         数据搬运被卸载到设备内部引擎

RDMA:    [本地设备] ──(PCIe+网络)──→ [远端设备] ──(PCIe)──→ [远端内存]
         数据搬运被卸载到两端网卡 + 传输层
```

**变化 1：引入了网络传输层**
- DMA 在 PCIe 域中工作，RDMA 需要跨越物理分离的服务器
- 需要封装/解封网络协议头
- 需要处理丢包、乱序、重传

**变化 2：远端地址保护**
- DMA 在本机受 IOMMU 保护
- RDMA 需要网卡自行验证远端地址合法性（通过 Memory Registration）

**变化 3：Kernel Bypass 成为必要条件**
- 节点内 DMA 还可以走 Kernel（如 NVMe 驱动）
- RDMA 如经 Kernel 则延迟从 1-3μs 膨胀到 10-40μs（系统调用开销）
- 因此必须用户态直接操作网卡硬件

### 7.2 RDMA 网卡（RNIC/HCA）的内部架构

```
┌─────────────────────────────────────────────────────────────┐
│                    RNIC / HCA (RDMA 网卡)                     │
│                                                              │
│  ┌──────────────────────────────────────────────────┐       │
│  │              Protocol Processing Engine           │       │
│  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐         │       │
│  │  │ Verbs│  │ Trans│  │ Net  │  │ Link │         │       │
│  │  │ Proc │  │ port │  │ work │  │      │         │       │
│  │  └──┬───┘  └──┬───┘  └──┬───┘  └──┬───┘         │       │
│  └──────┼─────────┼─────────┼─────────┼────────────┘       │
│         │         │         │         │                     │
│  ┌──────▼─────────▼─────────▼─────────▼────────────┐       │
│  │              PCIe DMA Engine                      │       │
│  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌─────────┐     │       │
│  │  │Desc  │  │Addr  │  │Data  │  │ Interrupt│     │       │
│  │  │Fetch │  │Trans │  │Queue │  │ (MSI-X) │     │       │
│  │  └──────┘  └──────┘  └──────┘  └─────────┘     │       │
│  └──────────────────────┬──────────────────────────┘       │
│                         │                                   │
│  ┌──────────────────────▼──────────────────────────┐       │
│  │           Embedded Processor (可选)               │       │
│  │  (BlueField DPU: ARM Cortex + 加速器)            │       │
│  └──────┬──────────────────────────────────────────┘       │
│         │                                                   │
│  ┌──────▼──────────────────────────────────────────┐       │
│  │           Internal Memory (SRAM + Cache)         │       │
│  │  (QP Context、CQ、PKeys、地址翻译缓存)           │       │
│  └─────────────────────────────────────────────────┘       │
│                                                              │
│  ┌────────────┐  ┌────────────┐  ┌────────────┐            │
│  │  Port 0    │  │  Port 1    │  │ 管理接口    │            │
│  │  400GbE/IB│  │  400GbE/IB│  │  (NC-SI)   │            │
│  └────────────┘  └────────────┘  └────────────┘            │
└─────────────────────────────────────────────────────────────┘
```

**关键内部单元**：

| 单元 | 功能 | 实现方式 |
|:-----|:-----|:---------|
| **Verbs Processor** | 解析 Verbs API 命令（Post Send/Recv） | 硬连线状态机或嵌入式处理器 |
| **Transport Engine** | 处理 IB 传输层（RC/UC/UD/XRC） | 硬件加速，生成/解析 BTH、PSN、ACK |
| **Network Processor** | 网络层（IB GRH 或 UDP/IP/ECN） | 硬件 offload |
| **DMA Engine** | PCIe 数据搬运 | 多通道 DMA，支持 SG |
| **QP Context Memory** | 存储队列对状态 | 片上 SRAM（典型 64K QP） |
| **ATC / TLB** | 地址翻译缓存 | 与 IOMMU 交互 |

### 7.3 RDMA 的关键数据结构：QP（Queue Pair）

QP 是 RDMA 通信的核心抽象——每个 QP 包含一对队列：Send Queue (SQ) 和 Receive Queue (RQ)：

```
应用程序
    │
    │ Post Work Request (WR)
    ▼
┌─────────────────────────────────┐
│       QP (Queue Pair)           │
│                                 │
│  ┌─────────────────────────┐    │
│  │    Send Queue (SQ)      │    │ ← 发送 WR: RDMA Write/Send/Read
│  │                         │    │
│  │ WR[0], WR[1], WR[2]... │    │
│  └─────────────────────────┘    │
│                                 │
│  ┌─────────────────────────┐    │
│  │    Receive Queue (RQ)   │    │ ← 接收 WR: 接收缓冲区
│  │                         │    │
│  │ WR[0], WR[1], WR[2]... │    │
│  └─────────────────────────┘    │
│                                 │
│  ┌─────────────────────────┐    │
│  │    Completion Queue     │    │ ← CQ: 完成通知
│  │    (CQ，可被多个QP共享)   │    │    (轮询或事件)
│  └─────────────────────────┘    │
└─────────────────────────────────┘
```

当应用调用 `ibv_post_send()` 时：
1. WR 进入 SQ 的硬件队列
2. RNIC 硬件自动取出 WR，解析操作类型和目标
3. 执行操作（DMA 读本地数据 → 封装网络包 → 发送）
4. 完成时在 CQ 中写入 Completion Entry
5. 应用程序轮询 `ibv_poll_cq()` 或等待事件

## 第8章：RDMA 传输语义与操作类型

### 8.1 四种基本 RDMA 操作

#### 8.1.1 RDMA Write

```
客户端 (Initiator)                    服务端 (Target)
┌─────────────────────┐           ┌─────────────────────┐
│   Application       │           │   Application       │
│   Buffer A          │           │   Buffer B          │
│   (数据源)          │           │   (数据目的)         │
└────────┬────────────┘           └────────▲────────────┘
         │ ① ibv_post_send(                │
         │   WR=RDMA_WRITE,                 │
         │   remote_addr=B)                 │
         ▼                                  │
┌─────────────────┐  RDMA Write Request    │
│ RNIC DMA Engine │ ──────────────────→    │
│ → 读取 Buffer A │   数据包(含 Payload)    │
│ → 封装网络包    │                        │
└─────────────────┘                        │
                                           │
                                   ┌───────┴────────┐
                                   │ RNIC DMA Engine │
                                   │ → 解析包        │
                                   │ → DMA Write     │
                                   │   Buffer B      │
                                   └────────────────┘
```

- **服务端 CPU 零参与**: 数据直接写入远端 Buffer B
- **单向操作**: 仅数据写入，不需要远端侧的 CPU 接收
- **适用于**: 数据发送（如梯度传输）

#### 8.1.2 RDMA Read

```
客户端 (Initiator)                    服务端 (Target)
┌─────────────────────┐           ┌─────────────────────┐
│   Application       │           │   Application       │
│   Buffer A          │           │   Buffer B          │
│   (数据目的)         │           │   (数据源)          │
└────────▲────────────┘           └────────┬────────────┘
         │ ① ibv_post_send(                │
         │   WR=RDMA_READ,                  │
         │   remote_addr=B)                 │
         │                                  │
         │ ② RNIC 发送 Read Request         │
         │ ────────────────────────────────→│
         │                                  │ ③ 远端 RNIC 解析
         │                                  │ → DMA Read Buffer B
         │ ④ RDMA Read Response (含数据)    │ → 封装
         │ ←────────────────────────────────│
         │                                  │
         ▼                                  │
┌─────────────────┐                        │
│ RNIC DMA Engine │ → 将数据 DMA 写入       │
│ → 写入 Buffer A │   Buffer A              │
└─────────────────┘                        │
```

- **服务端 CPU 零参与**: 客户端主动"拉"数据
- **双向操作**: Request + Response 两阶段
- **延迟更高**: 需要 RTT 往返
- **适用于**: 数据获取（如参数拉取、checkpoint 恢复）

#### 8.1.3 Send / Recv（通道语义）

```
发送端                                  接收端
┌─────────────────┐           ┌─────────────────┐
│ ibv_post_send(  │           │ ibv_post_recv(  │
│   WR=SEND,      │           │   WR=RECV,      │
│   buffer=C)     │           │   buffer=D)     │
└────────┬────────┘           └────────┬────────┘
         │                             │
         ▼                             ▼
┌─────────────────┐  SEND Request    ┌─────────────────┐
│ RNIC DMA Engine │ ────────────────→│ RNIC DMA Engine │
│ → DMA Read C    │  数据包          │ → DMA Write D   │
│ → 封装发送      │                 │ → 生成 CQE       │
└─────────────────┘                  └─────────────────┘
```

- **两端 CPU 都参与**: 发送端 Post Send，接收端需提前 Post Recv
- **数据语义**: 类似传统的消息传递（send/receive 配对）
- **适用于**: 控制消息、动态大小数据

#### 8.1.4 Atomic（原子操作）

```
客户端发起:
  ibv_post_send(WR=ATOMIC_CMP_AND_SWP,
    remote_addr=X, compare=10, swap=20)

远端执行（在 RNIC 硬件中）:
  if (*X == 10) {    // Compare
    *X = 20;         // Swap (条件满足)
  }
  return old_value;  // 返回原值给客户端
```

- **硬件事务**: 在 RNIC 内部锁住内存地址，执行读-改-写
- **操作类型**: Fetch&Add、Compare&Swap
- **适用场景**: 分布式锁、计数器、同步原语
- **延迟**: 通常比 RDMA Read 高 0.5-2μs（原子操作开销）

### 8.2 操作的延迟对比

| 操作 | 延迟 (同机房 400Gbps) | 往返次数 | CPU 参与度 |
|:-----|:--------------------:|:--------:|:----------:|
| **Send** (小消息) | ~1-3μs | 1 (单向) | 两端 |
| **RDMA Write** | ~1-3μs | 1 (单向) | 仅发送端 |
| **RDMA Read** (4KB) | ~3-5μs | 2 (Req+Resp) | 仅发起端 |
| **Atomic** | ~3-8μs | 2 (Req+Resp) | 仅发起端 |
| **TCP Send** (小消息) | ~10-40μs | 1+N (ACK) | 两端 Kernel |

## 第9章：RDMA 运输层机制深度分析

### 9.1 四种传输服务

| 服务 | 可靠 | 连接 | 保序 | 消息大小限制 | 典型用途 |
|:-----|:----:|:----:|:----:|:-----------:|:---------|
| **RC** (Reliable Connection) | ✅ | ✅ | ✅ | 不受限（可分段） | GPU 通信 |
| **UC** (Unreliable Connection) | ❌ | ✅ | ❌ | 单包大小 | 容忍丢包场景 |
| **UD** (Unreliable Datagram) | ❌ | ❌ | ❌ | MTU | 多播、控制 |
| **XRC** (eXtended RC) | ✅ | 半连接 | ✅ | 不受限 | 多对多消息传递 |

### 9.2 RC 模式的可靠性机制

```
发送端                                          接收端
┌─────────────┐                              ┌─────────────┐
│ QP Context   │                              │ QP Context   │
│ PSN: 1234   │                              │ Eager PSN    │
│ Retry: 7    │                              │: 1234        │
│ RNR: 0      │                              │              │
└──────┬──────┘                              └──────┬──────┘
       │                                            │
       │ ┌─ 数据包: PSN=1234, OpCode=RDMA Write ──→│
       │ │                                          │ 接收 → 检查 PSN
       │ │                                          │ 预期 1234 ✓
       │ │                                          │ DMA 写入内存
       │ │                                          │ 生成 ACK
       │ │←── ACK: PSN=1234 ────────────────────│
       │ │                                          │
       │ │←── 下一包: PSN=1235 ──────────────────│
       │ │   (乱序到达)                              │ 预期 1235 ✓
       │ │   ACK: PSN=1235                           │
```

**RC 保证**:
1. **按序交付** — PSN 严格递增，乱序包被缓存/重排
2. **无丢包** — 每包收到后发送 ACK，超时未收到 ACK 则重传
3. **无重复** — PSN 去重机制
4. **流控** — 基于信用的流控（IB）或 ECN+DCQCN（RoCEv2）

### 9.3 基于信用的流控（InfiniBand 特有）

```
┌─── 发送端 ────┐                    ┌─── 接收端 ────┐
│ Total Credits │                    │ 缓冲区大小    │
│ : 16          │                    │ : 16 包       │
│ Used : 4      │                    │ 已收: 4       │
│ Avail: 12     │                    │ 可用: 12      │
└───────┬───────┘                    └───────┬───────┘
        │ ─── 数据包 ─────────────────────→ │
        │ ─── 数据包 ─────────────────────→ │ 已收 6 → 可用 10
        │ ─── 数据包 ─────────────────────→ │
        │                                    │
        │ ←── ACK + Credits=10 ─────────── │
        │                                    │
        │ (更新 Avail=10)                    │
```

- 接收端在 ACK 中返回可用信用额度
- 发送端在信用耗尽前必须停发
- **零丢包**: 接收端永远不会因为缓冲区满而丢包

### 9.4 RoCEv2 的拥塞控制（DCQCN）

RoCEv2 基于 UDP，没有 IB 的信用于是流量控制，因此需要额外的拥塞控制：

```
┌─── 发送端 ────┐               ┌─── 交换机 ────┐     ┌─── 接收端 ────┐
│ Rate limiter  │               │ ECN 标记      │     │ CNP 生成      │
│ α: 0.5        │               │ Threshold:    │     │               │
│ Current Rate: │               │ Kmin=50KB     │     │               │
│ 100Gbps       │               │ Kmax=200KB    │     │               │
└───────┬───────┘               └───────┬───────┘     └───────┬───────┘
        │ ─── 数据包 ─────────────────→ │ ──────────────────→ │
        │                              │ 队列深度 > Kmin      │
        │                              │ 设置 ECN 标记        │
        │                              │                      │
        │                              │  ←── ECN 标记包 ──   │
        │                              │                      │
        │ ←── CNP (拥塞通知包) ────────│                      │
        │                              │                      │
        │ 降低速率:                     │                     │
        │ Rate = Rate × (1 - α/2)      │                     │
        │ = 100 × 0.75 = 75Gbps        │                     │
```

**DCQCN 的挑战**：
- 速率恢复缓慢（AI 场景频繁吞吐抖动）
- ECN 标记延迟：交换机队列 → 接收端 → CNP → 发送端
- 多流公平性问题

## 第10章：服务器节点内 RDMA 应用场景

### 10.1 场景矩阵

| 场景 | 通信模式 | 延迟要求 | 带宽要求 | RDMA 操作 | 典型规模 |
|:-----|:---------|:--------:|:--------:|:---------:|:--------:|
| **NCCL AllReduce** | 集合通信 | <5μs | 400Gbps+ | RDMA Write + Read | 8 卡内/跨节点 |
| **NVMe-oF** | 存储访问 | <10μs | 100-400Gbps | RDMA Read/Write | 每节点 |
| **DPU 加速** | 控制面+数据面 | <10μs | 200Gbps+ | 混合 | 每节点 |
| **分布式锁** | 点对点 | <5μs | 低 | Atomic | 跨集群 |

### 10.2 NCCL 中的 RDMA 使用

NCCL 是 RDMA 在 AI 训练中最核心的应用。以 **Ring AllReduce** 为例：

```
阶段 1: Reduce-Scatter (RDMA Write)
GPU0 → GPU1 → GPU2 → GPU3 → ... → GPU7
每个 GPU: 对接收到的块做 Reduce (加和)，再传给下一个

阶段 2: AllGather (RDMA Write)
GPU0 ← GPU1 ← GPU2 ← ... ← GPU7
每个 GPU: 将从上一个收到的数据块拷贝一份
```

**NCCL 的 RDMA 优化**：
1. **多对 QP**: 每对 GPU 之间建立独立的 RC QP
2. **多流并行**: 将 NCCL 通道映射到不同的 QP，跨多个 RNIC 端口
3. **轮询模式**: 忙等 CQ 而不是中断，<1μs 完成检测
4. **注册内存**: 预先注册 GPU 显存为 MR（Memory Region）

### 10.3 NVMe-oF (NVMe over Fabrics)

NVMe-oF 是 RDMA 在存储场景的核心应用：

```
传统存储路径:
  应用 → 系统调用 → VFS → 文件系统 → 块层 → SCSI/SAS → 远端存储
         ↓                    ↓
    每次操作需 CPU         经 Kernel

NVMe-oF + RDMA 路径:
  应用 → NVMe 驱动 → RDMA Fabric → 远端 NVMe 控制器
                      ↓
             用户态直通 + Zero Copy
```

**NVMe-oF 的 RDMA 操作**：

```
主机 (Initiator)                       存储端 (Target/NVMe SSD)
┌────────────────┐                    ┌───────────────────────┐
│ NVMe 驱动程序   │                    │ NVMe 控制器            │
│ → 构建 NVMe命令 │                    │                      │
│ → 将命令封装为   │                    │                      │
│   RDMA Send     │                    │                      │
└───────┬────────┘                    └───────────┬───────────┘
        │ RDMA Send (NVMe命令+数据的SG描述)      │
        │ ──────────────────────────────────────→ │
        │                                         │ RDMA Read (从)
        │                                         │ 主机拉取数据
        │ ←── RDMA Read Response ─────────────── │
        │                                         │ DMA 到 NVMe
        │                                         │ 执行 FTL 映射
        │ ←── RDMA Send (完成状态) ────────────── │
```

**性能对比**：

| 方案 | 4K 随机读延迟 | IOPS | CPU 利用率 |
|:-----|:------------:|:----:|:----------:|
| SATA SSD (AHCI) | ~100μs | ~100K | 高 |
| NVMe (PCIe) | ~10-20μs | ~1M | 低 |
| NVMe-oF (RoCE) | ~20-40μs | ~800K | 极低（远端） |
| NVMe-oF (TCP) | ~50-200μs | ~300K | 中 |

**每种拓扑的 RDMA 特性**：

### 11.2 RDMA 网卡到 GPU 的拓扑亲和性

```
                  ┌──────────────────────────────┐
                  │      PCIe Root Complex       │
                  │                              │
                  │  ┌────┐ ┌────┐ ┌────┐ ┌────┐ │
                  │  │CPU0│ │CPU1│ │CPU2│ │CPU3│ │
                  │  └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘ │
                  │    │      │      │      │      │
                  └────┼──────┼──────┼──────┼──────┘
                       │      │      │      │
                  ┌────▼──┐┌─▼────┐┌─▼────┐┌─▼────┐
                  │PCIe SW││PCIe  ││PCIe  ││PCIe  │
                  │┌──────┤│Switch││Switch││Switch│
                  ││GPU0-3││  ┌───┤│  ┌───┤│  ┌───┤
                  ││NIC0  ││  │GPU││  │GPU││  │GPU│
                  │└──────┘│  │4-7││  │8-11││  │12-│
                  └────────┘  │NIC││  │NIC││  │15 │
                              │1  ││  │2  ││  │NIC│

                                   └──────┘  └───┘
```

**关键拓扑约束**:
1. GPU 与 NIC 在同一个 PCIe Switch 下 → 最优 P2P 路径
2. 跨 Socket 的 PCIe 路径 → 额外 UPI 跳数
3. NUMA 距离直接影响 RDMA 延迟

### 11.3 整机柜 RDMA 设计要点

| 设计维度 | 最佳实践 | 理由 |
|:---------|:---------|:-----|
| **NIC 到 GPU 映射** | 同一 PCIe Switch 域 | 避免跨 UPI，降低延迟 |
| **QoS (PFC/ETS)** | 无损 RoCE 需 PFC 配置 | 丢包导致 RC 重传，吞吐骤降 |
| **拥塞管理** | DCQCN / TIMELY / HPCC | ECN 阈值要根据拓扑调优 |
| **子网划分** | IB: 子网 ≤ 2000 节点 | Subnet Manager 管理规模有限 |
| **冗余** | 2 个独立 IB/RoCE 平面 | 训练网 + 存储网物理隔离 |

### 12.2 用户态 Verbs 的 Kernel Bypass 路径

**关键路径**：以 `ibv_post_send()` 为例

```
应用程序:
  ibv_post_send(qp, wr)
       │
       ▼
libibverbs (用户态库):
  ┌─────────────────────────────────────────────────────┐
  │ 1. 将 WR 格式化为硬件可读的命令格式                  │
  │ 2. 写 QP 的 Doorbell 寄存器 (MMIO 映射)              │
  │    → 用户态直接写 (无需系统调用)                     │
  │    → `*((volatile uint64_t*)doorbell) = value`      │
  └──────────────────────┬──────────────────────────────┘
                         │
                         ▼
  RNIC 硬件:
  ┌─────────────────────────────────────────────────────┐
  │ 1. 检测 Doorbell 写 → 知道 SQ 有新 WR             │
  │ 2. DMA Read → 从内存读取 WR 内容                    │
  │ 3. 解析 WR → 获取 data/lkey/remote addr             │
  │ 4. 检查 lkey → 确认本地内存权限                      │
  │ 5. 执行操作 → RDMA Write/Read/Send                  │
  │ 6. Completion → 写 CQ Entry + (可选) MSI-X 中断     │
  └─────────────────────────────────────────────────────┘
```

**无系统调用的秘密**: Doorbell 寄存器被映射到用户态地址空间（通过 `mmap`），应用程序直接写即可通知硬件。

### 12.3 Memory Registration (MR)

MR 是 RDMA 安全的核心——它告诉 RNIC "这块内存可以被远端访问"：

```
ibv_reg_mr(pd, buffer, size, IBV_ACCESS_REMOTE_WRITE)
         │
         ▼
内核 rdma_core:
  1. 锁定物理页 (mlock 相当于) → 防止被 swap
  2. 建立物理页列表
  3. 写入 RNIC 硬件页表
  4. 生成 lkey / rkey (本地/远端 key)
         │
         ▼
  返回: mr 句柄 (含 lkey/rkey)
```

**MR 的代价**：
| 操作 | 延迟 | 说明 |
|:-----|:----:|:------|
| MR 注册 (1GB 连续) | ~100μs-1ms | 逐页处理 + IOMMU 操作 |
| MR 注销 | ~50-500μs | 释放锁定页 |
| MR 缓存命中 | ~0 | 注册后复用 |

### 13.3 带宽性能关键因素

```
理论带宽 = 链路速率 × 有效载荷比例

  1. 消息大小 (小消息效率低)
  2. QP 数量 (多 QP 可提升带宽)
  3. NUMA 亲和性 (跨 NUMA 降低 ~10-15%)
  4. PCIe 链路宽度和代际
  5. 拥塞程度 (丢包率)
```

**消息大小 vs 带宽效率 (典型 RoCE 400Gbps)**：

| 消息大小 | 实测带宽 | 效率 | 说明 |
|:--------:|:--------:|:----:|:-----|
| 4B | ~10 Mbps | 0.0025% | 协议头开销主导 |
| 256B | ~5 Gbps | 1.25% | 头开销仍显著 |
| 4KB | ~200 Gbps | 50% | 典型 AI 梯度大小 |
| 64KB | ~380 Gbps | 95% | 接近满载 |
| 1MB+ | ~395 Gbps | ~99% | 饱和 |

### 13.4 延迟分解测试方法

**单跳延迟分解（ib_read_lat 变体）**：

```
硬件延迟 = T1 (远程回环) - T_L (线缆)
T1: 远程回环模式（数据走网卡环回，不过网络）
T_L: 线缆传输延迟 (光纤 5ns/m)
```

**端到端延迟分解**：

```
端到端延迟 = T_app + T_kernel + T_pcie + T_nic + T_net + T_remote

T_app  = 应用程序处理时间 (~0.1-0.5μs)
T_kernel = Kernel Bypass (RDMA: ~0; TCP: ~10-30μs)
T_pcie = PCIe 传输延迟 (~0.1-0.5μs)
T_nic  = 网卡内部处理 (~0.2-0.5μs)
T_net  = 网络传输 (线缆 ~5ns/m + 交换机 ~0.3-2μs)
T_remote = 远端 PCIe + 内存访问 (~0.2-0.5μs)
```

### 14.1 一次 RDMA Write 的全路径延迟分解

以 4KB RDMA Write 在 400Gbps RoCEv2 同机房（1 跳交换机）为例：

```
时间轴 (每格 ~0.5μs)    累计延迟    占比
─────────────────────  ─────────  ────
[应用程序] ibv_post_send   0.2μs   10%
   │ 写 Doorbell (MMIO)  ~50ns
   ▼
[RNIC] 检测 Doorbell      0.5μs    25%
   │ DMA Read WR (PCIe)   ~100ns
   │ 解析 WR → 获取 data buf
   │ 检查 lkey/permission ~50ns
   ▼
[RNIC] DMA Read data     ~1.0μs    50%
   │ PCIe Memory Read    ~150ns
   │ (4KB 数据, PCIe Gen5)
   │ 完成等待             ~300ns
   ▼
[RNIC] 封装网络包          ~1.5μs    75%
   │ 添加 BTH/GRH/UDP/IP 头
   │ CRC 计算
   ▼
[线缆] 物理传输            ~1.6μs    80%
   │ 10m 光纤 ~50ns
   │ SerDes 编码 ~50ns
   ▼
[交换机] 转发              ~1.9μs    95%
   │ MAC 查找 ~200ns
   │ 内部 Crossbar ~100ns
   ▼
[远端 RNIC]               ~2.0μs   100%
   │ 解封包
   │ 检查 PSN/lkey
   │ DMA Write 到远端内存
   │ 生成 ACK
   ▼
完成确认 (CQ 轮询)         ~2.0μs   100%
```

| 组件 | 延迟 (ns) | 占比 | 是否可变 |
|:-----|:--------:|:----:|:--------:|
| PCIe Gen5 传输 (DMA Read 4KB) | ~150 | 7.5% | 固定 |
| RNIC 内部处理（包解析+封装） | ~500 | 25% | 设计相关 |
| 线缆传输 (10m) | ~50 | 2.5% | 距离相关 |
| 交换机转发 | ~350 | 17.5% | 拓扑相关 |
| 远端 RNIC 处理 | ~450 | 22.5% | 设计相关 |
| 应用程序开销 | ~200 | 10% | 实现方式 |
| 完成轮询 | ~200 | 10% | 轮询 vs 中断 |
| 协议头处理 | ~100 | 5% | 协议相关 |

### 14.2 各代 PCIe 对 DMA 延迟的影响

| PCIe 代际 | 每 GT/s | x16 带宽 | 每跳延迟 (TLP) | 4KB DMA 时间 |
|:---------:|:-------:|:--------:|:--------------:|:-----------:|
| Gen3 | 8 GT/s | ~16 GB/s | ~100-200ns | ~400ns |
| Gen4 | 16 GT/s | ~32 GB/s | ~70-150ns | ~200ns |
| Gen5 | 32 GT/s | ~64 GB/s | ~50-100ns | ~130ns |
| Gen6 | 64 GT/s(PAM4) | ~128 GB/s | ~40-80ns | ~80ns |

### 14.3 RDMA Read vs Write 延迟差异

```
RDMA Write (单向):
  T_write = T_prep + T_dma_read + T_packet + T_line + T_switch + T_dma_write
  ≈ 2.0μs (4KB, 1跳)

RDMA Read (双向):
  T_read = T_prep + T_dma_noop + T_req_pkt + T_line + T_switch + T_remote_dma_read
         + T_resp_pkt + T_line + T_switch + T_dma_write
  ≈ 3.5μs (4KB, 1跳)
```

RDMA Read 比 Write 多出：
- 远端 DMA Read 的时间 (~200ns)
- 额外的往返线缆传输 (~100ns)
- 额外交换机转发 (~350ns)
- 响应包的封装/解析 (~400ns)

**总计多 ~1.0-1.5μs**，这是为什么 Write 是梯度同步的默认选择。

## 第15章：一致性模型——PCIe Ordering 与 RDMA 语义

### 15.1 PCIe Ordering 模型：DMA 一致性的基础

DMA 的一致性本质是 **PCIe Ordering Domain** 内的事务排序问题。

#### 15.1.1 PCIe 事务分类

| 事务类型 | 示例 | 特性 |
|:---------|:-----|:-----|
| **Posted (PW)** | Memory Write | 不需要 Completion，发送即完成 |
| **Non-Posted (NP)** | Memory Read, Atomic | 需要 Completion 返回 |
| **Completion (Cpl)** | Read 的返回数据 | 与对应的 Non-Posted 匹配 |

#### 15.1.2 PCIe Ordering Rules (简化)

PCIe 规范定义了事务的排序规则。核心规则如下：

| 顺序规则 | 场景 | 允许重排？ |
|:---------|:-----|:----------:|
| **Rule 1**: PW → PW (同流) | 同一个设备连续写 | 可以（但通常不重排） |
| **Rule 2**: PW → NP (同流) | 先写后读同一设备 | 读可能超过写（读可被处理更快） |
| **Rule 3**: NP → PW (同流) | 先读后写 | 写不能超过读 |
| **Rule 4**: 跨流 | 设备A → 设备B | 无保证 |
| **Rule 5**: Cpl → PW | Read Response 与之后的写 | Read Response 必须先于后续写 |

#### 15.1.3 DMA 场景的排序问题

考虑以下场景（内存一致性窗口）：

```
CPU 写标志位 (PW)               设备通过 DMA 写数据 (PW)
      │                                │
      时间→                             │
                                        ▼
    内存此刻状态: [数据: ???, 标志位: READY]
```

**问题**: 如果 CPU 写标志位的 PW 在 DMA 写数据的 PW 之前被观察到（尽管 CPU 的 PW 是在 DMA 的 PW 之后发出的），设备读到的数据可能不全。

**解决方案**:
1. **MMIO 读作为 Drain**: 在设置完描述符后对 DMA 控制寄存器做一次 MMIO 读 → PCIe NP → 强制之前的 PW 全部排空
2. **Fence 指令**: CPU 发出 MFENCE/SFENCE 确保 store 顺序
3. **PCIe TLP 处理提示**: 使用 Relaxed Ordering 位控制排序

### 15.2 RDMA 的一致性模型

#### 15.2.1 RDMA 操作的排序语义

RDMA 定义了自己的排序规则，通过 **Work Request 的 Fence 标志** 控制：

```
QP 上的 WR 序列 (默认顺序):
  WR_0: RDMA Write (data A) → 立即生效
  WR_1: RDMA Write (data B) → 可能在 A 之前完成（允许重排！）
  WR_2: Send (通知)        → 可能在 A 和 B 之前完成

如果 WR_2 需要确保 A 和 B 都完成:
  WR_1.flags = IBV_SEND_FENCE  // 等待之前所有 WR 完成
  WR_2: Send (通知)
```

**RDMA 默认允许的重排**：
- 同 QP 内的不同 WR 可能乱序完成
- 不同 QP 之间无顺序保证
- ACK 的接收和 CQ 的生成有独立的顺序

#### 15.2.2 一致性保证的层次

| 层次 | 保证 | 机制 |
|:-----|:-----|:------|
| **PCIe 层面** | 同设备同流 Posted 事务最终被看到 | PCIe Ordering Rules |
| **传输层 (RC)** | 同一 QP 的包按 PSN 顺序交付 | PSN 递增 + 重传排重 |
| **完成层** | 同一 QP 的 CQ Entry 按发送顺序生成 | 需设置 Fence 标志 |
| **应用层** | 所有操作在 Send 完成前就绪 | 应用层 Barrier / Fence |

### 15.3 RDMA Atomic 的真正一致性

```
节点 A (发起)                     节点 B (目标)
┌────────────────┐              ┌────────────────┐
│ Atomic_CAS     │              │ RNIC 硬件       │
│ addr=0x1000,   │              │                 │
│ compare=5,     │ ──Request──→ │ • 锁定内存地址   │
│ swap=10        │              │ • 读 *0x1000    │
│                │              │ • 比较 (=5?)    │
│                │              │ • 写 10         │
│                │ ←─Response── │ • 解锁           │
│                │   (old=5)   │ • 返回原值       │
└────────────────┘              └────────────────┘
```

**Atomic 的一致性保证**：
1. **串行化**: 多个节点对同一地址的 Atomic 操作串行执行
2. **读-改-写原子性**: 在 RNIC 内部锁周期内不可中断
3. **对缓存不透明**: 不保证 CPU 缓存一致性（CPU 可能看到旧值）

**关键限制**: RDMA Atomic 不参与 CPU 缓存一致性协议。如果 CPU 同时写同一地址，可能出现不一致。

## 第16章：缓存一致性在 DMA 上下文中的角色

### 16.1 问题本质：DMA 绕过 CPU Cache

```
CPU Cache ──→ 内存 ──→ DMA (设备)
     │                    │
     │  CPU 写地址 A     │  设备 DMA 读地址 A
     │  (数据在 Cache 中) │  (直接读内存)
     ▼                    ▼
     Cache: [A=42]      Memory: [A=0] (旧值)
                                  ↑ 设备读到错误值!
```

**DMA 与 CPU Cache 的一致性问题**：
1. **CPU 写 → DMA 读**: CPU 修改的数据还在 Cache，未写回内存
2. **DMA 写 → CPU 读**: DMA 直接写内存，CPU Cache 中的副本变脏

### 16.2 解决方案

#### 16.2.1 软件一致性（最常用）

```
DMA 读之前 (CPU 确保数据已写回):
  // CPU 写入数据到 buffer
  memcpy(buffer, data, size);
  // 刷 Cache → 确保写入内存
  cache_flush(buffer, size);   // clflush / clwb 指令
  // 现在设备 DMA 读可以得到正确数据

DMA 写之后 (CPU 确保 Cache 失效):
  // 设备 DMA 写入 buffer
  // 等待 DMA 完成确认
  // 使 CPU Cache 行失效 → 下次读从内存取
  cache_invalidate(buffer, size);  // clflush / clzero / INVD
  // 现在 CPU 可以读到 DMA 写入的正确值
```

#### 16.2.2 硬件一致性（IO Cache Coherence）

现代 CPU（x86）通过 **自窥探（Self-Snooping）** 和 **总线嗅探** 维护 DMA 一致性：

```
x86 架构: 所有 PCIe Memory Write 自动触发 CPU Cache 窥探
  PCIe MWr (DMA 写入地址 A)
    → Root Complex 收到
    → 嗅探 CPU Cache → 如果 CPU Cache 有 A，标记失效
    → 数据写入内存
    → 下次 CPU 读 A → Cache Miss → 从内存读取新鲜值
```

**x86 实际上不保证 DMA 一致性！** 以上是简化说法。实际情况：

- x86 PCIe MWr **不自动触发 Snoop**
- 需要设备或软件设置 **NoSnoop** 位来控制
- 默认行为是 NoSnoop=0（设备请求窥探），但许多高性能设备设置 NoSnoop=1（跳过窥探）
- **跳过窥探时，软件必须显式处理一致性**

#### 16.2.3 AC 一致性（Non-Coherent DMA）

许多 ARM SoC 采用不同策略：

```
ARM 架构 (典型):
  DMA 区域使用 Normal Non-Cacheable 或 Device 内存属性
  → CPU 不会缓存这些区域
  → DMA 直接读写 → 天然一致
  → 代价: CPU 访问延迟增加（不能缓存）
```

### 16.3 DMA 一致性在各场景的实践

| 场景 | 一致性方法 | 代价 |
|:-----|:-----------|:-----|
| **NVMe (Linux)** | DMA 缓冲区使用 `dma_alloc_coherent()` → 非缓存内存 | CPU 读缓冲区延迟略高 |
| **GPU Direct RDMA** | GPU 显存作为 DMA 区域，GPU Cache 通过 NVLink 一致性协议维护 | 硬件复杂度高 |
| **NIC Ring Buffer** | 描述符使用一致性内存，数据缓冲区使用流式 DMA (`dma_map_single`) | 需要同步屏障 |
| **RDMA MR** | 注册时锁定 + DMA 映射，应用负责 CPU 侧 Cache 维护 | 软件复杂度 |

## 第17章：总结——DMA/RDMA 系统设计的核心权衡

### 17.1 三大核心权衡

```
─────────────────────────────────────────────────────────
 权衡 1: 吞吐 vs 延迟
─────────────────────────────────────────────────────────
  DMA 大块传输: 吞吐优先，延迟不敏感
  RDMA 小消息: 延迟优先，吞吐不重要

  NVMe 读 1MB 数据: 吞吐~7GB/s, 延迟~15μs
  RDMA 读 4B: 吞吐很低, 延迟~1.5μs

─────────────────────────────────────────────────────────
 权衡 2: 硬件复杂度 vs 灵活性
─────────────────────────────────────────────────────────
  InfiniBand: 硬件完整实现传输层 → 最低延迟 + 最高吞吐
              → 成本高、封闭生态
  RoCEv2:    硬件部分 offload + 标准以太网 → 灵活、成本低
              → 需要额外拥塞控制（DCQCN）
  iWARP:     TCP offload → 可在广域网使用
              → 延迟高于 RoCE/IB

─────────────────────────────────────────────────────────
 权衡 3: 一致性保证 vs 性能
─────────────────────────────────────────────────────────
  强一致性（每次 DMA 前刷新 Cache）: 正确但慢
  弱一致性（跳过 Snoop + 软件管理）: 快但复杂
  RDMA 传输层一致性: 可靠交付 vs 高性能原子操作
```

### 17.2 设计决策速查表

| 场景 | 推荐 DMA 策略 | 推荐 RDMA 策略 |
|:-----|:-------------|:--------------|
| AI 梯度同步 | GPU P2P DMA (NVLink) | RC RDMA Write |
| 分布式存储 | NVMe DMA + IOMMU | NVMe-oF (RoCE) |
| HPC MPI | DMA 描述符环 | RC RDMA + XRC |
| 控制消息 | PIO (低频率) | UD RDMA Send |
| 远距离复制 | 本地 DMA + 应用层 | iWARP RDMA |

### 17.3 未来方向

1. **CXL 对 DMA 的影响**: CXL.mem 提供了缓存一致性 DMA，可能改变节点内数据传输范式
2. **DPU/IPU 的 DMA 卸载**: BlueField / IPU 将更多 DMA 逻辑从 CPU 卸载到 SmartNIC
3. **800Gbps 以太网 + RDMA**: 线速率提升要求更高效的 DMA Engine
4. **计算式存储**: SSD 内部 DMA Engine 直接执行计算（Near-Storage Computing）
