## 1. 总览：IPD 五阶段与领域协同骨架

### 1.1 研发主线的五个节点

```
                CDCP          PDCP          ADCP
                  │             │             │
立项 ──────── 概念 ──────── 计划 ──────── 开发 ──────── 验证 ────→ 量产
  │             │             │             │             │        │
 TR1           TR2           TR3           TR4           TR5      EVT
  │             │             │             │             │
 需求分析     方案选型     详细设计        样机测试      小批验证
 可行性      规格锁定     原理图/PCB      单板/整机     DVT/PVT
```

| TR 点 | 核心问题 | 此节点「锁」什么 |
|:------|:---------|:----------------|
| **TR1** | 做不做？ | 市场需求、产品定位 |
| **TR2** | 怎么做？ | 方案选型、领域分工、PRD v1.0 |
| **TR3** | 做得对？ | **布局锁→原理图锁→BOM 锁**（核心收敛关口） |
| **TR4** | 做得好？ | 样机功能/性能达标 |
| **TR5** | 能量产？ | 认证通过、工艺成熟 |

### 1.2 八个领域及其协同关系

```
          ┌────────── 系统架构 (PM+Arch) ──────────┐
          │            定义PRD/规格/方案              │
          │                                         │
 ┌──── 硬件 ────┐          │          ┌──── 软件 ────┐
 │ 结构 │ 散热 ││                   │ BMC │ 驱动 ││
 ├──────┴──────┤│                   ├──────┴──────┤│
 │ SI │ Layout ││                   │ 集群 │ 测试  ││
 └─────────────┘│                   └─────────────┘│
          │                                         │
 ┌──── 供应链 ───┐          │          ┌──── 质量 ────┐
 │ 采购 │ 二供   ││                   │ 认证 │ 审核  ││
 └───────────────┘│                   └──────────────┘
          │                                         │
          └─────────────────────────────────────────┘
                    项目管理 (PM) — 节奏/决策/变更
```

**八个领域间的主要信息流**：

```
系统架构 ──PRD──→ 硬件/软件/供应链/质量/测试 (各领域评估)
     ↑                  │
     │         各领域评估反馈
     └──────────────────┘

硬件-结构 ──3D图──→ 散热(热仿真) ──温限──→ 硬件-Layout(布局调整)
     ↑                                        │
     └────────────────────────────────────────┘
          反馈(干涉/限高)

硬件-SI ──走线约束──→ 硬件-Layout ──叠层──→ PCB 加工厂
硬件-供电 ──VR规格──→ 硬件-Layout ──去耦──→ PCB
软件-BMC ──Redfish──→ 硬件-传感器布局
软件-集群 ──耦合规格──→ 硬件-接口定义
```

## 2. 立项启动（TR1→TR2）—— 拉通与对齐

### 2.1 领域小组组建与同步

立项后，拉通各领域成立专项小组，**同步目标**：

| 角色 | 组成 | 第一阶段任务 |
|:-----|:-----|:------------|
| **系统架构** | PM + 首席架构师 | 编写 PRD v0.1，定义产品定位和关键规格 |
| **硬件组** | 结构/散热/SI/Layout/供电 | 评估硬件可行性：空间/功耗/互联/信号 |
| **软件组** | BMC/BIOS/驱动/集群测试 | 评估系统目标：OS/框架/管理/测试方案 |
| **测试组** | 硬件测试/软件测试/可靠性 | 编写测试策略初稿 |
| **供应链组** | 采购/供应商管理 | 识别长周期物料，预研供应商 |
| **质量组** | 质量工程师 | 制定质量计划 |

### 2.2 沟通渠道（三个级别）

| 级别 | 类型 | 频率 | 参与 | 内容 |
|:-----|:-----|:----:|:-----|:------|
| **1级：决策** | 项目周例会 | 每周 | 各领域负责人+PM | 进度、风险、资源、关键决策 |
| **2级：对齐** | 技术拉通会 | 双周（或按需） | 架构+硬件+软件 | 方案对齐、接口定义、争议讨论 |
| **3级：应急** | 紧急拉通会 | 触发式 | 相关领域 | 重大阻塞、变更影响评估、风险应对 |
| **串讲/反串讲** | 关键节点 | A→B + B→A | 相关领域工程师 | A 领域串讲设计→B 反馈→B 反串讲验证理解 |

> **串讲/反串讲机制**：方案串讲是 A 领域向 B 领域介绍设计思路，接收反馈。反串讲是 B 领域「用自己的话」复述 A 的设计，检验是否真正理解。两者都过关才算拉通完成。这对硬件-软件接口、结构-散热约束等跨领域耦合点尤为重要。

### 2.3 PRD 的逐步收敛

PRD 不是一次写完的，而是随各领域评估反馈逐步收敛：

```
PRD v0.1 ──→ 各领域评估 ──→ PRD v0.2 ──→ 拉通讨论 ──→ PRD v1.0
 初始版本      反馈+争议      细化规格       方案对比      规格锁定
                                                           │
                                                    后续变更走 PCR
```

**收敛三原则**：

1. **早期分析到位，确认争议点**：端口数、板卡位置、按键指示灯、盘排序、盘带宽等—早期不敲定，后期翻案代价极大
2. **在给定范围内设计，确认极限**：这个机箱/功耗/成本范围内，能做到的极限是什么？
3. **区分硬性要求 vs 可协商要求**：硬性要求（端口个数/盘带宽/散热极限）决定物理边界；可协商项（走线方式/器件布局）留给后续优化

### 2.4 硬性要求清单（TR1 必须明确）

| 类别 | 硬性要求 | 锁定后影响 |
|:-----|:---------|:-----------|
| **端口** | 面板端口数量与类型（如 8×100G） | 面板开模、PCB 走线、散热风道 |
| **板卡位置** | GPU/OAM/Power 模块固定位置 | 结构框架、Manifold、水冷管路 |
| **按键/指示灯** | 位置与功能定义（前/后面板） | 结构开孔、BMC GPIO 分配 |
| **硬盘排序** | 盘位编号必须与 OS 识别一致 | 背板走线、SAS Expander |
| **盘带宽** | NVMe x4/x8/x16 链路带宽 | PCIe Switch 端口分配、PCB 走线 |
| **散热上限** | 最大可散走功耗（W） | 决定可塞入的板卡总数 |
| **机箱尺寸** | U 数 × 深 × 宽 | 所有领域的设计边界 |

### 2.5 硬件优先 + 软件同步

**硬件优先启动**的原因：硬件决定物理边界（机箱/功耗/信号/接口），是所有后续工作的前提。

但 **软件必须同步评估**，不等硬件定稿：

```
硬件启动 ──────────────────→ 硬件规格 ──→ 布局/原理图 ──→ 样机
   │                            │              │
软件同步评估 ──→ 系统目标 ──→ 耦合规格 ──→ 驱动开发 ──→ 集群联调
                ↑ 确定需要周边        ↑ 按早期接口
                  领域哪些支撑           假设推进
```

> **同步原则**：各领域启动评估，有问题及时反馈讨论。不反馈 = 默认方案可行。等发现问题才反馈 = 返工成本已经产生。

## 3. 方案收敛（TR2→TR3）—— 从模糊到明确

### 3.1 芯片选型：在 Datasheet 未到时的生存法则

这是研发中最常见的困境：芯片选型定了，但关键参数还没拿到。**不能等，先搭建可调整的框架**：

#### 第一阶段：用已有信息搭建骨架

| 未确认参数 | 临时方法 | 后续精调方式 |
|:----------|:---------|:------------|
| **功耗 (TDP)** | 按芯片厂商早期宣称值 + 20% 裕量 | Datasheet 到手后缩小裕量 |
| **Pin 角/封装尺寸** | 用芯片厂商的「初步机械图」或 Pinmap 草案 | 正式机械图到手后核实 |
| **信号功能定义** | 按接口协议通用假设（PCIe/CXL/HCCS 标准信号） | 风险很低，协议层兼容 |
| **时钟/时序** | 预设可配置时钟方案（PLL/Clock buffer 可调） | 通过软件配置适配 |
| **热仿真 model** | 用典型功耗 + 封装 Rth 包络值建仿真模型 | 正式 model 到手后刷新 |
| **电压/电流** | 按芯片代际典型值 + 20% 余量 | 正式值确认后调整 VR 配置 |
| **Design Guide** | 参考前代产品 Design Guide + 通用布局规则 | 正式版验证差异 |

#### 第二阶段：逐步明确（Datasheet 到手后）

Datasheet 确认后，逐项落实到对应领域：

| Datasheet 内容 | 解析后产出 | 受益领域 |
|:---------------|:-----------|:---------|
| 信号功能 + 电平 + Pin map | Pin 功能分配表、原理图符号 | 原理图设计 |
| 电压 + 电流 + 上电时序 | 供电拓扑、VR 选型、Power Sequence | 供电 + Layout |
| 时钟频率 + 抖动 + PLL 配置 | 时钟树、Clock Buffer 选型、走线约束 | SI + Layout |
| 建立/保持时间、Skew 预算 | 走线长度约束、等长规则 | SI + PCB |
| Design Guide | 布局禁布区、去耦建议、热管理参考 | 结构 + Layout + 散热 |
| 功耗/热仿真 model | 热仿真模型更新、散热方案验证 | 散热 + 结构 |
| 工作温度/湿度/海拔边界 | 可靠性测试用例 | 测试 |

### 3.2 极限评估：在约束中找最优

给定 PRD 范围内的约束，评估设计的**极限**，取最短的那块板：

```
约束 1 ── 结构空间：机箱可用尺寸 (U × 深 × 宽)
约束 2 ── 散热能力：风量/冷板能力决定最大可散功耗
约束 3 ── 供电能力：PSU 总额定功率 - 系统基础损耗
约束 4 ── 互联需求：需多少个 PCIe Switch / Retimer / 端口
约束 5 ── 信号极限：最长走线能否满足 112G/PCIe6 损耗预算
                          │
                   取最紧约束 = 设计极限
```

**典型调整场景**：

| 约束冲突 | 调整手段 | 决策者 |
|:---------|:---------|:-------|
| Cable tray 占空间 vs 需要更多板卡 | 调整 tray 数量/位置，牺牲部分维护便利 | 结构+架构 |
| Manifold 限高 11mm vs 器件高度 > 6mm | 改用更低 profile 的插座/connector | 硬件+结构 |
| 112G 走线受 54V 电源层干扰 | 增加地层隔离（加层/改层叠） | SI+Layout |
| PCB 面积利用率低 | 重新排布器件位置（见下方案例） | 硬件全体 |

> **面积利用率案例**：OAM 左边只有 MCIO 连接器，右边放了电容。评估发现：如果把电容挪到 OAM 右侧，左边只剩 MCIO——那半边 PCB 面积利用率很低。决策：**保留原 AI Rack GPU 左侧尺寸不动**，把拿掉 manifold 多出来的 35mm 全部放在 OAM 右侧给电源部分。同时评估右侧能否放下 OAM 到新 manifold 之间的水线连接件。

### 3.3 供应链：尽早锁定，同步备二供

**明确器件后尽早采购备货**，同步考虑二供：

| 时间窗口 | 行动 | 目的 |
|:---------|:-----|:-----|
| **TR1→TR2** | 识别长周期物料（PCB 板材/连接器/定制散热） | 提前锁定产能（国产 PCB 厂订单已排至 2028） |
| **TR2 确认选型** | 核心芯片（CPU/GPU/Switch/Retimer）下单 | Lead Time 3~6 个月 |
| **TR2→TR3** | 二供认证（备选芯片/替代连接器/替代板材） | 防止单一供应中断 |
| **TR3 锁 BOM** | EVT 样机物料下单 | 确保样机按时交付 |
| **EVT→DVT** | 正式 BOM 锁定、批量备料 | 量产准备 |

## 4. 硬件协同设计 Sprint（TR3）—— 多领域螺旋收敛

### 4.1 领域间依赖与串行→并行转换

硬件设计不是所有领域同时同步推进的，而是有明确的**前后依赖**：

```
             结构初版
                │
    ┌───────────┼───────────┐
    │           │           │
  散热        Layout     供电规划
  热仿真      预布局     电压轨定义
    │           │           │
    └─────┬─────┘           │
          │                 │
      结构调整           VR 选型
      布局优化              │
          │                 │
    ┌─────┴─────┐           │
    │           │           │
  散热        Layout     SI 介入
  再仿真     详细布局    走线约束
    │           │           │
    └─────┬─────┴─────┬─────┘
          │           │
       结构锁定     PCB 层叠确认
       3D 冻结     阻抗/叠层设计
          │           │
          └─────┬─────┘
                │
           布局锁定
         (Layout Freeze)
```

**关键转变**：从「串行等待」变为「并行迭代」——结构出初版→散热就用初版仿真→发现过热→反馈结构调整→结构出新版→散热再用新版仿真。**这个循环要转 2~3 轮才能收敛**。

### 4.2 内部互联拓扑确定

PCIe/CXL/HCCS 拓扑是硬件设计的核心骨架，必须在 Layout 前确定：

```
                      CPU (Host)
                     ──┬──┬──┬──
                       │  │  │
                  PCIe Switch
                ┌───┼──┼──┼──┼───┐
               GPU GPU GPU GPU ...
                │   │   │   │
              Retimer (112G 长距)
```

**互联拓扑拉通讨论的问题清单**：

| 问题 | 决策角度 | 拉通哪些领域 |
|:-----|:---------|:------------|
| 哪些器件能塞进去？哪些不行？ | 空间/功耗/散热/信号的极限评估结果 | 全领域 |
| 拆解到什么颗粒度？ | 功能模块聚合 vs 板级集成 vs 独立子卡 | 架构+硬件 |
| 用 connector 还是 mezz 连接？ | 信号质量 (SI) + 成本 + 维护性 | SI+结构+硬件 |
| 哪些用 CBB（可复用电路）？ | 投入产出比 + 已验证状态 | 硬件 |
| VR 规格能否覆盖所有电源需求？ | 各负载功耗 + 纹波 + 瞬态响应 | 供电 |

### 4.3 单板布局评估的三轮迭代

#### 第一轮：能不能摆下？

- 各单板在机箱内是否有位置
- 可安装性检查：螺丝/导轨/把手有空间吗？
- 去异型板：尽量减少异型单板，降低 PCB 成本和走线复杂度

#### 第二轮：信号能不能走通？

- 高速信号（112G SerDes）从 54V 电源下方穿过时，**必须隔地层**
- 评估走线长度能否满足损耗预算
- 如果走线绕太长 → 调整器件位置 / 加 Retimer / 换板卡布局
- **Cable Routing 评估**：线缆走线路径、弯折半径、绑扎位置

#### 第三轮：面积利用率够不够？

- 电容 + connector + 电源模块 + 去耦电容能否摆下
- 器件是否干涉（高度/禁布区/限高区）
- 结构+散热+Layout 三方确认

### 4.4 供电方案深化

#### 电压域决策树

```
问题：NPU/GPU 用 54V 还是 12V？
├── 54V: 直供，效率高，线损小，但需要 54V→Core 变换
│   └── 适合 >300W 的大功耗器件
└── 12V: 成熟方案，兼容性好，但中间总线路损大
    └── 适合 <300W 的器件
```

#### 逐项评估清单

| 评估项 | 详细内容 | 承接领域 |
|:-------|:---------|:---------|
| MosFET 参数 | Rds(on) / Qg / Qgd | 供电+散热 |
| MosFET 结温 | 评估满载温度 ≤125°C | 散热 |
| Retimer 供电 | 低噪声 1.8V/0.9V 方案 | 供电+SI |
| 电源砖数量 | **能否只用 1 块 DC/DC brick？** 评估风扇用 54V 后负载分布 | 供电+结构 |
| 线损评估 | 54V 走线 IR Drop，确定线宽和铜厚 | Layout |
| 输出电容 | 满足瞬态响应需求 | 供电+Layout |

> **风扇供电决策**：采用 54V 风扇后，能否只用一个 DC/DC brick？评估要点：风扇总功率 + 是否与 54V 其他负载共轨 + EMI 纹波要求是否能满足。

### 4.5 时钟方案

时钟方案**必须早于 Layout 确定**，后期改时钟 = 原理图和 PCB 大面积重做：

| 时钟域 | 频率 | 抖动要求 | 分配方式 |
|:-------|:----:|:---------|:---------|
| CPU RefClk | 100 MHz | ≤0.5 ps RMS | 专用 Clock Buffer |
| PCIe RefClk | 100 MHz | ≤0.3 ps RMS | SRIS / Common Clock |
| 以太网 RefClk | 156.25 MHz | ≤0.2 ps RMS | 本地晶振 |
| GPU/NPU RefClk | 按芯片要求 | 按 Datasheet | 独立分配 |
| USB/SATA | 24/25 MHz | 普通 | 晶振直连 |

## 5. 锁图定案（TR3 关口）—— 从设计走向落地

### 5.1 三道锁：结构锁 → 布局锁 → 原理图锁

这是 TR3 阶段最关键的三道关口，每道锁意味着该部分**冻结**，后续变更走 PCR：

```
   结构锁           布局锁           原理图锁
     │                │                │
     ▼                ▼                ▼
  板卡外形        器件位置         连接关系
  固定            固定             固定
     │                │                │
     ▼                ▼                ▼
  散热/SI 启动     PCB 布线启动     BOM 发布
  迭代             原理图定稿      软件接口锁定
```

**三道锁的含义**：

| 门禁 | 锁的条件 | 锁的意义 | 解锁代价 |
|:-----|:---------|:---------|:---------|
| **结构锁** | 3D 完成、无干涉、结构与散热/供电协商一致 | 板卡外形固定 → 散热/SI/Layout 可以启动 | 改外形 = 结构重做、散热重新仿真 |
| **布局锁** | DXF 发布、各领域签字确认 | 器件位置固定 → PCB 布线启动、原理图定稿 | 改布局 = 布线重做、时序重算 |
| **原理图锁** | 原理图评审通过、BOM 初版发布 | 连接关系固定 → 软件接口锁定、采购下单 | 改原理图 = BOM 变更、软件适配返工 |

### 5.2 结构出图后的领域确认（检查清单）

结构给出 3D 图后，各领域必须逐项确认（在布局锁之前）：

```
┌─────────────────────────────────────────────────┐
│                 结构 3D 图 (结构)                  │
├─────────────────────────────────────────────────┤
│ 散热:  MCIO 位置是否影响水冷管路？                │
│ 硬件:  Flash 是否有 Socket？高度多少？（正常~6mm） │
│ 散热:  冷管高度能做到 13mm 吗？限高多少？（11mm？）│
│ 散热:  OAM 到 manifold 间水线连接件位置是否充足    │
│ 系统:  面板端口/指示灯/按键是否对齐标准            │
│ 结构:  防反设计有吗？连接器防呆？                  │
│ 供电:  电源砖位置是否合理？线损路径最短？           │
│ 散热:  Cable routing 是否影响风道/水路？           │
└─────────────────────────────────────────────────┘
                      │
              各方确认无问题 → 布局锁
```

### 5.3 热仿真：从模型到结果

热仿真不是一次完成，而是与结构调整**交替迭代**：

```
3D 布局图 (结构)
     ↓
建立热仿真模型 (散热)
     ↓
选取最恶劣工况：高温环境 + 满载功耗 + 边界条件
     ↓
流场/温度场仿真
     ↓
┌── 发现问题？── 是 ──→ 调整器件/散热片/风道/管路 ──→ 更新 3D
│                             ↑
│                     结构调整后重新仿真
│                             ↓
└── 无 ──→ 布局锁 ──→ PCB Layout 启动
```

热仿真完成后的输出：

| 输出项 | 内容 | 使用方 |
|:-------|:-----|:-------|
| 温度云图 | 各器件结温，验证 ≤85°C | 硬件+散热 |
| 流场图 | 风道/水路流向、流速分布 | 结构 |
| 热点分析 | 最高温器件列表，需降频或调整 | 硬件+散热+供电 |
| PUE 预估 | 液冷/风冷效率 | 系统架构 |
| Cable Routing 建议 | 走线路径不影响散热 | 结构+Layout |

### 5.4 布局锁定 + DXF 发布 → 进入落地

布局锁定后，DXF 发布，进入「落地阶段」：

```
布局锁定 + DXF 发布
     │
┌────┴────┐
│  PRD    │  ← PRD 再做一次确认：当前实现是否满足需求
│ 再确认  │      与最初 PRD 比，有没有偏差需要更新
└────┬────┘
     │
┌────┴─────────────┐
│  PCB Layout 启动  │  ← 正式布线
│  原理图定稿       │  ← 锁定连接关系
│  BOM 发布         │  ← 开始采购
│  软件接口锁定     │  ← 按最终原理图定稿接口
└──────────────────┘
```

**核心转变**：

> 布局锁之后，重心从「方案设计」转为「落地执行」。PCB 工作、原理图定稿、BOM 发布——这些都是执行层面的工作，不再做大的方案调整。方案层面在布局锁之前就应该全部搞定。

### 5.5 加工考虑（布局锁后立即评估）

在启动 PCB 之前，先看加工端的要求：

| 考虑项 | 具体内容 |
|:-------|:---------|
| **防反** | 连接器防呆（防止错插/反插）、电源反接保护 |
| **扩展** | 是否需要预留升级接口（PCIe slot/内存槽/焊盘） |
| **降成本** | PCB 能否减少层数？板材能否降级？电源模块能否合并？ |
| **可制造性** | 线宽/线距/钻孔孔径满足加工厂能力范围？层压工艺 OK？ |

### 5.6 领域收敛与变更管理

各领域不断协商，但要知道**什么时候收敛**，变更的影响**怎么评估**：

```
收敛规则 (按锁的层次)：

结构未锁 → 任何结构变更代价低，畅所欲言
结构已锁 → 改结构=散热重仿+Layout重排 → 需要拉通会决策
布局已锁 → 改布局=PCB布线重做 → 需要项目层面审批
原理图已锁 → 改原理图=软件接口变更+BOM更新 → 走 PCR 流程
EVT 样机产出后 → 只修 Bug，不做功能变更 → 走 PCR + ECO
```

**变更影响评估模板**：

```
变更描述：_________________________
提出领域：_________
影响评估：
  ├── 结构：□ 无  □ 需改3D  □ 需新开模
  ├── 散热：□ 无  □ 需重仿  □ 需换冷板
  ├── SI：  □ 无  □ 需重仿  □ 走线重排
  ├── 供电：□ 无  □ 需改VR  □ 需换PSU
  ├── Layout：□ 无  □ 需局部重排  □ 大面积重排
  ├── 软件：□ 无  □ 需改接口  □ 需改驱动
  ├── 供应链：□ 无  □ 需换物料  □ 需重新采购
  └── 进度影响：___ 天
决策：□ 同意变更  □ 推迟到下一版  □ 否决
```

## 6. 软件并行轨（TR2→TR4）—— 测试策略与耦合管理

### 6.1 软件同步评估（与硬件并行）

硬件启动的同时，软件必须并行评估：

| 评估维度 | 需要问的问题 | 需要硬件/架构提供的输入 |
|:---------|:------------|:-----------------------|
| **系统目标** | 跑什么模型？多大参数量？需要多少显存？ | 算力规格、HBM 容量 |
| **OS 兼容** | 支持 Linux 哪些发行版？国产 OS（麒麟/欧拉）？ | 固件兼容列表、芯片驱动支持 |
| **BMC 方案** | OpenBMC 还是厂商方案？Redfish Schema 怎么定义？ | 传感器布局、FRU 信息、告警需求 |
| **BIOS/UBIOS** | 是否适配 UBIOS？启动流怎么走？ | 芯片内存支持、Boot 顺序 |
| **驱动** | GPU/NIC/RAID 驱动适配状态？驱动谁提供？ | 芯片选型结果、芯片厂商支持计划 |
| **集群管理** | 用 FusionDirector / 自研 / 第三方？ | 管理网络拓扑、Redfish 资源布局 |
| **测试策略** | 板级→整机→机柜→集群的分层方案？ | EVT 样机数量、交付时间 |

### 6.2 硬件→软件耦合规格（必须在 TR3 前明确）

这是最容易出问题的地方：**耦合规格不明确，后续只能自己领域搞定**。早期必须拉通、确认、纳入基线。

#### 第一优先级：接口耦合（TR2→TR3 期间拉通）

| 耦合项 | 硬件输出 | 软件使用 | 拉通时机 |
|:-------|:---------|:---------|:---------|
| **Redfish Schema** | 传感器路径、事件定义、FRU 布局 | BMC + 管理平台 | 结构布局完成后 |
| **I2C 地址分配** | 各器件 I2C 地址、Bus 编号、速率 | 固件读取传感器、CPLD 控制 | 原理图设计阶段 |
| **GPIO 定义** | 复位/中断/状态 LED/ID 定义 | 固件控制逻辑 | 原理图设计阶段 |
| **PCIe 拓扑** | BDF Mapping、Port 分配、链路宽度 | OS 设备枚举、驱动加载 | 互联拓扑确定后 |
| **Sensor 阈值** | Warning/Critical 限值、迟滞值 | BMC 告警上报 | 散热仿真完成后 |
| **功耗封顶** | PMBus 命令格式、功率上限阶梯 | BMC 功耗管控 | 供电方案确定后 |
| **Boot 顺序** | Boot Device 列表、启动延迟 | BIOS/UBIOS 配置 | 原理图阶段 |

#### 第二优先级：集群测试耦合

| 耦合项 | 硬件依赖 | 软件措施 |
|:-------|:---------|:---------|
| **NCCL/HCCL 通信** | 互联拓扑、链路带宽 | 集合通信库适配、拓扑优化 |
| **跨节点时钟同步** | IEEE 1588 PTP 支持 | NTP/PTP 配置 |
| **管理网络** | BMC 管理口 MAC/PXE 支持 | 批量发现、部署工具 |
| **故障注入接口** | NPU/GPU 模拟故障支持 | 故障注入测试脚本 |
| **Checkpoint/ Restore** | 存储带宽、NVMe 容量 | 分布式 Checkpoint 框架 |

> **核心原则**：不能事先明确的耦合规格，后续只能自己领域搞定。所以：
> 1. 早期拉通时双方确认「这个接口谁定义、谁使用」
> 2. 规格文档**双方签字**，纳入基线
> 3. 变更时互相通知、评估影响

### 6.3 集群测试四级方案

```
┌── Level 1: 板级测试 ──────────────────────────────┐
│  范围：单板 (主板/背板/PB)                          │
│  测试项：BMC 启动、传感器读值、电压/温度/Power Good  │
│          PCIe 链路训练、I2C 通信、GPIO 时序验证      │
│  通过标准：所有传感器读数正常，BMC 启动 < 60s        │
└────────────────────── ↓ ──────────────────────────┘
┌── Level 2: 整机测试 ──────────────────────────────┐
│  范围：单节点 (8×NPU + CPU + 网络 + 存储)          │
│  测试项：CPU/NPU 算力、内存带宽、存储 IOPS          │
│          网络吞吐/延迟、散热效率、供电效率             │
│          满负载运行 72h/168h 稳定性                  │
│  通过标准：性能 ≥ 规格 95%，无 P1/P2 级故障          │
└────────────────────── ↓ ──────────────────────────┘
┌── Level 3: 机柜测试 ──────────────────────────────┐
│  范围：整柜 (多节点 + 网络交换 + CDU + 供电)         │
│  测试项：整柜供电分配验证、液冷管路密封性              │
│          跨节点网络连通性、管理网络自动发现             │
│          机柜级散热均衡性（各节点温差 ≤5°C）           │
│  通过标准：所有节点可被管理平台发现，网络连通率 100%    │
└────────────────────── ↓ ──────────────────────────┘
┌── Level 4: 集群联调 ──────────────────────────────┐
│  范围：≥64 节点                                    │
│  测试项：HCCL/NCCL AllReduce 带宽 & 延迟             │
│          分布式训练跑通（LLaMA/DeepSeek 级模型）      │
│          故障注入（NPU 离线/网络断链/电源故障）       │
│          Checkpoint/Restore 验证                    │
│          长稳测试（集群满负载 7×24h）                │
│  通过标准：AllReduce 带宽 ≥ 理论 80%，训练收敛正常    │
└──────────────────────────────────────────────────┘
```

**测试失败逐层排除原则**：

```
单板失败 → 硬件/BMC 定位修复，不进入整机
整机失败 → 定位到具体器件/固件，闭环后才进入机柜
机柜失败 → 可能是网络/供电/散热问题，结构调优后再试
集群失败 → 逐层排除：硬件→网络→软件→训练框架
```

### 8.2 各领域需求包的统一节奏

时间线：         TR2 Gate         布局锁         原理图锁          EVT
                   │               │               │               │
统一节奏：  ┌───────┴───────┐   ┌──┴──┐    ┌──────┴───────┐   ┌──┴──┐
             各领域需求对齐      物理锁定     接口锁定        实物验证

领域 A:  ┌────────── 需求包 A ─────────────────┐  ← 验证 A
领域 B:  ┌────────── 需求包 B ─────────────────────┐  ← 验证 B
领域 C:  ┌────────── 需求包 C ───────────────┐  ← 验证 C
```

- 各领域在 TR2 前输出需求包初版，在 TR2 Gate 上对齐
- 后续需求迭代 → 通过 PCR 受控变更
- EVT 时各领域用样机验证各自需求包 → 完成 = EVT 通过条件之一

| **TR2** | 🟢 方案评审、技术选型决策、规格争议仲裁、各领域 PRD 拆分对齐 | 各领域评估反馈 | PRD v2.0、PDCP 材料 |
| **TR3** | 🟢 方案一致性检查（PRD vs 实际设计）、变更影响评估、PCR 审批 | 变更请求 (PCR) | 变更决策记录 |
| **TR4** | 🟡 EVT 验收、PRD 需求闭环确认 | EVT 测试报告 | EVT 验收报告 |
| **TR5** | 🟡 ADCP 材料、产品发布 | 全领域报告 | ADCP 材料 |

### 9.4 领域关注点的阶段迁移规律

> 各领域在 IPD 流程中的活跃度不是均匀的，以下是领域活跃度的「阶段迁移曲线」：

```text
领域        TR1    TR2    TR3    TR4    TR5
───────────────────────────────────────────────
系统架构    ████   ███    ██     █      ██
结构设计    ███    ████   █████  ██     █
散热设计    ██     ███    █████  ███    █
SI          █      ███    █████  ███    █
供电设计    █      ███    █████  ███    █
原理图      █      ███    █████  ██     █
PCB Layout  █      ██     █████  ██     █
BMC 固件    █      ███    ████   ████   █
集群软件    █      ██     ██     █████  █
供应链      ████   █████  ████   ████   ████
测试        █      ██     ██     █████  █████
质量         ██    ███    ███    ████   ████
```

**关键规律**：
- TR3 是所有硬件子领域的**峰值期**，资源投入最大，并行迭代最密集
- 软件领域（BMC/BIOS/驱动）峰值比硬件晚半个相位（TR3 中→TR4 初）
- 集群软件的峰值最晚（TR4），需要硬件定型后才能联调
- 供应链活跃贯穿始终，峰值在 TR2（核心芯片下单时不我待）和 TR5（量产备货）
- 测试和质量峰值在 TR4~TR5，它们是「设计做完了才真正忙起来的领域」

## 10. 三维跨领域交互矩阵

> 如果说 §9 是每个领域的「独奏谱」，本节就是 **全乐队的合奏谱**。
>
> **三维关系定义**：
> 1. **X 轴**：交互发起方（提供信息 / 驱动需求的领域）
> 2. **Y 轴**：交互接收方（需要输入 / 受影响的领域）
> 3. **Z 轴**：时间（IPD 阶段）——同一对领域在不同阶段的交互内容和关系方向是动态变化的
>
> 每个交互由「**输入→谁用→做什么→什么时候**」四要素定义。

### 10.1 三维交互模型定义

```text
                     ┌─── 什么阶段 ──→ Z: 时间轴
                     │     TR1  TR2  TR3  TR4  TR5
                     │
交互发起方 (X) ──────┼──── 交互内容 ────→ 交互接收方 (Y)
                     │
                     │ 交互类型:
                     │  ① 信息依赖: A 的输出是 B 的输入（串行依赖）
                     │  ② 约束反馈: A 的评价约束 B 的选择（闭环）
                     │  ③ 仲裁决策: 架构/PM 裁决 A 与 B 的争议
                     │  ④ 接口对齐: A 与 B 共同定义接口规格
```

**四大交互类型**：

| 类型 | 特征 | 典型场景 | 失败模式 |
|:-----|:------|:---------|:---------|
| **① 信息依赖** → | A 输出 → B 用。串行链，A 不完成 B 无法开始 | 结构出 3D → 散热才能做热仿真 | 结构一直不改，散热永远等 |
| **② 约束反馈** ↔ | B 评价 A 的输出，给出约束反馈，A 调整 | 散热说「这个器件太热需要移」→ 结构调整 | 反馈不及时，布局锁了才发现过热 |
| **③ 仲裁决策** △ | 架构/PM 在 A 与 B 的冲突中做决定 | 结构要更大的冷板 vs 供电要更大的电源砖 | 仲裁拖延，两方互等 |
| **④ 接口对齐** ○ | A 与 B 联合定义共享接口 | 硬件管脚分配与 BMC GPIO 联动定义 | 各定各的，联调时 GPIO 不匹配 |

### 10.2 按阶段展开的交互矩阵（含举例）

> 每个阶段只列该阶段**活跃的交互**。交互格式：`发起方 → 接收方：交互内容 | 交互类型`

#### 10.2.1 TR1~TR2（概念→立项）：架构驱动一切

**阶段特征**：系统架构是唯一的活跃发起方，所有领域都在接收输入后做初步评估。

| 发起方 | → | 接收方 | 交互内容 | 类型 | 举例说明 |
|:-------|:--|:-------|:---------|:----|:---------|
| **系统架构** | → | **结构** | PRD（板卡数量/尺寸/位置、面板需求） | ①信息依赖 | 架构决定机箱是 4U/8U，结构才能开始画框架。如果架构说「我要 8 卡+2 个 PCIe Switch+4 个 PSU」→ 结构评估放不放得下，反馈「8U 才够」→ 架构确认 |
| **系统架构** | → | **供电** | 功耗预算总表（各器件 TDP 预估值） | ①信息依赖 | 架构给各板卡分配功耗额度 → 供电评估 PSU 总功率和电压域划分 → 反馈「按这个预算需要 4 个 3kW PSU，机箱空间够吗？」 |
| **系统架构** | → | **散热** | 散热方案方向（风冷/液冷/混合） | ①信息依赖 | 架构说「目标是 8×700W NPU，必须液冷」→ 散热开始评估冷板方案和 CDU 选型 |
| **系统架构** | → | **供应链** | 芯片选型方向清单 | ①信息依赖 | 架构确认选 Ascend 910C → 供应链立即联系供应商锁定 Lead Time 和产能，不等原理图 |
| **结构** | → | **散热** | 初版 3D 概念模型（机箱外形/板卡摆放） | ①信息依赖 | 结构给出机箱概念图 → 散热初步看风道/水路走向是否可行 → 反馈「CPU 位置和风扇之间隔了硬盘，风道受阻」 |
| **供电** | → | **结构** | PSU 外形尺寸和位置要求 | ②约束反馈 | 供电选型确定 PSU 型号 → 给结构 PSU 3D 图 → 结构调整机箱后部空间放置 PSU |

> **TR1~TR2 交互口诀**：架构先出题，各领域做答案，结构是第一个交卷的，散热是第一个批改的。

#### 10.2.2 TR2~TR3（计划→详细设计）：硬件内部密集交互

**阶段特征**：所有硬件子领域进入密集的**串行→并行迭代**。结构→散热→Layout→SI 形成主交互链。

| 发起方 | → | 接收方 | 交互内容 | 类型 | 举例说明 |
|:-------|:--|:-------|:---------|:----|:---------|
| **结构** | → | **散热** | 3D 装配图 v0.5（板卡/散热器/冷板位置） | ①信息依赖 | 结构出了初版 3D → 散热开始建热仿真模型。如果结构不发布初版，散热只能「盲仿」—— 这是最常见的拖延原因 |
| **散热** | → | **结构** | 热点反馈：「器件 A 结温 105°C，超规格 85°C，需移走或加大散热片」 | ②约束反馈 | 散热仿真发现靠近出风口的 NPU 温度超标 → 反馈结构「这颗 NPU 需要更大的冷板或换个位置」→ 结构调整冷板尺寸 → 散热重仿验证 |
| **散热** | → | **Layout** | 温限约束：「这些器件必须放在进风区，这些要远离热源」 | ②约束反馈 | 散热给出热区分布图 → Layout 预布局时把高功耗器件靠近冷板/进风口，低功耗器件可放远端 |
| **结构** | → | **Layout** | DXF 布局图（板卡外形/安装孔/禁布区/限高区） | ①信息依赖 | 结构发布 DXF → Layout 知道「器件最高不能超过 11mm」「这个区域不能放器件（冷管经过）」 |
| **Layout** | → | **原理图** | 布局反馈：「这颗电容放不下，换个更小的封装」 | ②约束反馈 | Layout 布不下所有去耦电容 → 反馈原理图「这个 1210 封装太大，改为 0805 或 0603」→ 原理图更换物料 |
| **Layout** | → | **供电** | VR 位置和散热需求 | ②约束反馈 | Layout 把 VR 放在靠近负载的位置 → 供电确认走线损可接受 → 散热确认 VR 温度 OK |
| **SI** | → | **Layout** | 走线约束表（长度/阻抗/等长/间距/过孔数量） | ②约束反馈 | SI 要求 112G 差分对走线 ≤8 inch、阻抗 85Ω±5%、过孔 ≤2 个 → Layout 布线时必须遵守，否则 SI 不签收 |
| **Layout** | → | **SI** | 实际布线图（分段导入） | ②约束反馈 | Layout 布完一段差分线 → 导给 SI 做通道仿真 → SI 反馈「这根线长了 0.5 inch，BER 超了」→ Layout 调整布线 |
| **结构** | → | **供电** | PSU/电源砖安装空间确认 | ②约束反馈 | 结构确定了 PSU 位置 → 供电评估走线长度是否满足线损要求 → 如果太长需要结构允许 PSU 换个位置 |
| **供电** | → | **散热** | MosFET 功耗评估：「这颗 MosFET 满载 3.5W，结温多少？」 | ②约束反馈 | 供电选定 MosFET 型号 → 给散热功耗数据 → 散热评估结温是否超 125°C → 超了就换更大的封装或降额使用 |

> **案例：4 轮迭代才解决的交互冲突**
>
> 场景：一台 8×NPU 液冷服务器的结构-散热迭代过程。
>
> - **Round 1**：结构出初版 3D——NPU 排列为 2 行 ×4 列，冷板在正上方，Manifold 在侧面
> - **Round 2**：散热仿真发现——靠内的 4 颗 NPU（远离 Manifold 进液口）温差比外侧大 8°C，超出目标 5°C。反馈结构「Manifold 需要移到正上方或加一路旁通」
> - **Round 3**：结构调整——Manifold 从侧面移到正中间，但占用了 OAM 右侧 35mm 空间 → 供电说「电源砖没地方放了」→ 系统架构仲裁：「35mm 全部分配给 OAM 右侧电源区，左侧固定不动」
> - **Round 4**：散热重仿——确认温度平衡了，供电确认空间够放了
> - **结果**：结构锁定了 → Layout 拿到 DXF 开始布线。**如果散热不在 Round 1 就发初版仿真，Round 3 的布局冲突会在布线后才发现，代价大 10 倍。**

#### 10.2.3 TR3 关口（锁图定案）：全领域确认→收敛

**阶段特征**：所有交互从「迭代优化」转为「确认签字」。三道锁（结构→布局→原理图）是交互密度最高的时间窗口。

| 发起方 | → | 接收方 | 交互内容 | 类型 | 举例说明 |
|:-------|:--|:-------|:---------|:----|:---------|
| **结构** | → | **所有领域** | 3D 图发布 → 逐项确认清单 | ①+④ | 结构发出 3D 最终版 → 各领域按 §5.2 检查清单逐条确认（MCIO 是否影响水冷？Flash Socket 高度是否超限？冷管高度 13mm 能否做到？）。每一项都得签字 |
| **散热** | → | **结构** | 最终热仿真报告签收 | ②确认 | 最后一轮热仿真通过 → 散热确认「所有器件结温 ≤85°C」→ 结构才能锁 |
| **Layout** | → | **结构** | DXF 确认回签 | ②确认 | Layout 确认 DXF 无干涉、安装孔对齐、限高区标注清晰 → 回签给结构，结构才能发布正式版 |
| **供电** | → | **Layout** | VR 布局确认 | ②确认 | 供电检查 VR 的散热器、去耦电容位置、走线线宽是否满足载流 → 确认签收 |
| **原理图** | → | **BMC 固件** | GPIO 分配表终版 | ④接口对齐 | 原理图定稿后，GPIO 分配表（哪个 GPIO 接什么功能）发给 BMC → BMC 按此开发固件。**如果此处 GPIO 分配变了一次而 BMC 不知道，联调时就会发现复位信号接错了** |
| **原理图** | → | **测试** | 原理图评审邀请 | ④接口对齐 | 测试参与原理图评审 → 确认测试点、调试接口、JTAG/SWD 是否引出 → 不在原理图上加测试点 = 后面 Debug 只能靠飞线 |
| **系统架构** | → | **所有领域** | PRD 再确认 | ③仲裁 | 布局锁后，架构拿着初版 PRD 对比当前设计：「实际实现的规格 ≠ 原 PRD 需求？有没有偏差？需要更新 PRD 还是调整设计？」 |

#### 10.2.4 TR3~TR4（开发→样机）：硬件→软件→测试接力

**阶段特征**：硬件交互密度下降（设计已锁定），**软件-硬件接口拉通**和**测试-设计反馈**成为新的交互主链。

| 发起方 | → | 接收方 | 交互内容 | 类型 | 举例说明 |
|:-------|:--|:-------|:---------|:----|:---------|
| **PCB Layout** | → | **加工厂** | Gerber 文件 + 加工说明 | ①信息依赖 | Layout 发出 Gerber → 加工厂做 DFM 检查 → 反馈「最小线宽 3.5mil 只能做到 4mil」「钻孔孔径不够」→ Layout 微调 |
| **供应链** | → | **硬件** | EVT 物料到货状态 | ①信息依赖 | 供应链每周更新到货计划 → 硬件根据物料到货排产「CPU 到了，GPU 还在清关，先装半套测」 |
| **BMC** | → | **硬件** | 传感器读值异常反馈：「温度 sensor 读出来 -40°C，I2C 通信异常」 | ②约束反馈 | BMC 拿到 EVT 样机后读传感器 → 发现某个温度 sensor 读数异常 → 反馈硬件「I2C 地址冲突或上拉电阻不对」→ 硬件排查 |
| **测试** | → | **硬件** | 测试 Bug：「PCIe 链路只能训练到 Gen4，不能到 Gen5」 | ②约束反馈 | 测试发现 PCIe 链路只能到 Gen4 → 硬件排查「可能是 Retimer 配置不对或 PCB 走线损耗超标」→ SI 重新测量走线损耗 |
| **测试** | → | **软件** | 测试 Bug：「BMC WebUI 刷新 10 次有 1 次卡死」 | ②约束反馈 | 测试复现并录屏 → 给 BMC 开发定位 → 发现是 AJAX 轮询冲突 → 修复 |
| **架构** | → | **测试** | 「集群测试是否达到 PRD 目标？」 | ③仲裁 | 架构问测试「PRD 要求的 8×NPU 算力达到目标了吗？AllReduce 带宽达标了吗？」→ 测试出数据验证 |

> **案例：EVT 样机第一轮通电发现的问题链**
>
> 场景：工程师给 EVT 样机第一次上电，BMC 启动正常但系统无法 POST。
>
> ```
> 测试发现: 系统无法 POST
>     → 排查: BMC 正常，但 CPU 没起来
>     → 软件定位: BIOS 没读到内存 SPD
>     → 硬件定位: I2C 总线不通
>     → 原理图追溯: I2C 地址分配时，内存 SPD 的地址和某 sensor 冲突了
>     → BMC 确认: 确实读了同一个地址，返回的是 sensor 值不是 SPD 值
>     → 修复: 修改 I2C MUX 地址分配或改原理图
> ```
>
> **教训**：I2C 地址冲突本应在 §6.2 的「耦合规格拉通」阶段发现——硬件分配 I2C 地址时 BMC 就应该参与确认。因为没拉通，问题在 EVT 样机上才暴露，修起来要改原理图→改 PCB→等下一版，至少 2 周。

#### 10.2.5 TR4~TR5（验证→量产）：测试→供应链→质量闭环

**阶段特征**：设计交互基本结束，**测试问题反馈→修 Bug→验证闭环**是主交互链。

| 发起方 | → | 接收方 | 交互内容 | 类型 | 举例说明 |
|:-------|:--|:-------|:---------|:----|:---------|
| **测试** | → | **硬件/软件** | Bug 报告 → 修复 → 回归验证 | ②约束反馈 | 测试发现「NPU 满负载 72h 后性能下降 15%」→ 硬件排查散热节流、软件排查驱动内存泄漏 → 修复后验证 |
| **可靠性测试** | → | **硬件** | 「高温 65°C 环境下 48h 后有两块盘掉线」 | ②约束反馈 | 测试在环境实验室跑高温 → 发现硬盘掉线 → 散热实测确认硬盘温度已超 60°C 规格 → 结构调整硬盘位置或加风扇 |
| **供应链** | → | **系统架构** | 「原定 A 芯片缺货，建议切换 B 芯片」 | ③仲裁 | 供应链通知「GPU 芯片下季度缺货」→ 架构评审替代方案 → 决定是否切换二供 → 涉及重新认证 |
| **质量** | → | **全领域** | 「来料抽检 5% 不良，需启用二供」 | ③仲裁 | 质量发现某个连接器批次不良率偏高 → 供应链启用二供 → 硬件确认二供物料兼容性 |
| **认证实验室** | → | **硬件** | 「EMC 辐射超标 3dB，需整改」 | ②约束反馈 | EMC 认证失败 → 结构改屏蔽罩、Layout 调整滤波电路 → 重新送测 = 2 周 |

### 10.3 关键交互链深析——案例集

> 以下是跨领域交互中最容易出问题的 5 条链，每一条都附真实场景举例和规避建议。

#### 10.3.1 结构 ↔ 散热（最频繁的交互链）

这是 TR3 阶段**迭代次数最多**的交互链，平均需要 3~5 轮才能收敛。

| 轮次 | 结构做了什么 | 散热做了什么 | 交互结果 |
|:-----|:------------|:------------|:---------|
| R1 | 出初版 3D，板卡按「经典排列」 | 建粗糙模型→发现某 NPU 超温 | 「需要更大的冷板」 |
| R2 | 增大冷板面积→发现 Manifold 没位置 | 重仿→温度 OK 了但流量不够 | 「需要更高流量的 Manifold」 |
| R3 | 调整 Manifold 位置到机箱顶部 | 重仿→温度 vs 流量都 OK | 「可以收敛了」 |
| R4 | 修改供电模块位置让出空间 | 终仿确认 | **结构锁 🔒** |

**规避建议**：不要等到结构「完全定稿」再给散热——散热在 R1 初版就可以开始跑粗模型。**提前开始 = 提前发现问题 = 提前收敛**。

#### 10.3.2 Layout ↔ SI（信号质量的决定链）

这条链决定 PCB 能不能用，**一旦布线完成再发现 SI 问题，改板代价等于重做**。

| 阶段 | Layout | SI |
|:-----|:-------|:---|
| 预布局 | 粗略摆件，不知道走线空间够不够 | 给出走线约束表：「112G 走线 ≤8 inch，每段 ≤2 过孔」 |
| 布局中 | 按约束表调整器件位置→尽量缩短高速走线 | 粗略仿真：「这个方案理论可行」 |
| 布局完成 | 导出初步走线图 | 分段通道仿真→发现 3 根「红牌」走线超长 |
| 布线中 | 调整绕行方案→用蛇形走线补偿 | 再仿→「红牌」变「黄牌」→可接受 |
| 布线完成 | 导出最终版 | 全链路仿真→眼图/BER/Jitter → **SI 签收** |

**案例：一根 112G 走线的生死之旅**

```
布局时，GPU0 ↔ GPU1 之间的 112G 差分线最短距离是 6.5 inch → SI 说 OK（预算 8 inch）
但布线时为了避免穿过 54V 电源区，绕了一圈，变成了 9.2 inch → SI 重仿发现眼图开口不足
↓
Layout 调整：在 54V 区下方加一层 GND 隔离层 → 走线路径缩短回 7.8 inch
↓
SI 再仿：眼图开口 OK，BER ≤ 1e-12 → 签收
```

**规避建议**：SI 不要在「布线完成后再介入」，而是 **「每布完一条关键链路导出来跑一下」** ，迭代式验证。

#### 10.3.3 原理图 ↔ BMC 固件（接口定义的生死线）

这条链最容易被忽视，因为原理图和 BMC 是两个团队，**联调时才发现在同一个 I2C 地址上「打起来了」**。

| 交互点 | 原理图做了什么 | BMC 做了什么 | 对齐方式 |
|:-------|:-------------|:------------|:---------|
| I2C 地址分配 | 给每个 sensor/EEPROM/CPLD 分配了地址 | 确认地址不冲突、Bus 编号一致 | 原理图评审时 BMC 必须参加 |
| GPIO 分配 | 给每个功能（复位/中断/状态）分配了 GPIO | 确认 GPIO 电平逻辑（高有效/低有效） | GPIO 分配表+电平表 |
| Sensor 阈值 | 按器件规格填写 Warning/Critical 阈值 | 按散热仿真结果确认阈值是否合理 | 阈值表 BMC 确认 |
| PMBus 地址 | 给每个电源模块分配了 PMBus 地址 | 确认与 BMC 的 PMBus 驱动匹配 | 同步地址表 |
| Redfish Schema | — | 定义资源路径、事件定义 | 硬件确认传感器路径与实际一致 |

**案例：谁动了我的 I2C 地址？**

```
原理图团队给板上的 12 个温度 sensor 分配了 I2C 地址 0x48~0x53
BMC 团队按此开发固件
↓
逻辑修改：新增了一颗 sensor 用于冷板进出水温差监测，用了地址 0x50（和原有 sensor 冲突）
改原理图的人不知道 BMC 已经按原地址开发了
↓
联调时：BMC 读 0x50 得到水温（新 sensor），但这个地址原来对着 CPU 附近的一颗 sensor
BMC 以为 CPU 温度只有 28°C——实际 CPU 已经 72°C
↓
修复需要：改原理图→I2C 改地址→BMC 改驱动→回归测试 = 1 周
```

**规避建议**：I2C 地址和 GPIO 分配走**变更控制**——原理图改任何地址/管脚分配，必须同步给 BMC，BMC 确认不冲突才算闭环。

#### 10.3.4 供电 ↔ 散热（热与电的权衡）

高功耗器件时代，供电效率和散热能力是一对「零和博弈」——效率高发热小，但效率高的 VR 更贵更大。

| 决策点 | 供电角度 | 散热角度 | 最终方案 |
|:-------|:--------|:--------|:---------|
| MosFET 选型 | 选 Rds(on)=1.2mΩ 的，效率 95%，但结温 112°C | 建议选 Rds(on)=0.8mΩ 的，结温 98°C，但贵 30% | 选 Rds(on)=0.8mΩ——散热裕量更值钱 |
| PSU 功率 | 建议 2×3kW，冗余 N+1 | 评估 3kW PSU 散热需要独立风道 | 采用 2×3kW PSU + 独立风道设计 |
| 风扇供电 | 54V 风扇 → 能否只用一个 DC/DC brick？ | 如果只用 1 个 brick，故障时全部风扇掉电=整机宕机 | 用 2 个 brick 做冗余，成本增加但可靠性提升 |

#### 10.3.5 供应链 ↔ 全领域（最容易被低估的交互）

供应链不是「设计好了再去买东西」的，而是**从第一天就参与决策**的。

| 阶段 | 供应链介入 | 设计决策影响 | 灾难案例 |
|:-----|:----------|:------------|:---------|
| TR1 | 报 Lead Time：「这个芯片现在下单要 6 个月」 | 项目排期调整为 7 个月 | 不提前问→设计好了才发现芯片买不到→换芯片→设计重做 |
| TR1 | 报产能：「PCB 厂订单已排到 2028」 | 提前锁定合作 PCB 厂产能 | 到做 PCB 时才发现产线排满了→等 3 个月→EVT 延期 |
| TR2 | 报备选：「推荐备选 A/B 两种替代芯片」 | 硬件设计时兼容两套方案（Pin-to-Pin） | 不备二供→主芯片缺货→项目停摆 |
| TR2 | 报价格：「这颗物料月采购 10K vs 100K 价差 30%」 | 架构评估是否涨价/降规格/多买 | 设计时按小批量估价→量产时发现成本超预算 |

**核心结论**：供应链在 TR1 的输入决定了项目能不能做、多少钱做、什么时候能做。**供应链参与得越早，项目意外越少**。

### 10.4 交互管理三原则

#### 原则 1：提前拉通，别等「准备好了再沟通」

| 正确做法 | 错误做法 |
|:---------|:---------|
| 结构出初版 3D（即使只有外形骨架）就发给散热开始建模型 | 结构等「完全没干涉了」再发散热 |
| Layout 布完第一根关键链路就导给 SI 跑 | Layout 等「全布完了」再给 SI |
| 原理图 I2C 地址初稿就发给 BMC 确认 | 等原理图定稿了才发 BMC |
| 供应链在 TR1 就介入芯片选型讨论 | 等设计完了再问「芯片能买到吗」 |

#### 原则 2：变更必通知，别当「沉默的修改者」

任何领域决定变更（哪怕是微调），必须：
1. 评估影响范围（用 §5.6 变更模板）
2. 通知所有受影响领域
3. 受影响领域确认（或提出反对）
4. 有争议的走架构/PM 仲裁
5. 更新相关文档

**沉默修改的成本**：一个领域改了 1 件小事（改了一个 I2C 地址），没通知 BMC。联调时花了 1 周排查——7 天 vs 1 个通知。

#### 原则 3：在锁点之前「多吵」，在锁点之后「少动」

| 时间 | 交叉密度 | 决策方式 | 态度 |
|:-----|:---------|:---------|:-----|
| 结构锁之前 | 高 | 畅所欲言，鼓励争论 | **「这里不确定吗？那我们先跑一轮试试」** |
| 布局锁之前 | 中高 | 按结构锁后的约束修正 | **「这个问题现在不改，后面改代价 x3」** |
| 原理图锁之前 | 中 | 走评审+变更流程 | **「这个变更必须通知所有受影响领域」** |
| 原理图锁之后 | 低 | 严格 PCR | **「这个变更要算进度影响吗？还是推下版？」** |
| EVT 样机后 | 极低 | 只修 Bug | **「功能没问题就不动，性能问题放到下版」** |

**核心思想**：交互矩阵的价值不在于「列全了所有交互」，而在于 **让每个领域的工程师知道：我现在做的这个决定，会影响到谁、什么时候影响、我怎么和他对齐**。

### 12.17 认证与合规（新增领域，4个权衡点）

| # | 权衡点 | 活跃阶段 | 选项 A | vs 选项 B | 关键维度 | 举例与量化数据 |

| 1 | **EMI滤波方案** | TR3 | **全频段滤波(共模+差模+屏蔽)**——Class B余量8dB+，但+$15-25、PCB面积+5% | **最小滤波**——省成本省面积，但可能Class A不过、反复改版 | EMI余量vs成本 | 数据中心(Class A)→最小滤波+仿真+预扫。消费/边缘(Class B)+出口欧洲→全频段 |

| 4 | **认证顺序(目标市场决定)** | TR1 | **先中国(CCC/入网)**——国内上市最快，但出口需补CE/FCC/UL | **先全球兼容(同步做)**——一次设计多国，但认证12-16周、10-20万 | 时间vs覆盖 | 主攻国内→先CCC 6-8周3万。全球同步→同步认证，但设计需按全球兼容(EMI Class B、安全IEC 62368) |

| 1 | **固件签名强度** | TR3 | **强签名(ECDSA-384/SHA-384)**——抗量子前安全，但验证慢(500ms+启动) | **基本签名(RSA-2048/SHA-256)**——快(100ms)、兼容好，但安全性中等 | 安全vs性能 | 标准服务器→RSA-2048够。涉密/政务→ECDSA-384强制 |
