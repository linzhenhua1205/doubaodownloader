## 1. 方法论总纲

### 1.1 超节点分析的独特挑战

超节点（SuperPOD/SuperNode）作为一个由成百上千 GPU、多层互联拓扑、异构计算资源和复杂软件栈构成的巨型系统，其性能评估面临三个根本性挑战：

| 挑战 | 具体表现 | 传统方法为何失效 |
|:-----|:---------|:-----------------|
| **配置空间爆炸** | 模型规模、并行策略(TP/PP/DP/EP)、拓扑选择、调度策略、SLA 目标的组合数以万计 | 穷举实测不可行（45天只够跑一个案例的局部） |
| **非独立变量耦合** | 计算与通信重叠程度取决于内核粒度、流控与拓扑亲和性；显存可行域截断配置选择；故障恢复折扣长期 Goodput | 单一 benchmark 无法反映耦合效应 |
| **阶段异质性** | 训练有 Forward/Backward/Sync 三阶段差异；推理有 Prefill/Decode 资源瓶颈完全不同 | 平均化模型会把前沿拐点抹平 |

因此，超节点性能评估必须从"跑几个 benchmark 试试"升级为**系统化的边界比较框架**。

### 1.2 边界比较框架的总链路

本方法论遵循以下五步链路：

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  负载模型     │───▶│  系统模型     │───▶│  帕累托描边   │───▶│  校准验证     │───▶│  边界地图     │
│              │    │              │    │              │    │              │    │              │
│ 定义需求侧     │    │ 映射+编排推演│    │ 多目标空间中  │    │ 实测约束模型  │    │ 可比较可追踪 │
│ 的压力画像     │    │ 生成候选点   │    │ 画前沿       │    │ 收紧误差      │    │ 的方案族     │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
       │                    │                    │                    │
       │                    ▼                    │                    │
       │           ┌──────────────┐              │                    │
       └──────────▶│  资源模型     │◀─────────────┘                    │
                   │              │                                    │
                   │ 定义供给侧   │                                    │
                   │ 的供给边界   │◀───────────────────────────────────┘
                   └──────────────┘   （共演进闭环反馈）
```

**核心原则**：
1. **快速描边 + 关键点校准**：不用相同的保真度处理所有点
2. **硬件无关 + 策略中立**：负载模型不预先固化某种实现选择
3. **带条件输出**：每个结论附带适用范围、校准置信和前置条件

### 1.3 为什么不能用"跑几个 benchmark"替代

| 传统做法 | 问题 | 本框架的替代 |
|:---------|:-----|:-------------|
| 用 MLPerf 或单个 benchmark 代表全部负载 | 一个样例无法暴露边界拐点 | 参数化工作负载族，连续改变关键变量 |
| 用峰值 FLOPS/带宽衡量系统 | 忽略了同步放大、气泡率、重叠衰减 | 系统模型推演实际行为 |
| 跑几个配置点就下结论 | 可能恰好避开了关键跳变区 | 全空间扫描→帕累托前沿→关键点校准 |
| 把节能模式/故障恢复/碎片化当边缘条件 | 大规模下这些就是稳态的一部分 | RAS 和 Runtime 作为正式资源对象纳入 |

## 2. 负载模型：定义需求侧的压力画像

### 2.1 核心原则：不是一个 workload，而是一族 workload

如果只用单个代表性样例来描述负载，分析最多只能比较若干离散实验点，而无法**描述边界**。边界天然要求大量候选点，因此负载模型必须支持**参数化生成**：

- **训练侧**：连续改变模型规模、并行维度、同步频率、激活保留比例、MoE 路由分布
- **推理侧**：连续改变输入长度分布、输出长度分布、到达过程、并发度和 SLA
- **混合负载**：长短请求占比、多模型混部比例、冷热分布和突发性

只有这样，比较才会从"单点复现"升级为"边界塑形判断"。

### 2.2 取舍原则：保留什么、剥离什么

对超节点的边界判断而言，真正值得进入负载画像的，是那些**会改变边界形状和方案排序**的因素，而不是只改变局部实现常数的细节。

| 类别 | 保留（进入负载画像） | 剥离（不进入负载画像） | 判断依据 |
|:-----|:---------------------|:-----------------------|:---------|
| **计算结构** | 算子类型、shape、精度、层数、注意力模式 | 特定设备的 kernel 实现细节 | 前者决定计算/访存比与通信量级 |
| **通信意图** | 集合通信类型、消息大小、同步边界 | 运行时私有的 allocator/排队细节 | 前者定义负载对互联的需求 |
| **阶段差异** | Forward/Backward/Sync 边界、Prefill/Decode 分割 | 某次实验的偶然调度抖动 | 前者决定前沿拐点和阶段切换代价 |
| **状态生命周期** | KV Cache 尺寸演化、激活值峰值时序、梯度存活周期 | 固定 allocator 的碎片化行为 | 前者决定显存可行域边界 |
| **到达与 SLA** | 请求到达分布、并发度、TTFT/TPOT 约束 | 特定负载均衡器的路由实现 | 前者是推理侧边界的核心驱动 |
| **路由与稀疏性** | MoE 专家选择分布、top-k 策略、负载不均匀度 | 特定 MoE 框架的通信实现 | 前者决定 All-to-All 压力和前沿形变 |

**关键判断**：负载模型描述的是**需求**，而不是某条既定技术路线已经做出的**实现选择**。

### 2.3 静态图与动态执行迹的分工

两者并非彼此替代，而是分别服务于快速描边和关键约束：

| 表示形式 | 核心能力 | 适用阶段 | 典型工具/标准 |
|:---------|:---------|:---------|:--------------|
| **静态计算图** | 低成本暴露计算量、通信意图、关键路径 | 大范围配置扫描、前沿粗描 | Chakra ET、ONNX Graph、Alpa cost model |
| **动态执行迹** | 保留到达、切换、状态与尾部行为 | 关键点校准、跳变解释、极端验证 | Chakra Execution Trace、nsys trace、线上 trace |

**可切换性要求**：负载模型必须同时兼容两种使用方式：
1. 足够简化，以便和快速模型配合支撑数千点级的边界描绘
2. 足够贴近真实，以便在候选前沿点、敏感区回到真实 trace 校准

### 2.4 四类核心工作负载族

| 负载族 | 代表性场景 | 核心可变参数 | 对前沿的塑形作用 |
|:-------|:-----------|:-------------|:-----------------|
| **训练主线** | 稠密预训练、MoE 训练、长上下文训练、多模态训练 | 模型规模、TP/PP/EP/DP 维度、同步频率、激活保留比例、MoE 路由分布 | 决定训练侧前沿在同步放大、气泡率、显存和能效上的主形状 |
| **推理主线** | 短提示词交互、混合长短请求、长上下文生成、高并发在线服务 | 输入/输出长度分布、到达过程(Poisson/突发/trace)、并发度、SLA(TTFT/TPOT P99)、batch 策略 | 决定推理侧前沿在时延、吞吐、Goodput 和 KV 管理上的主形状 |
| **极端工况** | 超长序列(>128K)、专家负载极度偏斜、突发流量峰值、多模型混部干扰 | 序列长度上界、路由不均匀度、突发倍率、混部比例 | 不一定构成主前沿，但决定某条路线是否进入不可部署区域 |
| **未来变量** | 超稀疏 MoE(>256 专家)、多模态长交织、Agent 工作流、推理时计算 | 专家数量/激活比、模态交织密度、工具调用频率、自适应计算步数 | 为未来演进讨论提供边界变形的压力入口 |

### 2.5 训练侧关键塑形变量

训练系统不围绕单次延迟最优展开，而是围绕**稳态吞吐、同步放大和资源连续利用**展开。

| 塑形变量 | 具体机制 | 对前沿的效应 | 建模最低要求 |
|:---------|:---------|:-------------|:-------------|
| **同步边界放大** | 局部拖尾沿 DP/PP 同步路径传播 | Step time 被最慢 rank 决定，前沿向内收缩 | 迭代边界、全局同步点、拖尾放大因子 |
| **气泡与阶段失衡** | PP 阶段间计算量不均、warmup/cooldown 阶段闲置 | 气泡率 10-30% 常见，直接拉低 MFU | pipeline 调度时序、阶段计算量比 |
| **理论重叠 vs 实际重叠** | 计算-通信 overlap 受限于内核粒度和流控 | 理论 overlap 100% vs 实际 60-80% | 算子粒度、通信启动延迟、流依赖 |
| **多平面干扰** | 参数面(gradient sync)、数据面(activation)、控制面互扰 | 链路竞争导致通信长尾和抖动放大 | 流量矩阵、链路共享拓扑、优先级策略 |
| **显存可行域截断** | 激活、梯度、优化器状态峰值重叠 | 部分理论最优配置被显存不足直接排除 | 生命周期曲线、重计算/卸载交换代价 |
| **恢复路径** | 故障检测→隔离→checkpoint 回滚→回稳 | 长期 Goodput = 稳态吞吐 × 有效时间占比 | 检测延迟、回滚代价、MTTR、故障率 |

### 2.6 推理侧关键塑形变量

推理系统围绕**持续到达的请求流、阶段异质性和 SLA 约束**展开。

| 塑形变量 | 具体机制 | 对前沿的效应 | 建模最低要求 |
|:---------|:---------|:-------------|:-------------|
| **到达过程与调度入口** | 请求流形状直接影响排队状态；静态批/连续批/优先级批差异大 | 高并发短时长场景，调度器响应速度常常先于算子成为瓶颈 | 请求到达模型(Poisson λ/突发倍率/trace 回放速率) |
| **Prefill/Decode 阶段差异** | Prefill 接近计算上界，Decode 接近访存和状态管理上界 | 不拆分两阶段则无法解释某些架构平均吞吐合理但体验失败 | 阶段计算/访存差异、阶段切换成本 |
| **KV Cache 与显存状态** | 分页与碎片化、交换与驱逐、多模型混部下的状态干扰 | 决定"理论前沿"与"工程可达前沿"之间的距离 | 分页粒度、swap 阈值、驱逐策略 |
| **控制变量空间** | 合批策略、阶段映射策略(聚合/PD分离/异构PD)、并行度、限流 | 必须作为可扫描维度开放，不能固定 | 见下表 |

**推理系统可扫描的控制变量**：

| 控制变量 | 可扫描范围 | 对前沿的影响机制 |
|:---------|:-----------|:-----------------|
| 请求到达模型 | Poisson λ, 突发倍率, trace 回放速率 | 改变排队深度和调度器压力，直接影响 P95/P99 |
| 合批策略 | 静态 batch, 连续 batch, 最大 batch 上限 | 吞吐与时延的直接权衡杠杆 |
| 阶段映射策略 | 聚合, PD 分离, 异构 PD, chunk prefill | 决定 TTFT/TPOT 能否独立优化 |
| KV 管理策略 | 分页粒度, swap 阈值, 驱逐策略 | 决定长请求和混部场景下的显存效率 |
| 并行度 | TP, PP 度, 副本数 | 改变单请求路径效率 vs 系统级并行利用 |
| 限流与重平衡 | admission rate, 过载保护, 跨实例路由 | 决定高负载下 SLA 达标率和 Goodput |

## 3. 系统模型：映射与编排的推演引擎

系统模型回答的不是"这台系统理论上有多强"，而是更直接的问题：**给定一组工作负载、资源画像与策略选择，系统会在多目标空间中落到哪里，这个点为什么落在那里**。

系统模型做的事情只有两步：
1. **把逻辑负载映射到具体物理资源** → 决定候选点大致落在多目标空间的哪里
2. **把这些资源在时间轴上的竞争、重叠、排队、同步和恢复过程推演出来** → 决定这些点会不会因为抖动、拥塞、状态膨胀或恢复路径而发生显著偏移

### 3.1 供给侧资源对象与保真度分层

资源对象不应按"硬件类别"挑选，而应按**"是否会改变边界形状"**挑选。

#### 5 类核心资源对象

| 资源对象 | 理论上界参数 | 代价曲面（性能开始退化的机制） | 对前沿的塑形作用 |
|:---------|:-------------|:-----------------------------|:-----------------|
| **计算资源** | 峰值 FLOPS(FP16/FP8/INT8)、SM/CU 数量、功耗墙 | 精度降级损失、功耗限流、热降频、多租户争用 | 决定算子执行效率与单位能耗边界 |
| **存储资源** | HBM 容量与带宽、DDR 容量、NVMe 交换带宽 | 分页碎片化、缺页代价、HBM-DDR 交换长尾、驱逐抖动 | 决定显存可行域和 KV Cache 管理效率 |
| **互联资源** | 链路带宽(NVLink/PCIe/Ethernet)、端口数、拓扑 | 拥塞窗口收缩、多路径不均衡、重传代价、跨平面干扰 | 决定通信吞吐和同步放大程度 |
| **Runtime 资源** | 启动延迟、调度周期、内存分配开销 | kernel launch 累积、调度排队、allocator 碎片化 | 侵入关键路径时直接降低 MFU 和增大气泡率 |
| **RAS 资源** | 故障率(FIT)、检测延迟、隔离粒度、MTTR | 恢复路径引入的停机时间、checkpoint 写入开销、回稳抖动 | 决定长期 Goodput 与理论峰值之间的折扣系数 |

> **⚠️ 重要**：把 Runtime 和 RAS 视作资源而非附注，是超节点建模与传统设备评测的重要分界。

#### 必须纳入的系统级变量

| 系统级变量 | 影响路径 | 如果不纳入会怎样 |
|:-----------|:---------|:-----------------|
| 多平面网络组织(参数面/数据面/业务面) | 决定哪些流量会侵入主路径 | 低估通信长尾和跨平面干扰 |
| 统一调度(粒度、周期、重平衡延迟) | 决定理论资源何时变为可执行资源 | 高估资源利用率，低估调度开销 |
| 长期运行状态(故障率、碎片率、恢复时间) | 决定长期 Goodput 距离理论值多远 | 把稳态快照误判为长期表现 |
| 设施侧约束(机柜功率、液冷能力、PUE、建设周期) | 决定设计点是否可部署 | 把纸面最优当成工程可达 |

#### 资源保真度分层

| 保真度层级 | 数据来源 | 单点成本 | 适用阶段 |
|:-----------|:---------|:---------|:---------|
| **L1 解析层** | 公开规格、协议上界、Roofline 公式、TCO 参数 | 极低(毫秒级) | 全空间快速扫描，排除被支配区域 |
| **L2 画像层** | 微基准、子系统压测、collective benchmark、trace 拟合 | 中等(分钟级) | 甜点区识别、主要方案对比 |
| **L3 行为层** | 细粒度事件仿真、故障注入、控制面日志、拥塞演练 | 高(小时级) | 候选前沿点、跳变点、异常点校准 |

**原则：先足够便宜地覆盖大范围，再足够有针对性地收紧关键区域。**

### 3.2 系统模型的状态维护

系统模型不能只是一个"性能估算器"。它必须显式维护以下 5 类状态：

| 状态类别 | 代表性变量 | 为什么必须显式维护 |
|:---------|:-----------|:-------------------|
| **执行状态** | 算子占用时序、pipeline 阶段进度、micro-batch 调度槽位 | 不维护则无法解释气泡率和重叠效率 |
| **通信状态** | 链路占用率、Outstanding 消息数、拥塞窗口、多路径负载分布 | 不维护则无法解释同步放大和通信长尾 |
| **显存状态** | HBM 占用曲线、KV Cache 页表、激活值生命周期、swap 队列深度 | 不维护则显存可行域边界无从标定 |
| **调度状态** | batch 组成、请求排队深度、admission 阀值、优先级队列 | 不维护则推理侧吞吐-时延权衡无法建模 |
| **恢复状态** | 故障检测延迟、隔离粒度、checkpoint 回滚代价、回稳时间 | 不维护则 Goodput 只是理想稳态的乐观估计 |

### 3.3 训练前沿的塑形机制

训练前沿常常不是被峰值带宽或峰值算力直接决定，而是被结构性依赖、阶段失衡、理论重叠与实际重叠之间的差距，以及多平面干扰共同塑形。

以典型的 3D 并行训练为例，核心塑形链路：

```
同步边界放大 ──▶ 局部拖尾沿 DP/PP 同步路径传播 ──▶ 前沿向内收缩
       │
       ▼
气泡与阶段失衡 ──▶ PP 阶段间计算量不均 + warmup/cooldown ──▶ 气泡率 10-30%
       │
       ▼
重叠衰减 ──▶ 理论 overlap 100% vs 实际 60-80% ──▶ 拉大理论-可达差距
       │
       ▼
显存可行域截断 ──▶ 激活/梯度/优化器状态峰值重叠 ──▶ 部分配置被直接排除
       │
       ▼
恢复路径 ──▶ 故障检测→隔离→回滚→回稳 ──▶ 长期 Goodput 折扣
```

**同步放大因子的定量估算**：
- 假设 AllReduce 延迟函数为 `T_ar(M, N)`，其中 M 为消息大小，N 为参与节点数
- 若局部拖尾为 δ（某节点因热降频/链路退化多花的时间）
- 则 Step Time 放大 ≈ T_step + δ × sync_factor
- sync_factor 取决于拓扑（Fat-Tree vs Dragonfly）和同步频率

### 3.4 推理前沿的塑形机制

推理系统的建模核心不是估算单个 token 有多快，而是解释**请求流、调度器与状态变量如何共同决定 TTFT、TPOT 与系统级 Goodput**。

```
到达过程 ──▶ 请求排队深度 ──▶ 调度器压力
       │                      │
       ▼                      ▼
Prefill 阶段 ──▶ 计算上界 ──▶ TTFT 约束
       │
       ▼
KV Cache 管理 ──▶ 分页/驱逐/交换 ──▶ 显存可行域
       │
       ▼
Decode 阶段 ──▶ 访存上界 ──▶ TPOT 约束
       │
       ▼
阶段映射策略 ──▶ 聚合/PD分离/异构 ──▶ TTFT vs TPOT 权衡
```

### 3.5 三层结果输出：坐标、状态与归因

当训练侧和推理侧的建模都成立时，系统模型给出的不应该是几组分数，而应同时包含三层结果：

| 层次 | 内容 | 意义 |
|:-----|:-----|:------|
| **L1 坐标层** | 吞吐、时延、利用率、Goodput、能效、成本 | 直接放入多目标空间，支撑非支配集合计算和前沿描绘 |
| **L2 状态层** | 执行轨迹、队列状态、链路占用率、显存曲线、调度时序 | 校准验证时解释偏差来源，锁定模型需要修正的环节 |
| **L3 归因层** | 同步放大因子、重叠衰减率、瓶颈归因(计算/通信/显存/调度)、敏感性系数 | 解释边界为什么在某个位置、为什么会移动 |

三层之间不是简单的精度递进，而是服务于不同的分析角色：
- **L1** → 决策者
- **L2** → 校准工程师
- **L3** → 架构分析师

## 4. 延迟评估方案

### 4.1 延迟的全栈分解方法

超节点场景下的延迟，必须按**六层穿透**进行分解：

```
应用层延迟 ──▲  ├─ 调度排队延迟
              │  ├─ 序列化/反序列化
              │  
通信库延迟 ──┤  ├─ libibverbs/nccl 调用延迟
              │  ├─ Memory Registration 延迟
              │  
内核层延迟 ──┤  ├─ Doorbell MMIO (50-100ns)
              │  ├─ IRQ 处理/轮询切换
              │  
驱动层延迟 ──┤  ├─ PCIe 事务层处理
              │  ├─ DMA 描述符处理
              │  
传输层延迟 ──┤  ├─ 封包/解包
              │  ├─ 流控/拥塞控制等待
              │  ├─ 重传代价
              │  
物理层延迟 ──┘  ├─ SerDes 延迟
                 ├─ 光纤/电缆传播 (~5ns/m)
                 ├─ 交换机转发延迟
```

**关键洞察**：在超节点中，低层延迟可能存在数量级差异，但高层延迟（调度+排队）往往才是真正的瓶颈——尤其是推理侧。

### 4.2 超节点四大网络的延迟基线

| 网络 | 通信域 | 单向延迟(典型) | 关键决定因素 |
|:-----|:-------|:--------------|:-------------|
| **Scale-Up (NVLink)** | 节点内/超节点内 GPU↔GPU | 0.5-5μs | NVLink 4/5 的 flit 大小、NVSwitch 跳数、SHARP 内计算 |
| **Scale-Out (IB/RoCE)** | 节点间/机柜间 | 2-10μs | 拓扑跳数(Spine-Leaf 2-3跳 vs Dragonfly 1-2跳)、流控 |
| **存储网络** | GPU↔远端存储 | 10-100μs | NVMe-oF 开销、持久化写入确认、网络拥塞 |
| **管理网络** | BMC↔集中管理 | 100μs-1ms | 带外优先级低、轮询周期、不要求实时 |

### 4.3 端到端延迟预算分解法

延迟预算分解的核心思路：给定 SLA 约束，将总延迟预算按链路逐层拆解到每个子系统，验证各子系统是否在预算内。

**推理场景示例（Long 场景：TTFT ≤ 2500ms, TPOT ≤ 60ms）**：

```
TTFT 预算 2500ms 分解：
├── 网络输入延迟: ~50ms (请求传输 + 负载均衡)
├── 调度排队延迟: ~200-800ms (取决于到达率和并发度)
├── Prefill 计算: ~800-1500ms (取决于模型大小和输入长度)
├── 中间通信: ~50-200ms (AllReduce attention 等)
├── KV Cache 初始化: ~50-200ms
└── 预留余量: 10-20%

TPOT 预算 60ms 分解：
├── Decode 一次前向: ~20-40ms (模型越大越逼近上限)
├── Attention 通信: ~2-10ms
├── KV Cache 读写: ~1-5ms
├── 调度与启动: ~1-5ms
└── 预留余量: 10-20%
```

**训练场景示例（Step Time 预算）**：

```
Step Time 分解：
├── Forward 计算: ~40% total
│   ├── Embedding: 2-5%
│   ├── Attention 层: 15-25%
│   ├── FFN/MoE: 15-25%
│   └── 其他: 3-5%
├── Backward 计算: ~40% total
│   ├── Backward 激活: 30-35%
│   └── 梯度计算: 5-10%
├── 同步(AllReduce/Sync): ~10-15% total
│   ├── Gradient AllReduce: 5-10%
│   └── 同步等待(拖尾): 2-5%
├── 调度与启动: ~5-10%
└── 预留余量: 5%
```

### 4.4 训练场景的延迟评估：Step Time 分解

对于大型训练系统，Step Time 的完整计算公式：

```
T_step = T_fwd_max + T_bwd_max + T_sync_max + T_overlap_savings + T_schedule

其中：
  T_fwd_max   = max_over_all_ranks(T_fwd)      # 最慢 rank 的前向时间
  T_bwd_max   = max_over_all_ranks(T_bwd)      # 最慢 rank 的反向时间
  T_sync_max  = max(T_ar_grad, ...)            # 梯度同步最长通道路径的时间
  T_overlap   = 计算-通信重叠节省              # 理想省 20-40%，实际受限于核粒度
  T_schedule  = 调度器启动开销                   # kernel launch 累积
```

**关键评估点**：
1. **计算-通信重叠效率**：理想情况下 Backward 计算可与梯度通信完全重叠。但实际上：
   - 内核粒度不够时，通信端口需等待 compute kernel 释放 → 重叠率降至 60-80%
   - 多流依赖导致 pipeline stall → 需流依赖图分析
2. **拖尾放大效应**：1% 的局部性能偏离，经同步边界放大后可导致 3-5% 的 Step Time 损失
3. **显存生命周期冲突**：激活保存 + 梯度累积 + 优化器状态可能同时到达峰值

### 4.5 推理场景的延迟评估：TTFT + TPOT 分解

推理延迟必须区分两个阶段独立评估：

**TTFT（首 Token 生成时间）**：
```
TTFT = T_network_in + T_schedule_queue + T_prefill + T_sync_mid + T_kv_init

各分量估算：
- T_network_in: 请求从客户端到达负载均衡→调度器的时间
- T_schedule_queue: 在调度器中排队等待执行的时间（主要变体）
- T_prefill: 输入 prompt 的一次前向计算时间（计算密集型）
- T_sync_mid: Prefill 阶段的集合通信（如 Attention 跨 TP 通信）
- T_kv_init: 初始化 KV Cache 页表、填充首个 token 的时间
```

**TPOT（每 Token 生成时间）**：
```
TPOT = max(T_decode_comp, T_decode_mem) + T_kv_access + T_sync + T_schedule

各分量估算：
- T_decode_comp: Decode 的计算时间（通常远小于 Prefill，因只需 1 个 token）
- T_decode_mem: 内存带宽受限时间（KV Cache 读取 + 注意力计算）
- T_kv_access: KV Cache 的 HBM 读取延迟
- T_sync: TP 内的 AllReduce 通信（GQA 场景）
- T_schedule: kernel launch 调度开销
```

**PD 分离场景下的额外延迟**：
```
PD 分离额外延迟 = T_transfer + T_kv_cache_swap

- T_transfer: KV Cache 从 Prefill 实例传输到 Decode 实例的延迟
  - 不跨节点: 1-5μs (NVLink)
  - 跨节点: 5-50μs (IB/RoCE)
- T_kv_cache_swap: 交换到远端存储的延迟
  - HBM→DDR: 50-200μs
  - HBM→NVMe: 200μs-5ms
```

### 4.6 延迟尾部分析：长尾从哪来

推理场景的 SLA 通常约束的是 P95/P99，而非平均值。尾部延迟的主要来源：

| 来源 | 典型倍数(中位数 vs P99) | 根因 | 缓解方案 |
|:-----|:-----------------------|:-----|:---------|
| **调度排队** | 2-10× | 请求突发、admission control 不匹配 | 提前限流、请求优先级队列、预调度 |
| **传输重传** | 1.5-5× | 链路拥塞导致丢包/重传 | DCQCN 调优、优先级流控、多路径 |
| **显存缺页** | 3-20× | KV Cache 超出 HBM 容量触发 swap | 提前换出、分页粒度优化、驱逐策略 |
| **功耗限流** | 1.2-2× | 持续高负载触发热降频 | 动态功率预算、冷却规划 |
| **热迁移/故障切换** | 100-1000× | GPU 故障/节点故障触发恢复 | 预置热备、快速切换、任务迁移 |

**尾部延迟分解方法论**：
1. **全链路追踪**：用 nsys/perf 等工具记录每条请求的完整延迟瀑布图
2. **分位点对比**：对比 P50 vs P95 的延迟瀑布，锁定哪个环节被放大
3. **聚类分析**：将长尾请求按输入长度/排队时间/重传次数聚类，找共性
4. **根因归因**：确认是调度问题、网络问题还是显存问题

## 5. 校准验证与共演进闭环

### 5.1 为什么只能校准关键点，而不是重测全空间

建模仿真的职责是快速描边（数千点级），而评测和实验的职责只有一个：**为模型提供关键参数，并在关键区域校准模型误差**。

核心原则：**用实测约束模型，而不是用实测替代模型。**

真正值得投入实测的，通常只有三类区域：
1. **候选前沿点** — 帕累托前沿上的候选方案，方案排序的最终依据
2. **边界跳变点** — 结果发生剧烈变化的位置，需要解释为什么
3. **敏感区** — 靠近 SLA、显存上限、功耗边界或稳定性阈值的区域

### 5.2 三层校准层次

| 校准层次 | 校准对象 | 典型方法 | 产出 |
|:---------|:---------|:---------|:------|
| **原子级** | GEMM 吞吐、Attention kernel 耗时、P2P RTT、HBM/DDR 带宽 | 微基准(ncu, nsys, ib_write_lat) | 资源模型基础参数的置信锚点 |
| **子系统级** | AllReduce/AllToAll 耗时曲线、Paged KV 访问延迟、checkpoint 写入/恢复代价、swap 吞吐 | collective benchmark、子系统压测、trace 拟合 | 前沿甜点区和拐点的局部误差收紧 |
| **系统级** | step time、TTFT/TPOT P95/P99、tokens/s/GPU、Goodput、能效 | 端到端训练/推理 benchmark、线上 A/B trace | 整体解释力和排序结论的最终验证 |

**校准完成后的最重要产出**不是"模型已经正确"，而是一组置信边界：
- 哪些参数来自原子级实测
- 哪些现象已经被子系统级曲线收紧
- 哪些前沿点已被系统级结果验证
- 哪些区域仍只能作为受约束的推断

### 5.3 从散点到边界地图：量化卡设计

散落的帕累托点如果不被组织起来，就无法支撑方案比较和演进追踪。每张量化卡需要回答同一组问题：

| 量化卡字段 | 内容说明 |
|:-----------|:---------|
| **适用负载画像** | 该方案更适合什么类型的训练/推理负载 |
| **多目标定位** | 在规模、时延、功耗、内存、软件复杂度上位于边界的什么位置 |
| **理论 vs 可达差距** | 理论前沿与工程可达前沿之间的 gap 及主要归因 |
| **敏感度分析** | 负载、软件栈、拓扑变化时边界会怎样移动 |
| **前置条件** | 依赖的控制面能力、软件成熟度和器件条件 |
| **校准置信** | 哪些区域已被实测校准、哪些仍为模型推断 |

### 5.4 边界变化的持续更新机制

帕累托前沿不是静态的。真正的价值在于追踪边界如何随以下因素变化：

| 变化来源 | 典型时间尺度 | 对前沿的影响 | 分析方法的应对 |
|:---------|:-------------|:-------------|:---------------|
| **负载变化** | 周-月 | 新模型架构改变压力分布(MoE 占比上升、推理时计算增长) | 更新负载画像族，重新描边 |
| **软件迭代** | 周-季 | 同硬件上前沿被推出(通信库优化、调度策略升级) | 重新校准关键点，更新可达前沿 |
| **硬件代际** | 年 | 资源上界整体抬升，前沿在多维度外推 | 更新资源模型，重新生成全空间 |
| **运维与控制面** | 持续 | 长期 Goodput 折扣变化(故障率、恢复速度、碎片治理) | 更新 RAS 参数，修正 Goodput 折扣 |

这才是**共演进闭环**的真正含义：不是一次性描边，而是持续追踪边界位移。

## 6. 方法论落地实战：异构推理架构的帕累托分析

以下为 DeepLink 白皮书中的真实案例。方法只有真正跑通一次，才具备说服力。

### 6.1 案例定义：配置空间与 SLA 约束

**核心问题**：在给定 SLA 约束下，纯 A100 架构与 L40S+A100 异构分离架构之间，哪种方案更接近可部署前沿？

**候选架构**：

| 架构 | 类型 | 说明 |
|:-----|:-----|:------|
| A100_agg | 纯 A100 聚合架构 | Prefill 与 Decode 共享 GPU 资源 |
| A100_disagg | 纯 A100 分离架构 | Prefill 和 Decode 使用独立 GPU 池 |
| L40_P_A100_D | 异构混推 | L40S 承担 Prefill，A100 承担 Decode |
| A100_P_L40_D | 反向异构(对照) | L40S 承担 Decode，A100 承担 Prefill |

**请求场景与 SLA 约束**：

| 场景 | 输入/输出长度 | TTFT SLA | TPOT SLA |
|:---------|:------------|:---------|:---------|
| Short | 512 / 128 | 600 ms | 20 ms |
| Medium | 2048 / 512 | 1500 ms | 40 ms |
| Long | 4000 / 1000 | 2500 ms | 60 ms |

**评估指标**（三个指标构成帕累托问题）：
- **tokens/s/gpu** — 单位 GPU 吞吐，反映资源利用效率
- **tokens/s/user** — 用户感知生成速度，反映端到端体验
- **tokens/10k_rmb** — 单位成本产出，反映成本效益

### 6.2 资源模型：阶段瓶颈与硬件匹配

异构方案成立的前提是 Prefill 和 Decode 的资源瓶颈不同：

| 推理阶段 | 资源瓶颈 | L40S | A100 | 匹配结论 |
|:---------|:---------|:-----|:-----|:---------|
| **Prefill** | 计算密集型 | 362 TFLOPS (FP16) | 312 TFLOPS (FP16) | L40S 更适合 |
| **Decode** | 内存带宽密集型 | 864 GB/s | 2,039 TB/s | A100 更适合 |
| **成本** | — | ~1.5万/月 | ~3.5万/月 | L40S 性价比更高 |

**配置差异**（Long 场景）：

| 架构 | Prefill 配置 | Decode 配置 |
|:-----|:------------|:------------|
| 纯 A100 Disagg | TP=4, workers=2 | TP=8, workers=1 |
| L40_P_A100_D | TP=2, workers=5 | TP=2, workers=3 |

异构方案不仅硬件匹配更合理，更低的 TP 度允许更多 worker，提高了系统级并行利用率。

### 6.3 帕累托描边与核心结论

| 场景 | 推荐架构 | tokens/s/gpu | tokens/s/user | tokens/10k_rmb | 关键结论 |
|:-----|:---------|:------------|:-------------|:--------------|:---------|
| Short (512/128) | L40_P_A100_D | 210.1 | 72.99 | 672.31 | 用户体验最优，成本基本持平 |
| Medium (2048/512) | L40_P_A100_D | 280.08 | 25.78 | 689.43 | 成本效益提升约 31% |
| Long (4000/1000) | L40_P_A100_D | 243.4 | 19.58 | 599.13 | 成本效益提升约 45% |

> ⚠️ **结论带条件**：成立于给定请求分布和 SLA 目标下，本质上是"带约束的边界判断"。

### 6.4 校准验证：安全边际分析

帕累托最优点并不自动等于可上线点。引入**安全边际**作为稳健性度量：

```
安全边际 = (SLA 约束 - 仿真达成) / SLA 约束
```

| 场景 | SLA (TTFT/TPOT) | 仿真达成 | 安全边际(TTFT/TPOT) | 稳健性判断 |
|:-----|:---------------|:---------|:-------------------|:-----------|
| Short | 600ms/20ms | 246.7ms/13.7ms | 59%/32% | **高** |
| Medium | 1500ms/40ms | 925.6ms/38.8ms | 38%/3% | **中，TPOT 边界较紧** |
| Long | 2500ms/60ms | 1908.8ms/51.1ms | 24%/15% | **中，整体可行但余量有限** |

**关键洞察**：Medium 场景 TPOT 仅剩 3% 余量 → 对流量波动和实现偏差更敏感 → 需要额外缓冲或 admission control。

### 6.5 敏感性分析：成本、规模与反向验证

#### 成本敏感性

混推方案的成本优势依赖于 L40S 的价格区间：
- L40S 月成本 < 约 2.5 万 → L40_P_A100_D 保持明显优势
- L40S 价格继续上升 → 优势收敛，可能被纯 A100 反超

#### 规模扩展

| GPU 规模 | tokens/s/gpu | 总吞吐 tokens/s | tokens/10k_rmb | TTFT(ms) | TPOT(ms) |
|:---------|:------------|:---------------|:--------------|:---------|:---------|
| 8 GPUs | 252.54 | 2,020.29 | 404.06 | 925.61 | **49.27** |
| 16 GPUs | 280.08 | 4,481.28 | 689.43 | 925.61 | **38.79** |
| 32 GPUs | 308.09 | 9,858.82 | 857.29 | 925.61 | **49.27** |

**关键洞察**：总吞吐近似线性增长，但 **TPOT 并未随规模单调改善**——16 GPUs 时最优(38.79ms)，32 GPUs 反而回升到 49.27ms。说明规模扩展的主要收益在系统级吞吐，单请求时延需结合具体 TP/worker 组合单独验证。

#### 反向架构验证

A100_P_L40_D（反向异构）在多个场景中明显更差——证实了结论具有机理基础：不是"只要异构就更优"，而是**分工对了，异构才更优**。

### 6.6 方法论闭环验证

| 方法论环节 | 案例中的落地 | 验证了什么 |
|:-----------|:-------------|:-----------|
| **负载模型** | 3 类请求场景 × 参数化长度分布 | 不同负载确实把系统推向不同受压方向 |
| **资源模型** | L40S vs A100 的算力/带宽/成本三维画像 | 阶段瓶颈匹配是异构方案成立的前提 |
| **系统模型** | 4 架构 × 多并发 × 多配置的帕累托描边 | 2700+ 配置点无法穷举实测，分层描边可行 |
| **校准验证** | 安全边际公式 + 分场景稳健性判断 | Medium 场景 3% 余量——不校准会出问题 |
| **敏感性分析** | 价格敏感性 + 规模扩展 + 反向架构 | 帕累托前沿不是静态的 |

## 7. 分析工具链与实操指引

### 7.1 各环节推荐工具

| 环节 | 推荐工具 | 用途 |
|:-----|:---------|:------|
| **负载建模** | Chakra ET / ONNX Graph / Alpa cost model | 静态图生成、计算量估算 |
| **资源建模** | Roofline 分析、自定义 Python 脚本 | L1/L2 快速扫描 |
| **系统建模** | nsys / ncu / nsight compute | 微基准、kernel profiling |
| **通信分析** | ib_write_lat/ib_write_bw / nccl-tests | P2P RTT、AllReduce 曲线 |
| **延迟分解** | nsys timeline / perf / trace 分析 | 逐层延迟分解 |
| **帕累托描边** | 自定义扫描脚本(>1000 点级) | 全空间候选点生成 |
| **校准验证** | 端到端训练/推理 benchmark | 前沿点校准 |
| **敏感性分析** | 参数扫描 + 热力图 | 成本/规模/负载变化影响 |

### 7.2 典型工作流程

```
Step 1: 定义负载画像
  选择负载族 → 设定参数范围 → 生成参数化 workload 族

Step 2: 建立资源模型
  收集 L1 层参数(公开规格) → 可选 L2 层校准(微基准) → 建立代价曲面

Step 3: 系统模型扫描
  定义配置空间(架构×并行度×策略) → 运行快速扫描 → 输出候选点集

Step 4: 帕累托描边
  多目标排序 → 识别非支配解 → 绘制前沿 → 标注跳变区

Step 5: 关键点校准
  选择候选前沿点 → 子系统级验证 → 系统级验证 → 计算安全边际

Step 6: 敏感性分析
  成本变化 → 规模扩展 → 负载偏移 → 确认结论稳健性

Step 7: 输出边界地图
  量化卡 → 变化假设集 → 置信区间标注
```

### 7.3 常见陷阱与对策

| # | 陷阱 | 症状 | 对策 |
|:-:|:-----|:-----|:------|
| 1 | **平均化模型** | 把 Prefill/Decode 合并估算，导致 TTFT 和 TPOT 被低估/高估 | 严格区分两阶段建模 |
| 2 | **忽略调度排队** | 只算算子执行时间，推理时延比线上低 2-10× | 到达过程 + 排队模型必须纳入 |
| 3 | **静态/稳态假设** | 忽略故障恢复、碎片化、热降频的长期影响 | RAS 作为正式资源对象，Goodput 替代吞吐 |
| 4 | **单点结论泛化** | 一个场景的结论被当作普适真理 | 多场景 + 敏感性分析验证范围 |
| 5 | **校准不足** | 只用 L1 层数据就下结论 | 严格按三层校准路径执行 |
| 6 | **忽略跨平面干扰** | 只算参数面流量，低估通信长尾 | 确认多平面共享拓扑，纳入干扰模型 |
| 7 | **保真度混用** | 所有点用相同保真度处理，成本与精度不匹配 | 严格 L1→L2→L3 分层 |

### A. 关键术语表

| 术语 | 定义 |
|:-----|:------|
| **负载模型** | 定义系统承受的压力画像，以参数化工作负载族形式描述需求侧 |
| **资源模型** | 定义供给侧的能力边界，包括计算/存储/互联/Runtime/RAS 五类资源 |
| **系统模型** | 把负载需求映射到物理资源，推演系统行为的边界生成引擎 |
| **帕累托前沿** | 多目标空间中无法在所有维度上改善的非支配解集 |
| **描边** | 用低成本方法快速生成全空间候选点集，识别前沿位置 |
| **校准** | 用实测数据约束模型，收紧关键区域的误差 |
| **安全边际** | (SLA约束 - 仿真达成) / SLA约束，衡量方案稳健性 |
| **同步放大** | 局部拖尾沿同步边界传播导致整体 Step Time 损失 |
| **TTFT** | Time To First Token，推理首 token 生成延迟 |
| **TPOT** | Time Per Output Token，推理每 token 生成延迟 |
| **Goodput** | 有效吞吐，扣除故障恢复时间、碎片化损失后的净产出 |
| **PD 分离** | Prefill-Decode 分离架构，独立资源池处理不同阶段 |
| **共演进闭环** | 负载/软件/硬件/运维变化时持续更新边界地图的机制 |
