## 1. ODS 背景与定位

### 1.1 基本信息

| 维度 | 详情 |
|:-----|:------|
| **全称** | ODS — Open Disaggregated Supernode（开放解构超节点） |
| **性质** | 系统架构级技术规范 + 白皮书 |
| **牵头** | 中国移动（运营商立场，代表超大规模部署方的需求） |
| **联合编制** | 英特尔（CPU/互联芯片）、云尖信息（ODM 制造） |
| **核心参与** | 浪潮、曙光、超聚变、联想、宁畅、英业达、广达等 |
| **标准状态** | ✅ 白皮书 1.0 已发布 · ✅ 技术规范深化定稿 |

### 1.2 战略定位

ODS 是**国产 AI 基础设施在架构层面的独立宣言**。在 NVIDIA 定义了 NVLink 域（NVL72/NVL576）的封闭架构之后，ODS 对标实现**功能等效但架构开放**的超节点方案：

> **目标**: 用开放的、解耦的方式实现类似 NVIDIA NVLink 域的超节点级计算能力，但保持供应商无关性和供应链弹性。

### 1.3 ODS vs 其他超节点路线的定位差异

| 路线 | 代表 | 架构哲学 | 适用场景 |
|:-----|:------|:---------|:---------|
| **ODS（开放解构）** | 中国移动/Intel/ODM | 计算/互联/存储/管理物理分离，开放接口 | 推理/混合部署/国产化/运营商 |
| **NVLink 域** | NVIDIA GB200 NVL72 | 一体化集成，全栈绑定 | 极致训练性能 |
| **华为昇腾超节点** | Atlas 900/950 | 全栈自研，HCCS 私有不开放 | 国产化首选（有自主可控要求） |

## 2. ODS 核心架构 — 四层模型

```
ODS Architecture (bottom-up):
+--------------------------------------------------+
|  Application Layer                                  |
|  Training / Inference / Distributed Frameworks       |
|  (PyTorch / Megatron-LM / vLLM / Triton)            |
+--------------------------------------------------+
|  Orchestration Layer                                |
|  K8s / Slurm / Unified Scheduler                    |
|  Heterogeneous Resource Scheduling (XPU-aware)      |
+--------------------------------------------------+
|  Fabric Layer (Interconnect)                        |
|  Scale-Up:  UALink / CXL  (域内高带宽互联)          |
|  Scale-Out: RoCE / UEC   (域间集群扩展)            |
|  Mgmt:      Redfish / OpenBMC / NC-SI              |
+--------------------------------------------------+
|  Hardware Layer                                     |
|  ┌──────┬────────────┬───────────┬──────────┐     |
|  │ XPU  │ Interconnect│ Storage   │ Mgmt     │     |
|  │Module │ Module     │ Module    │ Module   │     |
|  │(OAM/ │ (Switch/   │ (NVMe/    │ (BMC/    │     |
|  │ UBB) │ Retimer/   │ CXL Pool) │ TPM/     │     |
|  │      │ Optical)   │           │ DSDB)    │     |
|  └──────┴────────────┴───────────┴──────────┘     |
+--------------------------------------------------+
```

### 2.1 各层职责

| 层次 | 职责 | 标准接口 | 关键特性 |
|:-----|:-----|:---------|:---------|
| **硬件层** | 物理计算/互联/存储/管理单元 | OAM/UBB · PCIe · CXL · NC-SI | 解耦设计，各模块独立演进 |
| **Fabric 层** | 域内 Scale-Up + 域间 Scale-Out | UALink 2.0 · CXL 4.0 · UEC/RoCE | 开放协议，非私有锁定 |
| **编排层** | 资源调度 + 任务分配 | K8s · Slurm · 统一调度器 | 支持异构 XPU 协同调度 |
| **应用层** | 训练/推理/分布式框架 | PyTorch · Megatron · Triton · vLLM | 框架无关，标准通信库 |

### 2.2 六大解构单元

| 解构单元 | 物理形态 | 互联要求 | 管理规范 |
|:---------|:---------|:---------|:---------|
| **计算单元 (XPU Module)** | OAM/UBB 兼容模组 | Scale-Up: UALink / CXL | Redfish + OpenBMC |
| **互联单元 (Switch Tray)** | 102.4T 交换机模组 | Scale-Out: UEC / RoCE | NVSwitch Mgmt |
| **存储单元 (Storage Tray)** | NVMe JBOF / CXL Memory Pool | PCIe 6.0/7.0 | 存储管理 API |
| **管理单元 (Mgmt Module)** | BMC 子卡 | OOB: NC-SI over 1GbE | IPMI + Redfish |
| **供电单元 (Power Tray)** | 800V HVDC BBU + 54V IBC | I2C / PMBus | 供电遥测 |
| **散热单元 (Cooling Tray)** | 液冷 CDU / 冷板组件 | — | 冷却液管理 |

## 3. 已定义边界：大禹整机柜参考框架

### 3.1 整机柜物理架构

| 维度 | ODS 标准定义 | 说明 |
|:-----|:------------|:-----|
| **机柜宽度** | 21 英寸（兼容 19 英寸） | 适应超节点内部大带宽互联线缆管理 |
| **机柜高度** | 42U ~ 52U（可扩展） | 适应液冷管路和盲插设计 |
| **节点形态** | 模组化 tray 设计 | XPU Tray / Switch Tray / Storage Tray 独立抽取 |
| **节点高度** | 4U ~ 8U per compute tray (按功耗) | 高功耗 XPU 需更大散热空间 |
| **盲插设计** | 供电/信号/液冷 Quick Disconnect | 支持免工具热插拔维护 |
| **前后风道** | 前→后（液冷为主，辅以风冷） | 液冷覆盖 >70% 热负荷后剩余风冷 |

### 3.2 液冷支持方案

| 维度 | ODS 标准方案 | 可选项 |
|:-----|:------------|:-------|
| **散热方式** | 冷板式直接液冷 (DLC) | 可扩展两相冷板（700W+ 场景） |
| **液冷覆盖率** | >70% 热负荷（CPU/GPU 全覆盖） | 可扩展到 >95%（含内存/SSD） |
| **冷却液** | 单相纯水 / 乙二醇溶液 | 两相氟化液（HFO 系列，GWP<150） |
| **CDU 部署** | 机柜内置 CDU | 行级 CDU / 列级 CDU |
| **盲插接口** | Quick Disconnect 统一规格 | 标准化中（液冷焦点组推进） |
| **入水温度** | 25°C ~ 40°C | 按芯片功耗柔性调节 |
| **漏液检测** | 电阻式 + 光纤光栅双冗余 | 覆盖冷板/管路/CDU 全链路 |

#### 3.2.1 冷却液选择矩阵

| 工质 | 类型 | 导热系数 W/(m·K) | GWP | 适用场景 | 成本 |
|:----|:----|:----------------:|:---:|:--------|:----:|
| 纯水 | 单相 | 0.60 | 0 | 冷板式（低功耗~700W） | ~0.001 元/L |
| 乙二醇溶液 25% | 单相 | 0.43 | 0 | 冷板式（中功耗） | ~10-20 元/L |
| HFO-1336mzz(Z) | 两相 | 0.07 | 2 | 两相冷板 / 浸没（>700W） | ~300-800 元/L |
| R-1234ze(E) | 两相 | 0.06 | 1 | 两相冷板替代方案 | 待定 |

### 3.3 供电架构

```
Grid 380V AC / 800V DC
        │
  [800V HVDC BBU / SSCB 三级断路器]
        │
  [54V IBC (Intermediate Bus Converter)]
        │
  ┌─────┴─────┐
  │            │
54V→12V POL  54V→0.8V VR (XPU)
(存储/管理)    (计算)
```

| 层级 | 电压 | 覆盖范围 | 标准进展 |
|:-----|:-----|:---------|:--------|
| **Grid → DC** | 380V AC / 800V DC | 整机柜输入 | 800V HVDC 草案讨论中 |
| **DC Bus** | 48V / 54V | 整机柜母线 | ODS 标准化中 |
| **IBC → VR** | 54V → 0.8V | XPU 核心供电 | 各厂商自定 |
| **IBC → POL** | 54V → 12V / 5V | 存储/管理/IO | 标准 PMBus |

### 3.4 散热与热管理

| 维度 | 参数 |
|:-----|:-----|
| **单机柜典型功耗** | 60 ~ 120 kW（按 XPU 数量和型号） |
| **液冷系统 PUE** | ≤ 1.10（全液冷，无风扇） |
| **降噪效果** | 75-85 dBA → 50-55 dBA（降噪 80%） |
| **芯片热流密度** | 100-200 W/cm²（下一代 XPU） |
| **两相冷板热阻** | ~0.01 °C·cm²/W（单相 0.1） |

### 3.5 互联拓扑

#### 3.5.1 Scale-Up 域内互联（机柜内部）

```
                   ┌─────────────────┐
                   │  Switch Tray    │
                   │  (UALink/CXL)   │
                   └──┬──┬──┬──┬───┘
     ┌──────────── ────┼──┼──┼──┼────────────┐
     │            │    │  │  │  │            │
     ▼            ▼    ▼  ▼  ▼  ▼            ▼
  ┌──────┐   ┌──────┐              ┌──────┐
  │XPU-1 │   │XPU-2 │    ...       │XPU-N │
  │OAM   │   │OAM   │              │OAM   │
  └──────┘   └──────┘              └──────┘
```

| 互联层 | 协议 | 单链路带宽 | 拓扑 |
|:-------|:-----|:---------:|:-----|
| XPU↔XPU (域内) | UALink 2.0 | 180 GB/s | Switch Fat-Tree / Dragonfly |
| XPU↔内存池 | CXL 4.0 | 128 GB/s (128 GT/s) | 内存池化 Fat-Tree |
| XPU↔存储 | PCIe 6.0 | 64 GT/s per lane | NVMe 直连 / JBOF |
| 管理 | NC-SI / Redfish | 1GbE | OOB 独立管理网 |

#### 3.5.2 Scale-Out 域间互联（跨机柜集群）

| 层级 | 协议 | 带宽 | 拓扑 |
|:-----|:-----|:----|:-----|
| 机柜间 EP | UEC (LLR/CBCF/MRC) | 400G ~ 800G per port | Leaf-Spine |
| 机柜间 DP | RoCEv2 | 400G per port | Fat-Tree |
| 存储网络 | NVMe-oF / RoCE | 400G per port | 独立存储平面 |
| 集群管理 | TCP/IP | 10G~25G | 带外管理 |

## 4. 可扩展空间一：单板/模组定义

ODS 中**并未锁定具体的单板形态**，而是通过 OAM/UBB 接口规范提供最大的灵活性：

### 4.1 XPU 模组（OAM 兼容）

| 维度 | ODS 定义 | 可扩展空间 |
|:-----|:---------|:-----------|
| **加速器接口** | OAM V2.0 兼容 | 可接入任意厂商 GPU/NPU（NVIDIA/AMD/华为/沐曦/燧原/寒武纪） |
| **互联接口** | UALink / CXL / PCIe | 可随 UALink 代际升级（1.0→2.0→3.0） |
| **功耗上限** | 600W ~ 1600W per XPU | 可适配未来 2kW+ XPU（需配合两相冷板） |
| **模组尺寸** | OAM 标准尺寸 | 可扩展半宽/全宽模组 |
| **供电针脚** | 12VHPWR / 54V 辅助 | 可定制高功耗针脚定义 |

### 4.2 通用底板（UBB）

| 维度 | ODS 定义 | 可扩展空间 |
|:-----|:---------|:-----------|
| **BaseBoard 尺寸** | 开放（ODS 未锁定尺寸） | 可适配 4/8/16 XPU 的不同密度需求 |
| **互联拓扑** | Switch-based (可配置) | 支持 Ring/Mesh/Fat-Tree 多种拓扑 |
| **冷却接口** | 冷板 Quick Disconnect | 可按 XPU 功耗选单相/两相冷板 |
| **传感器集成** | IPMI/SMBus 标准 | 可扩展智能传感器/遥测 |

### 4.3 管理子卡（BMC Module）

| 维度 | ODS 定义 | 可扩展空间 |
|:-----|:---------|:-----------|
| **管理协议** | Redfish + OpenBMC | 可扩展 PMCI/OEM 扩展 |
| **带外通信** | NC-SI over 1GbE | 可升级 2.5GbE/5GbE |
| **安全模块** | TPM 2.0 | 可扩展 SPDM/安全启动/固件验证 |

## 5. 可扩展空间二：外部互联

ODS 中互联协议**不绑定单一标准**，而是构建层次化的互联选择体系：

### 5.1 HBD 互联三档分级（ODCC 网络工作组定义）

| 等级 | 带宽 | 适用场景 | 物理层 | 技术成熟度 |
|:-----|:----|:---------|:-------|:----------|
| **HBD-1** | 100-200 GB/s 每方向 | TP 并行（域内高带宽需求） | 224G SerDes x 8-16 lanes | 🟢 可行（2025+） |
| **HBD-2** | 400-800 GB/s | MoE All-to-All（域内大吞吐） | NPO 光互联 / 224G x 32 | 🟡 实验性（2026-2027） |
| **HBD-3** | >1 TB/s | 全显存池化（未来需求） | CPO / 硅光集成 | 🔴 概念（2028+） |

### 5.2 UALink 路线（Scale-Up 主选）

| 代际 | 带宽 | 状态 | 兼容性 |
|:-----|:----|:-----|:-------|
| UALink 1.0 | ~180 GB/s | ✅ 已发布（2025.4） | 基础互联 |
| UALink 2.0 | ~360 GB/s | ✅ 已发布（2026.4），苹果加入 | In-Network Compute |
| UALink 3.0 | 待定 | 📋 规划中 | 与 CXL 内存语义融合 |

### 5.3 CXL 路线（内存池化/一致性互联）

| 代际 | 带宽 | 状态 | 扩展空间 |
|:-----|:----|:-----|:---------|
| CXL 3.0 | 64 GT/s | ✅ 已发布 | 内存池化 + 交换机互联 |
| **CXL 4.0** | **128 GT/s** | ✅ **2026.6 正式发布** | Bundled Ports · 增强 RAS |
| CXL 未来 | 256 GT/s⁺ | 路线图中 | 完全内存语义 + CXL Switch |

### 5.4 UEC 路线（Scale-Out 主选）

| 规范 | 功能 | 对标 NVIDIA 技术 |
|:-----|:-----|:----------------|
| **LLR** (Low Latency Routing) | 低时延逐跳转发 | InfiniBand 确定性路由 |
| **CBCF** (Cluster Buffer & Congestion Flow) | 集群拥塞控制 | DCQCN 增强版 |
| **MRC** (Multi-path Reliable Connect) | 多路径包级负载均衡 | 替代 ECMP，MoE 场景提升 20-40% |

### 5.5 光互联扩展（可选路线）

| 技术 | 优势 | 成熟度 | ODS 可集成性 |
|:-----|:-----|:-------|:------------|
| **224G NPO** | 功耗降 3x，密度升 5-10x | 🟡 白皮书输出中 | ✅ ODS UBB 预留光口 |
| **OCS 光电路交换** | 微秒级拓扑重配置 | 🔴 新立项 | ✅ ODS 互联层可插拔 |
| **448G SerDes** | 面向 1.6T/3.2T 光模块 | 🔴 新立项 | ✅ 未来兼容 |

## 6. 可扩展空间三：软件栈

### 6.2 管理栈

|:-----|:---------|:----------|:-----------|

| 硬件监控 | Redfish | IPMI / SNMP / Prometheus exporter | 标准 API |
| 集群调度 | K8s + Slurm | Univa Grid Engine / PBS | 支持混合工作负载 |

| 安全框架 | SPDM / TPM | SGX / SEV / CCA | 可信计算 |

### 6.3 国产化软件栈替代路线

| 层次 | NVIDIA 方案 | ODS 方案 | 国产替代候选 |
|:-----|:----------|:--------|:-----------|
| GPU 编程 | CUDA | Triton (OpenAI) | 华为 CANN / 百度 Paddle |
| 集合通信 | NCCL | RCCL / UCC | 华为 HCCL |
| 推理优化 | TensorRT | Triton Inference Server | 华为 MindIE |

## 7. ODS vs NVIDIA NVLink 域深度对标

| 对比项 | NVIDIA NVLink 域 | ODS 开放解构超节点 |
|:-------|:----------------|:-----------------|
| **Scale-Up 互联** | NVLink 私有协议 | UALink / CXL（开放标准） |
| **Scale-Out 互联** | InfiniBand / 以太网 | 标准 RoCE / UEC |
| **GPU 选择** | 仅 NVIDIA | 任意厂商 GPU/NPU |
| **交换机** | NVSwitch（绑定 GPU） | 独立交换机（可解耦演进） |
| **管理面** | NVAIE（私有） | OpenBMC / Redfish（开放） |
| **部署规模** | 固定倍数（72/576） | 弹性（几十卡~上万卡） |

### 7.1 带宽差距 vs 灵活性优势

| 维度 | NVLink 5.0 | UALink 2.0 | 差距 | 影响 |
|:-----|:----------:|:----------:|:----:|:-----|
| 单 GPU 互联带宽 | 900 GB/s | 180 GB/s | 1:5 | 训练 TP 效率差距显著 |
| 域内总带宽 | ~64 TB/s (72 GPU) | ~12.8 TB/s (72 XPU) | 1:5 | 集合通信吞吐差距 |
| 跨域带宽 | NDR800 (100 GB/s) | 400G UEC (50 GB/s) | 1:2 | 可接受差距 |
| 软件生态成熟度 | CUDA 20+ 年 | PyTorch/Triton 10 年 | 差距大 | 长期追赶中 |

## 8. ODS vs NVIDIA CMX/GB200/GB300 详细对标

| 维度 | ODS (开放解构) | NVIDIA CMX/GB200 NVL72 | NVIDIA GB300 NVL576 | 差距评估 |
|:-----|:--------------|:---------------------|:-------------------|:---------|
| **Scale-Up互联** | UALink 2.0 (180GB/s) / CXL 4.0 | **NVLink 5.0 (900GB/s per GPU)** | **NVLink 6.0 (1.8TB/s)** | ➡ 1:5 ~ 1:10 |
| **Scale-Out互联** | UEC / RoCEv2 400G | InfiniBand NDR800 / Spectrum-X 800G | InfiniBand XDR / Spectrum-4 1.6T | ➡ 1:2 |
| **GPU 互联拓扑** | OAM BaseBoard (可配置) | NVSwitch 全连接（均匀带宽） | NVSwitch 7th gen 全连接 | ⬇ 非均匀需软件补偿 |
| **GPU 选型** | **任意厂商 GPU/NPU（灵活）** | 仅 NVIDIA GPU | 仅 NVIDIA GPU | ⬆ ODS 灵活 |
| **内存带宽/容量** | 各厂商自定 | HBM3E 8TB/s, 576GB/GPU | HBM4 16TB/s, 1TB+/GPU | ➡ 单 GPU 差距大 |
| **软件栈** | PyTorch/Triton/OpenBMC | CUDA 12.x / NCCL / NVAIE | CUDA 13 / NCCL 4 / NVAIE | ➡ 生态差距大 |
| **管理面** | **Redfish / OpenBMC（开放）** | NVAIE / DGX Cloud | NVAIE / DGX Cloud | ⬆ ODS 开放 |
| **部署弹性** | **几卡到万卡灵活组合** | 固定 NVL72域 | 固定 NVL576域 | ⬆ ODS 弹性 |
| **供应商依赖** | **多源采购，无锁定** | NVIDIA 全栈绑定 | NVIDIA 全栈绑定 | ⬆ ODS 降低风险 |
| **成本模型** | **标准化摊薄，中等成本** | 高溢价 ~30-50% | 更高溢价 | ⬆ ODS 成本优势 |
| **时间窗口** | V1 原型 2027 | 已商用 | 2026H2 出货 | ➡ 落后 1-2 代 |

### 8.1 核心结论

1. **互联带宽差距最大 (1:5~1:10)** — UALink 2.0 的 180GB/s vs NVLink 5.0 的 900GB/s 是 ODS 最根本的硬件瓶颈
2. **ODS 核心优势在灵活性和供应链安全** — 推理场景、混合部署、国产化项目不可替代
3. **软件生态差距是长期挑战** — CUDA 深度绑定在 NCCL 级、算子级、Profiling 级的多层优化，开放生态短期难以复制
4. **可能的最优路线**：ODS 做 Scale-Out 和推理集群，NVLink 域做训练 Scale-Up — 两者互补而非替代

### 12.2 子系统一：Scale-Up 网络（计算互联域）

#### 12.2.1 定义与定位

| 维度 | 说明 |
|:-----|:------|
| **作用** | 超节点内 GPU/NPU 之间的高带宽、低时延互联，支撑 TP（张量并行）通信 |
| **物理范围** | 同一超节点内部，跨 XPU Tray 之间的互联背板/线缆 |
| **时延预算** | < 1-2 μs（端到端） |
| **带宽需求** | 每 GPU 100-900 GB/s（取决于代际和场景） |

#### 12.2.2 ODS 已定义边界

| 维度 | ODS 标准 | 固定程度 |
|:-----|:---------|:--------|
| **互联协议** | UALink 2.0 为主选 | 🔒 锁定（ODS 核心绑定 UALink） |
| **交换机方案** | 独立 Switch Tray（硬件解耦） | 🔒 锁定（架构设计核心） |
| **拓扑类型** | Switch Fat-Tree | 🔒 锁定 |
| **物理层** | 224G SerDes / 铜缆 DAC | 🔒 锁定 |

#### 12.2.3 可替换点详细矩阵

| # | 可替换维度 | 可选方案 | 替换成本/影响 | 技术影响 | 可替换时机 |
|:-:|:----------|:---------|:-------------|:---------|:----------|
| 1 | **GPU/NPU 供应商** | NVIDIA / AMD / 华为昇腾 / 沐曦 / 燧原 / 寒武纪 / Intel Gaudi | 🟡 **中**: 需重新适配 OpenBMC + 驱动 + 集合通信库 | 带宽、显存、算子兼容性随厂商变化 | **随时可换**（OAM 物理兼容） |
| 2 | **互联协议** | UALink / NVLink / HCCS / PCIe 6.0 | 🔴 **高**: 协议栈需全量替换，芯片物理接口不兼容 | 带宽 1:5（UALink vs NVLink），时延 1:3 | **架构级换**（代际更替时） |
| 3 | **互联拓扑** | 全连接 / Fat-Tree / Dragonfly / 2D Torus / Ring | 🟡 **中**: 影响 UBB 底板走线和交换机选择 | Fat-Tree 均匀带宽 vs Dragonfly 更低直径 vs Ring 低成本 | **代际换**（随 UBB 改版） |
| 4 | **交换机芯片** | Broadcom Tomahawk 5/6 / Marvell Teralynx / 华为 / 自研 | 🟡 **中**: 仅影响 Switch Tray 子卡 | 端口密度、功耗、延迟差异 | **热插拔换**（Switch Tray 独立抽取） |
| 5 | **物理层介质** | 铜缆 DAC（<3m）/ 有源光缆 AOC / NPO 线性直驱 / CPO 硅光 | 🟢 **低**: AOC 可进 DAC 可出 | 带宽可扩展 450G→225G，功耗降 3x，距离扩展 | **线缆级即可换** |
| 6 | **每域 XPU 数量** | 4/8/16/32/64 XPU per 超节点 | 🟢 **低**: 调整 BaseBoard 密度 | 随训练规模和 TP 维度灵活调整 | **机柜级规划时定** |
| 7 | **在网计算功能** | 无 / UALink In-Network Compute / 自研集合卸载 | 🟡 **中**: 需交换机支持 | 可降低 All-Reduce 带宽占用 30-50% | **交换机代际升级时** |
| 8 | **时钟同步** | IEEE 1588 PTP / SyncE / 自研 | 🟢 **低**: 软件配置 | 影响 MoE All-to-All 同步精度 | **随时可调** |

#### 12.2.4 可替换决策树

```
决定换 GPU/NPU 供应商？
├── 仅换物理 → 成本 ⬇（OAM 统一接口，换卡即可）
│   └── 需评估：OpenBMC 驱动、NCCL→RCCL/UCC 适配
├── 换互联协议 → 成本 ⬆（换 UBB、Switch Tray、固件栈）
│   └── 推荐时机：ODS V1→V2 代际更替
└── 仅换交换机芯片 → 成本 🟡（Switch Tray 热插拔）
    └── Broadcom↔Marvell↔华为，同一协议族内兼容
```

### 12.3 子系统二：Scale-Out 网络（集群互联域）

| **带宽需求** | 每端口 400G ~ 1.6T |

#### 12.3.2 ODS 已定义边界

| 维度 | ODS 标准 | 固定程度 |
|:-----|:---------|:--------|
| **互联协议** | UEC (LLR/CBCF/MRC) 为主，RoCEv2 为过渡 | 🔒 锁定（ODS 网络工作组推进 UEC） |
| **单端口带宽** | 400G 起步，800G 过渡期 | 🔒 锁定 |
| **拥塞控制** | UEC CBCF | 🔒 锁定 |

#### 12.3.3 可替换点详细矩阵

| # | 可替换维度 | 可选方案 | 替换成本/影响 | 技术影响 | 可替换时机 |
|:-:|:----------|:---------|:-------------|:---------|:----------|
| 1 | **网卡（NIC/DPU/IPU）** | NVIDIA ConnectX-8/9 / 华为 / 中兴 / Intel IPU / AMD Pensando / Fungible | 🟡 **中**: 驱动层需适配，DPU 卸载功能差异大 | 时延、吞吐量、RoCE 卸载能力差异 | **热插拔换**（标准 OSFP/QSFP） |
| 2 | **互联协议** | UEC / RoCEv2 / InfiniBand / 华为 HCCS / 自研 | 🔴 **高**: 端到端协议栈替换，不可混用 | 时延 UEC < RoCE < IB，拥塞控制差异大 | **POD 级规划时换** |
| 3 | **交换机品牌** | Arista / Cisco / 华为 / Juniper / Nokia / 中兴 | 🟢 **低**: 标准协议兼容 | 管理面（CLI/API）差异 | **随时可换**（标准协议） |
| 4 | **交换机 ASIC** | Broadcom Tomahawk / Jericho / Marvell / 华为 Solar | 🟡 **中**: 影响 SDN 控制器适配 | 端口密度、Buffer 大小、功耗差异 | **机柜生命周期换** |
| 5 | **拥塞控制算法** | DCQCN / Timely / UEC CBCF / HPCC / DCTCP | 🔴 **高**: 端网协同方案 | MoE 场景 20-40% 性能差异 | **不可混，需全网统一** |
| 6 | **拓扑** | Leaf-Spine / 3-Tier / Dragonfly+ / Torus / Fat-Tree | 🟡 **中**: 影响布线架构 | 等分带宽、跳数、容错性差异 | **POD 建设时定** |
| 7 | **负载均衡** | ECMP / UEC MRC (包级多路径) / Flowlet | 🟢 **低**: 软件配置 | MRC 在 MoE 场景提升 20-40% | **软件升级即可换** |
| 8 | **光模块速率** | 400G DR4 / 800G DR8 / 1.6T / CPO | 🟢 **低**: 物理可互换 | 带宽密度、功耗线性关系 | **热插拔换** |
| 9 | **RoCE 可信度增强** | ROCEv2 / ROCEv3 / Intel Enhanced RoCE | 🟢 **低**: 固件升级 | PFC 死锁风险差异 | **固件升级** |
| 10 | **SDN 控制器** | SONiC / Cumulus / ONOS / 自研 | 🟡 **中**: 全网统一 | 运维自动化程度差异 | **管理面迁移** |

#### 12.3.4 关键决策：UEC vs InfiniBand vs RoCE

| 场景 | UEC（ODS推荐） | InfiniBand | RoCEv2 | 推荐度 |
|:-----|:--------------|:-----------|:-------|:------|
| **TP 对带宽敏感** | LLR 确定性路由 良好 | 最优（NVIDIA 原生） | 需额外配置 | IB > UEC > RoCE |
| **MoE All-to-All** | MRC 包级负载均衡 优 | ECMP 不均衡 | Flowlet 缓解 | **UEC 最优** |
| **通用训练** | 良好 | 成熟稳定 | 成本最优 | 量体裁衣 |
| **推理集群** | 成本最优 | 过剩 | 最优 | RoCE ≥ UEC |
| **国产化** | **最推荐**（开放） | 被NVIDIA锁定 | 可但需调优 | **UEC 最优** |

### 12.4 子系统三：存储网络

#### 12.4.1 定义与定位

|:-----|:------|

| **存储形态** | NVMe JBOF / 分布式存储 | 🟡 半锁定（两种形态均可） |

#### 12.4.3 可替换点详细矩阵

| # | 可替换维度 | 可选方案 | 替换成本/影响 | 技术影响 | 可替换时机 |
|:-:|:----------|:---------|:-------------|:---------|:----------|
| 1 | **存储协议** | NVMe-oF / NFSv4.2 / POSIX / S3 / Ceph / Lustre | 🔴 **高**: 应用层需重写数据加载逻辑 | 协议语义差异大，S3 兼容性 vs POSIX 一致性 | **架构级换** |
| 2 | **存储介质** | NVMe SSD (TLC/QLC) / CXL 内存池 / HDD / SCM (Optane) | 🟡 **中**: 仅影响存储 Tray | 时延 SSD 100μs vs CXL 1μs，容量 1:10 | **Tray 热插拔换** |
| 3 | **存储控制器** | 厂商自研 / Ceph / Lustre / GPFS / 华为 Dorado / Dell PowerScale | 🔴 **高**: 全局替换 | 管理方式、故障恢复策略差异 | **集群级换** |
| 4 | **互联网络** | RoCE / FC (光纤通道) / SAS / NVMe-oF over TCP | 🟡 **中**: 需换网卡 | 时延 RoCE 5μs vs FC 10μs，带宽 RoCE 可扩展 | **POD 级换** |
| 5 | **Checkpoint 策略** | 同步全量 / 异步增量 / 多级分层 / 内存快照 | 🟢 **低**: 软件配置 | 影响恢复时间 1分钟→1小时 | **随时可调** |
| 6 | **数据缓存** | 本地 NVMe 缓存 / 分布式缓存 (Alluxio) / GPU 显存缓存 | 🟢 **低**: 软件层 | 数据加载加速 10-100x | **随时可加** |
| 7 | **压缩/去重** | 硬件卸载 / 软件（LZ4/Zstd） / GPU Direct 压缩 | 🟢 **低**: 开关配置 | Checkpoint 体积减 2-5x，代价计算开销 | **随时可调** |
| 8 | **存储安全** | 加密 (AES-256) / 完整性校验 / 不可变存储 | 🟢 **低**: 软件配置 | 计算开销 5-20% | **按需启用** |

#### 12.4.4 存储网络可替换性的关键设计原则

```
存储网络的设计本质上是在三条路径间做选择：

路径 A: 超节点本地存储        路径 B: 分布式共享存储        路径 C: 混合分层
┌────────────────────┐    ┌────────────────────┐    ┌────────────────────┐
│ NVMe JBOF in-shelf │    │ Ceph / Lustre      │    │ 本地NVMe 缓存层     │
│ 低延迟·低容量       │    │ 高容量·中延迟       │    │ + Ceph 容量层       │
│ Checkpoint快       │    │ 数据共享方便        │    │ 带宽+容量兼顾       │
│ 扩展受限            │    │ 扩展性好            │    │ 复杂度 ⬆           │
└────────────────────┘    └────────────────────┘    └────────────────────┘
                                                         │
                         可替换：三种路径可混合，可迁移，可分段替换
```

### 12.9 可替换性设计的黄金法则

> **替换不是无代价的，但不可替换等同于被锁定。**
> 对于超节点这样的战略基础设施，关键路径上的每个子系统至少应有 **2 个可竞争供应商方案**。

#### 最小可替换性清单（必须要保证可替换的维度）

| 优先级 | 可替换维度 | 原因 |
|:------|:----------|:------|
| 🔴 **必须** | GPU/NPU 供应商（OAM 物理兼容） | 供应链安全，制裁风险 |
| 🔴 **必须** | BMC 固件（OpenBMC 兼容） | 管理面不被锁定 |
| 🔴 **必须** | 训练框架（PyTorch 兼容生态） | 开发者生态不被锁定 |
| 🟡 **强烈建议** | 集合通信库（NCCL/RCCL/UCC 三选二） | 互联协议切换能力 |
| 🟡 **强烈建议** | 光模块速率（400G/800G/1.6T） | 带宽扩展路径 |
| 🟢 **建议** | 交换机品牌（≥2 家供应商） | 议价能力 |
| 🟢 **建议** | CDU 选型（≥2 技术路线） | 散热方案迭代 |

#### 不可替换的风险等级

| 不可替换级别 | 风险 | 举例 | 应对 |
|:-----------|:-----|:-----|:-----|
| **物理层锁定** | ⬆⬆⬆ | NVLink 域内唯一 GPU | 至少 Scale-Out 面开放 |
| **协议层锁定** | ⬆⬆ | NVLink 私有协议 | 建立开放协议过渡期 |
| **生态层锁定** | ⬆⬆ | CUDA 深度绑定 | PyTorch + Triton 双保险 |
| **运维层锁定** | ⬆ | 私有管理 API | 适配 Redfish 标准化 |
