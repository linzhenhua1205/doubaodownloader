## 第3章 训练可靠性——从「容忍故障」到「等效收敛」

### 3.1 四路线并进格局

本月训练可靠性领域共有 **4 条独立路线** 的论文密集出现，形成「四驾马车」格局：

```text
训练容错四路线比较（2026-06）
                      ┌─────────────────────────────────────┐

│  路线               │ 代表工作          │ 容错粒度         │
├─────────────────────┼──────────────────┼─────────────────┤
│  ① 数据并行副本容错  │ FT-HSDP (Meta)   │ 数据并行副本级    │

│  ③ MoE 专家弹性     │ Lazarus          │ Expert 级冗余    │

└─────────────────────┴──────────────────┴─────────────────┘
```

### 3.2 🥇 ReCoVer — 本月最大突破

| 维度 | 指标 |
|:-----|:-----|
| **核心思想** | 每个迭代保持微批次数恒定，确保每步梯度与无故障运行**随机等价** |
| **容错能力** | **256/512 GPU 失效**（50%）仍保持等效训练轨迹 |
| **对比 CR 基线** | **2.23× 更高有效吞吐量** |
| **额外 Token 处理** | 234 GPU-小时内多处理 **74.9% 的 token** |
| **并行性兼容** | 3D 并行 + HSDP，作为 drop-in 层 |
| **核心机制** | 容错集合通信 + 步内细粒度恢复 + 灵活工作负载重分配 |

**设计启示**: ReCoVer 标志着 RAS 设计从「容忍故障后恢复到原状态」到「故障下梯度与无故障等效」的范式转变。这对芯片级 RAS 意味着：

1. **芯片 RAS 不再是「检测→隔离→修复」的线性流程**，而是与训练框架的梯度语义直接耦合
2. **MCU/SCP 级错误上报需携带梯度语义信息**，不仅是硬件错误码
3. **训练框架容错能力的提升降低了对芯片级「完美 RAS」的要求**——允许部分硬件故障，只要框架能保证梯度等价性

### 3.3 其他关键工作

| 工作 | 核心指标 | 适用场景 |
|:-----|:---------|:---------|
| **FT-HSDP** (Meta, arXiv:2602.00277) | 10 万 GPU，停机 10min→3min，有效训练时间 44%→80% | 超大规模数据并行训练 |
| **Lazarus** (UC Berkeley/MSR, arXiv:2407.04656) | 频繁故障 5.7× 提升，spot instance 3.4× 提升 | MoE 专家并行训练 |
| **SPARe** (ICML'26, 600K GPU) | 冗余堆叠容错，大规模验证 | 通用流水线并行容错 |

### 3.4 趋势研判

> **核心判断**: 训练可靠性正在从「通用容错方案」走向**「并行策略定制化」**。未来每个并行策略（DP/TP/PP/EP/HSDP）都需要其专属的容错方案，而非一个通用的 CR 框架打天下。

### 4.2 关键发现汇编

| 发现 | 来源 | 工程含义 |
|:-----|:------|:---------|
| NaN/+INF/-INF 仅占 SDC 的 **1.01%** | Anatomy of SDC (DSN'26) | 依赖 NaN 检测的 SDC 防护方案覆盖范围极其有限 |
| 单比特翻转仅占 **<40%** | Anatomy of SDC (DSN'26) | 多比特错误模型必须纳入 SDC 仿真 |
| FP8 训练对 SDC **最敏感**，BF16 **最具韧性** | LLM-PRISM (7,664次运行) | 低精度训练需配套更强的硬件 ECC 保护 |
| SDC 可导致**收敛点漂移**到完全不同的权重空间 | SDC 生产实证 (AWS) | 训练轨迹等价性假设不成立 |
| SDC 可引发 **loss spikes** | SDC 生产实证 (AWS) | Loss Spike 预防方案需考虑硬件 SDC 因素 |
| GPU 间可靠性**差异巨大**（同一型号不同集群） | Ampere A100 实证 (6,777万GPU-hr) | 统一的 GPU 可靠性模型不成立 |

### 4.3 代数 SDC 防护——Grid-ECC

**Grid-ECC** (arXiv:2508.04355) 提供了芯片 RAS 之外的另一条数学路线：

| 指标 | 数值 |
|:-----|:-----|
| 纠错能力 | **确定性纠正**最多 2 个错误符号 |
| 计算开销 | GPU 上仅 **24%** |
| 覆盖范围 | 跨三个矩阵分布的矩阵乘法 |
| 理论保障 | 严格纠错性质证明 |

> **设计启示**: Grid-ECC 表明在矩阵乘法这一核心运算上，**数学纠错可以不依赖硬件 ECC 完全在软件层实现**。这为芯片 RAS 设计提供了一种权衡——在硬件冗余成本过高时，可通过代数纠错层弥补。

### 4.4 趋势研判

> **核心判断**: SDC 研究已从「提出方案」转向「实证评估」阶段。三验证体系的建立意味着未来 SDC 工作的可信度门槛大幅提高——仅有仿真不够、仅有门级注入不够、需要三层次交叉验证。

## 第5章 训练稳定性——Loss Spike 预防三路线

### 5.1 三条并行路线

本月训练稳定性领域出现三篇聚焦 Loss Spike 的互补工作：

```text
Loss Spike 预防三路线（2026-06）
┌─────────────────────────────────────────────────────┐
│                                                     │
│  ① 梯度裁剪 → AdaGC (ICML'26)                     │
│     ├─ 自适应 per-tensor 裁剪（EMA 动态阈值）        │
│     ├─ 7B/8B/10B 模型上 **Spike Score → 0**        │
│     ├─ 下游准确率提升 1.27%~2.48%                   │
│     └─ 优化器无关，通信更低（本地裁剪 vs GlobalGC）  │
│                                                     │
│  ② 动量重置 → SPAM                                 │
│     ├─ 检测到异常 → 重置 Adam 一阶/二阶动量          │
│     ├─ 梯度尖峰可达 **1,000× 正常幅度**             │
│     ├─ 60M→1B 预训练 + 4-bit + RL + 时间序列        │
│     └─ 与 AdaGC 可叠加（两道防线）                   │
│                                                     │
│  ③ 异常检测 → ZClip                                │
│     ├─ z-score 自适应裁剪阈值                        │
│     └─ 同期竞争路线，三条并行                        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 5.2 核心洞察——合流成因论

AdaGC 论文首次明确：**Loss Spike 不是单一因素导致**，而是 `data outliers × hardware faults × precision × hyperparameters` 的**合流效应**。

> **设计启示**: Loss Spike 预防方案不应假设单一根源——梯度裁剪、动量重置、异常检测可以叠加使用形成多道防线。

### 5.3 趋势研判

> **核心判断**: Loss Spike 预防正在成为训练可靠性的独立研究热点。从 RAS 设计角度看，**Loss Spike ≈ 软故障**——它与硬件 SDC 一样会破坏训练轨迹。芯片级 RAS（如 MCA 上报）如果能与训练框架的 Loss Spike 检测机制联动，可形成更完整的韧性训练闭环。

### 6.2 设计启示

> 检查点技术已经不是一个「怎么存得快」的单一问题，而是一个**四维优化空间**。芯片级 RAS 设计需要考虑：
> 1. **检查点频率的下限由硬件 MTBF 决定**——如果芯片 RAS 能显著降低 UE 故障率，检查点间隔可以拉长
> 2. **芯片级错误上报延迟直接影响检查点恢复的时效性**——ns-μs 级的芯片 RAS 响应比 ms 级的软件感知快 3-6 个数量级
> 3. **CXL 内存池 + KV Cache offloading** 可成为检查点 Tier 2（对等节点内存层）的新型硬件基础设施

## 第7章 互联与链路可靠性——幽灵故障的量化冲击

### 7.1 关键实证数据

**Ghost in the Datacenter** (arXiv:2603.03736) 提供了目前最全面的数据中心链路级可靠性实证：

| 来源 | 故障类型 | 量化数据 |
|:-----|:---------|:---------|
| **Meta** (LLaMA 3 训练) | 网络中断 | 54 天内 **419 次中断** |
| **ByteDance** (3 个月) | 显式 + 隐式故障 | 38,236 + 5,948 次 |
| **Alibaba** | NIC-ToR 链路月故障率 | **0.057%** |
| **2025 年集群规模放大** | 链路抖动 | **平均每 48 秒一次** |

### 7.2 根本问题——TAR 不可能性

核心论点：所有现有缓解方案（Phi Accrual、SWIM、BFD、OSPF/ISIS 快速收敛、SmartNIC offload、RoCE/PFC、K8s pod eviction）都基于 **Timeout And Retry (TAR)**，而 TAR 无法区分「慢」和「死」——这正是 **FLP 不可能定理** 在异步系统中的不可解问题。

### 7.3 工程含义

| 影响维度 | 含义 |
|:---------|:------|
| **集合通信可靠性** | NCCL AllReduce 的超时/重试机制本质上是 TAR 的一个实例——无法区分慢和死 |
| **RDMA 故障恢复** | 当前 RDMA failover 的「全部重传」策略需要精确化（见第8章 Varuna） |
| **训练框架设计** | **48 秒一次的链路抖动频率**意味着训练容错方案不能假设网络是可靠的 |

### 7.4 趋势研判

> **核心判断**: 链路抖动频率已到**不可忽视的量化级别**（48s/次），且根本问题（TAR vs FLP）在理论上不可解。未来的解决方案方向不是「消除抖动」，而是 **「让训练框架原生容忍抖动」**——这正是 ReCoVer / FT-HSDP / R²CCL 等工作的共同方向。

## 第8章 RDMA 故障恢复——精确化之路

### 8.1 Varuna——范式修正

**Varuna** (arXiv:2603.28001, USTC) 首次指出 RDMA failover 中「全部重传」的错误：

| 维度 | 传统方法 | Varuna |
|:-----|:---------|:-------|
| 故障区分 | 全部 in-flight 请求统一处理 | 区分 pre-failure vs post-failure |
| 重传策略 | 全部重传（浪费+语义错误） | **仅重传 pre-failure** |
| 重传时间减少 | — | **65% 消除** |
| 非幂等操作兼容性 | ❌ 重复执行 | ✅ 语义保持 |
| 连接重建开销 | 有 | **零** |
| 稳态延迟开销 | — | 仅 **0.6–10%** |

### 8.2 R²CCL——集合通信透明容错

**R²CCL** (arXiv:2512.25059) 从集合通信库层面实现 NIC 级链路故障容错：

| 指标 | 数值 |
|:-----|:-----|
| 训练开销 | **<1%** |
| 推理开销 | **<3%** |
| 对比 DejaVu | **47× 超越** |
| 对比 AdapCC | **12.18× 超越** |
| 机制 | 多 NIC 快速连接迁移 + 带宽感知负载重分配 |

### 8.3 趋势研判

> **核心判断**: RDMA 可靠性正在经历从「全部重传」到「故障类型感知精确恢复」的范式转变。Varuna + R²CCL 覆盖了 RDMA 故障的两个层面——连接级精确恢复 + NIC 级透明容错——形成互补的保护层。

## 第9章 GPU/MoE 芯片级 RAS 新进展

### 9.1 集群可靠性模拟器——AIReSim

**AIReSim** (arXiv:2603.07041, **DSN 2026 Industry Track**) 是本月唯一填补「工具空白」的工作：

| 模块 | 功能 |
|:-----|:------|
| **故障注入引擎** | 泊松/Weibull/经验分布，多粒度故障 |
| **检查点策略模块** | 固定/自适应/三级分层策略 |
| **恢复流程模块** | 回滚/重调度/修复队列 |
| **调度策略模块** | FIFO/优先级/弹性重调度 |
| **结果分析引擎** | 有效训练时间/瓶颈识别/敏感度分析 |

**意义**: 填补了集群可靠性设计「参数太多无法手工调优」的工具空白——在设计硬件冗余或修改容错策略前，可以通过 AIReSim 量化不同参数组合的影响。

### 9.2 Ampere GPU 内存错误大规模实证

**论文** (arXiv:2508.03513): 覆盖 **10,693 块 A100** × **6,777 万 GPU-小时**：

| 发现 | 含义 |
|:-----|:------|
| 三超算间差异显著 | 集群级因素（冷却/供电/负载）影响 > GPU 本身 |
| NVLink 错误是第二大来源 | 集合通信容错需考虑 NVLink 可靠性 |
| 跨代差异显著（A100 vs V100） | 容错策略需每代重新校准 |
| 检查点间隔可基于实证数据优化 | 替换经验公式 |

### 9.3 MoE 容错——Lazarus

**Lazarus** (UC Berkeley/MSR): 首个 MoE 专门容错系统：

| 技术 | 说明 |
|:-----|:------|
| 自适应专家副本分配 | 应对 expert workload 不均衡（3-5×差异） |
| 可证明最优的专家放置 | 最大化故障可恢复概率 |
| 灵活 Token Dispatcher | 故障后充分利用幸存节点 |
| **无 idle GPU** | 与 CR 的本质区别——无需等待任何 GPU |

**设计启示**: MoE 架构天然具有 expert 级容错基础——专家可被复制，token 可被重路由。未来芯片 RAS 设计可以考虑在 MoE 场景下，通过硬件 expert 复制/隔离来进一步提升容错能力。
