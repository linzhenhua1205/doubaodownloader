## 1. PCIe Switch 基本概念与工作原理

### 1.1 什么是 PCIe Switch？

PCIe Switch（PCIe 交换芯片）是一种**多端口 PCIe 互联设备**，它在逻辑上扩展了 PCIe 层次结构，使多个 Endpoint（GPU、NVMe SSD、网卡等）可以共享 Root Complex（CPU）的 PCIe 通道资源，或在多个主机之间共享设备。

从协议栈角度看，PCIe Switch 处于：

```
                    ┌──────────────────────┐
                    │    Root Complex (CPU) │
                    │  (PCIe Root Port x16) │
                    └──────────┬───────────┘
                               │
                    ┌──────────┴───────────┐
                    │    PCIe Switch       │
                    │  (上行: x16 接 CPU)   │
                    │  (下行: 多端口扇出)    │
                    └──┬────┬────┬────┬───┘
                       │    │    │    │
                  ┌────┘  ┌─┘  ┌─┘  └────┐
                  │       │    │          │
              ┌───┴──┐ ┌─┴──┐ ┌┴───┐ ┌───┴──┐
              │ GPU  │ │ GPU│ │NVMe│ │ 网卡  │
              │ PCIe │ │    │ │    │ │      │
              └──────┘ └────┘ └────┘ └──────┘
```

### 1.2 与传统以太网交换机的区别

| 维度 | PCIe Switch | 以太网 Switch |
|:-----|:-----------|:--------------|
| **协议层次** | PCIe 事务层（TLP）直接转发 | L2/L3 网络层封装 |
| **延迟** | ~100-300ns（直通转发） | ~0.5-10μs（MAC 处理） |
| **数据完整性** | CRC + ACK/NAK 硬件重传 | TCP/IP 软件重传 |
| **拓扑** | 树形（严格分离） | 任意拓扑（Mesh/Ring） |
| **路由方式** | 地址路由 / ID 路由 / 隐式路由 | MAC 地址 / IP 路由 |
| **流控** | 基于信用的硬件流控（Credit-based） | PFC/ECN 端到端流控 |
| **功耗** | ~8-25W（典型 100 通道 Switch） | ~100-400W（高端交换机） |

### 1.3 PCIe Switch 的内部架构

现代 PCIe Switch 芯片的内部架构如下：

```
                    ┌────────────────────────────────────────────┐
                    │              PCIe Switch SoC              │
                    │                                            │
                    │  ┌────────────────────────────────────┐    │
    Upstream Port   │  │      Internal Switch Fabric        │    │   Downstream
   ┌──────────────┐ │  │   (Crossbar / Shared Buffer)       │    │  ┌───────────┐
   │ ██ PCIe PHY  │─┼─▶│                                    │◀───┼──│ ██ PCIe   │
   │ ██ MAC/DLL/TL│ │  │  ┌───┐ ┌───┐ ┌───┐ ┌───┐          │    │  │ PHY/MAC  │─▶ Port A
   └──────────────┘ │  │  │VC0│ │VC1│ │VC2│ │VC3│  Arbiter │    │  └───────────┘
                     │  │  └───┘ └───┘ └───┘ └───┘  + Mux   │    │  ┌───────────┐
                     │  │  ┌──────────────────────┐          │    │  │ ██ PCIe   │
                     │  │  │ Routing Table (地址)  │          │    │  │ PHY/MAC   │─▶ Port B
                     │  │  ├──────────────────────┤          │    │  └───────────┘
                     │  │  │ Routing Table (ID)    │          │    │     ...
                     │  │  └──────────────────────┘          │    │
                     │  │  ┌──────────────────────┐          │    │
                     │  │  │ VC Buffer Pool (SRAM)│          │    │
                     │  │  │  每 VC 4-32 TLP 深度  │          │    │
                     │  │  └──────────────────────┘          │    │
                     │  └────────────────────────────────────┘    │
                     │                                            │
                     │  ┌────────────────────────────────────┐    │
                     │  │  管理处理器 (MIPS/RISC-V)            │    │
                     │  │  · 配置空间管理                      │    │
                     │  │  · 错误处理 (AER/DPC)               │    │
                     │  │  · 热插拔控制                        │    │
                     │  │  · 电源管理                          │    │
                     │  │  · 虚拟分区管理                      │    │
                     │  │  · NTB 管理                          │    │
                     │  └────────────────────────────────────┘    │
                     └────────────────────────────────────────────┘
```

关键组件：

| 组件 | 说明 | 实现方式 |
|:-----|:-----|:---------|
| **PCIe PHY** | SerDes 收发器，支持 NRZ (Gen5) 或 PAM4 (Gen6) | 模拟硬核 |
| **MAC/DLL/TL** | 数据链路层与事务层逻辑，包解析与组帧 | 硬逻辑（Hardened Logic） |
| **Switch Fabric** | 内部交叉矩阵，TLP 在端口间转发 | Crossbar 或 Shared Buffer |
| **VC Buffer Pool** | 每端口的 Virtual Channel 缓冲，基于信用流控 | SRAM (片上或片外) |
| **Routing Table** | 地址路由表 + ID 路由表 | CAM/TCAM 或 Hash Table |
| **管理处理器** | 片上微控制器，负责慢速路径处理 | 嵌入式 MIPS/RISC-V |

> **原理**: PCIe Switch 的转发机制是"内部直通"— 接收端口的 TLP 经过路由表查找后直接送入目标端口的 VC Buffer，无需任何协议转换。这是 PCIe Switch 延迟极低（~150ns）的根本原因。

## 2.2 关键参数详解

#### 通道数（Lane Count）

PCIe Switch 的总通道数是其最核心的规格指标。通道数的分配影响拓扑设计：

```
例: 100 通道 Switch 的端口分配示例

        ┌──────────┐
        │  CPU     │
        │  x16     │
        └────┬─────┘
             │ (上行, 消耗 16 通道)
        ┌────┴─────┐
        │  Switch  │   ← 总通道 = 100
        │ 100L     │     可用下行 = 100 - 16 = 84 通道
        └──┬───┬───┘
     x16   │   │  x16   x16   x16   x16
   ┌───┐   │   │ ┌───┐ ┌───┐ ┌───┐ ┌───┐
   │GPU│   │   │ │GPU│ │GPU│ │GPU│ │GPU│
   └───┘   │   │ └───┘ └───┘ └───┘ └───┘
           │   └── 4×x16 = 64 通道 (4 GPU)
           │
           └── 1×x16 = 16 通道 (1 网卡)
               + 1×x4  = 4 通道  (NVMe)
               剩余 = 0 (100% 利用)
```

**通道利用规则**:
- 每个物理端口消耗的通道数 = 端口宽度（x2/x4/x8/x16）
- 总消耗 = 上行端口 + 所有下行端口 ≤ 芯片总通道数
- 端口可以不对称（如上行为 x16，下行多个端口合计 > 上行）

#### 转发延迟

PCIe Switch 的延迟通常分为两种模式：

| 模式 | 延迟 | 适用场景 | 原理 |
|:-----|:-----|:---------|:-----|
| **Cut-Through（直通）** | ~100-200ns | 延迟敏感（GPU 通信） | 收到 TLP 头部即查表转发，不等完整包 |
| **Store-and-Forward（存储转发）** | ~300-1000ns | 错误检测（存储场景） | 收完整个 TLP 校验后再转发 |

大多数现代 PCIe Switch 支持**动态切换**：通常使用 Cut-Through，检测到错误后回退到 Store-and-Forward。

#### NTB（Non-Transparent Bridge）

NTB 允许将 Switch 划分为多个独立 PCIe 域，实现**多主机共享设备**：

```
无 NTB (透明):     Root Complex A ←→ Switch ←→ Endpoint A
                   Root Complex B ←→ Switch ←→ Endpoint B
                   (A 和 B 互不可见，各自拥有独立域)

有 NTB:            Root Complex A ←→ Switch ── NTB ──→ Root Complex B
                   (A 通过 NTB 可以访问 B 域的部分地址空间)
```

NTB 典型应用：多 CPU 服务器共享同一组 GPU/RDMA 网卡。

## 4. PCIe 5.0 vs 6.0 Switch 规格对照

### 4.1 基础参数对比

| 参数 | Gen5 (32 GT/s) | Gen6 (64 GT/s) | 性能差异 |
|:-----|:--------------|:--------------|:---------|
| **单 Lane 带宽 (x1)** | ~4 GB/s | ~8 GB/s | 2× |
| **x16 总带宽** | ~64 GB/s | ~128 GB/s | 2× |
| **最大芯片通道数** | 100 (Broadcom/Microchip) | 160 (Broadcom Gen6) | 1.6× |
| **单芯片最大带宽** | 100×4 = 400 GB/s | 160×8 = 1280 GB/s | **3.2×** |
| **转发延迟** | ~100-200ns (直通) | ~120-250ns (含 FEC) | 略增 |
| **SerDes 功耗** | ~8 pJ/bit | ~12-15 pJ/bit | 1.5-1.9× |
| **芯片功耗 (100L)** | ~16W | ~25W | 1.56× |
| **数据完整性** | CRC + ACK/NAK | FEC + CRC + ACK/NAK | 三级防护 |
| **链路宽度灵活性** | 全宽→降宽 | L0p 半宽度节能 | 更优能效 |

### 4.2 协议栈变化对 Switch 设计的影响

| 设计点 | Gen5 Switch | Gen6 Switch | 挑战 |
|:-------|:-----------|:-----------|:------|
| **PHY 复杂度** | NRZ 32G SerDes | PAM4 64G SerDes + ADC/DSP | 面积 2-3×，功耗 ↑60% |
| **FEC 引擎** | 不需要 | 需要 RS(544,514) 实时编解码 | ~20-50ns 额外延迟，面积大 |
| **内部 Buffer** | TLP 粒度管理 | FLIT (236B) 粒度管理 | 重组逻辑更复杂 |
| **TLP 跨 FLIT** | 无（TLP 边界连续） | TLP 可跨 FLIT 分割 | 需要 TLP 重组 FIFO |
| **路由表** | 按 TLP 查表 | 按 FLIT 查表（相同） | 无本质变化 |
| **热管理** | 常规 | 更高密度 → 更强调散热 | 可能需要散热片/主动散热 |

### 4.3 Gen6 开关芯片需要的新模块

```
Gen6 PCIe Switch 新增模块 (vs Gen5):

┌─────────────────────────────────────────────┐
│               Gen6 Switch SoC               │
│                                              │
│  Gen6 PHY (PAM4):                            │
│  ┌────────────────────┐  ┌────────────────┐  │
│  │ ADC (6-8bit, 32G)  │  │ DSP (FFE+MLSE) │  │
│  └─────────┬──────────┘  └───────┬────────┘  │
│            │                     │           │
│            ▼                     ▼           │
│  ┌──────────────────────────────────────┐    │
│  │     FEC Engine (RS 544/514)          │    │
│  │     · 编码器 (TX)                     │    │
│  │     · 解码器 + 纠错 (RX)              │    │
│  └────────────────┬─────────────────────┘    │
│                   │                          │
│                   ▼                          │
│  ┌──────────────────────────────────────┐    │
│  │     FLIT 组帧 / 解帧                  │    │
│  │     · TLP 切片 → FLIT 组装            │    │
│  │     · FLIT 解帧 → TLP 重组            │    │
│  └────────────────┬─────────────────────┘    │
│                   │                          │
│                   ▼                          │
│  ┌──────────────────────────────────────┐    │
│  │     Internal Switch Fabric (Crossbar) │    │
│  │     · 3.2× 带宽升级 (vs Gen5)         │    │
│  │     · 更大 VC Buffer Pool             │    │
│  └──────────────────────────────────────┘    │
└──────────────────────────────────────────────┘
```

### 4.4 Gen5 ↔ Gen6 向后兼容

Gen6 Switch 必须支持向后兼容 Gen5/Gen4/Gen3：

```
链路协商过程:

Gen6 Switch Port          Gen5 Endpoint
      │                        │
      │── TS1 (Gen6 能力声明) ─▶│
      │◀── TS1 (Gen5 能力) ────│
      │                        │
      │  协商结果: Gen5 32GT/s │ (下游只支持 Gen5)
      │                        │
      │── 降速至 Gen5 32GT/s ─▶│
      │── Gen5 Equalization ──▶│
      │◀── Equalization Done ──│
      │                        │
      │   现在以 Gen5 模式运行   │
      └────────────────────────┘
```

> **关键**: Gen6 Switch 需要对每个下行端口独立协商代次。在同一芯片上，A 端口可运行 Gen6×16（接 Gen6 GPU），B 端口可运行 Gen5×8（接 Gen5 NVMe）。这是 PCIe Switch 与生俱来的端口独立性。

## 5. 扩展能力：拓扑与规模

### 5.1 基本拓扑形态

#### A. 单级扇出（Fanout）

最常见形态：一个 CPU Root Complex 通过 Switch 扩展到多个 Endpoint。

```
                  ┌─────────────────────┐
                  │     CPU (x16)       │
                  └──────────┬──────────┘
                             │ UP
                  ┌──────────┴──────────┐
                  │   PCIe Switch       │  → 总通道 = 上行 + ∑下行
                  │   (100L, 52 Ports)   │
                  └──┬──┬──┬──┬──┬──┬──┘
                    │  │  │  │  │  │
                   x16 x16 x8 x8 x4 x4
```

**扩展能力**: 上行带宽决定总吞吐上限。
- 上行为 x16 Gen5 = 64 GB/s 双向
- 下行总和可达 84 GB/s（但同时间只能有 64 GB/s 通过上行）
- **瓶颈在上行** — 这是扇出架构的核心约束

#### B. 双级级联（Tiered）

在更大规模场景，使用双级 Switch：

```
┌─── CPU ───┐
└─────┬─────┘
      │ x16 (上行)
┌─────┴─────┐         第一级: 根 Switch
│ Switch-A  │         (100L, 部分下行接 GPU)
└──┬──┬──┬──┘
   │  │  │
   │  │  └────┬─────┐
   │  │  ┌────┴─────┐  第二级: 叶 Switch
   │  │  │ Switch-B │  (100L, 扩展更多 GPU)
   │  │  └──┬──┬──┬──┘
   │  │     │  │  │
  x16 x16  x16 x16 x16 ...
 GPU GPU   GPU GPU GPU ...
```

**注意**: 多级级联时，在第二级 Switch 下的 GPU 访问 CPU 需要**经过两个 Switch**，延迟增加 ~200-400ns。

### 5.2 多级级联（Tiered Topology）

#### 级联带宽计算

```
第一级可见带宽:
  Switch-A 上行 = x16 Gen5 = 64 GB/s
  Switch-A 总下行 = x16(CPU直连) + x16(Switch-B级联) + 其他
  其中，接 Switch-B 的 x16 通道被固定占用

第二级可用带宽:
  Switch-B 上行 → Switch-A = x16 Gen5 = 64 GB/s
  在 Switch-B 下挂的 GPU 共享 64 GB/s 带宽
  且所有流量必须经过 Switch-A 的上行 x16

因此: 第二级 GPU 的 CPU 访问带宽 = 64 GB/s（被所有二级 GPU 共享）
```

**结论**: 级联层数越多，叶节点到 CPU 的可用带宽越低。通常只推荐 **1-2 级**级联。

### 5.3 虚拟化与分区（Virtual Switch Partitioning）

现代 PCIe Switch 支持将物理芯片分割为多个**独立虚拟 Switch**：

```
┌──────────────────────────────────────────┐
│           PCIe Switch (物理)              │
│                                          │
│  ┌──────────────────┐ ┌────────────────┐ │
│  │ Virtual Switch #1 │ │ Virtual Sw #2 │ │
│  │ Port 1 (UP→CPU1)  │ │ Port 1 (UP→ ) │ │
│  │ Port 2-3 (GPU域)  │ │ Port 2-3 (...)│ │
│  │┌────────────────┐ │ │┌─────────────┐│ │
│  ││独立地址空间/路由 │ │ ││独立地址空间 ││ │
│  ││独立 VC 缓冲     │ │ ││  ...       ││ │
│  ││独立错误域       │ │ │└─────────────┘│ │
│  │└────────────────┘ │ └────────────────┘ │
│  └──────────────────┘                     │
└──────────────────────────────────────────┘
```

**典型分区用途**:
- **多租户隔离**: 同一台服务器 GPU 分给不同租户
- **多主机共享**: 每个虚拟 Switch 分配一个 Host 接口
- **安全隔离**: GPU 域与存储域完全隔离

### 5.4 NTB（Non-Transparent Bridging）

NTB 是实现多主机共享的关键技术：

```
                NTB 通道（地址映射）

  Host A 域                  Host B 域
  ┌────────┐               ┌────────┐
  │ Mem A  │─── NTB ───────│ Mem B  │
  │ Addr X │   映射         │ Addr Y │
  └────────┘               └────────┘
      │                        │
      │ PCIe Switch            │
      │ (NTB 将 Host A 的      │
      │  Mem 地址 X 映射到     │
      │  Host B 的 Y 地址)     │
      └────────────────────────┘

典型拓扑: 2-4 Host 共享 GPU/NVMe 池
```

**NTB 限制**:
- 每个 NTB 通道消耗一笔固定带宽（通常 x4/x8）
- NTB 地址映射需要软件初始化
- 不支持缓存一致性（需软件管理）

### 5.5 最大系统规模的理论极限

| 场景 | 单芯片 | 双级级联 | 三上级联 |
|:-----|:------:|:--------:|:--------:|
| **GPU 数 (x16)** | 4-5 (100L Switch) | 20-30 | 80-120 |
| **NVMe 盘 (x4)** | 20-24 (100L Switch) | 80-100 | 320-400 |
| **总通道数** | 100 | 400 | 1600 |
| **叶节点到 CPU 延迟** | ~150ns | ~350ns | ~550ns |
| **叶节点可用上行带宽** | 64 GB/s | 64 GB/s 共享 | 64 GB/s 深度共享 |

> **实际限制**: PCIe 的树形拓扑本质决定了**上行带宽是永远的单点瓶颈**。对于 GPU 密集通信（AllReduce），跨 Switch 的 GPU 通信绕行 CPU 的路径不高效。这也是 UALink/NVLink 用**全互联拓扑替代树形拓扑**的根本原因。

## 6. 应用场景分析

### 6.1 AI 超节点：GPU Scale-Up 互联

**这是 PCIe Switch 当前增长最快的场景**。ODS 规范和国产超节点方案都大量使用 PCIe Switch。

#### 典型拓扑 1: 8-GPU 节点

```
          ┌─────────────────────────────────┐
          │          CPU (双路)              │
          │     x16          x16             │
          └─────┬────────────┬──────────────┘
                │            │
          ┌─────┴────┐  ┌────┴─────┐
          │ PCIe Sw  │  │ PCIe Sw  │  2× 100L Switch
          │   #1     │  │   #2     │
          └──┬──┬──┬─┘  └─┬──┬──┬──┘
          x16 │  │       │  │  x16
         ┌───┐│  │  x16 ┌┘  │ ┌───┐
         │GPU││  └──────┴─┐ │ │GPU│
         │ 0 ││          │ │ │ │ 7 │
         └───┘│ x16  x16 │ │ └───┘
              │ ┌───┐ ┌──┐│ │
              │ │GPU│ │GPU││ │
              │ │2-3│ │4-6││ │
              │ └───┘ └──┘│ │
              └───────────┘ └─┘
```

**关键考量**:
- GPU-GPU 在同一 Switch 下通信: 直通，~150ns 延迟
- GPU-GPU 跨 Switch 通信: 需要经过 CPU 转发 → 延迟显著增加
- 训练通信模式: DeepSpeed/FSDP 需要 AllReduce 跨所有 GPU
  → **跨 Switch 的 GPU 通信是高带宽瓶颈**

#### 典型拓扑 2: 大规模 GPU Pool（多级 Switch）

```
                 ┌──────┐ ┌──────┐
                 │CPU_A │ │CPU_B │  双 CPU，每 CPU 接 1+ Switch
                 └──┬───┘ └──┬───┘
                 x16│         │x16
              ┌─────┴──┐ ┌───┴─────┐
              │Sw-A    │ │Sw-B     │  根层 Switch (100L)
              │(根)    │ │(根)     │
              └─┬──┬───┘ └───┬──┬──┘
           x16  │  │       x16 │  │
         ┌─────┐│ x16   ┌────┐│  │
         │GPU 0││       │GPU4││  │
         └─────┘│       └────┘│  │
                │x16         │  │x16
          ┌─────┴──┐   ┌────┴──┴──┐
          │Sw-C    │   │Sw-D      │  叶层 Switch (100L)
          │(叶)    │   │(叶)      │
          └──┬──┬──┘   └──┬──┬──┬─┘
          x16│  │x16    x16│  │  │x16
         ┌──┐│ ┌──┐  ┌──┐│ ┌──┐│
         │G1││ │G2│  │G5││ │G6││
         └──┘│ │  │  └──┘│ │  ││
             └─┘  └──┘   └─┘  └┘
```

**此拓扑的数学限制**:

| GPU 对 | 通信路径 | 跳数 | 总延迟估计 | 可用带宽 |
|:-------|:---------|:----:|:---------:|:--------:|
| G0↔G1（同叶Sw） | G0→Sw-A→G1 | 1 跳 Switch | ~150ns | 64 GB/s |
| G0↔G4（同根Sw） | G0→Sw-A...→G4 | 2 跳 Switch | ~300ns | 64 GB/s ▲ |
| G0↔G5（不同根） | G0→Sw-A→CPU_A→Sw-B→G5 | 2 Switch + CPU | ~500ns+ | 受限 CPU 带宽 |
| G0↔G6（跨 CPU） | G0→Sw-A→CPU_A→C2C→CPU_B→Sw-B→G6 | 最长路径 | ~700ns+ | 瓶颈严重 |

> **▲ 关键洞察**: PCIe Switch 在单根下可高效扩展 GPU（1-2 跳），但跨 CPU 的全互联通信是 PCIe 拓扑的天然短板。这是为何 NVLink/UALink 用直连拓扑替代 PCIe 作为 GPU-GPU 主互连的原因。

#### GPU 互联场景中 PCIe Switch 与 NVSwitch/UALink 的关系

| 维度 | PCIe Switch | NVSwitch / UALink Switch |
|:-----|:-----------|:------------------------|
| **协议** | PCIe (TLP) | NVLink / UALink (自定义) |
| **拓扑** | 树形（CPU 为中心） | 全互联（任意 GPU 直通） |
| **延迟** | ~150-300ns/跳 | ~50-100ns/跳 |
| **带宽 (x16 等价)** | 64 GB/s (Gen5) | 450 GB/s (NVLink5) |
| **缓存一致性** | 无 | 支持（NVLink） |
| **多 CPU 通信** | 需绕行 CPU/C2C | 交换机内部直通 |
| **适用场景** | Host↔Device、存储 | Device↔Device（GPU 直连） |

**PGIe Switch 在 AI 超节点中的实际角色**:

```
每个 GPU 实际上需要两种互联:

    GPU
    ├── x16 PCIe ─── PCIe Switch ─── CPU
    │      (控制面、数据预处理、Boot)
    │
    └── UALink/NVLink ─── UALink Switch ─── 其他 GPU
           (AllReduce、P2P 显存访问)
```

即：**PCIe Switch 是 GPU 的控制面路径，UALink 是 GPU 的数据面路径**。

### 6.2 分布式存储：NVMe 高密度扩展

**这是 PCIe Switch 最成熟的传统应用**。

#### 典型拓扑: 24× E3.S NVMe 存储节点

```
             ┌────────────────────┐
             │  CPU (x64 通道)    │
             └──┬──┬──┬──┬──┬──┬─┘
          x16 │  │  │  │  │  │  x16
     ┌────┐   │  │  │  │  │  │   ┌────┐
     │PCIe│◀──┘  │  │  │  │  └──▶│PCIe│
     │Sw #│      │  │  │  │      │Sw #│
     │ 1  │      │  │  │  │      │ 3  │
     └─┬──┘      │  │  │  │      └──┬─┘
     x4│x4...    │  │  │  │    x4...│x4
  ┌──┐ ┌──┐       │  │  │  │    ┌──┐ ┌──┐
  │NV│ │NV│       │  │  │  │    │NV│ │NV│
  │Me│ │Me│  ┌────┘  │  │  └───┐│Me│ │Me│
  └──┘ └──┘  │       │  │      │└──┘ └──┘
             │  ┌────┴──┴──┐   │
             │  │PCIe Sw #2│   │
             │  └──┬──┬──┬─┘   │
             │  x4 │  │  │ x4  │
             └──┐ ┌──┐│ ┌──┐  │
                │ │NV││ │NV│  │
                └─┘Me│└─┘Me│  │
                     └─────┘  └──
```

**可扩展性关系**:

| 参数 | 值 | 说明 |
|:-----|:---|:------|
| CPU 总 PCIe 通道 | 64-128 (双路) | 每路 EPYC/Xeon |
| 每 NVMe 消耗 | x4 | 典型 E3.S/U.2 |
| 直接接 CPU 的 NVMe | 16-32 | 受 CPU 通道限制 |
| 通过 Switch 扩展 | 24-48 | 以 PEX8000 48L 为例 |
| 扩展后总 NVMe | 40-80 | 典型高密度存储节点 |

**存储场景的特殊要求**:
- **NVMe 热插拔**: 所有端口必须支持单盘热插拔和意外移除
- **QoS 带宽控制**: 存储分层需要 IO 隔离
- **低延迟**: 4K 随机 IO 延迟 < 100μs（含 PCIe Switch）
- **与 CPU PATH 配合**: 通过 AER + DPC 实现磁盘故障隔离

### 6.3 数据中心服务器：Fanout 扇出架构

**最普遍的应用**，几乎每台服务器内部都有 PCIe Switch。

#### 典型场景:

| 用途 | 拓扑 | 典型 Switch 型号 | 价值 |
|:-----|:-----|:----------------|:------|
| **GPU 扇出** | CPU x16 → Switch → 4-5× GPU x16 | 100L Gen5 | 单 CPU 支持多 GPU |
| **NVMe 扇出** | CPU x16 → Switch → 8-16× NVMe x4 | 48L Gen4/5 | M.2/E3.S 高密度 |
| **网卡扩展** | CPU x16 → Switch → 2-4× 网卡 x8 | 16-32L Gen4/5 | 多网口/多网络 |
| **通用 I/O** | CPU x8 → Switch → 多路 USB/SATA | 8-16L Gen3 | 主板 I/O 扩展 |

### 6.4 多主机互联（Multi-Host）

**将多台服务器的主机通过 NTB 互联，共享 PCIe 设备池**。

```
           ┌─────────┐  ┌─────────┐  ┌─────────┐
           │ Host A  │  │ Host B  │  │ Host C  │
           └────┬────┘  └────┬────┘  └────┬────┘
                │            │            │
                └──────┬─────┴─────┬──────┘
                       │           │
                 ┌─────┴───────────┴─────┐
                 │  PCIe Switch (PSX)    │
                 │  NTB: A↔B↔C 互联      │
                 └──┬──┬──┬──┬──┬──┬──┬──┘
                   x16│  │  │  │  │  │
                  ┌───┐│  │  │  │  │ ┌────┐
                  │GPU││  │  │  │  └─│ NVMe│
                  │   ││  │  │  │    │池   │
                  └───┘│  │  │  │    └────┘
                       │  │  │  └──────┐
                       │  │  └──────┐  │
                       │  └───┐    │  │
                       │ ┌──┐ │ ┌──┐│  │
                       │ │R │ │ │S ││  │
                       │ │DM│ │ │to││  │
                       │ │A │ │ │ra││  │
                       └─└──┘ └─└──┘└──┘
```

**典型应用**: 超融合架构双控 / 存储双活 / GPU 资源池化

### 7.3 技术趋势

| 趋势 | 驱动力 | 时间线 | 对 Switch 的影响 |
|:-----|:-------|:------|:----------------|
| **Gen6 普及** | AI 超节点带宽需求 | 2025-2027 | 160L 芯片成旗舰，需要更高功耗管理 |
| **端口灵活性增强** | GPU/NPU 异构部署 | 2025-2026 | 端口 Bifurcation 颗粒度更细（x2/x4/x8/x16） |
| **更低延迟** | AllReduce 通信 | 2025-2027 | 优化 Cut-Through 到 <100ns |
| **虚拟化深度集成** | 多租户 GPU 池化 | 2025-2026 | 单芯片分区数从 26→更多 |
| **CXL 集成** | 内存池化需求 | 2026-2028 | CXL 协议在 PCIe 之上运行，Switch 需要理解 CXL 语义 |
| **能效优化** | 数据中心 PUE 要求 | 2025-2027 | L0p 节能、动态 Lane 开关 |
| **RAS 增强** | 万卡集群可靠性 | 2025-2026 | 更多预测性故障检测，FEC 日志 |

| 厂商 | 进展 | 代次 | 详细 |
|:-----|:-----|:-----|:------|

> **国产化挑战**:
> 1. **SerDes 难度**: Gen5 32G NRZ / Gen6 64G PAM4 SerDes 是最大的模拟设计壁垒
> 2. **生态壁垒**: 需兼容 Broadcom Switch 的寄存器级行为（OEM BIOS/驱动适配）
> 3. **验证周期**: 大型 Switch（100L+）的验证工作量巨大，通常 2-3 年
> 4. **成本劣势**: 国产小批量 vs Broadcom 年出货数千万颗，单价差距悬殊

## 8. 关键决策因素（选型指南）

### 8.1 PCIe Switch 选型决策树

```
Q: 我的应用需要什么？

1. 连接 GPU/NPU?
   → 必须 Gen5 (32GT/s) 或 Gen6 (64GT/s)
   → 每 GPU 需要 x16 通道
   → GPU 数量 × 16 + 上行 x16 ≤ Switch 总通道
   → 如：5×GPU ×16 = 80 + 上行 16 = 96L → 选 100L Switch

2. 连接 NVMe SSD?
   → Gen4 已够（NVMe 通常不需要 Gen5 带宽）
   → 每盘 x4 通道
   → 考虑热插拔 → 必须选 Managed Switch (PEX8000/PFX)
   → 需要考虑 QoS 隔离

3. 是否需要多 Host 共享?
   → 需要 NTB 功能 → 选 PSX 或 PFX (NTB 数量≥需求)
   → 需要虚拟分区 → PSX (可编程) 或 PFX (固定分区)

4. 是否需要高级 RAS?
   → AER/DPC/ACS/E CRC → 所有数据中心级 Switch 都支持
   → 热插拔 → 必须 Managed Switch
   → 在线固件升级 → PSX 系列支持

5. 成本敏感?
   → 非 Managed Switch (Fanout 型) 价格约 Managed 的 60-70%
   → 但损失热插拔、RAS、虚拟化功能

6. 国产化要求?
   → 当前无成熟替代方案
   → 策略：保留 Broadcom 方案 + 跟踪国产进展
```

### 8.2 代次选择矩阵

| 应用场景 | 推荐代次 | 推荐通道 | 推荐厂商 | 理由 |
|:---------|:--------|:--------|:---------|:------|
| AI 训练 GPU (NVIDIA B200/B300) | Gen5 | 100L | Broadcom | 成熟量产，NVIDIA 参考设计 |
| AI 推理 (L40S/A100) | Gen4 | 48-64L | Broadcom | 性价比最佳 |
| AI 超节点 (Gen6 GPU) | Gen6 | 160L | Broadcom/Microchip | 带宽翻倍，适配未来 GPU |
| 分布式存储 (NVMe) | Gen4 | 48-64L | Broadcom PEX8000 | 标配，热插拔支持完整 |
| 通用服务器 (M.2扩展) | Gen4 | 16-24L | Broadcom/Microchip | 低成本扇出 |
| 嵌入式/工业 | Gen3/4 | 8-16L | Microchip | 宽温、加固、长供货 |

## 9. 总结与展望

### 9.1 核心结论

1. **PCIe Switch 是服务器/存储/AI 基础设施的"骨架芯片"**：几乎所有服务器内部都在使用，覆盖从 8L 到 160L 的广泛需求

2. **Broadcom 事实垄断（>70% 份额）**：从 PLX 时代的积累、全产品线覆盖、OEM 生态绑定，使其地位难以撼动

3. **Gen6 是协议栈重构而非简单提速**：PAM4 + FLIT + FEC 使 Switch 芯片设计复杂度剧增（面积 2-3×，功耗 ↑60%），但也打开了 160L 128 GB/s/端口的新维度

4. **AI 超节点是 PCIe Switch 的最大增量市场**：每台 8-GPU 服务器需要 2×100L Switch，万卡集群需要数千颗 Switch
   - 但 GPU-GPU 通信的主力正从 PCIe 转向 UALink/NVLink
   - PCIe Switch 在 AI 场景中的角色从"数据面"转为"控制面+存储面"

5. **国产替代极难**：SerDes 壁垒 + 生态壁垒 + 验证壁垒使国产 PCIe Switch 在 3-5 年内难以形成有效竞争

### 9.2 未来 3 年趋势判断

| 时间 | 关键事件 | 影响 |
|:----|:---------|:-----|
| 2025-2026 | Gen6 160L 芯片量产 | 超节点带宽翻倍，单芯片 1280 GB/s |
| 2026-2027 | CXL 3.0 在 Switch 上原生支持 | PCIe Switch 从纯 I/O 芯片演变为内存互联芯片 |
| 2026-2028 | UALink 交换机开始部署 | PCIe Switch 的 GPU 互联职能被 UALink 逐步取代 |
| 2027+ | PCIe 7.0 规范预计发布 (128 GT/s PAM4) | 但 Switch 产品化至少要 2029+ |
| 长期 | 光电合封 (CPO) Switch | PCIe 光互联可能改变 PCIe Switch 形态 |

### 9.3 对超节点架构师的建议

1. **短期（2025-2026）**：AI 训练节点继续使用 Broadcom Gen5 100L Switch（成熟、稳定），开始验证 Gen6
2. **中期（2026-2027）**：跟踪 UALink 的进展，PCIe Switch 向"控制面"角色收敛；5nm/3nm 制程的 Gen6 Switch 将大幅降低功耗
3. **存储侧**：Gen4/5 PEX8000 系列仍是 NVMe 扩展的最优解，无需急于升级
4. **国产化**：短期无法替代，中长期关注华为海思是否开放供应 + 是否有新创企业突破 SerDes
