## 📐 六维设计框架总览

整机系统设计的核心是**六维联动决策**：

```
                        使用场景 (场景驱动)
                            ↓
                    ┌───────────────────┐
                    │   整机规格定义      │ ← 友商分析 (竞争定位)
                    │   (算力/功耗/尺寸)  │
                    └────────┬──────────┘
                             ↓
              ┌──────────────┼──────────────┐
              ↓              ↓              ↓
         配置策略 ─────→ 资源复用 ←──── 面板/板卡定义
         (维度/矩阵)      (已有资产)      (前后接口)
```

**决策顺序**: 场景 → 规格 → (配置 + 复用 + 面板) 并行迭代，友商分析贯穿始终。

## 一、友商分析方法论 — 从"看别人"到"定自己"

### 1.1 五维对标框架

友商分析不是简单地罗列别人做了什么，而是通过系统化对标找到**差异化空间**。

| 维度 | 问题 | 信息来源 | 分析工具 |
|:-----|:------|:---------|:---------|
| **① 产品矩阵** | 友商覆盖了哪些形态/密度/方案？ | 官网产品页、OFC/COMPUTEX展品 | 产品矩阵拓扑图 |
| **② 技术路线** | 友商选择了什么互联/散热/供电方案？ | Whitepaper、公开演讲、Patents | 技术路线对比表 |
| **③ 成本结构** | 友商的BOM/ODM/定价策略？ | TrendForce、供应链情报、竞品报价 | BOM逆向估算 |
| **④ 客户覆盖** | 友商拿下了哪些客户/场景？ | 客户案例、中标公告、财报 | 客户份额地图 |
| **⑤ 生态绑定** | 友商绑定了哪些芯片/软件/生态？ | 合作伙伴公告、认证列表 | 生态依赖图 |

### 1.2 产品矩阵拓扑图（以主流厂商为例）

产品矩阵不是列表，而是**在2D平面上的定位图**：

```
算力密度（GPU数量）→
 32 ├──────────┬──────────┬──────────┐
    │          │          │ 华为     │
    │          │          │ Atlas 800│
 16 ├──────────┼──────────┼──────────┤
    │ 浪潮     │ 宁畅     │ 新华三   │
    │ NF5688M6 │ X680 G3  │ R5500 G5 │
  8 ├──────────┴──────────┴──────────┤
    │     标准 H100 PCIe 方案         │
    └─────────────────────────────────┘
      风冷         冷板         浸没    ← 散热深度
```

**读图方法**：
- **左上角**（高密度+低成本）→ 不可能三角，需警惕
- **右下角**（极致散热+极致密度）→ NVIDIA/华为的领地
- **左下角**（中等密度+风冷）→ 最大众市场，但也最卷
- **中间区域**（8-16卡+冷板）→ **国内竞争最激烈的核心战场**

### 1.3 友商深度拆解 — 5家主流厂商设计哲学

| 维度 | 浪潮 | 华为 | 新华三 | 宁畅 | 超聚变 |
|:-----|:-----|:-----|:-------|:-----|:-------|
| **设计哲学** | 标准化+规模效应降本 | 全栈自主+系统优化 | 兼容性优先+企业级 | 场景定制+快速响应 | 传承华为+逐步自主 |
| **主力车型** | NF5688M6 (8卡)/NF5488A5 | Atlas 800 A2 (昇腾) | R5500 G5 / R5300 G5 | X680 G3 (GPU) / B8000 | FusionServer 9000 |
| **GPU方案** | NVIDIA为主+昇腾可选 | 昇腾全系 | NVIDIA+AMD+昇腾混合 | NVIDIA全系 | NVIDIA+昇腾混合 |
| **互联偏好** | PCIe Switch标准化 | HCCS+PCIe混合 | PCIe Switch为主 | PCIe Switch + NVLink | PCIe Switch为主 |
| **散热路线** | 兼容风冷/液冷双模 | 冷板自研TMU | 液冷伙伴生态 | 液冷全系（B8000标杆） | 兼容液冷 |
| **BMC路线** | OpenBMC开放化 | iBMC+FusionDirector | H3C CAS自主 | 自研+OpenBMC | BMC自主 |
| **电源方案** | 标准AC/DC | 三总线盲插 | 标准AC | DPS分布式 | 标准AC |
| **核心优势** | 成本、规模、供应链 | 系统级优化、软硬协同 | 客户覆盖广、兼容性好 | 定制灵活、响应快 | 服务渠道广 |
| **核心劣势** | 差异化不足、同质化 | 生态封闭、成本高 | 技术创新偏慢 | 规模效应弱 | 品牌认知重建中 |
| **定价策略** | 性价比导向（最低价） | 自主溢价（高毛利） | 中高端定价 | 中端+服务溢价 | 中等定价 |

### 1.4 差异化空间矩阵

通过友商分析，识别出**别人没做好或没做**的领域：

| 差异化维度 | 市场现状 | 差异化机会 | 可行性评估 |
|:-----------|:---------|:-----------|:-----------|
| **异构融合** | 各厂商在单一GPU路线站队 | **多GPU平台兼容**（N+昇腾双轨） | 高（板卡级兼容设计） |
| **液冷一体化** | 大多外采CDU+液冷方案 | **自研液冷集成**（整柜级交付） | 中高（需液冷团队） |
| **运维智能化** | BMC功能同质化 | **AI预测运维**（如提前7天预警故障） | 高（软件投入即可） |
| **场景深度优化** | 通用机型覆盖全场景 | **垂直场景专机**（如推理专用、训练专用） | 中（需市场验证） |
| **生态兼容性** | NVIDIA > 国产的切换成本高 | **"一键迁移"方案**（硬件+软件平滑过渡） | 高（软件+测试投入） |

### 1.5 反向BOM估算 — 从友商定价反推成本

这是友商分析中最硬核的能力。通过产品定价和公开规格，逆向推断友商的BOM成本和毛利空间：

**示例：某厂商8卡H100通用服务器**

| 成本项 | 估算 | 依据 |
|:-------|:-----|:------|
| GPU × 8 | 约 ¥120-160 万 | H100 约 20-25 万/颗（行情波动） |
| CPU × 2 | 约 ¥5-8 万 | Xeon Platinum 约 2.5-4 万/颗 |
| 主板+PCB | 约 ¥2-4 万 | 12+层板，含板级器件 |
| 内存 2TB | 约 ¥6-10 万 | DDR5 约 3-5 万/512GB |
| 网络 400G × 8 | 约 ¥8-12 万 | CX-7 ≈ 1-1.5 万/张 |
| PSU+风扇+机箱 | 约 ¥2-3 万 | 冗余电源+机箱结构件 |
| 组装+测试+软件 | 约 ¥3-5 万 | ODM加工+品控+认证分摊 |
| **BOM合计** | **≈ ¥146-202 万** | — |
| **市场售价** | **≈ ¥200-280 万** | 毛利率 ≈ 25-35% |
| **友商利润空间** | ≈ ¥50-80 万 | 扣除销售/研发/管理费用 |

**关键洞察**: GPU在BOM中占比高达65-75%，整机厂商的议价空间核心在"非GPU部件"的降本。

## 二、整机规格定义 — 从需求到可执行规格书

### 2.1 规格定义六步法

```
客户需求 → 场景抽象 → 技术指标 → 关键参数 → 约束边界 → 规格书
    │           │           │          │          │          │
    ├─ 场景字    ├─ 归一化   ├─ 性能     ├─ 可量化   ├─ 不可协商 ├─ 正式文档
    ├─ 市场数据  ├─ 分类     ├─ 功耗     ├─ 可验证   ├─ 可协商   ├─ 版本控制
    └─ 竞品情报  └─ 优先级   └─ 互联     └─ 可测试   └─ 期望    └─ 评审纪要
```

### 2.2 规格定义的核心维度矩阵（16项）

一个好的整机规格书应该覆盖以下维度，每项都要有明确的**数值目标**和**验证方法**：

| # | 维度 | 定义项 | 典型目标 | 验证方法 |
|:-:|:-----|:-------|:---------|:---------|
| 1 | **算力规格** | GPU型号/数量、CPU型号/数量、FP16/TF32算力 | 16×H100 = 64 PFLOPS FP8 | 基准测试 |
| 2 | **内存规格** | GPU显存类型/容量/带宽、系统内存容量/通道 | HBM3e 141GB ×16, DDR5 2TB | memtest, HPL |
| 3 | **互联规格** | GPU-GPU拓扑、CPU-GPU协议、Scale-out方案 | NVLink 5 392GB/s ×16, 800G RoCE ×8 | NCCL/allreduce测试 |
| 4 | **存储规格** | 本地NVMe容量/数量、远端存储协议、KV Cache方案 | 8×7.68TB NVMe, NFS/CXFS | fio, IOR |
| 5 | **散热规格** | 散热方案、TDP上限、环境温度范围、PUE目标 | 冷板液冷, GPUs 1000W TDP, 35°C inlet | CFD仿真+实测 |
| 6 | **供电规格** | 供电架构、PSU容量/冗余、峰值功率、备援方案 | 4×3kW 80Plus钛金, N+1冗余, BBU备援 | 功率测试 |
| 7 | **物理规格** | 机箱高度/U数、深度、宽度、重量、安装方式 | 4U 机架式, 深度≤800mm, 重量≤60kg | 尺寸测量 |
| 8 | **网络规格** | 管理口、业务口、存储口数量/速率/形态 | 2×25G管理, 8×400G QSFP-DD业务口 | 吞吐量测试 |
| 9 | **管理规格** | BMC协议、Redfish版本、告警方式、固件升级 | Redfish 2024.1, IPMI 2.0, HTTPS | API自动化测试 |
| 10 | **安全规格** | 可信启动、TPM/TCM、安全启动、固件签名 | TPM 2.0, Secure Boot, 国密SM3/SM4 | 安全评审 |
| 11 | **可靠性规格** | MTBF、MTTR、热插拔次数、故障覆盖率 | MTBF ≥ 50,000h, FIT覆盖100% | 加速寿命测试 |
| 12 | **环境适应性** | 工作温度、湿度、海拔、振动、冲击 | 0-40°C, 8%-90% RH, ≤3000m | 环境箱测试 |
| 13 | **认证合规** | 安规/EMC/能效认证、信创认证、行业准入 | CCC, CE, FCC, 80Plus, 信创目录 | 认证测试 |
| 14 | **运维规格** | 前/后维护方式、指示灯定义、线缆管理 | 前IO, 热插拔GPU/PSU/风扇, LED故障指示灯 | 运维可用性评审 |
| 15 | **噪音规格** | 噪音等级(dBA) | ≤45dBA (25°C), ≤75dBA (满载) | 噪音测量 |
| 16 | **成本规格** | 目标BOM成本、目标售价、NRE分摊 | BOM ≤ ¥180万, 目标售价 ¥220万 | 成本核算 |

### 2.3 规格冲突仲裁表

当上述16项维度之间出现冲突时，以下面的优先级规则决策：

| 冲突场景 | 仲裁原则 | 典型处理 |
|:---------|:---------|:---------|
| 算力 vs 散热 | **散热优先** | 若散热方案无法支撑目标算力 → 降算力/换散热方案 |
| 性能 vs 成本 | **成本底线保护** | 性能提升的边际收益 < 成本增加 → 放弃 |
| 密度 vs 维护性 | **维护为先** | 高密度但不可维护 = 不可用 |
| 新特性 vs 兼容性 | **兼容优先** | 除非新特性是核心差异化 |
| 规格 vs 时间线 | **时间线为王** | 迟到的产品 = 不存在的产品 |
| 自主 vs 成熟 | **成熟优先** | 除非自主是战略要求（国产化） |

### 2.4 规格书的"三道门"评审标准

| 评审门槛 | 问题 | 通过标准 |
|:---------|:-----|:---------|
| **门1：必要性** | 这个规格真的需要吗？ | 至少回答 3 个 Yes：(1)有明确客户需求 (2)有竞品对标依据 (3)符合产品路线图 |
| **门2：可行性** | 我们能做得到吗？ | 内部资源+供应链+技术储备 三方面均可支撑 |
| **门3：可验性** | 能证明做到了吗？ | 有明确的测试用例、通过标准、验收方法 |

## 三、配置策略 — 从"一个机型打天下"到"配置矩阵"

### 3.1 配置维度树

整机配置不是随机的，而是**五维交叉**的矩阵决策：

```
                        服务器整机配置
                              │
          ┌───────────────────┼────────────────────┐
          │                   │                    │
      核心配置              可选配置            软件配置
          │                   │                    │
  ┌───────┼───────┐    ┌─────┼─────┐      ┌──────┼──────┐
  │       │       │    │     │     │      │      │      │
 GPU    CPU    网络  存储  扩展卡  电源  BMC  驱动  管理平台
配置   配置   配置  配置   配置   配置  版本  版本   版本
```

### 3.2 配置梯度策略

| 梯度 | 定义 | 管理方式 | SKU数量控制 | 适用阶段 |
|:-----|:-----|:---------|:-----------:|:---------|
| **L0 标准配置** | 面向广泛覆盖，1-3个默认配置 | 基线固化，常备库存 | 1-3个 | 通用市场 |
| **L1 可选配置** | 在标准基础上，提供A/B选型 | 按订单配置(CTO) | 5-10个变体 | 主流市场 |
| **L2 场景配置** | 针对特定场景的预定义配置包 | 需求驱动，快速出图 | 视场景数 | 垂直场景 |
| **L3 定制配置** | 完全按客户需求定制 | 项目制，非标管理 | 极少 | 大客户/项目 |

### 3.3 典型AI服务器的配置矩阵

以 **16卡液冷AI推理/训练通用服务器** 为例：

| 配置项 | 基础版 (L0) | 均衡版 (L1-A) | 性能版 (L1-B) | 极致版 (L2) |
|:-------|:-----------|:-------------|:-------------|:-----------|
| **GPU** | 8×H100 80GB | 16×H100 80GB | 8×B200 192GB | 16×B200 192GB |
| **CPU** | 2×Xeon 4510Y (12核) | 2×Xeon 5515+ (20核) | 2×Xeon 6518P (28核) | 4×Xeon 6980P (64核) |
| **内存** | 512GB DDR5 (8×64GB) | 1TB DDR5 (16×64GB) | 2TB DDR5 (16×128GB) | 4TB DDR5 (32×128GB) |
| **系统盘** | 2×480GB SATA SSD | 2×960GB SATA SSD | 2×1.92TB NVMe | 4×3.84TB NVMe |
| **数据盘** | — | 4×3.84TB NVMe | 8×7.68TB NVMe | 16×15.36TB NVMe |
| **业务网** | 4×25G SFP28 | 4×100G QSFP56 | 8×400G QSFP-DD | 8×800G QSFP-DD |
| **管理网** | 2×1G RJ45 | 2×10G SFP+ | 2×25G SFP28 | 2×25G SFP28 |
| **PSU** | 2×2000W 冗余 | 4×2000W 冗余 | 4×3000W 冗余 | 6×3000W+N+1 |
| **典型场景** | 小模型推理/开发测试 | 中等推理/微调 | 大模型训练/推理 | 超大模型训练 |
| **BOM参考** | ¥80-100万 | ¥180-220万 | ¥280-350万 | ¥500万+ |

### 3.4 配置管理的三大原则

| 原则 | 说明 | 执行方法 |
|:-----|:-----|:---------|
| **① 易配不易改** | 配置项一旦定义就固定，但允许灵活组合 | 用配置器管理，避免手工改BOM |
| **② 80/20 配置覆盖** | 20%的配置覆盖80%的市场需求 | 优先定义L0+L1-A+L1-B三个核心配置 |
| **③ 配置与验证对齐** | 每增加一个配置变体，增加一条验证链路 | 配置新增需同步更新测试矩阵 |

**警告**：每增加一个配置变体，测试工作量增加约15-20%。**SKU数量应以5的倍数增长，而非指数增长**。

### 3.5 配置器需求（6步上库法）

配置器是将配置策略落地为IT工具的流程：

| 步骤 | 操作 | 输出 |
|:-----|:-----|:-----|
| ① **定义配置维度** | 明确哪些维度可配置（GPU/CPU/内存/存储/网络/PSU） | 配置维度清单 |
| ② **设定约束规则** | 哪些组合合法（如A+B可配，A+C不可配） | 配置规则矩阵 |
| ③ **构建BOM模板** | 每个配置项映射到物料编码 | BOM模板库 |
| ④ **确定价格策略** | 基础价+选配加价 | 价格配置表 |
| ⑤ **验证配置路径** | 自动化测试所有合法路径 | 配置验证报告 |
| ⑥ **上库投产** | 配置器发布、订单系统对接、培训 | 上线通知 |

## 四、使用场景 — 场景驱动设计的完整方法

### 4.1 场景驱动的"五步设计法"

整机设计不是先设计再找场景，而是**由场景反推设计**：

```
典型场景 ─→ 导出核心需求 ─→ 映射到技术指标 ─→ 确定权衡取舍 ─→ 固化设计基线
    │              │               │                 │               │
    ├─ 推理场景    ├─ 低延迟       ├─ GPU互联拓扑    ├─ 延迟vs吞吐  ├─ 推理专机
    ├─ 训练场景    ├─ 高吞吐       ├─ 散热方案        ├─ 成本vs性能  ├─ 训练专机
    └─ 混合场景    └─ 弹性         └─ 供电方案        └─ 通用vs专用  └─ 通用机
```

### 4.2 AI服务器四大典型场景及其设计影响

#### 场景A：在线推理（延迟敏感）

| 维度 | 需求特征 | 设计影响 |
|:-----|:---------|:---------|
| **GPU数量** | 4-8卡（过多浪费） | 可降配到8卡以下，降低BOM |
| **GPU互联** | 低要求（单卡/小规模TP） | 可以PCIe直连，不需要NVSwitch |
| **内存** | 注重显存容量 > 带宽 | 高显存型号优先（如H200 141GB） |
| **存储** | 模型加载快，KV Cache 大 | NVMe + 大内存KV Cache方案 |
| **延迟要求** | P50<50ms, P99<200ms | 均衡设计，不过度配置网络 |
| **散热** | 非持续满载 | 可接受风冷（较低功耗） |
| **供电** | 负载波动大 | 关注能效曲线而非峰值效率 |
| **典型TCO分布** | GPU 60%, 电力 15%, 机房 10% | 优化GPU利用率比降低GPU单价更关键 |

**推荐配置**: 8卡H200(141GB) + 风冷/冷板可选 + 400G网卡

#### 场景B：训练（吞吐优先）

| 维度 | 需求特征 | 设计影响 |
|:-----|:---------|:---------|
| **GPU数量** | 16卡+（越多越好） | 高密度设计，16卡起步 |
| **GPU互联** | 全互联，高带宽 | NVSwitch/NVLink关键，或PCIe全拓扑 |
| **内存** | 带宽 > 容量 | HBM3e高带宽是关键限制因素 |
| **存储** | Checkpoint大，IO密集 | 分布式文件系统 + 本地NVMe缓存 |
| **散热** | 持续满载100% | 液冷必选项，风冷无法持续支撑 |
| **供电** | 稳定持续高功率 | 高效率PSU + 大功率冗余 |
| **网络** | 高速Scale-out | 800G RoCE/IB，多端口聚合 |
| **典型TCO分布** | GPU 75%, 电力 15%, 其他 10% | GPU间互联效率是训练吞吐的核心瓶颈 |

**推荐配置**: 16卡H100/B200 + 冷板液冷 + NVSwitch + 800G网络

#### 场景C：混合推理+训练（弹性场景）

| 维度 | 需求特征 | 设计影响 |
|:-----|:---------|:---------|
| **GPU数量** | 8-16卡灵活调度 | 可分区设计（部分训练部分推理） |
| **GPU互联** | 需兼顾两种模式 | **多拓扑设计**（可切换互联拓扑） |
| **散热** | 负载动态变化 | 动态散热控制（PID调参+液冷泵调速） |
| **供电** | 功率波动大 | 动态功率分配 + 超级电容平滑 |
| **网络** | 两者需求折中 | 考虑400G/800G共存方案 |
| **设计复杂度** | **最高** | 需要在两种场景之间找到平衡点 |

**设计哲学**: 混合场景是**最难的设计**，因为需要在矛盾的需求中找到平衡点。如果两种场景的需求冲突严重，**建议做两种专机而非一台通用机**。

#### 场景D：推理集群边缘节点

| 维度 | 需求特征 | 设计影响 |
|:-----|:---------|:---------|
| **功率密度** | 低，≤10kW/柜 | 2U/4U标准风冷 |
| **环境适应性** | 数据中心/机房边缘 | 宽温设计(0-45°C)，防尘 |
| **运维需求** | 少人/无人值守 | 远程管理完善，故障自愈 |
| **成本敏感度** | 极高 | 成本最优设计，性能够用即可 |
| **部署规模** | 大批量 | 统一配置，版本锁定 |

### 4.3 场景与配置的映射关系

```
场景 → GPU密度 → 散热方案 → 互联方案 → 供电方案 → 机型规格
在线推理: 4-8卡 → 风冷优先 → PCIe直连 → 标准AC → 2U/4U
训练: 16卡+ → 液冷必选 → NVSwitch → HVDC → 8U+
混合: 8-16卡 → 液冷必备 → 双拓扑 → HVDC → 8U
边缘: 2-4卡 → 风冷 → PCIe → 标准AC → 2U
```

## 五、已有资源分析 — 从零复用最大化

### 5.1 复用决策四步法

```
盘点已有资产 → 评估复用可行性 → 确定复用方式 → 验证复用效果
     │               │               │               │
     ├─ 硬件模块     ├─ 物理兼容性   ├─ 直接复用     ├─ 功能验证
     ├─ 设计图纸     ├─ 电气兼容性   ├─ 设计复用     ├─ 兼容性测试
     ├─ 固件/软件    ├─ 软件适配度   ├─ 参数复用     ├─ 回归测试
     └─ BOM/器件     └─ 供应链可用性  └─ 微调后复用    └─ 老化测试
```

### 5.2 可复用资产分类矩阵

| 资产类别 | 示例 | 复用潜力 | 复用风险 | 推荐策略 |
|:---------|:-----|:--------:|:--------:|:---------|
| **机箱结构件** | 机箱/托架/抬手/面板 | ★★★★★ | 低（物理兼容性已验证） | **直接复用** |
| **供电模块** | PSU/PDB/BBU | ★★★★☆ | 中（功率/接口需求变化） | 功率匹配后复用 |
| **管理模块** | BMC板/CPLD逻辑/BMC固件 | ★★★★☆ | 中低（CPU/GPU平台兼容） | 平台化复用 |
| **散热模块** | 风扇模组/液冷板/CDU | ★★★☆☆ | 中高（TDP变化需重新验证） | 微调后复用 |
| **背板/线缆** | NVMe背板/PCIe Riser/线缆 | ★★★☆☆ | 中（拓扑变化影响） | 拓扑相似可复用 |
| **主板设计** | 原理图/PCB Layout | ★★☆☆☆ | 高（每代需要改版） | **设计方法复用** |
| **测试用例** | 功能/性能/可靠性测试 | ★★★★★ | 低（软件资产天然可复用） | **强复用** |
| **生产治具** | 测试夹具/烧录工具 | ★★★★★ | 低 | 直接复用 |
| **软件/BMC/BIOS** | 固件基线/管理平台 | ★★★★☆ | 中（硬件变化需适配） | 基线+增量配置 |

### 5.3 模块复用优先级指南

```
优先级:      第一层            第二层              第三层
          直接复用资产      微调复用资产        设计方法复用
              │                │                   │
          ┌───┴───┐        ┌───┴───┐           ┌───┴───┐
          │       │        │       │           │       │
       机箱结构  测试用例  风扇模组  BMC固件   主板设计 PCB设计
       托架/抬手 测试脚本  PSU     CPLD逻辑   SI方案  散热方案
       线缆/标签  BOM模板
```

**关键原则**: 尽量在第一层和第二层解决问题，第三层复用是"不得已的复用"（因为需要重新设计验证）。高效研发团队的目标是将**第一层+第二层复用率提升到60%以上**。

### 5.4 基于复用的研发效率提升模型

| 复用率 | 新机型研发周期 | 新增测试工作量 | 风险等级 | 典型场景 |
|:------:|:-------------:|:--------------:|:--------:|:---------|
| **<30%** | 10-14个月 | 100% | 🔴 高 | 全新架构/平台切换 |
| **30-50%** | 7-10个月 | 60-80% | 🟡 中 | 半代升级 |
| **50-70%** | 5-7个月 | 30-50% | 🟢 低 | 相同平台的提配/降配 |
| **>70%** | 3-5个月 | <20% | 🟢 极低 | 配置变体/客户定制 |

## 六、面板接口与板卡信息定义 — 面向运维的设计

### 6.1 面板设计核心哲学

**"运维人员在紧急情况下，不应需要查阅手册就能操作这台机器"**

面板设计不是"有什么接口就放哪里"，而是按照**运维逻辑**组织：

```
监控运维视角的接口分类：
┌──────────────────────────────────────────────────┐
│  运维人员：                                           │
│  1. 插电 → 插网 → 上架 → 开机 → 调测 → 关机 → 下架  │
│                       ↓                               │
│  面板布局：                                           │
│  前面板（热区→频繁操作）：电源 + 管理网口 + 指示灯     │
│  后面板（冷区→低频操作）：业务网口 + 扩展卡 + 风扇     │
│  侧面/内部（仅维护时）：标签 + 序列号 + 维修门        │
└──────────────────────────────────────────────────┘
```

### 6.2 前面板定义规范

**使命**: 让运维人员**一眼看清当前状态，30秒完成基本操作**

| 区域 | 组件 | 数量 | 规格 | 设计要点 |
|:-----|:-----|:----:|:-----|:---------|
| **状态灯区** (左上角) | 系统状态LED | 1 | 三色(绿/黄/红) | 绿=正常, 黄=告警, 红=故障 |
| | 故障定位LED | 1 | 蓝色 | 运维人员定位用，软件可触发闪灯 |
| | 电源状态LED | 1 | 绿/灭 | 亮=供电正常 |
| | 硬盘活动LED | 1/盘 | 绿/闪 | 闪灯=读写中 |
| | 网络状态LED | 2/口 | 绿/黄 | 绿=Link, 闪=Activity |
| **操作区** (中部) | 电源按钮 | 1 | 带保护盖 | 防误触，有明确开关行程 |
| | 复位按钮 | 1 | 凹入式 | 防止误按，需工具操作 |
| | UID按钮 | 1 | 触发定位LED | 数据中心定位用 |
| | USB接口 | 2 | USB 3.0 Type-A | 调试/升级用，建议带标签盖 |
| **IO区** (侧下部) | 管理网口 | 1-2 | RJ45 10/100/1000 | **带外管理，独立于业务网络** |
| | 串口 | 1 | DB9/RJ45 | 调试用，BMC串口直连 |
| | VGA | 1 | DB15 | 可选，调试用 |
| | Micro USB | 1 | — | 本地BMC调试口（可选） |

### 6.6 板卡配置的Top-Down决策树

```
给定一个整机规格 → 需要哪些板卡？
                        │
          ┌─────────────┼─────────────┐
          │             │             │
      ┌───┴───┐    ┌───┴───┐    ┌───┴───┐
      │ 主板   │    │ 扩展板 │    │ 功能板 │
      │ (必选) │    │ (按需) │    │ (可选) │
      └───┬───┘    └───┬───┘    └───┬───┘
          │             │             │
    ┌─────┼─────┐  ┌───┴───┐    ┌───┴───┐
    │     │     │  │       │    │       │
  CPU板  IO板  PDB  PCIe  存储    BMC   CPLD
                    Riser  背板   板     板
```

**板卡定义建议原则**：

| 板卡类型 | 建议集成还是分离 | 理由 |
|:---------|:---------------|:-----|
| CPU+内存 | 集成在主板上 | 信号完整性关键，分离布线困难 |
| PCIe插槽 | 可分离（Riser卡） | 模块化后方便兼容不同GPU数量 |
| BMC | 集成或独立小卡 | 独立方便复用跨平台，集成省成本 |
| PSU接口 | 独立PDB板 | 电源布线独立，安全隔离 |
| 硬盘接口 | 独立背板 | 标准化接口，方便维护升级 |

## 七、六维联动决策矩阵（总表）

将六个维度整合为一个决策框架，用于**新机型设计启动时的系统性复盘**：

| 决策维度 | 输入来源 | 输出成果 | 关键负责人 | 评审节点 |
|:---------|:---------|:---------|:-----------|:---------|
| **① 友商分析** | 市场情报+竞品拆解+定价分析 | 竞品矩阵、差异化机会列表 | 产品管理(PM) | TR1前 |
| **② 整机规格** | 客户需求+友商对标+技术可行性 | PRD/规格书 v1.0 | 系统架构(SA) | TR1→TR2 |
| **③ 配置矩阵** | 规格书+市场需求+供应链能力 | 配置清单+SKU策略 | 产品管理(PM) | TR2→TR3 |

### A2 管理网口与业务网口区分设计

| 维度 | 管理网口 | 业务网口 |
|:-----|:---------|:---------|
| **物理位置** | 前面板（便于运维插拔） | 后面板（集中布线） |
| **颜色标识** | 黄色/黑色边框 | 蓝色/无边框 |
| **标签前缀** | MGT/MGMT | DATA/ETH |
| **速率** | 1G/10G | 10G/25G/100G/400G/800G |
| **协议** | HTTP/HTTPS/Redfish/IPMI | RoCE/IB/TCP |
| **网络隔离** | 独立管理VLAN | 业务网络 |
| **BMC绑定** | 带外管理（独立path） | NCSI共享（可选） |
| **冗余** | 1+1（可选） | 取决于业务需求 |

### A3 典型机型标签规范

```
┌──────────────────────────────────────┐
│  [厂商Logo]  [机型代号]  [Spec版本]   │
│                                       │
│  SN: Y23G012345678                    │
│  MAC1: 00:1A:2B:3C:4D:5E            │
│  MAC2: 00:1A:2B:3C:4D:5F            │
│  资产号: ZC-2026-000123               │
│                                       │
│  生产日期: 2026-05-15  |  产地: 深圳  │
│  [Barcode: SN]  [Barcode: 资产号]     │
└──────────────────────────────────────┘
```

**三码合一原则**: SN(整机序列号) = 资产号(客户系统) = Barcode(仓储系统)，任一个编码可以追溯到其他两个。
