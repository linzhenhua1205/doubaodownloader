# 📐 LLM 推理的 PD 分离部署设计

> PD 分离（Prefill-Decode Disaggregation）将 LLM 推理的传统「前向传播全流程」拆分为 Prefill 和 Decode 两个独立阶段，分别部署在异构 GPU 集群上。本文详细讨论为什么要做、怎么做、以及落地中的关键工程决策。

---

## 1. 为什么需要 PD 分离？

### 1.1 传统「单阶段」推理的核心矛盾

传统 LLM 推理中，每个 GPU 同时承担 Prefill 和 Decode：

```
  请求到达 → Prefill (GEMM 密集) → Decode (Memory 密集) → 输出
                 ↑                                    ↑
           GPU 利用率 80-95%                     GPU 利用率 10-30%
```

| 阶段 | 计算特性 | 利用瓶颈 | 典型利用率 |
|:----|:--------|:--------|:---------:|
| **Prefill** | Compute-bound, GEMM 密集 | 算力 | 80-95% |
| **Decode** | Memory-bound, KV 读取密集 | HBM 带宽 | 10-30% |

**问题**：当同一 GPU 混合处理 Prefill 和 Decode 时，会出现严重的**资源争抢**和**利用率不平衡**：

- 新请求的 Prefill（需要大量算力）与已有请求的 Decode（需要 HBM 带宽）争抢 GPU
- Decode 阶段的低利用率意味着大量 GPU 算力被浪费
- Prefill 延迟（TTFT）和 Decode 延迟（TPOT）互相影响，无法独立优化

### 1.2 PD 分离的核心思路

```
  请求 → Prefill 集群 (集中处理 Prefill)   →  KV Cache 传输  →  Decode 集群 (持续 Decode)
         ┌───────────┐                      ┌───────────┐
         │ Compute   │  KV Cache via GDA    │ Memory    │
         │ Optimized │ ──────────────────→  │ Optimized │
         │   (FP8)   │                      │ (混合精度) │
         └───────────┘                      └───────────┘
             80-95% util                         60-80% util
```

**三个核心动机**：

| 动机 | 说明 |
|:----|:-----|
| 🎯 **独立资源扩缩** | Prefill 集群扩算力（加 GPU），Decode 集群扩容量（加 HBM） |
| 💰 **异构硬件** | Prefill 用高算力 GPU（如 H100/B200），Decode 用大显存 GPU（如 Gaudi 3） |
| 🏎️ **独立延迟优化** | TTFT 和 TPOT 不再互相耦合，可分别调优 |

## 2. PD 分离的三种架构模式

### 2.1 模式 A：节点级分离（Node-Level）

```
  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
  │ Router      │     │ Router      │     │ Router      │
  │ (负载均衡)   │     │ (负载均衡)   │     │ (负载均衡)   │
  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
         │                   │                   │
  ┌──────▼──────┐     ┌──────▼──────┐     ┌──────▼──────┐
  │ Prefill 集群│     │ Decode 集群 │     │ Decode 集群 │
  │ 2-4 Nodes   │     │ 16-32 Nodes │     │ 16-32 Nodes │
  │ (高算力)     │     │ (大容量)     │     │ (大容量)     │
  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
         │  KV Cache         │                   │
         └───────────────────┴───────────────────┘
                             │
              RoCE / InfiniBand Fabric
```

| 维度 | 描述 |
|:----|:-----|
| **粒度** | 整节点（8 GPU 为单位） |
| **灵活性** | 中（节点级调配） |
| **网络开销** | 低（同 Fabric 内传输） |
| **适用场景** | 中等规模集群（32-256 GPU） |
| **KV 传输** | 节点间 GDA RDMA / NCCL |

### 2.2 模式 B：GPU 级分离（GPU-Level）

```
  同一 Node 内的 PD 混部（High-Level）
  
  ┌──────────────────────────────────────┐
  │  GPU 0   GPU 1   GPU 2   GPU 3      │
  │  Prefill Prefill Decode  Decode     │
  │  (TP=2)   (TP=2)   (TP=2)   (TP=2)  │
  │                                      │
  │  GPU 4   GPU 5   GPU 6   GPU 7      │
  │  Prefill Decode  Decode  Decode     │
  │  (TP=2)   (TP=2)   (TP=2)   (TP=2)  │
  └──────────────────────────────────────┘
  │◄─────── NVLink / GDA 域 ────────────►│
```

| 维度 | 描述 |
|:----|:-----|
| **粒度** | 单 GPU（细粒度） |
| **灵活性** | 高（动态调整 Prefill/Decode 配比） |
| **网络开销** | 极低（片内通信） |
| **适用场景** | 小集群（<32 GPU）或实验环境 |
| **KV 传输** | NVLink 域内直接读写 |

**优势**：可以在同一节点内动态调整 Prefill/Decode GPU 比例，响应流量波动。
**限制**：受限于单节点 GPU 数量，扩展能力有限。

### 2.3 模式 C：分层分离（Hierarchical）

```
    全域负载均衡 (Global Router)
           │
    ┌──────┴──────┐
    │  区域 Router  │
    └──────┬──────┘
           │
    ┌──────┴──────┐
    │  Prefill 池  │
    │  (多节点)    │
    │  KV Cache → │─────────────┐
    │  分发服务   │             │
    └──────────────┘             │
                          ┌─────┴──────────┐
                          │ Decode 池 1    │
                          │ (区域 A)       │
                          └────────────────┘
                          ┌─────┴──────────┐
                          │ Decode 池 2    │
                          │ (区域 B)       │
                          └────────────────┘
```

| 维度 | 描述 |
|:----|:-----|
| **粒度** | 池级（全局调配） |
| **灵活性** | **最高**（跨区域池化） |
| **网络开销** | 高（跨区域传输） |
| **适用场景** | 大规模集群（>256 GPU）、多可用区 |
| **KV 传输** | 需分层缓存 + 全局调度 |

## 3. KV Cache 传输：PD 分离的关键技术

### 3.1 传输路径设计

PD 分离中，Prefill 节点生成的 KV Cache 需要传输到 Decode 节点。这是**影响端到端延迟的核心因素**。

```
  传输策略一览：

  A. 全量传输 (Eager)：
     Prefill 完成 → 立即传输完整 KV Cache → Decode 开始
     ✔ 简单直接    ✘ 等待时间长，TTFT = Prefill + 传输时间

  B. 流式传输 (Streaming)：
     Prefill 生成了部分 KV → 立即开始传输 → Decode 边接收边处理
     ✔ TTFT 更低    ✘ 实现复杂，需处理乱序

  C. 预取传输 (Prefetch)：
     预测下一个请求 → 预先将热门 KV 放到 Decode HBM
     ✔ 零等待时间    ✘ 需要准确的预测
```

### 3.2 KV Cache 的序列化与压缩

```
  ┌─────────┐     ┌──────────┐     ┌─────────┐     ┌─────────┐
  │ KV 原始 │────→│ 量化压缩 │────→│ 序列化  │────→│ RDMA 发送│
  │ (FP16)  │     │ (FP8/INT8)│    │ (Flat)  │     │         │
  └─────────┘     └──────────┘     └─────────┘     └─────────┘
                                                     │
                                                     ▼
                                           ┌─────────────────┐
                                           │ KV Cache 传输   │
                                           │ 带宽占用推估     │
                                           └─────────────────┘
```

**KV Cache 传输带宽推估**（以 70B 模型为例）：

| 上下文长度 | KV 大小 (FP16) | KV 大小 (FP8) | 传输延迟 @200Gbps | 传输延迟 @800Gbps |
|:---------:|:-------------:|:------------:|:----------------:|:----------------:|
| **4K** | ~1.5 GB | ~0.75 GB | ~30ms | ~7.5ms |
| **32K** | ~12 GB | ~6 GB | ~240ms | ~60ms |
| **128K** | ~48 GB | ~24 GB | ~960ms | **~240ms** |
| **1M** | ~384 GB | ~192 GB | ~7.7s | **~1.9s** |

> 🔑 **关键结论**：长上下文下，KV Cache 传输延迟是 PD 分离的最大瓶颈。必须配合流式传输 + 量化压缩 + GDA 直接通道。

### 3.3 传输优化技术

| 技术 | 效果 | 实现复杂度 |
|:----|:---:|:--------:|
| **FP16 → FP8 量化** | 带宽减半，几乎无精度损失 | 🟢 低 |
| **INT4 量化** | 带宽减 75%，有精度影响 | 🟡 中 |
| **KV Cache 稀疏化** | 跳过不重要 Token 的 KV | 🔴 高 |
| **Streaming 流水线** | 传输与 Decode 重叠 | 🟡 中 |
| **RDMA 直接写入** | 零拷贝，跳过 PCIe | 🟢 低（GDA 原生） |
| **Async Pipeline** | Prefill 边算边发 | 🔴 高 |

## 4. 调度与负载均衡设计

### 4.1 请求调度流程

```
  用户请求
      │
      ▼
  ┌────────────────────────────────────┐
  │ 全局 Router                        │
  │ 1. 解析请求 (模型/上下文长度/优先级) │
  │ 2. 选择 Prefill 节点               │
  │    - 负载最低的 Prefill 节点        │
  │    - 或与用户最近的 Prefill 节点    │
  └──────────────┬─────────────────────┘
                 │
                 ▼
  ┌────────────────────────────────────┐
  │ Prefill 节点                       │
  │ 1. 执行 Prefill 计算               │
  │ 2. 生成 KV Cache                   │
  │ 3. 向 Router 报告 KV 元数据        │
  │ 4. 等待 Router 分配 Decode 节点    │
  └──────────────┬─────────────────────┘
                 │
                 ▼
  ┌────────────────────────────────────┐
  │ Router (第二次决策)                │
  │ 1. 检查所有 Decode 节点的 HBM 余量 │
  │ 2. 选择最优 Decode 节点            │
  │    - 有足够空间存放 KV Cache       │
  │    - 负载最低                      │
  │    - 尽量与已缓存该用户 KV 的同节点  │
  │ 3. 通知 Prefill 发送 KV Cache     │
  └──────────────┬─────────────────────┘
                 │
                 ▼
  ┌────────────────────────────────────┐
  │ Decode 节点                        │
  │ 1. 接收 KV Cache (GDA RDMA)       │
  │ 2. 开始 Decode                     │
  │ 3. 每步输出 Token                  │
  │ 4. KV Cache 保留在 HBM             │
  └────────────────────────────────────┘
```

### 4.2 关键调度策略

#### 策略 1：Cache-aware 调度

```
  Decode 节点状态表 (Router 维护)：
  
  ┌─────────┬────────────┬───────────┬───────────┐
  │ Node ID │ Free HBM   │ 已缓存 User│ 当前负载  │
  ├─────────┼────────────┼───────────┼───────────┤
  │ D-01    │ 45 GB      │ User A, C │ 3 个活跃  │
  │ D-02    │ 62 GB      │ User B    │ 1 个活跃  │  ← 最优
  │ D-03    │ 28 GB      │ —         │ 5 个活跃  │
  └─────────┴────────────┴───────────┴───────────┘
```

**原则**：
- 同一个用户的连续请求 → **优先分配到同一 Decode 节点**（避免重复传输 KV）
- 如果 Decode 节点已缓存该用户的 KV → 直接开始 Decode，零传输
- 如果 Decode 节点 HBM 不足 → 触发 KV Cache Offloading

#### 策略 2：Batching-aware 调度

```
  Decode 节点的 Batching 效率：
  
  同批次请求 → 共享 Decode 节点 → Attention 合并
  (提高 GPU 利用率)
  
  请求 A (User X): "HBM 剩余 0G"  ← 快结束了
  请求 B (User Y): "HBM 剩余 80G"  ← 刚来
  
  调度决策：将 B 分配给 A 所在节点，利用 A 释放的空间
```

#### 策略 3：弹性 Prefill/Decode 比例

根据实时流量动态调整：

```
  流量类型          Prefill:Decode 比    说明
  多短请求              1:1~1:2         大量小 Prompt
  多长文档              1:1             每请求需大量计算
  多轮对话 Agent        1:4~1:8         持续 Decode，少 Prefill
  混合负载              动态调整         根据队列深度实时调整
```

## 5. 容量规划与硬件配比

### 5.1 Prefill/Decode 配比公式

给定模型 $M$、上下文长度 $L$、请求速率 $R$、目标 TTFT 和 TPOT：

```
设：
  P_prefill = 单 GPU 的 Prefill 吞吐 (tok/s)
  P_decode  = 单 GPU 的 Decode 吞吐 (tok/s)
  L_avg     = 平均 Prompt 长度 (tokens)
  G_avg     = 平均 Generate 长度 (tokens)
  R         = 请求速率 (req/s)

所需 Prefill GPU = ceil(L_avg × R / P_prefill)
所需 Decode GPU  = ceil(G_avg × R / P_decode)
推荐配比         = Decode_GPU : Prefill_GPU
```

**示例（70B 模型, Gaudi 3）**：

| 场景 | L_avg | G_avg | R | Prefill GPU | Decode GPU | 配比 |
|:----|:----:|:----:|:-:|:----------:|:---------:|:----:|
| 聊天 | 512 | 1024 | 100 | 1 | 4 | **1:4** |
| 文档分析 | 4096 | 512 | 50 | 4 | 4 | **1:1** |
| 代码生成 | 2048 | 2048 | 80 | 4 | 12 | **1:3** |
| Agent 对话 | 1024 | 8192 | 30 | 2 | 16 | **1:8** |

### 5.2 不同模型规模的推荐配比

| 模型 | 权重 (FP8) | 每 GPU KV (128K) | 推荐 Prefill GPU | 推荐 Decode GPU | 总 GPU | 配比 |
|:----|:---------:|:---------------:|:--------------:|:--------------:|:-----:|:----:|
| **7B Dense** | ~7 GB | ~12 GB | 1 | 2-4 | 3-5 | 1:2~1:4 |
| **70B Dense** | ~70 GB | ~12 GB | 1-2 | 4-8 | 5-10 | 1:4 |
| **405B Dense** (~350B active) | ~200 GB | ~12 GB | 4-8 | 16-32 | 20-40 | 1:4 |
| **671B MoE (~37B active)** | ~335 GB | ~12 GB | 2-4 | 16-32 | 18-36 | 1:8 |
| **1T MoE (~100B active)** | ~500 GB | ~12 GB | 8-16 | 32-64 | 40-80 | 1:4 |

> ⚠️ **重要说明**：以上数值为估算，实际配比取决于具体部署环境中的 Token 吞吐目标、延迟要求和流量模式。

## 6. 性能优化决策树

### 6.1 何时应该采用 PD 分离？

```
  我的集群是否应该做 PD 分离？
  
  请求量 > 10 req/s? 
  ├── 否 → 单阶段推理就够了
  └── 是 → 继续
      
  平均上下文 > 4K tokens?
  ├── 否 → 单阶段推理就够了
  └── 是 → 继续
      
  Prefill/Decode 利用不平衡 > 2x?
  ├── 否 → 单阶段推理可能可行
  └── 是 → 继续
      
  硬件资源可以独立扩缩?
  ├── 否 → 考虑 GPU 级分离 (模式 B)
  └── 是 → 继续
      
  集群规模 > 32 GPU?
  ├── 否 → GPU 级分离 (模式 B)
  └── 是 → 节点级分离 (模式 A)
  
  集群规模 > 256 GPU?
  ├── 否 → 节点级分离 (模式 A)
  └── 是 → 分层分离 (模式 C)
```

### 6.2 性能瓶颈定位

```
  端到端延迟 = TTFT + TPOT × Generate_Length
  
  如果 TTFT 过高:
  ├── Prefill 节点不够 → 增加 Prefill GPU
  ├── KV Cache 传输慢 → 启用 Streaming / 压缩
  └── Router 调度慢 → 优化调度策略
  
  如果 TPOT 过高:
  ├── Decode 节点不够 → 增加 Decode GPU
  ├── HBM 带宽饱和 → 检查 batch size/KV 碎片
  └── KV 传输延迟 → 预取 / Cache-aware 调度
  
  如果吞吐不足:
  ├── Prefill 瓶颈 → 加 Prefill GPU 或升级算力
  ├── Decode 瓶颈 → 加 Decode GPU 或增大 Batch
  └── 网络瓶颈 → 升级互联带宽 / 启用压缩
```

## 7. 实际部署中的注意事项

### 7.1 容错设计

| 失效场景 | 影响 | 容错方案 |
|:--------|:----|:--------|
| Prefill 节点宕机 | 新请求无法处理 | 主备 Router + 自动故障转移 |
| Decode 节点宕机 | 活跃请求中断 | KV Cache 多副本 + 快速迁移 |
| KV 传输超时 | 单个请求超时 | 重试 + 备用 Decode 节点 |
| Router 宕机 | 全集群不可用 | Router 主备 + 一致性哈希 |

### 7.2 KV Cache 生命周期管理

```
  请求到达 → Prefill → KV 传输 → Decode → 请求结束
                                  │
                              KV Cache 存活策略：
                              ├── 立即释放（标准模式）
                              ├── 保留 N 秒（等待用户下一轮对话）
                              ├── 保留到 HBM 满（LRU 淘汰）
                              └── Offload 到 CXL 池（长上下文 Agent）
```

**建议策略**：
- **短上下文 (≤4K)**：立即释放，KV Cache 小，重新计算代价低
- **中等上下文 (4K-32K)**：保留 30-60 秒，兼顾多轮对话体验
- **长上下文 (＞32K)**：Offload 到 CXL 池，按需加载
- **Agent 场景**：保留整个 Session 的 KV 在 CXL 池，HBM 只保持窗口

### 7.3 PD 分离的成本收益分析

```
  收益：
  ✅ 相同 GPU 数量下，PD 分离可提升系统吞吐 30-70%
  ✅ TTFT 降低 40-60%（独立 Prefill 集群算力集中）
  ✅ TPOT 降低 20-30%（Decode 无 Prefill 干扰）
  ✅ 资源利用率提升 15-25%（Prefill 高利用，Decode 也提升）
  
  成本：
  ❌ 增加 KV Cache 传输延迟（长上下文下显著）
  ❌ 增加系统复杂度（Router、调度、KV 管理）
  ❌ 增加网络带宽消耗（KV 传输）
  ❌ 需要更精细的监控和调优
```

## 8. 总结

PD 分离不是银弹——它在特定场景下效果显著，但也引入了额外的复杂度和网络开销。

**✅ 特别适合**：
- 长上下文推理（>4K tokens）
- 高 QPS 场景（>10 req/s）
- Agentic AI 的多轮对话
- 需要独立优化 TTFT 和 TPOT 的场景

**❌ 不太适合**：
- 短上下文小模型（<7B, <1K tokens）
- 低 QPS 实验环境
- 单节点即可满足需求的小规模部署

**核心工程原则**：

1. **KV Cache 传输是 PD 分离的最大瓶颈**——优先优化传输路径
2. **Cache-aware 调度比算力调度更重要**——避免重复传输 KV
3. **配比不是固定的**——根据负载动态调整 Prefill:Decode 比例
4. **从模式 B（GPU 级）开始**——先在同一节点内验证，再扩展到跨节点
5. **始终考虑 Fallback**——PD 分离架构应有退化到单阶段的能力
