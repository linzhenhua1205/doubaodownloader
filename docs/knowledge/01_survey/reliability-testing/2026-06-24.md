## 1️⃣ 🏗️ ReCoVer：512 GPU 韧性 LLM 预训练系统（268 核失效仍保训练轨迹）

**arXiv:2605.11215** — 提交 2026-05-11，更新 2026-05-21

**核心贡献**：首个将 LLM 预训练容错从"恢复后不偏离"提升到"每一步梯度与无故障等效"的系统。

**关键设计**：
- **唯一不变式**：每个迭代保持微批次数恒定，确保每步梯度与无故障运行随机等价
- 三层解耦协议：(1) **容错集合通信** — 隔离故障不跨副本传播；(2) **步内细粒度恢复** — 保持迭代内进展，防止梯度污染；(3) **灵活工作负载策略** — 动态在幸存 GPU 之间重分配微批次配额
- **并行性无关**：直接集成 3D 并行和 HSDP，作为 drop-in 层

**实验结果**（512 GPU 端到端预训练）：
- 尽管运行过程中 **256 块 GPU 陆续失效**，仍然保持与无故障参考相同的训练轨迹
- 对比 checkpoint-restart 基线：**2.23× 更高有效吞吐量**
- 在 234 GPU-小时内处理 **多 74.9% 的 token**，差距随训练时间扩大

**意义**：证实了 50% GPU 失效率下 LLM 预训练仍可通过容错集合 + 微批重分配维持等效收敛，大幅刷新了 CR 范式在大规模训练中的适用边界。

## 2️⃣ 👻 Ghost in the Datacenter：链路抖动与拓扑认知失效 — 大型数据中心可靠性实证

**arXiv:2603.03736** — 提交 2026-03-04

**核心论点**：每一次链路断开或抖动都损坏网络的自认知（graph），产生"幽灵"——表现为可达但实际不可达的节点、显示 up 但静默丢包的链路、解析到已分区机器的 IP 地址。

**生产实证数据**：
| 来源 | 规模 | 核心数据 |
|:-----|:-----|:---------|
| Meta | LLaMA 3 训练 | **54 天内 419 次中断** |
| ByteDance | 3 个月生产 | **38,236 次显式故障 + 5,948 次隐式故障** |
| Google | TPUv4 OCS | 光路交换网络中断实证 |
| Alibaba | 集群网络 | 0.057% NIC-ToR 链路月故障率 |

**规模放大**：在 2025 年集群规模（~300 万 GPU，>1000 万光链路）下，**平均每 48 秒发生一次链路抖动**。

**根本原因诊断**：所有现有缓解方案（Phi Accrual、SWIM、BFD、OSPF/ISIS 快速收敛、SmartNIC offload、RoCE/PFC 无损以太网、Kubernetes pod eviction）本质上都基于 **超时机制（Timeout And Retry, TAR）**，而 TAR 无法区分"慢"和"死"——这正是 FLP 不可能定理证明在异步系统中不可解的问题。

**连接**：将幽灵故障与灰色故障（Huang et al., HotOS 2017）和元稳定故障（Bronson et al., HotOS 2021, 跨 11 家组织 22 个故障验证，OSDI 2022）建立统一框架。

**意义**：提供了目前最全面的数据中心链路级可靠性实证，以 48 秒/次的频率将 TAR 根本矛盾推到工程不可忽视的程度。

## 3️⃣ 🚆 TrainMover：中断弹性 LLM 训练运行时（1,024 GPU ~20s 恢复）

**arXiv:2412.12636** — 提交 2024-12-17，更新 2026-05-15

**核心贡献**：利用弹性/备用机器实现中断处理，**零内存开销**、极短停机时间。

**三项关键技术**：
- **两阶段、基于 delta 的通信组建立** — 避免完全重建
- **免通信沙箱预热** — 恢复过程不占用网络带宽
- **通用备用设计** — 任何角色故障均可恢复

**实验结果**：
- **1,024 GPU** 规模下各类中断停机时间 **~20 秒**
- 在 **64K GPU** 规模下预测比最佳替代方案减少 **55% 的浪费 GPU 小时**
- 每周可节省 **140 万 GPU-小时**

**意义**：将中断恢复从分钟级推入秒级，使"备用机器 + 弹性恢复"模式在千卡~万卡区间成为可行的 CR 替代方案。

## 4️⃣ 🔄 R²CCL：可靠韧性集合通信库（<1% 训练开销，47× 超基线）

**arXiv:2512.25059** — 提交 2025-12-31

**问题**：现代 ML 训练/推理扩展到数万 GPU，网络故障浪费 **10-15% GPU 小时**。常见链路错误和抖动触发超时，导致整个作业终止，训练需昂贵 CR 回滚，推理需重新处理请求。

**方案**：R²CCL 利用多 NIC 硬件实现无损、低开销故障切换：
- **快速连接迁移** — 故障 NIC 流量秒级切换
- **带宽感知负载重分配** — 保持集体操作效率
- **韧性集合算法** — 故障下维持训练/推理进度

**实验结果**（8-GPU H100 + IB 双服务器 + 大规模 ML 模拟器）：
- 训练开销 **<1%**
- 推理开销 **<3%**
- 超越 AdapCC **12.18×**，超越 DejaVu **47×**

**意义**：证明在集合通信层面实现透明容错是可行的，为万卡集群的 NIC 级链路故障提供了秒级不中断替代方案。
