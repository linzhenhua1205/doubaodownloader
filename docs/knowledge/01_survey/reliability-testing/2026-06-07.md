## 🆕 1. SHIFT — 跨 NIC RDMA 容错的理论边界与工程实现

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2512.11094, 2025.12 → 2026.02 v2 |
| **作者** | Lin et al. (HKUST + UW + Microsoft Research) |
| **领域** | RDMA 网络容错 / 集合通信可靠性 |

### 🎯 核心贡献

**理论发现**：首次证明跨 NIC（Cross-NIC）RDMA 故障切换的**三元悖论（Trilemma）**——无法同时满足：
1. **Exactly-Once 执行语义** — 数据包不重不丢
2. **Receiver-NIC Opacity** — 接收端 NIC 状态对上层透明
3. **Zero-Copy 数据通路** — 无内核拷贝

三者最多得其二。

**工程突破**：观察到 NCCL 等主流训练框架使用**幂等批量传输**，容忍**松弛内存序**（仅需保持 notification ordering）。基于此设计 **SHIFT**，一个用户态 RDMA 层，在 rdma-core 中实现跨 NIC 容错。

### 📊 实验数据

| 指标 | 值 |
|:-----|:----|
| 正常运行时开销 | **可忽略**（negligible） |
| 故障场景 | 致命 NIC 故障 / 链路异常 → **无缝掩蔽** |
| 评估环境 | PyTorch 分布式训练 |
| 恢复方式 | 无需重启训练，无需 Checkpoint Rollback |

### 💡 为何重要
- 与 R²CCL（容错通信库，06-05 已跟踪）形成**互补**：R²CCL 聚焦多 NIC 连接迁移 + 带宽重分配，SHIFT 从 RDMA 协议层面揭示理论边界
- 填补了 **RDMA 容错的理论空白**——第一个系统级 cross-NIC failover 方案
- 幂等传输+松弛序的观察适用于主流框架，实际部署门槛低

## 🆕 2. MeCeFO — 轻量计算高效容错优化算法

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2510.16415, 2025.10 |
| **会议** | **NeurIPS 2025 Poster** 🏆 |
| **作者** | Hu et al. (PKU + Alibaba) |

### 🎯 核心贡献

提出 **MeCeFO**（Memory- and Computation-efficient Fault-tolerant Optimization），一种**算法级**容错方案，节点故障时无缝迁移训练任务到邻居节点，通过三种轻量技巧最小化额外负载：

1. **Skip-connection**（关键创新）：故障节点补训练时，反向传播**跳过 MHA（Multi-Head Attention）模块**，仅保留 FFN 梯度计算。理论依据：MHA 计算量大但参数增量小
2. **Recomputation**：FFN 部分仅重计算激活而非全量存储，减少峰值激活内存
3. **Low-rank gradient approximation**：FFN 权重矩阵梯度用低秩近似高效估计

### 📊 实验数据

| 指标 | 值 |
|:-----|:----|
| 高故障率下吞吐下降 | **仅 4.18%** |
| 对比 SOTA 韧性提升 | **5.0× — 6.7×** |
| 收敛速度理论保证 | **O(1/√(nT))** — 与传统分布式训练同阶 |
| 额外资源需求 | **0**（不需要备用节点或显存预留） |

### 💡 为何重要
- 区别于系统级容错（检查点/通信库），MeCeFO 是**优化算法级**方案，与检查点/通信容错**正交可叠加**
- Skip-connection 跳过 MHA 的设计巧妙利用了 LLM 训练中 MHA 梯度稀疏性——与 LLMTailor 按层裁剪（06-05）形成互补：一个从内容裁剪检查点，一个从计算路径减负
- NeurIPS 2025 接收证明了学术界的认可

## 🆕 3. SKT × Lablup 504 GPU 生产集群故障运营实证分析

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2605.09370, 2026.05 (42页技术报告) |
| **作者** | Kang et al. (SKT, Upstage, Lablup, NVIDIA Korea, VAST Data 五方联合) |
| **亮点** | 跨国界、跨组织的真实生产数据共享与联合诊断示范 |

### 🎯 核心贡献

基于 **63 节点 NVIDIA B200 集群（504 GPU）** 的 **55 天 Prometheus 时序数据 + 73 天运营日志 + 224 次多节点训练任务**，给出了**罕见的公开生产级故障实证分析**，三个定量分析得出四项发现：

**发现一：多信号 GPU 故障检测**
- 751 个 Prometheus 指标 + 10 次 XID GPU 故障 → **10/10 检测率**（2/10 可在 XID 触发前检出）
- 误报 ~**0.84/天**
- 结论：无单一指标对所有故障类型一致有效 → **多信号融合策略**才是务实路径

**发现二：检查点带宽悖论**
- 523 次检查点事件从 GPU VRAM → NFS 全路径剖析
- 200Gbps RoCE 链路上实际利用率仅 **1.4% — 10.4%**
- 根因：**128 槽 NFS RPC 层饱和**，GPU/NIC 端远未到达瓶颈
- 💡 这与 TierCheck（内存内检查点，06-03）和 Cleave（边缘训练容错，06-04）的优化方向一致——检查点瓶颈不在网络带宽而在存储栈软件层

**发现三+四：故障分布与恢复统计**
- 故障高度集中于少数节点：**Top 3/63 节点占 >50%** 所有排除事件
- 自动重试链成功率 **33.3%**（12 链/73 次尝试），手工重恢复率仅 **12.5%**
- 中位重试间隔 **11 分钟**（IQR 10-11）

### 📊 关键数据

| 指标 | 数据 |
|:-----|:-----|
| 集群规模 | 63 节点 × B200 = 504 GPU |
| 监控周期 | 55 天 Prometheus + 73 天日志 |
| 训练任务 | 224 次多节点任务 |
| GPU 故障检测率 | 10/10（2/10 故障前预警）|
| 误报率 | ~0.84/天 |
| 检查点 RoCE 利用率 | **1.4% — 10.4%** |
| Top-3 故障节点集中度 | **>50%** |
| 自动重试成功率 | **33.3%** vs 手工 **12.5%** |

### 💡 为何重要
- 罕见的**生产级公开数据**——大多数容错研究基于仿真/小规模实验，这篇来自 504 GPU 真实预训练
- 跨组织协作（韩国的 SKT + Upstage + Lablup + NVIDIA Korea + VAST Data）的运作模式本身具有示范意义
- 检查点带宽悖论直接指导工程优化方向：**与其升级网络，不如优化 NFS RPC 层**
- 故障节点高度集中（二八法则）为**主动更换/隔离策略**提供了运营数据支持
