## 🆕 1. GhostServe — KV Cache 纠删码影子检查点：推理容错的「零重量」方案 (MLSys 2026)

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2605.00831, 2026-03-26 | **MLSys 2026 接收** 🏆 |
| **作者** | Shakya Jayakody, Youpeng Zhao, Chinmay Dhanraj Nehate, Jun Wang (UCF) |
| **领域** | LLM 推理容错 / KV Cache 保护 / 纠删码检查点 |

### 🎯 核心贡献

**问题：** 百万 token 级长上下文推理和 Agent 任务的 KV Cache 非常巨大且**有状态**。一旦设备故障，KV Cache 丢失需要**全量重计算**才能恢复，代价极其高昂。

**GhostServe 方法：** 一种轻量级「影子」检查点方案，利用**纠删码（erasure coding）**为流式 KV Cache 生成**校验分片（parity shards）**并存储到主机内存（host memory），在设备故障时可直接**原位重建**丢失的 KV Cache：

1. **Shadow 架构**：主路径零侵入——KV Cache 正常在 GPU 上生成和使用，纠删码校验计算在**后台异步执行**
2. **主机内存存储**：校验分片存放在 CPU DDR 而非远端存储，恢复路径极短
3. **无缝重建**：故障时从校验分片恢复丢失的 KV Cache，推理流无需中断

**与现有方案对比：**

| 方案 | 保护方式 | 恢复代价 |
|:----|:--------|:---------|
| **镜像复制（replication）** | 双份 GPU 内存 | 2× GPU 内存开销 |
| **周期性全量检查点** | CPU/存储 | 高频→I/O 瓶颈；低频→丢失窗口长 |
| **GhostServe** ✅ | 纠删码 → 主机内存 | **无额外 GPU 内存，恢复无需重计算** |

### 📊 实验数据

| 指标 | GhostServe | SOTA 方案 | 改进 |
|:-----|:----------|:---------|:----|
| **检查点延迟（单 batch）** | — | — | **↓ 2.7×** |
| **恢复延迟（单 batch）** | — | — | **↓ 2.1×** |
| **中位响应延迟（故障时）** | — | — | **↓ 1.2×** |
| 额外 GPU 内存 | **无** | 2×（replication） | — |
| 恢复路径 | **本地主机内存** | 远端存储 | 最短 |

### 💡 为何重要

1. **推理容错的新战场**：此前容错研究几乎全聚焦**训练**（检查点/灾备/弹性），GhostServe 首次系统性解决**推理服务**的容错问题——在大规模 Agent 推理部署中，KV Cache 丢失将导致 `O(N)` 级别的全部重计算
2. **纠删码在推理场景的巧妙应用**：纠删码常用于存储系统（RS 码），GhostServe 将其迁移到 KV Cache 的保护上，既避开 2× GPU 内存开销（replication 的死穴），又绕开了存储级检查点的 I/O 瓶颈
3. **MLSys 2026 接收**说明该工作在 ML 系统社区获得认可，**可能成为 LLM 推理服务高可用的重要参考方案**
4. **与推理框架的独立集成性**：GhostServe 设计上作为推理系统的**可插拔子系统**，理论上可与 vLLM/TensorRT-LLM 等主流框架集成

## 🆕 2. Crab — AI Agent 沙箱语义感知检查点：75% 的检查点都是浪费的

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2604.28138, 2026-04-30 |
| **作者** | Tianyuan Wu, Chaokun Chang, Lunxi Cao, Wei Gao, Wei Wang (HKUST) |
| **领域** | Agent 沙箱 / 操作系统级检查点 / 语义感知容错 |

### 🎯 核心贡献

**问题：** AI Agent 通过 sandboxed 容器和 microVM 执行，其状态跨文件系统、进程和运行时产物。现有方案陷入两难：
- **应用层恢复**：只保存聊天历史（chat history），忽略 OS 副作用 → **恢复正确率仅 8%**
- **全量检查点**：每轮（per-turn）保存完整状态 → **正确但开销巨大**，无法密集共置

**根因：** Agent-OS **语义鸿沟（semantic gap）**：
- Agent 框架看到的是**工具调用**，但不了解其 OS 侧效应
- OS 看到的是**所有状态变更**，但不知道哪些与当前 turn 的恢复相关

**颠覆性发现：** 超过 **75% 的 Agent turns 不产生任何与恢复相关的状态变化**——即绝大多数检查点都是浪费的。

**Crab 方法：** 一个透明的主机侧运行时（host-side runtime），桥接语义鸿沟：

| 组件 | 功能 |
|:----|:------|
| **eBPF Inspector** 🔍 | 分类每个 turn 的 OS 可见效应，动态决定检查点粒度（full/diff/skip） |
| **Coordinator** 🔄 | 对齐检查点与 turn 边界，利用 LLM 等待时间**重叠执行 C/R** |
| **Host-scoped Engine** 🎛️ | 在共置沙箱间调度检查点流量，避免 I/O 竞争 |

### 📊 实验数据

| 指标 | Chat-only | Full per-turn | Crab | 改进 |
|:-----|:---------|:-------------|:----|:----|
| **恢复正确率** | 8% | 100% | **100%** | 8% → 100% ✅ |
| **检查点流量** | — | 100% (基线) | **↓ 87%** | **7.7× 减少** |
| **执行开销** | 0% (不可靠) | ~5-10% (预期) | **< 1.9%** | **接近零开销** |

**测试负载：** Shell-intensive + code-repair 两类 Agent 工作负载。

### 💡 为何重要

1. **Agent 容错的「正确 + 高效」首次统一**：之前要么正确但贵，要么便宜但不可靠——Crab 做到了**两者兼得**
2. **eBPF 在 AI Agent 领域的创新应用**：eBPF 本是 Linux 内核可观测性工具，Crab 将其用于 Agent state 的分类，展示了 OS-LLM 跨层协同的可能
3. **75% 零状态变化**的发现具有设计哲学意义——提示 Agent 框架设计者不需要**每步都检查点**
4. **与 RL 训练的结合潜力**：Agent 强化学习需要大量 rollout 分支（branched rollout），Crab 为 RL 环境的高效快照提供了基础设施

## 🆕 3. STAR — 同构 GPU 集群的 Straggler 真相与弹性训练系统（48-84% TTA 降低）

| 元数据 | 内容 |
|:-------|:-----|
| **来源** | arXiv:2512.09685, 2025-12-10 | **开源代码** ✅ |
| **作者** | Zeyu Zhang, Haiying Shen (UVA) |
| **领域** | Straggler 检测与缓解 / 同构 GPU 训练 |

### 🎯 核心贡献

**关键前提：** 人们以为「同构 GPU = 没有 straggler」，但这是错的。

**STAR 作者的实证发现：**
1. **同构 GPU 中 straggler 仍然普遍存在**——根因不是算力差异，而是 **CPU 和带宽使用不均衡**
2. **从 SSGD 切换到 ASGD 不一定有效**——ASGD 的资源消耗更高，**反而会制造新的 straggler**

### 方法设计

STAR 包含三层策略：

| 层 | 策略 | 说明 |
|:-:|:----|:-----|
| ① | **自适应同步模式选择** | 将 worker 分组，为每组选择最优同步模式（heuristic + ML 双路决策） |
| ② | **资源感知重分配** | 选择模式后，动态重分配 CPU/带宽资源以支持选择的模式，最小化对共置作业的影响 |
| ③ | **主动预防** | 避免在 Parameter Server 分配和梯度传输中**过载 CPU 和带宽**，**预防** straggler 产生 |

### 📊 实验数据

| 架构 | STAR vs SOTA | 指标 |
|:----|:-----------|:-----|
| **Parameter Server** | **↓ 48-84% TTA** | Time-To-Accuracy |
| **All-Reduce** | **↓ 51-70% TTA** | Time-To-Accuracy |

**核心指标说明：** Time-To-Accuracy (TTA) 是衡量分布式训练效率的**综合指标**——融合了每轮速度与收敛质量。48-84% 的降低意味着**大幅减少达到目标精度的时间**。

**实现细节：** Trace-driven 实验基于 AWS 集群模拟，使用了真实生产负载的 trace 数据。

### 💡 为何重要

1. **破除同构 GPU 的「straggler 免疫」迷思**——这是一项被忽略已久的系统性问题
2. **揭示 SSGD → ASGD 切换的不良后果**——这是业界常用的 straggler 缓解手段，但 STAR 证明它可能**适得其反**（更高资源消耗 → 更多 CPU/带宽竞争 → 更多 straggler）
3. **主动预防 vs 被动响应**的范式升级——现有方案大多在 straggler 出现后**被动响应**，STAR 的第三层做了**预防**
4. **开源代码**可用，降低了复现和落地的门槛
5. 与 **Lit Silicon（ISCA'26, 热耦合 straggler, 06-06 已跟踪）** 形成**互补**——Lit Silicon 关注 GPU 自身的热学因素，STAR 关注 CPU/带宽资源竞争因素
