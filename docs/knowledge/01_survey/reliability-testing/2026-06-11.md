## 📋 今日 2 篇新补录论文

### 1️⃣ BitSnap — Checkpoint 稀疏化+量化，16× 压缩 🆕
**arXiv:2511.12376v2, 2025-11 (v2. 2026-05) | 不同规模 LLM 验证**

- **问题**: LLM 训练中 Checkpoint 存储与加载开销随模型增长急剧膨胀，现有方法未综合考虑压缩率、速度与精度影响间的平衡
- **方案**: 两阶段压缩——① **Bitmask-based 稀疏化**：动态识别训练中参数更新，只保存显著变化的权重子集（与 LLMTailor 按层裁剪形成正交互补）；② **Cluster-based 量化**：对保留权重的聚类中心进行编码，进一步缩小体积
- **数据**:
  - 位掩码稀疏化：**16× 压缩率**，模型质量无损
  - 聚类量化：额外 **2× 压缩**，精度损失极小
  - 联合应用：**32× 总体压缩率**，训练中动态调整策略适配不同阶段
- **交叉引用**: 与 LLMTailor（4.3× 层裁剪）、MoEvement（Expert 子集检查点, NSDI'26）构成检查点优化三路线——**层级裁剪 × 激活稀疏 × 量化编码**

### 2️⃣ Optimus — Aurora 超算 12K+ GPU MoE 训练，内置容错特性 🆕
**arXiv:2604.00785, 2026-04 | Intel Aurora 超算（127,488 Intel PVC GPU tiles）**

- **背景**: Aurora 超算训练 Mula 系列 MoE 模型（Mula-220B-A10B），从 384 扩展到 **12,288 GPU tiles**，扩展效率约 90%
- **Optimus 库**: 自研训练库，支持标准大模型训练技术 + 端到端 MoE 预训练（含自定义 GPU Kernel + EP-Aware Sharded Optimizer，1.71× 加速）
- **容错特性**: "As part of the Optimus library, we also developed a robust set of reliability and fault tolerant features to improve training stability and continuity at scale."
  - 面向 Intel PVC (Ponte Vecchio) GPU 的特殊容错需求
  - 保障 1000+ GPU 规模下训练的**稳定性**与**连续性**
  - 端到端 4T tokens 全量预训练成功完成（Mula-1B/Dense、Mula-7B-A1B/MoE）
- **意义**: 工业级容错工程实践，证明大规模 ExaScale 系统上训练的鲁棒性可达——与 FT-HSDP(100K GPU)、SPARe(ICML'26) 共同组成超大规模容错实践经验库
