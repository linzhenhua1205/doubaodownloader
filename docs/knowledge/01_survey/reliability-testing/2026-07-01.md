## 1️⃣ 🛡️ ItoyoriFBC: 动态任务依赖下的 SDC 防护——全复制仅 2× 开销

**来源**: arXiv:2606.30771 · **2026-06-29 提交** · cs.DC

**作者**: Rüdiger Nather, Claudia Fohry, Mia Reitz

**背景**: 随计算规模增长，静默数据损坏（SDC）发生概率快速上升。现有 SDC 防护方案多数假设**静态任务和依赖关系**，但实际 AMT（异步多任务）运行时中存在动态任务派生、promise/future 通信、条件触发、负载均衡等工作窃取等复杂场景。在这些场景下，传统基于多数投票的复制方案面临**任务/数据追踪**的根本困难。

**核心方法 — ItoyoriFBC**:

| 组件 | 说明 |
|:-----|:------|
| **跨验证复制** | 将原始计算与副本计算紧密耦合——不仅复制计算，还**跨验证所有对外作用**（runtime 系统交互时的输出效果） |
| **选择性重计算** | 出现不一致时，仅对受影响的子任务选择性重计算，而非全任务回滚 |
| **动态依赖链追踪** | 支持 C++11 风格 promise/future 通信、条件触发的任务派生、work-first 工作窃取的集群级负载均衡 |
| **运行时实现** | ItoyoriFBC 运行时系统，针对动态 AMT 场景做了专门优化 |

**实验数据**:

| 指标 | 数值 |
|:-----|:-----|
| **无故障运行时间增加** | **< 2×**（全复制前提下，因任务数量增加→负载均衡机会改善，实际开销远低于预期的 2×）|
| **每次 SDC 修正开销** | **≈ 0.5%** 总运行时间 |
| **基准测试** | Fibonacci + 模拟 H-矩阵 LU 分解 |
| **覆盖场景** | 动态任务派生 · promise/future 通信 · 条件触发 · 集群级工作窃取 |

**意义**:

1. **动态 SDC 防护的稀有探索** 🔍：大多数 SDC 复制工作关注静态 DAG（如 TensorFlow 计算图）。ItoyoriFBC 面向**运行时动态派生**的 AMT 场景——这更接近真实 HPC 应用中不可预知的控制流

2. **「耦合复制 + 选择性重计算」的设计模式** 🧩：跨验证所有对外作用（而非仅验证计算结果），是 SDC 防护中较罕见的设计选择——它更「防御性」，但更适合副作用密集的 AMT 运行时

3. **与 SDC 研究体系的定位** 🏛️：此前的 SDC 三验证体系（门级→RTL级→生产级，如 06-28 跟踪的 AWS SDC 实证）聚焦**故障微观机制和宏观影响**；ItoyoriFBC 聚焦**运行时级 SDC 防护策略**——形成一个「理解 SDC → 防护 SDC」的完整链条

4. **局限** ⚠️：实验限于 Fibonacci 和 LU 分解，未在大规模 LLM 训练场景验证；全复制策略的内存开销在超大模型场景下可能是约束

**📌 交叉引用**:
- 与 [AWS SDC 生产实证](./2026-06-28.md)（arXiv:2502.12340, 真实 SDC 硬件效应）互补——理解 SDC 效应（实证）vs 防护 SDC（ItoyoriFBC）
- 与 [CCGrid2026 SDC](./2026-06-06.md)（指令级 SDC 注入检测）——检测 vs 防护，可级联使用
- 与 [LLM-PRISM](./2026-06-25.md)（RTL 级仿真, 7,664 次训练, SDC 影响面分析）——SDC 影响面分析为 ItoyoriFBC 类方案提供故障模型输入

## 2️⃣ ⚡ StreamGuard: 低开销 HPC 数据流弹性——异步检查点 + 进度感知重分配, ≤1% 开销

**来源**: arXiv:2606.30848 · **2026-06-29 提交** · cs.DC · **已接收: ICS'26**（ACM 国际超级计算会议）

**作者**: Hai Duc Nguyen, Bogdan Nicolae, Tekin Bicer, Amal Gueroudji, Matthieu Dorier, Kyle Chard, Ian Foster（ANL / UChicago）

**背景**: 实时科学工作流处理连续数据流，必须在故障频发的复杂基础设施上产出及时、高质量的结果。硬件故障、网络中断、资源争用和异构性导致的性能异常会严重违反实时约束。生产-消费（producer-consumer）流模式是科学流工作流的基本构建块——其弹性直接决定整体可靠性。

**核心方法 — StreamGuard**:

| 组件 | 说明 |
|:-----|:------|
| **动态异步非阻塞检查点** | 不中断计算的前提下保存数据流进度——与传统同步检查点不同，StreamGuard 的检查点操作与数据流处理**完全重叠** |
| **进度感知负载重分配** | 检测慢节点并**主动重平衡**任务——无需等待节点完全失效，在性能降级初期就介入 |
| **保持前进** | 两机制协同确保即使在高度故障环境中也能保持**前进而不停滞** |

**双机制协同**:

```
数据流进入 → 动态异步检查点（保存进度，不中断计算）
          ↘ 进度感知监测（实时检测慢节点）
            → 快速节点 ← 任务重分配 → 慢节点（负载减轻）
              ↘ 持续输出 ← 进度合并
```

**实验数据**:

| 指标 | 数值 |
|:-----|:-----|
| **故障/异常影响减少** | **≤ 6×**（与无弹性基线相比） |
| **无故障执行开销** | **< 1%** |
| **覆盖异常类型** | 硬件故障 · 网络中断 · 资源争用 · 性能异常 · 异构性干扰 |
| **接收会议** | **ICS'26**（ACM International Conference on Supercomputing） |
| **核心模式** | 生产-消费（producer-consumer）流模式 |

**意义**:

1. **ICS 顶会论文，可信度高** 🏆：作为超算领域旗舰会议 ICS 2026 的收录论文，StreamGuard 的质量和评审标准有保障

2. **异步检查点 + 主动重负载平衡 = 「检测 + 预防」双模式弹性** 🔗：多数弹性方案仅被动响应故障（故障后恢复）。StreamGuard 结合**异步检查点（故障后恢复）** 和**进度感知重分配（故障前预防）**——在性能降级初期就主动调整，比纯故障后恢复更高效

3. **<1% 开销的工程价值** 📉：低开销在 HPC/AI 训练中极为重要——检查点方案引入显著开销的历史教训（如传统同步检查点在 175B 模型上的瓶颈）表明，<1% 的开销使 StreamGuard 的异步检查点可被默认启用，无需权衡

4. **与 LLM 训练弹性的关联** 🔄：虽然 StreamGuard 面向实时科学流（如显微镜数据流、传感器数据流），其核心机制——**异步无阻塞检查点 + 进度感知负载均衡**——与 ResiHP（LLM 训练的运行时弹性调度）的设计哲学相同

5. **局限** ⚠️：未在 LLM 训练规模的同步 AllReduce 密集场景下验证；进度感知负载均衡在 NPU/GPU 上的可移植性需额外工程适配

**📌 交叉引用**:
- 与 [ResiHP](./2026-06-23.md)（并行策略级运行时弹性调度）——同属「动态重分配」弹性范式，ResiHP 面向 LLM 训练，StreamGuard 面向 HPC 数据流
- 与 [GoCkpt](./2026-06-30.md)（跨步重叠检查点, 38.4% 吞吐提升）——GoCkpt 的跨步分段检查点与 StreamGuard 的异步非阻塞检查点，均属「检查点操作与计算重叠」设计空间
- 与 [TierCheck](./2026-06-23.md)（分级检查点存储, ≥3 层, 10s 内完成 CKPT）——TierCheck 优化检查点存储层级，StreamGuard 优化检查点执行时序，正交可叠加
- 与 [StreamGuard（同组）AI 透明 C/R](#3️⃣-🤖-透明检查点ai-驱动代码生成llm-50-分钟完成-mpi-应用检点接入)（本日条目 3）——同一 ANL 团队的两个互补产出：StreamGuard 是弹性运行时方案，AI 透明 C/R 是自动生成方案

## 3️⃣ 🤖 透明检查点：AI 驱动代码生成——LLM 50 分钟完成 MPI 应用检点接入

**来源**: arXiv:2606.30921 · **2026-06-29 提交** · cs.DC · FlexScience'26 @ HPDC'26

**作者**: Hai Duc Nguyen, Tekin Bicer, Kyle Chard, Ian Foster, Bogdan Nicolae（同一 ANL 团队）

**背景**: 为 MPI 科学应用添加可靠的检查点/恢复（C/R）支持是一项耗时费力的专家工作——需要同时深入了解应用和可靠性工程。该工作问了一个根本问题：**前沿 LLM 能否端到端地自动完成这项任务，无需人工介入？**

**核心方法 — LLM 驱动的 C/R 代码生成管线**:

| 组件 | 说明 |
|:-----|:------|
| **MPI 应用基准套件** | 覆盖多个领域的 6 个 MPI 应用，多样的计算模式 |
| **迭代代码生成循环** | 使用 Anthropic Claude Opus 4.7，通过 OpenCode CLI 驱动 |
| **端到端无干预** | 从输入 MPI 应用到输出 C/R 代码，全程无人工介入 |
| **验证方法** | 注入进程故障，对比人类专家编写的 C/R 实现的恢复效率 |

**实验数据**:

| 指标 | 数值 |
|:-----|:-----|
| **平均 C/R 代码生成时间** | **50 分钟 / 每个应用** |
| **平均 token 消耗** | **3.4 M tokens / 每个应用** |
| **无故障执行开销** | **6 个应用中 5 个开销可忽略** |
| **恢复效率** | 与人类工程化 C/R 实现**相当** |
| **LLM 模型** | Anthropic Claude Opus 4.7 |
| **工具链** | OpenCode CLI |
| **应用范围** | 6 MPI 科学应用，跨域多样性 |

**意义**:

1. **AI 辅助可靠性工程的里程碑** 🏅：首次系统证明 LLM 可以**端到端自动生成**可工作的 C/R 代码，这是「可靠性工程自动化」从概念验证到实际可用的一步

2. **对检查点研究的反向启示** 💡：C/R 实现是「人类专家平均耗时数天-数周」的繁重工作。50 分钟的自动生成效率意味着：**检查点研究的落地成本将大幅降低**——更多 HPC 应用可以快速获得 C/R 支持，从而为检查点优化研究的大规模实验提供更多试验床

3. **ANL 团队的「弹性组合拳」** 🥊：同一团队（Nguyen, Bicer, Chard, Foster, Nicolae）同时产出：
   - **StreamGuard**（入口 2）→ 弹性运行时方案
   - **AI 透明 C/R**（入口 3）→ 自动化 C/R 代码生成
   - 两者构成「让更多应用获得弹性 + 让获得弹性的应用有更好的弹性运行时」的正反馈

4. **局限** ⚠️：
   - 仅测试了 6 个 MPI 应用，规模有限
   - 1 个应用有不可忽略的开销——并非所有应用类型都适合
   - 使用 Anthropic 的闭源模型，可复现性受限于模型版本
   - 针对 MPI 应用，非 PyTorch/NCCL 分布式训练场景——LLM 训练 C/R 自动生成仍待验证
   - 未比较不同类型 LLM 的 C/R 代码生成能力差异

**📌 交叉引用**:
- 与 [StreamGuard](#2️⃣-⚡-streamguard-低开销-hpc-数据流弹性异步检查点--进度感知重分配-≤1-开销)（本日条目 2）——同一团队的弹性组合
- 与 [GoCkpt](./2026-06-30.md)（跨步检查点, 38.4%）——GoCkpt 提供更优的检查点执行方式，AI C/R 降低检查点接入门槛，组合效应显著
- 与 [TRANSOM](./2026-06-30.md)（TOL+TEE+TCE 三引擎, 20× CKPT）——工业级全栈容错系统工程 vs AI 辅助 C/R 自动化，不同维度推动可靠性工程
- 与 [MeCeFO](./2026-06-07.md)（NeurIPS'25, 优化算法级容错, 4.18% 吞吐下降）——算法级容错 vs 系统级 C/R 自动化，正交可叠加
