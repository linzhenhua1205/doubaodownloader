## 1️⃣ MoEvement: MoE 模型稀疏检查点系统（NSDI'26）🆕✨

- **论文**: [Sparse Checkpointing for Fast and Reliable MoE Training](https://arxiv.org/abs/2412.15411)
- **来源**: arXiv:2412.15411 v5 | 2026-03-19 Camera-Ready | **NSDI'26 接收**
- **作者**: Swapnil Gandhi, Christos Kozyrakis (Stanford/EPFL)
- **背景痛点**: MoE 模型参数量显著大于同规模 Dense 模型（训练状态膨胀 2-4×），传统全量检查点导致 **长时间训练 stall** 和 **慢速恢复**，MoE 的密集通信特性使得该问题更加突出

- **方法 — 三大核心设计**:

  1. **稀疏检查点 (Sparse Checkpointing)** — 每轮迭代仅对**部分 expert 子集**做增量快照，而非全量 expert，从而将每次检查点的 I/O 量降低数倍
  2. **稀疏→稠密转换机制 (Sparse-to-Dense Conversion)** — 在后台增量地从多个稀疏快照**重构出完整一致的稠密检查点**，不阻塞训练进度
  3. **上流日志记录 (Upstream Logging)** — 在 pipeline-stage 边界记录 activation/gradient，节点故障时**仅恢复受影响 worker**，无需重放未受影响的 GPU

- **实验数据**（多种 MoE 模型，最多 64 experts）:

  | 指标 | 数据 |
  |:-----|:-----|
  | 检查点开销降低 | **≤ 4×** vs SOTA |
  | 恢复开销降低 | **≤ 31×** vs SOTA |
  | ETTR (MTBF=10min) | **≥ 0.94**（极端频繁故障下） |
  | 整体训练加速 | **≤ 8×** |
  | 同步训练语义 | **完整保留** |

- 📌 **深度洞察**: MoEvement 是**首个针对 MoE 架构优化的检查点系统**，其「稀疏快照+渐进式稠密化」理念与 LLMTailor 的「按层裁剪」有异曲同工之妙——都在回答一个问题：「训练状态中的哪些部分可以跳过同步？」二者互补：LLMTailor 沿模型深度（按层裁剪），MoEvement 沿模型宽度（按 expert 子集）

## 2️⃣ LLM-PRISM: GPU 永久故障引发 SDC 的硬件接地特性分析 🆕✨

- **论文**: [LLM-PRISM: Characterizing Silent Data Corruption from Permanent GPU Faults in LLM Training](https://arxiv.org/abs/2604.10390)
- **来源**: arXiv:2604.10390 | 2026-04-12
- **作者**: Abhishek Tyagi, Saurabh Hukerikar, Nirmal Saxena, Yanxiang Huang, Philip Shirvani, Chung-Hsuan Tung, Yuhao Zhu
- **核心问题**: 制造逃逸和硅老化导致的 GPU 永久性硬件缺陷在训练中产生 SDC，但**缺乏硬件接地**的 LLM 预训练 SDC 弹性特性分析
- **方法 — 双层方法论**:
  - **RTL 级 GPU 故障仿真** — 模拟永久性 stuck-at 故障在硬件逻辑门的传播行为
  - **随机注入引擎嵌入 Megatron-LM** — 在训练循环中注入故障，观察对梯度/参数的扰动
  - 覆盖 **FP16、BF16、FP8** 三种精度格式
  - 共计 **7,664 次训练运行**，系统性扫描故障类型×故障率×数值格式

- **关键发现**:

  | 发现 | 描述 |
  |:-----|:------|
  | 低频故障抵抗性 | LLM 对低频故障表现一定的鲁棒性 |
  | **高度非均匀** | 影响分布极度不均衡——一些 datapath 能够抵抗，另一些导致灾难性发散 |
  | **关键 datapath** | 存在**少数关键路径**，故障时引发不可逆的精度崩塌 |
  | **精度格式敏感** | FP8 训练下，**中等故障率即导致灾难性发散**，BF16 最鲁棒 |
  | **与 DSN'26 GPU SDC 解剖论文同团队** | 前者关注瞬态故障模式，本工作关注永久故障效应，形成「时序故障 (DSN'26) + 缺陷故障 (本工作)」双线 SDC 特性覆盖 |

- 📌 **深度洞察**: 这是**首个硬件接地**的 LLM 预训练 SDC 特性研究。与 DSN'26 的 GPU SDC 解剖（2605.04213, 06-04 已收录）来自**同一 NVIDIA 研究团队**：前者用 300 万+模拟器小时做**门级瞬态故障定征**，本工作用 7,664 次训练运行做**RTL 永久故障分析**——二者形成「瞬态 vs 永久」双维度 SDC 图谱，对 GPU 架构的容错加固和训练框架的软错误感知设计有直接指导意义

## 3️⃣ SDC 轻量检测与恢复：LLaMA 训练中的可控故障注入研究 🆕✨

- **论文**: [Exploring Silent Data Corruption as a Reliability Challenge in LLM Training](https://arxiv.org/abs/2604.00726)
- **来源**: arXiv:2604.00726 | 2026-04-01 | **CCGrid 2026 接收**
- **作者**: Anton Altenbernd, Philipp Wiesner, Odej Kao (TU Berlin)
- **方法**:
  - **可控故障注入** — 在 GPU 矩阵乘法指令层面注入间歇性 SDC
  - **敏感性分析** — 覆盖不同 bit 位置、kernel 函数、执行阶段
  - **轻量检测算法** — 基于观察到的 SDC signature（损失尖峰/梯度范数激增/注意力 logits 异常），在训练过程中**实时识别有害参数更新**
  - **恢复策略** — 检测到后**重算最近一个训练步骤**，而非全局回滚

- **实验数据**（LLaMA 60M/350M/1.3B）:

  | 观察维度 | 结果 |
  |:---------|:-----|
  | NaN 传播 | 局部故障可产生广泛的 NaN 传播 |
  | Loss 尖峰 | 短生命周期（数步内恢复）但不及时处理会积累 |
  | 梯度范数 | 梯度范数激增是 SDC 的可靠早期信号 |
  | Attention logits | 输出层故障导致 attention 分布畸变 |
  | **参数发散** | 部分故障导致**持久性参数偏移**，不恢复则不可逆 |
  | **重算恢复** | 重算最近一步即可有效缓解上述影响 |

- 📌 **深度洞察**: 本工作与 DSN'26/LLM-PRISM 形成**三条 SDC 防线**：

  | 防线 | 论文 | 技术路线 | 粒度 |
  |:----|:-----|:---------|:----|
  | ① 特性理解 | DSN'26 SDC 解剖 | 门级注入 → 故障模式分布 | 硬件微架构 |
  | ② 特性理解 | LLM-PRISM | RTL 注入 → 永久故障 | 硬件+训练框架 |
  | ③ 在线防御 | **本工作** | 可控注入 → 轻量检测+重算恢复 | 训练框架层 |

  CCGrid 的工程实用价值在于提出了**可直接嵌入训练管道的零开销检测器**——不引入额外计算开销，仅利用训练中已有的 loss/gradient/logits 信号

## 4️⃣ Lit Silicon: 热不平衡耦合多 GPU 执行导致的 Straggler 新机制（ISCA'26）🆕✨

- **论文**: [Lit Silicon: A Case Where Thermal Imbalance Couples Concurrent Execution in Multiple GPUs](https://arxiv.org/abs/2511.09861)
- **来源**: arXiv:2511.09861 v3 | 2026-05-12 更新 | **ISCA 2026 接收** — 计算机体系结构顶级会议
- **作者**: Marco Kurzynski, Shaizeen Aga, Di Wu (AMD Research)
- **核心发现 — Lit Silicon 效应**: 在同一节点内，**GPU 间的热不平衡**导致较热的 GPU 降频变慢（Straggler），而较冷的 GPU 因 C³（Computation & Communication Concurrency，计算通信重叠）技术需等待热 GPU 的同步通信完成，从而也被拖慢——即热 GPU 的降频通过**同步屏障**「传染」给冷 GPU

- **问题机制**:
  ```
  热不平衡 → 热 GPU 降频（throttling）→ 集合通信变慢
        ↓
  冷 GPU 因 C³ 与热 GPU 通信重叠 → 冷 GPU 计算完成后等待
        ↓
  **整个节点性能坍缩至最热 GPU 的水平**
  ```

- **实验平台**:
  - 两台 AMD Instinct™ MI300X GPU 系统（8-GPU 节点）
  - 两个 LLM 训练框架
  - 两种 workload

- **实验数据**:

  | 方案 | 性能提升 | 能耗节省 | 说明 |
  |:----|:--------:|:--------:|:-----|
  | GPU TDP 内功耗优化 | **6%** | **4%** | 限制热 GPU 峰值功耗但不触发降频 |
  | 节点级 GPU Power Capping | 持平 | 约数百万美元电费 | 统一限制节点功耗上限 |
  | CPU Power Sloshing | 约 3-5% | — | 将 CPU 功率转移至 GPU |

  > **经济影响**: 单数据中心可节省数千万美元电费

- 📌 **深度洞察**: Lit Silicon 揭示了**一个此前被忽视的性能损失来源**——热耦合不是简单的单卡降频，而是通过 C³ 同步机制将 thermal straggling 传播到整个节点。这与 Guard（MLSys'26, 1.7× MFU）的 straggler 检测形成互补：Guard 关注计算不均匀（如数据倾斜），Lit Silicon 关注**物理热耦合**引发的性能波动。对大规模部署而言，这意味着**温度平衡应成为 GPU 集群调度的新目标维度**
