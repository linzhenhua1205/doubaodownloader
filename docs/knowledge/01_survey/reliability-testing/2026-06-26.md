## 1️⃣ 🔥 DiffCp：利用压缩梯度复用的低开销频繁检查点框架

**来源**: arXiv:2509.04084 · **2025-09-04 提交（v3: 2025-12-24）** · cs.DC · 华中科技大学（Chenxuan Yao, Yuchong Hu, Feifan Liu 等）

**概述**: 大规模分布式训练中检查点频率与恢复速度是一对根本矛盾——频繁检查点可以加速故障恢复，但产生巨大的 I/O 开销。已有差分检查点（differential checkpointing）方案仅针对推荐系统有效，无法推广到通用分布式训练。DiffCp 提出**复用压缩梯度作为差分检查点**的思路，从根本上消除了额外存储写入成本。

**核心技术**:

| 组件 | 说明 |
|:-----|:-----|
| **核心思想** | **复用（reuse）** 训练中已存在的压缩梯度作为差分检查点，无需额外计算或 I/O 生成检查点数据 |
| **Batched Gradient Write** | 批量梯度写入优化，将多个微批次的差分数据聚合后持久化到存储，减少小 I/O 放大 |
| **动态调参** | 自动调节检查点频率和批次大小以最大化性能 |
| **DiffCp⁺ 扩展** | 非压缩场景下采用逐层复用快照（layer-wise-reuse snapshotting）+ 增量合并持久化（incremental-merging persistence）双策略 |

**实验数据**:

| 指标 | DiffCp | DiffCp⁺ |
|:-----|:-------|:--------|
| 训练时间减少（per-iteration 频率） | **最高 89.2%** | **最高 81.2%** |
| 适用场景 | 梯度压缩场景 | 非压缩/通用场景 |
| 检查点频率 | 可高达每迭代一次 | 可高达每迭代一次 |

**意义**: DiffCp 的创新在于「复用已有数据」而非「额外生成」——压缩梯度已是 NCCL/Ring AllReduce 中的中间产物，DiffCp 将其「废物利用」为差分检查点。这使得频繁检查点的 I/O 成本趋近于零。在 checkpointing 技术路线中，DiffCp 与 LLMTailor（层级选择性）、BitSnap（稀疏+量化）形成**正交补充**——前者解决「存什么」，后两者解决「怎么存」。

**📌 交叉引用**: 
- 与 [LLMTailor](../bmc-system/2026-06-25.md)（层级选择性检查点）互补——LLMTailor 决定「哪些层要存」，DiffCp 决定「存的成本趋零」
- 与 [BitSnap](../bmc-system/2026-06-11.md)（稀疏+量化 16×+2× 压缩）互补——BitSnap 优化单个检查点的存储体积，DiffCp 优化「存得有多频繁」
- 三者构成检查点技术的「存什么→怎么存→多频繁」三维优化空间

## 2️⃣ 🔄 Lazarus：MoE 模型的弹性容错训练（5.7× 性能提升）

**来源**: arXiv:2407.04656 · **2024-07-05 提交（v9: 2025-10-24）** · cs.DC · UC Berkeley / Microsoft Research（Yongji Wu, Wenjie Qu, Ion Stoica 等）

**概述**: MoE（混合专家）架构在大模型中得到广泛采用，但其专家并行（expert parallelism）特性使得现有容错方案——尤其是依赖流水线并行弹性的方案——无法直接应用。同时，云上 spot instance 的使用虽大幅降低成本（达 70-90%），但其频繁的抢占本质上也构成了训练过程中的系统性故障。Lazarus 是首个专门为 MoE 模型设计的弹性容错训练系统。

**核心技术**:

| 组件 | 说明 |
|:-----|:-----|
| **自适应专家副本分配** | 动态调节 expert replicas 数量以应对 expert workload 天然不均衡（某些 expert 负载是其他的 3-5×），同时通过冗余加速训练 |
| **最优专家放置算法** | 可证明最优（provably optimal）的 expert placement，最大化故障后可恢复概率 |
| **灵活 Token Dispatcher** | 故障后充分利用所有幸存节点，**不闲置任何 GPU**，非简单的负载均衡 |
| **无 idle GPU** | 与 checkpoint/restart 的本质区别——故障节点移除后训练继续，不需要等待任何 GPU |

**实验数据**:

| 指标 | 值 |
|:-----|:---|
| 频繁节点故障场景 | **5.7×** 优于现有 MoE 训练系统 |
| 实际 spot instance trace | **3.4×** 优于基线 |
| 支持故障类型 | 节点故障（node failures）+ 抢占（preemptions） |
| 映射的 MoE 规模 | 生产级 MoE 模型（具体规模未公开） |

**意义**: Lazarus 填补了 MoE 模型弹性容错训练的空白。与 FT-HSDP（数据并行副本容错）和 SPARe（冗余堆叠）不同，Lazarus 利用了 MoE 架构本身的 expert 冗余特性——专家天然可被复制，token 天然可被重路由。这一思路使得故障恢复无需所有 GPU 同步等待，真正实现了「故障就是少几个 expert，训练继续」。

从方法论上看，Lazarus 与 FT-HSDP 和 SPARe 走的是**不同路径**：
- FT-HSDP → 以数据并行副本为容错单元
- SPARe → 以冗余数据堆叠掩盖故障
- Lazarus → 以 MoE 专家自身的分布式冗余为弹性基础

**📌 交叉引用**: 
- 与 [FT-HSDP](../bmc-system/2026-06-23.md)（数据并行副本容错，100K GPU）对比——针对不同并行策略
- 与 [SPARe](../bmc-system/2026-06-23.md)（冗余堆叠，ICML'26）对比——三种容错路径的最新进展
- 与 [ReCoVer](../bmc-system/2026-06-23.md)（容错聚合，512 GPU 验证）对比——微批次恒定的容错方案
