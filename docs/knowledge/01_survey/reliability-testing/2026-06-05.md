## 1️⃣ R²CCL: 面向 LLM 训练与推理的可靠弹性集合通信库 🆕✨

- **论文**: [Reliable and Resilient Collective Communication Library for LLM Training and Serving](https://arxiv.org/abs/2512.25059)
- **来源**: arXiv:2512.25059 | 2025-12-31 | **全新收录，此前未跟踪**
- **作者**: Wei Wang, Nengneng Yu, Sixian Xiong, Zaoxing Liu（与 OptCC 同团队）
- **方法**: 基于多 NIC 硬件的容错通信库，提供无损低开销故障切换：

  1. **快速连接迁移**（Rapid Connection Migration）— 检测到 NIC 故障后立即将活跃连接迁移至备用 NIC，避免 TCP/IB 连接超时重连
  2. **带宽感知负载重分配**（Bandwidth-Aware Load Redistribution）— 根据幸存 NIC 的带宽动态调整数据分片，避免慢节点拖慢集体
  3. **弹性集体算法**（Resilient Collective Algorithms）— 在任意 NIC 组合（非对称带宽）下保持 AllReduce 进度

- **问题规模**: 网络故障浪费 GPU 10-15% 算力（因慢速恢复触发 timeout → 任务终止 → 检查点回滚/推理重处理）

- **实验数据**:
  - **训练开销 <1%**（8-GPU H100 IB 服务器实测）
  - **推理开销 <3%**
  - 对比 **AdapCC 提升 12.18×**
  - 对比 **DejaVu 提升 47×**
  - 大规模 ML 仿真器（数百 GPU + 多样故障模式）验证

- 📌 **深度洞察**: R²CCL 与 OptCC 来自同一研究团队（Wang/Yu/Xiong/Liu），形成「容错通信双子星」——OptCC 提供网络故障下的 AllReduce 信息论下界 + 算法，R²CCL 提供工程化的多 NIC 故障切换实现。二者的关系可以类比为理论算法（OptCC）与工程系统（R²CCL）的互补

## 2️⃣ LLMTailor: LLM 训练中基于逐层裁剪的高效检查点 🆕✨

- **论文**: [LLMTailor: A Layer-wise Tailoring Tool for Efficient Checkpointing of Large Language Models](https://arxiv.org/abs/2602.22158)
- **来源**: arXiv:2602.22158 | 2026-02-25 | **PDSW'25 接收** | **全新收录，此前未跟踪**
- **作者**: Minqiu Sun, Xin Huang, Luanzheng Guo, Nathan R. Tallent, Kento Sato, Dong Dai

- **核心洞察**: LLM 各层的更新幅度高度**非均匀**——部分层在训练步骤中变化显著，其他层保持稳定甚至不变。现有方法无论 I/O 策略如何，都定期存储全部模型 + 优化器状态，造成大量存储浪费和资源争抢

- **方法**: LLMTailor 是一个**检查点合并框架**，允许选择性保留更新显著的层：

  1. **检测** — 按层追踪权重和优化器状态变化量
  2. **筛选** — 仅保留变化超过阈值的层（可配置）
  3. **合并** — 从多个检查点中筛选 + 组装层形成复合检查点

- **实验数据**:
  - **Llama3.1-8B**: 检查点大小缩小 **4.3×**
  - **Qwen2.5-7B**: 检查点时间加速 **2.8×**
  - 保持模型质量不下降
  - 兼容不同选择性检查点策略

- 📌 **深度洞察**: LLMTailor 代表了检查点技术从「均匀全量存储」向「更新感知选择性存储」的范式转变。与 TierCheck 的三层架构（本地/peer/远程 + 差分检查点）形成互补——TierCheck 从存储层次优化，LLMTailor 从数据内容优化。二者结合有潜力在保留恢复速度的同时，将检查点 I/O 开销降低一个数量级

## 3️⃣ SPARe: 100K+ GPU 集群的堆叠并行自适应重排序容错 ⭐

- **论文**: [SPARe: Stacked Parallelism with Adaptive Reordering for Fault-Tolerant LLM Pretraining Systems with 100k+ GPUs](https://arxiv.org/abs/2603.00357)
- **来源**: arXiv:2603.00357 | v3 2026-05-27 更新 | **ICML 2026 接收**
- **作者**: Jin Lee, Zhonghao Chen, Xuhang He, Robert Underwood, Bogdan Nicolae, Franck Cappello, Xiaoyi Lu, Sheng Di, Zheng Zhang
- **核心问题**: 100K+ GPU 集群中，**重启成本占主导地位**——但现有容错机制对此状态准备不足

- **方法**:
  - **Stacked Parallelism** — 在梯度同步过程中跨并行组堆叠冗余数据分片，在 GPU 节点故障时，**无需重启全局训练**，冗余分片自动补偿丢失梯度
  - **Adaptive Reordering** — 根据故障模式动态调整执行顺序，最小化计算开销
  - 推导了 **可容忍故障数的封闭式表达式** + **计算开销封闭式表达式**
  - 联合优化冗余度和检查点频率以最小化总训练时间
  - 基于 SimGrid 的离散事件仿真验证

- **关键特性**:
  - 对比传统复制的**接近常数计算开销**（仅 2-3×），而传统复制随冗余度线性膨胀
  - 实现与传统复制相当的可用性

- **实验数据**（SimGrid 仿真）:
  - **600K GPU 极端规模下**，SPARe 减少 **40-50% 的训练时间**（time-to-train）vs 传统复制
  - 计算开销 **2-3×**（近常数，不随冗余度线性增长）
  - ICML 2026 接收（顶级 ML 会议验证）

- 📌 **深度洞察**: SPARe 与 ReCoVer 代表了容错领域的**两个正交路线**——ReCoVer 通过保持 microbatch 数量恒定 + 故障后快速恢复保证训练轨迹，SPARe 通过训练前的冗余布局 + 自适应调度**预防**故障干扰。前者是"发生后快速修复"，后者是"提前布局以容忍发生"。SPARe 在超大规模（600K GPU）场景下优势更加明显

## 4️⃣ LiveR: 弹性训练的实时重配置（交叉引用）

- **论文**: [LiveR: Real-Time Reconfiguration for Elastic LLM Training on Cloud Resources](https://arxiv.org/abs/2605.22014)
- **来源**: arXiv:2605.22014 | 2026-05-26
- **方法**: 替代传统 "stop-and-restart" 范式——当前世界继续训练，**异步准备目标世界**，通过高带宽互联（跨 TP/PP/DP 维度）在线重塑模型状态
- **性能**:
  - 重配置时间：分钟级 → **秒级**
  - 对比 checkpoint/restart 加速 **14-23×**
  - volatile（易变）资源下维持 **99% goodput**
- 📌 **深度分析**: LiveR 对可靠性领域的关键贡献在于**将「重配置」从训练中断操作变为后台无缝操作**——这是弹性训练从实验走向生产的关键一步
- 🔗 详见 `knowledge/cluster-training/2026-06-05.md` §5.3

## 5️⃣ Guard: 大规模训练的 Straggler 检测（交叉引用）

- **论文**: [Guard: Proactive Straggler Detection and Mitigation for Large-Scale Pre-training](https://arxiv.org/abs/2605.17879)
- **来源**: arXiv:2605.17879 | **MLSys 2026 接收**
- **方法**: 轻量级在线性能监控 + 离线节点扫描，同时检测**急性故障**和**长期 fail-slow** 行为
- **性能**:
  - MFU 提升 **1.7×**
  - Step 方差从 **20% 降到 1%**
- 📌 **深度分析**: Straggler 检测是可靠性工程的重要组成部分——Guard 的商业价值在于无需额外硬件即可解决「集群中 20% 的慢节点拖慢全局」这一普遍问题
- 🔗 详见 `knowledge/cluster-training/2026-06-05.md` §5.4

## 💡 趋势洞察

1. **容错进入「多路并行」时代**: 集合通信容错已有 VCCL（可观测）、OptCC（理论下界+算法）、R²CCL（工程实现）三条独立路线同步推进，表明该方向已成为系统研究的热点
2. **检查点从"存什么"向"不存什么"转变**: TierCheck 按存储层次差异化 → LLMTailor 按更新幅度选择性跳过，未来两者融合可能出现存储+内容双重优化的智能检查点
3. **十万卡容错被 ICML 认可**: SPARe 被 ICML 2026 接收，标志着容错训练论文从系统会议突破到 ML 顶级会议——社区开始认识到容错是 ML 训练的核心基础设施问题
4. **工业界 vs 学术界合力**: Guard (MLSys'26) 来自大规模预训练的一线经验，SPARe (ICML'26) 来自 Argonne/ANL 的学术研究、R²CCL 来自华中科技大——工业界经验+学术界理论共同推动可靠性技术演进
