## 1️⃣ ⚙️ TRANSOM：工业级 LLM 容错训练系统——TOL+TEE+TCE 三引擎

**来源**: arXiv:2310.10046 · **2023-10-16 提交（v3: 2023-10-18）** · cs.DC

**作者**: Baodong Wu, Lei Xia, Qingping Li, Kangyu Li, Xu Chen, Yongqiang Guo, Tieyao Xiang, Yuheng Chen, Shigang Li

**背景**: LLM 训练集群上硬件/软件故障不可避免，大量训练时间浪费在检查点保存/加载、任务重调度、人工异常检查上。本文提出 TRANSOM，一个工业级全栈容错训练系统，聚焦 GPT-175B 量级模型的实际部署瓶颈。

**核心架构 — 三引擎子系统**:

| 组件 | 名称 | 功能 |
|:-----|:-----|:-----|
| **TOL** | Transom Operator & Launcher | 训练任务**生命周期管理** + 自动容错恢复编排。管理任务启动、节点分配、故障后节点剔除/替换、自动重启 |
| **TEE** | Transom Eagle Eye | 多维度指标**自动异常检测**系统。监控训练进度、loss 曲线、吞吐量、硬件健康等指标，发现异常即向 TOL 报告 |
| **TCE** | Transom Checkpoint Engine | 检查点**异步访问自动容错恢复**引擎。支持检查点异步保存/加载，显著缩短容错开销 |

**工作流**: TEE 检测到异常 → 报告 TOL → TOL 自动执行容错策略：剔除异常节点 + 保留健康节点 + 启动替换节点 + TCE 加载最新检查点 + 恢复训练

**关键实验数据**:

| 指标 | 数值 |
|:-----|:-----|
| **GPT3-175B 预训练时间减少** | **28%**（容错开销大幅削减） |
| **检查点保存/加载性能提升** | **20×**（TCE 异步引擎 vs 传统同步检查点） |
| **评估模型** | GPT3-175B 规模 |
| **覆盖场景** | 硬件故障 · 软件异常 · 网络波动 · 训练发散 |

**意义**:

1. **工业级全栈容错实践** 🏭：与 FT-HSDP（Meta, 100K GPU）、Optimus（Aurora, 12K GPU）类似，TRANSOM 代表了另一家大型科技公司在大规模 LLM 训练容错上的系统工程实践。三个子系统（生命周期管理 × 异常检测 × 异步检查点）覆盖了「检测→决策→恢复」容错全链路
2. **异常检测与容错编排的耦合设计** 🔗：TEE（Eagle Eye）作为独立检测子系统，与 TOL 的自动响应系统紧密耦合——这比单纯依靠超时触发容错更主动，但前提是检测准确率足够高
3. **异步检查点是核心加速器** ⚡：20× 的检查点性能提升揭示了同步检查点在 175B 量级模型上的瓶颈严重程度，验证了后来 DiffCp、GoCkpt 等检查点优化方向的必要性
4. **局限**：论文发布于 2023 年，实验基于 GPT3-175B，未涵盖后来的 MoE 架构、3D 并行等更复杂并行策略。该工作中的异步检查点方案对比当前 SOTA（如 MoEvement 的稀疏检查点、DiffCp 的差分检查点）可能已有差距

**📌 交叉引用**:
- 与 [FT-HSDP](./2026-06-03.md)（arXiv:2602.00277, 100K GPU, 有效训练 44%→80%）——同属工业级容错训练系统，FT-HSDP 聚焦数据并行副本容错，TRANSOM 聚焦全栈三引擎编排
- 与 [Optimus](./2026-06-11.md)（Aurora 12K GPU, Optimus 库容错）——同属超大规模实践，Optimus 侧重 Intel GPU 生态，TRANSOM 侧重通用 GPU 集群
- 与 [GoCkpt](#2️⃣-🧩-gockpt-梯度辅助多步重叠检查点——38.4-吞吐提升)（本日条目 2）——TRANSOM 的 TCE 使用同步→异步检查点获得 20× 性能提升，GoCkpt 进一步通过多步重叠再提升 38.4% 训练吞吐

## 2️⃣ 🧩 GoCkpt：梯度辅助多步重叠检查点——38.4% 训练吞吐提升

**来源**: arXiv:2511.07035 · **2025-11-10 提交** · cs.OS

**作者**: Keyao Zhang, Yiquan Chen, Zhuo Hu, Wenhai Lin, Jiexiong Xu, Wenzhi Chen

**背景**: LLM 训练中，检查点保存时的 GPU→CPU 传输会暂停/延迟 GPU 计算，导致显著训练中断和吞吐下降。现有异步检查点方案虽然将保存操作移到后台，但 GPU-CPU 带宽竞争和完整检查点传输的 I/O 瓶颈仍造成可观开销。核心问题是：**检查点保存的 I/O 时间是否可以被进一步隐藏？**

**核心方法 — GoCkpt**: 将检查点保存重叠到**多个训练步**中完成

| 组件 | 说明 |
|:-----|:------|
| **多步分段传输** | 将完整检查点状态分成多个片段，每个训练步只传输一个片段到 CPU，而非集中在一个步完成 |
| **梯度辅助一致性恢复** | 利用部分梯度数据辅助恢复——每个步传输的片段在 CPU 端累积，当所有片段传完后通过梯度辅助机制恢复**一致版本**的检查点 |
| **带宽/吞吐双优化** | GPU-CPU 带宽利用率最大化 + SSD 持久化吞吐优化，消除 I/O 瓶颈 |
| **一致性保证** | 跨步分段传输的核心难题——避免「部分新 + 部分旧」的不一致状态——通过梯度辅助机制解决 |

**关键创新点**:

| 维度 | 传统异步检查点 | GoCkpt |
|:-----|:---------------|:-------|
| **检查点保存时间窗口** | 单个训练步内完成全部传输 | **分散到多个训练步**，每步传输部分 |
| **GPU 计算中断** | 单步内 CPU 端 I/O 操作造成不可忽略的延迟 | 每步仅传输小片段，**延迟被训练计算自然覆盖** |
| **一致性维护** | 快照+同步点机制 | **梯度辅助的多步版本一致性恢复** |
| **可扩展性** | 模型参数增长 → 单步传输时间线性增长 | 参数增长 → 更多步数分担，**不产生单步瓶颈** |

**实验数据**:

| 指标 | 数值 | 对比基线 |
|:-----|:-----|:---------|
| **训练吞吐提升** | **最高 38.4%** | 行业传统异步检查点方案 |
| **训练中断时间减少** | **86.7%** | SOTA 检查点传输方法 |
| **最终吞吐提升** | **4.8%**（含中断时间后净增益） | SOTA 检查点传输方法 |
| **评估平台** | LLM 训练集群 | — |

**意义**:

1. **「跨步」设计是检查点 I/O 优化的方向性创新** 🧭：传统方案（无论同步还是异步）都假设「一个检查点必须在某步内完整保存」。GoCkpt 打破了这一假设——既然检查点本身的创建频次（每 N 步一次）远高于单步时间，完全可以将保存操作**时间维度打散**
2. **对检查点全栈优化的补充** 🏛️：GoCkpt 属于「系统 I/O 层」优化（与 06-27 的 liburing 聚合优化同层），但与差分检查点（DiffCp, 算法层）、稀疏检查点（MoEvement）、分级存储（TierCheck）**正交可叠加**：
   - 算法层消除 I/O 量 × 系统层优化 I/O 路径 × 跨步分散 I/O 时间 = 乘法效应
3. **与异步检查点的关系** 🔄：GoCkpt 不是替代异步检查点，而是在异步基础上增加**多步粒度的时间重叠**。异步已经将 GPU 计算与检查点 I/O 从「串行」变为「并发」，GoCkpt 进一步将并发从「单步粒度」细化为「多步粒度」
4. **局限**：论文未公开最大可支持的模型参数规模，跨步分段的一致性问题在大规模（>100B 参数）下是否仍能保持高效的验证不足

**📌 交叉引用**:
- 与 [DiffCp](./2026-06-26.md)（差分检查点, 89.2% 训练时间减少）——算法层 + 系统 I/O 层 + 时间分散层，三层正交叠加
- 与 [liburing Checkpoint I/O 优化](./2026-06-27.md)（3.9-7.6× 提升）——同属系统 I/O 层，liburing 聚焦 I/O 效率，GoCkpt 聚焦时间重叠
- 与 [TRANSOM](#1️⃣-⚙️-transom工业级-llm-容错训练系统toltcee-三引擎)（本日条目 1）——TRANSOM TCE 异步引擎 20×，GoCkpt 多步重叠再提升 38.4%——可叠加
- 与 [MoEvement](./2026-06-06.md)（NSDI'26, 稀疏检查点, 4× 开销减少, 31× 恢复加速）——稀疏减少开销，GoCkpt 隐藏开销

## 3️⃣ 🧮 ECRM：纠删码容错——推荐模型训练检查点的替代方案

**来源**: arXiv:2104.01981 · **2021-04-05 提交** · cs.LG/cs.DC · CMU

**作者**: Kaige Liu, Jack Kosaian, K. V. Rashmi（CMU 计算机科学系，存储系统方向）

**背景**: 深度学习推荐模型（DLRM）规模持续增长——embedding 表可占用数百 GB 显存，跨数十/数百台服务器分布式训练。在此规模下服务器故障频繁，检查点作为主要容错手段在正常训练和故障恢复时均产生显著开销，随着 DLRM 增长该问题只会恶化。本文提出**用纠删码（Erasure Coding）替代检查点作为 DLRM 训练的容错手段**。

**核心方法 — ECRM**:

| 组件 | 说明 |
|:-----|:------|
| **参数量选择编码** | 不编码所有 DLRM 参数，而是**选择性地对关键参数**（embedding tables + MLP 权重）进行纠删码编码 |
| **高效奇偶更新** | 纠删码需要维护奇偶校验信息——ECRM 设计了一套机制在训练过程中**正确且高效地更新奇偶校验**，避免每次参数更新都重算全部编码 |
| **零暂停训练** | 编码/解码过程完全异步，训练过程**无需暂停等待容错操作** |
| **一致性保证** | 恢复的参数保持与检查点等价的**一致性水平** |

**与传统检查点的对比**:

| 维度 | 检查点（Checkpointing） | 纠删码（ECRM） |
|:-----|:----------------------|:---------------|
| **训练时开销** | 定期保存完整/增量快照 → 训练中断 + I/O 瓶颈 | **仅维护奇偶校验编码** → 无 I/O 冲刺，开销持续均摊 |
| **恢复时过程** | 加载最新检查点（可能损失若干步进度） | **从编码即时重建**丢失参数（不损失训练进度） |
| **恢复不影响训练** | ❌ 恢复期间训练暂停 | ✅ **恢复期间训练可继续** |
| **存储开销** | 1× 参数大小（或增量） | 约 1.5×（取决于编码参数） |
| **计算开销** | 序列化/反序列化 | 编码/解码计算（GPU 友好） |

**实验数据**（基于 XDL 工业级 DLRM 训练框架实现）:

| 指标 | 数值 |
|:-----|:-----|
| **训练时开销减少** | **最高 88%** vs 检查点 |
| **故障恢复速度** | **最高 10.3× 更快** vs 检查点 |
| **恢复期间训练** | ✅ **可继续**进行（无暂停） |
| **实现基础** | XDL（开源工业级 DLRM 训练框架） |
| **适用模型** | 深度学习推荐模型（DLRM，含 embedding tables） |

**意义**:

1. **检查点的「替换」而非「优化」思路** 🔄：传统容错研究都是「优化检查点」（更快保存/更快加载/节省存储），ECRM 的思路是**完全用纠删码替代检查点**——这一范式转换对特定场景（DLRM 的强一致性需求 vs 编码计算的可矢量化性）有吸引力
2. **恢复期训练继续的特性对生产环境至关重要** ⏱️：大规模训练集群中，一次完整恢复（加载检查点 + 恢复训练）可能耗时数分钟。ECRM 允许恢复期间训练继续，有效训练时间大幅提升
3. **局限性** ⚠️：
   - **DLRM 特化**：该方案高度依赖 DLRM 的参数结构（embedding table 为主，MLP 为辅），未验证在 LLM 的纯 Transformer 架构上的适用性
   - **编码计算开销**：虽然编码在 GPU 上可高效执行，但每步更新奇偶校验仍增加额外计算，论文中「最高 88% 减少」是相对于高频率检查点场景
   - **数据新鲜度**：编码保护的是当前参数版本，故障时损失的是最后几步的增量更新——与检查点相比，损失粒度的语义不同
4. **在同类型场景的后续发展** 📈：纠删码容错思想在 DLRM 领域后续有延续工作（如 2022-2024 年期间 CMU Rashmi 组的序列工作），但未广泛扩散到 LLM 训练领域——这可能是因为 LLM 的交错异构参数（attention weights + MLP + layer norms）的编码效率不如 DLRM 的均匀 embedding table

**📌 交叉引用**:
- 与 [FT-HSDP](./2026-06-03.md)（数据集副本容错, 100K GPU）——数据级冗余（FT-HSDP 的数据并行副本）vs 编码级冗余（ECRM 的纠删码），不同冗余策略的对比
- 与 [SPARe](./2026-06-23.md)（ICML'26, 冗余堆叠, 600K GPU 规模）——冗余堆叠（SPARe, 完全副本）vs 纠删码（ECRM, 部分编码），开销—恢复时间权衡空间的不同探索
- 与 [Inshrinkerator](./2026-06-06.md)（动态量化检查点压缩, 39×）——检查点压缩（减少体积）vs 消除检查点（用编码替代），完全不同的容错哲学

## 📊 本轮 vs 上轮对比

| 维度 | 上轮（2026-06-29） | 本轮（2026-06-30） | 演进 |
|:-----|:-------------------|:-------------------|:-----|
| **RDMA 连接可靠性** | Varuna（65% 重传减少） | — | 🔴 未重复覆盖 |
| **容错 Reduce 理论** | 校正型 Allreduce（理论证明） | — | 🔴 未重复覆盖 |
| **工业级全栈容错** | — | **TRANSOM**（TOL+TEE+TCE, 20× CKPT, 28% 时间减少） | 🆕 **全新维度——三引擎编排系统** |
| **检查点时间隐藏** | — | **GoCkpt**（跨步重叠, 38.4% 吞吐提升, 86.7% 中断减少） | 🆕 **全新维度——多步分层检查点** |
| **纠删码替代检查点** | — | **ECRM**（DLRM, 88% 开销减少, 10.3× 恢复, 恢复期继续训练） | 🆕 **全新维度——检查点替代方案** |
| **检查点全栈体系** | 五层（算法 → I/O → 存储 → 压缩 → 选择） | **新增「时间隐藏」层 (GoCkpt) + 「替代」路径 (ECRM)** | 🔵 从五层栈扩展为六层 + 一替代路径 |

**趋势观察**:

1. **检查点全栈优化已形成六层 + 一替代的完备格局** 🏛️：
   - **算法层** → DiffCp（复用梯度消除 I/O）
   - **系统 I/O 层** → liburing 聚合（3.9-7.6×）+ GoCkpt 跨步重叠（38.4% 吞吐）
   - **存储层** → TierCheck（分级存储放置）
   - **压缩层** → BitSnap/Inshrinkerator（39× 压缩比）
   - **选择层** → LLMTailor（逐层选择性保存）
   - **时间隐藏层** → 🆕 **GoCkpt**（跨步分散 I/O 时间）← **本轮新增**
   - **替代路径** → 🆕 **ECRM**（用纠删码替代检查点）← **本轮新增**
   - 除存储层外，其余五层加替代路径**互不冲突、可叠加**

2. **容错训练系统的「工业级对抗」格局成型** 🏭：
   - **Meta FT-HSDP** → 数据并行副本为容错单元，100K GPU，80% 有效训练
   - **INTEL Optimus** → Aurora 超算 12K Intel GPU，MoE 训练
   - **TRANSOM** → TOL+TEE+TCE 三引擎编排，GPT-175B，28% 时间减少
   - **SPARe** → 冗余堆叠容错，ICML'26
   - 各系统面向不同并行策略和硬件生态，形成多元工业级容错实践库

3. **容错范式的「二分法」浮现** 🔄：
   - **检查点优化派**：多数研究（DiffCp/MoEvement/TierCheck/GoCkpt/Inshrinkerator）——接受检查点的必要性，从头优化各个环节
   - **检查点替代派**：少数探索（ECRM/冗余堆叠/数据级副本）——试图找到不依赖检查点的容错手段
   - 两派并存是健康的——说明社区对「检查点是否是最优容错原语」尚未有共识
