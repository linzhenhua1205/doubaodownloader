## 1️⃣ 💣 PuDGhost：Processing-using-DRAM 操作的计算结果损坏实证

**来源**: arXiv:2606.19119 · **2026-06-17 提交** · ISCA 2026 录用 · Onur Mutlu 团队 (ETH Zürich + 共同作者) · cs.AR/cs.DC

**概述**: Processing-using-DRAM (PuD) 是一种通过同时多行激活 (SiMRA) 将 DRAM 列作为计算引擎的新型计算范式。本研究首次揭示了 **PuDGhost** 干扰现象——给定列中的 PuD 操作因来自非参与行的干扰和并发计算列的串扰而产生错误结果。

**核心发现**:
| 干扰源 | 对 SiMRA 输出的影响幅度 | 说明 |
|:-------|:------------------------|:-----|
| 相邻非激活行的数据 | **高达 10%** | 未参与计算的行数据泄漏 |
| 并发计算列的数据 | **高达 48%** | 同 SiMRA 操作的其他列串扰 |

**实验规模**: 96 片真实 DDR4 DRAM 芯片（来自 12 个模组），15 项新实证观察

**缓解措施**:
1. **鲁棒列筛选** (Robust Column Screening) — 降低在 PuDGhost 存在下使用不可靠列的风险
2. **计算行布局** (Compute Row Layout) — 通过计算行之间设置专用行来缓解 PuDGhost

**意义**: 这是首个对 PuD 操作鲁棒性的实验表征。DRAM 密度扩展使得行和列间距缩小，干扰问题将日益严重。PuDGhost 威胁未来 Processing-using-DRAM 系统的可靠性基础。

**📌 交叉引用**: 与已有知识库中的「DRAM CE→UE 故障率」论点形成互补——传统 DRAM 可靠性分析关注的是标准读写操作，而 PuD 场景引入了全新的计算结果损坏维度。

## 2️⃣ 📡 ARGUS：10,000+ GPU 集群的生产级追踪与性能诊断系统

**来源**: arXiv:2606.20374 · **2026-06-18 提交** · cs.DC

**概述**: 大规模 LLM 训练需要始终在线的细粒度可观测性。粗粒度资源监视器无法定位根因，细粒度性能分析工具产生 5%-30% 开销和巨大追踪体积。ARGUS 实现了 **<2% 组合开销**、**~3,700× 压缩率**（每 rank 每步 10MB → 2.7KB）的始终在线追踪系统。

**架构亮点**:
- **三层观察分解**: CPU 调用栈 + 框架语义 + GPU Kernel 执行
- **统一数据管道**: 原始 kernel 事件 3,700× 压缩
- **渐进式诊断框架**: 迭代时间级 → 阶段级 → kernel 级三级渐进分析，自动隔离异常窗口、straggler rank 和退化 kernel

**生产验证**: 已在 **10,000+ GPU 生产集群部署超过 6 个月**，持续支持：
- **fail-slow 检测**: 计算 straggler、链路退化、流水线气泡放大
- **性能优化**: FlashAttention JIT 停顿、被通信症状掩盖的计算 straggler

**案例覆盖的异常类型**:
| 异常类型 | 说明 |
|:---------|:-----|
| 计算 Straggler | 单 rank 计算性能退化 |
| 链路退化 | 网络互联性能下降 |
| 流水线气泡放大 | 流水线并行中因故障导致的空闲放大 |
| FlashAttention JIT 停顿 | 即时编译引起的间歇性延迟尖峰 |
| 计算 Straggler 被通信掩盖 | straggler 表现为通信瓶颈而非计算瓶颈 |

**意义**: 代表了大规模 GPU 集群 fail-slow 诊断的 **SOTA 工程方案**——首次在 10K+ 规模下实现 <2% 开销的始终在线细粒度追踪，并经过 6 个月生产验证。

## 3️⃣ 🧬 Execution-State Capsules：完整 GPU 执行状态检查点与恢复

**来源**: arXiv:2606.20537 · **2026-06-18 提交** · cs.LG/cs.DC · Liang Su (独立作者)

**概述**: 当前主流 LLM 推理系统仅管理 KV cache 这一状态片段。本文针对低延迟、小批量、设备端物理-AI 推理（交互式 Agent、语音系统、机器人策略）提出 **execution-state capsules**——在承诺边界处对完整可恢复状态的图绑定检查点与恢复机制。

**核心技术**: FlashRT——白盒、后端面向的 kernel runtime，将捕获的图计划运行在连续静态缓冲区上，无块表间接引用。胶囊可快照、恢复、分支或回滚完整执行边界，涵盖 KV、循环状态、卷积状态、MTP 状态和元数据。

**性能数据**:
| 指标 | 值 |
|:-----|:---|
| GPU 内快照和恢复 | **<1ms** (子毫秒) |
| 2k tokens TTFT vs 冷 prefll | **3.9×** |
| 16k tokens TTFT vs 冷 prefll | **27×** |
| 字节级精确性 | ✅ (存储状态级 byte-exact) |
| Token 一致性 | ✅ (贪婪解码下 token-identical) |
| KV-only ablation 结果 | ❌ 发散——循环状态不可省略 |

**平台验证**: RTX 5090 ✅ / Jetson AGX Thor ✅ / DGX Spark ✅

**局限**: 不替代高吞吐 KV cache 推理；定义了一个互补的延迟优先复用点。

**意义**: 将 GPU 检查点从「KV 片段寻址」范式推向「完整执行状态边界复用」范式。子毫秒级恢复能力对交互式 AI Agent 场景有重要影响。

## 4️⃣ 🎭 The Correctness Illusion：LLM 生成的 GPU Kernel 的正确性幻觉

**来源**: arXiv:2606.20128 · **2026-06-18 提交** · cs.SE/cs.DC/cs.LG

**概述**: 当前 LLM 生成 GPU kernel 的基准测试（KernelBench、TritonBench、GEAK）通过**固定形状、小样本的 allclose 风格检查**来评估正确性。本文构建了一个受控语料库（24 个 Triton 和 CPU 替代内核：15 个正确对照 + 9 个注入 LLM 典型转录错误的有 bug 变体），展示了 allclose 检查的不足。

**核心发现**:
- **allclose 单形状 oracle**：无法检测注入的 bug（报"正确"）
- **op-schema 感知的种子 fuzzing + 高精度 CPU 参考**：9/9 buggy kernel 被捕获，15/15 正确对照通过
- **扩展到 26 个 ops**（含 FlashAttention 对）：在 5 类 GPU（RTX 3060 / A10 / L40S / A100 SXM4 / H100 NVL）上结果一致——10/10 幻觉捕获, 16/16 对照清白

**意义**: LLM 生成 kernel 的 allclose 单形状检查存在系统性漏洞——LLM 在测试输入上产生正确结果，但在非测试输入上计算错误。这种「正确性幻觉」对 AI 辅助代码生成场景下的 GPU kernel 可靠性构成实际威胁。

**📌 交叉引用**: 与之前跟踪的「GPU SDC 模式表征」(arXiv:2605.04213, DSN'26) 形成互补——SDC 来自硬件故障，而此论文指向 LLM 生成代码的正确性缺陷这一新来源。
