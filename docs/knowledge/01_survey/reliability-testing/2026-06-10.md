## 1️⃣ ResiHP — 动态混合并行：驯服 LLM 训练的设备性能倾斜

**来源**: arXiv:2605.06374, 2026-05 | 256-GPU 集群验证
**作者**: Tenghui Ma, Jihu Guo, Wei Gao 等（CUHK-Shenzhen）

### 问题
混合并行（TP+PP+DP）在万卡级 LLM 训练中，单个设备的**硬件故障**导致跨设备**性能偏斜（performance skew）**，整体训练效率急剧下降。

### 现有方案的盲区
1. **序列长度变异性被忽视**：数据集中 batch 间序列长度不同 → 迭代时间自然波动 → 被误判为 fail-slow
2. **混合并行中的性能偏斜**：一个设备的故障在 TP/PP/DP 三个维度上的传播方式不同，现有方案逐维度做适应性调整 → 检测开销冗余

### ResiHP 方案

**① Detector：工作量感知执行时间预测器**
- 输入：当前 batch 的序列长度分布 + 模型架构配置
- 输出：该次迭代的正常执行时间预测
- 作用：从迭代时间波动中**解耦（disentangle）**真正的设备故障，避免"正常波动→误报→误触发容错"

**② Scheduler：动态自适应调度**
- 自适应调整：并行组大小、模型分区、工作量调度策略
- 弹性：保留幸存设备全部参与训练，零闲置

### 性能数据

| 指标 | ResiHP vs SOTA |
|:-----|:---------------|
| 训练吞吐量提升 | **1.04–4.39×**（跨多种故障场景，256 GPUs） |
| 检测准确率 | 避免序列长度变异导致的**虚假 fail-slow 检测** |
| 检测开销 | 轻量级在线检测，无额外训练停顿 |

### 意义

与 Guard 形成**互补**——Guard 关注**在线性能监测 + 离线节点扫屏**（宏观节点健康度），ResiHP 关注**故障后混合并行的动态重调度**（微观并行策略调整）。两者结合可覆盖从检测到恢复的完整链路。

## 2️⃣ Guard — 大规模训练的可扩展 Straggler 检测与节点健康管理（MLSys 2026）

**来源**: arXiv:2605.17879, 2026-05 | **MLSys 2026 正会论文**
**作者**: Guanliang Liu, Abhinandan Patni, Congzhu Lin 等（大型科技公司联合团队）

### 问题
现有健康检查（NCCL 测试、GPU burn-in）仅关注**功能正确性**，无法检测**fail-slow 行为**（性能缓慢退化而非完全崩溃）。在数万 GPU、多月的训练中，微小的性能退化累积成巨大的效率损失。

### Guard 方案

**双组件架构**：

**① 在线性能监测（During Training）**
- 轻量级检测：在训练过程中实时采集 per-GPU 性能指标
- 检测目标：**急性故障**（崩溃）和**慢速退化**（fail-slow 行为）

**② 离线节点扫屏（Node Sweep）**
- 在节点进入生产前，对其进行**系统性评估与资质认证**
- 捕获传统诊断无法发现的**长期运行降低行为**

### 性能数据

| 指标 | 效果 |
|:-----|:-----|
| Mean FLOPs Utilization | **提升 up to 1.7×** |
| Run-to-run 训练步方差 | **从 20% 降至 1%**（20× 改善） |
| Mean Time To Failure (MTTF) | **显著提升**（作者报告） |
| 运维/调试开销 | **大幅降低** |

### 生产部署
- 部署于**大规模基础模型预训练**（frontier-scale foundation model pretraining workloads）
- 来自工业界的实战经验（17 位作者，跨工程与研究团队）

### 意义
1. **首次将 off-line node sweep 引入 LLM 训练健壮性流水线**，与在线监测形成闭环
2. **方差从 20% → 1%** 是惊人的指标——意味着训练稳定性质的飞跃，对大规模集群运维有直接价值
3. 与 FALCON (arXiv:2410.12588, 2024, 生产集群 10K+ GPUs, 99% 检测准确率, 60.1% 缓解效果) 和 ResiHP 构成 straggler 检测与缓解的完整拼图
