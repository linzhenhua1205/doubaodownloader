## 🏢 Databricks (6月11日集中发布6篇)

### 1️⃣ ⭐ Spatial SQL GA：几何/地理类型 + Iceberg v3 + Delta Sharing 全链路 (Jun 11)

**来源**: [Geospatial Unbounded: Spatial SQL GA](https://www.databricks.com/blog/geospatial-unbounded-spatial-sql-ga-aibi-maps-delta-sharing-and-iceberg-v3)

Databricks 正式 GA **Spatial SQL**，将地理空间能力深度嵌入 Lakehouse：

- **原生数据类型**: GEOMETRY (GA) + GEOGRAPHY (Public Preview) 列类型，直接存储在 Iceberg 或 Delta 表中
- **90+ 个 ST_\* 函数**: 符合 OGC 标准，覆盖 WKT/WKB/GeoJSON/EWKT/EWKB 导入导出、测量、构造、谓词、变换等
- **性能数据 (SpatialBench)**:
  - SpatialBench 12 个查询中 **8 个有显著提升**，幅度 **20% 到 15×**
  - 布尔集操作 (ST_Intersection, ST_Difference, ST_Union) 平均 **2× 加速** — 无需改代码
  - S&P Global 客户实测: **70% 更快查询**
- **AI/BI Dashboards 原生地图渲染**: Geometry/Geography 列直接呈现为地图，Genie 可用自然语言生成空间查询
- **Iceberg v3 支持**: Spatial SQL 可读写托管 Iceberg 表，且支持 Iceberg v3 地理空间数据类型 (Databricks Runtime 18.2+)
- **Delta Sharing for Geo**: 带 geo 列的表可通过 Delta Sharing 发布，接收方直接读取，无需数据提取
- **Apache Spark 4.2 贡献**: Databricks 正在向 Apache Spark 4.2 贡献 GEOMETRY / GEOGRAPHY 类型 (预计 2026 夏季发布)
- **分析意义**: Spatial SQL 消除了"空间数据库 + 仓库 + 可视化工具"三层拼凑架构，让所有空间分析在 Lakehouse 单一平台完成，且是完全开放的 (Iceberg v3 + Delta Sharing + Apache Spark 开源)

### 2️⃣ ⭐ Zerobus Ingest: 无服务流式摄取 — 12 GB/s 单表吞吐 (Jun 11)

**来源**: [Ingesting the Milky Way: Petabyte-Scale with Zerobus Ingest](https://www.databricks.com/blog/ingesting-milky-way-petabyte-scale-zerobus-ingest)

Databricks 发布 **Zerobus Ingest** — 完全托管的无服务流式摄取服务，Push API 直接写入 Delta 表，已 GA：

- **架构核心创新**:
  - **动态分区 (Dynamic Partitioning)**: 将排序保证从"分区级"提升到"流连接级"，连接 = 逻辑身份。Pod 热路由实现真正弹性伸缩 — 无需预配分区分区
  - **ZeroParser 零拷贝 Protobuf 解码器**: 自定义 Rust 实现，单遍解析 + 零内存分配，动态描述符下每 CPU 核 ~1 GB/s 吞吐，超越业界 codegen 实现
  - **写入前日志 (WAL)**: 延迟优化的 WAL 保障数据持久性，gRPC 双向流异步确认 offset
- **性能基准 (NASA NEOWISE 数据集, 200B 数据点)**:
  - **12 GB/s** 单表持续吞吐 (proto2 wire)
  - **1.04 万亿行** / 24 小时内完成
  - **2,048** 并发 Locust worker 模拟真实 fan-in 模式
  - 零预配置、稳定延迟
- **支持的格式**: protobuf (最优) > Arrow (批量) > JSON
- **路线图**: Kafka Producer API / MQTT / Rescue Column / Avro
- **开源**: ZeroParser 以 Zerobus SDK 形式开源 ([GitHub](https://github.com/databricks/zerobus-sdk))
- **分析意义**: Zerobus 标志着 Databricks 正式开始替代 Kafka 在 Lakehouse-inbound 流式场景中的角色 — 将流式摄取完全 Serverless 化，消除分区拓扑的管理负担

### 3️⃣ Mercedes-Benz Korea: Talk to Data — 500+ KPI 语义层 + Multi-Agent 系统 (Jun 11)

**来源**: [Unlocking semantics for AI: Mercedes-Benz Korea built trusted "Talk to Data" at scale](https://www.databricks.com/blog/unlocking-semantics-ai-how-mercedes-benz-korea-built-trusted-talk-data-scale)

- **核心架构**: Power BI DAX (500+ KPI) → **Unity Catalog Metric Views** (AI-ready 语义层) → Genie Spaces + Agent Bricks 多 Agent
- **DAX-to-Metric-View 自动转换器**: Databricks 为此案例构建的 Pipeline，解析 Power BI 语义模型 → 提取 DAX Measures → 生成 UC Metric Views (含 source/joins/dimensions/measures) → 标记不可自动转换项。已作为 Genie Code 技能 (Private Preview)
- **五种 AI-ready 流程**: Prepare → Build → Organize by Domain → Test Incrementally → Validate & Release
  - 每个 Genie Space ≤30 个 UC 条目，按业务领域组织
  - 目标: Genie 回答与 Power BI 报告 **100% 对齐**
- **Agent Bricks 多 Agent 架构**: 父监督 Agent 路由 → 角色 Agent (CFO/销售经理/市场分析) → 跨 Genie Spaces 整合 → Unity Catalog 行列权限
- **可复制的 Playbook**: 6 步法供其他 Mercedes-Benz 市场采用
- **分析意义**: 这是目前公开最完整的"企业级 Talk to Data"落地案例 — 展示了从现有 BI (Power BI) → AI-ready 语义层 (UC Metric Views) → 多角色 Agent 的完整迁移路径，且强调 100% 回答对齐而非"大概正确"

## 📄 arXiv 新论文 (6月8-12日 cs.DB 最新提交)

### 8️⃣ Joseph Hellerstein: Determination Provenance (Jun 10)

**来源**: arXiv:2606.10270, cs.DB

UC Berkeley 数据库泰斗 Joseph Hellerstein 的最新力作，将 **Determination Provenance**（确定性溯源）形式化为代数系统：

- **核心贡献**: 从数据溯源的基础问题出发 — "如果输出改变了，是哪些输入改变导致的？" — 构建完整的 Determination Provenance 代数
- **理论深度**: 15 页正文 + 34 页总篇幅，横跨 Databases + Distributed Computing + Logic 三域
- **分析意义**: 这是溯源领域的奠基性理论工作，对 Lakehouse 中的数据变更追踪、数据合约验证、Agent 执行溯源有深远的基础支撑意义

### 9️⃣ Flash-GMM: 内存高效的软聚类 Kernel (Jun 10)

**来源**: arXiv:2606.10896, cs.LG/cs.DB/cs.IR

Intel 团队提出 **Flash-GMM** — 专为大规模软聚类设计的内存高效 Kernel：

- **技术**: 通过内存布局优化 + 向量化来加速高斯混合模型 (GMM) 的 EM 算法
- **适用场景**: 大规模数据湖中的聚类分析、匿名分群、异常检测
- **跨域价值**: 同时优化 DB 查询 (聚类) + IR 排序 (语义分群) + ML 训练

### 🔟 Robust Active Learning for Few-Shot Text-to-SQL 示例选择 (Jun 10)

**来源**: arXiv:2606.10125, stat.ML/cs.DB

- **问题**: Text-to-SQL 中的 Few-Shot 示例选择高度依赖于评估指标，不同指标导致不同示例策略
- **方案**: 鲁棒主动学习方法，自动选择对多个评估指标都稳健的示例组合
- **分析意义**: Few-Shot 示例选择是当前 Text-to-SQL 系统的关键瓶颈，这项工作提供了系统的自动化解决方案

### 1️⃣1️⃣ Bespoke-Card: 合成工作负载特定基数估计器 (Jun 9)

**来源**: arXiv:2606.09361, cs.DB (AIDB@VLDB'26)

- **核心思想**: "与其调优，不如生成" — 用 LLM 生成工作负载特定的训练数据，训练轻量级基数估计器
- **方法**: 分析查询负载模式 → 合成针对性训练集 → 训练专用估计器，替代通用成本模型
- **分析意义**: 开创新的 Query Optimization 范式 — 从通用 Cardinality 估计 → 工作负载特定估计器的按需生成

### 1️⃣2️⃣ Fast LLM-Based Semantic Filtering: 统一框架 + 自适应两阶段方法 (Jun 9)

**来源**: arXiv:2606.08090, cs.DB/cs.AI (Ailamaki 组, EPFL)

- **统一框架**: 将 LLM-based semantic filtering 的各种方法统一建模
- **自适应两阶段**: 粗筛 (快速近似) + 精筛 (LLM 精确判定)，根据选择性动态调整两阶段切换点
- **分析意义**: 为 AI-powered 查询处理中"LLM 调用成本"优化提供了系统性方案

### 1️⃣3️⃣ DataEvolver: 面向 LLM 训练的自动数据准备 (Jun 8)

**来源**: arXiv:2606.07001, cs.DB/cs.AI

- **技术**: 多层级自演化 — 从记录级 → 字段级 → 模式级，逐步提升训练数据质量
- **方法**: 结合数据分布分析 + 质量评估 + 自动修复，无需人工标注
- **分析意义**: 为 Lakehouse 中存储的训练数据质量治理提供了自动化方案

### 1️⃣4️⃣ Auto-Relate: 统一发现可靠函数依赖关系 (Jun 8)

**来源**: arXiv:2606.07060, cs.DB (Microsoft Research + 多校)

- **方法**: 利用统计检验 (Fisher Exact Test, Chi-squared) 统一检测多种函数关系 (FD, CFD, DC 等)

- **分析意义**: 数据湖中的数据关系发现从"规则驱动"走向"统计检验驱动"

### 1️⃣5️⃣ Architectural Evolution for DB Systems in AI-Ready Data Platforms (Jun 9)

## 📊 综合趋势判断

1. **Streaming 基础设施全面 Serverless 化**: Zerobus Ingest (12 GB/s/表, 无分区管理) 标志着流式摄取进入 Serverless 时代，Databricks 正在构建端到端的 Lakehouse-inbound streaming 替代方案（替代 Kafka Pipeline 角色）

2. **地理空间智能成为 Lakehouse 原生能力**: Spatial SQL GA + Iceberg v3 地理数据类型 + Apache Spark 4.2 贡献，消除了空间数据需要独立引擎的历史——2026 年可能是"Lakehouse 原生地理空间能力"的元年

3. **企业语义层成为「Talk to Data」质量的关键差距**: Mercedes-Benz Korea 案例明确展示了 500+ KPI 作为 Metric Views 治理 → Genie 回答 100% 对齐 BI 报告的路径。Snowflake Cortex Sense (83% vs 47%) 同样证实：**AI 回答质量的最大瓶颈不是模型，是企业语义上下文的完备性**

4. **Data + AI Summit 2026 (6/15-18, SF) 即将开幕**: 预期 Lakebase/Apache Spark 4.2/下一代 Genie 等重大发布——此次 Spatial SQL GA 和 FDE 组织发布可以视为预热

5. **数据库理论基础研究的持续活力**: Hellerstein 的 Determination Provenance 代数 + Bespoke-Card 合成估计器 + Fast LLM-Based Semantic Filtering 统一框架，表明 cs.DB 社区仍在为 AI 时代的 Lakehouse 提供核心理论支撑
