## 1. 🏢 Databricks — ETL Migration Decision Framework (Jun 26)

**来源**: [Databricks Blog](https://www.databricks.com/blog/decision-framework-etl-migration-databricks)

### 核心主张: 三条路径，不唯一解

Databricks 发布了一套完整的 ETL 从传统数仓迁移到 Lakehouse 的决策框架，明确定义三条技术路径：

| 路径 | 适用团队 | 核心逻辑 |
|:-----|:---------|:---------|
| **Lakehouse (Databricks SQL)** | SQL-heavy 团队、DBA | SQL tasks + stored procedures，Photon 加速 |
| **Spark Declarative Pipelines (SDP)** | 数据工程师 | 声明式管线的执行顺序/重试/扩缩容由引擎决定，Enzyme 自动决定增量 vs 全量重算 |
| **PySpark / Spark SQL notebooks** | Python/Spark 工程师 | 完整控制权，适合复杂业务逻辑/ML 特征工程/API 集成 |

### 架构关键细节

- **SDP (Lakeflow 子集)**: 内置数据质量约束、自动依赖解析、统一 batch+streaming
- **Photon 向量化引擎**: 加速大规模 joins/aggregations/window functions，也加速 Arrow-based Pandas UDFs
- **Four-Phase 迁移方法论**:
  1. **Assess** — 采集 legacy 系统的 CPU time/runtime/频率/源目标表，用 Lakebridge Analyzer 自动化
  2. **Quick Wins** — 低风险高可见性 workload 先行，运行双系统对比行数/校验和
  3. **Modernize** — 重构管线：数据质量约束替换手动检查、CDC 替换 batch、物化视图减少复杂性、Unity Catalog 统一元数据/权限/审计
  4. **Optimize** — 合并冗余 ETL pipelines、重访 batch vs streaming 边界

- **Lakebridge 工具链**: 覆盖 profiling→code conversion→validation，声称自动化率达 **80%**，项目时间缩短约 **2×**
- **三个反模式**: 一刀切选路径 ✅ 只测迁移百分比不测并行运行 ✅ 在 Lakehouse 上重建旧的 schedule 层 ✅

## 2. 🔓 Delta Lake 4.3.0 (Jun 18) — UC API 架构细节

**来源**: [GitHub: delta-io/delta v4.3.0](https://github.com/delta-io/delta/releases/tag/v4.3.0)

> ⚠️ 该版本在 6/26 已收录，本次补充 UC Delta REST API 和 V2 Checkpoint 的具体架构实现细节

### Unity Catalog Delta REST API 架构

Delta 4.3.0 引入了 **UC Delta REST API** 作为 Spark 访问 UC 管理表的默认路径，架构变化如下：

```
之前: Spark → Delegate Connector (legacy) → UC
之后: Spark → UC Delta REST API (默认) → UC
                                   ↕ 服务端 commit 验证
                                Delta 事务日志
```

关键实现细节：
- **服务端 commit 验证**: UC 作为 single source of truth，验证 schema 兼容性、协议特征一致性
- **服务端告知 table features**: UC 主动告知客户端当前表的已启用 feature，避免客户端推测
- **Intent-based 元数据更新**: 写操作提交 "intent"，由服务端决定最终的元数据变更，而非客户端直接写 log
- **支持操作范围**: table loads、CREATE/CTAS、REPLACE、所有 DML（INSERT/MERGE/UPDATE/DELETE）、schema evolution、auto-merge、ALTER TABLE
- **非 Delta 表继续走 legacy delegate**: name-based 和 path-based 访问不受影响
- **未来扩展**: 为 Flink、Trino 等引擎的 REST 协议访问奠定基础

### V2 Checkpoint 性能硬化

- 默认每个 sidecar 文件 **50,000 actions**（之前无此分割），sidecar 文件自动分片
- Checkpoint 写入自动并行化，大表场景下 checkpoint 写入吞吐显著提升

### 其他架构改进

- **replaceOn / replaceUsing API**: 按列匹配（`replaceUsing`）或按条件（`replaceOn`）实现部分数据替换，不需要全表 OVERWRITE
- **UniForm 增量元数据转换**: Iceberg 元数据跟随 Delta commit 原子写入，只转换变更的 log range（非全量扫描）
- **Variant column 统计收集**: write-time 收集 min/max statistics，支持数据跳过（data skipping）

## 3. 📐 arXiv 2606.23537 — SQLConductor: Search-to-Policy Learning for Text-to-SQL (Jun 22)

**来源**: [arXiv 2606.23537](https://arxiv.org/abs/2606.23537)
**核心成果**: BIRD-Dev **73.2% EX**，超越此前所有方法

### 范式突破: 从固定管线 → 搜索→策略蒸馏

SQLConductor 提出 **Search-to-Policy Learning** 新范式：

```
传统管线: Schema Linking → SQL Generation → Execution → (fixed order)
SQLConductor: MCTS 搜索候选工作流 → 稳定性估计 → 策略蒸馏 → 推理时逐步编排
                                                  ↕ 课程强化学习
```

### 架构创新

1. **Step-wise Orchestration**: 将 Text-to-SQL 子任务编码为专门的动作（schema link、decompose、generate、verify、refine），每一步策略模型根据中间产物和反馈选择下一个动作
2. **MCTS 搜索**: 在训练阶段用 Monte Carlo Tree Search 探索候选工作流空间，收集多样化的成功路径
3. **稳定性估计 (Stability Estimation)**: 识别哪些编排模式在各种条件下都稳定成功，作为高质量监督信号
4. **Stability-weighted SFT**: 优先学习高稳定性编排模式
5. **课程强化学习**: 从简单 query 的编排策略逐步过渡到复杂 query
6. **策略模型轻量化**: 协调 frozen 的大 action model，推理开销远低于直接训练大 backbone

### 关键结果

- **BIRD-Dev: 73.2% EX** — 大幅超越此前 SOTA
- **强泛化性**: Out-of-distribution 数据集同样表现优异
- **动作模型无关性**: 可插拔替换 backbone LLM，无需重新训练策略层

### 意义

标志着 Text-to-SQL 的编排范式从 "hand-crafted 固定管线" 正式进入 "搜索→策略蒸馏" 时代。策略模型轻量化意味着可以低成本部署在 agent 系统中，而非每个 query 都调用超大模型做完整端到端推理。

## 4. 🔬 arXiv 2606.05906 — ACE-SQL: 联合优化 Schema 检索与 SQL 生成 (Jun 4)

**来源**: [arXiv 2606.05906](https://arxiv.org/abs/2606.05906)
**核心成果**: BIRD Dev **65.3% EX**，仅用 **0.93k** 输出 token/query

### 核心洞察

现有 schema linking 方法的致命问题：用 **静态 gold-column 标注** 训练 retriever，但这些标注对当前生成器策略可能不是最优的。

### 架构: 双向自适应 RL

```
Generator 输出 → 构建在线列集池 → 识别执行正确的 rollout 的列集
        ↕                                      ↕
Retriever 调整目标                                               Generator 调整策略
```

1. **Empirical Credit Assignment**: 从 generator rollout 中构建在线列集池，识别与执行正确 rollout 最常关联的列集作为 retriever 目标
2. **双向自适应**: retriever 适配 generator 能正确执行的列集，generator 适配 retriever 的 schema 选择（在 execution feedback 下）
3. **轻量训练**: 约 **3k 合成 question-database pairs** 即可完成 RL 训练
4. **输出高效**: 平均仅 0.93k tokens/query，远低于全 schema 生成方案

### 意义

ACE-SQL 证明了 schema linking 和 SQL generation 的联合优化在轻量 RL 设置下即可显著超越独立优化的两阶段方案。输出 token 数量的压缩意味着生产环境中的成本和延迟大幅降低。

## 5. 🔬 arXiv 2606.05836 — ProSPy: SQL-Python Agentic Framework for Enterprise Text-to-SQL (Jun 4)

**来源**: [arXiv 2606.05836](https://arxiv.org/abs/2606.05836)
**核心成果**: Spider 2.0-Lite **60.15%**, Spider 2.0-Snow **60.51%** (Claude-4.5-Opus)

### 核心挑战

企业级数据库面临三大障碍：
- **大且异构的 schema**: 难以在 prompt 中完整包含
- **不完整的元数据**: 表注释、字段描述往往缺失
- **复杂的分析问题**: 单条 SQL 无法解决，需要多步推理

### 架构: 四阶段 Profiling-driven 流程

```
Phase 1: Automatic Profiling → 提取细粒度数据级证据（值分布、NULL 率、唯一值数）
Phase 2: Schema Pruning → 将大 schema 逐步裁剪到任务相关上下文
Phase 3: SQL 中间视图 → 用 dialect-agnostic SQL 接口获取中间结果
Phase 4: Python 灵活分析 → 在中间结果上进行 Python 灵活分析
```

### 关键设计

- **Profiling 驱动**: 不依赖不可靠的元数据，而是通过自动数据 profiling 获取实际数据特征
- **SQL + Python 组合**: 用 SQL 高效操作大规模表，用 Python 做灵活的下游分析
- **Dialect-agnostic**: SQL 层抽象掉了不同 SQL 方言的差异
- **无多数投票**: 即使不使用多次采样和投票，也取得 SOTA

### 意义

ProSPy 展示了企业级 Text-to-SQL 的正确方向：不是让模型一次性生成完美 SQL，而是结构化为 **探索→缩小范围→执行→分析** 的多阶段 agent 流程。Profiling 驱动替代元数据依赖是处理真实企业数据库的关键实践。

## 6. 📐 arXiv 2606.08317 — AI-Ready 数据库架构选择框架 (Jun 6)

**来源**: [arXiv 2606.08317](https://arxiv.org/abs/2606.08317)
**作者**: Mohit Srivastava

### 核心贡献

首个跨范式统一评估框架，对 **13 种数据库范式** 进行结构化对比：

| 维度 | 内容 |
|:-----|:-----|
| 评估维度 | 9 个架构维度（存储计算分离、workload 特化、AI 集成度等） |
| 对比范式 | 事务型(6) + 分析型(4) + AI 型(3) = 13 |
| 选择流程 | workload 特征化 → 约束过滤 → 兼容性评分 → 推荐 |

### 三个演化模式

1. **存储计算分离 (Decoupling)** — 几乎所有现代系统都在走这条路
2. **Workload-driven 特化** — 系统为特定 workload 模式进行深度优化
3. **AI-ready 平台融合** — 趋向集成 lakehouse 存储 + 特征处理 + 语义检索的 AI 参考架构

### AI Ready 参考架构

```
         ┌──────────────────────────────────────┐
         │   AI Applications (RAG / ML / Agent)  │
         ├──────────────────────────────────────┤
         │   语义检索层 (Vector DB)              │
         ├──────────────────────────────────────┤
         │   特征处理层 (Feature Store)          │
         ├──────────────────────────────────────┤
         │   Lakehouse 存储 (Iceberg/Delta)      │
         └──────────────────────────────────────┘
```

**Delta Lake UC REST API 架构确立**

Delta 4.3.0 的 UC Delta REST API 不再只是实验特性，而是 **Spark 默认路由**。服务端 commit 验证 + 服务端告知 features + intent-based 更新，使得 Delta 从 "文件系统级别的事务" 正式升级为 "目录服务级别的一致事务"。这对多引擎互操作（Flink/Trino 通过同一 REST 协议访问 Delta）意义深远。
