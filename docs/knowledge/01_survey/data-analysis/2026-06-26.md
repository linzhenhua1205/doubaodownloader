## 🏗️ Apache 基金会

### ① Apache Flink 2.3.0（Jun 25, 2026）

> **重磅发布**：15 个 FLIP 全面落地，Flink SQL 能力大幅跨越，原生 S3 文件系统移除 Hadoop 依赖

- **SQL 层 changelog 操作符** — `FROM_CHANGELOG` / `TO_CHANGELOG`（FLIP-564）:
  - `FROM_CHANGELOG`: 将携带操作列的 append-only 流转换为动态表，可配置 `op_mapping` 适配自定义 CDC 格式
  - `TO_CHANGELOG`: 首次提供 SQL 级操作符，将 retract/upsert 流物化为 append-only changelog 流，适配件库审计、append-only sink
  - 当前覆盖基础场景，后续将扩展 PARTITION BY、invalid_op_handling 等

- **物化表进化**（FLIP-550/557）:
  - DDL 对等：`CREATE MATERIALIZED TABLE` 现在支持显式列定义（含 watermark / primary key），`ALTER MATERIALIZED TABLE` 获得 ADD/MODIFY/DROP/RENAME TO
  - `START_MODE` 子句：FROM_BEGINNING / FROM_NOW / FROM_TIMESTAMP / RESUME 四种策略，避免每次表演化都全量重算

- **SinkUpsertMaterializer 重构**（FLIP-558）:
  - upsert key ≠ primary key 时，新增 `ON CONFLICT` 子句：`DO NOTHING` / `DO ERROR` / `DO DEDUPLICATE`
  - **默认改为**：key 不匹配时规划阶段直接报错，必须显式指定冲突策略
  - 基于 watermark 的记录压缩，通过 `table.exec.sink.upserts.compaction-mode` 控制

- **原生 S3 文件系统**（FLIP-555）:
  - 全新构建，基于 **AWS SDK v2**，移除 Hadoop/Presto 依赖
  - 统一提供 FileSystem + RecoverableWriter（exactly-once streaming sink）
  - 非阻塞 I/O，原生 IRSA/IAM，`s3.*` 独立配置命名空间
  - 实验性功能（功能完整）

- **运行时改进**:
  - **自适应分区选择**（FLIP-339）：下游加载感知的负载均衡，偏斜场景吞吐提升 **~3×**
  - **AdaptiveScheduler rescale 历史**（FLIP-487/495）：新的 Web UI "Rescales" 标签页 + REST API
  - **Watermark alignment buffer**：消除 backlog 处理速度限制（之前 30 event-time sec / 1 real sec 的瓶颈被打破）
  - **恢复期间 checkpointing**（FLIP-547）：大 unaligned checkpoint 场景下恢复效率大幅提升
  - **应用级生命周期管理**（FLIP-549/560）：cluster-application-job 三层模型，新 Applications tab
  - **OTel 导出健壮性**（FLIP-553）：gzip 压缩 + batch 分片

- **版本升级注意**: Scala 2.13 默认，**Flink 2.2 → 2.3 升级需关注 SinkUpsertMaterializer 行为变更**

## 🔓 开放格式

### ③ Delta Lake 4.3.0（Jun 18, 2026）

> 继 4.2.0（Flink Kernel connector）之后，4.3.0 聚焦 Unity Catalog 深度集成 + 选择性数据替换 + UniForm 增量转换

- **Spark: Unity Catalog Delta REST API 集成**:
  - Spark 默认通过 UC Delta API 路由 UC 管理表的操作（CREATE/CTAS/REPLACE/DML/schema evolution 等）
  - 服务端 commit 验证 + 服务端宣告表特性 + 基于意图的元数据更新
  - 为 Flink、Trino 等其他引擎的后续支持奠定基础

- **选择性数据替换 DataFrame API**:
  - `replaceUsing`: 替换指定列匹配的行
  - `replaceOn`: 替换满足用户自定义条件的行
  - 替代传统全表 overwrite，适合增量修正场景

- **UniForm 进展**:
  - Iceberg 元数据**原子性写入 Delta commit**，不再有元数据不一致窗口
  **增量转换**：只转换变更的 log range，大幅降低开销
  - 支持 **Spark 4.1**

- **Sharing 流式 + CDF 支持**:
  - Delta Sharing 增强：Spark Structured Streaming + Batch CDF，自动 Delta 响应解析，Parquet→Delta 流式转换
  - `Trigger.AvailableNow` 支持共享表

- **Kernel 增量 version-checksum**: `tableSizeBytes` / `numFiles` 增量更新，大表健康诊断不再需要全量 log replay

- **V2 Checkpoint 性能优化**: Sidecar 默认 50,000 actions/part，大表 checkpoint 自动并行化

## 📝 arXiv 论文精选

### ④ EXPO-SQL: Clause-level Policy Optimization（2606.23693, ACL 2026 Findings）

> **关键创新**: 将 Text-to-SQL RL 从 query-level 粗粒度奖励提升到 clause-level 细粒度

- **问题**: 现有 RL 方法对 SQL 中所有 clause 施加统一奖励，正确 clause 和错误 clause 获得相同信号
- **方案**: 分析执行结果（错误信息 + clause 级增量执行）定位错误 clause，分配 clause 级奖励
- **结果**: 显著超越 SFT / prompting / 现有 RL 方法，已吸收为 ACL 2026 Findings
- **意义**: clause-level 粒度是 Text-to-SQL RL 的下一个方向，与 CAPER（clause-aligned PRM）形成互补

### ⑤ DecoSearch: 复杂度感知路由 + 计划级修复（2606.17821, Jun 16）

> **训练无关**的 Text-to-SQL 框架，70.53% BIRD，token 消耗低一个数量级

- **架构三阶段**:
  1. **Schema Selector**: 轻量级 schema 剪枝，保留相关表和列
  2. **LLM Judger**: 判断是否需要分解（简单→直接生成，复杂→DAG 子问题）
  3. **Topology Refiner**: 执行失败时重构推理计划（区分"分解错误" vs "SQL 生成错误"）
- **RAG 增强**: 语义相似的训练示例作为 decomposer 的 grounding
- **结果**: BIRD 70.53% + Spider 88.31%（DeepSeek backbone），训练无关，可作为任意模型的 wrapper
- **Token 效率**: 比竞争方法少一个数量级

### ⑥ RefGRPO: 校准反射 + 免费校准奖励（2606.14211, Jun 12）

> **核心发现**: LLM Agent 存在系统性"反射差距"——看到执行反馈后反而错误评估自身输出

- **Reflection Gap**: 观察到环境反馈后 LLM 对自身输出评估失准，标准 RL 因 credit-assignment 错配基本无效
- **RefGRPO**: 在标准 RL 上增加两个关键组件：
  1. **免费校准奖励（free calibration bonus）**：对比 agent 自身反射与实际结果，无需额外 reward model
  2. **动态系数调度**
- **结果（5 个 Text-to-SQL 基准）**:
  - 过度自信率：**44.4% → 7.7%**
  - 任务准确率：**75.1% → 76.5%**
- **衍生能力**: 校准后的反射可作为验证器，支持更好的 self-improvement 和 test-time selective prediction
