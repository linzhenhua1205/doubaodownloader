## 一、产品动态

### 1️⃣ Apache Spark Real-Time Mode GA — 单引擎流批一体里程碑

- **来源**: [Databricks Blog, 2026-06-03](https://www.databricks.com/blog/apache-spark-real-time-mode-gaming-better-way-do-real-time-sessionization)
- **摘要**: Databricks 正式发布 Spark Real-Time Mode GA（已于 2026-06-08 跟踪过上线消息，本次为完整性能数据披露）。配合新 `transformWithState` 算子实现有状态流处理，timer-Driven 语义独立于输入数据触发。
- **性能数据**:
  - 游戏会话化场景：**500K 输入事件/分钟** → 8M 心跳输出/分钟（16× 放大比）
  - 活跃会话数 **4M** 并发
  - 端到端延迟：**432ms p99**，较微批模式 **20× 加速**
  - 单 `StatefulProcessor` 类通过 `handleInputRows()` + `handleExpiredTimer()` 完成完整会话生命周期
- **意义**: 替代 Flink 和自建方案的统一引擎路线正式落地，切换只需改 trigger 参数，无需重写代码

### 2️⃣ Snowflake CoWork — 个人工作 Agent 全能力发布

- **来源**: [Snowflake Blog, 2026-06-02](https://www.snowflake.com/blog/snowflake-cowork-personal-work-agent/)
- **摘要**: Snowflake 正式发布 CoWork（原 Snowflake Intelligence），定位为每位知识工作者的个人工作 Agent。核心能力矩阵：
  - **Cortex Sense**（Private Preview 即将到来）— 自动学习企业业务语义层，从查询历史、元数据、Power BI/Tableau 仪表板信号中理解收入定义、财务日历、快照表等概念
  - **Deep Research**（即将 GA）— Agent Swarm 编排系统，跨结构化+非结构化数据的混合深度研究
  - **Analytical Search**（即将 Public Preview）— 对非结构化数据的多步聚合/比较/趋势分析
  - **Multi-Agent Orchestration**（即将 Public Preview）— 自动路由到最佳数据源和工具
  - **Memory**（Public Preview）— 持续个性化，跨会话记住偏好和工作模式
  - **User Skills**（Public Preview）— 将多步工作流录制成可复用技能
  - **Artifacts**（即将 GA）— 保存可刷新、RBAC 感知的分析产出和交互式仪表板
  - **Automations & Time-Based Subscriptions**（Public Preview）— 后台巡检，自动推送异常到 Slack/邮件
  - **Async Agent API**（即将 GA）— 耗时任务异步执行，支持数分钟到数小时
  - **Slack App**（Private Preview）— 内联对话+图表+线程跟进的 Slack 集成
  - **iOS App**（即将 GA）— Face ID + 完整历史
  - **Agent Studio**（即将 GA）— Agent 构建、评估、治理的中心化平台
- **性能数据**:
  - Cortex Sense 使 CoWork 在复杂企业查询上的**准确率从 47% 提升至 83%**（对比无 Sense 的原始 CoCo+CoWork）
  - 对比 Frontier Coding Agent + Snowflake MCP（23% 准确率），Cortex Sense 加持后 **3.6× 准确率提升**
  - Deep Research 在混合基准上超越单 Agent 系统 **>33%**
- **意义**: Snowflake 从"数据湖仓"向"AI 工作 Agent 平台"的全面转型，Cortex Sense 解决了企业 AI 落地的最大痛点——业务语义理解

### 3️⃣ Databricks Query Tags Public Preview

- **来源**: [Databricks Blog, 2026-06-02](https://www.databricks.com/blog/query-tags-context-your-warehouse-queries-have-been-missing)
- **摘要**: 为每条 SQL 执行附加自定义 KV 标签（如 `project:finance_planning`），标签记录在 Query History System Table 中，支持标准 SQL 查询和 Genie 自然语言分析。
- **关键能力**:
  - **Partner Tool 自动标签**: dbt 自动注入 model name/core version/materialization type；Power BI/Tableau 支持连接级标签配置
  - **API 级标签**: SQL Statement Execution API、Python/Go/Node.js 等连接器均支持
  - **Ad-hoc 标签**: SET QUERY_TAGS 单行 SQL 即为会话打标签
  - **Genie 集成**: 可直接用自然语言问 "Which dbt model had the most queries?"
- **采用数据**: 已数百客户、**数百万查询/周**
- **意义**: 解决共享 Warehouse 成本归属和查询溯源这一长期痛点，是湖仓可观测性的关键基础设施

## 二、学术界新论文

### 4️⃣ NL2Pipe: 自然语言→语义算子管线的三阶段编译

- **来源**: [arXiv:2606.04641, 2026-06-03](https://arxiv.org/abs/2606.04641), Wenkai Dong 等
- **摘要**: 提出 NL2Pipe 中间件系统，将自然语言问题编译为可执行的语义算子管线（filtering/joining/mapping/aggregating over tables, text, images）。将问题分解为三阶段编译：
  1. **Query-Data Linker** — 将问题实体与数据绑定，发现桥接实体
  2. **Semantic Planner** — 生成后端无关的算子动作计划
  3. **Code Generator** — 根据后端签名自动生成可执行代码
- **性能数据**:
  - 复杂跨源 Workloads 上 F1 比基线 **提升 60%**
  - 成本可预测、延迟有竞争力（具体数据见论文实验部分）
  - 三层分离使同一计划逻辑支持多个后端（Palimpzest, LOTUS 等），无需重新适配
- **意义**: 将 Text-to-SQL 从"仅关系数据"扩展到"异构数据（表+文本+图片）+语义算子"，是 AI Agent 与数据湖仓结合的关键突破

---

### 5️⃣ GraftDB: 并发分析查询的动态折叠引擎

- **来源**: [arXiv:2606.04303, 2026-06-03](https://arxiv.org/abs/2606.04303), Genki Kimura & Kazuo Goda
- **摘要**: 提出 GraftDB 多查询执行引擎，通过 **State-Centric Execution（状态中心执行）** 动态折叠并发到达的查询——即将运行中查询的算子状态视为共享状态，后到的查询可"嫁接"到已有状态上进行观察或贡献。核心机制：
  - **State Lens** — 每个查询通过自己的状态透镜只观察符合其语义的合法状态
  - **Query Grafting** — 自动识别已有状态可满足的部分 + 剩余部分可继续共享执行的部分
- **性能数据**:
  - TPC-H 动态并发负载：**2.17× 吞吐量提升**
  - 过载开放到达：P95 响应时间降至基线 **0.17×**
- **意义**: AI Agent 场景下大量并发相似查询（如多 Agent 对相同数据集的迭代分析），GraftDB 提供了一种无侵入的工作共享方案

---

### 6️⃣ Puffin-Backed Vector Indexes: Iceberg 原生向量索引

- **来源**: [arXiv:2606.04196, 2026-06-02](https://arxiv.org/abs/2606.04196), Artur Borycki
- **摘要**: 提出将分布式 ANN 索引嵌入 Apache Iceberg 表格式的设计模式——使用 Puffin 侧边文件存储 Vamana 图索引，通过 Snapshot Summary 绑定。核心技术：
  - **分片图索引的 Puffin 二进制布局**
  - **Coordinator-Executor 协议** 实现分布式索引构建、探测和增量刷新
  - **Tiered Probe 策略**：小 centroid 索引在 Coordinator，大 DiskANN 图在 Executor SSD
  - 利用 Iceberg REST Catalog 的乐观并发提交路径
- **继承特性**（零实现成本）:
  - 原子性、Time Travel、多引擎可读、孤立文件 GC
  - 评估面向 **10⁹ 向量** 规模的查询和构建性能
- **意义**: 将向量索引管理简化为标准的 Iceberg Snapshot 操作，无需独立索引存储层，是"Lakehouse + 向量搜索"融合的关键一步

---

### 7️⃣ SPA: SQL-Plan-Aware RL 框架—用物理执行反馈训练 LLM 重写查询

- **来源**: [arXiv:2606.08620, 2026-06-07](https://arxiv.org/abs/2606.08620), Xinyi Huang & Zhengjie Miao
- **摘要**: 将 SQL 查询重写建模为策略优化问题，扩展 GRPO 引入涵盖**语义等价、文本距离、物理计划差异、运行时加速**的四维奖励函数。关键创新：
  - **Probability-Gated Adaptive Reward Shaping** — 基于查询难度的课程学习，只有当当前组掌握低级目标后才解锁高级奖励
  - **On-Policy Self-Improvement** — 回收当前策略产生的慢速重写作为针对性训练信号
- **性能**: 在 IID 和 OOD 工作负载上均优于规则基和强 LLM 基线，大幅减少有害的慢速重写，尾部延迟显著改善（具体数值见论文实验）
- **意义**: 解决了 LLM 重写 SQL 的经典问题——重写后语义正确但物理计划未改善甚至变差。将"执行反馈"引入训练循环，是 Text-to-SQL 工程化的关键进展

---

### 8️⃣ Data Flow Control: AI Agent 数据安全策略基础设施

- **来源**: [arXiv:2606.05679, 2026-06-04](https://arxiv.org/abs/2606.05679), Charlie Summers & Eugene Wu
- **摘要**: 提出 Data Flow Control（DFC）框架，将数据安全策略嵌入 DBMS 的元组级数据流中。核心组件 Passant 是一个可移植的查询重写层，通过**聚合谓词 over provenance monomials** 的形式化来保证策略执行。
- **性能数据**:
  - 跨 **5 个 DBMS 引擎**（DuckDB, Umbra, PostgreSQL, DataFusion, SQLServer）均实现 **~0% 开销**
  - 替代方案（物化 provenance）对比，Passant 快数个数量级
- **意义**: 第一个将 AI Agent 数据安全策略从"prompt 层和后置检查"迁移到数据基础设施层的系统性工作，对 Agent 时代的数据湖仓安全架构有根本性影响

---

### 9️⃣ Data Agents Under Attack: LLM 驱动分析系统的安全漏洞

- **来源**: [arXiv:2606.08661, 2026-06-07](https://arxiv.org/abs/2606.08661), Kuncan Wang 等
- **摘要**: 首次对 Data Agent 安全进行系统性研究。提出**三层漏洞框架**（解释层/执行层/策略层），识别 8 种 Data Agent 特有风险，构建含 3 个攻击目标、7 种战术、14 种技术的攻击分类体系。
- **实验**:
  - 评估 **6 个系统**：4 个开源 Data Agent + 2 个生产云分析服务
  - 搭配基于真实数据库 Schema 的 LLM 驱动的 Payload 生成管线
- **关键发现**:
  - Data Agent 将数据库安全和通用 LLM-Agent 安全问题重新组合成二者单独无法捕获的故障模式
  - 当前系统普遍存在严重的漏洞
- **意义**: 随着 Data Agent 进入生产环境（Snowflake CoWork、Databricks Genie），安全已成湖仓+AI 融合的硬约束
