## 1. 🏗️ Apache Iceberg 1.11.0 架构深潜（May 20, 2026）

> **来源**: [Apache Iceberg 1.11.0 Release](https://github.com/apache/iceberg/releases/tag/apache-iceberg-1.11.0)
>
> ⚠️ 此前跟踪仅提及未深潜，本次补充 1.11.0 的架构级变更

### 1.1 加密：Manifest List Encryption（PR #7770）

Iceberg 1.11.0 引入了 **Manifest List 级加密**，这是继 1.10.0 的 Spec-level encryption 定义（加密密钥 + 元数据标记字段）后的首个具体实现：

```
加密架构:
  Spec v3 定义 → 加密密钥字段 + 加密 metadata keys
  1.10.0: 表元数据加密标记
  1.11.0: Manifest List 加密 (首个具体实现)
          ↕
  每个 manifest list 文件可独立加密
  加密元数据存储在 table metadata 的 encryption keys 字段
```

- **范围**: Manifest list 内容加密（文件路径、分区信息、统计信息等）
- **方法**: 与文件级加密相同的密钥体系，支持多密钥轮换
- **意义**: 为敏感数据湖场景提供端到端加密保护，元数据层面不可见

### 1.2 Variant 类型：从 Core 移动到 API 层

Variant 类型从 1.9.0 开始在 Core 层实现（半结构化 JSON-like 数据类型），1.11.0 正式将 Variant 接口和相关序列化实现 **移动到 API 层**（PR #12304）：

```
1.9.0: Variant in Core  → Parquet reader/writer + 统计收集
1.10.0: Variant 表达式支持 (extract expressions)
1.11.0: Variant → API 层 → 所有引擎可见
                               ↕
          Spark 列统计支持       Flink 2.1 变体读取
```

**架构演进信号**: Variant 从实验性功能 → Core 实现 → API 标准化，表明 Apache Iceberg 正将半结构化数据作为一级公民。这比单纯将 variant 存为 string 带来 **schema-aware 的查询优化**（min/max 统计跳过等）。

### 1.3 Flink 2.1.0 支持（PR #13714）

Iceberg 1.11.0 增加对 Flink 2.1.0 的正式支持，包括：
- Flink 2.1 sink 适配（DynamicIcebergSink）
- Flink 2.1 表维护操作（ExpireSnapshots、RewriteDataFiles）
- `_row_id` 和 `_last_updated_sequence_number` 读取器支持

结合 Flink 2.3.0（6月25日发布）的原生 S3 文件系统，**Iceberg + Flink 2.3** 形成完整的云原生湖仓实时写入链路。

### 1.4 Delete File Index 架构改进（PR #13161）

- **不再复制 delete file stats**: `DeleteFileIndex` 在构建时不再复制 delete files 的统计信息，改为引用原数据
- **删除文件的 Position Delete 数据去重优化**: 行级删除场景下的内存占用显著降低
- **与 Deletion Vector (V3) 的集成准备**: 为后续 DVs 原生支持奠定基础

### 1.5 CountNull 聚合函数（PR #13981）

新增 `CountNull` 聚合支持，允许在 Iceberg 层面直接计算 null 值数量，而不需要将所有数据加载到引擎层。这对数据质量监控和大规模数据 profiling 场景意义重大。

### 1.6 其他架构级变更

| 变更 | 影响 |
|:-----|:-----|
| **Partition Statistics for V3**（PR #12098） | V3 表格式的分区统计正式纳入 Spec |
| **REST Catalog 认证属性文档化**（PR #14065） | OAuth2 / token 刷新标准化 |
| **Position delete 文件带 row data 标记 Deprecated**（PR #14045） | 向 Deletion Vector 方向演进 |
| **Spark 4.0 过程重构: ProcedureInput 统一参数处理**（PR #13913） | Spark 4.0 集成成熟度提升 |

## 2. 🤖 AgenticDB：Agent 驱动数据库调优（arXiv 2606.20318, Jun 18→26）

> **来源**: [arXiv 2606.20318](https://arxiv.org/abs/2606.20318)
> **核心成果**: 118.1% 平均性能提升，超越最强 baseline

### 核心问题

现有自动调优工具的三大局限：
1. **仅 DBMS knob 搜索**: 局限于预定义的数据库参数空间（如 buffer pool size），无法触及 OS 级瓶颈（如 I/O scheduler、透明大页）
2. **高执行成本**: 每次 knob 调整都需要迭代测试，收敛慢
3. **无安全约束**: 缺乏 runtime 反馈保障，调优可能破坏生产稳定性

### 架构：Context-Grounded Harness

```
AgenticDB 调优循环:
  ┌─ 上下文感知 Harness ─────────────────────┐
  │  ↓ 提出变更 (DBMS + OS 级)                │
  │  ↓ 安全约束过滤 (避免不安全操作)            │
  │  ↓ 应用变更 → 观察 workload + runtime 状态 │
  │  ↓ 执行反馈 → 修正后续决策                 │
  │  ↓ 经验积累 (跨 task 迁移)                 │
  └──────────────────────────────────────────┘
  ⬆ 记忆增强规划 (memory-enhanced planning)
```

### 架构创新

1. **双域动作空间**: 同时操作 DBMS knob（MySQL/PostgreSQL 参数）和 OS 级配置（I/O scheduler、THP、NUMA balancing），诊断瓶颈面更广
2. **安全约束执行生命周期**: runtime 交互中包含 `propose → check safety → apply → observe → rollback on anomaly` 的完整生命周期
3. **记忆增强规划**: 跨调优任务积累经验，同一 workload 模式无需从头搜索
4. **运行时反馈驱动**: 不依赖离线 benchmark 数据，直接在真实 workload 上观察性能变化

### 结果

| 维度 | 指标 |
|:-----|:-----|
| 性能提升 | **118.1%** 平均改善（vs 最强 baseline） |
| 收敛速度 | time-to-best 减少 **22.6%** |
| 测试覆盖 | MySQL + PostgreSQL, YCSB/Sysbench/TPC-H |
| OS 级贡献 | OS-level actions 在 PostgreSQL 上贡献显著 |
| 安全约束 | 有效阻止不安全操作（如生产环境 OOM） |

### 意义

AgenticDB 标志着数据库调优从 **knob 搜索** 正式进入 **Agent 编排** 时代。将 OS 级调优纳入动作空间 + runtime 反馈驱动的安全约束，使得自动调优首次在生产级可用性上闭环。这与整个数据领域的 Agent 化趋势（Databricks Genie Agents、Snowflake CoCo）一致。

## 3. 🗄️ BtrLog：低延迟云数据库日志服务（arXiv 2606.27051, VLDB'26）

> **来源**: [arXiv 2606.27051](https://arxiv.org/abs/2606.27051)
> **被接收**: VLDB 2026 (Boston)

### 核心问题

云数据库的 WAL 持久化三难选择：
| 方案 | 延迟 | 成本 | 复用性 |
|:-----|:----|:----|:------|
| EBS（远程块存储） | ❌ 高延迟 + 高成本 | ❌ 高 | ✅ 易接入 |
| 对象存储（S3） | ❌ OLTP 不可用 | ✅ 低 | ✅ 通用 |
| 自建日志后端 | ✅ 低延迟 | ✅ 可控 | ❌ 专有 + 紧耦合 |

### 架构

```
BtrLog 架构:

App ─→ BtrLog Quorum (SSD-backed log nodes) ─→ 异步归档 → Object Store (S3)
        ↕ quorum 写入 (单网络 RTT)
        降低 straggler 敏感度
```

**关键设计决策**:

1. **单 writer 聚焦**: 面向云数据库最常见的 single-writer 架构（非 multi-master），简化一致性协议
2. **Quorum 复制**: 日志记录跨 SSD 日志节点仲裁复制，单网络 RTT 完成持久化
3. **异步对象存储归档**: 日志节点将记录聚合为大 segment 异步写入对象存储，**不在延迟关键路径上**
4. **降低 straggler 敏感度**: 仲裁复制天然屏蔽慢节点（对比 EBS 的单点写入）

### 结果

- **延迟**: 低于 EBS
- **吞吐**: 集成到 DBMS 后端到端事务吞吐更高
- **成本**: 日志节点 SSD + 异步对象存储归档，存储成本可控

### 意义

BtrLog 为云数据库提供了一个可复用的 WAL 持久化层，解决了当前 cloud-native 数据库必须自建日志系统的痛点。其架构——SSD quorum 低延迟写入 + 异步对象存储归档——可能成为云数据库日志层的参考架构。

## 5. 🔮 趋势研判

### 本周最大信号：Iceberg 加密 + Variant 标准化，开放格式正向企业级全栈推进

- 1.11.0 的 Manifest List 加密是首个具体加密实现，结合 Spec v3 的加密定义，Iceberg 正在完成从 **开源开放格式 → 企业级安全格式** 的跨越
- Variant 进入 API 层表明半结构化数据支持正式成为 Iceberg 一级能力，与 Delta Lake 的 Variant column 统计形成对标竞争

### 数据库调优进入 Agent 时代

AgenticDB 的 118.1% 性能提升不是增量改进，而是范式切换——从 knob 搜索到 Agent 编排。结合上周的 Databricks Genie Agents 和 Snowflake CoCo，**Agent 正在渗透数据平台的所有层面**：查询（Text-to-SQL）、调优（AgenticDB）、运维（AIOps）。

### 云数据库日志架构标准化信号

BtrLog 被 VLDB 2026 接收，为 cloud-native 数据库的 WAL 层提供了一个通用范本。如果这一模式被开源（如 FoundationDB 作为底层），可能催生类似 etcd 之于 K8s 的**云数据库日志标准层**。
