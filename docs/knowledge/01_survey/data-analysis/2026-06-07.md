## 🥇 TOKI：LLM-Agent 持久化记忆的双时序算子代数（arXiv:2606.06240, Jun 5）

> **核心突破**：首次将 Agent 持久化记忆的写入一致性契约形式化为**双时序算子代数**，把 Agent 推理中的矛盾检测从 prompt 工程下沉到数据库基础设施层

- **来源**: arXiv:2606.06240, Ziming Wang (Jun 5, 2026)
- **架构设计**：
  - **双时序维度**：每个事实携带**有效时间**（真实世界当其时）和**事务时间**（Agent 何时写入），双轴交互相邻状态
  - **矛盾分类体系**：形式化定义 3 类矛盾（事实冲突 / 时序矛盾 / 逻辑不一致）→ 自动检测与解决
  - **算子代数量要操作**：状态归并（merge）/ 状态回退（retract）/ 状态分解（split），均保时序一致性
- **实现**：代码 + 数据 + 可复现性工单已开源
- **意义**：Agent 长期运行时必然面对「同一个事实的多个版本」。TOKI 将记忆一致性从「靠 prompt 约束」推进到「数据库代数保证」——这是一个从应用层到基础设施层的范式跨越

## 🥈 QDAG：LinkedIn 声明式分析方法论编译系统（arXiv:2606.05662, Jun 4）

> **架构创新**：将分析方法论声明式表达为 DAG（有向无环图），运行时编译为可复用 Dataset + 对比分析 + 洞察管道

- **来源**: arXiv:2606.05662, Peter Ho 等 (LinkedIn, Jun 4, 2026)
- **架构设计**：
  - **声明式 DAG**：方法论表达为节点（数据操作）+ 边（依赖关系），与具体数据集**解耦**
  - **运行时编译**：给定新数据集 → 自动绑定 DAG 节点 → 生成完整分析流水线（含统计测试/可视化）
  - **可复用有机生长**：相同方法论 DAG 跨不同阶段/团队/领域复用，DAG 本身随业务进化
- **意义**：LinkedIn 生产环境验证，标志着**分析标准化的新路径**——从「手动复制分析模板」升级为「声明式编译方法论」

## 🥉 SpecDB + Sophrosyne：LLM 时代的数据库设计双突破（arXiv, May 28-29）

### SpecDB：LLM 生成自定义数据库（arXiv:2605.31097, May 28）

- **来源**: arXiv:2605.31097, Yunkai Lou 等 (May 28, 2026)
- **核心**：用户用自然语言描述业务场景 → LLM 生成完整数据库 schema（表/视图/索引/约束）+ 填充示例数据
- **Feature-Oriented Decomposition**：将用户需求拆解为业务特征 → 每个特征生成独立 schema 片段 → 自动组合为完整设计
- **意义**：DDIA 降维打击——AI 时代的数据库设计正从「人写 DDL」走向「人描述意图，LLM 生成 DDL」

### Sophrosyne：Agent 探索关系型数据库需要"节制"（arXiv:2605.30862, May 28）

- **来源**: arXiv:2605.30862, Madhav Jivrajani 等 (May 28, 2026)
- **核心发现**：LLM Agent 自由探索 SQL 数据库时会产生大量**低效/破坏性查询**（全表扫描、跨 join 暴力枚举、无 limit 的聚合）
- **Moderation 层设计**：Agent 和数据库之间的审计/节流中间件，检查查询的 cost estimate + 访问模式 + 结果集大小
- **意义**：AI Agent 查询不是越多越好——数据库基础设施需要"管教"Agent，目前业界的缺失环节

## 📦 产品更新（Snowflake Jun 2 深度补录）

### 🔹 Snowflake Interoperable Lakehouse：Agency Over Data

- **发布时间**: Jun 2, 2026
- **核心理念**：**Agency Over Data**（数据自主权）——用户数据不受限于单一引擎，通过 Iceberg REST Catalog 实现跨引擎互操作
- **架构设计**：
  - **Iceberg REST Catalog** 作为统一元数据层，任何兼容引擎可读取相同数据
  - **Horizon Context**（同步发布）：治理上下文层，标签/分类/PII 掩码/行级策略——**一次定义，跨引擎处处生效**
  - **与 Cross-Engine ABAC（Databricks）平行演进**：Snowflake 选择在 Horizon 层做元数据治理，Databricks 选择在 Unity Catalog 层做策略执行，但底层都是 Iceberg REST Catalog API

### 🔹 Snowflake Adaptive Compute：工作负载自适应

- **发布时间**: Jun 2, 2026
- **核心**：突破传统固定资源配给模式，根据查询执行特征**动态调整计算资源分配**
- **意义**：AI Agent 和分析查询的负载模式高度不可预测，Adaptive Compute 直接应对「突发+间歇+混合」的 Agentic 查询模式
