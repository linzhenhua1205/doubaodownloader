## 一、Apache Iceberg 突破：Puffin-backed 向量索引原生嵌入湖仓 🚀

**来源**: arXiv:2606.04196 — *"Puffin-Backed Vector Indexes: Attaching Approximate Nearest Neighbor Indexes to Apache Iceberg Snapshots for Compute-Disaggregated Query Engines"* (Artur Borycki, Jun 2, 2026) ⚠️【深度跟踪】

### 核心创新
将 **分布式 ANN（近似最近邻）索引直接嵌入 Apache Iceberg 表格式**，使用 **Puffin 侧边文件** 作为存储容器、**snapshot summary** 作为绑定机制，实现 **向量搜索无独立索引层**：

| 组件 | 作用 |
|:-----|:------|
| **Puffin 格式** | 原为 Iceberg 的 portable 列级统计信息和 deletion vectors 设计，本文证明可承载 **十亿规模 Vamana 图索引** |
| **Snapshot summary** | 通过 `statistics-file.snapshot-summary` property 将 Puffin blob 链接到 snapshot，使 ANN 索引管理退化为标准 Iceberg snapshot 操作 |
| **Sharded 图索引** | 分片 Vamana 图的二进制布局，支持分布式 build/probe/增量刷新 |
| **分层 probe 策略** | centroid 小索引放在 coordinator，DiskANN 大图放在 executor SSD |

### 架构意义
- **计算-存储分离** 引擎（无状态 executor、弹性扩展、从对象存储读取）无需专用索引存储层
- **继承 Iceberg 原子性**：原子提交、time travel、多引擎可读、孤儿文件 GC — 零实现成本
- **构建在 FlockDB 上**（基于 DuckDB 的分布式 MPP 引擎）
- **这意味着：Iceberg 表可以原生支持向量相似性搜索**，消除 ANN 索引与湖仓之间的格式鸿沟

## 二、Text-to-SQL 三篇突破性论文（Jun 2-3 arXiv 集中投稿）

### 2.1 CAPER：子句对齐的过程监督 ⚠️【深度跟踪】

**来源**: arXiv:2606.03327 — *"CAPER: Clause-Aligned Process Supervision for Text-to-SQL"* (Ban et al., Jun 2)

- **问题**：SQL 查询级正确性信号（执行准确率）无法定位中间哪个 SQL 决策出错；token 级密集监督不适用
- **方案**：通过 **反事实干预 SQL 抽象语法树 (AST)** 自动推导子句级监督信号
- **成果**：训练 **CAPER-9B**，轻量 Clause-PRM（Process Reward Model）
  - **在 BIRD/Spider 上比 GPT-5.4 相对提升 15.3% EX**
  - **故障定位准确率 84.53%，MRR 90.60%**
- **架构**：子句级别（SELECT/WHERE/GROUP BY/HAVING/ORDER BY 等）的中间反馈，用于 policy optimization + candidate verification

### 2.2 Cost-Aware Agentic Query Execution

**来源**: arXiv:2606.03152 — *"Cost-Aware Optimization for Agentic Query Execution"* (Nie et al., Jun 2)

- **问题**：LLM-backed 算子进入查询后，传统查询优化假设（代数等价仅成本不同）不再成立
- **方案**：形式化 **agentic query execution** 新范式，提出 **EnumGRPO** 自改进优化器
  - 在 **SWAN 4 个数据库**上达到 **35.4% 执行准确率 @ $0.011/查询**
  - 相对 hybrid query baseline **成本降低 ~317x**，答案准确率**相对提升 18%**
- **决策维度**：执行范式、算子类型、算子放置位置、selectivity scope、projection width
- **方法**：枚举计划 → 通过 **in-context reinforcement learning (GRPO)** 蒸馏质量-成本反馈为可复用启发式

### 2.3 Text-to-SQL Friendly Logical Database Design

**来源**: arXiv:2606.03145 — *"The Case for Text-to-SQL Friendly Logical Database Design"* (Zhang et al., Jun 2)

- **新范式**：LLM-based Text-to-SQL 改变了 schema 的消费者，逻辑数据库设计应新增 **LLM-friendly** 目标
- **三种语义保持转换**（可组合、不改变 SQL 语义）：

| 算子 | 含义 | 效果 |
|:-----|:------|:------|
| **+A (abstraction)** | 逻辑视图物化重复 join 路径 | 降低 LLM 理解负担 |
| **+P (partitioning)** | 工作负载感知的逻辑分区，剪枝无关上下文 | 减少 prompt 噪声 |
| **+R (renaming)** | 描述性标识符改善列链接和谓词构造 | zero-shot 直接适用 |
| **+A+P+R 组合** | 三种算子叠加 | **BIRD/Spider 准确率 +4.2%** |

- 优势：**数据库侧的优化**，与 Text-to-SQL 模型无关，各 pipeline/backbone 均有效

## 三、Databricks 产品+研究动态

### 3.1 Query Tags — 通用 SQL 工作负载标记（Jun 2, Public Preview）

**来源**: Databricks Blog — *"Query Tags: The Context Your Warehouse Queries Have Been Missing"*

- 为每个 SQL 执行附加 **自定义 K-V 业务标签**
- 自动集成 **dbt（model name）、Power BI（report id）、Tableau（workbook name）**
- 可通过 **System Tables 用标准 SQL 查询**，或直接向 Genie 提问
- **数百客户、数百万查询/周** 已投入使用
- 架构意义：**将观测性延伸至业务语义层**，从"谁/什么工具/什么仓库"到"哪个项目/哪个成本中心/哪个团队"

**来源**: Databricks Blog — *"Introducing Cross-Engine ABAC"*

- 基于 **Iceberg REST Catalog scan API** 开放规范
- 已支持：Apache Spark（Iceberg-Spark / Delta-Spark connector）
- 即将支持：Starburst、DuckDB
- 向 Iceberg 社区贡献 **标签跨目录交换协议** 提案

### 3.3 Genie Code 个性化（Jun 1）

### 3.4 Databricks AI Research: 3x Faster Search（Jun 4）
