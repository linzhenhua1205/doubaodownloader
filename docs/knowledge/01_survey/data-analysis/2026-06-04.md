## 一、Databricks 最新动态（Jun 2-3, 2026）

### 1️⃣ Apache Spark Real-Time Mode GA — Gaming Sessionization 实战验证（Jun 3）⚠️【深度跟踪】

**发布**: Databricks Blog — *"Apache Spark Real-Time Mode for Gaming: A Better Way to Do Real-Time Sessionization"*

**核心内容**:
- **Spark Real-Time Mode 正式 GA**：此前 06-03 记录仅提及 Spark 4.0，今新增 Real-Time Mode 实战详情
- **transformWithState 新算子**：一个 `StatefulProcessor` 类管理完整会话生命周期
  - `handleInputRows()` → 响应式处理 Kafka 事件（Session Start/End）
  - `handleExpiredTimer()` → **主动式定时输出**（Heartbeat/Timeout），**独立于输入数据流**
- 结合 Real-Time Mode 实现 **432ms p99 端到端延迟**，比 micro-batch 快 20x

**性能数据**:
| 指标 | 数值 |
|:-----|:-----|
| 输入事件（启动+结束） | ~500K/分钟 |
| 活跃会话数 | ~4M 并发 |
| Heartbeat 输出 | ~8M 条/分钟 |
| 输入输出放大比 | ~16x |
| p99 延迟 | **432ms**（Real-Time Mode）vs micro-batch |

**架构意义**:
- 替代传统方案对比：Apache Flink → 需要独立集群/状态后端/部署栈；Akka 自建 Actor → 不扩展
- Real-Time Mode 仅需 **单 trigger 切换**，无需重写代码 → 「一个引擎统一流批」
- 应用场景外推：IoT 心跳、实时告警、设备监控等定时驱动型工作负载

### 2️⃣ Cross-Engine ABAC Beta（Jun 2）⚠️【深度跟踪】

**发布**: Databricks Blog — *"Introducing Cross-Engine ABAC"*

**核心内容**:
- Unity Catalog 成为**首个实现跨引擎 ABAC 的 Catalog**
- 基于 **Iceberg REST Catalog scan API** 开放规范实现
- 策略执行发生在 **Catalog 层（数据到达引擎之前）**，引擎无需实现任何策略逻辑

**架构原理**:
```
用户查询 → 引擎发送 scan 请求到 Unity Catalog
          → UC 评估用户权限 + 策略
          → UC 返回过滤后的 scan plan（仅权限内数据）
          → 引擎执行查询
```

**当前支持**:
- 策略类型: 基于标签的行过滤器（Row Filter）、列掩码（Column Mask）
- 引擎支持: Apache Spark（Iceberg-Spark / Delta-Spark 连接器）
- 即将支持: Starburst、DuckDB

**路线图**:
- 短期: Catalog 层集中式策略执行（已 Beta）
- 长期: 向 Apache Iceberg 社区提案**标签跨目录交换协议**，实现引擎原生策略执行

### 3️⃣ Query Tags Public Preview（Jun 2）

**发布**: Databricks Blog — *"Query Tags: The Context Your Warehouse Queries Have Been Missing"*

**核心内容**:
- 给每条 SQL 执行附加自定义 KV 业务标签（如 `project:finance_planning`）
- 自动标签集成: dbt（模型名）、PowerBI（报告ID）、Tableau（工作簿名）
- 标签数据写入 `Query History System Table`，支持 SQL 查询 + Genie 自然语言分析

**典型场景**:
- 按团队/项目分摊共享 warehouse 成本
- 追踪 dbt 模型回归 → 找出哪个模型变慢
- 区分 dev vs prod 流量

### 4️⃣ Liquid Clustering vs Partitioning 8 大迷思深度揭秘（Jun 1）

**发布**: Databricks Blog — *"Debunking 8 data layout myths: why Liquid Clustering outperforms partitioning"*

**关键数据（补充 06-03 记录）**:

| 客户场景 | 数据规模 | 改进幅度 |
|:---------|:---------|:---------|
| Arctic Wolf 安全遥测 | 3.8 PB | **7.7x 查询加速**，文件数从 4M→2M |
| Bolt CDC 表 | TB 级 | 写入吞吐 +138%，读取 -63% |
| Databricks 内部 PB 表 | 1.1 PB→0.8 PB | **5.9x 加速**，字节读取减少 86% |
| Co-Clustered Join（Private Preview） | — | Join 快 51%，Shuffle 减少 87% |

**关键架构特性**:
- **行级并发**（非分区表文件级）：两个 writer 更新不同行不冲突
- **元数据操作**：metadata-only DELETE 比全量重写快 ~90%，聚合查询快 27x
- **OPTIMIZE 改进**：10PB 表 planning 从 12h→23min，execution 5x 加速
- **Petabyte scale 生产验证**: 几十个客户 PB 级生产表已上线
