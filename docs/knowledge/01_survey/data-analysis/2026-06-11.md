## 🏢 Databricks

### 1️⃣ Custom Model Serving：双轴弹性伸缩引擎 (Jun 10)

**来源**: [AI Serving Platform That Adapts to Your Model](https://www.databricks.com/blog/ai-serving-platform-adapts-your-model)

Databricks 正式发布下一代 **Custom Model Serving** 平台，核心创新是 **AutoPilot Pod Autoscaler (APA)** — 双轴自动伸缩引擎：

- **技术架构**: 每个 serving endpoint 是完全隔离的 K8s Deployment，自带独立 Pod + 模型版本专用容器镜像
- **双轴弹性**: 
  - **水平伸缩 (Horizontal)**：基于活跃并发请求数，每 1s 采样、每 5s 决策，10→10K QPS 可在 **<60s** 内完成
  - **垂直伸缩 (Model-Aware Vertical)**：每隔 30s 学习模型资源画像 (CPU/GPU 利用率、I/O 等待、内存带宽、FP16/BF16 FLOPS)，自动调整每 Pod 的并发上限
  - **不对称策略**: 向上激进 (防延迟飙升)、向下保守 (考虑最近 5 分钟避免冷启动)
- **性能数据**:
  - **300K+ QPS** 峰值，p99 延迟开销 **<10ms**
  - **90%+** 成本节省 (客户从自建迁移后的典型值)
  - **99.99%** 生产可用性
  - p99/p50 延迟相比 DIY 方案改善 **2×**
- **冷启动优化**: 预热节点池 + 并行拉取模型 + Provisioned Concurrency 保底
- **分析意义**: 这是 Lakehouse 平台将 AI 推理能力"平台化"的关键一步 — 从 Spark SQL 分析 → Model Serving 在同一平台、同一治理体系下完成，不再需要独立 ML 平台

### 2️⃣ Databricks Genie 新一代：跨空间推理 + 企业知识连接 (Apr 26, 持续迭代至 Jun)

**来源**: [The next generation of Databricks Genie](https://www.databricks.com/blog/next-generation-databricks-genie)

新一代 Genie 从单 Genie Space 回答问题进化为跨空间的 Agent 推理引擎：

- **跨空间推理**: 单一聊天界面可同时引用多个认证的 Genie Spaces、治理仪表盘和 Databricks Apps，元数据驱动路由保证高信任源优先级
- **企业知识连接**: 内置 Google Drive、SharePoint 连接器 + 通用 MCP 连接，通过 Unity Catalog AI Gateway 统一治理
- **规模数据**: 2026 年仅一年已创建 **1.5M Genie Spaces**
- **移动端**: 原生 iOS 和 Android App，全量 Genie 体验（仪表盘 + App + Chat）
- **大规模租户**: Workspace 可扩展到 **100K+** 用户
- **分析意义**: Databricks 正在以 Genie 为入口整合所有业务用户交互，将数据查询、BI 仪表盘、AI Agent 统一为一个入口

### 3️⃣ Plenitude 案例：PDF → Delta Lake → Genie Agent (Jun 8)

**来源**: [Transforming solar and wind maintenance reports with Genie](https://www.databricks.com/blog/transforming-solar-and-wind-maintenance-reports-genie-and-ai-agents)

Plenitude 基于 Databricks Genie 构建的风电/太阳能运维分析系统：

- **技术架构**: PDF 事件驱动摄取 → `ai_parse_document` 提取 (文本/表格/图片/元数据) → 序列化 JSON → Delta Lake
- **Agent 交互**: Genie Agent Mode 进行 Deep Research，Unity Catalog 语义元数据 + Genie Space 本地知识库双重指导
- **数据安全**: Row-Level Security + Unity Catalog Function，按国家控制数据可见性
- **分析意义**: 展示了 Lakehouse + AI Agent 处理完全非结构化数据的完整路径：解析→结构化→语义化→Agent 推理

## ❄️ Snowflake

### 4️⃣ CoWork：每个知识工作者的个人工作 Agent (Jun 2)

**来源**: [Snowflake CoWork: The Personal Work Agent for Every Knowledge Worker](https://www.snowflake.com/blog/snowflake-cowork-personal-work-agent/)

Snowflake CoWork (原 Snowflake Intelligence) 重新定位为个人工作 Agent 平台：

- **Cortex Sense** (即将 Private Preview): 自动学习企业数据语义层，从查询历史、元数据、Power BI/Tableau 仪表盘等信号中理解业务概念（收入定义、财年日历、快照表等）
  - **性能数据**: 使用 Cortex Sense 后，CoCo + CoWork 准确率从 **47% → 83%**；无 Snowflake 上下文的 Frontier Coding Agent 仅 **23%**
- **Deep Research** (即将 GA): 基于 AI Research Team 的 Agent Swarm 编排，跨结构化和非结构化数据深度调研，比单 Agent 系统提升超 **33%**
- **功能矩阵**:
  - **Multi-Agent Orchestration** (即将 Public Preview): 自动路由问题到最佳数据、技能和工具
  - **Memory** (Public Preview): 跨会话持久化用户偏好
  - **User Skills** (Public Preview): 用户用自然语言描述多步工作流 → 可复用 Skill
  - **Automations & Time-Based Subscriptions** (Public Preview): 后台定时运行 (如"每周一检查消费变化 >20% 的客户")
  - **Artifacts** (即将 GA): 可保存、分享且动态刷新（含 RBAC）的分析成果，包含交互式仪表盘
  - **Conversation Sharing** (即将 GA): 可分享完整分析线程（含引用）
  - **Async Agent API** (即将 GA): 支持数分钟到数小时的异步后台任务
  - **iOS App** (即将 GA): Face ID + 完整会话历史
  - **CoWork Slack App** (Private Preview): 行内渲染图表 + 多用户 OAuth 权限继承
- **MCP 连接器**: Gmail、Jira、Slack、Salesforce 等 → 在 Snowflake 治理边界内执行操作
- **Agent Studio** (即将 GA): 统一的 Agent 构建、管理与监控界面
- **分析意义**: CoWork 代表了 AI 数据分析从"被动回答问题 → 主动工作代理"的范式跃迁，Cortex Sense 83% 的准确率数据也说明: **企业上下文学习是 Agent 准确率的最大差距来源**

## 📄 arXiv 新论文

### 5️⃣ TAHOE: 提示优化即数据管理 (Jun 10)

**来源**: arXiv:2606.12387, cs.DB

TAHOE 将 Text-to-SQL 的提示优化重新定义为**动态数据管理问题**：

- **Hint Bank**: 错误驱动的编译反馈 → Syntax Hints (方言规则) + 执行/用户反馈 → Semantic Hints (Schema 与用户特定逻辑)
- **Strategy Layer**: 建模冲突用户意图为竞争策略，附带近期性信号和后学习归因统计
- **性能数据**: Spider 2.0-Snow 上 pass rate 从 **61.95% → 79.42%**；pass@4 从 **72.57% → 87.61%**；Snowflake 语法 pass rate **100%**；平均编译反馈轮次从 2.79 → **0.12**
- 同组 Hint Bank 迁移到弱模型（Doubao-2.0-lite）仍产生 **19.7 pp** pass rate 提升

### 6️⃣ CAPER: 子句级过程监督 (Jun 2)

**来源**: arXiv:2606.03327, cs.DB

CAPER 通过 SQL AST 的反事实干预自动推导子句级监督信号：

- **技术**: 轻量级 Clause-PRM (CAPER-9B) 提供子句边界反馈，用于策略优化和候选验证
- **性能数据**: BIRD + Spider 上 **15.3%** 相对 EX 提升 over GPT-5.4；故障定位准确率 **84.53%**，MRR **90.60%**

### 7️⃣ ProSPy: Profiling 驱动的 SQL-Python Agentic 框架 (Jun 4)

**来源**: arXiv:2606.05836, cs.CL

面向企业级 Text-to-SQL 的四阶段 Agentic 框架：

- **技术架构**: 自动 Profiling 提取细粒度数据证据 → 渐进裁剪大 Schema → 方言无关 SQL 接口提取中间视图 → Python 灵活下游分析
- **性能数据**: Spider 2.0-Lite **60.15%**；Spider 2.0-Snow **60.51%** (Claude-4.5-Opus, 无多数投票)
- 对 SQL 方言变化鲁棒，在 Schema Recall 和 Precision 间实现良好权衡

### 8️⃣ Larch: 语义谓词 AI SQL 查询优化 (Jun 5)

**来源**: arXiv:2606.07923, cs.DB

- **技术**: Larch-A2C (GGNN 编码 AI_FILTER 表达式树 + MDP 规划评估顺序) + Larch-Sel (监督学习预测选择性 + DP 找近优顺序)
- **性能数据**: 比 Palimpzest 和 Quest **3×-19×** 更低的 Token 总成本，跨不同真实/合成数据集

### 9️⃣ ACE-SQL: Schema 检索 + SQL 生成联合强化学习 (Jun 4)

**来源**: arXiv:2606.05906, cs.CL

- **技术**: 在执行反馈下联合优化 Schema 检索和 SQL 生成，双向自适应：Retriever 向 Generator 能正确执行的列集学习，Generator 向 Retriever 不断演进的 Schema 选择学习
- **性能数据**: BIRD Dev **65.3%** 贪婪执行准确率，仅用 **0.93K** 输出 Token/查询，使用约 3K 合成数据对训练

## 📊 综合趋势判断

1. **Text-to-SQL 进入"RL + 过程监督"阶段**: CAPER/ACE-SQL/Progress-SQL 三篇同时聚焦 RL + 细粒度监督，Signal 从执行结果扩展到子句级/选择级
2. **企业上下文成为最大瓶颈**: EntSQL 最佳 15.9% vs TAHOE 最佳 79.4%，差距在**企业私有知识 + Schema 复杂度** — 这正是 Snowflake Cortex Sense 和 Databricks Genie Space/Unity Catalog 商业化竞争的核心战场
3. **Agent 查询引擎新品类**: Hyperparam 70KB JS-native 引擎的出现，代表"Agent 端直接查询 Lakehouse"的新架构模式，可能改变数据工程与 AI 应用的交互边界
4. **Databricks vs Snowflake 双平台路线分化**:
   - **Databricks** 以 Genie 为入口整合业务用户，强调跨空间推理 + MCP 连接 + 移动端
   - **Snowflake** 以 CoWork 为个人工作 Agent，强调 Cortex Sense 自动学语义 + Agent Swarm Deep Research + 后台自动化
   - 共同方向: **Data + AI + Agent 统一平台**，数据平台即是 Agent 平台
5. **Data + AI Summit 2026** (Jun 15-18, SF) 在即，预计将有重大发布
