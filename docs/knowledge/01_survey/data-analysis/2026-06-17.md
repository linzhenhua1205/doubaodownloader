#### ① Lakehouse//RT: 统一湖仓实时数仓

**来源**: Databricks Blog, 2026-06-16 (Nong Li, Shoumik Palkar 等)
**核心发布**: Databricks 推出全新实时数仓 Lakehouse//RT，由 Reyden 引擎驱动，直接在湖仓上实现毫秒级查询，无需数据搬运和独立 serving 层。

**技术详情**:
- Reyden 引擎为实时工作负载全新设计，直接读取 Delta/Iceberg 开放格式
- **12,000 QPS** @ TPCH 基准测试，sub-100ms 延迟
- AUTO sizing：自动确定基线计算资源，无需 t-shirt size 猜测
- Incremental autoscaling：按节点增删而非整机复制，精细控制成本

**客户实测性能数据**:
| 客户 | 行业 | 加速效果 |
|:-----|:-----|:---------|
| Meta Enterprise | 社交 | 10s of ms 返回，高并发下稳定 |
| SES | 太空通信 | **20×** 比之前快 |
| Bally's | 博彩/游戏 | **7×** 更快，60TB Delta Lake |
| Enverus | 能源数据 | **100×** 于专有实时引擎（部分查询），2.7PB |
| Equilibrium Energy | 能源交易 | **3.6×** vs SQL Serverless 中位延迟 |
| Cisco | 安全 | **5×** 响应时间改进 |
| Halcyon | 安全 | **4×** 更快，无需缓存系统 |
| PointClickCare | 医疗健康 | **10×** 部分查询，**1/3** 以上平均提升 |
| Magnite | 广告 | 持续 **sub-200ms** 数百 QPS |

**定价**: Beta 阶段，2027 年 1 月前 **30% 折扣**

**意义**: 消除了长期困扰行业的"实时 serving 层 vs 统一湖仓"矛盾选择。之前用户需要将数据复制到专用实时引擎（如 Druid、ClickHouse），产生三倍成本（重复存储、治理碎片化、工程运维），且复杂查询（多表 join、window functions）仍无法处理。Lakehouse//RT 首次在单一引擎上实现了实时 serving + 复杂分析的统一。

#### ② Genie Code: 全页命令中心 + ML 工作流 + 定时任务

**来源**: Databricks Blog, 2026-06-17 (Julia Powell, Gal Oshri 等)
**核心发布**: Genie Code 三大升级 — 全新全页命令中心、原生 ML 工程集成、定时任务自主执行。

**技术详情**:
- **全页命令中心**: 替代侧边面板，支持管理 **15+ 并行线程**，可重命名/搜索/追踪状态
- **ML 工作流集成**: 原生集成 MLflow（读取实验/制品/模型血缘/质量指标）、Model Serving（诊断健康与性能）、Compute Awareness（自动切换到 AI Runtime 使用 GPU）
- **Genie Ontology**: 学习团队的建模模式（特征工程方向、模型评估标准），使 Genie Code 遵循团队特有规范而非通用默认值
- **Scheduled Tasks (即将推出)**: 定时任务自主执行，如检查隔夜作业结果、汇总流水线运行、解释仪表盘指标变化

**关键数据**:
- Genie 产品线 **10× 增长**，**90%** Databricks 客户已在使用
- **Danfoss 案例**: 从原始数据到生产就绪的受治理 ML 工作流，**90 分钟**完成
- **Genie ZeroOps**: 扩展至生产运维，监控生产系统、调查问题、准备修复方案供审核

**意义**: 从"被动回答问题"的辅助工具进化到"主动执行工作"的自治操作平台，Scheduled Tasks + ZeroOps 意味着数据团队可以配置 Agent 在无人值守时完成运维工作。

#### ③ Genie App Builder + App Spaces + Serverless Micro Apps

**来源**: Databricks Blog, 2026-06-16 (Evan Pandya 等)
**核心发布**: 三大 App 开发新能力 — Genie App Builder（AI 辅助 app 创作）、App Spaces（治理边界）、Serverless Micro Apps（微 VM 运行时）。

**技术详情**:
- **Genie App Builder**: 自然语言描述 → 生成计划 → 实时预览迭代；底层基于 AppKit（TypeScript SDK），内置缓存/遥测/重试逻辑，原生感知 Unity Catalog 语义层
- **App Spaces**: 管理员在空间级别定义资源/数据访问/API scopes/安全策略，所有 app 自动继承。治理从"逐个审批"转向"预置环境"
- **Serverless Micro Apps**: 基于微 VM 运行时，每个 app 独立隔离，快速冷启动，scale-to-zero

**关键数据**:
- Databricks Apps 较去年 **5× 增长**，**5,000+ 账户**每周运行生产 App
- Top 100 客户平均部署 **250+ 生产 App**
- 活跃 App 数量 **近翻倍**，周交互用户 **3× 增长**

**意义**: 使"vibe coding"在企业中可落地 — 业务分析人员可以用自然语言构建 app，同时 governance 在 App Space 层级预置而非逐 app 审批。

#### ④ OpenSharing: 首个 AI 时代开放分享协议

**来源**: Databricks Blog, 2026-06-16/17 (Stephen Orban)
**核心发布**: 行业首个开源、厂商无关的 AI 时代分享协议 OpenSharing + 开放湖仓存储生态。

**技术详情**:
- 支持结构化数据（Delta/Iceberg）+ **非结构化数据**、volumes、**Agent Skills**、**AI 模型**、semantics 等新型数据资产
- **SecureConnect**: 跨云存储简化网络配置
- **Global Distribution**: 主动创建/缓存跨区域复制，降低出站费用
- **Open Lakehouse Storage Ecosystem**: Cohesity, Commvault, HPE, **MinIO**, **NetApp**, **Nutanix**, **Qumulo**, **Rubrik**, **VAST Data** 等已或即将支持 OpenSharing
- **Genie Agents 分享**: 可通过 OpenSharing 分享带自然语言界面和业务上下文的 Agent，支持"**按问题付费**"（pay-per-question）商业模型

**关键数据**: 20,000+ 客户，$5B+ ARR（年同比增 65%+），70%+ Fortune 500

**意义**: Delta Sharing 的进化版 — 不再局限于结构化数据，将分享协议扩展到 AI 时代的全部资产类型。Genie Agents 的分享开启了全新的数据变现模式。

## 📄 arXiv 论文精选

#### ⑨ Hyperparam: Agent 查询引擎（arXiv:2605.27785）

**发表于**: 2026-05-26 | **分类**: cs.AI, cs.DB | **作者**: Kenny Daniel

**论文链接**: https://arxiv.org/abs/2605.27785

**核心问题**: AI 原生客户端应用（Claude Code、Cursor、Claude Desktop、浏览器内 Agent）的数据增长速度最快，但 Spark、Trino、托管仓库等传统引擎无法嵌入 JS 运行时。

**解决方案 — Hyperparam**: 3 个开源 JS 库，总计 **<70KB**:
- **Hyparquet**: 从对象存储直接读取 Parquet 文件
- **Squirreling**: 运行 LLM 形状的异步 UDF
- **Icebird**: 直接读取 Apache Iceberg

**性能数据**:
- **300×** 快于 DuckDB-WASM（过滤有界查询）
- **192×** 快于 DuckDB-WASM（排序有界查询）
- 10 个 Agent 分析师任务套件，完成成本降低 **2/3**

**架构创新**: 逐像元、异步原生的 SQL 执行——昂贵的像元仅当下游操作符请求时才触发执行（lazy evaluation）。

**意义**: 代表"**数据工程需要为 AI 原生客户端应用而更新**"的论点——未来数据查询将大量来自 Agent 内部的 JS 运行时，而非传统的 SQL 终端或仪表盘。

#### ⑩ Data Lakehouse 架构对比: Delta Lake vs Iceberg vs Hudi

**发表于**: 2026-04-23 | **分类**: cs.DC, cs.DB | **作者**: Ivan Borodii, Halyna Osukhivska

**论文编号**: arXiv:2604.21449

**核心内容**: 首次使用 Apache Spark 对三大 Data Lakehouse 系统进行系统化对比，处理结构化（CSV）和非结构化（JSON）数据，文件大小达 7GB，经过 4 个 ETL 流程。

**关键发现**:
- **Delta Lake**: 对于任何数据量的加载速度最优 → 适合加载性能优先的场景
- **Apache Iceberg**: 存储效率和稳定性最优 → 适合节省磁盘空间和稳定性的场景
- **Apache Hudi**: 在数据加载和存储评估中表现不佳，但可能在增量更新和流处理场景中有效

**意义**: 为数据工程师提供了首个基于实证的湖仓选型决策依据，明确告知识别出自己的**首要约束（加载速度 vs 存储效率）**后再做选型。

**论文链接**: https://arxiv.org/abs/2604.21449

## 🏛️ Apache 基金会更新

#### ⑪ Apache Livy 晋升为顶级项目

**来源**: Apache Software Foundation Blog, 2026-06-04
**核心发布**: Apache Livy — Spark 的 REST 服务接口 — 正式从孵化器晋升为顶级项目（Top-Level Project, TLP）。

**技术详情**: Livy 提供 REST API 来管理 Spark 会话、提交作业、查询状态，无需直接管理 Spark 集群。支持同步/异步作业执行、多用户会话隔离、Kerberos 认证等企业特性。

**意义**: Livy 作为 Spark 生态的关键组件正式毕业，意味着它在 API 稳定性、社区治理、代码质量方面达到了 ASF 最高标准。对于希望通过 REST API 而非原生 Spark 接口来运行的湖仓架构，Livy 是标准选择。

## 🎯 核心趋势判断

1. **"实时湖仓"成为现实** — Lakehouse//RT 的发布标志着湖仓架构补齐了最后一块拼图（实时 serving），用户不再需要独立实时引擎
2. **Agentic Data Engineering 成型** — Genie Code 的定时任务 + ZeroOps + Lakeflow 形成了"构建→部署→运维"的 Agent 闭环
3. **AI 时代数据分享协议正在定义** — OpenSharing 将分享范围从结构化数据扩展到 Agent Skills、AI 模型、语义层
4. **Postgres 成为数据平台标准接口** — Snowflake 和 Databricks 都拥抱了 Postgres 兼容层，降低开发者迁移门槛
5. **Data Agent 正从对话式向自主执行演进** — 从"你问它答"到"定时自动执行"再到"自主运维"的路线清晰
6. **JS-native 数据引擎兴起** — Hyperparam 仅 70KB 就实现了 Iceberg 读取 + LLM UDF，预示 Agent 时代的数据引擎会越来越轻量、嵌入化
7. **Lakehouse 选型从直觉走向实证** — arXiv 的 Delta vs Iceberg vs Hudi 实证对比为数据工程师提供了首个量化选型依据
