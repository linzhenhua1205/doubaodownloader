## 2. ⚡ Databricks Lakehouse//RT：Reyden 引擎驱动的实时湖仓

> **来源**: [Databricks Blog](https://www.databricks.com/blog/introducing-lakehousert-real-time-performance-unified-lakehouse) (Jun 16, 2026)

### 2.1 核心产品发布

Databricks 正式推出 **Lakehouse//RT**（读作 "Lakehouse Double Slash RT"），这是建立在全新 **Reyden 引擎** 上的实时数据仓库，承诺直接在湖仓数据上实现毫秒级查询响应，无需数据复制或单独 serving 层。

### 2.2 架构级变革

```
传统架构:
  Delta/Iceberg 湖仓 → ETL 管道 → 专用实时 serving 层 (专有存储)
                                                   ↕
                             三倍成本: 数据复制 + 治理分裂 + 工程维护

Lakehouse//RT 架构 (Reyden 引擎):
  Delta/Iceberg 湖仓 → Reyden 引擎 → 毫秒级响应
                        (直接在开放格式上运行)
                     ↕
  统一治理: Unity Catalog 单点管控
  无数据移动成本
```

### 2.3 性能数据

| 维度 | 指标 |
|:-----|:-----|
| **小数据集响应** | 低至 **10ms** |
| **大数据集响应** | **sub-100ms** |
| **高并发** | **12,000 QPS** 时 sub-100ms |
| **vs 实时 serving 层** | 最高 **16× 性能提升** |
| **复杂查询 (TPC-DS)** | 替代品最高 **25× 更慢**（大数据集下替代品直接失败） |
| **大数据集** | TPCH 1TB 级低延迟，替代品在相同规模下 **2/3 失败** |

### 2.4 新计算模型

Lakehouse//RT 引入了全新的计算计费模型：

1. **AUTO sizing**: 不再需要用户选 "t-shirt size"，系统自动确定基准计算资源
2. **Incremental autoscaling**: 传统仓库 2×/3×/4× 整车扩展 → 增量式单节点添加/移除，按需付费

### 2.5 客户验证

| 客户 | 场景 | 效果 |
|:-----|:-----|:-----|
| **Meta Enterprise** | 内部分析 (供应链/财务) | 10ms 级响应 |
| **SES (卫星通信)** | 数十亿行实时遥测 | 20× 性能提升，省去单独 serving 层成本 |
| **Enverus (能源)** | 2.7PB 数据，350M+ 记录 | 最快 10ms，最高 100× 提升 |
| **PointClickCare (医疗)** | 医疗数据集 | 平均 1.3× 提升，部分 10× |
| **Bally's (博彩)** | 60TB Delta Lake | 7× 加速，sub-second 响应 |

### 2.6 架构信号

Lakehouse//RT 是 **2026 年数据领域最重要的平台级发布之一**。其核心信号：
- **湖仓 vs 实时不再二选一**: 开放格式（Delta/Iceberg）可直接支撑毫秒级查询，数据湖的实时能力不再是"中间层"的专利
- **治理统一是刚需**: Unity Catalog 单点管控 vs 传统多系统碎片化治理，在合规驱动的企业场景中尤为关键
- **Reyden 引擎是黑盒**: Databricks 未披露 Reyden 的技术细节（列存？向量化？MVCC？），但显然是 PhoTon 的进一步演进，可能引入了新的存储层或缓存架构

## 3. 🔬 arXiv cs.DB（Jun 30）— 新提交 9 篇 · 重点 3 篇

### 3.1 DBCC：数据库上下文压缩 — Text-to-SQL 新瓶颈认知

> **论文**: [Database Context Compression for Text-to-SQL on Real-World Large Databases](https://arxiv.org/abs/2606.28601)
> **作者**: 北大团队 (Jingwen Liu 等)
> **核心主张**: **Text-to-SQL 的瓶颈不再是推理能力，而是数据库表示**

#### 核心问题

现有 Text-to-SQL 系统（包括 Spider 2.0、BIRD）的真实数据库 schema 巨大：
- **重复审计列**: 每张表都有 `created_at`/`updated_at`/`is_deleted`
- **同构表群组**: 大量结构相似的表（如按月份分区的日志表）
- **不透明标识符**: 含义仅存在于文档中的列名
- **庞大的数据字典**: 大量与查询无关的元信息

#### SGCF 原理与架构

```
SGCF (Support-Gain Component Factorization) 统一压缩框架:

输入: 原始 Schema + 语义描述 + 文档
  ↓
  ├── 重复列提取 (Repeated Column Extraction)
  │   ← 识别并合并多表共有的审计/元数据列
  ├── 同构表模板化 (Isomorphic Table Templating)
  │   ← 将结构相似的表合并为带参数模板
  ├── 语义组件化 (Semantic Componentization)
  │   ← 将不透明标识符↔文档关联后语义化
  └── 证据净化 (Evidence Purification)
      ← 过滤与查询无关的数据字典内容
  ↓
输出: 紧凑 schema 表示 (2.6M → 34.7K tokens)
```

#### 关键结果

| 指标 | 原始 | DBCC 后 | 提升 |
|:-----|:----|:--------|:----|
| 输入 Token（最大子集） | **2,600,000** | **34,700** | **~75× 压缩** |
| Schema-linking 严格召回 (DeepSeek-V3.2) | **0%** | **56.5%** | ∞ |
| Schema-linking 严格召回 (Claude Opus 4.7) | — | **63.1%** | — |
| 端到端执行准确率 | — | **+1.8~1.9%** (恒增) | 模型无关 |

#### 架构意义

DBCC 提供了一个**模型无关的数据库侧中间件**，直接压缩 schema 表示。这意味着：
- **任何 LLM 都能受益**（已测试 DeepSeek-V3.2、Claude Opus 4.7 等）
- **0%→56.5% 的召回跳跃**表明当前 schema-linking 的最大问题是冗余而非算法
- **query-agnostic 预压缩 + online evidence purification** 的组合是实用化关键

### 3.2 开源 LLM Text-to-SQL 能力边界：BIRD 基准的诚实基准

> **论文**: [How Far Do On-Prem Open LLMs Get on Text-to-SQL?](https://arxiv.org/abs/2606.29733) (Jun 29)
> **核心问题**: 不能使用云端 API的企业，开源模型在 Text-to-SQL 上的真实表现如何？

#### 方法

- 基准: BIRD dev split (n=1,534, Execution Accuracy)
- 模型: Qwen2.5-Coder (7B/14B/32B), CodeLlama (7B/13B/34B), Llama-3.x (8B/70B)
- 方法消融: schema linking + self-correction + self-consistency → McNemar 检验

#### 四大发现

| # | 发现 | 数据 |
|:-|:-----|:-----|
| ① | **模型代际 >> 参数规模** | Qwen2.5-Coder 7B: **39.1** vs CodeLlama 7B: **20.9** |
| ② | **Self-correction 是可靠的免费午餐** | 所有族系中显著有效（有提升空间时） |
| ③ | **Schema linking 没帮助**（统计不显著） | 96.5% 召回率也救不了 |
| ④ | **Self-consistency 性价比极差** | +0.13 pp 提升，消耗 ~5× tokens |

#### 最佳实践建议

```
推荐管线 (基于证据):
  Qwen2.5-Coder 32B (最好) 或 Llama-3.3-70B (有竞争力)
    → Schema linking: 跳过
    → Self-correction: 必开 (显著提升)
    → Self-consistency: 跳过 (性价比低)
```

#### 架构意义

这篇论文的 value 不在数字，而在**方法论**——它是第一个在真实企业场景（on-prem、开源模型、成本约束）下做系统性 ablation 的研究。三大结论颠覆了常见认知：
1. **Schema linking 不工作的原因不是 "linker 不够强"**，而是模型本身在 BIRD 上的瓶颈已不是 table 选择——96.5% 召回率仍然统计不显著说明问题在其他环节
2. **Self-correction 第一次被证实跨模型族系普适有效**，且是免费（不显著增加推理成本）
3. **Self-consistency 的性价比被钉死**：+0.13 pp / 5× tokens，工业级不可接受

### 3.3 CADENZA：编译自然语言意图为任务 DAG（SIGMOD 2027）

> **论文**: [CADENZA: Compiling Natural-Language Intent into Task-Specific Operator DAGs](https://arxiv.org/abs/2606.29151)
> **被接收**: SIGMOD 2027
> **方向**: 语义查询处理（Semantic Query Processing）

#### 核心问题

语义查询处理引擎（SQPE）将关系查询扩展到语义算子（模型推理），但优化是多目标的：质量 vs 延迟 vs 成本。已有优化器未将"语义算子中间任务输出"作为关系优化对象暴露，无法做过滤/路由/阈值/联合调优。

#### CADENZA 架构

```
自然语言意图
  ↓
TxRA (Task-extended Relational Algebra)
  ├── 逻辑规划器
  │   ├── 种子 TxRA 计划合成
  │   ├── 结构重写 (安全条件从算子依赖检查)
  │   └── 语义引导替代生成
  └── 物理规划器
      ├── 异构后端路由器 (每个算子)
      └── 贝叶斯联合调优 (路由 + 后端参数 + 关系阈值)
  ↓
可执行任务 DAG
```

#### 结果 (SemBench)

| 维度 | 提升幅度 |
|:-----|:--------|
| 质量 | **+0.49** |
| 延迟 | **165.7×** |
| 成本 | **310.3×** |

#### 架构意义

CADENZA 是语义查询处理从"启发式"走向"编译优化"的范式转折。它将自然语言意图编译为**可优化、可重写、可枚举**的任务 DAG，相当于 SQL 查询优化器对语义算子的等价物。SIGMOD 2027 接收意味着学术界认可了这一方向。

## 5. 🎯 趋势研判

### 最大信号：实时湖仓正式成为产品级现实

Databricks Lakehouse//RT 的发布是 **2026 年数据架构最重要的里程碑**。核心判断：
- **开放格式（Delta/Iceberg）可以直接做实时在线服务**，不再需要 Kappa 架构中的单独 serving 层
- **传统三件套（湖 + 仓 + 实时引擎）→ 统一平台**，治理成本线性降低
- **Reyden 引擎的具体实现是黑盒**，但毫无疑问是列式 + 向量化 + 某种形式的实时索引的融合
- **Snowflake 需要回应**：Snowpipe Streaming + CoCo 是 Snowflake 的实时方案，但仍在 Snowflake 自有存储上，不如 Lakehouse//RT "直接在开放格式上"激进

### Text-to-SQL 的瓶颈正式从推理切换到表示

今天两篇论文共同指向一个方向：
- **DBCC** 证明 schema 表示问题比推理更重要（2.6M→34.7K 的压缩带来 recall 0%→56.5%）
- **On-Prem 基准** 证明 schema linking 对当前模型"统计不显著"——问题不在 table 选择，而在查询理解
- 这意味着 **DBCC 类中间件 + Qwen2.5-Coder 可能构成当前最优的 on-prem T2S 方案**
- **Self-correction 是唯一可靠的免费增益**，self-consistency 被钉死在耻辱柱上

### Spark 4.2.0 临近 GA

Spark 4.2.0-preview5（May 1）已发布，按照 Spark 的发布节奏（通常 preview 阶段 6-12 个月），**4.2.0 GA 可能在 2026 Q3/Q4**。重点关注其 Spark Connect 深化和 K8s 集成改进。
