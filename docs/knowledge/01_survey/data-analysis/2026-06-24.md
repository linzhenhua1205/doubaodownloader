## ⚡ 引擎加速

### ① Presto+GPU: Presto/Velox 原生 GPU 加速查询引擎（arXiv, 2606.24647, 6/23）

> IBM/NVIDIA/Intel 联合团队，开放到开源 Presto/Velox

- **架构改造**: 扩展 Presto 为 GPU-aware 查询引擎，解决两大核心挑战：
  - 存储到 GPU 算子间的高效数据传输路径
  - 跨分布式节点算子间数据交换**不离开 GPU 内存**
- **技术路线**: 基于 NVIDIA cuDF（C++ 列式 data-frame 库）构建 GPU 算子，多 GPU 集群 TPC-H 基准测试
- **性能**: CPU Presto 对比，**6× cost/performance 改善**（性价比提升），已用于客户生产工作负载
- **开源**: 代码已合入开源 Presto/Velox 主干

**意义**: 首个主流分布式 SQL 引擎将 GPU 加速作为一等公民加入架构，标志着**湖仓分析引擎 GPU 化的正式开端**，非 NVIDIA 专有方案

## 🧠 Agent 数据系统

### ② 我们准备好 Agent 原生记忆系统了吗？（arXiv, 2606.24775, 6/23）

> 清华/阿里研究团队 — 从数据管理视角评测 12 个 Agent 记忆系统

- **系统分析框架**: 将 Agent 记忆分解为四大核心模块：
  - 记忆表示与存储（Memory Representation & Storage）
  - 提取（Extraction）
  - 检索与路由（Retrieval & Routing）
  - 维护（Maintenance）
- **评测规模**: 12 个代表性记忆系统 + 2 个基线，5 个基准工作负载跨越 11 个数据集
- **核心发现**: **没有单一架构能通吃所有场景**；记忆结构的有效性完全取决于其设计是否匹配工作负载瓶颈
- **细粒度消融**: 量化各模块对表示保真度、检索精度、更新正确性、长期稳定性的独立贡献
- **成本-性能权衡**: 定位维护比全局重组更高效
- **跨领域**: cs.CL / cs.DB / cs.IR 三领域交叉

**意义**: 首次将 Agent 记忆从"黑盒端到端性能"推进到**系统级数据管理分析**，是 Data Agent 基础设施层的关键参考

### ③ DIA — 数据智能代理（arXiv, 2606.19319, 6/17）

> 三个 Agent 的系统（数据解释器+模式创建器+查询生成器）部署在生产环境

- **架构**: 以**自主编码代理（ACA）**作为一等抽象 — 不输出文本，而是生成、执行、验证和修复具体程序工件
- **共享记忆**: 三个 Agent 共享一个经验复用记忆池，领域专家可审查每个输出
- **性能**: 在 **7 个 SQL 基准测试**上横跨 4 种任务类型和 4 种 SQL 方言，**全部匹配或超越当前最佳结果**
- **部署**: 已为企??客户提供生产服务

**意义**: **全栈 Data Agent（解释→建模→查询）的生产级验证**，证明"基于执行+共享记忆"架构可泛化

### ④ ProSPy — Profile 驱动的 SQL-Python 代理框架（arXiv, 2606.05836, 6/4）

> 四阶段: 数据 Profile → 模式剪枝 → SQL 中间视图 → Python 灵活分析

- **设计**: SQL 在大数据库上的高效 + Python 的灵活性 — 通过自动 Profile 提取细粒度数据证据，渐进式将大模式修剪为任务相关上下文
- **方言无关**: 通过方言无关的 SQL 接口获取中间视图，大幅降低对不可靠元数据的依赖
- **性能**: Spider 2.0-Lite 和 Spider 2.0-Snow 上 **60.15% / 60.51%** EX（Claude-4.5-Opus），无 majority voting
- **鲁棒性**: 对 SQL 方言变化鲁棒，模式召回率和精度之间实现有利权衡

### ⑤ RUBICON — 表中心 vs 文本中心的企业数据代理（arXiv, 2604.21413, 4/23）

> Stonebraker（Ingres/PostgreSQL 创始人）联合团队 — 证明表中心架构完胜 LLM 文本管道

- **核心论点**: 当前 Agent 将整个查询工作流委托给单一前缘 LLM，是**企业数据的无效抽象**
- **替代方案**: 对每个数据源使用受约束的查询接口（非自由文本），表中心集成层由查询处理器驱动
- **性能对比**:
  - RUBICON-Bench（跨异构企业源查询）: RUBICON **100%** 端到端准确率，所有 Agentic 基线（单/多 Agent ReAct）**0 条正确答案**
  - SemBench: 准确率比 LOTUS/Palimpzest **高 14.7%**，延迟降低 **62.64%**，token 成本降低 **98.64%**
- **结论**: **企业数据场景应走"受约束查询+表集成"路线，而非"LLM 自由文本管道"**

### ⑥ TAHOE — 经验驱动的 Text-to-SQL 提示优化（arXiv, 2606.12387, 6/10）

> 核心洞察: Text-to-SQL 的提示优化本质是**数据管理问题**

- **Hint Bank 机制**: 编译器反馈→语法 Hint，执行反馈→语义 Hint
- **策略层**: 建模冲突的用户意图为共享自然语言触发器下的竞争策略
- **性能**: Spider 2.0-Snow 上 pass rate 从 **61.95%→79.42%**（+17.5pp），Snowflake 语法通过率 **100%**
- **Hint 迁移性**: 相同 Hint Bank 迁移到较弱模型可提升 19.7pp（Doubao-2.0-lite）

## 📈 趋势判断

1. **查询引擎 GPU 化启动**: Presto+cuDF 6×性价比**不是终点，是起点**。首个主流开源 SQL 引擎集成 GPU 加速意味着更多引擎（Spark、Trino）将跟进
2. **Agent 记忆系统成为新赛道**: 本次论文首次将 Agent 记忆从黑盒评测推进到系统性数据管理评估。结合 DIA 的共享记忆机制，**可观察、可维护、可审计的 Agent 记忆系统**将成为 Data Agent 的基础设施
3. **表中心 vs 文本中心路线之争**: RUBICON 证明表中心架构在企业数据场景下完胜 LLM 自由文本管道。TAHOE 的 Hint Bank 策略（将经验结构化存储）与 RUBICON 的表中心集成分属同一设计哲学：**结构化＞非结构化**
4. **行业 AI 从平台到解决方案**: Databricks Genesis Workbench 和 Snowflake BioNeMo 标志着数据平台厂商从"提供平台能力"转向直接交付可部署的行业 AI 解决方案
