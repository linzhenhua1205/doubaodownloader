## 1️⃣ Apache Spark Real-Time Mode 正式 GA — 子秒级流处理

**来源**: Databricks Blog, 2026-06-03, Neha Prabhu & Murali Talluri
**链接**: https://www.databricks.com/blog/apache-spark-real-time-mode-gaming-better-way-do-real-time-sessionization

### 技术详情
Spark Structured Streaming 的 **Real-Time Mode** 正式 GA，配合新一代 `transformWithState` 状态化算子，实现了无需第二引擎的端到端子秒级流处理。解决了此前微批模式（Micro-Batch Mode, MBM）在需要**定时器驱动输出**场景下的不足。

**关键特性**:
- `transformWithState` 算子：面向对象状态管理、复合数据类型、**定时器驱动逻辑**（`handleInputRows()` + `handleExpiredTimer()` 双方法）、自动TTL、Schema演化
- Real-Time Mode 实现输入处理和定时器触发的**子秒级精度**

**性能数据**（游戏会话跟踪场景验证）:
- **吞吐量**: 每分钟 ~500K 输入事件，~400万活跃会话，8M 心跳记录输出，输入→输出放大比 **16×**
- **延迟**: **432ms p99** — 相比微批模式 **20× 加速**
- 引擎统一：替代需要额外部署 Flink 集群或自建状态服务（Akka Actor）的旧架构

### 影响判断
标志着 **Spark 在流处理领域正式进入 Flink 的传统领地**。对于已有 Spark 生态的团队，Real-Time Mode + transformWithState 可实现"单引擎流批一体"，减少架构碎片化。不仅仅是游戏场景，还适用于 IoT 心跳、会话跟踪、实时告警、设备监控等需要定时器驱动输出的场景。

## 2️⃣ Databricks Genie 新一代发布：跨空间 Agentic 分析

**来源**: Databricks Blog, 2026-04-26, Ken Wong, Dillon Morrison & Richard Tomlinson
**链接**: https://www.databricks.com/blog/next-generation-databricks-genie

### 技术详情
Genie 从单一"Genie Space"升级为**跨空间统一智能分析助手**，成为 Databricks 面向业务用户的核心入口。

**核心增强**:
1. **跨空间Agentic推理**：单个聊天界面可跨多个Genie Spaces、治理化仪表板和Databricks Apps聚合答案，无需预先建模。由最新推理模型和新Agent架构驱动，可合成整个生态系统中跨越多个数据域的非显式关系
2. **外部知识源连接**：内置 Google Drive、SharePoint 等连接器 + MCP 协议支持，通过 Unity Catalog AI Gateway 统一管理
3. **全局统一入口**：消除 workspace 边界，支持 >100,000 用户的单 workspace，统一的 Entra/Okta 身份管理、自定义域名、Domains 按业务概念分组
4. **原生移动端**：iOS + Android 原生 App，含仪表板、应用和聊天功能

**采用数据**: 2026 年仅前5个月就创建了 **超过 150 万个 Genie Spaces**

### 影响判断
Genie 从"Text-to-SQL on a single dataset"进化到"**跨数据域、跨结构化+非结构化、跨设备**的Agentic数据分析"。结合 Query Tags（见下文），Genie 可以自然语言直接回答"哪个dbt模型最慢？哪个成本中心开销最大？"等运维问题，**正在模糊数据分析与运维监控的边界**。

## 3️⃣ Databricks Query Tags Public Preview — 查询级成本归因

**来源**: Databricks Blog, 2026-06-02, JooHo Yeo & Jiabin Hu
**链接**: https://www.databricks.com/blog/query-tags-context-your-warehouse-queries-have-been-missing

### 技术详情
Query Tags 让用户通过自定义 KV 标签 attach 业务上下文到每条 SQL 执行，存储在 Query History System Table 中，支持标准 SQL 查询和 Genie 自然语言分析。

**核心能力**:
- **Partner Tools 自动标记**：dbt（模型名+版本+物化类型）、Power BI（报表ID）、Tableau（工作簿名）自动注入标签
- **自定义应用标记**：SQL Statement Execution API、Python Connector 支持 statement-level/connection-level 标签（`customerid`, `applicationname` 等）
- **Ad-hoc 标记**：`SET QUERY_TAGS` 语句在 SQL Editor/Notebooks/Dashboards/Alerts 中生效

**采用数据**: 已有 **数百家客户每周标记数百万条查询**

### 影响判断
Query Tags 填补了湖仓体系中查询语义归因的关键空白——以前知道"谁跑的"但不知道"为什么跑的"、"为哪个项目跑的"。对 **FinOps、成本分摊、性能归因** 场景是重要的基础设施能力。

## 4️⃣ Databricks Cross-Engine ABAC Beta — 跨引擎统一治理

**来源**: Databricks Blog, 2026-06-02, Alex Jiang, Alex Reid & Michelle Leon
**链接**: https://www.databricks.com/blog/introducing-cross-engine-abac

### 技术详情
Unity Catalog 通过 **Iceberg REST Catalog scan API** 实现跨引擎的属性级访问控制（ABAC），让用户在外部引擎（如 Apache Spark）上也能强制 Unity Catalog 中定义的标签级行过滤和列掩码策略。

**架构要点**:
- 策略执行在 **Catalog 层完成**，数据到达引擎之前已被过滤——引擎不需要理解任何策略逻辑
- 基于 Iceberg REST Catalog scan API 开放规范，任何兼容引擎均可接入
- 当前支持：Apache Spark（Iceberg-Spark / Delta-Spark connector）；即将支持：Starburst、DuckDB 等
- Apache Iceberg 社区正在推进 **Exchange Labels** 提案，实现跨 Catalog 的治理元数据共享

### 影响判断
解决了湖仓领域长期存在的治理断层问题——开放表格式（Iceberg/Delta）实现了数据互操作，但**策略无法跟随数据流动**。Cross-Engine ABAC 让 Unity Catalog 成为业界首个也是唯一一个提供跨引擎 ABAC 的 Catalog。

## 5️⃣ Snowflake CoWork — 面向每个知识工作者的个人工作 Agent

**来源**: Snowflake Blog, 2026-06-02, Arun Agarwal + Effie Goenawan
**链接**: https://www.snowflake.com/blog/snowflake-cowork-personal-work-agent/

### 技术详情
Snowflake CoWork（前称 Snowflake Intelligence）是面向**每个知识工作者的个人工作 Agent**，融合了 Cortex Sense（自动理解企业语义）、Deep Research（Agent 集群编排）、多 Agent 编排、持久化记忆、用户技能（User Skills）和 Artifacts 等功能。

**关键性能数据**:
- **Cortex Sense**（自动上下文层）：自动学习企业定义的数据和业务运营方式，理解收入定义、财年日历、快照表等概念。在复杂企业查询的内部测试中：**CoCo/CoWork + Cortex Sense 准确率 83%**，vs 无 Sense 时的 47%，vs Frontier Coding Agents + Snowflake MCP 的 23% —— **提升几乎翻倍**
- **Deep Research**（深度研究）：由 Snowflake AI Research Team 构建的 Agent 集群编排系统，在 Hybrid Deep Research Benchmark 上 **超过单 Agent 系统 1/3**

**新增功能矩阵**:
| 功能 | 状态 | 说明 |
|:-----|:-----|:------|
| Cortex Sense | Private Preview 即将到来 | 自动上下文层 |
| Deep Research | 即将 GA | Agent 集群深度研究 |
| Analytical Search | Public Preview 即将到来 | 文档级聚合/比较/趋势分析 |
| Multi-agent orchestration | Public Preview 即将到来 | 自动路由查询到最佳工具 |
| Memory | Public Preview 即将到来 | 跨会话持久化用户偏好 |
| User Skills | Public Preview 即将到来 | 自然语言定义可复用技能 |
| Automations | Public Preview 即将到来 | 定时触发/条件触发的后台运行 |
| Async Agent API | 即将 GA | 分钟级/小时级异步任务 |
| Artifacts | 即将 GA | 可刷新的多格式输出 |
| Agent Studio | 即将 GA | Agent 构建/管理/治理统一平台 |

### 影响判断
Snowflake 正在将自身从"数据分析平台"重构为"**AI 工作操作系统**"。CoWork 本质上不是 BI 工具——它是结合了 Cortex Sense（企业语义理解）+ Deep Research（Agent 编排）+ MCP 连接器（跨系统动作）+ Memory（个性化）+ Skills（自动化）的全栈 Agent 平台。核心差异点在于 **83% 准确率**——这得益于 Cortex Sense 自动学习企业语义，这是单纯 Text-to-SQL 无法做到的。

## 6️⃣ Lakehouse 查询运行时方差系统研究 — arXiv:2606.03464

**来源**: arXiv, 2026-06-02, James Nurdin et al. (CLOUD 2026)
**链接**: https://arxiv.org/abs/2606.03464

### 技术详情
首次系统性研究分布式湖仓分析查询的 **run-to-run 运行时方差**，在 3 个公有云 + 1 个私有云上使用 K8s 部署、多数据库规模和三个分析基准进行实证。

**核心发现**:
1. 相同查询的**重复执行可产生近 2× 的运行时方差**
2. 关键方差来源：数据本地性、共租户负载、缓存效果
3. 解决关键方差来源后，**查询性能预测（QPP）误差可降低高达 80%**
4. 下游影响：以低碳调度为例，考虑运行时方差可**显著降低碳排放成本**

**影响**: 对于湖仓平台的成本优化（自适应资源扩缩、低碳调度）来说，理解运行时方差是不可或缺的前提。

## 7️⃣ Hyperparam: JS-native 湖仓查询引擎 — arXiv:2605.27785

**来源**: arXiv, 2026-05-26, Kenny Daniel
**链接**: https://arxiv.org/abs/2605.27785

### 技术详情
面向 AI Agent 应用的新一代 JS-native 湖仓引擎。三个开源 JS 库（Hyparquet, Squirreling, Icebird）仅 **< 70 KB**，直接从对象存储读取 Parquet 和 Apache Iceberg。

**性能数据**:
- Squirreling 运行 LLM 异步 UDF：在 filter-bounded 查询上比 **DuckDB-WASM 快 300×**，在 sort-bounded 查询上 **快 192×**
- 在 10 任务的 Agent 分析师套件上，成本比 DuckDB-WASM **低 2/3**

**核心洞察**: AI Agent 应用（Claude Code, Cursor, 浏览器 Agent）需要在 JS 运行时中操作数据，传统湖仓引擎（Spark/Trino）无法嵌入。Hyperparam 填补了这个缺口——它代表湖仓引擎向**AI-native 客户端应用**方向演进的新趋势。

## 8️⃣ Data-Centric 优化湖仓 Agent Skills — arXiv:2606.01185

**来源**: arXiv, 2026-05-31, Nicole Rose Schneider et al. (Bauplan)
**链接**: https://arxiv.org/abs/2606.01185

### 技术详情
研究分支湖仓（Bauplan）中如何优化 Agent 技能（skills）和环境文件。核心观察：**分支湖仓将 Agent 评估从输出匹配问题转化为状态验证问题**——Agent 生成的管线代码会引发具体的、可检查的湖仓状态变化。

**关键结果**:
- 提出的数据为中心的优化管线生成 task-verifier pairs，在隔离沙箱中执行候选技能，使用 trace-level 信号 + 程序化检查评分
- 在 25 个任务上的初步评估：**优化后的技能准确率提升 31.9%**
- 写入路径（write-path）数据工作流比只读任务更适合优化 Agent 技能

### 影响判断
Agent 正在成为湖仓的"新用户"，但它们的成功率不仅取决于模型质量，还取决于技能和配置文件的优化。这项工作是 Agentic Lakehouse 方向的重要基础——**湖仓需要为 Agent 设计新的接口（分支、沙箱、状态验证）**，而非简单暴露传统的只读 SQL 接口。

## 📋 综合观察

| 方向 | 核心趋势 | 关键玩家 |
|:-----|:---------|:---------|
| **流处理** | Spark Real-Time Mode GA → 单引擎流批一体，432ms p99 进入 Flink 领地 | Databricks/Spark |
| **AI-Native 分析** | Genie 跨空间 Agentic + CoWork 个人Agent → 数据平台变成 Agent 平台 | Databricks, Snowflake |
| **湖仓治理** | Cross-Engine ABAC + Iceberg Labels → 策略随数据流动 | Databricks/Unity Catalog |
| **运维可观测** | Query Tags → 查询级语义归因，FinOps 基础设施 | Databricks |
| **JS-native 湖仓** | Hyperparam 70KB → 湖仓引擎嵌入 AI Agent 运行时 | 开源社区 |
| **Agentic Lakehouse** | 分支+沙箱+状态验证 → Lakehouse 为 Agent 重新设计接口 | Bauplan 社区 |

### 关键对比：Databricks vs Snowflake 本周动态

| 维度 | Databricks | Snowflake |
|:-----|:-----------|:----------|
| **Agent 形态** | Genie = 业务用户的数据分析 Agent（结构化 + 非结构化 + 外部知识） | CoWork = 全栈个人工作 Agent（分析 + 自动化 + 跨系统动作） |
| **上下文层** | 依赖 Genie Spaces + Unity Catalog 语义 | Cortex Sense 自动学习企业语义（83% vs 47% 准确率） |
| **引擎** | Spark Real-Time Mode + Photon | Cortex AI + Snowpark Container Services |
| **流式能力** | Spark Real-Time Mode GA（子秒级） | 依赖外部流处理引擎 |
| **治理** | Cross-Engine ABAC Beta（Iceberg REST Catalog 架构） | Horizon Context + RBAC |
| **开放性** | Iceberg/Delta + Unity Catalog 开源 | Snowflake 专有格式 + Interoperable Lakehouse |
