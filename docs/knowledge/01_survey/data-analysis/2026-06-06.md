## 🥇 NL2Pipe：自然语言→语义算子管线的三阶段编译（arXiv:2606.04641, Jun 3）

> **核心突破**：将自然语言查询编译为语义算子管线（Palimpzest/LOTUS 后端），打通非专家用户使用语义分析系统的"最后一公里"

- **来源**: arXiv:2606.04641, Wenkai Dong 等 (Jun 3, 2026)
- **三阶段编译架构**：
  1. **Query-Data Linker** — 将问题实体 grounded 到实际数据，发现表/文本/图像间的隐式桥接实体
  2. **Semantic Planner** — 生成后端无关的算子计划（filter/join/map/aggregate）+ 自然语言谓词
  3. **Code Generator** — 利用自动生成的参考文档（算子签名+示例管线+后端约束），翻译为后端可执行代码
- **关键架构特性**：数据感知推理与后端代码生成解耦，同一 planning 逻辑支持多后端
- **效果**：跨源复杂工作负载上 F1 高达 **+60%**，同时保持可控成本和竞争性延迟
- **意义**: 标志着 Text-to-SQL 从纯 SQL 生成进入 **Text-to-Semantic-Operator-Pipeline** 时代，让 Palimpzest/LOTUS 这类系统对非专家真正可用

## 🥈 Data Flow Control / Passant：AI Agent 数据安全基础设施（arXiv:2606.05679, Jun 4）

> **核心突破**：将数据安全策略从 prompt 和后置检查移入 DBMS 基础设施层，形式化定义"数据安全"为 provenance monomial 上的聚合谓词

- **来源**: arXiv:2606.05679, Charlie Summers & Eugene Wu (Jun 4, 2026)
- **架构设计**：DFC 框架→ Passant 可移植查询重写层
  - **策略语言**：定义 tuple 级数据流上的监管/隐私/业务约束，**优化器无关**（optimizer-invariant）
  - **Passant 引擎**：provenance monomial 上的聚合谓词 → 查询重写，无需 materialize provenance
  - **跨引擎兼容**：DuckDB / Umbra / PostgreSQL / DataFusion / SQLServer 五引擎
- **性能**：**~0% 开销**，比替代方案快数个数量级
- **意义**：Query correctness ≠ Safety。Agent 生成的 SQL 可能语义正确但违反监管约束。这是首个将 data safety 纳入数据基础设施的系统性方案（开源）

## 🥉 GraftDB：动态折叠并发分析查询（arXiv:2606.04303, Jun 3）

> **架构创新**：多查询执行引擎，将后到达查询动态折叠进正在运行的执行中，复用已完成工作+共享后续工作

- **来源**: arXiv:2606.04303, Genki Kimura & Kazuo Goda (Jun 3, 2026)
- **State-Centric Execution**：算子状态不再是单查询私有，而是任何兼容查询可观测/贡献的共享状态
  - **Per-Query State Lens**：每个查询通过专属 lens 观察共享状态，仅能看到其有效行/状态片段
  - **Query Grafting**：识别已满足部分需求的算子状态 + 仍可共享的后续工作
- **效果**：TPC-H 变体工作负载 **吞吐量提升 2.17×**，过载开放到达下 **P95 响应时间降至 0.17×**
- **意义**：Agent 频率越来越高（AI agents issue many analytical queries），GraftDB 的 state-centric 设计直击多查询冗余执行痛点
