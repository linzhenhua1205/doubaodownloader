### 1.2 SQL 层变革

#### FROM_CHANGELOG / TO_CHANGELOG — SQL 层 changelog 流操作（FLIP-564）

这是 Flink SQL 首次提供 changelog 流的原生操作能力：

```sql
-- 将 CDC 流（含操作码列）转换为动态表
SELECT * FROM FROM_CHANGELOG(
  TABLE cdc_stream,
  'op_mapping' = 'c:INSERT, u:UPDATE, d:DELETE',
  'default_op' = 'INSERT'
);

-- 将动态表逆向为 append-only changelog 流（用于归档/审计）
SELECT * FROM TO_CHANGELOG(
  TABLE retract_table
);
```

**架构意义**: 此前只有 DataStream API 能操作 changelog 流。FLIP-564 在 SQL 层打通了这一能力，意味着：
- CDC 管道可以**纯 SQL 实现**，无需 Java UDF
- `TO_CHANGELOG` 允许**将 upsert/retract 流还原为 append-only**，解决了 write to append-only sink 的经典难题
- ⚠️ 2.3.0 覆盖基础场景，`PARTITION BY`、`invalid_op_handling`、`produces_full_deletes` 等高级功能在后续版本

#### 物化表（Materialized Table）DDL 对等 + 刷新控制（FLIP-550/557）

| 能力 | 说明 |
|:-----|:-----|
| **DDL 完全对等** | `CREATE MATERIALIZED TABLE` 支持显式列定义（watermark + PK）、`ALTER` 支持 ADD/MODIFY/DROP/RENAME |
| **START_MODE 刷新控制** | `CONTINUE`（从上游 offset 续跑，不重扫历史）或 `FULL`（全量重扫） |
| **架构变化不丢数据** | 添加 nullable 列等兼容变更不再需要全量 reprocess |

#### SinkUpsertMaterializer 重构（FLIP-558）— ⚠️ 升级注意

当 upsert key ≠ primary key 时，Flink 需要 SinkUpsertMaterializer。此前它因保持完整更新历史导致 state 无限增长。2.3.0 的改动：

```
之前: 自动启用 SinkUpsertMaterializer → 隐式 deduplicate → 高 state 消耗

之后 (2.3.0 默认行为):
  当 upsert key ≠ primary key → 规划期报错
  用户必须显式指定 ON CONFLICT 策略:
    → ON CONFLICT DO NOTHING   (静默跳过冲突)
    → ON CONFLICT DO ERROR     (失败作业)
    → ON CONFLICT DO DEDUPLICATE (显式 dedup, 原行为)
```

同时引入** watermark-based compaction** 减少 state 大小：
- `table.exec.sink.upserts.compaction-mode`: WATERMARK（默认）或 CHECKPOINT
- `table.exec.sink.upserts.compaction-interval`: 无 watermark 时的回退间隔

**⚠️ 升级影响**: 2.2 → 2.3 时，任何 upsert key ≠ primary key 的 SQL 作业都会在规划期报错，必须在 SQL 中**显式添加 `ON CONFLICT` 子句**。

## 2. 📦 Flink 原生 S3 文件系统深潜

> **来源**: [Introducing Flink's Native S3 FileSystem](https://flink.apache.org/2026/06/26/announcing-native-s3-fs/)

### 2.1 背景问题的历史

Flink 在 2.3 之前有**两个** S3 插件，都注册在 `s3://` scheme 上，只能选其一：

| 维度 | flink-s3-fs-hadoop | flink-s3-fs-presto |
|:-----|:------------------|:------------------|
| 底层 | Hadoop S3A client + SDK v1 | Hadoop + SDK v1（Presto 风格读取） |
| Exactly-once sink | ✅ (RecoverableWriter) | ❌ |
| Checkpoint 性能 | 慢（Hadoop 适配层开销） | 较快（专有读路径） |
| JAR 大小 | ~30 MB | ~93 MB |
| 运维代价 | hadoop-common 全依赖树带来的 CVE 负担 | 同 Hadoop + 已知 bug |

**困境**: 要么选 exactly-once（Hadoop 插件但慢），要么选 checkpoint 性能（Presto 插件但无 exactly-once），无法兼得。

### 2.2 原生 S3 FS 架构设计

```
Before (Flink 2.2):
  Flink Job → Hadoop FileSystem 适配层 → Hadoop S3A → SDK v1 → S3
                                    ↕
              两大缺陷: ① 间接层增加开销 ② SDK v1 已 EOL (Dec 2025)

After (Flink 2.3):
  Flink Job → Native S3 FileSystem → AWS SDK v2 (S3TransferManager) → S3
                          ↕
          直接基于 Netty NIO 异步 I/O，线程不阻塞
```

### 2.3 性能数据

| 指标 | Presto 插件 | 原生 S3 FS | 提升 |
|:-----|:-----------|:-----------|:----|
| **平均吞吐** | ~92 MB/s | ~200 MB/s | 2.17× |
| **平均 checkpoint 时长** | 90.1 s | 48.8 s | 1.85× |
| **P90 checkpoint 时长** | 155.0 s | 72.5 s | 2.14× |
| **P99 checkpoint 时长** | 165.3 s | 76.7 s | 2.15× |
| **同窗口完成 checkpoint 数** | 40 | 78 | 1.95× |
| **每个 checkpoint 平均存储** | 415 MB | 312 MB | 25% 更小 |

| 小状态 (0-2 GB) | Presto | 原生 | 4.58× |
| 大状态 (14-16 GB) | Presto | 原生 | 2.16× |

### 2.4 架构关键设计决策

| 决策 | 说明 |
|:-----|:-----|
| **AWS SDK v2 S3TransferManager** | 异步、非阻塞、连接池感知并发控制，替代 s5cmd 等外部工具 |
| **NativeS3RecoverableWriter** | 基于 S3 multipart upload，断点续传，同时支持 exactly-once 和 checkpoint |
| **无 Hadoop 依赖** | ~13 MB JAR vs 30-93 MB；CVE 面缩小；无配置 key 镜像问题 |
| **干净配置命名空间** | `s3.access-key` / `s3.secret-key` / `s3.region`，无 `fs.s3a.*` 兼容层 |
| **SSE-KMS 加密上下文** | 唯一支持自定义 key-value 元数据的插件，支持 IAM 细粒度策略 |
| **Entropy injection** | checkpoint 路径中可配置的随机子串，避免 S3 热键节流 |

### 2.5 迁移路径

```
步骤 1: 移除旧插件    rm -rf plugins/flink-s3-fs-hadoop/ 或 plugins/flink-s3-fs-presto/
步骤 2: 添加原生插件  mkdir -p plugins/flink-s3-fs-native && cp opt/flink-s3-fs-native-*.jar plugins/flink-s3-fs-native/
步骤 3: 更新配置     Hadoop 前缀 → s3.* 前缀（可选，旧 key 仍兼容）
步骤 4: 重启集群
```

**回滚保护**: 两种插件可在同一 Flink 集群共存，通过优先级选择。原生 FS 读写兼容 Hadoop/Presto 插件写入的数据，切换是双向安全的。

### 2.6 路线图

| Flink 版本 | 里程碑 |
|:----------|:-------|
| **2.3 (当前)** | 实验性 opt-in 插件。功能完整，已在大型科技公司生产验证 |
| **2.4 (规划)** | ① Per-bucket 配置 (`s3.bucket.<name>.*`) ② AWS CRT 客户端（HTTP/2） ③ S3 操作指标暴露 ④ Stream-based S3 R/W |
| **2.4+** | 社区投票成为推荐默认插件 |
| **后续** | Hadoop/Presto 插件正式废弃 |

### 2.7 架构意义

Flink 原生 S3 FS 的发布不仅是性能改进，更是**运营和安全层面的范式转变**：
- **SDK v1 EOL**（2025/12/31）是时间压实的合规风险，原生 FS 直接换到 SDK v2
- **无 Hadoop 依赖**意味着 Flink 用户不再需要为 `hadoop-common` 的 CVE 做 triage（Jackson/Guava/protobuf/Jetty/Kerberos/ZK 等非 S3 相关的传递依赖全部消失）
- **JAR 从 ~93MB 降到 ~13MB**，对 Kubernetes sidecar 容器镜像大小敏感的场景是直接收益

## 3. 🔓 Delta Lake 4.3.0（Jun 18, 2026）

> **来源**: [Delta Lake 4.3.0 Release](https://github.com/delta-io/delta/releases/tag/v4.3.0)

### 3.1 核心新特性

| 特性 | 组件 | 说明 | 架构信号 |
|:-----|:-----|:-----|:---------|
| **UC Delta REST API 集成** | Spark | Delta Spark 默认通过 UC API 路由 UC 管理表操作，服务端做 commit 验证 | **Delta 从文件系统协议走向目录服务协议** |
| **replaceOn / replaceUsing** | Spark | 选择性数据替换，按条件（`replaceOn`）或按列匹配（`replaceUsing`）替换部分行 | 不再只能全表 `OVERWRITE` |
| **UniForm 增量 Iceberg 转换** | UniForm | Iceberg 元数据**原子写入 Delta commit**，只转换增量 log range | **Delta ↔ Iceberg 双向互操作进一步打通** |
| **Sharing 流式+CDF** | Sharing | Spark Structured Streaming + Batch CDF 自动 Delta 响应解析 | Delta Sharing 从静态走向实时 |
| **Kernel 增量 version-checksum** | Kernel | `tableSizeBytes`/`numFiles` 增量更新，无需全量 log replay | 大表 table-health 诊断瞬间完成 |

### 3.2 UC Delta REST API 集成 — 架构级变革

这是 Delta 4.3.0 最重要的变化。本质上将 Delta table 的操作从：

```
Before (Delta 4.2):  Spark → ∆ log (文件系统原子提交)
                   ↕ 依赖文件系统层面 rename+list+monotonic clock

After (Delta 4.3):  Spark → UC Delta REST API → UC 服务端 → 提交 + 验证
                   ↕ 服务端负责 commit validation、table feature advertising、intent-based metadata updates
```

**受 UC API 管理的操作**: table loads、CREATE/CTAS、REPLACE、DML（MERGE/UPDATE/DELETE）、schema evolution、auto-merge、ALTER TABLE。

**不受影响的操作**: 非 UC 表（name-based / path-based）继续走旧的 `∆ log` 文件系统协议。

**未来规划**: 同样 UC Delta REST API 底座将在后续支持 Flink、Trino 等多引擎。

### 3.3 UniForm 增量 Iceberg 转换

```
Before: 整个表的 Iceberg 元数据 → 全量重建 → 写放大
After:  Delta commit → 原子写入 Iceberg 元数据 → 只转换增量 log range → 写近无放大
```

Delta 4.3.0 也恢复了 UniForm 在 Spark 4.1 上的兼容性。

### 3.4 Delta Lake 版本线（2026 年 7 月现状）

```
Delta 4.x 版本线:
  4.0.0 (Jun 2025)  ← 里程碑: Spark 4.0、Variant 数据类型、catalogManaged preview、Kernel 初版
  4.0.1 (Jan 2026)  ← 特性重命名 catalogOwned-preview → catalogManaged + UC OAuth
  4.1.0 (Feb 2026)  ← Spark 4.1 支持、V2 Spark Connector (Kernel-based)、Server-Side Planning preview
  4.2.0 (Apr 2026)  ← UC Managed Table 增强 (RTAS/DPO)、Kernel-based Flink 连接器(experimental)、Variant GA
  4.3.0 (Jun 2026)  ← ← **最新**: UC Delta REST API、replaceOn/replaceUsing、UniForm 增量 Iceberg
```

## 4. ❄️ Snowflake 博客（Jun 17–30）— 三大能力发布

> **来源**: [Snowflake Blog](https://www.snowflake.com/blog/)

### 4.1 Cortex Sense（Jun 30）— 未建模数据的 Grounded Context

**核心问题**: Snowflake 上大量数据是**未经建模的**（unmodeled data）——非结构化文本、JSON blob、半结构化日志、API 响应。传统分析需要先建 schema 再查询，延迟高、门槛高。

**Cortex Sense** 是 Cortex AI 家族的新成员，直接在未建模数据上提供 grounded context：
- 无需预先定义 schema 或 ETL
- 利用 LLM 自动理解数据语义
- 将 unstructured/semi-structured 数据转化为可查询、可分析的 context

**架构意义**: 这是 Snowflake 从 \"schema-on-write\" 的仓库范式向 \"schema-on-read\" 的湖仓范式的重要延伸。在 Data Cloud 内直接为 AI 应用提供 grounded context，与 Databricks Lakehouse//RT 形成对应——不同路径（Snowflake 从仓→湖、Databricks 从湖→仓）但目标相同：**统一平台上的实时 + AI 分析**。

### 4.2 Snowpipe Streaming + Snowflake CoCo（Jun 17）— 实时管道

> [Snowpipe Streaming and Snowflake CoCo](https://www.snowflake.com/blog/real-time-pipelines-snowpipe-streaming-snowflake-coco/)

**Snowpipe Streaming** 是 Snowflake 的实时数据摄入管道（逐行写入，无需文件 staging 延迟），而 **CoCo** 是新推出的编排层：

- 在 Snowflake 内定义**声明式实时管道**（声明 source → transform → sink 的 DAG）
- 自动处理 schema drift、重试、exactly-once
- 基于 Snowpark Container Services 运行，无需外部编排工具

**架构信号**: Snowflake 正在构建自己的实时管道能力，与 Databricks Delta Live Tables 直接竞争。

### 4.3 Postgres ML Feature Store（Jun 22）— 毫秒级特征服务

> [Millisecond Latency ML Feature Store on Snowflake Postgres](https://www.snowflake.com/blog/millisecond-latency-ml-feature-store-snowflake-postgres/)

Snowflake 在其 PostgreSQL 服务上构建 ML Feature Store：

| 维度 | 说明 |
|:-----|:-----|
| 存储后端 | Snowflake Postgres（强调：Postgres 是 Snowflake 自研的 PG 兼容数据库服务） |
| 延迟 | **毫秒级**特征查询（在线推理场景） |
| 架构 | 特征计算在 Snowflake 数据平台内完成 → 特征向量写入 Postgres → 在线推理时毫秒级读取 |
| 集成 | 与 Cortex AI、Feature Store API 原生集成 |

**架构意义**: Snowflake 采用**计算与服务分离**——特征工程在 Snowflake 数据平台上（利用 SQL + Python UDF），特征服务在 Postgres 上（毫秒级点查）。这是对 Databricks Feature Store（基于 Delta Lake + 向量化）的回应，且 Postgres 优势在于生态兼容性。

### 5.6 Text-to-SQL 趋势总结（July 2026）

```
趋势 1: RL 训练从粗粒度→细粒度
  query-level reward (e.g. SPIN) → clause-level (EXPO-SQL) → progressive multi-turn (Progress-SQL)
  + 自适应协同优化 retriever↔generator (ACE-SQL)

趋势 2: 推理管线从固定 pipeline→agent 化编排
  固定阶段 (schema link → generate → self-correct) 
    → SQLConductor: MCTS 探索最优路径
    → ProSPy: SQL + Python 双语言 Agent
    → DecoSearch: 模型无关的最简路线

趋势 3: 瓶颈认知从模型能力→表示质量
  DBCC: schema 压缩 75× 召回 0%→56.5% （比大模型更明显）
  Schema-First Retrieval: catalog 选择是检索问题
  ZAS-SQL: 零样本规则蒸馏已超越 few-shot

趋势 4: 评估从单方言→多方言
  UniQL 揭示 SQLite 成功的模型在其他方言上大幅下降
  TACO 揭示跨库/歧义查询的实用化差距
```

### 6.1 3 篇特别值得拓展阅读

#### "15-Year Data Lake Reality Check"（2606.08266）

这篇来自金融/电信行业实践者的 **500 条现场记录** 驱动的反思极具参考价值：

> **Seven Deadly Sins**:
> 1. No metadata strategy → data swamp
> 2. Ingest everything → cost explosion
> 3. No access control → compliance nightmare
> 4. Schema-on-read as excuse to defer structure
> 5. Over-reliance on tools → people gap
> 6. Batch thinking in streaming world
> 7. Governance as afterthought → Governance Debt

**核心论点**: Governance Debt（治理债务）是根本问题——每次推迟治理决策都会产生复利。当治理变得困难，"Governance Gravity"（治理引力）会把组织拉回仓库式的结构化方法。Data Lakehouse 和 Data Mesh 在技术上进步了，但组织层面的记录"几乎没变"。

#### Hyperparam（2605.27785）— JS 原生的 Agent 查询引擎

> 这篇定位未来 Agent 计算范式的重要论文：

```
Agent 时代的查询引擎三要求:
  ① JS-native 分发（在 Agent 运行的 JS 运行时内）
  ② 小到能在冷启动标签页 / 每次 Agent sandbox 中分发（<70 KB）
  ③ 分析算子与模型推理可交错执行

Hyperparam 方案:
  Hyparquet (Parquet 读取) + Squirreling (SQL 执行) + Icebird (Iceberg) = 3 个 JS 库 <70KB
  Per-cell async SQL 执行 → 昂贵的"模型推理 cell"只在下游算子需要时才触发
  → Filter-bounded 查询比 DuckDB-WASM 快 300×，整个 Agent 分析套件成本低 2/3
```

**架构信号**: 当 Agent（Claude Code、Cursor 等）和 AI 应用成为数据的主要消费者，传统的湖仓读取路径（Spark/Trino/托管仓库）不适合 JS 运行时。数据工程需要为 Agent-native 的客户端应用更新架构。

#### Skill Issues: Optimizing Lakehouse Agents（2606.01185）

> 这篇将"Agent 技能"视为数据工程产品的论文：

分支湖仓（Bauplan）上的 Agent → Agent 的代码产生**可检查的湖仓状态变更** → 评估从 output-matching 变成 state-verification：
- 自动生成 task-verifier 对
- 在隔离 sandbox 中执行 candidate skills
- 用 trace-level signals + 程序化状态检查评分
- 优化后技能准确率提升 **31.9%**

Delta 从文件系统协议→目录服务协议的转换开始加速。当前 Delta 正在经历的核心转变：
- **协议层**: file-based atomic commits → catalog-mediated commits（UC REST API）

### 🥉 Text-to-SQL 的深化与固化

6 月末的 arXiv 冲刺呈现 Text-to-SQL 的高热度（~20 篇/周）。关键趋势：
- **RL 训练方法论成熟**: clause-level reward (EXPO-SQL)、progressive reward (Progress-SQL)、adaptive co-optimization (ACE-SQL)、calibrated reflection (RefGRPO)

- **零样本方法逼近有样本上限**: ZAS-SQL 88.6% Spider Test (zero-shot SOTA)，超过 GPT-4/4o few-shot
- **实用化基础设施涌现**: DBCC (2.6M→34.7K schema 压缩)、TAHOE (Hint Bank 提示管理)、ProSPy (SQL+Python 双语言)
