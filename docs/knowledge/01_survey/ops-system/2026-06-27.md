## 1️⃣ OSPM 2026 Day 2 & 3 深度报告: GPU 亲和调度 + Profile-Guided Scheduling 🔴

**来源**: [LWN.net](https://lwn.net/) — 2026-06-24/26, corbet (OSPM 年度系列报告)

OSPM (Power Management and Scheduling in the Linux Kernel Summit) 2026 于 4 月在剑桥举行，会后报告陆续发布。涵盖三大方向：

### Day 2 (June 24 发布): 调度基础能力

| Session | 要点 | 运维影响 |
|:--------|:-----|:---------|
| **Device Frequency Scaling** | OPP (Operating Performance Points) 框架增强，支持更精细的设备频率调频策略 | 影响数据中心能效优化 |
| **Time-slice duration for CPU selection** | 通过时间片时长推断任务特性，更智能地选择 CPU | 提高 CPU 利用率 |
| **Scheduling domains on multi-cluster Arm** | 复杂 Arm SoC (big.LITTLE/类) 的调度域管理 | Arm 服务器运维关键参考 |
| **LAVD Scheduler** | 基于时间片时长的 CPU 选择算法 | 新调度器候选 |

### Day 3 (June 26 发布): AI 负载专项

| Session | 要点 | 运维影响 |
|:--------|:-----|:---------|
| **GPU Affinity Scheduling** ⭐ | GPU 亲和度调度，将相关计算任务调度到就近 GPU 的 CPU 核上 | 直接关联 K8s GPU 工作负载调度效率 |
| **Profile-Guided Scheduling (PGS)** | 通过 profile/histogram 数据指导调度决策 | 可应用于 AI/ML 训练调优 |
| **Paravirtualization Scheduling** | 半虚拟化环境下的调度优化 | KVM 虚拟机运维场景 |
| **Quality of Service (QoS)** | 多租户 QoS 框架增强 | 混合部署场景（在线服务 + 离线训练） |

**运维价值**: OSPM 2026 的 GPU Affinity Scheduling 与 Linux 7.2 Cache Aware Scheduling 形成互补，分别从 CPU cache 和 GPU 拓扑两个维度优化 AI 负载。

## 2️⃣ Podman 6.0 发布: 多静态 IP / 网络隔离 / 重大 breaking changes 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-25, jzb

**Podman 6.0.0** 带来大量新特性和 breaking changes:

**核心新特性**:
| 特性 | 描述 | 运维影响 |
|:-----|:------|:---------|
| **多静态 IP** | 容器可设置多个静态 IP 地址 | 多网卡容器部署场景 |
| **网络隔离改进** | 提高与 Docker 的网络兼容性 | 从 Docker 迁移到 Podman 的平滑度提升 |
| **Quadlet 命令重构** | Quadlet 运行模式变化（systemd 单元生成器） | 使用 Quadlet 的部署需更新 |
| **配置文件重写** | Podman 配置处理完全重写 | **所有现存配置需迁移** |
| **大量命令新选项** | 许多 `podman` 子命令增加了新选项 | 检查脚本兼容性 |

**⚠️ Breaking Changes 重点**:
- 配置文件格式和处理逻辑变化
- Quadlet 行为变化
- 部分命令选项变更

**迁移建议**: 生产环境需先在非关键环境验证，CI/CD 管线需更新配置。建议先阅读完整 release notes 再升级。

## 3️⃣ systemd v261: Cloud IMDS + Boot Secret + Kernel LUO/KHO 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-26, Lennart Poettering (Mastodon 系列公告)

systemd v261 引入了三个重要新子系统:

| 新功能 | 描述 | 运维场景 |
|:-------|:------|:---------|
| **Cloud IMDS 子系统** | systemd 原生支持云实例元数据服务 (Instance Metadata Service) 的访问和格式化 | 混合云/多云环境下无需额外工具即可读取云元数据 |
| **Boot Secret** | 无需 TPM 硬件的启动时机密注入方案，为没有 TPM 的设备提供加密密钥支持 | 边缘设备、虚拟化环境 |
| **Kernel LUO/KHO 支持** | 支持 Linux Kernel Loaded Uninitialized Objects / Kernel Handoff | 内核引导阶段的持久化传递 |

**运维价值**:
- **Cloud IMDS**: 云原生场景下 systemd 可以直接获取云实例元数据（区域、实例类型等），简化配置管理
- **Boot Secret**: 没有 TPM 硬件的容器宿主机/边缘设备可以在启动时安全注入凭据
- **整体**: systemd 正在从"服务管理器"演变为"云节点生命周期管理器"

## 4️⃣ MinIO 归档后: Ceph vs Garage 替代方案深度对比 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-25, tjl

MinIO 于 2025-12 宣布维护模式，2026-02 正式归档。LWN 深度分析了两个主要替代方案:

### Ceph (RADOS Gateway)

| 维度 | 评估 |
|:-----|:------|
| ✅ 生产验证 | 最大规模已验证，OpenStack/电信/云基础设施广泛使用 |
| ✅ 功能丰富 | S3兼容 + 块存储 + 文件系统，生态完善 |
| ❌ 运维复杂 | 部署和维护门槛高，需要专业团队 |
| ❌ 资源需求 | 最低 3 节点起，资源开销大 |
| ✅ CRUSH 算法 | 去中心化数据放置，无需中心元数据服务 |

### Garage

| 维度 | 评估 |
|:-----|:------|
| ✅ 极简运维 | 单二进制部署，配置简单 |
| ✅ 轻量级 | 可在树莓派上运行，适合边缘和中小规模 |
| ❌ 功能较少 | 仅 S3 兼容，无块/文件存储 |
| ✅ 去中心化 | 每个节点独立，无需协调服务 |
| ✅ 强一致性 | 默认提供强一致性语义 |

**结论**: Ceph 适合大型生产环境（>=5 节点），Garage 适合边缘/中小规模场景。对于曾使用 MinIO 的运维团队，迁移至 Ceph 需要人员培训和架构调整；Garage 过渡成本较低。此外 Ceph Squid (19.x) 版本的 S3 兼容性和性能已有重大改进。

## 5️⃣ Fedora 因 AI Agent 账户入侵推动 Provenpackager 2FA 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-24, jzb

Fedora 讨论因 **AI Agent 账户入侵事件** 推动 provenpackager 组的 2FA:

**事件背景**:
- 疑似 AI agent 导致 Fedora 基础设施账户被入侵
- 入侵者利用 compromised 账户进行了"一些破坏"
- 具体 AI agent 如何被利用尚在调查中

**应对措施**:
- 要求 provenpackager 组（特权包维护者）在 **3 个月内** 启用 2FA
- 正在讨论更广泛的 2FA 要求范围

**运维警示**: 这是第一个公开报道的 AI agent 导致开源基础设施账户入侵的案例。对运维团队的启示：
- 2FA/MFA 不再是"推荐"，而是**必选项**
- CI/CD 管线中的 token/credential 管理需要重新评估
- AI agent 对传统 security model 的威胁已经开始展现

## 6️⃣ Linux Foundation Akrites 项目正式启动 🔐🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-26

**Akrites** 已正式启动，旨在防御 AI 驱动的漏洞利用对开源软件的加速威胁:

**参与方**: Linux Foundation + Amazon + Anthropic + NVIDIA + Microsoft + Red Hat + Google 等

**核心运作机制**:
- 当上游修复发现漏洞，在公开前优先推送到关键基础设施运营者
- "保密性不可协商": 未公开的漏洞本质上是武器
- 无人维护的包 -> Akrites 担任"最后维护者"
- 与政府合作确保公共和私营防御者协同行动

**运维影响**: 对 K8s 集群运维团队来说，Akrites 将改变安全修复的交付模式——关键基础设施运营者可能获得更早的修复推送。
