## 1️⃣ Systemd v261: Cloud IMDS 新子系统 🔴

**来源**: [LWN.net](https://lwn.net/) — 2026-06-19

Systemd v261 正式发布，最值得关注的特性：

**Cloud Instance Metadata Service (IMDS)** 子系统：
- 统一抽象各云厂商的实例元数据服务（AWS IMDS、GCP Metadata、Azure IMDS）
- 减少对云厂商自带 Agent 的依赖（如 AWS SSM Agent、Azure Linux Agent）
- 提供标准化的本地元数据访问接口
- 适用于多/混合云场景统一管理

其他重要特性：
- **Boot Secret 功能**: 为缺少物理 TPM 的设备提供类似 TPM 的启动秘密管理
- **KHO (Kexec Handover) 支持**: 热升级场景的内核无中断切换

## 2️⃣ Fedora 强制 2FA: 起因是 AI Agent 入侵事件 🔴

**来源**: [LWN.net](https://lwn.net/) — 2026-06-24

**事件回溯**: Fedora 项目的一个 provenpackager 账户被攻破，攻击者利用 AI Agent 自动化地制造了大量破坏。

**应对措施**:
- Fedora 要求 provenpackager 组在 **3 个月内强制启用 2FA**
- 这是继 AURpocalypse（AUR 攻击）后的又一起开源供应链安全事件
- **趋势**: 开源基础设施正加速向 2FA/MFA 转型，尤其是拥有软件包制作者权限的账户

**启示**: AI 不仅被用来防御，也成为攻击工具——被攻破的账户 + AI Agent 自动化可造成远超传统手动攻击的破坏规模。

## 4️⃣ AURpocalypse: AUR 供应链攻击复盘

**来源**: [LWN.net](https://lwn.net/) — 2026-06-19

**攻击手法**:
1. 攻击者批量创建新账户
2. 认领无人维护的 AUR 包（orphaned packages）
3. 推送恶意更新（包含后门/木马）
4. 维护者发现后如同打地鼠——关掉一个，新账户认领另一个

**后果**:
- AUR 暂停新用户注册（至今未恢复）
- 现有用户受影响范围尚不明确
- **根本问题**: AUR 的信任模型基于"谁先认领"而非"谁值得信任"
- 社区正在讨论长期解决方案（code review、签名验证、可信包维护者制度）

**对比 Fedora 事件**: 同样是开源包仓库安全问题，Fedora 选择强化账户验证（2FA），而 AUR 面临的是从信任模型层面的重构挑战。

## 5️⃣ Linux 7.2 合并窗口关键进展

**来源**: [LWN.net](https://lwn.net/) — 2026-06-18 ~ 2026-06-24

**规模**: ~7,000+ changesets，主要核心模块已合入

| 特性 | 说明 | 运维价值 |
|:-----|:-----|:---------|
| **BPF 协程化** | BPF 程序可用协程挂起/恢复，支持长时间运行任务 | 监控/安全 Agent 可编写更复杂的 BPF 逻辑 |
| **libarena** | BPF Arena 通用工具库，零校验内存结构构建 | eBPF 可编程观测能力显著增强 |
| **KASAN for BPF JIT** | KASAN 内存检查器覆盖到 JIT 编译的 BPF 代码 | 发现 BPF JIT 编译器中的内存管理 bug |
| **RMR/BRMR** | 新内存区域机制（Reserved/Bluetooth Reserved Memory Region） | 影响嵌入式/IoT 运维场景 |
| **OSPM 2026** | 设备频率缩放、LAVD 调度器、多集群 Arm 调度域 | 能效调度优化，ARM 服务器运维关键 |

## 6️⃣ CNCF: Telemetry That Matters — Pipeline 可持续设计

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-22

**核心困境**:
- 观测数据洪流：系统越复杂 telemetry 越多
- 真实价值数据不到 1%，其余都是噪音
- 盲目全量采集 → 存储爆炸 → 成本失控

**Pipeline 改造方案**:
1. **Triage-First 采集**: 先收关键信号，检测到异常再展开全量
2. **自适应采样**: OTel Collector 的动态 Tail Sampling → 根据健康状态调整
3. **成本归属**: 每个 Pipeline 阶段建立 ROI 指标，追踪 telemetry 成本
4. **与 OTel-Arrow Phase 2 结合**: 列式格式贯穿 Collector 内部，降低序列化开销

**实践建议**: 在引入新 telemetry 源时先问「这个数据能直接导致一个自动化动作吗？」—— 如果不能，它很可能只是噪音。

## 7️⃣ Jaeger v2.18 ClickHouse 对运维的影响

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-23

| 指标 | Elasticsearch | ClickHouse | 变化 |
|:-----|:-------------|:-----------|:-----|
| 存储压缩比 | ~3× | **8.6×** | 减少 65%+ |
| 写入吞吐 | ~20k spans/s | **>50k spans/s** | 2.5× |
| 索引性能 | 写入放大严重 | MergeTree 原生高效 | 大幅降低 |

**运维直接收益**:
- 相同存储预算下，追踪保留期可从 7 天延长至 60+ 天
- 降低 Jaeger 集群的硬件需求（少 60%+ 存储节点）
- ClickHouse 比 ES 更易运维（无 GC 问题、无分片重平衡烦恼）
