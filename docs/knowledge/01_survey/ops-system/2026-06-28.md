## 1️⃣ Fedora AI Agent 账户入侵 → 强制 Provenpackager 2FA 🔴

**来源**: [LWN.net](https://lwn.net/) — 2026-06-24, jzb

**事件概要**: Fedora 项目发生**首次公开报道的 AI Agent 账户入侵事件**。一个 AI agent 被利用在 Fedora 基础设施中造成了一系列问题，随后被追溯为 provenpackager 账户被攻破所致。

**后续措施**: Fedora 正在讨论强制 2FA 要求，经过讨论后决定：
- **Provenpackager 组成员必须在 3 个月内启用 2FA**
- 更广泛的包维护者 2FA 要求正在评估中

**运维意义**: 
- ✅ **首个公开报道的 AI agent 供应链攻击案例** — 标志着 AI agent 安全已从理论讨论进入实际攻击阶段
- ✅ AI agent 权限管理需要纳入安全和合规清单
- ✅ 开源基础设施的 CI/CD 安全需重新评估

**关联参考**:
- [Dapr 1.18 Verifiable Execution](../bmc-system/2026-06-28.md) (5️⃣) — 与 AI Agent 信任链方案互补

## 2️⃣ Akrites 漏洞缓解项目正式启动 🔴

**来源**: [LWN.net](https://lwn.net/) — 2026-06-26, corbet

**项目概要**: Linux Foundation 联合大量组织和公司宣布启动 **Akrites 项目**，旨在加速漏洞修复进入项目的流程。

**联合发起方**: Linux Foundation + NVIDIA + Microsoft + OpenAI 等

**核心理念**:
- AI 驱动的逆向工程使漏洞修复必须比以往更快部署
- "一个未公开的广泛软件包缺陷就是武器"（开源的脆弱性）
- 当关键包无人维护时，Akrites 将作为**最后的维护者**来提供修复
- 修复将流回各个项目的主仓库

**关键设计原则**:
| 原则 | 描述 |
|:-----|:------|
| **保密性不可妥协** | 防止漏洞在修复前被武器化 |
| **上游优先** | 修复流回项目自身的维护者 |
| **与政府协同** | 公共和私人防御者步调一致 |
| **维护者兜底** | 无人维护的关键包由 Akrites 站出 |

**运维价值**: 对大型企业来说，Akrites 填补了"开源漏洞响应"和"AI 加速的攻击"之间的时间差。建议关注 Akrites 的 patch 部署渠道。

## 3️⃣ OSPM 2026 Day 3: GPU 亲和调度 + Profile-Guided 调度 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-26, corbet (OSPM 系列)

**Day 3 核心 Session** (剑桥，4月举办，6月底发布报告):

| Session | 技术要点 | 运维 / K8s 影响 |
|:--------|:---------|:----------------|
| **GPU Affinity Scheduling** ⭐ | 将相关计算任务调度到就近 GPU 的 CPU 核上 | 直接提升 K8s GPU 工作负载性能 |
| **Profile-Guided Scheduling (PGS)** | 通过 profile/histogram 数据指导调度决策 | 可用于 AI 训练任务调优 |
| **Paravirtualization Scheduling** | 半虚拟化环境下的调度优化 | KVM 虚拟机运维 |
| **Quality of Service (QoS)** | 多租户 QoS 框架增强 | 在线服务+离线训练混合部署 |

**与已有跟踪的关联**:
- 与 Linux 7.2 Cache Aware Scheduling 互补（CPU cache 维度）
- 与 OSPM Day 2 (device frequency scaling, LAVD scheduler) 形成完整框架

## 4️⃣ Podman 6.0 发布: 多静态 IP + 网络隔离 + 配置重写 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-25, jzb

**核心更新**:

| 特性 | 描述 | 运维注意 |
|:-----|:------|:---------|
| **多静态 IP** | 容器可设置多个静态 IP 地址 | 多网卡/多网络命名空间场景 |
| **网络隔离改进** | 提高与 Docker 的网络兼容性 | Docker→Podman 迁移更平滑 |
| **Quadlet 命令重构** | systemd 单元生成器行为变化 | **使用 Quadlet 需更新部署** |
| **配置处理重写** | 配置文件格式完全改变 | **所有现存配置需迁移** |
| **大量 breaking changes** | API/CLI 行为变化 | 需检查 CI/CD 脚本兼容性 |

**运维价值**: Podman 6.0 标志着 Podman 作为 Docker 替代方案更加成熟，但升级需谨慎处理 breaking changes。

## 5️⃣ systemd v261 新特性概览 🟡

**来源**: [LWN.net](https://lwn.net/) — 2026-06-26, corbet

Lennart Poettering 发布了一系列 Mastodon 帖子介绍 systemd v261 的特性：

| 新特性 | 描述 | 运维影响 |
|:-------|:------|:---------|
| **Cloud IMDS 子系统** | 直接支持从云实例元数据服务 (IMDS) 获取配置 | 云原生环境下减少配置依赖 |
| **Boot Secret** | 无 TPM 的情况下也能使用加密密钥 | 边缘设备/无 TPM 服务器适用 |
| **Kernel LUO (Last Used Organization)** | 内核级内存页面最后使用组织跟踪 | 内存回收策略优化 |
| **KHO (Kexec Handover)** | kexec 热接管增强 | 内核升级不中断 |

**来源**: [LWN.net](https://lwn.net/) — 2026-06-25, corbet

| 技术 | 原理 | 效果 |
|:-----|:------|:-----|
