## 1️⃣ NVIDIA Dynamo Snapshot：K8s 推理冷启动 21x 加速 🚀

**来源**: NVIDIA Technical Blog · 2026-05-27 · Schwinn Saereesitthitipak 等
**标签**: `NVIDIA Dynamo` `CRIU` `cuda-checkpoint` `K8s冷启动` `GPU推理`

### 核心问题
生产推理集群需要弹性伸缩，但 K8s 上冷启动推理负载需要数分钟，期间 GPU 已分配但空闲不产生 token，SLA 风险高。

### 技术方案：CRIU + cuda-checkpoint 组合
- **设备状态 (GPU侧)**：CUDA contexts/streams/device memory → `cuda-checkpoint` 序列化到 CPU 内存
- **主机状态 (CPU侧)**：process tree/file descriptors/namespaces → `CRIU (Checkpoint/Restore in Userspace)` 序列化到磁盘
- 恢复后在**精确指令位置**恢复执行，workload 无感知

### K8s 架构设计
- **snapshot-agent DaemonSet**：通过 Helm chart 部署，每个节点独立处理 checkpoint/restore
- **Quiesce/Resume Hooks**：worker 在 engine 初始化完成但分布式运行时尚未启动时 checkpoint，通过信号文件同步恢复流程
- 无需修改 runc，不依赖云厂商 checkpoint/restore feature gates

### 三大优化

| 优化 | 方案 | 效果 |
|:-----|:-----|:------|
| **KV Cache Unmap** | cuMemUnmap+cuMemRelease 去分配KV Cache | Qwen3-0.6B checkpoint: 190GiB→**6GiB** |
| **Parallel memfd restore** | 线程池并行恢复 shmem-backed memfd | gpt-oss-120b restore: 119s→**15s** |
| **Linux Native AIO** | io_submit 并发 128 个 read in-flight | 结合并行 memfd 共 **7.9x** 加速 |
| **GPU Memory Service (GMS)** | 将权重解耦到独立 artifact，并发恢复 | 结合本地 NVMe：**21x** 加速 (gpt-oss-120b, <5s) |

### 结果数据

| Model | Checkpoint | Upstream CRIU | Optimized | Speedup |
|:------|:----------|:-------------|:---------|:--------|
| Qwen3-0.6B | 6.2 GiB | 6.8s | 2.4s | 2.8x |
| Qwen3-8B | 26 GiB | 24s | 4.7s | 5.1x |
| gpt-oss-120b | 129 GiB | 119s | 15s | **7.9x** |

**GMS + 本地 NVMe Striped SSD**: gpt-oss-120b 端到端恢复 **<5s**，**21x 加速**

### 路线图
- ✅ 实验版已支持：单 GPU vLLM/SGLang（非 GMS 路径）
- 🚧 GMS restore path（GDS/UCX后端，待 CUDA driver patch）
- 🚧 TensorRT-LLM 支持
- 🚧 多 GPU/多节点支持（通过 quiesce/resume hooks for PyTorch/NCCL/NIXL）

## 2️⃣ NVIDIA GPU Usage Monitor：开源 K8s GPU 可观测性 🖥️

**来源**: NVIDIA Technical Blog · 2026-05-21 · Guy Saltoun (Run:ai)
**标签**: `GPU监控` `DCGM` `Prometheus` `Grafana` `K8s可观测性`

### 背景
平台团队在 GPU K8s 集群上面临两个常见问题：
1. **过度分配**：工程师请求整 GPU 但通常只用 30-50% 资源
2. **Pod 饥饿盲区**：GPU 请求堆积导致 Pod Pending，通常用户报告才被发现

标准 K8s 监控栈 (kube-state-metrics + node-exporter) **不暴露 GPU 级信号**。

### 架构
```
DCGM Exporter ─┐
               ├──→ Prometheus → Grafana Dashboard
kube-state-metrics ┘
```
- **DCGM Exporter**：暴露 NVIDIA GPU 硬件指标
- **kube-state-metrics**：暴露 K8s Pod 和资源指标
- **Prometheus**：统一存储
- **Grafana**：预构建 GPU 仪表盘

### 核心特性
- ⚡ **单 Helm 命令部署**（`helm install gpu-usage-monitor`）
- 📊 **预构建仪表盘**：GPU 分配趋势、计算利用率阈值、Pod 级内存消耗、运行中 vs Pending Pod 计数
- 🔍 **GPU 类型过滤**：按平台过滤 (Hopper/Blackwell/Blackwell Ultra)
- 🔌 **外部 Prometheus 集成**：可接入已有监控栈
- 🆓 **Apache 2.0 开源**

### 可观测信号
- Namespace/Workload 级别的 GPU 分配趋势（发现已分配未使用的空闲容量）
- 逐 GPU 利用率百分比 + 可配置告警阈值
- Pod 级 GPU 内存消耗（右缩放的依据信号）
- 集群级运行中 vs Pending GPU Pod 计数（调度压力早期预警）
- GPU 类型过滤（异构集群场景）

## 3️⃣ etcd 3.7.0-beta.0 发布：RangeStream 功能 🗄️

**来源**: Kubernetes Blog (SIG-etcd) · 2026-05-20
**标签**: `etcd` `K8s组件` `RangeStream`

### 核心更新
- **RangeStream 功能**：长期请求的流式 Range 操作，显著优化大规模 Watch 场景
- **重构与清理**：代码库现代化，移除遗留代码
- **性能优化**：减少大规模 K8s 集群中 etcd 的 watch 延迟

### 影响
- K8s v1.36 中的 Server-Side Sharded List and Watch 依赖 etcd 底层能力
- 对 5000+ 节点集群的 API Server 性能有直接改善

## 4️⃣ Kubernetes Dashboard → Headlamp 迁移 🧭

**来源**: Kubernetes Blog · 2026-06-01 · Will Case (Headlamp)
**标签**: `K8s Dashboard` `Headlamp` `K8s运维工具`

### 核心内容
- Headlamp 作为 Kubernetes Dashboard 的现代化替代品正式确立
- Headlamp 是由 CNCF 托管的 K8s UI 项目，支持插件系统、自定义主题、多集群管理
- K8s Dashboard 团队建议用户规划迁移路径
- **关键区别**：Headlamp 是桌面应用（desktop app）而非纯 Web UI，支持本地运行 + 任意集群连接

### 迁移建议
- 使用 Headlamp 的内置插件生态（已支持 Karpenter 等插件）
- 支持通过 Helm chart 部署到集群作为 Web UI
- 支持 OIDC / OpenID Connect 认证集成
