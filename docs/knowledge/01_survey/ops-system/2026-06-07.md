## 1️⃣ 🔥 OTel GenAI Observability — GenAI 语义约定正式落地

**来源**: [OpenTelemetry Blog · 2026-05-14](https://opentelemetry.io/blog/2026/genai-observability/)  
**作者**: James Newton-King (Microsoft)  
**版本**: OTel GenAI Semantic Conventions (GA 候选阶段)

### 关键特性

**GenAI 语义约定标准化**了一套完整的 LLM 可观测属性：

| 属性 | 说明 |
|:-----|:------|
| `gen_ai.request.model` | 调用的模型名称（如 `gpt-4o`） |
| `gen_ai.usage.input_tokens` | 输入 token 数 |
| `gen_ai.usage.output_tokens` | 输出 token 数 |
| `gen_ai.response.finish_reasons` | 停止原因（`stop` / `tool_calls`） |
| `gen_ai.system_instructions` | 系统提示（可选，需启用 content capture） |
| `gen_ai.input.messages` | 输入消息内容（可选） |
| `gen_ai.output.messages` | 输出消息内容（可选） |

### 落地情况

- **VS Code Copilot** 已全面支持 OTel GenAI telemetry——可 emit trace/metric/event，需设置 `github.copilot.chat.otel.enabled=true`
- **OpenAI Codex** 导出结构化日志事件 + OTel 指标（API 请求、工具调用、会话级）
- **Claude Code** 通过 OTel 导出指标和日志事件，trace 支持处于 Beta
- **演示工具链**: 搭配 Aspire Dashboard（开源 Docker 容器），接收 OTLP 后可视化 trace 树（`invoke_agent` → `chat` × N → `execute_tool` × N）和 metrics（`gen_ai.client.operation.duration` 延迟直方图、`gen_ai.client.token.usage` token 直方图）

### 运维意义

这是 **AI Agent 可观测性的里程碑**。过去调试 AI Agent "为什么慢/为什么答错"全靠猜测，现在有了标准化的 trace 链路——能精确看到哪一步 LLM 调用延迟高、哪个工具调用失败、输入输出 token 是否异常。GenAI Metrics 还能做**按模型过滤的成本归因**和**延迟回归检测**。

## 2️⃣ 🔥 KEDA GPU Autoscaling — 开源参考实现完整架构

**来源**: [CNCF Blog · 2026-05-27](https://www.cncf.io/blog/2026/05/27/gpu-autoscaling-on-kubernetes-with-keda-building-an-external-scaler/)  
**作者**: Pavan Madduri (Grainger, CNCF Golden Kubestronaut)  
**版本**: keda-gpu-scaler (社区参考实现)

### 核心架构问题

KEDA 核心使用 `CGO_ENABLED=0` 编译，而 **NVIDIA NVML 需要 CGO**，无法直接在 KEDA 内置 scaler 中调用。同时 NVML 是**节点本地接口**，无法跨节点查询。

### 解决方案：DaemonSet + gRPC ExternalScaler

```
┌─────────────────┐     gRPC      ┌───────────────────────┐
│  KEDA Operator  │ ←──────────→ │  keda-gpu-scaler      │
│  (集群级)        │               │  DaemonSet (每节点)    │
└─────────────────┘               │  go-nvml → NVML       │
                                   └───────────────────────┘
```

### 暴露的 GPU 指标（每 GPU）

| 指标 | 说明 |
|:-----|:------|
| `gpu_utilization` | SM (compute) 利用率 % |
| `memory_utilization` | 内存控制器利用率 % |
| `memory_used_percent` | VRAM 使用 % |
| `temperature` | GPU 核心温度 ℃ |
| `power_draw` | 当前功耗 W |

**多 GPU 聚合**: `max` / `min` / `avg` / `sum`，或指定单 GPU 索引

### 预置 Profile

| Profile | 指标 | 目标值 | 激活值 | 场景 |
|:--------|:-----|:-------|:-------|:-----|
| `vllm-inference` | memory_used_percent | 80% | 5% | LLM serving, scale-to-zero |
| `triton-inference` | gpu_utilization | 75% | 10% | Triton 模型服务 |
| `training` | gpu_utilization | 90% | 0% | 训练任务，无 scale-to-zero |
| `batch` | memory_used_percent | 70% | 1% | 批推理，激进缩容 |

**快速部署**: `helm install gpu-scaler deploy/helm/keda-gpu-scaler`，配合 ScaledObject 声明式配置，minReplica=0 实现 idle 时 scale-to-zero。

### 运维意义

填补了 K8s GPU 弹性伸缩的空白——此前 CPU/memory 指标的 HPA 对 GPU 工作负载几乎无效。现在 vLLM 推理服务可以在 idle 时缩零、流量上升时按 GPU 内存利用率弹性扩容，**显著降低 GPU 闲置浪费**（作者特意提到这也是 GreenOps 问题——浪费的 GPU 周期=浪费的能耗+Scope 3 排放）。

## 3️⃣ 📋 CNCF IAM Whitepaper — 云原生 IAM 最佳实践

**来源**: [CNCF Blog · 2026-06-04](https://www.cncf.io/blog/2026/06/04/identity-and-access-management-whitepaper/)  
**作者**: CNCF TAG Security and Compliance  
**版本**: 白皮书首次发布

### 核心内容

- **身份即新安全边界**：传统基于网络的 perimeter 安全模式在分布式动态环境中失效
- **SPIFFE 工作负载身份**：服务到服务认证的标准方案
- **PEP/PDP 架构**：策略执行点+策略决策点分离的授权模式
- **零信任参考架构**：有状态/无状态工作负载的 IAM 实现模式

### 与运维的关联

IAM 白皮书直接关联到 K8s 权限管理的深层问题——RBAC 只是起点，在 AI 工作负载（多租户 GPU 集群、跨 namespace 推理服务）场景下，**SPIFFE 工作负载身份 + 细粒度授权策略**才是生产级方案。

## 4️⃣ 🛡️ OTel Security for Legacy Environments — 工业场景可观测安全

**来源**: [OpenTelemetry Blog · 2026-05-19](https://opentelemetry.io/blog/2026/security-legacy-environments/)  
**作者**: Lukasz Ciukaj (Cisco Splunk)  
**版本**: 最佳实践指南

### 核心方法论

针对**制造/工业/OT 环境**中系统不可修改、网络无分段、补丁周期长等约束，提出：

1. **Collector-as-Bridge 模式**：Collector 运行在系统外部（独立 VM/容器），不侵入遗留系统，可独立更新打补丁
2. **决策模型**：安全关键系统→低影响信号+被动监控；可修改系统→SDK 自动埋点；不可修改但有数据→Collector receiver；仅有日志→filelog receiver
3. **数据最小化**：避免全量 payload、优先聚合信号、定期审查属性
4. **处理器示例**：`transform` 处理器做 hash + delete_key，`redaction` 处理器做 allowed_keys 白名单

### 运维意义

随着 AI 推理向边缘/工厂下沉（如 NVIDIA IGX/Jetson 的工业推理场景），OTel 在 OT 环境的安全部署模式变得重要——不再是"能不能监控"的问题，而是"如何在不能打补丁的设备上安全监控"。

## 5️⃣ ℹ️ CNCF CI/CD Supply Chain Security — 供应链安全深度分析

**来源**: [CNCF Blog · 2026-06-04](https://www.cncf.io/blog/2026/06/04/securing-ci-cd-for-an-open-source-project-controlling-who-runs-what/)  
**作者**: André Martins (Isovalent at Cisco), Feroz Salam (Isovalent at Cisco)  
**版本**: 系列文章 Part 1

### 核心观点

- **Axios npm 的 RAT 植入**和 **LiteLLM PyPI 账户劫持**两个真实案例表明：CI/CD 管道本身是最大的攻击面
- **CI runner 信任模型**：不是"谁提交了代码"，而是"谁的 runner 在执行代码"
- 建议：使用硬件隔离 runner（如 GitHub Actions larger runner）、限制外部 PR 的 CI 触发权限、对 artifact 做签名校验

### 与运维的关联

直接关联 K8s 供应链安全——`kyverno`/`ratify`/`cosign` 等 CNCF 安全工具链需要在 CI/CD 环节就建立信任基线。
