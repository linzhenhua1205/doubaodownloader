### 🔒 Cilium CI/CD 供应链安全七层防线（June 4）

**来源**: [CNCF Blog](https://www.cncf.io/blog/2026/06/04/securing-ci-cd-for-an-open-source-project-controlling-who-runs-what/)

Cilium 核心团队系统公开了其 CI/CD 供应链安全的完整防御架构（Part 1 触发器控制），从运维安全视角有 4 个可复用的实践模式：

**🛡️ 模式 1: Ariane bot 工作流调度器**
- 仅组织验证成员可从 PR comment 触发 CI
- 工作流名**硬编码在配置白名单**，支持依赖关系（`depends-on`）
- 随机外部评论者无法启动昂贵的云提供商 conformance 测试

**🛡️ 模式 2: pull_request_target 安全双阶段 checkout**
- 第一阶段：checkout 基分支（已审核代码）→ 加载 composite actions/签名逻辑
- 第二阶段：checkout PR head → **仅作为 Docker build context**
- 关键安全约束：`run:` 步骤不执行任何 untrusted 脚本，composite actions 全来自基分支
- BuildKit 隔离运行：无 GITHUB_ENV 访问、无 repo secrets、无 Docker credential store
- Docker login 仅用 CI 级凭据（只能推 `-ci` 开发 registry）

**🛡️ 模式 3: CODEOWNERS 安全审查门禁**
- `.github/` 下所有变更 → `@cilium/github-sec` + `@cilium/ci-structure`
- `auto-approve.yaml` → `@cilium/cilium-maintainers`
- 无人可绕过安全团队修改 CI 管线

**🛡️ 模式 4: 依赖和运行时安全**
- SHA-pinned actions + container images（40 位 commit SHA + @sha256:digest）
- Renovate 自动更新但有 5 天冷却窗口
- Go vendor 目录检入，团队评审
- CodeQL 强制 `permissions:` 声明 + actionlint 检测 unsafe patterns

**尚未补齐**: SLSA provenance、PR 时依赖审查、govulncheck、内部 @main 引用迁移

> **运维启示**: Cilium 的安全模式是**层级递进**设计——每个控制层只阻止一类攻击向量，不依赖任何单点防御。这比常见的一层防火墙式的安全设计更可运维。

### 👤 Prometheus UX Research 工作组正式成立（April 8）

**来源**: [Prometheus Blog](https://prometheus.io/blog/2026/04/08/introducing-ux-research-working-group/)

Prometheus 社区正式成立用户体验研究工作小组，标志着从纯技术驱动向设计驱动的范式转变：

- **动因**: PromCon 2025 用户研究揭示了显著的体验痛点（onboarding 流程、配置复杂度、文档、心智模型、OTel 互操作性）
- **目标**: 通过结构化用户研究和设计视角，系统性地改善 Prometheus 的整体体验
- **范围**: 支持 Prometheus 所有子项目，提供统一的 UX 研究和可用性视角

> 这是 Prometheus 从"功能完整性"走向"用户完整性"的重要信号。

### 🧠 Prometheus 原生存储复合类型路线（Feb 14）

**来源**: [Prometheus Blog](https://prometheus.io/blog/2026/02/14/modernizing-prometheus-native-storage-for-composite-types/)

虽然距今稍早，但这是 Prometheus 3.0 时代最重要的架构演进之一，值得纳入跟踪：

- **核心理念**: 为 Prometheus TSDB 引入复合类型的原生存储（native histograms 的延伸）
- **解决痛点**: 用户必须在 PromQL 中手动聚合高基数数据（如通过 `sum(rate(...)) by (le)`），原生支持后无需在查询时计算
- **与 OTel 的关系**: OTel metrics 的复杂类型可以无损存储，无需在 ingestion 时打平
- **社区目标**: 让 Prometheus 成为 OpenTelemetry metrics 的**默认存储后端**

### 🧩 GenAI Observability 与 OTel 语义约定已投入生产（May 14）

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/genai-observability/)

OpenTelemetry GenAI 语义约定（Semantic Conventions）正式用于生产：

- **三大 GenAI 工具的 OTel 支持状态**:
  - VS Code Copilot: 已完整支持 traces/metrics/events
  - OpenAI Codex: 结构化日志 + OTel metrics
  - Claude Code: metrics + 事件，trace beta

- **关键指标**: `gen_ai.client.operation.duration`（延迟直方图）, `gen_ai.client.token.usage`（Token 消费直方图，可按 input/output 过滤）
- **安全保障**: 默认不捕获 prompt 内容，需显式启用 `captureContent`
- **Aspire Dashboard**: 免费开源的可视化工具，支持 GenAI span 树（invoke_agent → chat → execute_tool）

> **运维启示**: AI Agent 的可观测性已从概念走向标准——O Tel GenAI 语义约定是大模型运维的基线要求。
