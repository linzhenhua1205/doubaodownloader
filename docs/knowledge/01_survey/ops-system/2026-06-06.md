## 1️⃣ 🏭 NVIDIA DSX OS 正式发布：AI Factory 操作系统开源化

- **来源**: NVIDIA Technical Blog · 2026-05-31 · Warren Barkley
- **版本**: DSX OS v1.0 (初始发布)
- **关键特性**:

NVIDIA DSX OS 是一套**开源、模块化**的软件组件集合，专门为 AI Factory 的规模化运营而设计，解决三大核心目标：

| 目标 | 技术手段 | 量化指标 |
|:-----|:---------|:---------|
| ⚡ **更快创收** | DSX OS 组件以 GitHub 开源形式发布，生态伙伴可直接复用 NVIDIA 在 DGX Cloud 上已验证的运维软件栈 | 消除数月定制开发 |
| 💡 **更高能效** | DSX MaxLPS（动态功耗策略引擎）将电力视为可编程资源，在 GPU/机架/冷却/负载四级动态分配 | 固定功率预算内 **多运行 40% GPU**，推理性能影响极小 |
| 🛡️ **更高可靠性** | 从被动告警转向自动化修复 + 跨区域运行时版本一致性 + Fleet Intelligence 全局可见性 | — |

**核心组件矩阵**：

| 组件 | 功能 | 关键技术 |
|:-----|:------|:---------|
| **DSX Exchange** | IT/OT 通信枢纽，连接计算/网络/电力/冷却系统 | MQTT 总线 + MCP 协议 Server，支持 AI Agent 发现工厂全表面为统一工具目录 |
| **DSX MaxLPS** | 动态功率策略引擎，GPU→机架→冷却→负载级细分 | 回收获回功率供电更多计算 |
| **DSX Flex** | 电网服务连接库，支持需求响应/削峰/可再生能源适配 | — |
| **NVIDIA Infra Controller (NICo)** | API 驱动的裸金属生命周期管理 | BlueField DPU + DOCA 硬件级租户隔离 |
| **AI Cluster Runtime (AICR)** | 版本锁定运行时配置，消除配置漂移 | — |
| **NVIDIA NVSentinel** | K8s 原生 GPU 故障检测与自动修复 | 秒级封锁不健康节点 + 排空工作负载 |
| **NVIDIA Fleet Intelligence** | 全局舰队可见性/完整性验证/健康监控 | — |
| **KAI Scheduler + Run:ai** | GPU 感知智能调度（拓扑感知 + 分片分配 + 层级配额）| — |
| **NVIDIA Dynamo + Grove** | 分布式推理服务（PD 分离 + 每阶段自动伸缩）| — |
| **NVCF (Cloud Functions)** | 统一推理/微调/批处理 API + 多租户 | — |

**生态伙伴**：CoreWeave、Lambda、Nscale、Red Hat、Supermicro、Mirantis、Rafay、Spectro Cloud、Vultr 等已集成在生产中使用。

> 🔗 https://developer.nvidia.com/blog/nvidia-dsx-os-delivers-open-modular-software-for-operating-ai-factories-at-scale/

## 3️⃣ 🤖 Jaeger v2 进化：以 OTel 为底座追踪 AI Agent

- **来源**: CNCF Blog (Jaeger Project Maintainer) · 2026-05-26 · Jonah Kowall
- **版本**: Jaeger v2
- **关键特性**:

Jaeger 正在从分布式微服务追踪全面转型为 **AI Agent 可观测性平台**，核心变化：

**🔧 架构重构**：Jaeger v2 将底层收集机制替换为 OpenTelemetry Collector 框架，原生 OTLP 接入，消除中间转换，实现 Metrics/Logs/Traces 三信号统一部署。

**🤝 三大协议集成**：

| 协议 | 作用 |
|:-----|:------|
| **MCP** (Model Context Protocol) | 标准化 AI 模型安全访问外部数据源 |
| **ACP** (Agent Client Protocol) | 用户界面与 AI Agent 通信的统协议 |
| **AG-UI** (Agent-User Interaction) | Agent-人类协同交互协议 |

**🖥️ 新 UI 工作空间**：前端基于 Zustand + React Query（替代 Redux），集成 AG-UI 辅助助手的 in-app assistant，支持自然语言查询（如"找出支付服务中延迟 >2s 的 500 错误"）→ 自动翻译为确定性 Trace 查询。

**🧬 GenAI 可视化**：跟踪 RAG 流水线（嵌入模型延迟 + 外部工具调用 + Token 用量），支持 OpenTelemetry GenAI Semantic Conventions（Issue #2664 Agentic Systems、Issue #3583 AI Sandboxes）。

**本地→生产一致性**：Jaeger v2 容器可在本地运行 SLM 小模型私密测试，生产环境替换为云端 LLM 处理大规模事故分析。

> 🔗 https://www.cncf.io/blog/2026/05/26/how-jaeger-is-evolving-to-trace-ai-agents-with-opentelemetry/

## 4️⃣ 🚀 KEDA GPU Autoscaling 开源参考实现

- **来源**: CNCF Blog (Community Post) · 2026-05-27 · Pavan Madduri (Grainger)
- **版本**: keda-gpu-scaler (开源参考实现)
- **关键特性**：

解决 K8s 中 GPU 工作负载无法按 GPU 原生指标自动伸缩的痛点：

**💡 架构设计**：因 NVML 依赖 CGO（与 KEDA `CGO_ENABLED=0` 冲突），采用 **DaemonSet 模式**：
- 每个 GPU 节点运行一个 Daemon Pod，通过 go-nvml 读取本地 GPU 指标
- 通过 gRPC 暴露 KEDA ExternalScaler 接口
- KEDA Operator 连接 scaler 驱动 HPA 决策
- 与 K8s Device Plugin / Metrics Server 相同的每节点收集模式

**📊 支持指标**：
- `gpu_utilization` — SM 计算利用率 %
- `memory_utilization` — 内存控制器利用率 %
- `memory_used_percent` — VRAM 用量 %
- `temperature` — GPU 核心温度 ℃
- `power_draw` — 当前功耗 W

**📋 预置 Profile**：

| Profile | 指标 | 目标 | 激活阈值 | 场景 |
|:--------|:-----|:-----|:---------|:-----|
| `vllm-inference` | memory_used_percent | 80% | 5% | LLM serving，支持 scale-to-zero |
| `triton-inference` | gpu_utilization | 75% | 10% | Triton 模型服务 |
| `training` | gpu_utilization | 90% | 0% | 训练作业，不 scale-to-zero |
| `batch` | memory_used_percent | 70% | 1% | 批量推理，激进缩容 |

**部署**：单 Helm 命令 `helm install gpu-scaler deploy/helm/keda-gpu-scaler`，自动发现 `nvidia.com/gpu.present=true` 节点。

> 🔗 https://www.cncf.io/blog/2026/05/27/gpu-autoscaling-on-kubernetes-with-keda-building-an-external-scaler/

## 5️⃣ ⚡ NetEase Games：Fluid + Alluxio 实现 30 秒 LLM 冷启动

- **来源**: CNCF Blog (End User Post) · 2026-05-21 · Haifeng Liao (Senior Infra Engineer)
- **版本**: Fluid (CNCF Incubating) + Alluxio
- **关键特性**：

网易游戏在 Tmax AI 平台上的实战案例，解决了**服务器端 LLM 推理冷启动**的核心瓶颈：

**📉 性能提升路径**：

| 方案 | 模型加载时间 | 倍数提升 |
|:-----|:------------|:---------|
| 跨地域直读存储 | 42 分钟 | 1× |
| 传统 Alluxio 缓存 | 14 分钟 | 3× |
| Fluid 预取工作流启用后 | 3 分钟 | 14× |
| **生产调优后** | **<30 秒** | **84×+** |

**🧩 Fluid 的关键价值**——为什么比直接跑 Alluxio 更好：

| 维度 | 直接 Alluxio | + Fluid 抽象层 |
|:-----|:-------------|:---------------|
| K8s 集成 | 手动部署管理 Master/Worker | **自动化运行时生命周期 + HPA/KEDA 缓存弹性 + 数据感知调度** |
| LLM 优化 | 通用缓存，需自定义预热脚本 | **预取工作流（定时/事件/主动）+ vLLM/SGLang 原生模型加载优化** |
| 数据抽象 | 绑定单一缓存实现 | **Dataset 与 Runtime 分离（Alluxio/JindoCache/JuiceFS 可切换）** |
| 多租户隔离 | 手动 Namespace/配额设计 | **Dataset 级逻辑隔离 + 跨 Namespace 共享 + K8s 原生访问控制** |
| 异构环境 | 需要额外集成工作 | **CSI + Sidecar 双模式注入** |

**💰 实际收益**：42 min → <30s 的模型加载时间差，将 serverless 推理从架构概念变为可运营的生产方案；共享基础模型免除重复缓存浪费。

> 🔗 https://www.cncf.io/blog/2026/05/21/how-netease-games-achieved-30-second-llm-cold-starts-on-kubernetes/
