### 1️⃣ 🛡️ Cilium CI/CD 供应链安全三部曲（Part 1 & 2）

**来源**: [CNCF Blog](https://www.cncf.io/blog/)
**项目**: Cilium（CNCF Graduated）
**作者**: André Martins & Feroz Salam（Isovalent at Cisco）

Cilium 团队发布了两篇深度文章，详细记录其 CI/CD 供应链安全实践。这是**已投产的、经过实战验证的**运维模式，不是理论框架。

#### Part 1：控制谁可以触发构建（June 4）

**问题**：CI 供应链攻击的第一步往往是**欺骗系统用高权限执行攻击者控制的代码**。

**Cilium 的四层防线**:

| 层 | 控制措施 | 具体实现 |
|:---|:---------|:---------|
| 🚪 **谁触发构建** | **Ariane 机器人 + 白名单** | 仅已验证的组织成员可通过 PR 评论触发 CI，且可触发的工作流按白名单枚举 |
| 📦 **CI 执行什么代码** | **双阶段 Checkout** | 第一阶段 checkout 基分支（可信代码：composite actions/签名逻辑）；第二阶段 checkout PR 头（仅作 Docker build context，**从不执行脚本**） |
| 👀 **谁审查 CI 变更** | **CODEOWNERS 严控** | `.github/` 下所有文件需安全 CI 团队审查；`auto-approve.yaml` 需 maintainer 审查 |
| 🔍 **工作流格式检查** | **静态分析** | CodeQL 强制每个 workflow 显式声明权限；actionlint 检查语法错误和不安全模式 |

**关键模式：双阶段 checkout 的安全边界**

```
Checkout 1: base branch (可信)
  ├─ 加载 composite actions
  ├─ 加载 Cosign 签名逻辑
  └─ 加载脚本
Checkout 2: PR head (不可信)
  └─ 仅作为 docker build context
```

安全报告经常标记此模式为漏洞，但 Cilium 团队解释：**`run:` 步骤全部内联写在 workflow YAML 中**，不从 PR branch 执行任何脚本；composite actions 也来自基分支。Docker BuildKit 在隔离环境中运行 PR 的 Dockerfile。

**⚠️ 运维教训**：Automated scanners 对 `pull_request_target + second checkout` 模式产生大量误报。理解设计意图比屏蔽告警更重要。

#### Part 2：锁定依赖（June 12）⭐ 新增

**问题**：控制了谁触发构建之后，下一步是**控制构建拉取了什么代码**。一个固定的工作流却拉取了被篡改的依赖，仍然是被攻破的工作流。

**六层依赖防护**:

| 层 | 措施 | 运维要点 |
|:---|:-----|:---------|
| 🔐 **GitHub Actions SHA Pin** | 所有 `uses:` 引用 40 位 commit SHA，版本号仅作注释 | `actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2` |
| 🐳 **容器镜像 Digest Pin** | 工作流步骤中用 `@sha256:` 引用容器镜像 | 内容寻址，标签被覆盖也不影响 |
| 🤖 **Renovate 自动化更新** | 自托管 Renovate bot，小时级调度，`pinDigests: true` | 使用专用 GitHub App（非 PAT），细粒度权限 |
| ⏳ **5 天 Cooldown** | 新版本发布 < 5 天不自动采纳 | 新版本被植入恶意代码后通常几天内即被发现并 yanked |
| 📦 **Go 模块 Vendoring** | 所有 Go 依赖 vendor 并提交到仓库 | 构建不接触外部 module proxy，信任决策从构建时移到审查时 |
| 🔬 **Workflow 静态分析** | CodeQL + actionlint + GitHub 表达式注入检测 | `${{ }}` 注入是最隐蔽的漏洞：bash 无法区分注入的命令和正常输入的值的边界 |

**Renovate 自动合并且安全验证**:

```yaml
# 白名单内的依赖自动合并
"automerge": true,
"groupName": "auto-merge-trusted-deps"
# 但自动审批工作流验证：
# 1. PR 确实来自 cilium-renovate[bot]
# 2. review request 确实由 bot 触发（而非人工冒充）
```

**"Fork vs Pin" 决策**:

| 策略 | 安全性 | 运维成本 | Cilium 的选择 |
|:-----|:-------|:---------|:--------------|
| Fork 所有依赖到自己的 org | 🟢 最高（上游被攻破也不影响） | 🔴 极高（维护数十分 fork 的同步） | ❌ 放弃 |
| SHA Pin（当前策略） | 🟡 中（transitive 依赖不可见） | 🟢 低（Renovate 自动更新） | ✅ 采用 |

**关键教训**：transitive 依赖是盲点。如果 `actions/checkout@[SHA]` 自身引用了另一个 action by tag（`uses: some-org/some-helper@v1`），解析在运行时进行且完全不可见。**GitHub 2026 Actions Security Roadmap 承诺的工作流级依赖锁定（类似 `go.mod + go.sum`）将解决此问题**。

#### 全套供应链安全控制矩阵（TL;DR）

```
Layer                     Control
─────────────────────────────────────────────────
谁触发构建                Ariane + org-member 白名单
执行什么代码              双阶段 checkout（可信/不可信分离）
谁审查 CI 变更            CODEOWNERS @cilium/github-sec
依赖拉取什么              SHA-pinned actions + images
Go 模块送什么进二进制      Vendored + @cilium/vendor 审查
工作流长什么样             CodeQL + actionlint 静态分析
可访问什么凭据             CI CI凭据仅可推 -ci 开发 registry
消费者怎么验证             Sigstore Cosign 签名 + SBOM
尚未解决的问题             无 SLSA provenance、无 PR-time dep review
```

### 2️⃣ 🏛️ CNCF IAM Whitepaper：身份边界的运维实践

**来源**: [CNCF Blog · 2026-06-04](https://www.cncf.io/blog/2026/06/04/identity-and-access-management-whitepaper/)
**作者**: CNCF TAG Security and Compliance

#### PEP/PDP 架构模式（运维视角）

策略执行点（PEP）和策略决策点（PDP）分离的架构模式是**云原生环境中企业级授权的最佳实践**：

```
请求 → [PEP] ──问策略──→ [PDP] ──→ 策略引擎
        │                     ↑
        │                     │
        └──执行决定───────────┘
        允许/拒绝/评估中
```

| 组件 | 职责 | 运维考虑 |
|:-----|:------|:---------|
| **PEP**（如 OPA Gatekeeper/Envoy ext_authz） | 拦截请求，询问 PDP 决策 | 需保证 PEP 的可用性——PDP 不可用时如何处理？ |
| **PDP**（如 OPA/SPIRE/OAuth2 Server） | 基于策略做出决策 | 需要 PDP 有独立的持久化层和高可用 |
| **策略存储** | 策略版本化管理 | GitOps + CI/CD 用于策略变更流水线 |

#### SPIFFE 运维要点

SPIFFE 不再是"未来概念"——Dapr 1.18、Istio、Spire 三个毕业级项目已将其作为基础设施组件落地。运维团队需关注：

- **SPIFFE 证书轮换** — 短期证书的生命周期管理
- **SPIFFE Federation** — 跨集群/跨组织的身份联合
- **Workload Attestation** — 如何证明一个工作负载确实就是它声称的

### 3️⃣ 🧪 OTel 安全实践在传统环境中的应用（May 19）

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/security-legacy-environments/)
**项目**: OpenTelemetry（CNCF Graduated）

传统环境（制造业、工控系统等）引入 OTel 时面临不同的**安全挑战**。文章覆盖的关键点：

| 挑战 | 建议 |
|:-----|:------|
| Collector 部署在隔离网络 | 使用 OTel Collector Gateway 架构，边界处做协议转换 |
| 遗留系统无原生 OTel 支持 | 使用 File Log Receiver/Syslog Receiver 桥接日志 |
| 有限的 TLS/加密支持 | OTel Collector 作为 TLS 终止层，转发加密数据到后端 |
| 工控协议的特殊性 | 使用 OTel Custom Exporter + 协议适配器 |
