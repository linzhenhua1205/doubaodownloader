## 🔥 重点信号

### 🔍 Inspektor Gadget 首次安全审计深度解读（June 3）

**来源**: [CNCF Blog](https://www.cncf.io/blog/2026/06/03/inspektor-gadget-results-from-the-first-security-audit/)
**项目**: Inspektor Gadget（eBPF-based K8s 可观测性与 Linux 主机检查工具包）
**审计方**: OSTIF（协调）+ Shielder（执行）

#### ⚠️ 为什么这很重要

任何以提升权限运行在共享基础设施上的工具都需要建立信任。Inspektor Gadget 以 **root 级权限** 在节点上运行，这是其功能的基础，也是安全审查的理由。过去十年 OSTIF 已在 120+ 开源项目中发现 800+ 漏洞。

#### 🔎 审计发现

**3 个 CVE（无 Critical/High）**:

| CVE | 严重度 | 问题描述 | 修复版本 |
|:----|:-------|:---------|:---------|
| CVE-2026-24905 | **Medium** | `ig image build` 命令注入 — 用户输入在 Makefile 中未正确转义，CI/CD 构建不可信 gadget 时风险最大 | v0.48.1 🔧 |
| — | **Medium** | eBPF ring buffer 事件淹没 — 硬编码 256KB，恶意容器可用噪声**隐藏自己的恶意活动**，直接影响安全监控流水线 | v0.50.1 🔧 |
| CVE-2026-25996 | **Low** | columns 输出模式 ANSI 转义未清理 — 受陷容器可向运维终端注入转义码 | v0.49.1 🔧 |

#### 🛡️ 6 条强化建议

| # | 建议 | 现状 | 优先级 |
|:-:|:-----|:-----|:------|
| 1 | TCP 监听器**默认启用 TLS**（当前仅日志警告，需显式 opt-out） | 讨论中 | 🔴 高 |
| 2 | CI/CD 中**pin 外部依赖 + 验证签名** | 部分已修复 | 🟡 中 |
| 3 | **K8s namespace blocklist** — 阻止追踪 kube-system 等敏感命名空间 | 规划中 | 🟡 中 |
| 4 | 限制远程客户端通过 Daemon 启用**主机级追踪**：文档化风险 | 讨论中 | 🟢 低 |
| 5 | 自动化**第三方依赖漏洞扫描** | PR 进行中 | 🟢 低 |
| 6 | 降低 DaemonSet 的 RBAC — 特别是 `nodes/proxy` GET 权限 | 长期项目 | 🟡 中 |

#### 🧪 Gadget Bypass 测试

研究人员识别了 **6 种绕过场景**——受陷容器可执行操作而不触发 gadget 事件：

| 绕过方式 | 原理 |
|:---------|:------|
| `openat2` 替代 `openat` | 某些 gadget 未 hook 较新的 Linux syscall |
| `io_uring` 异步 I/O | eBPF 对 io_uring 的覆盖仍不完整 |
| 静态链接库 | 绕过动态库 hook |
| seccomp 绕过 | 使用未过滤的系统调用 |
| 命名空间 escape | 利用 K8s Pod 中未限制的命名空间 |
| 定时规避 | 在 gadget 采样间隙执行 |

**核心教训**: eBPF 追踪不是万能的。Linux 持续演进，新的 syscall 和子系统不断出现，eBPF 工具需要持续跟上。Inspektor Gadget 已修复多个已识别缺口，并正在文档化其固有局限。

#### 📋 行动项

- **立即**: 升级至 v0.50.1（修复所有 3 个 CVE）
- **短期**: 关注 namespace blocklist 和 TLS 默认启用的进度
- **长期**: 理解 eBPF 工具的局限性，**不将 eBPF 追踪视为唯一安全屏障**

### 📜 K8s 项目修正历史 CVE 记录（May 26）

**来源**: [K8s Blog](https://kubernetes.io/blog/2026/05/26/reconciling-the-past-correcting-records-for-unfixed-kubernetes-cves/)
**作者**: Pushkar Joglekar (Broadcom / SIG Security), Tabitha Sable (Datadog / K8s Security Response Committee)

K8s 项目进行了一项重要的**历史数据质量治理**工作——审核并修正了 CVE 数据库中一些未修复 K8s 漏洞的记录。

**背景**:
- K8s 历来通过发布 CVE 记录到公共数据库来保持透明度
- 但发现某些 CVE 记录**不准确**——被标记为"未修复"的实际上已有缓解措施
- 这些问题可能来自记录错误或审核流程变化

**修正内容**:
1. 审查所有标记为"unfixed"的 K8s CVE 记录
2. 区分"未被上游修复"和"已有缓解措施/工作区可减轻影响"
3. 更新 CVE 记录以反映更准确的修复状态

**Ops 影响**:
- 对安全运维团队意味着 CVE 数据库的可靠性提升
- 使用自动化 CVE 扫描工具时，减少误报（false positive）——这是修复状态不准确的最大代价
- 体现了 K8s 安全团队对**数据治理**的认真态度

**更广泛的含义**: 当 CVE 数据库成为自动化供应链安全工具的输入时，CVE 记录的准确性直接影响安全决策质量。K8s 项目这次修正为其他 CNCF 项目树立了一个**透明度和数据质量**的榜样。
