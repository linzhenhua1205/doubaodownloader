## 1️⃣ OpenTelemetry 正式毕业为 CNCF 毕业项目 🏆

**来源**: OpenTelemetry Blog + CNCF · 2026-05-21
**标签**: `OpenTelemetry` `CNCF毕业` `可观测性` `OTel`

### 核心内容
CNCF 官方宣布 **OpenTelemetry 正式成为毕业项目 (Graduated Project)**，这是可观测性领域最重要的里程碑之一。

### 毕业指标
| 维度 | 数据 |
|:-----|:------|
| **社区规模** | 500+ 组织贡献者，遍及全球 |
| **代码仓库** | 30+ 语言 SDK、Collector、Operator 等核心组件 |
| **采用率** | 90%+ 的 CNCF 调查受访者报告使用 OTel |
| **SIG/Committee** | 9个 SIG + Governance Committee + Technical Committee |
| **认证** | OTCA (OpenTelemetry Certified Associate) 考试推出 |
| **市场覆盖** | 几乎所有主流可观测性厂商提供原生 OTel 集成 |

### 毕业意义
- OTel 已成为**云原生可观测性的事实标准**，从 Tracing 扩展到 Metrics、Logs、Profiles
- Collector 生态持续扩展，已成为可观测性数据管道的核心枢纽
- **OpenTracing 兼容性已正式弃用**（PR #4938, 2026-03-19 合并），标志 OpenTracing 时代彻底结束
- OTel 的毕业意味着可观测性领域完成了从百家争鸣到**统一标准**的过渡

### 近期补充动作

- **Ecosystem Explorer Project**（4/30）：探索 Java 240+ 自动插桩 + Collector 数百组件的生态地图
- **GenAI Observability**（5/14）：Inside the LLM Call — 追踪 LLM 调用链（模型调用→工具调用→Token交换）

## 2️⃣ Jaeger v2 演进：基于 OTel 的 AI Agent 分布式追踪 🤖🔍

**来源**: CNCF Blog (Project Maintainer Post) · 2026-05-26 · Jonah Kowall, Jaeger Maintainer
**标签**: `Jaeger` `AI Agent` `MCP` `ACP` `AG-UI` `GenAI可观测性`

### 核心演进路径
Jaeger 正在进行两阶段转型：
1. **Jaeger v2**：已发布，内核替换为 **OpenTelemetry Collector** 框架，原生 OTLP 接入，统一 Metrics/Logs/Traces
2. **AI Agent 可观测性**：采用三大开放协议 — **MCP + ACP + AG-UI**

### 三大协议含义

| 协议 | 全称 | 作用 |
|:-----|:------|:------|
| **MCP** | Model Context Protocol | 标准化 AI 模型安全访问外部数据源 |
| **ACP** | Agent Client Protocol | 统一 UI 与 AI Agent/侧车通信方式 |
| **AG-UI** | Agent-User Interaction Protocol | 人类与 Agent 协作交互协议 |

### 架构设计
```
Jaeger UI ──AG-UI协议──→ Agent Gateway (ACP翻译层) ──ACP──→ AI Agent (LLM/SLM)
                                    │
                                    └──MCP──→ Jaeger MCP (查询生成)
```
- **后端 ACP 层**：无状态翻译器，将自然语言约束（如"找出支付服务中延迟>2s的500错误"）转换为确定性 Trace 查询
- **模型灵活配置**：可用云端 LLM（复杂推理）或本地 SLM（数据隐私严格场景）
- **限制 AI 到协议翻译+查询生成**范围，最小化幻觉风险

### GenAI 执行路径可视化
- Jaeger issue #8401 (GenAI integration)：可视化 OTel GenAI 语义约定
- 追踪 RAG 管道的嵌入模型延迟、外部工具调用、Token 使用量
- 遵循 OTel 社区正在推进的 **GenAI Agentic Systems 语义约定** (Issue #2664) 和 **AI Sandbox 约定** (Issue #3583)

### 统一二进制
- Jaeger v2 基于 OTel Collector → 本地测试和生产运行同一二进制
- 本地测试用 SLM 私密沙箱，生产切换为云端 LLM

### 趋势判断
**Jaeger 正在从"追踪微服务"演进到"追踪 AI Agent"**，这是可观测性领域对标 AI 工作负载的最重要适应性改造。

## 3️⃣ KEDA GPU 自动缩放器：基于 GPU 指标的原生弹性伸缩 📈🎮

**来源**: CNCF Blog · 2026-05-27 · Pavan Madduri (Grainger)
**标签**: `KEDA` `GPU Autoscaling` `NVML` `vLLM` `GreenOps`

### 核心问题
KEDA 内置缩放器（Prometheus/Kafka 等）不支持 GPU 指标，因为：
1. KEDA 使用 `CGO_ENABLED=0` 编译，而 **NVML (NVIDIA Management Library)** 需要 CGO
2. KEDA operator 是单 Deployment，无法读取其他节点的本地 GPU 指标

### 架构方案：DaemonSet + ExternalScaler
```
┌─────────────────┐      gRPC ExternalScaler      ┌──────────────────┐
│ keda-gpu-scaler  │◄──────────────────────────────┤   KEDA Operator  │
│ (DaemonSet)      │   IsActive/GetMetricSpec/     │                  │
│   ├─ NVML 调用   │      GetMetrics               └──────┬───────────┘
│   └─ gRPC Server │                                      │ HPA
└─────────────────┘                                ┌──────┴───────────┐
  每节点一个 Pod                                    │  vLLM Deployment │
                                                   └──────────────────┘
```

### 暴露的 GPU 指标
| 指标 | 说明 | 缩放依据 |
|:-----|:------|:---------|
| `gpu_utilization` | SM 计算利用率 % | Triton 推理 |
| `memory_utilization` | 内存控制器利用率 % | — |
| `memory_used_percent` | VRAM 使用率 % | vLLM 推理 (scale-to-zero) |
| `temperature` | 芯片温度 °C | 过温保护 |
| `power_draw` | 功率 W | GreenOps 节能 |

### 预置 Profile（开箱即用）

| Profile | 指标 | Target | Activation | 场景 |
|:--------|:----|:-------|:-----------|:-----|
| `vllm-inference` | memory_used_percent | 80% | 5% | LLM 推理，scale-to-zero |
| `triton-inference` | gpu_utilization | 75% | 10% | Triton 模型服务 |
| `training` | gpu_utilization | 90% | 0% | 训练，不放缩到零 |
| `batch` | memory_used_percent | 70% | 1% | 批量推理，激进缩容 |

### 亮点
- **scale-to-zero 支持**：通过 `activationThreshold` 实现 GPU 空闲时缩零
- **多 GPU 聚合**：max / min / avg / sum 四种聚合方式
- **Mock Collector 模式**：无需 GPU 硬件即可在 CI 中跑 11 个 E2E 测试
- **GreenOps 视角**：浪费的 GPU 周期 = 浪费的能源 = Scope 3 排放

## 4️⃣ NetEase Games：Fluid + Alluxio 实现 K8s 上 LLM 推理 30 秒冷启动 🚀

**来源**: CNCF Blog (End User Post) · 2026-05-21 · Haifeng Liao, Xiang Zhang (NetEase Games)
**标签**: `LLM冷启动` `Fluid` `Alluxio` `K8s推理` `Serverless GPU`

### 核心挑战
NetEase 的 AI 平台 Tmax 在 K8s 上运行 LLM 推理服务，面临三个耦合问题：
1. **GPU 资源碎片化**：异构 GPU 种类多，不同 workload 需要不同卡型
2. **流量不均匀**：游戏流量随时间段波动，静态预留效率低
3. **冷启动被模型加载主导**：70B 模型几百 GB 权重下载，消除弹性伸缩价值

### 方案：Fluid (CNCF Incubating) 数据编排层

| 维度 | Alluxio 直接部署 | Fluid + Alluxio |
|:-----|:---------------|:-----------------|
| K8s 集成度 | 需手动管理 Master/Worker 集群 | Fluid 自动部署和管理 Runtime 生命周期 |
| 预加载 | 需自定义预热逻辑 | **Prefetch Workflow**：定时/事件驱动/主动预热 |
| 数据抽象 | 绑定具体缓存实现 | **Dataset 抽象**与 Runtime 解耦，可切换 Alluxio/JindoCache/JuiceFS |
| 多租户隔离 | 手动设计 Namespace/Quota | **跨 Namespace 共享**，基础模型单次预热多服务复用 |
| 环境适配 | 需额外集成 | CSI + Sidecar 双模式，Webhook 注入无需改应用 |

### 加载时间演进
```
跨区直连存储: 42 分钟
    ↓ 传统缓存层 (Alluxio)
传统缓存:    14 分钟
    ↓ Fluid Prefetch 工作流
Fluid 预热:  3 分钟
    ↓ 生产进一步调优
生产优化:    ~30 秒 🎯
```

### 关键发现
- 跨 Namespace 模型共享减少了大量缓存内存浪费
- 分布式缓存吸收了启动突发流量，避免压垮后端存储
- 预取(prefetch) + 自动缩放 + 缓存缩放能力将"弹性推理"从概念变为可操作

## 5️⃣ CNCF 圆桌共识：云原生 = AI Native — 生产就绪 AI 三要素 🏗️

**来源**: CNCF Blog (Staff Post) · 2026-06-02 · KubeCon EU 圆桌 (AWS/Google/Microsoft/solo.io)
**标签**: `AI Native` `生产就绪AI` `K8s AI Conformance` `Agentic SRE`

### 核心三要素
1. **平台成熟度** — K8s AI Conformance 认证，确保跨环境互操作性
2. **安全设计** — Agentic 工作流的内置安全，特别是 Prompt Injection 防护和模型供应链完整性
3. **社区贡献** — 从消费工具到主动参与 CNCF SIG 驱动创新

### K8s 适配 AI 的三项关键工作
- **Pod Groups (Workload API)**：将 Pod 组视为单一故障域，保障大规模 AI 矩阵初始化的亲和性
- **Dynamic Resource Allocation (DRA)**：将专用芯片/GPU 集成到调度器，管理硬件 nuances
- **Inference Gateways**：利用 Gateway API 构建 AI Gateway，管理 Prompt 和生成模型响应

### Agentic SRE 方向
审核瓶颈正在成为新问题——AI 生成大量代码需要人工 Review。未来方向是 **Agentic SRE**：AI Agent 辅助根因分析和修复(remdiation)，但人工保留关键决策权。
