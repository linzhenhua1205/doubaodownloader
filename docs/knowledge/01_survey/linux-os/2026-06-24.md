## 🐧 Linux 内核

### 🆕 OSPM 2026 第二天报告（6月24日发布）

[LWN] 剑桥举办的 OSPM 2026（Power Management and Scheduling in the Linux Kernel Summit）第二天议题覆盖：

- **设备频率缩放（Device Frequency Scaling）** — 跨设备协同频率控制策略
- **时间片时长用于 CPU 选择** — 将时间片 Duration 作为 CPU 选择调度的信号，优化工作负载匹配
- **多集群 Arm 系统调度域** — 异构 big.LITTLE / DSU 集群架构下调度域设计
- **LAVD 调度器** — 新的 Linux 调度器探索方案（Latency-Aware Virtual Deadline）
- **idle-state 选择** — 延续第一天的 idle 状态选择深入讨论
- **lock-holder preemption** — 持有锁的线程被抢占的处理策略

**影响**: sched_ext + 用户态调度器持续成为热话题，传统内核调度演进的步伐在加速。Arm 多集群场景下调度域设计对 AI 推理节点的异构计算至关重要。

### 🆕 KASAN for JIT-compiled BPF（6月23日）

Alexis Lothoré 在 LSF/MM/BPF 2026 上报告了将内核 KASAN（Kernel AddressSanitizer）支持扩展到 JIT 编译的 BPF 代码的工作：

- **目标**: 捕获 BPF JIT 编译器的内存管理 bug
- **现状**: KASAN 只监控内核自身代码，JIT 生成的 BPF 机器码不可见
- **挑战**: JIT 编译的 BPF 代码在运行时动态生成，需要特殊的插桩
- **进展**: 原型已可在 BPF JIT 路径上检测出内存越界/use-after-free

**影响**: 随着 eBPF 在网络安全（Cloudflare bpf-lsm, Datadog Workload Protection）和可观测性中广泛部署，JIT 编译器的正确性变得至关重要。

### BPF 协程化 — 暂停/恢复 BPF 程序（6月19日）

Kumar Kartikeya Dwivedi 在 LSF/MM/BPF 2026 上的工作持续被讨论：

- **核心思想**: BPF 程序作为 coroutine，可以暂停执行并在后续恢复
- **突破**: 打破"BPF 必须一次执行完"的限制（currently BPF programs must run to completion）
- **应用场景**: 长时运行 BPF 任务（如周期性监控、遍历型操作）
- **状态**: 仍为实验性，但社区反响积极（9 条评论）

## 🐧 eBPF 应用进展

### 🆕 LinkedIn: eBPF off-CPU profiling 定位内核锁竞争（5月27日）

LinkedIn 工程师遇到反复出现的短时服务中断——数据库不可用后自动恢复，无有效日志痕迹。通过 **eBPF off-CPU profiling**：

- **根因**: 内核中特定锁的竞争导致关键路径阻塞
- **方法**: 利用 eBPF 跟踪线程被调度出 CPU 时的内核栈
- **价值**: 传统方法无法捕获此类间歇性锁竞争，eBPF 提供了"进程被阻塞在哪里"的直接可见性

### Lambda: eBPF 将 125,000 条 iptables 规则压缩到 144 条（4月22日）

Lambda 网络团队采用 eBPF 实现网络隔离和快照：

- **iptables 规则膨胀**: SaaS 多租户场景下，iptables 规则膨胀到 125,000 条
- **eBPF 替换**: 基于 eBPF 的包过滤实现，仅需 144 条程序
- **快照网络 20× 加速**: 快照创建/恢复的网络吞吐提升 20 倍
- **价值**: 规则数量减少 99.9% 的同时性能大幅提升

### GitHub: eBPF 改进部署安全性（4月16日）

GitHub 使用 eBPF 检测和防止部署工具中的循环依赖：

- **场景**: 多服务部署时的依赖图检测
- **方法**: eBPF 实时追踪依赖调用关系
- **效果**: 在部署前发现循环依赖，避免级联故障

## ☸️ Kubernetes & CNCF 生态

### 🆕 Jaeger v2.18: ClickHouse 后端 Alpha（6月23日）

CNCF Jaeger 项目发布 v2.18.0，首次原生支持 ClickHouse 作为存储后端：

| 指标 | 数值 |
|:-----|:-----|
| 压缩比（spans 表） | **8.6×** — 10M spans 从 ~6 GiB → ~722 MiB |
| 写入吞吐 | **>50k spans/sec**（单节点） |
| Trace 检索延迟 | **~100 ms** |
| 搜索查询延迟 | **<50 ms**（多数），复杂过滤 ~140 ms |
| 单节点规模 | 10M spans / 1M traces |

**架构设计要点**:
- 主键选择 `(service_name, name, start_time)` 而非 `trace_id` → 搜索性能提升 6×
- Bloom filter skip index 补偿 trace_id 查询性能
- 物化视图预计算服务名/操作名/trace 时间范围
- 5 层属性存储（resource/scope/span/event/link）+ `attribute_metadata` 表

**OpenTelemetry 数据模型适配**: Jaeger v2 采用 OTel 数据模型，需重新设计 schema
