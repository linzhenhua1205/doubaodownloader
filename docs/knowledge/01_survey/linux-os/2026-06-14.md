## ⭐ 核心发现

### ① 🔥 Linux 7.1 稳定版今日发布在即 + 7.2 特性预告（含 Cache Aware Scheduling）

**来源**: Phoronix · Michael Larabel · 2026-06-12~14

#### Linux 7.1 稳定版
- Phoronix 预计 **Linux 7.1 稳定版于 6 月 14 日（周日）发布**
- 截至北京时间 22:30，kernel.org 仍显示 7.1-rc7，但预计数小时内完成 tagging
- **7.1 核心特性**：
  - **FRED（Flexible Return and Event Delivery）** — Intel 提出的新异常/中断处理机制，替代 IDT 门描述符方案，大幅简化内核异常分发路径
  - **新 NTFS 内核驱动** — 替代旧的 ntfs-3g FUSE 方案，原生内核支持 NTFS 读写
  - 性能优化：调度器、内存管理持续迭代

#### Linux 7.2 特性预告（merge window 即将开启）
- **Cache Aware Scheduling（缓存感知调度）** — 对 K8s CPU Manager 和拓扑感知调度有直接影响，可感知 LLC 缓存拓扑做更精确的调度决策，提升多容器场景的缓存命中率
- **Apple M3 支持** — 更多 Apple Silicon 硬件使能
- **AMDGPU HDMI 2.1 FRL（Fixed Rate Link）** — 高带宽显示支持
- **USB4STREAM** — USB4 流传输能力改进
- **更多性能基准数据**预计在 7.2 merge window 期间公布

> **对 K8s 影响**: Cache Aware Scheduling 可与 K8s CPU Manager Static Policy + Topology Manager 协同，在 Pod 级别优化 L2/LLC 缓存亲和性

### ② 🔥 AMD openSIL + Coreboot 在 EPYC 服务器上成功运行——开源固件里程碑

**来源**: Phoronix · Michael Larabel · 2026-06-12 · 深度评测

#### 背景
- AMD 于 2023 年初宣布 **openSIL（open Silicon Initialization Library）**，旨在以开源方式替代闭源 AGESA（AMD Generic Encapsulated Software Architecture）
- 此前 openSIL 仅针对参考平台，预期在 **Zen 6 时期达到生产状态**
- **3mdeb** 团队将 openSIL + Coreboot 移植到 **Gigabyte 服务器主板**，在 **Zen 5 (EPYC 9005 系列)** 上实际验证成功

#### 技术要点
- openSIL 提供 CPU 硅初始化代码（内存训练、PCIe 枚举、核心启动等），以开源形式替代 AGESA 的闭源二进制 blob
- Coreboot 作为轻量级固件框架，结合 openSIL 形成**完整的开源服务器固件栈**
- 评测验证：系统能成功启动、运行 Linux、通过基本稳定性测试

#### 对服务器安全的意义
- **供应链安全**：服务器 BIOS/固件是攻击面中最大但也最不透明的环节之一，openSIL 开源后**固件代码可被第三方审计**
- **K8s 机密计算**：在 confidential computing（CoCo VM）场景下，开源固件栈提供从 TCB（可信计算基）到 Linux 内核的完整可审计链路
- **对国产化替代的参考**：AMD openSIL 模式为服务器主板开源固件提供了可复用的工程范式

> **里程碑意义**: 这是首次在 EPYC 服务器主板上实现 openSIL + Coreboot 全开源固件栈，是服务器透明化/安全化的分水岭事件

### ③ 🔥 Arch Linux AUR 第二波恶意软件攻击——更复杂、更隐蔽

**来源**: Phoronix · Michael Larabel · 2026-06-14

#### 事件演进
- **第一波（6月12日）**：攻击者攻陷 400+ AUR orphaned 包，注入恶意 npm 包 `atomic-lockfile` 窃取敏感数据
- **最终统计**：第一波清理后发现 **超过 1,500 个受影响包**
- **第二波（6月14日）**：新一波攻击，**更复杂、更隐蔽**
  - 使用**代码混淆（code obfuscation）** 隐藏恶意载荷的真实意图
  - 攻击手法升级，绕过第一波后部署的检测机制

#### 容器安全警示
- AUR 包在 Docker/container 构建过程中常被用作 base image 的软件源
- 在 CI/CD 流水线中，若镜像构建依赖 AUR 包，**一次成功的供应链攻击即可渗透整个容器镜像供应链**
- **启示**：
  - 容器镜像构建应**锁定包源版本（SHA pin）**，避免使用「latest」标签
  - CI/CD 流水线应集成 SBOM（软件物料清单）校验
  - 对第三方包注册表来源进行依赖冷却（dependency cooldown）和人工审查

### ④ Fedora 45 考虑精简 GRUB 用于机密计算

**来源**: Phoronix · Michael Larabel · 2026-06-13

#### 提案内容
- Fedora 45 正在评估引入**精简版 GRUB UEFI 引导加载程序**，专为**机密计算（Confidential Computing, CoCo）** 场景设计
- 主要动机：标准 GRUB 包含大量驱动和功能模块，对 CoCo VM 的可信计算基（TCB）来说过于臃肿
- 精简版 GRUB 只保留最必要的启动功能，将攻击面降至最低

#### 对 K8s 的意义
- K8s 的 **Confidential Containers（CoCo）** 项目依赖宿主机固件提供可信执行环境
- 精简 GRUB 可缩小 CoCo VM 的 TCB，减少潜在侧信道攻击面
- 这是「云原生安全从内核延伸到固件」的又一信号

> **趋势**: K8s 安全工作正从容器层→内核层→固件层逐层下沉，CoCo GRUB 是这一趋势的最新体现

### ⑤ Intel Thermald 2.5.12 发布——初步支持 ARM 架构

**来源**: Phoronix · Michael Larabel · 2026-06-13

- Intel 的 Linux 热管理守护进程在 2.5.12 版本中**引入了对 ARM 架构的初步支持**
- 此前 Thermald 仅支持 Intel x86 处理器，扩展 ARM 支持意味着 Intel 正在将其热管理方案跨架构泛化
- **对服务器的影响**：在 Arm Neoverse 服务器（AWS Graviton、Ampere、NVIDIA Grace）和 DPU（BlueField）场景下，Thermald 可作为跨架构统一热管理工具使用

### ⑥ Intel 终止 BigDL 开源 AI 项目

**来源**: Phoronix · Michael Larabel · 2026-06-13

#### 背景
- BigDL 是 Intel 开源的 AI/LLM 项目，旨在跨 Intel XPU（CPU/GPU/NPU）以低延迟方式运行 LLM
- 覆盖从 Core Ultra 笔记本到数据中心硬件的全场景
- 是 Intel 减少维护的开源项目之一

> **信号**: 在 Intel Foundry 重心转移和 AI 芯片市场格局变化下，Intel 正持续收缩非核心开源项目，开发者需关注长期依赖风险
