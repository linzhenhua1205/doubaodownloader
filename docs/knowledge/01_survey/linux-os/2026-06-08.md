## 2️⃣ ⚠️ splice()/vmsplice() 因 LLM 发现漏洞洪流面临完全移除

**来源**: [LWN.net](https://lwn.net/Articles/989172/) · Jonathan Corbet · 2026-06-04
**讨论**: 34 条评论

### 背景

`splice()` 和 `vmsplice()` 系统调用通过**最小化（或完全避免）系统调用和数据复制**来提高特定数据移动任务的性能。但它们也有**长久的安全问题历史**。

### 触发事件

最近 LLM（大语言模型）发现的漏洞洪流（LLM-discovered vulnerabilities）再次将 splice() 和 vmsplice() 推上风口浪尖。这些 AI 生成的漏洞报告集中在：
- 竞态条件（race conditions）
- 数据竞争（data races）
- 内存安全边界违规

### 技术影响

如果 splice()/vmsplice() 最终被移除，将影响：

| 影响范围 | 说明 |
|:---------|:-----|
| **高性能网络** | splice() 广泛用于零拷贝数据传输（如 Web 服务器、反向代理） |
| **容器运行时** | 部分容器 I/O 路径依赖 splice 机制 |
| **pipe 操作** | vmsplice() 是用户空间向内核 pipe 批量注入页面的唯一系统调用 |
| **现有应用** | 大量使用 splice() 的生产代码需要迁移 |

### 讨论焦点

- 社区分化为两派：**安全优先派**（完全移除）vs **兼容派**（增加安全限制后保留）
- 核心问题：splice() 提供的**零拷贝性能收益**是否值得持续的 **CVE 补丁维护成本**？
- 统计数据显示：近 12 个月 splice/vmsplice 相关的 CVE 数量呈**指数级增长**，与 LLM 辅助代码扫描工具的大规模使用直接相关

### 运维影响

如果 splice() 被移除，Nginx、HAProxy、Redis、以及 Kubernetes 网络数据路径中的零拷贝优化将需要替代方案。生产环境中应**提前审计**是否依赖 splice() 进行数据传输。

## 3️⃣ 🏗️ spawn templates：超越 fork()+exec() 的新进程创建原语

**来源**: [LWN.net](https://lwn.net/) · Jonathan Corbet · 2026-06-05
**讨论**: 64 条评论（本周最高讨论热度）

### 背景

自 Unix 诞生以来，`fork()`（创建子进程作为父进程的副本）+ `exec()`（替换当前进程为新程序）一直是进程创建的核心模型。在 Linux 中对应 `clone()` + `execve()`。

虽然模型优雅，但有**固有缺陷**：

| 缺陷 | 说明 |
|:-----|:------|
| **COW 页面故障** | fork() 后子进程触发大量写时复制页面故障 |
| **地址空间浪费** | 复制整个地址空间只为了立即 exec 替换 |
| **安全风险** | fork() 临时暴露父进程的敏感数据（如密钥） |
| **性能开销** | 大进程（～GB 级内存）的 fork() 延迟显著 |

### Li Chen 的 spawn templates 提案

Li Chen 提出的方案是在内核中增加 **「spawn templates」** 机制——一种新的进程创建原语，允许：

- **预配置**进程模板（包括安全上下文、cgroup、namespace 等）
- **从模板直接派生**新进程，绕过 fork() 的地址空间复制
- 与 execve() 结合实现**更高效、更安全**的进程创建

### 当前状态

- **不会以当前形式被合入**主线内核
- 但讨论指出了**未来的可能方向**——内核社区正在认真考虑是否需要一个新的进程创建 API
- 64 条评论反映了社区对此问题的高度关注

### 与容器运行时的关系

`fork()` + `exec()` 的性能开销在**容器启动场景**中被显著放大——每个容器的初始进程都需要完整的 fork→exec 链。spawn templates 如果最终实现，将直接改善：
- 容器冷启动延迟（Docker/containerd/CRI-O）
- 短生命周期函数计算的启动效率（Knative/OpenFaaS）
- 沙箱/安全容器（gVisor/Kata Containers）的创建速度

## 4️⃣ 🔬 BPF in the Agentic Era：Alexei Starovoitov 的「觉醒之吼」

**来源**: [LWN.net](https://lwn.net/) · daroc · 2026-06-03
**讨论**: 22 条评论

### 核心内容

在 2026 年 Linux 存储、文件系统、内存管理和 BPF 峰会（LSFMM+BPF）的 BPF 专场上，BPF 维护者 **Alexei Starovoitov** 进行了一场被他自称为「与其说是演讲，不如说是觉醒之吼」（scream of realization）的报告。

### 核心论点

**BPF 必须改变，以避免被 LLM 和 Coding Agent 的浪潮冲垮。**

具体问题：

| 问题 | 描述 |
|:-----|:------|
| **LLM 生成 BPF 代码的质量** | AI 工具生成的 BPF 程序大量存在安全和正确性问题 |
| **bpftrace 的 LLM 滥用** | Coding Agent 使用 bpftrace 进行诊断时产生大量噪音 |
| **补丁审阅洪流** | BPF 子系统面临大量低质量补丁，维护者不堪重负 |
| **BPF 编译工具链的自动化风险** | LLM 生成的 BPF 字节码可能绕过安全检查 |

### 后续讨论

在跟进讨论中，参与者覆盖了：
- Coding Agent 如何使用 bpftrace 等工具的问题
- BPF 子系统中当前需要审查的补丁洪流
- 是否需要为 BPF 代码引入**自动化质量门禁**（类似 AI 生成代码的沙盒测试）

### 运维影响

BPF 是云原生基础设施的**核心技术**——Cilium、Falco、Inspektor Gadget、以及 K8s 安全策略都深度依赖 BPF/eBPF。如果 BPF 因为 LLM 生成的低质量代码而面临「信任危机」，将间接影响整个云原生安全生态的发展节奏。

## 5️⃣ 📝 BTF 函数签名追踪：优化编译器与内核追踪的摩擦

**来源**: [LWN.net](https://lwn.net/) · daroc · 2026-06-01

### 技术问题

优化编译器在某些情况下会推断函数参数不需要并移除它。对于常规执行这没问题，但当内核的追踪（tracing）或 BPF 子系统需要知道**如何调用该函数或其参数存储位置**时，就出问题了。

### 解决方案

Alan Maguire 和 Yonghong Song 在 LSFMM+BPF 2026 上介绍了他们在 BTF（BPF Type Format）调试信息中记录**变化后的函数签名**信息的工作：

- 当编译器优化移除未使用参数时，BTF 记录**原始签名 vs 优化后签名**的映射
- `bpftrace`、`perf` 等工具可以利用这些信息正确解析函数参数
- 对内核可观测性的**准确性**至关重要

### 应用场景

- AI Agent 使用 bpftrace 诊断时，需要**正确的函数签名**来追踪参数值
- 内核开发人员调试时，频繁遇到「参数不存在」的问题
- 这是内核可观测性基础设施的**基础性改进**

## 6️⃣ 🔄 x32 ABI 第三次面临移除

**来源**: [LWN.net](https://lwn.net/) · Jonathan Corbet · 2026-06-01
**讨论**: 30 条评论

### 背景

x32 ABI 被设计为**两全其美**——提供 x86-64 的扩展寄存器和指令集，同时保留 32 位系统的低内存使用。Linux 内核自 3.4（2012）起支持 x32。

### 现状

- **初始热情没有持续**——x32 几乎没有实际使用者
- 内核开发者**再次**考虑移除 x32 支持——这已经是第三次了
- 困难在于：**即使是最不被喜爱的特性，也总有几个用户**，使得移除变得困难

### 技术原因

- x32 需要在内核中维护**独立的系统调用表**和兼容层
- 新增功能的兼容性测试增量成本持续累积
- 使用场景极其有限（主要是一些内存受限但需要 64 位能力的环境）

## 7️⃣ 📦 Ruby Bundler 4.0.13 引入依赖冷却功能对抗供应链攻击

**来源**: [LWN.net](https://lwn.net/) · jzb · 2026-06-05

### 技术详情

Ruby 的包管理器 **Bundler 4.0.13** 新增了 **dependency cooldown**（依赖冷却）功能：

> 「大多数针对 RubyGems 的供应链攻击利用的是一个狭窄的时间窗口：账户被入侵 → 恶意版本发布 → 之后的 `bundle install` 会直接解析到它。Bundler 4.0.13 引入了 cooldown——一个基于时间的过滤器，拒绝解析到发布未满 N 天的版本。」

### 关键特性

| 特性 | 说明 |
|:-----|:------|
| **时间窗口过滤** | 拒绝解析到发布未满 N 天的版本，给社区留出审查时间 |
| **选择性启用** | opt-in，不强制 |
| **补充现有机制** | 与强制 2FA 和 trusted publishing 互补而非替代 |
| **跨生态参考** | 设计参考了其他语言生态如何解决同样问题 |

### 运维影响

这是**包管理供应链安全**领域的一个行业趋势信号——从「事后检测」转向「事前冷却」防御。Bundler 的 cooldown 模式可能被其他生态（npm、PyPI、cargo）跟进。
