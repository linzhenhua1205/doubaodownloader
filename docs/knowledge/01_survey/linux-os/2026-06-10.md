### ① BPF 循环验证突破：标量演化（SCEV）分析 — 解决嵌套循环验证上限问题

**来源**: [LWN.net](https://lwn.net/) · daroc · 2026-06-09 · LSFMM+BPF Summit 报道

**Eduard Zingerman** 在 LSFMM+BPF 峰会上汇报了一项关键进展：将**编译器标量演化（Scalar Evolution, SCEV）技术**引入 BPF 验证器。

- **背景问题**：BPF 验证器对循环的基本策略是逐次迭代展开，遇到嵌套循环时易触及 `BPF_MAXINSNS` 指令数上限，导致有效 BPF 程序被误拒
- **SCEV 方案**：从 LLVM/GC C 编译器中借鉴 SCEV 分析技术，分析循环归纳变量（induction variable）的演化模式，推导循环边界和内存访问模式，**无需逐次迭代展开**
- **直接解决**：嵌套循环触及 `BPF_MAXINSNS` 上限的问题
- **当前状态**：进行中的工作，需解决 SCEV 分析本身的 BPF 验证问题
- **K8s/容器关联**：Cilium 的 eBPF 网络策略、Falco/Inspektor Gadget 的 eBPF 安全监控均使用复杂嵌套循环 BPF 程序，**验证器改进直接影响生产可用 BPF 程序的复杂度**

### ② Trusted Publishing：消除长期凭证——供应链安全新范式

**来源**: [LWN.net](https://lwn.net/) · jzb · 2026-06-09 · Open Source Summit North America 2026

**Mike Fiedler（PyPI 安全团队）** 发表演讲，阐述了 trusted publishing（可信发布）机制的完整设计哲学与实践路径。

- **核心机制**：基于**短期凭证**替代长期 API token/密码，用于向包注册表（PyPI、npm、RubyGems 等）发布
- **工作原理**：包注册表与 OIDC 身份提供商（GitHub Actions、GitLab CI、Google Cloud Build 等）建立信任关系 → CI/CD 流水线在每次发布时获取一次性短期凭证
- **安全收益**：即使 CI/CD 平台被攻破，攻击者拿到的也只是**短期凭证**，无法长期控制发布权限
- **已知采纳**：PyPI 已强制推行；RubyGems 在 OIDC 支持中；npm 正在实验阶段
- **不是银弹**：不能防御注册表自身漏洞、依赖混淆、typosquatting，但有效防御凭证泄露型供应链攻击
- **K8s 关联**：CNCF 背景下的 Helm Chart 发布、OCI artifact 发布均可参考此模式；K8s 自身的 Image Promoter 和 Release Engineering 考虑类似 OIDC 集成
