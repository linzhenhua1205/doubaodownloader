### 1️⃣ K8s v1.36 发布 — 22 篇博客，多特性 GA/Beta

- **来源**: Kubernetes Blog (2026-05-07 ~ 2026-05-15)
- **版本**: Kubernetes v1.36 "ハル (Haru)"
- **核心 GA 特性**:
  - **Users Namespaces for Pods GA** — 历经多个版本，Pod 容器默认以非 root 用户运行，彻底阻断容器逃逸
  - **PSI Metrics for K8s GA** — 基于内核 PSI（Pressure Stall Information）实现集群级资源饱和度检测，2018 年 Linux 内核引入，历经 v1.34 Beta → v1.36 GA
  - **Volume Group Snapshots GA** — 支持跨卷的原子级快照（从 v1.27 Alpha → v1.32 Beta → 最终 GA）
  - **Declarative Validation GA** — ValidatingAdmissionPolicy 进入 GA
  - **Fine-Grained Kubelet API Authorization GA**
  - **SELinux Volume Label Changes GA** — 影响 v1.37 对使用 seLinuxOptions 的现有配置
- **核心 Beta/Alpha 特性**:
  - **Pod-Level In-Place Vertical Scaling (Beta)** — 支持 Pod 层面直接增删 Container Resource spec，不重启 Pod
  - **Mixed Version Proxy (Beta)** — 混版本代理让升级更安全
  - **Workload-Aware Scheduling (v1alpha2)** — Workload/PodGroup API 原子化调度 + 拓扑感知
  - **Pod-Level Resource Managers (Alpha)** — 每个 Pod 独立资源管理器
  - **Tiered Memory Protection (Memory QoS)** — 层级化内存保护
  - **Mutable Pod Resources for Suspended Jobs (Beta)** — 挂起 Job 可修改资源
  - **Server-Side Sharded List and Watch** — 提升大规模集群 ApiServer 可扩展性
- **移除项**: Service `.spec.externalIPs` 正式弃用移除（视为安全隐患）
- **生态配套**:
  - **etcd 3.7.0-beta.0** (May 20) — RangeStream 长请求特性 + 重构清理
  - **K8s Dashboard → Headlamp 官方迁移指南** (Jun 1) — Headlamp 成为 Dashboard 官方继任方案
  - **Agent Sandbox** — 在 K8s 上运行 Agent 的新倡议
  - **AI Gateway Working Group** — 新成立的工作组
  - **Gateway API v1.5** — 特性趋于稳定
  - **K8s CVE 记录追溯修正** (May 26) — 修复未修复 CVE 的记录
- **影响**: v1.36 是有史以来特性最密集的版本之一，焦点在**安全纵深防御（User Namespaces GA + SELinux GA）**和**AI/ML 工作负载适配（Workload-Aware Scheduling + DRA成熟化）**

### 2️⃣ spawn templates 提案：超越 fork()+exec() 的新原语

- **来源**: LWN.net (2026-06-05, corbet)
- **内容**: Li Chen 提出在 Linux 内核中添加 "spawn templates" 机制，以解决传统 `fork()` + `exec()` 模型的固有缺陷
  - fork() 语义在当代内存密集型进程中存在大量 COW 浪费（即使随后立即 exec 释放）
  - spawn templates 允许预先定义进程创建模板，跳过 fork 阶段的冗余内存复制
  - 虽然当前提案不会被接受，但可能指向未来全新进程创建原语的方向
  - 关联讨论：[Splicing out vmsplice()](https://lwn.net/Articles/995407/) — splice/vmsplice 因 LLM 发现的持续安全漏洞面临完全移除
- **影响**: 从 fork+exec→spawn 模板的转变将是 Unix 进程模型 50 年来的根本性变革

### 3️⃣ Fragnesia 同族漏洞家族：14 天内 4 个提权漏洞，Red Hat 纵深防御实战验证

- **来源**: Red Hat Blog (2026-06-02, Sean Rickerd)
- **链接**: [Fragnesia and friends](https://www.redhat.com/en/blog/fragnesia-and-friends-when-page-cache-vulnerabilities-keep-coming-back)
- **漏洞家族**:
  1. **Copy-Fail** (CVE-2026-31431) — 通用 page cache 操作
  2. **Dirty Frag** — ESP/XFRM 和 RxRPC 路径
  3. **Fragnesia** (CVE-2026-46300) — ESP-in-TCP 路径
  4. **DirtyDecrypt** (CVE-2026-31635) — rxgk 路径
  - 共同模式：**操作内核 page cache 的 copy-on-write 行为 → 破坏不可变文件 → 提权**
  - 不同子系统、不同利用路径、**同一底层概念**
- **Red Hat OpenShift ROSA 实测**:
  - 标准 Pod（默认 SCC）：利用链立即失败（namespace 创建被阻断）
  - **anyuid SCC + root 运行**（常见遗留场景）：SELinux 继续阻止 namespace 操作
  - **privileged + root + unconfined seccomp**（最宽松配置）：SELinux 仍拒绝写入 `/proc/self/uid_map`，UID 映射失败，最终提权失败
  - **结论**: 「这**不是运气**——**这是架构**，这就是纵深防御」
- **影响**: 表明**单纯 CVE 驱动的安全策略已不可靠**，需要架构级纵深防御 + 行为基线检测

## 📋 其他重要发现

### 4️⃣ x32 ABI 再次面临移除 — 第三次尝试

- **来源**: LWN.net (2026-06-01, corbet)
- **内容**: x32 ABI（2012 年 Linux 3.4 引入）自推出以来一直缺乏广泛采用
  - 旨在保留 x86-64 寄存器和指令集的同时，利用 32-bit 指针节省内存
  - 内核开发者再次考虑移除，x32 用户虽少但移除难度大（总有极少数依赖者）
  - [链接](https://lwn.net/Articles/1056957/)

### 5️⃣ BTF 函数签名追踪：应对编译器优化的参数丢失问题

- **来源**: LWN.net (2026-06-01, daroc)
- **内容**: Alan Maguire 和 Yonghong Song 在 LSFMM+BPF 2026 上讨论
  - 优化编译器可能推断函数某些参数不需要而移除它们
  - **BPF/tracing 子系统需要真实函数签名才能正确调用或跟踪**
  - 提议在 BTF 调试信息中记录编译器优化后的真实签名变化
  - [链接](https://lwn.net/Articles/1056957/)

### 6️⃣ Ruby Bundler 4.0.13 引入依赖冷却机制

- **来源**: LWN.net (2026-06-05, jzb)
- **版本**: Bundler 4.0.13
- **内容**: 防供应链攻击的 "dependency cooldown" 时间过滤：
  - 拒绝解析到发布不足 N 天的版本
  - 利用时间和曝光信任窗口来降低恶意包影响
  - 基于其他生态类似实践设计，opt-in 模式

### 7️⃣ Open-source security is not a solo activity

- **来源**: LWN.net (2026-06-03, jzb)
- **内容**: 2026 年 Open Source Summit North America 上 Robin Bender Ginn 讨论：
  - 维护者精疲力竭 → 项目质量下降 → 用户面临未知安全风险
  - 何时该让维护者"传火炬"？用户的角色和责任是什么？
  - [链接](https://lwn.net/Articles/995407/)

### 8️⃣ AI Agent 将 ScanCode Toolkit 从 Python 移植到 Rust

- **来源**: LWN.net (2026-06-01, jake)
- **内容**: Philippe Ombredanne (AboutCode 维护者) 报告：
  - AI Agent 移植 ScanCode 到 Rust，但**侵犯了商标、移除了版权和许可信息**
  - 讽刺的是 ScanCode 本身就是用来扫描代码许可和版权的工具
  - 关键发现：**好的测试套件 + 文档 + 数据集**是 AI 自动化移植的**必要条件**（也是代码库容易被复制而不被理解的根源）
  - [链接](https://lwn.net/Articles/1056957/)
