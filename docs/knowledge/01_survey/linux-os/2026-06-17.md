### 🔥 ① Linux 7.1 开发统计出炉——1,987 开发者·23% 新贡献者·社区结构在变
- **来源**: LWN.net (corbet, Jun 15) — LWN 传统开发统计深度分析
- **来源**: kernel.org (Linus Torvalds, Jun 14) — 7.1 稳定版发布
- **核心数据**:
  - **11,895 commits** 合入 7.1 开发周期
  - **1,987 位开发者**参与贡献，其中 **23% 是首次贡献者**（约 457 人）——新兴社区参与度显著提升
  - 与前几个版本相比，新贡献者比例持续走高，AI 编程工具（如 LLM 辅助代码生成）可能是一个驱动因素（corbet 在文章中专门探讨了这一趋势）
- **7.1 关键特性回顾**: FRED 中断架构 · BPF io_uring 支持 · 重写 NTFS 驱动 · ubk 零拷贝 I/O · sched_ext 子调度器（初始版）· i486 架构移除
- 🎯 **K8s 相关性**: sched_ext + BPF io_uring 为容器粒度的可编程调度和 I/O 策略奠定内核基础。新贡献者比例上升也意味着 **K8s/Cilium 等上游 BPF 相关子系统的维护者需适应更多贡献者涌入带来的审查负担**

### 🔥 ② Linux 7.2 合并窗口：自动 mTHP（多尺寸透明巨页）即将到来
- **来源**: LWN.net (corbet, Jun 11)
- **贡献者**: Nico Pache（Red Hat）
- **核心机制**: 此前 THP（透明巨页）的尺寸由硬件决定（通常仅 2MB 和 1GB 两种），软件层面的折衷灵活性有限。mTHP（multi-size THP）已在内核中存在一段时间，但需要应用程序/系统管理员手动通过 `thp_file_alloc` 等接口指定期望尺寸
- **7.2 新增**: 使 mTHP 的创建**更加透明**——内核根据页面使用模式自动选择最优的 mTHP 尺寸（16KB/32KB/64KB/128KB/256KB/512KB/1MB/2MB），无需用户干预
- **性能影响**: 在 TLB 覆盖率和内存浪费之间自动平衡。对于运行在 K8s Pod 中的 **Redis / Memcached / 数据库/ AI 推理服务**等大内存工作负载，mTHP 自动选择可降低 10-40% TLB miss，提升内存密集型工作负载吞吐
- 🎯 **K8s 协同**: 结合 K8s Pod-level THP 配置（`sysctl` 或 `/sys/kernel/mm/transparent_hugepage`），mTHP 自动选择使 Pod 无需 Per-application THP 调优即可受益

### ③ overlayfs 更新：嵌套层 + composefs 驱动创新
- **来源**: LWN.net (jake, Jun 12) — LSFMM+BPF Summit 2026 filesystem track
- **主讲人**: Amir Goldstein
- **关键进展**:
  - **嵌套层支持**: overlayfs 现在支持多层嵌套（此前 max 限制为 500 层），不再因层数过多被拒绝。这对 **containerd / Podman 容器镜像拉取**场景至关重要——多层镜像（如 AI 基础镜像 + 多层 conda 叠加）不再触发 kernel stack 问题
  - **composefs 驱动改变**: 2023 LSFMM 讨论的 composefs 整合已经成熟，composefs 现在利用 overlayfs 的 **metadata-only 层**和**数据分离模式**，实现了**只读镜像层 + 可写上层**的安全容器文件系统方案
  - i_version 和 statx() 改进提升容器文件系统变更检测精度
- 🎯 **K8s 相关性**: containerd/CRI-O 镜像拉取和多层 overlayfs 挂载的可靠性和层深度限制显著改善，对 AI 工作负载的大型多段镜像集特别有利

### ④ CNCF 数字主权架构：vCluster Tenant Cluster 成为主权边界原语
- **来源**: CNCF Blog (Hrittik Roy, Jun 16)
- **监管背景**: EU Data Act（2025.1.11 全面生效）· NIS-2 · DORA · UK Data Use and Access Act 2025
- **核心模式**: **Tenant Cluster（租户集群）**——每个主权边界一个独立的 K8s 控制平面，运行在共享下层集群之上
  - 每个租户集群有独立的 API Server、Controller Manager、Scheduler、etcd 存储
  - 用 **vCluster** 实现：租户集群控制平面作为 Pod 运行在共享的 Control Plane Cluster 上
  - 可插拔后端存储：租户集群的 etcd/SQL 数据存放在所选管辖区域内的加密卷上
  - 与 **Kyverno/OPA Gatekeeper** 策略引擎 · **SPIFFE/SPIRE** 身份层 · **Argo CD/Flux** GitOps 控制器协同
- **主权事件降级**: 一租户的 admission webhook 被攻破不会影响其他租户；CLOUD Act 请求不自动暴露所有租户的 etcd 内容
- 🎯 **AI 云场景**: Polarise（德国主权 AI 云提供商）已在生产中运行此模式——下层共享 GPU 容量（裸机 Metal³ 部署），上层每个客户一个 tenant cluster，全部在 EU 管辖下
- 🎯 **DRA 联动**: AI 工作负载通过 **Dynamic Resource Allocation (DRA)** 获取 GPU 访问，结合 tenant cluster 的主权边界，可在满足 EU AI Act 第 12 条日志与治理要求的同时，实现 GPU 资源的弹性共享

### ⑤ Arm64 CNCF OCI 信用额度项目：下一代云原生基础设施已转向 Arm
- **来源**: CNCF Blog (Dave Neary, Ampere Computing, Jun 15)
- **市场数据**: 截至 2025 年底，**AWS 超 50% 新实例是 Arm64**、**Azure 超 33% 新实例是 Arm64**
- **Oracle $3M OCI 信用捐赠**: 为 CNCF 项目提供自托管的 Arm64 GitHub Actions Runner（无大小限制，相比 GitHub 托管的 Arm64 runner CPU/内存更充裕，适合大型构建矩阵和多架构容器镜像构建）
- **项目收益情况**: OpenTelemetry（"Arm 性能远超旧 x86 服务器"·Arm64 下载量显著增长）· Falco（真正需要 Arm64 CI runner 以提升架构支持）· containerd（现已完全信任 Arm64 作为目标平台——CI 如果 Arm 失败会 blocking merge）
- **CNCF 审批流程**: 维护者提交 Service Desk 工单，初始预算上限 **$5,000 USD/月**，可根据实际需求申请增加
- 🎯 **K8s 意义**: 多架构容器镜像（manifest list）成为标准实践。KubeCon India 2026（6/18-19 孟买）将有 Arm64 最佳实践专题讨论

### ⑥ 云原生 + Agentic AI：Orange 在生产级 K8s 上构建多 Agent 安全平台
- **来源**: CNCF Blog (Willem Berroubache, Orange Innovation, Jun 17) — KubeCon Europe 2026 演讲
- **架构全景**: Coordinator Agent（LangGraph + A2A 协议）编排 4 个专业 Agent（Detect Falco+ML · Analyse 威胁分析 · Remediate · Notify Mattermost）+ 人类在环 + Isolation Forest 经典异常模型做预过滤
- **五大技术洞见**:
  1. **每个 Agent 是一个独立 K8s Deployment**（HPA / 命名空间隔离 / canary 发布，不共享进程空间）
  2. **Agent 间 mTLS 无需 Service Mesh**：cert-manager + 应用层 mTLS + CiliumNetworkPolicy，比 Sidecar 方式简单
  3. **安全约束 = Policy-as-Code**（OPA / Kyverno），不是 LLM prompt——政策版本控制、单元测试、代码审查
  4. **A2A trace_id 贯穿可观测性**：每个 Agent 输出带 trace_id 的结构化 JSON 日志 + Prometheus 指标（速率 / MCP 延迟 / 自动执行/拒绝/升级比例）+ Cilium Hubble 网络流
  5. **Isolation Forest 预过滤**：17 维特征 · 微秒级得分 · 仅最新颖的一小部分触发 LLM Agent -> 单次推理延迟和 Token 成本可控
- 🎯 **模式总结**: 整个 Agent 平台通过 Argo CD 从 Git 同步 CRD（system prompt、工具列表、输出 schema）、OPA 策略包，回滚 = `kubectl apply` 上一个版本
