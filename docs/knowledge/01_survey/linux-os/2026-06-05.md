## 📌 1. Kubernetes v1.36「ハル (Haru)」正式发布

- **来源**: [Kubernetes Blog](https://kubernetes.io/blog/) (2026-05) + K8s Release Notes
- **版本**: Kubernetes v1.36
- **关键特性**（22篇博客，以下为核心亮点）：

### 可观测性
| 特性 | 状态 | 说明 |
|:-----|:-----|:------|
| **PSI Metrics** | ✅ **GA** | Pressure Stall Information 从 Linux kernel 4.20+ 集成到 Kubelet，80+ Pod 高密度测试验证：Kubelet 开销仅 ~0.1 core (2.5%)，Kernel 开销仅 ~0.037-0.125 core。v1.36 新增 OS 级 PSI 支持检测，无 PSI 时不发出误导零值 |
| **Controller Staleness Mitigation** | 新增 | 增强 Controller 可观测性和陈旧数据检测 |
| **CCM Route Sync 新指标** | Alpha | `route_controller_route_sync_total` 计数器 |

### 调度与资源管理
| 特性 | 状态 | 说明 |
|:-----|:-----|:------|
| **Workload-Aware Scheduling** | v1alpha2 | **架构重大重构**：Workload API 降级为静态模板，新 PodGroup API 处理运行时状态。kube-scheduler 新增 **PodGroup 调度周期**（原子化调度），引入 **拓扑感知调度**（Topology-Aware Placement）和 **Workload-Aware Preemption**。Job 控制器与新 API 集成 |
| **Pod-Level Resource Managers** | Alpha | 按 Pod 级别独立管理资源，不再依赖 Container 级别 |
| **In-Place Vertical Scaling (Pod-Level)** | Beta | Pod 级别的原地垂直扩缩容（之前是 Container 级别） |
| **Tiered Memory Protection** | 新增 | 通过 Memory QoS 实现分层内存保护 |
| **Mutable Pod Resources (Suspended Jobs)** | Beta | 挂起 Job 的 Pod 资源可变 |

### 网络与安全
| 特性 | 状态 | 说明 |
|:-----|:-----|:------|
| **User Namespaces** | ✅ **GA** | 用户命名空间最终 GA，Container 可以完全非 root 运行 |
| **Service ExternalIPs** | ❌ 移除 | 因安全风险（假设所有用户完全信任）被移除 |
| **Mixed Version Proxy** | Beta | 混合版本 API Server 代理，安全升级跨版本集群 |
| **Fine-Grained Kubelet API Auth** | ✅ GA | 细粒度 Kubelet API 授权 |
| **Admission Policies (不可删除)** | 新增 | 关键准入策略获得删除保护 |
| **Declarative Validation** | ✅ GA | 声明式验证正式 GA |

### 存储
| 特性 | 状态 | 说明 |
|:-----|:-----|:------|
| **Volume Group Snapshots** | ✅ **GA** | 从 v1.27 Alpha → v1.32 Beta → v1.34 Beta2 → v1.36 GA |
| **DRA (Dynamic Resource Allocation)** | 持续演进 | 新一代 DRA 功能、驱动支持 |

### 其他
- **SELinux Volume Label Changes GA** — v1.37 可能有影响
- **etcd 3.7.0-beta.0** — 引入 RangeStream 功能（长期请求的流式 List）
- **Gateway API v1.5** — 多项特性进入 Stable
- **Agent Sandbox** — 在 K8s 上运行 Agent

## 📌 2. Linux kernel: splice()/vmsplice() 面临完全移除

- **来源**: [LWN.net](https://lwn.net/) (2026-06-04)
- **版本**: Linux mainline 7.1-rc6 / stable 7.0.11
- **关键发现**:
  - splice() 和 vmsplice() 系统调用有长期的安全问题历史
  - **LLM 发现的大量新漏洞**将这些问题推上前台
  - 内核开发者正在考虑**完全移除**这两个系统调用
  - 这是继 BPF 安全审查压力之后，AI/LLM 生成的代码审计报告对内核子系统影响的又一案例
  - Dave Airlie 在 SE Radio 访谈中提及引入 Rust 对吸引年轻内核开发者有价值

## 📌 3. BPF in the Agentic Era: Starovoitov 的「觉醒之吼」

- **来源**: [LWN.net](https://lwn.net/) (2026-06-03, LSFMM+BPF 2026)
- **版本**: Linux mainline 7.1-rc6
- **关键发现**:
  - BPF 维护者 Alexei Starovoitov 在 LSFMM+BPF 2026 上做了"更像觉醒之吼而非演讲"的分享
  - 核心论点：**BPF 必须变革才能避免被 LLM/Agent 编程范式淘汰**
  - 讨论包括：AI 生成的 patch 数量激增 → BPF 子系统审核不堪重负
  - 后续会议还讨论了 coding agent 对 bpftrace 等工具的滥用问题

## 📌 5. Kernel 函数签名与 BTF 跟踪增强

- **来源**: [LWN.net](https://lwn.net/) (2026-06-01, LSFMM+BPF 2026)
- **版本**: Linux mainline 7.1-rc6
- **关键发现**:
  - Alan Maguire 和 Yonghong Song 在 LSFMM+BPF 讨论**记录优化编译器移除的函数参数信息**
  - 编译器可能推断某些参数不需要并移除它们，但内核的 tracing/BPF 子系统需要知道如何调用该函数
  - 方案：在 BTF (BPF Type Format) 调试信息中记录变更后的函数签名
  - 对 eBPF 程序编写和内核追踪工具链有直接影响

## 📌 6. rsync 安全加固回应 + LLM 争议

- **来源**: [LWN.net](https://lwn.net/) → [Tridgell: rsync and outrage](https://lwn.net/Articles/1044712/) (2026-06-03)
- **版本**: rsync (开发版)
- **关键发现**:
  - Andrew Tridgell 回应 LLM 批评，详细说明使用 AI 工具的原因
  - AI 生成的漏洞报告"洪水"迫使 rsync 必须大规模加固
  - 正在进行的加固措施：更彻底的测试套件、代码覆盖率分析、多平台 CI、深度防御加固技术
  - 已吸引到新的优秀开发者加入（"rage storm clouds have silver linings"）
  - 同时: **ScanCode Toolkit 被 AI Agent 移植到 Rust**，但移除了版权/许可信息，引发商标侵权争议

## 📌 7. x32 ABI 再次面临移除讨论

- **来源**: [LWN.net](https://lwn.net/) (2026-06-01)
- **版本**: Linux mainline 7.1-rc6
- **关键发现**:
  - x32 ABI (2012年 Linux 3.4 引入) 再次面临移除
  - 虽提供 x86-64 的寄存器优势 + 32 位低内存消耗，但多年来关注度低
  - 内核开发者第 N 次讨论移除 — 但最不受待见的特性总有一些用户

## 📌 8. 容器安全与供应链事件

- **来源**: LWN.net Security Updates, K8s Blog
- **关键事件**:
