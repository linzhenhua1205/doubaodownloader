### 1️⃣ spawn templates：超越 fork()+exec() 的新进程创建提案

| 属性 | 内容 |
|:-----|:------|
| **来源** | LWN.net, Jonathan Corbet, 2026-06-05 |
| **版本/状态** | 内核讨论中，不会立即合并 |
| **标题** | [Moving beyond fork() + exec()](https://lwn.net/Articles/1012769/) |

- **Li Chen** 提出 **spawn templates** 内核提案，旨在创建超越传统 `fork()`+`exec()` 模型的新进程创建原语
- 自 Unix 早期以来，`fork()`（创建子进程）+ `exec()`（运行新程序）一直是核心系统调用，在 Linux 中对应 `clone()` + `execve()`
- 该模型虽有优雅之处，但存在缺点：fork() 在 COW 下对大地址空间仍有性能问题、安全攻击面大
- **关键判断**: 虽不会以当前形式合入，但指明了未来进程创建原语的发展方向

### 2️⃣ splice()/vmsplice() 因 LLM 漏洞洪流面临完全移除

| 属性 | 内容 |
|:-----|:------|
| **来源** | LWN.net, Jonathan Corbet, 2026-06-04 |
| **版本/状态** | 内核讨论中，可能完全移除 |
| **标题** | [Splicing out vmsplice()](https://lwn.net/Articles/1013897/) |

- `splice()` 和 `vmsplice()` 系统调用旨在通过最小化系统调用和数据复制来提升数据移动性能
- 但它们有**长期的安全问题历史**
- **LLM 发现的漏洞洪流**再次引起关注 → 开发者正在考虑**完全移除**这两个系统调用
- 这是 LLM 对内核安全影响的一个标志性事件：AI 辅助漏洞发现正在推动内核 API 设计决策
