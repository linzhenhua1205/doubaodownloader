## 1️⃣ Copilot CLI — 专用 `/security-review` 安全审查命令 🛡️

- **来源**: [GitHub Changelog (Jun 10, 2026)](https://github.blog/changelog/2026-06-10-dedicated-security-review-command-now-available-in-copilot-cli/)
- **工具**: GitHub Copilot CLI
- **标签**: `copilot-cli` `security-review` `ai-security` `experimental`

**核心内容**:

Copilot CLI 现在内置了一个**专用的 AI 驱动的安全审查命令** `/security-review`，作为实验性功能以公开预览形式发布。

**能力**:
- 扫描本地代码变更，返回**按严重性和置信度评分**的安全发现
- 提供**可操作的修复建议**，无需离开终端即可应用
- 专注高频高危漏洞类型：注入缺陷、XSS、不安全数据处理、路径遍历、弱加密

**定位**:
> 这不依赖 GitHub code scanning、Dependabot 或 secret scanning——它是一个**轻量级、按需的 AI 安全扫描**，在 commit 前使用

**使用方式**:
```bash
# 开启实验模式后
copilot /security-review
```

**价值判断**: 这是 Copilot CLI 从"代码补全/解释"向"安全审查"的重要扩展。相比 CodeQL 深度但慢的扫描，`/security-review` 定位在**commit 前的快速安全检查**——与 Copilot CLI `/explain` 和 `/fix` 形成了"解释→修复→审查"的终端内 AI 工作流闭环。

**可用性**: 实验性，需开启 experimental mode

## 2️⃣ GitHub — 周期性代码扫描不活跃仓库 🔄

- **来源**: [GitHub Changelog (Jun 9, 2026)](https://github.blog/changelog/2026-06-09-periodic-code-scanning-of-inactive-repositories/)
- **工具**: GitHub Code Scanning
- **标签**: `code-scanning` `security` `inactive-repos` `enterprise`

**核心内容**:

组织现在可以为**超过 6 个月无推送或 PR 活动**的仓库启用周期性安全扫描。

**关键参数**:
- 仅适用于**启用 code scanning default setup** 的仓库
- 扫描频率：**每 30 天一次**自动扫描
- 配置位置：Settings → Advanced Security → Global Settings → **"Keep scheduled scans running every 30 days for inactive repositories"**

**价值判断**: 这是企业安全治理中一个**长期被忽视的缺口**——不活跃仓库（归档遗留代码、暂缓项目）往往是安全盲区。GitHub 此举将安全扫描从"活跃开发周期"扩展到了"资产全生命周期"，与秘密扫描、Dependabot 形成了完整的存量代码安全覆盖。对于拥有数千仓库的大型企业，这填补了一个关键盲点。

## 3️⃣ GitHub Enterprise Server 3.21 GA 🏢

- **来源**: [GitHub Changelog (Jun 11, 2026)](https://github.blog/changelog/2026-06-11-github-enterprise-server-3-21-is-now-generally-available/)
- **工具**: GitHub Enterprise Server (GHES)
- **标签**: `ghes` `enterprise` `release` `3.21`

**核心内容**:

GHES 3.21 是自 GHES 3.20（2026年3月）以来的重要版本更新，主要增强集中在**部署效率、监控能力、代码安全和策略管理**四个维度。

**主要更新**:

| 功能 | 状态 | 说明 |
|:-----|:----:|:------|
| **组织自定义属性** | GA | 企业管理员可为组织打标签元数据，自动定向企业规则集 |
| **Projects 层级视图** | GA | 在项目表格视图中查看完整 issue 层级结构，无需切换视图 |
| **REST API v2026-03-10** | 新版本 | 引入 breaking changes；旧版 v2022-11-28 至少再支持 24 个月 |
| **Actions 工作流页性能** | 改进 | >300 job 的工作流支持懒加载+按状态筛选 |
| **Secret Scanning** | 增强 | 告警级/企业级权限改进，自定义模式+推送保护绕过管理 |
| **多数据盘配置** | GA | MySQL + 仓库数据分离，支持独立部署和 HA 拓扑 |
| **Codespaces 数据驻留** | GA | Enterprise 用数据驻留版 Codespaces GA |

**价值判断**: GHES 3.21 的 REST API 版本切换值得注意——v2022-11-28 已在 3.5 年+，新 API 版本的 breaking changes 意味着集成方需要规划升级。多数据盘 GA 对大规模 GHES 部署（数 TB 级 MySQL + 仓库数据）的运维效率有直接提升。

## 4️⃣ GitHub Actions — 新 Runner 镜像公开预览 🖥️

- **来源**: [GitHub Changelog (Jun 11, 2026)](https://github.blog/changelog/2026-06-11-new-runner-images-in-public-preview/)
- **工具**: GitHub Actions
- **标签**: `actions` `runner-images` `ubuntu-26-04` `windows-arm64`

**核心内容**:

两个全新的 GitHub-hosted Runner 镜像进入公开预览：

### Ubuntu 26.04
- 支持 **x64** 和 **arm64** 双架构
- 标签：`ubuntu-26.04` / `ubuntu-26.04-arm`
- 工具和工具版本与早期镜像有差异（见 runner-images 仓库）

### Windows 11 arm64 with Visual Studio 2026
- 标签：`windows-11-vs2026-arm`
- 提供 VS 2026 工具链的 Windows arm64 CI 验证环境
- **2026年9月初** 现有 `windows-11-arm` 标签将迁移至 VS2026 镜像

**价值判断**: Ubuntu 26.04 arm64 镜像**尤其值得关注**——随着 Arm 服务器（AWS Graviton、Ampere、NVIDIA Grace）的持续扩张，arm64 CI/CD 需求快速增长。Ubuntu 26.04 arm64 的 Runner 镜像上架意味着团队可以在不管理自托管 arm64 Runner 的情况下验证 arm64 构建兼容性。
