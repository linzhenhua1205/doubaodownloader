## 1️⃣ JetBrains Rider 2026.2 EAP 5 — IDE 级 Agent 代码质量检查钩子 🎯

- **来源**: [JetBrains .NET Tools Blog — Rider 2026.2 EAP 5 (Jun 8, 2026)](https://blog.jetbrains.com/dotnet/2026/06/08/rider-2026-2-code-quality-check-hooks-for-ai-agents/)
- **工具**: JetBrains Rider
- **标签**: `jetbrains` `rider` `ai-agent` `code-quality` `posttooluse-hook` `mcp`
- **核心内容**:

Rider 2026.2 EAP 5 为 **Claude Code 和 OpenAI Codex** 引入了内置的质量检查钩子（Quality-Check Hooks），这是 IDE 层面的 Agent 输出质量门禁机制。

**架构原理**:

```
AI Agent (Claude Code / Codex)
    │
    ├─ 编辑文件 ──────────────────┐
    │                              │
    │  PostToolUse Hook ────────── │
    │      │                       │
    │      └─ Rider 自动运行:      │
    │          • 内置代码分析      │
    │          • 代码检查 (Inspections)
    │          • 格式一致性检查    │
    │                             │
    │  ┌─ 有错误? ──→ 阻塞 Agent  │
    │  │                          │
    │  └─ 有警告? ──→ 反馈给 Agent │
    │                Agent 自行修正│
    │                             │
    └─ 无问题 → Agent 继续 ──────┘
```

**关键设计**:
- **PostToolUse 钩子**: Agent 编辑文件**之后**、继续下一步**之前**，Rider 自动运行 IDE 级别的代码验证
- **错误阻断**: 检测到的错误**阻止 Agent 将任务标记为完成**，必须修正后才能继续
- **警告反馈**: 警告和其他发现作为**可修复反馈**返回给 Agent，Agent 自行修正输出
- **配置方式**: Settings → Tools → MCP Server → Bundled Hooks，一键启用

**价值判断**: 这是目前业界**唯一一个在 IDE 层面充当 AI Agent 代码质量门禁**的实现。此前所有 Agent 代码审查方案（Cursor Bugbot、Copilot Code Review、Linear Agent 自审）都是**Agent 审查 Agent 的输出**，而 Rider 的方案是**IDE（编译器和静态分析引擎）审查 Agent 的输出**——后者天然具有更低的假阳性、更高的可信度。

**与竞品对比**:

| 维度 | JetBrains Rider Hooks | Cursor Bugbot | GitHub Copilot Code Review |
|:-----|:----------------------|:--------------|:---------------------------|
| 审查方 | IDE 内置分析引擎 | Composer 2.5 Agent | Copilot 云端 Agent |
| 触发时机 | PostToolUse（实时） | Push/PR 时 | PR 提交后 |
| 阻断能力 | **可阻断任务完成** | 仅建议 | 仅建议 |
| 依赖 | Rider 自身分析能力 | Cursor 模型 | Copilot 模型 |
| 覆盖语言 | .NET 为主 | 多语言 | 多语言 |

## 2️⃣ GitHub — Copilot 代码审查新增企业级控制 🏢

- **来源**: [GitHub Changelog — Copilot code review new configurations (Jun 12, 2026)](https://github.blog/changelog/2026-06-12-copilot-code-review-new-configurations-and-controls/)
- **工具**: GitHub Copilot Code Review
- **标签**: `copilot` `code-review` `enterprise` `runner-control` `content-exclusion`

**三条增量更新**:

### ⚙️ 组织级 Runner 控制
- 可在组织级别统一设置 Copilot 代码审查的 Runner 类型（托管 / 自托管 / Large Runner）
- **一次配置，全组织仓库生效**，无需每个仓库单独设置
- **锁定机制**: 组织管理员可锁定 Runner 设置，覆盖单个仓库的配置
- 配置位置: Organization → Copilot → Runner type → Runner type configuration
- 同时适用于 Copilot Code Review 和 Copilot Cloud Agent

### 🛡️ 内容排除（Content Exclusion）
- Copilot 代码审查现在**尊重仓库/组织/企业级内容排除设置**
- 可排除指定文件/目录不被 Copilot 读取或审查
- 路径规则配置在仓库设置中
- **意义**: 解决企业担心 Agent 访问敏感文件（如密钥文件、合规文档、第三方代码）的核心顾虑

### 📝 自定义指令字符限制取消
- 此前 `copilot-instructions.md` 和 `*.instructions.md` 文件超过 **4,000 字符**后会被截断
- 该限制已**完全取消**，现支持任意长度的自定义指令

**价值判断**: 这三条更新展示了 Copilot Code Review 从"个人生产力工具"向"企业级治理工具"的转型。Runner 控制解决的是 **Agent 代码审查对基础架构的控制权**问题，内容排除解决的是**数据边界**问题，无限指令解决的是**策略表达能力**问题。三者合在一起，形成了企业级 Agent 代码审查的**管控三件套**。

## 3️⃣ GitHub — Bot 创建的 PR 经批准后可运行 Workflows 🔐

- **来源**: [GitHub Changelog — Bot-created PRs can run workflows if approved (Jun 11, 2026)](https://github.blog/changelog/2026-06-11-bot-created-pull-requests-can-run-workflows-if-approved/)
- **工具**: GitHub Actions / GitHub Copilot
- **标签**: `actions` `bot-pr` `workflow` `approval-gate` `agent-governance`

**更新内容**:
- Agent/Bot 创建的 Pull Request 现在**经人工批准后即可运行 Workflows**
- 此前 Bot 创建的 PR 需要手动触发 Workflow 或通过其他变通方式
- 审批 + 自动运行形成了**安全合规的自动化流水线**

**价值判断**: 这是 Agent 写代码的**审批链最后一环**——Agent 创建 PR → 人工审批 → 自动运行 CI/CD。结合同日发布的 Agentic Workflows 公开预览（六层安全架构）和 PAT-free，GitHub 正在系统性地解决"Agent 写的代码如何安全地进入生产"这个核心问题。

## 4️⃣ GitHub — CodeQL CLI 增量分析支持 Go / C / C++ ⚡

- **来源**: [GitHub Changelog — Incremental analysis for Go, C/C++, CodeQL CLI (Jun 10, 2026)](https://github.blog/changelog/2026-06-10-incremental-analysis-for-go-c-cpp-and-codeql-cli/)
- **工具**: CodeQL CLI
- **标签**: `codeql` `semgrep` `incremental-analysis` `go` `c++` `security`

**更新内容**:
- CodeQL CLI 扩展增量分析支持至 **Go, C 和 C++** 语言
- 增量分析仅扫描变更部分而非全量扫描，大幅降低分析耗时
- 配合不久前发布的 Security Validation for Third-Party Coding Agents GA（第三方 Agent 安全验证），增量分析使得 **Agent 每轮代码变更后的安全扫描成本大幅降低**

**价值判断**: 安全扫描的延迟是 Agent 编码工作流中的瓶颈之一。CodeQL 增量分析让 Agent 的每轮修复只需扫描变更部分而非全量仓库，直接降低 Agent 编码周期中的安全验证时间，使得"每次 Agent 变更都跑安全扫描"成为可行策略。

**三个趋势判断**：

1. **IDE 成为 Agent 质量门禁**: JetBrains Rider 的 PostToolUse 钩子对比 Cursor Bugbot / Copilot Code Review 的 Agent 自审，代表了两种不同的质量控制哲学——"让 IDE/编译器说不行" vs "让 Agent 审 Agent"。前者在可靠性上更有优势，后者在灵活性和语言覆盖上更强。

2. **企业治理进入 Agent 代码审查层**: GitHub 一周内连续推出 Runner 控制 + 内容排除 + 无限指令 + Bot PR 审批流，加上上周的第三方 Agent 安全验证 GA，企业级管控能力正在快速追上 Agent 的生产力提升速度。

3. **安全扫描转变为 Agent 工作流的低延迟组件**: CodeQL 增量分析、Copilot CLI `/security-review` 等更新表明，安全扫描正从"CI 阶段的大延迟步骤"转变为"可嵌入 Agent 循环的低延迟步骤"。
