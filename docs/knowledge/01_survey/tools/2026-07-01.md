## 2️⃣ Copilot Agent 进驻 JetBrains AI Assistant 🏗️

- **来源**: [GitHub Changelog (Jun 30, 2026)](https://github.blog/changelog/)
- **工具**: GitHub Copilot Agent / JetBrains AI Assistant
- **标签**: `copilot` `jetbrains` `agent` `ide` `intellij`

**核心内容**: GitHub Copilot Agent 功能在 JetBrains AI Assistant 中正式可用，标志着 Copilot Agent 跳出 VS Code 生态。

### 🔹 能力

- 在 IntelliJ IDEA、PyCharm、WebStorm 等 JetBrains IDE 中通过 AI Assistant 使用 **Copilot Agent 的完整能力**
- 支持 Agent 自主规划、执行、调试代码的循环
- 与 JetBrains 生态原生集成（项目模型、运行配置、调试器等）

### 🔹 背景：JetBrains 上的 Copilot 演进

| 时间 | 事件 |
|:----:|:------|
| Jun 22 | JetBrains IDE 中引入 Claude 作为 Agent Provider 预览 |
| Jun 02 | JetBrains IDE 中 Copilot CLI 和 Agent 能力增强 |
| **Jun 30** | **Copilot Agent 在 JetBrains AI Assistant 正式可用** |

### 🔹 战略意义

| 维度 | 分析 |
|:-----|:------|
| **IDE 垄断破局** | Copilot Agent 此前深度绑定 VS Code 及其衍生品（Cursor 为此而起）。进入 JetBrains 意味着 GitHub 不再满足于 VS Code 生态的领先，而是主动进入 Python/Java/Kotlin 开发者的核心 IDE |
| **用户覆盖扩展** | JetBrains 在 Java/Kotlin 后端、Android、数据科学领域占据统治地位。这些用户此前只能用「Copilot 补全」而非 Copilot Agent。现在 Agent 的规划-执行能力惠及这个群体 |
| **Agent Provider 模型** | 注意 Jun 22 引入的是「Claude 作为 Agent Provider」，Jun 30 是「Copilot Agent」——两者同在 JetBrains AI Assistant，用户可根据需求选择 |

> **信号**: AI IDE 竞争的战场正在从「编辑器补全」转向「Agent 范式」，并且从 VS Code 一家独大走向多 IDE 覆盖。Cursor 抢了 VS Code 的用户，GitHub 则进 JetBrains 抢后端开发者。

## 3️⃣ Open Source License Compliance — Public Preview 📜

- **来源**: [GitHub Changelog (Jun 30, 2026)](https://github.blog/changelog/)
- **工具**: GitHub / 供应链安全
- **标签**: `github` `open-source` `license` `compliance` `supply-chain` `security`

**核心内容**: GitHub 推出**开源许可证合规性检测**功能（Public Preview），帮助开发者和管理者识别仓库中的许可证风险。

### 🔹 能力

- 自动扫描仓库依赖中的**开源许可证类型**并与项目目标许可证兼容性比对
- 识别 **许可证冲突**（如 GPL 与专有软件混用）
- 提供**合规建议**和**修复指引**
- 与 Dependabot、Secret Scanning 等其他供应链安全功能形成互补

### 🔹 企业价值

开源许可证合规是**企业法务和工程团队的长期痛点**。传统上依赖 Snyk、FOSSA、Black Duck 等第三方工具做 License 扫描。GitHub 将其内置到平台层面：

- **零额外成本**：对 GitHub 用户，不需要额外采购第三方工具
- **CI/CD 集成**：可在 PR 级别阻断不合规依赖引入
- **与 Dependabot 联动**：License 问题 + 版本漏洞，一站式解决

> **意义**: 继 Code Scanning、Dependabot、Secret Scanning 之后，License Compliance 补上了供应链安全最后一块拼图。GitHub 的「应用安全平台」拼图日益完整。

## 4️⃣ Per-user AI Credit Budgets for Cost Centers 💰

- **来源**: [GitHub Changelog (Jun 30, 2026)](https://github.blog/changelog/)
- **工具**: GitHub Copilot / Enterprise 管理
- **标签**: `copilot` `enterprise` `cost-center` `budget` `governance` `ai-credits`

**核心内容**: GitHub Copilot 为 Enterprise 的 Cost Center 管理新增**按用户的 AI Credit 预算控制**能力。

### 🔹 能力

- 在 Cost Center 中为**每个用户设置 AI Credit 预算上限**
- 结合此前（Jun 19）AI Credits consumed per user API，实现 **预算设置 + 用量追踪 + 超限告警** 闭环
- 与 Jun 25 的 Cost Center + Enterprise Teams 支持叠加，**Cost Center 体系已覆盖团队+用户双层**

### 🔹 Enterprise 管理工具链成熟度

| 功能 | 日期 | 说明 |
|:----|:----:|:-----|
| Cost Centers 创建（上限 500 个） | Jun 10 | 组织级成本中心 |
| Budget & Usage APIs GA | Jun 04 | 预算和用量 API 正式版 |
| Cost Centers + Enterprise Teams | Jun 25 | 与组织架构对齐 |
| **Per-user AI Credit Budgets** | **Jun 30** | **用户级预算控制** |
| AI Usage Report 更新 | Jun 11 | 管理层级报表 |

> **意义**: 企业大规模采用 Copilot 的最大障碍之一是**成本失控风险**。Per-user Credit Budget 让管理员可以在用户级别精细控制 AI 用量，避免少数用户过度消耗 Credits 导致预算超支。这是 GitHub Copilot Enterprise 从「工具可用」到「治理完备」的关键一步。

## 5️⃣ Code Coverage Merge Protection for PRs 🔒

- **来源**: [GitHub Changelog (Jun 30, 2026)](https://github.blog/changelog/)
- **工具**: GitHub / Code Quality
- **标签**: `github` `code-coverage` `merge-protection` `pr` `quality-gate`

**核心内容**: 代码覆盖率结果现可作为 **PR 合并保护条件**，不达标的 PR 将被阻止合并。

### 🔹 能力

- 在仓库设置中配置 **Code Coverage 阈值**作为 PR 合并的必过条件
- 与 GitHub Code Quality（即将在 Jul 20 GA）集成
- 覆盖率的来源可以是任何 CI 工具（不限于 GitHub 自身）

> **意义**: 与 Jun 16 的 GitHub Code Quality GA 预告（Jul 20）配合，GitHub 正在构建一个完整的**代码质量门禁体系**（Code Scanning + Code Review + Code Coverage），在 PR 级别层层把关安全与质量。
