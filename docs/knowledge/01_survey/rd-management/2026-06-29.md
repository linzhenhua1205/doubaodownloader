## 一、GitHub Copilot Agentic Harness 基准测试（Jun 25）— 首个公开头对头对比

> **来源**: [GitHub Blog](https://github.blog/ai-ml/evaluating-performance-and-efficiency-of-the-github-copilot-agentic-harness-across-models-and-tasks/) · Jun 25, 2026 · Shibani Basava & Carlos Castro
> **类别**: 研发工具 · AI编程工具评测 · Agent 工程化实践

### 背景

GitHub Copilot 内部有一个 **共享 Agentic Harness**（SDK 层），同一个 harness 同时驱动 **Copilot CLI、Copilot App、Copilot Code Review** 以及 GitHub 和 Microsoft 内部的数十个 Agent 体验。提升 harness = 所有 surface 同时受益。

本文首次公开了该 harness **与模型原厂 harness**（Claude Code 对应 Sonnet/Opus、Codex CLI 对应 GPT）的对比数据。

### 基准测试设置

| Benchmark | 项目数 | 用途 | 难度 |
|:----------|:------:|:-----|:-----|
| **SWE-bench Verified** | 500 | 已验证的 Python bug-fix 任务 | 行业标准 |
| **SWE-bench Pro** | 300+ | 多步工程任务，需更深推理和更大范围代码改动 | 高（贴近真实） |
| **SkillsBench** | — | Agent 使用工具/Skills 的能力 | 评估可扩展性 |
| **TerminalBench 2.0** | 89 | Agent 在终端任务上的表现 | CLI 工作流 |
| **Win-Hill** | 内部 | Windows 容器内的任务 | 跨 OS 验证 |

**控制变量方法**：同一模型 × 同一任务 × 相同推理力度 × 相同上下文窗口大小 × 相同工具集。

### 核心发现

#### 1. Token 效率 — Copilot Harness 在多数配置下更省 Token

> 「在**固定模型和任务**下比较，Copilot CLI 在大多数配置中实现了**更低的 Token 消耗**，同时保持任务完成率与模型原厂 harness 相当。」

#### 2. 任务完成率 — 持平（差异在随机方差内）

> 「固定模型和基准测试下，Copilot CLI 的任务完成率与模型原厂 harness **相当**。两个方向的差异都在模型随机性的 ±1σ 范围内。」

#### 3. TerminalBench 2.0 的细致分析

关键图表解读（三个发现）：

| 发现 | 细节 |
|:-----|:------|
| **Copilot 未在任何配置下落后** | 紫色（Copilot）标记及其同模型竞品的椭圆在完成率和成本两个轴上重叠，Copilot 从未低于竞品的完成率或向右（更贵） |
| **GPT 系模型性价比最优** | 图表左侧：GPT 模型以最低成本达到强完成率 |
| **Claude Opus 完成率最高但代价最大** | 右上角：Opus 达到最高分辨率但成本也最高 |
| **Copilot 提供灵活选择** | 同一 harness 覆盖 20+ 模型，可在效率和质量之间按任务切换 |

#### 4. 多模型架构的独特能力

Copilot 的多模型架构实现了**模型原厂 harness 无法提供的能力**：

- **Rubber Duck**：跨模型家族批评——一个模型审查另一个模型的工作，产出超过任何单一模型
- **Auto Model Selection**：根据任务意图和模型健康度自动选择最优模型
- **BYOK**：支持自有密钥接入开源/本地模型

### 方法论要点

| 参数 | 值 |
|:-----|:---|
| 超时 | 2 小时 |
| 交互模式 | 非交互式单轮 |
| Web 工具 | 禁用 |
| 推理力度 | medium |
| 各 agent-model 配对 | 相同上下文窗口、相同 prompt token 限制 |
| 各 agent 复现 | TerminalBench: 5 次独立运行；SWE-bench: 5 次取 best pass@1 |
| Copilot 测试 | 两个独立评估批次（以与 Claude Code、Codex CLI 对比） |

- 📌 **【深度跟踪】** 这是目前业界第一个**严格受控的跨 harness 对比**。关键信号：(1) harness 层的差异化空间是 real 的——即使是同一模型，不同 harness 的 Token 效率可以差出一截；(2) Copilot 的多模型架构带来了 harness 级别的独特能力（Rubber Duck, Auto Model Selection），这是单一模型厂商的 CLI 工具无法复制的差异化；(3) 但对用户来说，**模型的选择比 harness 的选择影响更大**——GPT 系性价比最优、Opus 完成率最高，Copilot 同时提供这两种选择本身就是差异化。

## 二、GitHub Copilot App GA（Jun 17）— Agent-Native Desktop + Canvases + Sandboxes + Code Review

> **来源**: [GitHub Blog](https://github.blog/news-insights/product-news/github-copilot-app-the-agent-native-desktop-experience/) · Jun 2, 2026 · Mario Rodriguez · [GitHub Changelog](https://github.blog/changelog/) · Jun 17, 2026
> **类别**: 研发工具 · Agent 工程化实践 · Enterprise 治理

### 背景：Agent 原生时代的桌面

> 「如果 Agent 要成为软件开发中持久的一部分，它们需要在开发者工作流中有一个真正的位置。然而大多数开发者工具在设计时并没有考虑并行指挥多个 Agent。上下文分散在各个窗口中，你无法追踪什么在运行，代码以 PR 形式提交却没有清晰的 trace 记录 Agent 尝试了什么、验证了什么、哪里需要人工判断。」

GitHub 的宏观数据佐证了这一趋势的剧烈程度：

> 「在 GitHub 上，**提交次数同比几乎翻倍，突破 14 亿次/月**，加上**每周 20 亿分钟**的 GitHub Actions 运行时间。仓库创建、PR 活动和 API 使用都在加速且没有放缓迹象。」

### Copilot App 核心能力

#### 1. My Work 控制台

单一视图查看所有进行中的工作——横向连接的仓库中的活跃 Agent 会话、Issue、PR 和后台自动化。

**每会话独立 Git Worktree**：每个 Agent 会话运行在自己的独立 git worktree（隔离的分支副本）。多个并行 Agent 会话互不干扰，App 自动处理 worktree 的创建和清理。

#### 2. Agent Merge — 从 PR 到合入的自动化

Agent Merge 自动推动 PR 走完 Review → CI → 合并全流程：
- 监控 CI 状态
- 追踪要求的 Reviewers
- 处理失败的检查
- 等待所有条件满足后才合并
- 开发者选择 Copilot 可以走多远：修复 CI / 处理 Review 反馈 / 条件满足后自动合并

#### 3. Canvas 交互面 — 可视化的双向协作

> 「Chat 适合指令和歧义讨论，但一旦 Agent 开始做实际工作，Chat 就变成了长条状的决策、日志和修正记录。你需要一个**工作本身可见**的地方。」

**Canvas** = 人与 Agent 的双向工作界面。一个 Canvas 可以展示：Plan / PR / 浏览器会话 / 终端 / 部署 / 仪表板 / 工作流状态。

Agent 在工作时更新 Canvas，开发者可以在同一界面上编辑、重排序、批准或重定向工作。

> 📌 这是 **Agent Experience (AX)** 的雏形——Chat 用于指令和歧义协商，Canvas 用于意图到可见工作的转化。

#### 4. 云沙箱 + 本地沙箱（Public Preview）

| 维度 | 本地沙箱 | 云沙箱 |
|:-----|:---------|:-------|
| 位置 | 开发者本地机器 | GitHub 托管的临时 Linux 环境 |
| 隔离 | 隔离环境，限制文件系统/网络/系统能力 | 完全隔离 + 临时性 |
| 策略 | 组织级中心配置和强制 | 组织自行定义策略 |
| 跨设备 | 本地 | 任意设备远程连通 |

**Copilot Memory for Business/Enterprise**（Jun 2）：Agent 跨设备的连续性——查询从 App/CLI/VS Code/github.com 启动的会话上下文。

#### 5. Copilot Code Review — 适应性 Agent 审查系统

关键更新（Jun 2026）：

| 功能 | 状态 | 说明 |
|:-----|:----|:------|
| **Medium 审查等级** | 🆕 Jun 25 | 高推理模型审查 PR，更高精度和召回率 |
| **按仓库配置审查等级** | 🆕 Jun 12 | Admin 可为低风险仓库设 "low"，高风险设 "medium" |
| **AGENTS.md 支持** | 🆕 Jun 18 | 自定义 Agent 行为的 repo 级说明文件 |

- 📌 **【深度跟踪】** Copilot Code Review 正在形成一种 **分级审查体系**：(1) Low tier — 低成本模型快速过筛；(2) Medium tier — 高精度深入审查；(3) Security review — 专项安全审查；(4) Rubber Duck — 跨模型批判。这种分级体系的核心价值是**按风险配置成本**——低风险变更用低成本模型，高价值变更投入更多计算资源。结合 AGENTS.md + Customs Skills + MCP，企业可以注入自己的代码规范和安全策略，让代码审查从"通用检查"升级为**符合组织标准的适应性审查**。

## 三、Cursor 3.8（Jun 18）— Automations 增强 + /automate skill + 新 GitHub/Slack 触发器

> **来源**: [Cursor Changelog](https://changelog.cursor.com/) · Jun 18, 2026
> **类别**: 研发工具 · AI编程工具更新 · Agent 工程化实践

Cursor 3.8 是对 **3.7 Cloud Subagents** 和 **Automations** 事件驱动 Agent 引擎的增强版本。核心更新：

### /automate skill

在本地 Agent 会话中直接使用 `/automate` 创建自动化：
- 用自然语言描述想自动化的任务
- Cursor 自动配置触发器、指令和工具
- 无需离开当前会话即可创建自动化

### Slack Emoji 触发器

> 「对任何 Slack 消息用指定 emoji 回复即可触发自动化。Cursor 团队内部用这个机制直接从 Slack 触发特定自动化。」

### 五个新 GitHub 触发器

| 触发器 | 触发时机 |
|:-------|:---------|
| `Issue comment` | 非 PR 的 Issue 上有人评论 |
| `PR review comment` | PR diff 上留下内联评论 |
| `PR review submitted` | PR Review 被提交 |
| `Review thread updated` | PR Review thread 标记为 resolved/unresolved |
| `Workflow run completed` | GitHub Actions workflow 在 PR/分支上运行完毕 |

**配套 Marketplace 模板**：
- 排查失败的 GitHub Actions
- 自动修复 PR Review 评论

### Automations 的 Computer Use 工具

被 Automation 触发的 Cloud Agent 现在可以使用自己的计算机制作 Demo 或产出成果物。默认启用——只需在自动化指令中加入 "include a demo of your work"。

### 其他改进

- 自动化可保存为**不完整状态**（离开设置 MCP auth 后回来继续）
- 自动化默认可以**打开 PR**（无需在 UI 中指定）
- 可在 UI 中 **删除 memory 文件**，或在自动化运行时提示删除过期记忆

- 📌 **【深度跟踪】** Cursor 3.7→3.8→3.9 构成了一个清晰的演进弧线：3.7 Cloud Subagents（后台并行能力）→ 3.8 Automations（事件驱动触发）→ 3.9 Customize + Enterprise Governance（统一管理+治理）。Slack Emoji trigger 表面看起来是"趣味功能"，但其架构含义重要——**Agent 从"在 IDE 中被调用"变成了"在 Slack 中被触发"**，意味着 Agent 的能力边界突破了 IDE。这是 Agent 从"工具"到"团队成员"的位置迁移。

## 五、Atlassian: Knowledge Architect — AI-First 组织的新角色（Jun 2026）+ AI Agents Drift + 研发瓶颈转移

> **来源**: [Atlassian Blog](https://www.atlassian.com/blog/) · Jun 2026
> **类别**: AI辅助研发 · 研发管理 · HBR 组织管理研究

### 1. Knowledge Architect — 每个 AI-First 组织即将需要的角色

> 「虽然大多数组织在讨论如何进行 AI 转型，知识管理和所需的上下文编排正在成为 AI 能否有效工作的关键因素。这一转变正在催生一个叫做 **Knowledge Architect** 的新角色。」

**核心论点**：
- 就像有 Systems Architect 设计软件整体架构一样，AI 时代需要 **Knowledge Architect** 来设计知识的结构、流动和可用性
- 关键职责：设计知识分类体系、维护上下文质量、确保 Agent 可访问组织知识库
- 与 "Context Wrangler"（上一期 Atlassian 访谈）一脉相承——CEO 级别的上下文管理员是第一步，但规模化后需要专门的 Knowledge Architect

**Atlassian 的实践**：**System of Work**（Jira + Confluence + Loom）作为 AI 上下文基础设施——知识记录 + 项目追踪 + 异步视频沟通的整合，成为 Agent 获取组织上下文的管道。

### 2. Why AI Agents Drift Mid-Task（Jun 2026）

> 「你给了一个能干的 Agent 一份真正的工作——有边界情况和验收标准的完整 spec——头几分钟它很精准。然后它开始漂移。它忘记了自己做的决定。它朝着一个从未讨论过的方向前进。」

**根本原因**：Agent 在长任务中缺乏**持续指向原始 spec 的锚定**。大型上下文窗口中，Agent 的注意力从原始需求逐渐偏移到中间结果。

**Atlassian Rovo Dev 的解决方案**：**多 Agent 编排 + Jira Spec 锚定**
- 一个 orchestrator agent 持续持有原始 Jira Issue spec
- 解耦为多个 specialist agents（搜索、实现、测试）
- 每个 specialist agent 完成后，orchestrator 对照原始 spec 验证
- 发现漂移时触发回退

### 3. The Bottleneck Keeps Shifting（May 6 补充深度解读）

> 「过去几十年，构建产品的最大约束一直是工程师不够、时间不够、算力不够。第一次，这个时代正在结束。」

| 旧瓶颈 | 新瓶颈 | 影响 |
|:-------|:-------|:-----|
| 写代码 | **决定该写什么** | PM 的超能力 = Discernment（辨别力） |
| 工程师数量 | **上下文质量** | 文档化成为 Agent 效能的关键依赖 |
| 开发速度 | **对齐速度** | 团队对齐 > 个人产出 |
| 职能边界（PM/Eng 分离） | **角色模糊化** | PM 开始写代码 + 做 Bug Triage |

- 📌 **【深度跟踪】** Atlassian 三条内容（The Bottleneck Keeps Shifting + Knowledge Architect + AI Agents Drift）构成一个完整的叙事：(1) **编码不再是瓶颈** → (2) **决策和上下文成为新瓶颈** → (3) 需要 **Knowledge Architect** 来系统化解决上下文问题 → (4) 即使有上下文，**Agent 仍会漂移** → (5) 需要 **多 Agent 架构 + spec 锚定** 来约束 Agent 行为。这种连锁反应揭示了 AI 引入研发体系后的**二阶效应**——解决一个问题会暴露更深层的问题，"瓶颈在转移"本身是 AI 重塑研发组织的核心动态。
