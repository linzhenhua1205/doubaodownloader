## ① Cursor SDK 3.7 大幅成熟 — Custom Tools + Auto-Review + Nested Subagents + JSONL（6月4日）

> **来源**: [Cursor Changelog — Custom stores, custom tools, and auto-review for the Cursor SDK](https://changelog.cursor.com/) | Jun 4, 2026

Cursor SDK（TypeScript/Python）发布重大功能批次，标志着从"用户端Agent工具"向"可编程Agent基础设施"的转变：

**🔧 Custom Tools（自定义工具）**: 通过 `local.customTools` 直接传入函数定义即可暴露给本地Agent，无需启动独立的stdio/HTTP MCP Server。SDK内置 `custom-user-tools` MCP server作为桥接，自定义工具在父Agent及其所有子Agent中全局可见。

**✅ Auto-Review（自动审查）**: 原本headless SDK模式下所有工具调用自动运行（无人工审核），现在可通过 `local.autoReview` 路由到classifier子Agent，该classifier决定哪些调用自动执行、哪些需要暂停审批。通过 `permissions.json` 的 `autoRun.allow_instructions` / `autoRun.block_instructions` 用自然语言定义审核策略，例如：
- 允许：`"Read-only inspections of build artifacts under ./dist are fine."`
- 拦截：`"Always pause delete operations so I get a chance to review them."`

**🪺 Nested Subagents（嵌套子Agent）**: 子Agent可以递归创建自己的子Agent，每层保持独立的prompt和模型配置。例如：reviewer子Agent → 委托test-writer子Agent → test-writer再委托更细粒度的子Agent。

**📦 JSONL Stores**: 新增JSONL持久化存储选项，替代默认SQLite。JSONL是纯文本、append-only文件，可读、可diff、可纳入版本控制。同时提供 `LocalAgentStore` 接口，可自定义实现（in-memory for CI、Postgres for production等）。

**🆔 requestId 追踪**: 每个 `send()` 携带平台生成的 `requestId`，跨in-memory/SQLite/JSONL所有存储持久化，将CI脚本或运行与后端日志、分析系统、支持工单关联。

**🏷️ 性能优化**: `@cursor/sdk` 导入不再急切加载完整的本地Agent栈（cloud-only/type-only消费者跳过），.d.ts文件不再引用未发布的workspace包（修复TS2305/TS2307），本地shell运行使用捆绑的ripgrep（不修改全局PATH）。

> **意义**: Cursor SDK从"可用的Agent SDK"进化为"可编程的Agent基础设施"，支持生产环境中的CI自动化、审核治理、多级Agent编排。这为企业在CI/CD中集成AI Agent提供了完整的工程化支撑。

## ② Cursor Design Mode 增强 — 多元素多选 + 语音排队交互（6月5日）

> **来源**: [Cursor Changelog — Design Mode Improvements](https://changelog.cursor.com/) | Jun 5, 2026

Design Mode在两个维度增强：

**多元素多选**: 点击浏览器中两个或多个元素，Cursor同时看到所选元素的代码、周围布局和页面上的视觉关系。可请求Agent让一个匹配另一个、删除重复内容或批量调整一组组件。

**语音输入**: 通过Design Mode覆盖层语音描述更改。mic在Agent运行期间保持可用，因此**无需等待前一个编辑完成**即可排队下一个语音指令——多个子Agent可并行工作。

> **意义**: 语音+多选的组合本质上创造了"非阻塞并行交互"模式，从「人等机器」变成「机器并行处理+人持续输入」，交互吞吐量大幅提升。

## ③ Cursor Canvas Design Mode + Context Usage Report（6月4日）

> **来源**: [Cursor Changelog — Canvas Design Mode and Context Usage Report](https://changelog.cursor.com/) | Jun 4, 2026

**Canvas Design Mode**: Agent创建的交互式画布（仪表盘、报告、内部工具）现支持Design Mode——直接在画布上选择UI元素进行标注引导编辑。

**Context Usage Report（上下文使用报告）**: Agent的上下文使用情况以交互式报告在Canvas中展示。上下文资源管理器分解token在系统提示、工具定义、rules、skills等各个部分的分布。内置"Debug with Agent"按钮可让Agent分析如何减少上下文使用量。

**Canvas增强**: 共享画布支持浏览器全屏展示、Agent可在画布中嵌入按钮（点击后执行指定prompt）、Agent修复canvas类型错误的能力提升、图表定制功能增加。

> **意义**: Context Usage Report首次提供了对Agent上下文使用量的可视化洞察，使开发者能诊断token占用问题并优化上下文预算。

## ④ Claude Code 2.1.154 — Dynamic Workflows + Opus 4.8 默认（5月28日）

> **来源**: [Claude Code Changelog v2.1.154](https://docs.anthropic.com/en/docs/claude-code/changelog) | May 28, 2026

**Dynamic Workflows（动态工作流）**: 当用户请求Claude创建一个workflow时，它可在后台编排**数十到数百个Agent**并行工作，支持更大、更复杂的任务。通过 `/workflows` 查看运行状态。

**Opus 4.8 作为默认模型**: 默认 `high effort` 模式，`/effort xhigh` 处理最困难任务。Fast Mode定价为**标准费率2倍换取2.5倍速度**。

**其他关键更新**:
- `/code-review --fix`：审查结果直接应用到工作区，同时报告reuse/simplification/efficiency建议
- Streaming tool execution 始终启用（之前需要feature flag）
- 插件从 `.claude/skills` 目录自动加载，无需marketplace
- 子Agent（Agent Tool）现在自动保留MCP配置继承

> **意义**: Dynamic Workflows标志着AI coding agent从"单会话单任务"到"多Agent并行编排"的范式升级，适用于大型重构、跨模块代码生成等复杂场景。

## ⑤ Claude Code 2.1.163-2.1.168 — Fallback Model + 版本治理（6月4-6日）

> **来源**: [Claude Code Changelog v2.1.163-168](https://docs.anthropic.com/en/docs/claude-code/changelog) | Jun 4-6, 2026

**Fallback Model（回退模型）**: 新增 `fallbackModel` 设置，可配置最多**3个回退模型**，当primary模型过载或不可用时按序尝试。`--fallback-model` 现在也适用于交互式会话。

**企业版本治理**: 新增 `requiredMinimumVersion` 和 `requiredMaximumVersion` 托管设置——当Claude Code版本超出允许范围时拒绝启动，引导用户安装经审批的版本。

**安全加固**: 
- 全局模式在deny rule的tool-name位置支持（`"*"` 拒绝所有工具）
- 跨会话消息安全：通过 `SendMessage` 从其他Claude会话中继的消息不再携带用户权限——接收者拒绝中继的权限请求，auto mode阻止它们
- `MAX_THINKING_TOKENS=0` 和 `--thinking disabled` 现在可禁用默认思考的模型

**并行工具调用修复**: 失败的Bash命令不再取消同批次的其他调用——每个工具独立返回结果。

> **意义**: Fallback Model 解决了API高负载时的服务降级问题；版本治理为企业在多环境部署AI Agent提供了合规管控能力，是Agent进入正式生产环境的关键基础设施。

## ⑥ Atlassian + Cursor 深度集成 — Cursor in Jira 正式发布（5月20日）

> **来源**: [Atlassian Blog — Introducing Cursor in Jira](https://www.atlassian.com/blog/announcements/cursor-in-jira) | Matthew Canham, May 20, 2026

**核心能力**: Jira团队可直接将工作项分配给Cursor Cloud Agent，形成 **Jira（编排层）→ Cursor（执行层）→ PR（产出）** 的闭环。

**四种使用模式**:
1. **Jira触发Agent**: 分配给 `@Cursor` 或评论中提及 `@Cursor` 即可启动Agent会话
2. **自动工作流**: 创建规则自动分配特定任务给Cursor（重复性工作/代码质量）
3. **全员赋能**: 任何团队成员可使用Cursor启动任务并开PR，无需本地环境
4. **说明驱动开发**: Confluence中的Agent可读规格自动同步到工作项

**Teamwork Graph 上下文集成**: 通过Atlassian Teamwork Graph CLI 或 Rovo MCP，Cursor可直接获取Jira issue、依赖项、Confluence页面等完整上下文。内部测试显示：
- **答案质量提升 44%**
- **Token消耗减少 48%**

**Atlassian首席AI产品官 Tamar Yehoshua 评价**: "Jira是工作定义的地方，Cursor是工程师构建的地方。现在这两者之间的上下文可以轻松共享。"

**Cisco工程运营VP Jason Andrews**: "问题不是如何让工程师使用更多AI，而是如何构建一个人类和Agent从相同上下文、向着相同目标工作的系统。"

> **意义**: 这是AI coding agent首次与项目管理工具实现深度双向集成，标志着Agent从「单兵作战」进化到「被PM工具编排、与团队协同」的阶段。Jira成为Agent编排层（orchestration layer），这一范式可能重塑软件开发工作流。

## ⑦ Atlassian MCP Compression — 工具描述Token消耗降低70-97%（3月29日）

> **来源**: [Atlassian Blog — MCP Compression: Preventing tool bloat in AI agents](https://www.atlassian.com/blog/development/mcp-compression-preventing-tool-bloat-in-ai-agents) | Tim Esler, Mar 29, 2026

**问题**: MCP Server工具描述极其消耗token——单个GitHub MCP Server暴露94个工具，每次请求消耗约 **17,600 tokens** 的工具描述。Atlassian MCP Server（Jira + Confluence）也消耗约10,000 tokens。

**方案 — mcp-compressor**: 将MCP Server的完整工具清单替换为两个通用wrapper工具：
- `get_tool_schema(tool_name)` — 按需获取单个工具的全部输入schema
- `invoke_tool(tool_name, tool_input)` — 按需调用

可选增加第三个辅助工具 `list_tools()`。

**Token压缩效果**（94工具的GitHub MCP示例）:

| 压缩级别 | Token用量 | 对比原始 |
|:---------|:----------|:---------|
| 原始（无压缩） | 17,600 | 100% |
| Low | 3,900 | 22% (↓78%) |
| Moderate | 3,300 | 19% (↓81%) |
| Strong | 2,200 | 13% (↓87%) |
| Max | 500 | 3% (↓97%) |

**四种详细程度级别**:
- `full`：工具名+参数名+完整描述（最低压缩，最高可发现性）
- `brief`：工具名+参数名+一行描述（适合大Server的平衡方案）
- `minimal`：仅工具名+参数名（更激进压缩）
- `none`：无嵌入工具列表，Agent必须显式调用 `list_tools()`

**质量影响测试**: 工具压缩对端到端质量"几乎没有影响"——只要保留按需schema访问能力，模型不需要所有可能的工具schema。

> **意义**: MCP工具膨胀是Agent规模化部署的核心瓶颈之一。mcp-compressor的发布解决了70-97%的token浪费，使得同时连接多个大型MCP Server成为可行方案。已作为开源项目发布在 github.com/atlassian-labs/mcp-compressor。
