## ① ⚡ **GitHub Actions: 自托管 Runner 最低版本强制执行时间表**（6月12日）

**来源**: [GitHub Changelog](https://github.blog/changelog/2026-06-12-github-actions-minimum-version-enforcement-timeline-for-self-hosted-runners/)
**标签**: `github-actions` `self-hosted-runner` `ci-cd-governance` `platform-engineering`

### 背景：Actions 架构重建已完成

GitHub Actions 团队从 2024 年初开始重建支撑作业执行和 Runner 通信的后端服务。新架构现已处理 **每天超过 1.2 亿个作业**，是迁移前的 **3 倍以上**，同时支持企业**每分钟启动 7 倍于之前的作业量**。强制执行最低版本是这一架构迁移的最后一步。

### 两个核心要求

| 要求 | 描述 | 影响 |
|:----|:-----|:----|
| **配置/注册 Runner** | Runner 版本必须 ≥ **2.329.0** | 低于此版本的 Runner 无法注册 |
| **持续执行作业** | Runner 必须在每个新版本发布后 **30 天内更新** | 未及时更新的 Runner 将停止接收作业 |

> 2.329.0 只是**注册最低要求**，不是永久可执行版本。Runner 不持续更新，即使已注册也不会收到新作业。

### 强制执行时间线

#### GitHub Enterprise Cloud with Data Residency
- **强制日期**: 2026年7月31日
- **Brownout 阶段**: 6月29日–7月24日（配置验证 → 配置+运行时验证）
- **强制后**: 低于最低版本的 Runner 无法注册/执行作业

#### GitHub Enterprise Cloud
- **强制日期**: 2026年9月25日
- **Brownout 阶段**: 8月24日–9月18日（配置验证 → 配置+运行时验证）

### 影响与应对

**影响范围**:
- 自托管 Runner 用户（github.com + GHEC + GHEC-DR）
- **GitHub Enterprise Server 不受影响**
- 启用自动更新的 Runner 自动满足 30 天要求

**需要做的事**:
1. 将所有自托管 Runner 升级到最新版本
2. 更新安装脚本/VM 镜像/容器镜像/部署自动化
3. 重新创建基于旧缓存镜像或模板构建的 Runner

**已提供的工具**:
- 运行时作业注释（在过期 Runner 上运行时会显示）
- REST API 新增 runner version 字段
- Audit Log 查询 runner 注册事件（`org.register_self_hosted_runner` / `repo.register_self_hosted_runner` / `enterprise.register_self_hosted_runner`）

### ⭐ 对研发管理的意义

> 这是 **CI/CD 基础设施治理的一次强制升级**——Agentic Workflows / Agentic Pipelines 等新一代 Agent 驱动的 CI/CD 系统完全依赖自托管 Runner 的可靠性和版本一致性。Runner 版本管理从"建议"变为"强制"，企业平台团队需要建立 Runner 版本跟踪和自动更新机制。

与上周发布（Jun 11）的 **Agentic Workflows** 直接关联——Agent 创建的代码需要通过 Actions 运行测试和部署，Runner 版本合规是前置条件。

## ② 🔐 **GitHub: Bot 创建的 PR 经批准后可运行 Workflows**（6月11日）

**来源**: [GitHub Changelog](https://github.blog/changelog/2026-06-11-bot-created-pull-requests-can-run-workflows-if-approved/)
**标签**: `agent-pr` `approval-gate` `ci-cd-security` `agent-governance`

### 更新内容

Pull Request 由 `github-actions[bot]` 创建后，现可经 **人工批准后自动运行 CI/CD Workflows**。

**此前的限制**:
- Bot 创建的 PR 无法触发 CI/CD Workflows
- 导致 Bot 生成的代码可能被**意外合并而不经过 CI 检查**

**更新后的行为**:
- Bot PR 经具有 **write 权限的用户批准**后即可运行 Workflows
- 需要批准是安全措施，确保生成的代码不会自动运行有敏感信息访问权限的 Workflows
- **与 Copilot 生成的 PR 行为一致**

### ⭐ 对研发管理的意义

> 这是 **Agent 代码进入生产的审批链闭环**——Agent 创建 PR → 人工审批 → 自动运行 CI/CD。结合 Agentic Workflows（六层安全架构）和 PAT-free（GITHUB_TOKEN 直结组织账单），GitHub 正在系统解决"Agent 写的代码如何安全地进入生产"这个核心问题。

**与本周其他治理更新的关系**:

```
Agentic Workflows 六层安全（Jun 11）
    ↓  Agent 可创 Workflow YAML
Bot-created PRs can run workflows if approved（Jun 11）
    ↓  Agent 创建的 PR 可触发 CI/CD
Copilot Code Review Runner 控制（Jun 12）
    ↓  企业控制 Agent 审查在哪个基础设施上运行
Self-hosted Runner 版本强制（Jun 12）
    ↓  确保 CI/CD 基础设施版本合规
```

**四层递进形成了一个完整的"Agent CI/CD 安全治理链"**。

## ④ 🔗 **Atlassian: Agentic Pipelines 增加 Claude Code 支持**（May 19 公布，当前 Open Beta）

**来源**: [Atlassian Blog — Agentic Pipelines now supports Claude Code](https://www.atlassian.com/blog/bitbucket/agentic-pipelines-now-supports-claude-code)
**标签**: `bitbucket-pipelines` `agentic-pipelines` `flaky-test` `claude-code` `ci-cd-agent`

### 更新内容

Agentic Pipelines 从最初的 Rovo Dev-only 扩展为支持 **Claude Code** 作为 Agent 提供商。

**配置方式**（`bitbucket-pipelines.yml`）:

```yaml
definitions:
  agents:
    my-agent:
      prompt: "Explain this repository"
      provider: claude  # ← 添加这一行即可使用 Claude Code
pipelines:
  default:
    - step:
        script:
          - agent: my-agent
```

### 深度案例：Flaky Test 自动修复

Agentic Pipelines 内置的 **Flaky Test 修复 Agent** 工作流程：

1. **信息收集**：测试的完整执行历史、失败模式、周围代码上下文
2. **复现执行**：运行测试，观察失败
3. **根因假设**：时序问题/共享状态/环境依赖/测试顺序等
4. **定向修复**：提出计划并进行有针对性修改
5. **验证**：再次运行测试确认修复生效
6. **创建 Draft PR**：打开草稿 PR，自动@触发修复的人作为 Reviewer

**配置触发**（仅需在 `bitbucket-pipelines.yml` 中添加 `fix-flaky-test` trigger）:

```yaml
triggers:
  fix-flaky-test:
    - condition: "<custom-condition>"
      pipelines:
        - fix-flaky-test-pipeline
```

### ⭐ 对研发管理的意义

> 这是 **CI/CD 层面的 Agent 编排平台化**的典型代表。Bitbucket Pipelines 从"CI/CD 自动化引擎"转变为"Agent 编排平台"，覆盖的不再只是构建和测试，而是**代码创建前后的所有低价值高工作量重复性工作**（更新 README、分类安全报告、清理 Feature Flag、生成 PR 描述、修复 Flaky Test）。

**对各厂商 CI/CD Agent 化的对比**:

| 维度 | GitHub Agentic Workflows | Atlassian Agentic Pipelines |
|:-----|:------------------------|:---------------------------|
| 发布时间 | Jun 11（公开预览） | May 5（公开 Beta） |
| Agent 提供商 | Copilot | Rovo Dev + Claude Code |
| 配置方式 | Markdown → Actions YAML | YAML agent step |
| 安全体系 | 6 层安全架构 | Third-Party Product 协议 |
| 核心场景 | CI 失败分析/文档更新/多仓库变更 | Flaky Test/README/Feature Flag/安全报告 |
| **关键差异** | Markdown 转 YAML 的自然语言范式 | YAML 定义 Agent 的声明式范式 |
