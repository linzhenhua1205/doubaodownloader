## 一、Cursor 3.9（Jun 22）— Customize 统一管理 + Plugin Canvases

> **来源**: [Cursor Changelog](https://changelog.cursor.com/)
> **类别**: 研发工具 · AI编程工具更新 · Enterprise治理

Cursor 3.9 的核心主题是 **统一自定义管理**，将之前分散的配置入口全部收敛到新的 Customize 页面。

### 1️⃣ Customize 页面：统一配置中心

将 **Plugins、Skills、MCPs、Subagents、Rules、Commands、Hooks** 全部纳入一个管理页面：
- 支持 **User / Team / Workspace 三级作用域**
- 可直接添加自有自定义 MCP
- 所有类型的扩展在同一界面管理，减少心智负担

### 2️⃣ Marketplace Leaderboard

团队内显示最流行的插件、Skills、MCP 排行榜：
- 一键从 Customize 页面添加到自己的配置
- 推动团队级扩展发现的"社会证明"机制

### 3️⃣ Plugin Canvases

插件现在支持**预构建的共享 setup 模板（Canvases）**，团队可以打开并复用：

| Canvas 类型 | 用途 |
|:-----------|:-----|
| **Hex Canvas** | 构建数据可视化 |
| **Atlassian Canvas** | 实时查看所有 Issues、Projects、Documents |

### 4️⃣ Team Marketplaces 跨平台导入

团队 Marketplace 现在支持从以下平台**导入插件仓库**并分发给团队：
- **GitLab**
- **BitBucket**
- **Azure DevOps**

- 📌 **【深度跟踪】** Cursor 3.9 标志着 Cursor 从"个人编辑器工具"向**团队平台**的关键转变。Customize 统一管理 + 三级作用域（User/Team/Workspace）+ 跨平台插件仓库导入 + 排行榜，这三件事拼在一起构成了 Cursor Enterprise 治理的基础设施。结合 3.8 的 Automations 和 3.7 的 Cloud Subagents，Cursor 已经建立了一个**完整的 Agent 平台栈**：Local Agent（交互）→ Cloud Subagent（后台并行）→ Automation（事件驱动）→ Enterprise Governance（统一管理+治理）。

## 二、Cursor Design Mode（Jun 5）— Multi-select + Voice Input

> **来源**: [Cursor Changelog](https://changelog.cursor.com/)
> **类别**: 研发工具 · AI编程工具更新

Cursor Design Mode 新增两个关键交互能力：

### Multi-select Elements
在浏览器中同时点击 2 个以上元素，Cursor 同时看到：
- 选中元素的代码
- 周围布局
- 页面上的视觉关系

可要求 Agent：让一个匹配另一个、移除重复内容、或同时调整一组组件。

### Voice Input
通过 Design Mode 覆盖层用语音描述更改。**Mic 在 Agent 运行中也可用**——无需等待前一次完成即可通过语音排队下一次更改。

- 📌 **【深度跟踪】** Multi-select + Voice Input 将 Design Mode 的交互模式从"对话式（描述→等待→下一个）"升级为**操作式（选中→语音→并行）**，更接近设计师使用 Figma/Sketch 的直觉交互模式。这是 UI 开发"所见即所得"体验的重要演进方向。

## 三、Claude Code 密集更新（Jun 2→25）— Fable 5 + Agent Teams 简化 + MCP CLI + 安全加固

> **来源**: [Claude Code Changelog](https://docs.anthropic.com/en/docs/claude-code/changelog) — 版本 2.1.161→2.1.193
> **类别**: 研发工具 · AI编程工具更新 · Enterprise治理

过去三周 Claude Code 发布了 **33 个版本**（2.1.161→193），几乎是每天一个版本。以下按主题聚合关键更新：

### 🆕 Fable 5 Mythos-class 模型（Jun 9, v2.1.170）

> 「Claude Fable 5：一个 Mythos-class 模型，能力超过 Anthropic 迄今所有公开发布的任何模型。」

这是继 Opus、Sonnet 之后的新能力层级（Mythos），标志着模型能力新台阶。需 v2.1.170+ 版本访问。

### 🆕 MCP CLI 管理（Jun 22, v2.1.186）

新增 `claude mcp login <name>` 和 `claude mcp logout <name>`：
- 从 CLI 完成 MCP Server 鉴权，无需进入 `/mcp` 交互菜单
- 支持 `--no-browser` stdin redirect 用于 SSH 环境
- 新增 `claude mcp get/remove` 命令，支持 server name 模糊匹配

**影响**：MCP Server 从交互式 GUI 配置进入 CLI 自动化管理阶段，为 CI/CD 和自动化流水线中的 MCP 使用铺平道路。

重大架构简化：
- **移除了 TeamCreate / TeamDelete 工具**
- 设置 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 后，每个会话隐式拥有一个团队
- 直接用 Agent 工具的 `name` 参数 spawn teammate，无需 setup 步骤
- Sub-agents 深度限制：最多 5 层嵌套（v2.1.172）
- 旧版 `team_name` 参数仍被接受但会被忽略

### 🆕 /cd 命令（Jun 8, v2.1.169）

新增 `/cd <dir>` 命令：
- 在会话中切换工作目录，**不破坏中间的 prompt cache**

## 四、Claude Agent for Jira（Jun 18）— 首个 Project-to-Code 全链路闭环

> **来源**: [Atlassian Blog — Introducing Claude Agent for Jira](https://www.atlassian.com/blog/jira-software/claude-agent-for-jira)
> **类别**: 研发工具 · Agent工程化实践 · Enterprise治理

Atlassian 发布了 **Claude Agent for Jira**，基于 Anthropic 的 Claude Managed Agents 基础设施构建。

### 核心工作流

```
Jira Issue 分配 → Claude Agent 读取上下文 → 沙箱执行 → 打开 Draft PR → 人工 Review + Merge
```

### 关键架构特性

| 特性 | 说明 |
|:-----|:------|
| **分配方式** | 注释中@Claude、Assignee 下拉选择 Claude、自动化规则自动分配 |
| **上下文来源** | 直接从 Issue 读取：接受标准、目标仓库、设计/工程标准文档 |
| **执行环境** | Anthropic 隔离基础设施中的沙箱——克隆仓库、分析代码库、实现变更、推送代码 |
| **Review 机制** | 打开 Draft PR（GitHub），实时状态同步回 Jira Issue。工程团队保留代码 Review、验证、合并的完全控制权 |
| **上下文保持** | 无论从 Jira Cloud 还是本地 IDE 启动，Agent 始终拥有完整的 Issue 上下文 |

### 纵向研究数据：最值得关注的数据点

> 「一项覆盖 **400 个工程组织**的纵向研究发现，即使 AI 编码采纳率高达 **90%**，**整体生产力提升仍然徘徊在 10-15%**。对于常规任务，开发者通常花费大量时间将代码库和业务上下文复制到聊天界面、设置本地分支、将 Issue 描述翻译成 Agent 可读的 spec。」

**关键解读**：
- **90% 采纳率 × 10-15% 提升** = AI 编码工具的边际收益正在递减
- 瓶颈不在"编码"，而在**上下文传递**（从 Issue 到 Agent 的翻译成本）
- Claude Agent for Jira 的核心价值主张就是消除这种 Context Switching——将 Issue 直接连接到代码执行

### 影响分析

这一发布标志着：
1. **Issue-to-PR 的上下文传递成本**是当前 AI 编码的最大瓶颈（400 组织研究揭示）
2. **Atlassian + Anthropic 合作** = 项目管理平台 × AI 编码基础设施，两个层面的数据打通
3. Claude Agent for Jira 的**沙箱执行** + **Draft PR** 流程是当前最安全的 Enterprise Agent 实现模式

- 📌 **【深度跟踪】** **400 组织纵向研究是今天最重要的数据点**。它揭示了一个反直觉的事实：即使几乎所有开发者都在用 AI 编码工具，生产力提升仅 10-15%。这意味着当前 AI 编码工具解决的只是"写代码"这个相对容易的问题，而"理解上下文→转化为可执行规范"这个更难的问题尚未被解决。Claude Agent for Jira 和 Cursor Automations 的兴起，本质上都是在用不同的方式解决同一个问题——**减少 Agent 和业务上下文之间的摩擦**。

## 五、Atlassian：The Bottleneck Keeps Shifting（May 6）— PM:Eng 1:4 时代到来

> **来源**: [Atlassian Blog](https://www.atlassian.com/blog/how-we-build/the-bottleneck-keeps-shifting-what-ai-is-changing-about-how-we-build)
> **类别**: AI辅助研发工作流 · HBR AI组织管理研究

Atlassian CPO Tamar Yehoshua 与 Gamma 联合创始人 Jon Noronha 的对话，揭示 AI 时代研发瓶颈的实时转移。

### 关键发现

#### PM:Engineer 比例从 1:10 转向 1:4
> 「Gamma 目前 PM 与工程师的比例是 **1:4**，而几十年的经验法则一直是 1:10。我认为 PM 没有像工程师那样提速 2-3 倍，因为 PM 处理的是更多的人类约束。」

> 「一个 PM + 4 个工程师的团队，正在完成以前 8-10 个工程师才能完成的工作。」

#### PM 开始写代码
- Gamma **一半的 PM 和设计师开始提交代码**
- PM 用 AI 做 **bug triage**：客户报告 Bug → PM 用 AI 做完整调查 → 决定是否需要工程师介入
- 以前 PM 需要咨询工程师代码工作原理，现在可以直接问 AI

#### "Context Wrangler" 新角色
Jon Noronha 将自己定位为 **context wrangler**（上下文管理员）：
> 「我觉得我的工作就是把只在我脑子里的信息写成文档，让我的团队成员和他们的 Agent 都能访问。」

这不仅是领导力的新定义，也是 AI 时代知识管理的核心挑战——**信息必须被记录下来，Agent 才能有效运作**。

#### PM 的新超能力：Discernment（辨别力）
- 工程产出的瓶颈不再是"能不能写出来"，而是"该不该写"
- PM 的超级能力从"协调开发"转向**辨别力**——决定该构建什么、不该构建什么
- 实验和主观评价仍然至关重要——用户在 A/B 测试中说的和做的经常不一致

- 📌 **【深度跟踪】** "1:4 比例 × PM 写代码 × Context Wrangler" 三者拼在一起描绘了一个**新型研发组织形态**：(1) AI 将编码效率提升 2-3× → 工程产出不再是瓶颈 → (2) 瓶颈转移到"决定该做什么" → (3) PM 需要更快地将意图转化为可执行规范 → (4) PM 开始直接用 AI 编码工具执行 → (5) 组织需要更好的上下文记录能力来支持 Agent 和人类协作。这形成了一个正反馈循环：**编码效率↑ → 决策瓶颈↑ → PM 能力升级 → 上下文管理需求↑ → Agent 效能↑**。

2. **策略控制**：组织级工作流策略管理

### 值得重温：Agent-Driven Development 方法论（Mar 31）

GitHub Copilot Applied Science 团队的核心方法论已在昨日（6/25）详细记录。今天补充两个未覆盖的洞见：

**Copilot Applied Science 的关键教训：**
> 4 人 × <3 天 = 11 个 Agent + 4 个 Skill + 28K LOC

核心方法论三支柱：
1. **Prompt 策略**：对话式而非指令式，Agent 擅长范围明确的问题
2. **架构策略**：重构优先于新功能——Agent 对整洁代码库的表现远优于混乱代码库
3. **迭代策略**：责备流程而非 Agent——Agent 犯错 → 增加 guardrail

**Agent 工作流闭环**：
```
/plan → 迭代 Plan → 先写文档再写代码 → /autopilot 实施 → Copilot Code Review → 循环修复 → 人工审查 → 合入
```

**核心洞察**：Agent 不是替代工程纪律，而是强制纪律——因为 Agent 对混乱代码库的表现远差于整洁代码库。

- 📌 **【深度跟踪】** GitHub 的 Agent 方法论与 Cursor 的 Automations + Cloud Subagents 形成对比：GitHub 更强调**工程纪律 + 文档优先**，Cursor 更强调**环境弹性 + 事件驱动**。两者互补而非对立——最好的 Agent 工作流应该同时具备纪律和弹性。
