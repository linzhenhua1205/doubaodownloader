## 一、Cursor 3.9（Jun 22）— Customize Page 统一插件管理

> **来源**: [Cursor Changelog](https://changelog.cursor.com/)
> **类别**: 研发工具 · AI编程工具更新 · 团队协作基础设施

Cursor 3.9 的主要更新是 **Customize 页面**，将之前分散的配置入口统一到一个界面：

### 1️⃣ 统一的 Customize 页面

| 能力 | 说明 |
|:----|:----|
| **统一管理** | 用户/团队/工作空间级别统一管理 plugins、skills、MCPs、subagents、rules、commands、hooks |
| **自建 MCP** | 支持带入自己的自定义 MCP 服务 |
| **一站式操作** | 所有配置在同一个页面完成，不再需要在多个菜单中切换 |

### 2️⃣ Marketplace Leaderboard

Cursor 新增团队内的 **Marketplace 排行榜**，展示最受欢迎的 plugins、skills 和 MCPs。团队成员可以一键添加到自己的配置中。

### 3️⃣ Plugin Canvases（插件画布）

插件现在支持**预构建画布**——团队可以共享和复用的设置模板：
- **Hex Canvas**：构建数据可视化
- **Atlassian Canvas**：实时查看所有 Issues、项目、文档

### 4️⃣ 团队市场多平台导入

Team Marketplaces 新增支持从 **GitLab、BitBucket、Azure DevOps** 导入插件仓库，使大企业能够在内部基础设施上分发插件。

**🛠️ 产品分析**: Cursor 3.9 的三项更新（统一管理页、市场排行榜、画布/多平台导入）构成了一条清晰的**团队采用基础设施**升级路径。从"个人 AI 工具"向"企业 AI 开发平台"的转型持续推进。

## 二、Claude Code v2.1.186-187（Jun 22-23）— MCP CLI 认证 + 凭证沙箱

> **来源**: [Claude Code CHANGELOG](https://docs.anthropic.com/en/docs/claude-code/changelog)
> **类别**: AI编程工具更新 · Agent工程化实践 · 安全治理

### 2.1.187（Jun 23）— 凭证安全与超时

| 特性 | 说明 | 影响 |
|:----|:-----|:-----|
| **`sandbox.credentials` 设置** | 阻止沙箱化命令读取凭证文件和 secret env vars | 🚀 **企业安全里程碑** — 首次将凭证文件级别的访问控制加到沙箱中 |
| **模型限制显示** | org 配置的模型限制在 model picker、`--model`、`/model`、`ANTHROPIC_MODEL` 中显示 | 企业感知：用户选受限模型时展示"受组织设置限制" |
| **MCP 远程调用超时** | 远程 MCP 工具调用 5 分钟无响应后**自动中止并报错**（而非无限阻塞） | 可用性改进；可覆盖 `CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT` |
| **工作树泄漏清理** | 被 Kill 的 Agent 残留 `.git/worktrees/` 条目自动清理 | 工程整洁度改进 |
| **`/plugin` 改进** | 展示近期未使用的插件以便清理 | 工程整洁度改进 |

### 2.1.186（Jun 22）— MCP CLI 认证 + 技能改进

| 特性 | 说明 |
|:----|:-----|
| **`claude mcp login/logout` CLI** | 从 CLI 直接认证 MCP 服务器，无需打开交互式 `/mcp` 菜单；支持 `--no-browser` 用于 SSH 环境 |
| **Skills 标签** | `/plugin` 的已安装标签页新增 "Skills" 分区 |
| **`! bash` 自动回复** | 以 `!` 开头的 bash 命令现在**自动触发 Claude 回复输出**（不再只是上下文保持） |
| **子 Agent 权限提示** | 后台子 Agent 的权限提示会悬浮在主会话中显示，不再自动拒绝；Esc 仅拒绝该工具 |
| **记忆压缩提醒** | Agent 在接近大小限制时被提醒压缩 MEMORY.md 索引 |
| **Skill frontmatter 兼容** | `display-name`, `default-enabled`, `fallback`, `metadata.*` 现在接受 kebab-case、snake_case、camelCase |
| **远程调用改进** | 远程控制连接失败显示持久红色 `/rc failed` 提示 |
| **`/workflows` 详情视图** | 支持状态过滤（按 `f` 键） |

### 2.1.183（Jun 19）— 破坏性命令自动阻断

> **来源**: 同 CHANGELOG
> **类别**: Agent工程化实践 · 安全治理

| 破坏性操作 | 阻断条件 |
|:----------|:---------|
| `git reset --hard` | 用户未要求丢弃本地工作 |
| `git checkout -- .` | 用户未要求丢弃本地工作 |
| `git clean -fd` | 用户未要求丢弃本地工作 |
| `git stash drop` | 用户未要求丢弃本地工作 |
| `git commit --amend` | commit 不是本会话 Agent 创建的 |
| `terraform destroy` | 用户未要求该特定 stack |
| `pulumi destroy` | 用户未要求该特定 stack |
| `cdk destroy` | 用户未要求该特定 stack |

**🛠️ 产品分析**: 2.1.186 的 `claude mcp login/logout` 标志着 Anthropic 将**连接管理**（MCP 生态的关键门槛）提升为一级 CLI 能力。结合 2.1.187 的凭证沙箱和 2.1.183 的破坏性命令阻断，Claude Code 的企业安全三层体系日渐清晰：**工具级权限（Tool:param）→ 命令级阻断（git/terraform）→ 文件级沙箱（credentials）**。

## 四、Product School — PM-AI 技能缺口职业分化加速

> **来源**: [Product School Blog](https://www.productschool.com/blog)
> **类别**: 研发效能度量 · AI辅助研发工作流

### 4.1 Claude Code for Product Managers（Jun 11）

CEO Carlos Gonzalez de Villaumbrosia 的长文中提出了一个关键论断：

**"真正的分界线不再是技术型PM和非技术型PM，而是AI-capable PM 和 AI-aware PM。"**

| AI-capable PM | AI-aware PM |
|:-------------|:------------|
| 独立验证原型 | 仍然等待工程容量 |
| 直接查询数据 | 依赖分析师 |
| 带着真实成果进会议室 | 带着规格文档进会议室 |
| 将用户访谈转化为结构化报告（分钟级） | 手动标记转录稿 |
| 部署自主 Agent 监控产品健康 | 手动检查 Dashboard |

#### 关键数据点：
- **LinkedIn 2026年1月劳动力市场报告**：产品管理是**AI技能渗透率 Top 3** 的职位功能，达 **10%**，**与 Engineering 持平**
- **Lightcast 岗位数据**：要求 AI 技能的岗位发布量 2023→2024 增长 **73%**，2024→2025 再增长 **109%**
- Claude Code 为 PM 带来的**六大工作流转换**：用户研究→结构化洞察（分钟级）→原型验证（小时级）→代码库阅读→数据查询（自然语言）→PRD 流程自优化→自主 Agent 监控

### 4.2 The AI Operating Model — 80% 产品团队卡住

该文指出当前产品团队在 AI 运营模型上呈**极端分布**：
- 80% 团队卡在 L1-L2（AI 作为辅助工具，个体使用）
- 仅 Top 1% 团队突破到 L4-L5（AI 作为开发流程的核心基础设施，改变团队结构和交付模式）
- 瓶颈不是工具本身，而是**运营模式**

### 4.3 "AI Builder" 新角色

**AI Builder** 是一个使用 AI 进行原型设计、分析、评估和交付的产品型角色，将产品思维转化为制品，速度远超传统构建周期。

**🛠️ 要点**: Product School 正在系统性地将 AI 工具能力与 PM 职业发展绑定（推出 Claude Code 认证课程）。LinkedIn 数据表明 PM-AI 融合已从趋势变为**人才市场硬指标**。

## 五、Atlassian — 团队仪式驱动 AI 对齐 + 实操工作坊

> **来源**: [Atlassian Work Life Blog](https://www.atlassian.com/blog/teamwork)
> **类别**: AI辅助研发工作流 · 研发效能度量 · HBR AI组织管理研究

### 5.1 团队仪式驱动 34% AI 对齐提升

Atlassian Teamwork Lab 的发现：给现有的团队仪式（如站会、回顾会）加上 **AI 话题环节**，使 AI 对齐度提升 **34%**。

核心发现：
- 与其推出新的 "AI 计划"，不如升级**已有的团队仪式**
- 关键是将 AI 使用从个人行为变为**团队讨论的话题**
- 定期对话比任何正式培训更能改变行为

### 5.2 实操工作坊提升 AI 信心

"当团队通过动手学习时，AI 不再感觉抽象。" 实操工作坊带来：
- 更高的 AI 工具使用率
- 更强的日常技能
- 真实的行为改变

### 5.3 其他重要内容

- **Ace Hardware 现代化 IT**：使用 Jira Service Management 实现 IT 现代化，含 Salesforce 集成与自动化
- **Nonprofit Teams 2026 报告**：团队如何实施 AI（而非是否实施）决定了 AI 带来的价值
- **Brain drain: Zombie Projects**：僵尸项目正在吃掉团队生产力（方法论参考）

**🛠️ 要点**: Atlassian 的实证研究提供了两条成本极低的"助推"路径：**改造现有仪式**（而非新建项目）和**动手工作坊**（而非讲座）。这两条路径对研发效能转型有直接借鉴价值。

## 六、HBR — 组织变革五阶段 + 技术局限增长

> **来源**: [HBR](https://hbr.org/)
> **类别**: HBR AI组织管理研究

### 6.1 Designing Organizational Change That Actually Sticks（May 29）

BCG 合伙人 Julia Dhar 和 Kristy Ellmer 的 HBR Executive Masterclass 提出组织变革的**五阶段框架**：

1. **诊断**（Diagnosis）— 理解人们为何会抗拒变化
2. **设计**（Design）— 将变革设计得符合人的心理模式
3. **试运行**（Pilot）— 在小范围内验证
4. **规模化**（Scale）— 逐步推广
5. **固化**（Sustain）— 将变革嵌入文化和流程

核心论点：大多数变革失败不是因为缺乏策略，而是因为**误解了人们如何体验变革**——领导者低估了变革带来的不确定性和认知负担。

### 6.2 When Technology Limits Growth（Jun 5, 2026）

"当技术限制增长：重新思考上云的案例"——过时的系统不仅仅是成本问题，它们会：
- 抑制创新
- 使数据变得不可用
- 在团队中制造摩擦
- 引发对"看起来很差劲"的变革的抵制

**🛠️ 要点**: HBR 的内容为研发效能度量提供了**组织心理学的理论支撑**——我们在跟踪技术效率的同时，也需要跟踪组织变革的心理阻力。
