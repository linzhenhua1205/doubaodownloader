## ① 🚀 **GitHub Agentic Workflows 公开预览 — 自然语言Markdown定义Agent流水线**（6月11日）

**来源**: [GitHub Changelog](https://github.blog/changelog/2026-06-11-github-agentic-workflows-is-now-in-public-preview/)

**技术详情**:
- 用 **自然语言 Markdown 文件** 定义自动化逻辑，GitHub Agentic Workflows 自动编译为标准 **Actions YAML**
- 覆盖场景：**Issue 分类、CI 失败分析、文档更新、多仓库变更**
- 重复使用现有 **Runner Groups** 和 **Policy Constraints** → 零迁移成本
- 安全体系（6层）:
  - ✅ **Integrity Filter** — 完整性过滤规则
  - ✅ **默认只读权限** — 最小权限原则
  - ✅ **沙箱容器** + **Agent Workflow Firewall** — 运行时隔离
  - ✅ **Safe Outputs 验证** — 输出安全校验
  - ✅ **Dedicated Threat Detection Job** — 专用威胁检测，扫描所有变更后提交
- **PAT 不再需要**（同一天发布）：Agentic Workflows 现在可以使用 **GITHUB_TOKEN**，AI 信用直结组织账单（需启用 `copilot-requests: write` 权限） → 消除长期 PAT 的安全风险
- **500 个成本中心上限**（同日发布，从 250 翻倍）

**数据**:
- Carvana（Alex Devkar, SVP Engineering）: "跨多仓库的工程任务以前需数小时，现在自主完成，团队可专注创新"
- Marks & Spencer（James Hoare, CTO）: "之前每 sprint 数小时用于 issue 分类、漏洞修复、依赖维护，现已构建可复用工作流目录"

**意义**: 这是 GitHub 从 Copilot 补全 → Chat → Agent Tasks → **Agent Workflows** 的第四次跃迁。Agent 从"辅助写代码"进化为"CI/CD 流水线的一等公民"。

## ② 🎛️ **Copilot CLI `/settings` — 统一Schema驱动的配置管理**（6月11日）

**来源**: [GitHub Changelog](https://github.blog/changelog/2026-06-11-copilot-cli-configure-everything-from-one-place-with-settings/)

**技术详情**:
- 替代零散的 `/theme`、`/streamer-mode`、`/experimental` 等命令
- 三种使用方式：
  - `/settings` → 全屏、分节对话框，按类型提供专属编辑器（Boolean 开关、枚举选择器、多行文本、数组编辑器、JSON 回退）
  - `/settings <key> <value>` → 内联单行设置
  - `/settings reset <key>` → 恢复默认
- Key 为 Schema 中的 **dotted paths**，支持 **Tab 自动补全** + 描述 + 可选值提示
- 设置变更 **即时生效**（如 colorMode 或 streamerMode 保存即切换）
- **Schema 验证**：设置文件仅在新值通过校验后才写入 → typo 不会静默破坏下一会话

**关键细节**:
- `/` 搜索、`Ctrl+R` 恢复默认、`Ctrl+E` 打开设置文件直接编辑
- `/settings security-review.autoScan true` 打开一处即可启用

**意义**: Copilot CLI 从"命令行工具"向"完整开发环境"演进，配置界面专业化程度接近 IDE。

## ③ 🔬 **Claude Code v2.1.174 — Skill Hot-Reload 粒度优化 + VS Code 用量归因**（6月12日）

**来源**: [Claude Code Changelog](https://docs.anthropic.com/en/docs/claude-code/changelog)

**技术详情**:
- **Skill hot-reload 增量更新** (关键修复): 原实现每次变更都重发整个技能列表，现**仅重发变更的技能** → 大型技能目录下显著减少不必要的上下文刷新
- **VS Code `/usage` 用量归因对话框**：新增 **Cache Misses、Long Context、Subagents、Per-Skill/Agent/Plugin/MCP** 过去 24h/7d 的分组用量钻取 → 企业用户可精确定位成本来源
- **v2.1.175**（同一天）: `enforceAvailableModels` 托管设置——启用后 availableModels 白名单也约束 Default 模型（Default 若解析到禁止模型则回退到第一个允许模型），且用户/项目设置不能拓宽托管 availableModels 列表
- **v2.1.173** (6月11日): Fable 5 模型名 `[1m]` 后缀自动标准化（Fable 5 默认包含 1M 上下文）

**意义**: Claude Code 在企业治理粒度上持续深化——从"能不能用"（allowed models）到"成本归属"（usage attribution by skill/agent/plugin）再到"配置锁死"（enforceAvailableModels），企业管控能力完整闭环。

## ⑤ 🤝 **Atlassian: Cursor in Jira — Jira 作为 Agent 编排平台**（5月20日，6月持续发酵）

**来源**: [Atlassian Blog](https://www.atlassian.com/blog/announcements/cursor-in-jira)

**技术详情**:
- **Jira → Cursor Agent 直接指派**：@Cursor 或 @Cursor 评论 → 启动 Agent 会话
- **自动工作流**：创建规则自动将特定任务指派给 Cursor（重复性工作/代码质量）
- **Rovo × Teamwork Graph 上下文丰富**：Agent 利用 Teamwork Graph 上下文达到 **44% 质量提升 + 48% Token 减少**
- **Spec-Driven Development**：自动同步 Confluence 中的 agent 可读 spec 到 Jira 工作项
- 基于多年 DX 研究的核心发现：开发者速度未能跟上模型能力，因为 **Agent 缺乏上下文**——上下文切换、规划、对齐、Bug 分类、审查是 IDE 外的主要摩擦点

**关键引用**:
> "问题不是如何让工程师多用 AI，而是如何构建一个人和 Agent 从同一上下文、朝同一目标工作的系统。" — Jason Andrews, VP Eng Ops, Cisco
