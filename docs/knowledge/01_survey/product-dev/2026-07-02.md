## 1️⃣ AI Agent 何时有用：三条件决策框架

**来源**: Product School — "When Should Product Teams Use AI Agents?"
**日期**: 2026年 | **类型**: 方法论框架

### 核心观点
AI Agent 不是所有场景的答案。产品团队应使用三条件测试来判断是否适合用 Agent：

| 条件 | 说明 | 反例 |
|:-----|:-----|:-----|
| 🎯 **开放目标** | 问题没有唯一固定答案，需要多步探索 | 有固定答案的问题 → 传统 ML 模型 |
| 🔄 **多步推理** | 需要将任务分解为多个步骤，前后依赖 | 单步预测/分类 → API 调用即可 |
| 🌐 **环境反馈** | Agent 可观察执行结果并自我修正 | 无反馈信号 → 无法闭环优化 |

### 管理启示
- Agent ≠ 万能工具，强行套用会导致过度工程化
- Agent 的价值在于**自主探索和自适应**，不是自动化执行
- 产品团队应优先用传统 ML/规则引擎解决确定性高的问题
- Agent 适合**复杂、不确定、多步骤**的业务场景

## 2️⃣ AI 实验方法论升级：非确定性输出的测试体系

**来源**: Product School — "AI Experimentation: How AI PMs Test and Learn Faster"
**日期**: 2026年 | **类型**: 实战方法论

### 核心观点
AI 产品的非确定性（non-deterministic）输出本质，要求 PM 建立新的实验体系：

| 实验类型 | 何时用 | 测什么 |
|:---------|:-------|:-------|
| 🧪 **离线评估** | 上线前 | 准确率、BLEU/ROUGE、对抗测试 |
| 🔄 **在线实验** | 生产中 | A/B 测试、用户行为指标 |
| 🛡️ **对抗式测试** | 持续 | Red team、边缘案例、安全边界 |
| 📊 **持续监控** | 生产后 | 漂移检测、回归捕获 |

### 关键要点
- 传统 A/B 测试不够：AI 输出每次不同，需要**统计显著性阈值调整**
- 用户感知指标（满意度/信任度）比纯准确率更重要
- **离线评估不能替代在线实验**——用户行为受 AI 输出影响后，环境会变化
- 建议建立三层实验管道：离线 → 灰度 → 全量

### 来源链接
- https://www.productschool.com/blog/artificial-intelligence/ai-experimentation-how-ai-pms-test-and-learn-faster

## 3️⃣ AI Builder：重塑产品创造方式的新角色

**来源**: Product School — "What Is an AI Builder? The New Role Reshaping How Products Get Made"
**日期**: 2026年 | **类型**: 角色定义

### 核心观点
AI Builder 是产品专业人员用 AI 工具（Claude Code、Copilot、Cursor 等）将产品思维直接转化为可运行产物的新角色范式。

**传统 PM vs AI Builder：**

| 维度 | 传统 PM | AI Builder |
|:-----|:--------|:-----------|
| 输出 | PRD + 原型 + 文档 | 可运行原型 + 实验代码 + 数据 |
| 周期 | 周级（PRD→设计→开发→验证） | 小时级（Prompt→产物→测试→迭代） |
| 核心能力 | 沟通协调、文档撰写 | 描述精度、Prompt 工程、快速验证 |
| 团队依赖 | 高度依赖工程/设计资源 | 低依赖，独立探索验证 |
| 风险 | 需求传译失真 | 原型与生产差距 |

### 管理启示
- PM 需要学「用 AI 做原型」——这是新的基本技能集
- 描述精度（Precision of Description）成为 PM 最核心的新能力
- AI Builder 模式下 PM 的职责从「文档管理者」变为「产品加速器」
- 组织需重新定义 PM 的能力模型和招聘标准

### 来源链接
- https://www.productschool.com/blog/artificial-intelligence/ai-builder-new-role

## 4️⃣ Claude Code for Product Managers：新一轮工具变革

**来源**: Product School — "Claude Code for Product Managers: Why It Matters"
**日期**: 2026年 | **类型**: 实操指南

### 核心观点
Claude Code 代表了 AI 辅助开发的范式跃迁——从对话式聊天（Chat）到**直接操作用户文件的 Agent**。

### 对 PM 的意义
| 变化 | 说明 |
|:-----|:-----|
| 🔨 **原型能力跃升** | PM 可直接用自然语言生成交互原型、数据模拟、UI 代码 |
| 🔍 **数据自取** | PM 可自行分析用户数据、查询数据库、生成报告 |
| 📐 **规格→代码** | 需求规格可直接转化为可执行的代码原型 |
| 🔄 **验证周期压缩** | 从「写 PRD → 排期 → 开发 → 测试」到「描述 → 生成 → 验证」 |

### 管理启示
- Claude Code 类工具将降低 PM 对工程资源的依赖
- 但需要 PM 提升「描述精度」——模糊的需求会导致错误产物
- 组织应鼓励 PM 学习原型工具链，建立新的最佳实践

### 来源链接
- https://www.productschool.com/blog/artificial-intelligence/claude-code-for-product-managers-why-it-matters
