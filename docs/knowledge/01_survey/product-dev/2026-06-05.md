## 一、Product School CEO Carlos：AI Agent 决策框架完整发布 ⭐⭐

**来源**: Product School Blog, Carlos Gonzalez de Villaumbrosia, 2026-03 初版 / 2026-04-27 更新
**文章**: *"When Should Product Teams Use AI Agents?"* — 25 min 深度文章

### 📌 核心发现

这是目前看到关于"产品团队何时/如何使用 AI Agent"**最实操、最完整的决策框架**。Carlos 写成了一篇 25 分钟的深度文章，内含可直接使用的**12 维度评分决策矩阵**。

### 🔬 三大核心框架

**1. 能力分层：Assistant vs Automation vs Agent**

| 类型 | 定义 | 适用场景 |
|:----|:-----|:---------|
| Assistant（聊天/副驾驶） | 帮你思考、写作、头脑风暴 | 你依然在"驾驶位" |
| Workflow Automation（确定性自动化） | 执行预定步骤 | 需求稳定、需可预测性 |
| Agent（智能体工作流） | 决定下一步、调用工具、处理异常、循环直到完成 | 工作流混乱、异常多、规则不适用 |

**2. 三大思想阵营**

| 阵营 | 代表 | 核心论点 |
|:----|:-----|:---------|
| 🚀 加速派 | Elena Verna (Lovable Growth Head) | "AI 可将 shipping velocity 提升到可防御的程度"，需要给团队"完全自主权" |
| 🔒 信任派 | Jeetu Patel (Cisco CPO) | "安全与信任不是生产力的敌人，而是生产力的先决条件" |
| ⚖️ 综合派 | Carlos 本人 | 取决于团队的风险承受能力、控制成熟度和成本结构 |

**3. 12 维度决策评分矩阵（0-24 分）** ⭐ 实操可用

| 维度 | 0分 | 1分 | 2分 |
|:----|:---|:---|:---|
| 规则自动化可行性 | 规则完全可行 | 部分边界情况 | 规则经常失效 |
| 多步骤复杂度 | 单步 | 少量步骤 | 多系统跨步骤 |
| 工具交互需求 | 无工具 | 一个系统 | 多个系统 |
| 成功可衡量性 | 难定义"正确" | 部分信号 | 明确成功指标 |
| 错误操作成本 | 高+不可逆 | 中等 | 低或可逆 |
| 过度信任风险 | 非常可能 | 可能 | 不太可能 |
| 可逆性 | 很难撤销 | 可部分回滚 | 容易修复 |
| 执行频率 | 罕见 | 偶尔 | 频繁/重复 |
| 输入变化性 | 高度可预测 | 混合 | 混乱/非结构化 |
| 协作开销 | 极少 | 中等 | 大量 |
| 判断需求 | 无 | 一些 | 大量 |
| 延迟容忍度 | 必须即时 | 中等 | 灵活 |

**评分解读**：
- **0-8**：不要用 Agent。用脚本、规则、模板
- **9-16**：考虑推荐模式（Agent 建议，人决定）
- **17-20**：适合 Agent + 审批模式
- **21-24**：强 Agent 候选 + 护栏

### 📊 关键数据

- **Gartner**：40% 以上 Agentic AI 项目将在 2027 年前被取消
- **MIT NANDA 报告**：95% 组织从 GenAI 获得**零回报**
- **Autonomy budget（自主权预算）** 概念 — "当你愿意给一项工作分配自主权预算时，才使用 Agent"
- **核心原则**：Agent 能做 50-70% 的工作，人掌握剩余 30%，人永远是问责层
- **Dave Bottoms (Upwork GM/VP)**："AI 做 50-70%，人进来完善、定制、增强"

### 🎯 对产品管理的启示

1. **"Autonomy is earned, not assumed"** — 自主权是挣来的，不是假设的。从推荐模式开始
2. **"Human-in-the-loop 不是万能的"** — 新研究指出"契约悖论"：AI 越准确，人越难保持警惕
3. **四个前提条件**：可衡量的成功 → 约束与护栏 → 可观察性 → 适合的 HITL 设计
4. **与 Product Partners 四阶模型互补**：Agent 决策矩阵回答了"什么时候用"的技术问题，四阶模型回答了"组织层面怎么过渡"的管理问题

## 二、Notion Custom Agents Beta 运营经验：Agent 成了新的组织技能 ⭐

**来源**: Notion Blog, Akshay Kothari (Co-founder), 2026-05-04
**文章**: *"What we learned during the Custom Agents Beta"*

### 📌 核心发现

Notion Custom Agents 从 2 月进入 Beta 到 5 月正式 GA（含使用量定价），**两个月内用户创建了超过 100 万个 Custom Agents**。最关键的不是这个数字，而是运营模式上的发现。

### 🔬 关键洞察

**1. 最快的团队把 Agent 当内部产品来构建**
- 从一小群熟悉工作流的人开始，迭代构建直到 Agent 达到成功标准，然后分享学到的经验
- **每个 Agent 让下一个更容易** — 知识在团队中复合增长
- **Ramp 案例**：已发展出 **300+ 个 Custom Agents**
- Ramp 运营/内部 AI 负责人 Ben Levick 的描述：
  > "一路有两个多巴胺高峰。一是你破解问题、自动化了你不想做的工作；二是另一侧的事情安静了，你一天中有了更多时间。"

**2. Agent 构建成了一种新的运营技能**
- 最有持久力的 Agent 来自**离问题最近的人**，而非最懂技术的人
- 没有一个 Agent 是由工程师构建的：
  - **Heidi** 产品团队 — 内部 Q&A Agent，**每月节省 60 小时**
  - **Equals Money** 产品营销团队 — 知识库更新 Agent，确保文档随版本更新
  - **Notion 客服团队** — 帮助客服更快响应的 Agent
- 核心洞察：**"If you can describe what you'd tell a new hire to do, you can build a Custom Agent"**

**3. 最大收益来自重新设计工作流而非任务自动化**
- **Braintrust 案例**（竞争情报 Agent "Competitive Intelligence Buddy"）：
  - Before：市场团队每天早上手动研究竞争对手
  - After：Agent 每天早上扫描竞品 changelog + Gong 转录 → 更新 Notion 销售一页纸 → 标记重要发布
  - 🎯 销售带着情报上场，市场团队每天拿回数小时

### 📊 关键数据

- **100 万+ Custom Agents** 在 2 个月内创建
- **Ramp** 300+ 个 Agent，从内部运营切入
- **Heidi** 产品团队每月节省 **60 小时**
- **50% 成本降低**：引入轻量模型（Haiku 4.5、GPT-5.4 Mini），使用量减少 10 倍
- **定价**：$10/1,000 credits，使用量定价，弹性扩容

### 🎯 对产品管理的启示

1. **Agent 不再是工程团队的专属** — "能描述怎么教新人，就能建 Agent"意味着 PM 和运营团队可以直接构建
2. **最大的杠杆不在"任务自动化"，而在"工作流重新设计"** — Braintrust 案例展示了端到端重构的威力
3. **内部 Agent 产品化思维** — 像做产品一样迭代 Agent（定义成功标准 → 迭代 → 分享 → 规模化）
4. **Notion 的 5 月定价转变** — 使用量定价意味着 AI 产品的商业模型从"按座收费"向"按价值收费"加速迁移

## 三、Intercom 深度研究：AI Agent 部署的 5 种组织就绪度 ⭐

**来源**: Intercom Blog, Jennifer Murphy, 2026-05-18
**文章**: *"Agents can do the work"*

### 📌 核心发现

Intercom 研究了横跨 6 个行业（能源、医疗、电商、游戏、金融服务、物业）的公司后发现：**大多数组织对 AI Agent 的约束不是技术能力，而是组织就绪度。**

### 🔬 五维就绪度框架

| 层级 | 类型 | 定义 | 大多数组织的状态 |
|:----|:-----|:-----|:--------------|
| 1️⃣ | Content Readiness | 能否清晰一致地解释产品和策略 | ✅ 大多数可以 |
| 2️⃣ | Scope Readiness | 是否定义了边界：Agent 何时介入、何时退出 | ✅ 可通过努力解决 |
| 3️⃣ | Procedural Readiness | 流程是否足够清晰，能让非人类执行 | ⚠️ "快乐路径没问题，但失败路径、决策分支从未被记录" |
| 4️⃣ | Data Readiness | 能否实时可靠地识别正确的用户/账户/对象 | ❌ **第一个真正的悬崖**。"部分可以，但不确定何时会断" |
| 5️⃣ | Execution Readiness | 技术上能执行 + 组织上能承担后果 | 🔴 **最高门槛**。谁承担错误退款？谁检测？谁恢复？谁接受风险？ |

### 📊 关键数据

- **能力评估的陷阱**：大多数 Agent 评估只测能力（can the AI do this?），但能力 ≠ 组织实际可用
- **研究结论**："Readiness enables capability, not the other way around" — 就绪度才是能力的前提
- 大多数公司拥有前两层，少数拥有第三层，**更少的拥有第四和第五层**

### 🎯 对产品管理的启示

1. **重新定义 Agent 评估问题**：从"Agent 能不能做？"变成"我们准备好让它做了吗？"
2. **5 种就绪度是诊断工具**：当你发现 Agent 部署卡住，用这个框架找出是哪一层的瓶颈
3. **组织就绪度是产品管理的长期投资** — 比买更强的 Agent 更值得投入
4. **与 Product Partners L2→L3 瓶颈互补**：四阶模型说明了"80% 团队卡在 L1"，五维框架解释了"为什么"

## 四、Linear 产品创新：Code Intelligence + Team Documents + Agent MCP ⭐

**来源**: Linear Changelog, 2026-04 至 2026-06
**关键发布**：Code Intelligence (May 14)、Team Documents (Jun 4)、Agent MCP (Apr 23)、Linear Diffs + Guided Reviews (May 28)

### 📌 核心发现

Linear 在 2026 Q2 密集发布了一系列产品能力，从"项目管理系统"向"AI-native 产品协作平台"转型。

### 🔬 关键产品创新

**1. Code Intelligence（5月14日，Public Beta）**
- Linear Agent 获得对代码库的受控访问
- PM 可以问："这个功能怎么实现的？""这个变更会影响什么？"
- **PMs can write sharper specs** — 通过代码上下文撰写更精准的产品需求
- Support/Sales 可以回答技术问题，不用打断工程师
- 权限控制：管理员选择哪些仓库纳入，是否与 GitHub 权限一致

**2. Team Documents（6月4日）**
- 每个团队有专属的文档空间
- 跨 Issue/Project 的团队上下文集中管理
- 可创建团队级文档、从 Project/Issue 添加已有文档、分组管理
- **意义**：从"Issue 驱动"到"上下文驱动"，文档成为 Agent 的推理基础

**3. Agent MCP 集成（4月23日）**
- Linear Agent 通过 MCP 连接外部工具
- 示例：从 Granola 拉取会议要点创建项目更新、从 Glean 获取企业上下文写 Spec、从 PostHog 数据验证产品假设
- **允许管理员控制访问**：白名单+工作区级 MCP 权限

**4. Linear Diffs + Guided Reviews（5月28日）**
- 原生代码审查能力，无需离开 Linear
- **Guided Reviews**（Beta）：智能分组，先展示核心变更，分支/胶水代码折叠
- 背景 Agent 在审查过程中可以实时更新 Diffs
- **Review Inbox**：审查通知与日常工作流集成

### 🎯 对产品管理的启示

1. **Code Intelligence 是 PM 能力的重要扩展** — PM 不再只是在 Jira/Linear 上写需求，可以主动从代码层面理解产品
2. **AI-native PM 工具的特征**：文档（Team Documents）+ 代码（Code Intelligence）+ 外部上下文（MCP）= 统一的 Agent 工作区
3. **Linear 的产品路线展示了从"项目管理工具"向"产品 Agent OS"的演进**
4. **交叉引用**：与本文件第四条（Notion Custom Agents）形成对照 — Notion 是 Agent 创建平台，Linear 是 Agent 协作平台
