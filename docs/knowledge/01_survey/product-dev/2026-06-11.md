### 核心论题：Autonomy Budget

> **"Use AI agents when you're willing to give a piece of work an autonomy budget."**

每个 AI Agent 决策本质上是在"赋予自主权"与"管控风险"之间做权衡。自治预算是决定 Agent 是否值得引入的核心度量。

### 行业背景数据

| 数据点 | 出处 |
|:------|:-----|
| Gartner 预测 **40%** 的 Agentic AI 项目将在 2027 年前被取消 | Gartner |
| MIT NANDA 报告：**95%** 的组织从 GenAI 计划中获得**零 ROI** | MIT |
| 企业 AI 中大量产出仍是个体增益（individual gains），而非业务 OKR 指标 | 业界共识 |

### Agent Spectrum 自主权光谱（完整版）

| 级别 | 名称 | 自主权 | 风险管理 | 适用场景 |
|:----|:-----|:------|:--------|:--------|
| A | **反应式（Reactive）** | 无自主权 | 安全，即时价值 | 聊天机器人、文档自动补全 |
| B | **工具使用（Tool-using）** | 辅助自主权 | 快速 ROI，最少治理 | 搜索、总结、API 调用 |
| C | **半自主（Human-in-the-loop）** | 多步工作流+人工审核 | 高杠杆+可控风险 | 研究起草→人工审核→发布 |
| D | **全自主（Goal-driven）** | 最小干预 | 最大价值+最高治理需求 | 自动化 bug 修复、持续监控 |

> **Product School 的明确建议**：大多数产品团队应长期停留在 B 或 C 级别。在 D 级别获得真实杠杆之前，已先支付完整的风险账单。

### ⭐ 0-24 分评分矩阵（完整版）

以下是文章中提供的完整评分矩阵。评分方式：每个维度 0-2 分。

| 维度 | 0 = 不需要 Agent | 1 = 可以考虑 | 2 = 强烈适合 Agent |
|:----|:---------------|:------------|:-----------------|
| 规则自动化可行性 | 规则可以清晰处理 | 有些边缘情况 | 规则经常失效 |
| 多步工作流复杂度 | 单个步骤 | 少量步骤 | 跨多个工具的多个步骤 |
| 工具交互需求 | 无需工具 | 一个系统 | 多个系统 |
| 成功可衡量性 | 难定义"正确" | 部分信号 | 明确的成功指标 |
| 错误动作成本 | 高+不可逆 | 中等 | 低或可逆 |
| 人类过度信任风险 | 非常可能 | 可能 | 不太可能 |
| 输出的可逆性 | 难撤销 | 部分可回滚 | 容易可逆 |
| 执行频率 | 罕见任务 | 偶尔 | 频繁/重复 |
| 输入变异性 | 高度可预测 | 混合 | 混乱/非结构化 |
| 协调负担 | 最小 | 中等 | 沉重 |
| 判断需求 | 无 | 一些 | 高 |
| 延迟容忍度 | 必须即时 | 中等 | 灵活 |

**评分解释**：

| 总分 | 推荐 |
|:----|:----|
| 0-8 | **不要用 Agent**。用脚本、规则或模板 |
| 9-16 | 考虑 **Agent 推荐模式**（人类做最终决定） |
| 17-20 | 适合带**审批的 Agent** |
| 21-24 | 强 Agent 候选，但需要**保护措施** |

### 5 个最佳使用场景（详细版）

1. **持续研究综合（Living Research Pipeline）** — 持续摄入访谈、调查、支持工单，自动标记主题、链接证据到主张、标记变化
2. **决策支持（Decision Support）** — Agent 从 Jira/Linear/Slack/docs 拉取上下文，合并成清晰的时间线，提取真实决策和约束条件，映射利益相关者立场，运行结构化预检
3. **工作流自动化** — 关键模式：**Agent 计划，确定性系统执行**
4. **发布准备和 QA 分类** — 使日志/指标/调用链对 Agent 可读，Agent 验证变更并推理行为
5. **指标叙事** — 不只是仪表盘，Agent 解释"什么变了、为什么、该问什么问题"

### 4 个不应使用 Agent 的场景

1. **后果严重且难撤销** — 定价、删除数据、授权、触碰生产、广泛客户消息 → 有限自主权模式
2. **Agent 读取混乱的、不受信任的文本并采取行动** — **Prompt Injection 是 OWASP #1 风险**
3. **规则已经很好用** — 确定性自动化更便宜、更可预测、更易治理
4. **「Human-in-the-loop」变成时间黑洞** — Agent 大部分正确 → 人类不自觉变成质量检查员

## 2️⃣ Product School: Pretotyping vs. Prototyping — 完整深度 ⭐

**来源**: Product School Blog — Carlos Gonzalez de Villaumbrosia（CEO, Product School）
**发表日期**: 2026年4月5日（16 min read）
**原文标题**: "Pretotyping vs. Prototyping: No, It's Not a Typo"

> **注**：此文章的概要已在 06-10 中记录，本节补充完整文章中的**详细比较维度**和**AI 时代的应用变化**。

### 六个关键差异

| 维度 | Prototyping（原型化） | Pretotyping（预原型化） |
|:----|:--------------------|:---------------------|
| **回答的问题** | "我们能构建这个吗？它能在真实用户流程中工作吗？" | "这个值得构建吗？有人真的想要它吗？" |
| **流程位置** | 在对 idea 有足够信心后 | 在 idea 刚形成时 |
| **细节程度** | 接近真实产品，有意义地测试体验 | 极简设计，核心体验模拟 |
| **真实程度** | 通常包含真实交互元素和一定逻辑 | 通常非功能性或人工驱动（Wizard of Oz） |
| **时间成本** | 天-周（仍远低于完整开发） | 小时-天 |
| **产出类型** | 执行质量的证据：可用性反馈、可行性、性能约束 | 需求证据：点击、注册、支付意愿、重复使用 |

### McKinsey 数据背景

> McKinsey 研究显示：产品和服务发布的平均失败率超过 **40%**。
> Pretotyping 解决的是**需求风险**（有人在乎吗？），Prototyping 解决的是**产品风险**（能用吗？）

### AI 对 Pretotyping 的赋能

- **AI 使预验证成本趋近于零** — 无需定制模型，直接用现成 LLM 包装成基本聊天 UI，观察用户交互
- **Wizard of Oz 测试升级** — AI 在幕后做重活（草拟回复、总结输入、生成变体），人类仍可掌舵
- **核心原则始终不变**：先验证需求，再验证实现

### ⚠️ 常见陷阱

团队常称粗糙的 demo 为 "MVP"，然后市场无反应时感到困惑。正确链条是：

```
Pretotype → 验证需求
    ↓
Prototype → 验证解决方案
    ↓
MVP → 在现实中学习价值
```

## 3️⃣ Product School: AI 实验方法论 — 细节补充

**来源**: Product School Blog
**发表日期**: 2026年6月

> 此文章的概要已在 06-10 中记录，本节补充完整文章中的关键细节。

### AI 实验的五大类

| 实验类型 | 适用场景 | 关键挑战 |
|:--------|:--------|:--------|
| 在线 A/B 测试 | 功能上线后评估效果 | AI 的变异性需要更大样本量和更长周期 |
| 离线评估 | 模型质量基准测试 | 离线指标可能不反映在线体验 |
| 用户研究 | 定性理解用户感知 | AI 的不可预测性可能影响研究设计 |
| AI 模拟评估 | 用 AI 评估 AI | 评估者偏差的风险 |
| 回退率分析 | 衡量 AI 何时被人类干预 | AI 成功率 vs 用户信任的动态关系 |

### AI 特有的指标

- **幻觉率**（Hallucination Rate）— AI 输出的不准确信息比例
- **回退率**（Fallback Rate）— 用户或系统选择不信任 AI 输出的频率
- **信任度**（Trust Score）— 用户持续使用 AI 输出的意愿
- **可解释性**（Explainability）— 用户能否理解 AI 为何做出某个输出
- **过度信任风险**（Over-reliance Risk）— 用户不加批判地接受 AI 输出
