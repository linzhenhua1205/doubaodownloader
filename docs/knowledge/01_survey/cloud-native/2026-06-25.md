## 2️⃣ Jaeger v2.18 ClickHouse Backend: 8.6× Compression GA

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-23, Mahad Zaryab (Jaeger Maintainer @ Meta)

| 指标 | 数值 |
|:-----|:-----|
| 压缩比 | **8.6×**（10M spans 基准测试）|
| 写入吞吐 | >50k spans/sec |
| 存储节省 | 对比 Elasticsearch 减少 80%+ 存储成本 |
| 状态 | **Alpha → 可生产体验** |

**架构要点**:
- 使用 ClickHouse 列式存储 + MergeTree 引擎替代 Elasticsearch
- Span 存储采用按 Trace ID 分片策略，支持快速 Trace 检索
- 配合 OTel Collector 原生 OTLP 导出，Jaeger 端无需额外适配

**运维价值**: 8.6× 压缩比意味着在相同存储预算下可保留 8 倍多的追踪数据，或降低 80%+ 存储成本。对于大规模分布式追踪场景尤为关键。

## 3️⃣ Dapr 1.18: Verifiable Execution for Workflows & AI Agents

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-11

Dapr 1.18 引入 **Verifiable Execution**（可验证执行），核心能力:
- **Attestation（证明）**: 每次 Workflow/AI Agent 执行附带加密签名证明
- **Provenance（来源追溯）**: 完整记录执行链路与数据来源
- **Tamper-Evident Execution History（防篡改历史）**: 所有执行记录加密链式存储

**意义**: 面向 AI Agent 工作流的安全治理——当 Agent 需要跨服务、跨信任域调用时，Verifiable Execution 提供审计追踪和不可否认性。与 Istio/SPIFFE/SPIRE 构成完整的 Agent Auth 信任链。

## 4️⃣ OTel: "Don't Wrap OpenTelemetry" — API 包装反模式 🚫

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/) — 2026-06-24

OTel 团队系统性批判了常见的 API 包装反模式:
- ❌ **强制堆分配**: 包装层引入不必要的 `make()` 调用
- ❌ **查找包装器破坏热路径**: map/slice 查找导致延迟散布
- ❌ **维护负担**: 每个新 OTel 版本需同步更新包装层
- ✅ **零分配设计**: 原生 OTel API 的 `TagList` / `&[]` 已是零分配
- ✅ **OTel Weaver 代码生成方向**: 替代手动包装的正确方案

## 8️⃣ CNCF: Telemetry That Matters — 可持续观测管道设计

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-22

**核心挑战**: 观测数据爆炸——更复杂的系统产生更多 telemetry，但真正的价值数据可能不到 1%。

**应对策略**:
- **Triage-first 采集**: 先采集关键指标，有异常再展开全量
- **自适应采样**: 根据服务健康状态动态调整采样率
- **Telemetry 成本归属**: 每个 pipeline 阶段建立 ROI 量化机制
- 与 OTel-Arrow Phase 2 的高效管道结合，降低数据洪流
