## 1️⃣ OTel Blog: "Don't Wrap OpenTelemetry" — 包装 OTel 反模式深度剖析 🔴

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/dont-wrap-opentelemetry/) — 2026-06-24, Cijo Thomas (Microsoft)

OTel 维护者发布重磅文章，指出许多团队"善意的包装"正在损害 OTel 的性能与可维护性。

### 两大反模式

**反模式 #1: 签名强制堆分配**
- 团队自行编写 `IMetric` 接口或 `RecordHistogram(name, val, List<KeyValuePair>)` 包装
- OTel .NET SDK 专门为 1/2/3 属性设计了**零分配**重载，`TagList` 是 struct 也避免堆分配
- Rust SDK 使用借用的 `&[KeyValue]` 切片（栈分配数组）
- **包装层强制使用集合类型后，每次测量都堆分配，性能模型被完全破坏**

**反模式 #2: 按名称查找 + 内部缓存 Instrument**
- 包装每次调用时用 `ConcurrentDictionary.GetOrAdd` 或 `Mutex<HashMap>` 查找 Instrument
- OTel Instrument 设计为**启动时一次性创建**并持有引用，每次查找是破坏性的性能退化
- .NET ConcurrentDictionary 每次测量含字典查找+哈希+比较；Rust 的 Mutex 成为热点路径串行点

### 更深层问题

| 问题 | 描述 |
|:-----|:------|
| **开发者学的是包装而不是 OTel** | 换团队/读官方文档需从头学起 |
| **维护负担随时间增长** | OTel API 新能力需在包装层重新暴露 |
| **调试更难** | 多一层间接，问题难追踪 |
| **伪抽象** | 实际上 OTel 已经提供 InMemoryExporter/stdout exporter 用于测试 |

### OTel 官方建议

> **Skip the wrapper.** 直接使用 OTel SDK 的零分配 API。如果必须包装（如遗留系统迁移期），确保不破坏 OTel 的零分配路径。

**未来方向**: OTel Weaver 工具可从语义约定定义生成类型安全 OTel 代码，取代自定义包装。

**运维价值**: 对企业内部规范 OTel 使用模式有直接指导意义，避免团队重复踩坑。

## 2️⃣ GenAI 可观测性: OTel 标准化下的 LLM 调用全链路追踪 🔴

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/genai-observability/) — 2026-05-14, James Newton-King (Microsoft)

OTel GenAI 语义约定正式用于生产环境，以 VS Code Copilot 为例展示全链路追踪：

### 已支持 GenAI OTel 的编码工具

| 工具 | 支持能力 |
|:----|:---------|
| **VS Code Copilot** | 完整的 traces/metrics/events，支持 content capture |
| **OpenAI Codex** | 结构化日志 + OTel metrics（API请求/工具调用/会话） |
| **Claude Code** | metrics + 日志事件，trace 支持 beta |

### 可观测性数据

**Traces**: 顶级 `invoke_agent` span → 子 `chat` spans（每次LLM调用）+ `execute_tool` spans（每次工具调用）
- `gen_ai.request.model` — 使用的模型（如 gpt-4o）
- `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens`
- `gen_ai.response.finish_reasons` — stop 或 tool_calls

**Metrics**: 前缀 `gen_ai.`
- `gen_ai.client.operation.duration` — LLM 延迟直方图，可按 model 过滤
- `gen_ai.client.token.usage` — token 消耗直方图，按 token.type 区分 input/output

**运维价值**: 在 AI Agent + LLM 调用场景下，OTel 提供了开箱即用的全链路追踪方案。

## 3️⃣ OTel 正式弃用 OpenCensus 兼容性 🟡

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/deprecating-opencensus-compatibility/) — 2026-06-23

**要点**:
- OTel 规范项目合并 PR #5138，正式将 OpenCensus 兼容性标记为**已弃用**
- 这是对 OpenCensus 已基本消亡的现实生态的最终确认（OTel 本身由 OpenTracing + OpenCensus 合并而来）
- 实际上 OTel 诞生就是为了替代 OpenCensus，这一步表明**旧兼容层可以清理了**

**运维影响**: 还在使用 OpenCensus 导出器的组织应尽快迁移到纯 OTel。

## 5️⃣ Dapr 1.18: 可验证执行 (Verifiable Execution) 🟡

**来源**: [CNCF Blog](https://www.cncf.io/blog/2026/06/11/introducing-verifiable-execution-in-dapr-1-18/) — 2026-06-11

**三大新能力**:

| 能力 | 作用 | 关键技术 |
|:-----|:------|:---------|
| **Workflow History Signing** | 工作流历史加密签名，防篡改 | 签名验证链 |
| **Workflow History Propagation** | 执行血缘随请求传递 | 分布式上下文传播 |
| **Workflow Attestation** | 可验证的执行上下文证明 | SPIFFE 工作负载身份 + 签名 |

**核心洞察**: "Observability tells you what happened. Verifiable Execution helps you prove it."

**AI Agent 场景**: 当 Agent 调用工具、委托工作、触发工作流时，下游系统需要证明"谁发起+怎么来的"——SPIFFE 回答"你是谁"，Verifiable Execution 回答"你怎么到这的"。

## 6️⃣ Security Profiles Operator v1: 8 个 CRD 稳定版 GA 🟡

**来源**: [CNCF Blog](https://www.cncf.io/blog/2026/06/26/security-profiles-operator-v1-stable-apis-security-hardened-and-shaping-upstream-kubernetes/) — 2026-06-26, Sascha Grunert (Red Hat)

**里程碑**: SPO 8 个 CRD API 全部升级到 v1 稳定版，历时 6 年 API 演进。

**安全审计结果**: 0 个严重漏洞，审计发现已全部修复：
- RawSelinuxProfile 增加门控开关 + 验证 webhook
- AppArmor 输入严格 regex 验证
- 防 regex 回溯攻击、防 OOM、防 PID 重用缓存污染
- 指标高基数问题修复

**向上游 K8s 演进**: KEP 6061 (OCI Artifact-Based Security Profile Distribution) 将 SPO 的 OCI 分发概念引入 kubelet 原生支持。
