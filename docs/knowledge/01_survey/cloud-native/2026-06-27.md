## 1️⃣ 构建集群感知 AI Agent: K8s + Argo CD + GitOps 实践指南 🔴

**来源**: [CNCF Ambassador Post](https://www.cncf.io/blog/) — 2026-06-25, Maryam Tavakkoli

CNCF Ambassador 发布了一篇**完整的 K8s AI Agent 实践指南**，展示如何运行自托管、只读 AI Agent:

**架构要点**:
- **自托管 AI Agent 运行在 K8s 集群内部**，数据不出集群，无云 AI API 依赖
- 完整 CI/CD 链由 **GitHub Actions + Argo CD Image Updater** 处理
- 通过 K8s RBAC 限制 Agent 为**只读访问**，确保安全
- 数据隐私: Agent 直接在集群内部访问数据，不需要将敏感信息外传

**技术栈**: Kubernetes + Argo CD + GitHub Actions + Argo CD Image Updater

**运维意义**: 这篇博客提供了一个"落地"方案框架——在 CNCF "Cloud Native is AI-native" 的宏观判断下，给出了**具体的工程实现路径**。

**与近期趋势关联**:
- 与 Dapr 1.18 Verifiable Execution 形成互补（Dapr 解决 Agent 信任链，Argo 解决交付链）
- Agent 的 GitOps 化交付是 MLOps/AIOps 运维落地的重要一步
- 自托管 Agent 方案对**数据合规**（GDPR/数据主权）有天然优势

## 2️⃣ Jaeger ClickHouse 后端: 8.6× 压缩比 + 10M+ span 吞吐 🔴

**来源**: [CNCF Project Maintainer Post - Mahad Zaryab (Meta)](https://www.cncf.io/blog/) — 2026-06-23

Jaeger v2.18.0 正式交付 **ClickHouse 后端**，这一功能社区已期待多年。项目维护者（来自 Meta）详解了背后的架构决策:

**性能数据**:
- **压缩比 8.6×**: 在 1000 万 span 数据集上验证
- **吞吐能力**: 持续保持 >50k spans/s 的摄取
- **与 Elasticsearch 对比优势**: 列式存储天然适合 trace 数据的聚合和分析

**架构决策要点**:
| 维度 | ClickHouse 方案 | Elasticsearch 方案 |
|:-----|:---------------|:-----------------|
| 存储模型 | 列式 | 行式 |
| 压缩比 | 8.6× | ~1.5-3× |
| 聚合查询 | 极快（列式原生优势） | 需扫描全量 |
| 运维复杂度 | 较低（无 JVM） | 较高（JVM 调优） |

**运维影响**:
- 对大量使用 Jaeger 的可观测性平台是重大利好
- ClickHouse 作为 OTel 后端存储的生态系统持续扩展（从 tracing 到 metrics）
- 考虑迁移的用户需注意: ClickHouse 是**追加**到现有后端选项，而非替代

## 3️⃣ 设计可持续、高影响力的可观测性管道 🟡

**来源**: [CNCF Community Post](https://www.cncf.io/blog/) — 2026-06-22, VictoriaMetrics/Bindplane/Cisco

聚焦可观测性管道的 **可持续性** 和 **信号质量**，描述了社区面临的一个关键问题：

**核心问题**: **我们淹没在自己的遥测数据中**。分布式系统的复杂性使得采集信号越来越容易，但真正有价值的信号却越来越难以提取。

**设计原则**:
1. **面向业务的信号选择**: 不是所有指标都值得采集，优先采集与 SLO 直接关联的信号
2. **高压缩比设计**: 利用列式格式（如 OTel-Arrow）和高压缩编解码器控制数据体积
3. **从被动告警到主动洞察**: 可观测性管道应支持 AI 驱动的异常检测，而非仅依赖阈值告警
4. **降采样与保留策略**: 高精度数据短期保留，聚合数据长期保留

**与 OTel-Arrow 关联**: 该文与 OTel-Arrow Phase 2 的列式管道方向一致，都是应对数据爆炸的工程方案。
