## 1️⃣ CNCF Blog · 数字主权云原生架构模式

> **发布日期**: 2026-06-16
> **来源**: [From data residency to digital sovereignty: Architectural patterns for cloud native platforms](https://www.cncf.io/blog/2026/06/16/from-data-residency-to-digital-sovereignty-architectural-patterns-for-cloud-native-platforms/)
> **作者**: Hrittik Roy, CNCF Ambassador

**核心论点**: 数字主权已从政策讨论演变为平台工程实践。EU Data Act 自2025年1月全面实施、NIS-2 和 DORA 已影响日常平台决策、UK Data Use and Access Act 2025 正在 2026 年推进。平台团队不仅需要证明工作负载在哪里运行，还需要证明**基础设施如何被运营、保护和管理**。

### 监管驱动的四大属性

| 属性 | 含义 |
|:-----|:------|
| **司法管辖隔离** | 所有可读租户数据的组件（包括控制面）在可命名和捍卫的法律管辖区内运行 |
| **运营自主** | 团队无需依赖单一供应商的托管服务即可重建、迁移和审计工作负载 |
| **加密与访问控制** | 密钥、etcd内容和管理凭证不被管辖区外的实体访问 |
| **可移植性** | 底层硬件/供应商/国家变更时，工作负载无重写迁移 |

### 核心架构模式：Tenant Cluster

**为什么单个 K8s 集群不够**：
- 一个控制面服务所有租户 → 一个租户数据面事故影响全部
- Namespace 不是隔离边界（CRD/Webhook 共享）
- 集群扩张是常用的后备方案 → 运维重、昂贵、变更慢

**Tenant Cluster 模式**：每个租户拥有独立控制面（API Server/Controller Manager/Scheduler/数据存储），运行在共享底层集群之上，以 vCluster 为典型实现。

#### 对数字主权的关键特性

| 特性 | 价值 |
|:-----|:------|
| **独立控制面** | 每个租户独立 CRD/Admission Webhook/审计日志，版本可不同 |
| **可插拔后端存储** | 租户 etcd 可置于自有硬件上的加密卷，实现存储主权 |
| **租户隔离** | 租户内工作负载无法回访底层集群 API → 配合 gVisor/Kata 实现更强隔离 |
| **工作负载可移植** | 可移植到任何符合标准的 K8s 集群 |

### 实践模式：Jurisdiction as a Cluster

SaaS 公司服务 EU 和 UK 客户时的模式：

```
一 jurisdiction一 tenant cluster → GitOps 声明管理
- Node selector 硬约束 → Pod 钉在对应管辖节点
- 后端存储 → 在选定管辖区内
- 审计日志 → 管辖区本地 sink
- 策略包（Kyverno/OPA Gatekeeper）→ 内置执行
```

**主权事故的隔离能力**：
- CLOUD Act 请求 → 如果 etcd 存储由管辖区本地运营商管理，则不能自动获取租户集群的 etcd 内容
- EU 租户集群中受损的 Webhook → 无法触及 UK 集群（不共享控制面）
- CRD 升级可按集群分阶段进行 → 版本偏差成为特性

### AI 云场景

GPU 工作负载 + EU AI Act Article 12 日志与治理要求：
- 模式：GPU 承载的底层集群在主权裸金属上 + 每客户/每管辖区的 Tenant Cluster（通过 DRA 获取 GPU 访问）
- 生产案例：**Polarise**（德国主权 AI 云提供商）正运行此模式

### 约束与边界

- Tenant Cluster 模式不改变底层集群运营商的司法管辖区
- 仍需：Kyverno/OPA Gatekeeper、SBOM 管线、审计日志管线、SPIFFE/SPIRE 身份层、GitOps 控制器
- 每租户集群有运维成本，仅在有真实法律/风险权重时使用

### 2026 年主权平台形态

> 少量底层 K8s 集群（需要时建在主权基础设施上）→ 每管辖区/受监管工作负载一个 Tenant Cluster → Git 声明（Argo CD/Flux）→ Kyverno/Gatekeeper 策略 → 管辖区本地审计日志 sink → 标准 K8s API 可移植工作负载

**一句话**: 主权不再是采购条款，而是集群中的一个对象——有名字、有模板、有提交历史。

## 2️⃣ K8s Blog · SIG Storage Spotlight 深度访谈

> **发布日期**: 2026-06-15
> **来源**: [Spotlight on SIG Storage](https://kubernetes.io/blog/2026/06/15/sig-storage-spotlight-2026/)
> **受访者**: Xing Yang (VMware by Broadcom, SIG Storage Co-Chair)

### 受访者背景

Xing Yang 是 SIG Storage 联合主席（另一位是 Google 的 Saad Ali），还有两位 Tech Lead：Michelle Au (Google) 和 Jan Šafránek (Red Hat)。她也是 Data Protection Working Group 的联合主席。

### 近期关键特性

| 特性 | 状态 | 说明 |
|:-----|:-----|:------|
| **VolumeGroupSnapshot** | **v1.36 GA** ✅ | 多卷崩溃一致性快照，确保数据库等多卷应用数据完整性 |
| **CSI Changed Block Tracking (CBT)** | **v1.36 Beta** ✅ | 增量备份，仅报告自上次快照以来变更的块 |
| **VolumeAttributesClass** | **v1.34 GA** ✅ | 动态调整 IOPS/吞吐量，无需重新创建卷 |
| **Container Object Storage Interface (COSI)** | v1alpha2 | 标准化对象存储 Bucket 管理，目标未来 Beta |
| **Volume Health** | 规划中 | 存储驱动/控制面主动报告持久卷健康状态，支持自动化修复 |
| **Mutable PV Affinity** | Alpha | v1.35 引入，支持跨区域/磁盘类型迁移卷 |

### 存储 + AI 趋势

1. **更智能的数据管理**：CSI 驱动将支持自动分层、快照、迁移、复制
2. **对象存储一等公民**：AI 数据集达 Exabyte 级 → COSI 标准化 Bucket 管理
3. **高性能/低延迟**：NVMe-over-Fabrics (NVMe-oF) 原生管理
4. **数据感知调度**：Scheduler 将根据数据局部性决策 Pod 放置

### 状态工作负载三大挑战

| 挑战 | 说明 |
|:-----|:------|
| **Data Gravity** | Pod 秒级移动但数据有"重量"→ 增强 Volume Health 实现自动恢复决策 |
| **Day 2 复杂度** | 数据库搭建容易，长期运维难（Schema 升级/引擎修补/K8s 升级） |
| **数据移动性** | 跨存储层/可用区/集群的持久数据迁移仍是重大障碍 |
