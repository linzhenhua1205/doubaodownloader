## 1️⃣ 🌍 k0smos：跨数据中心异构 GPU 集群的云原生实战

**来源**: [CNCF Blog · 2026-06-08](https://www.cncf.io/blog/2026/06/08/breaking-free-of-a-single-datacenter-practical-geo-distributed-ai-operations-with-the-k0smos-platforms/)
**作者**: Prithvi Raj (Mirantis), Alexander Acker & Soeren Becker (Logsight.ai)
**项目**: k0s · k0smotron · k0rdent（k0smos 栈）

### 背景：地理分布式 AI 成为 K8s 问题

现代 AI 架构假设集中式同质数据中心，但现实是计算资源分散在私有云、研究环境和边缘。当工作负载跨越集群地理边界，**多集群编排**成为必需。

### k0smos 三层架构

| 层 | 项目 | 功能 |
|:---|:-----|:-----|
| 🏗️ **运行时** | **k0s** | 零依赖 K8s 发行版，单二进制，可在任何 Linux 上运行 |
| 🎮 **控制平面** | **k0smotron** | 将控制平面以隔离 Pod 形式部署在中心管理集群中，Worker 节点可跨地理位置 attach |
| 📋 **编排层** | **k0rdent** | 声明式多集群生命周期管理，GitOps 驱动，多 provider 支持 |

### 实战验证：跨洲异构 GPU 训练

与德国 SPRIND（联邦颠覆性创新局）合作的两个现场研究：

**研究一：静态地理分布式训练**

- 架构：**魁北克 A100 + 亚特兰大 MI300X**，控制平面在**法兰克福** CPU-only 节点
- 网络：Cilium + WireGuard P2P 隧道（~35ms 延迟，~600MB/s），**不经过中心 VPN**
- AI 框架集成：PyTorch Elastic、Ray、vLLM，通过 k0rdent ServiceTemplates + Helm Charts 部署
- 训练结果：GPT-NeoX（LLM）、ResNet（CV）、GCN（图学习）、PPO（强化学习）、Wav2Vec2（音频）——跨厂牌 GPU 成功训练
- 关键使能技术：**解耦动量优化**（de-coupled momentum optimization），大幅减少跨站点通信需求

**研究二：动态能量感知训练**

- 放宽静态假设：GPU 资源根据**电价窗口**动态加入/退出
- 联邦学习范式 + Flower AI（Kubernetes Operator 管理）
- WattTime API 实时获取可再生能源信号，动态启停 GPU 容量
- 成果已发表于 Flower AI Summit 2026 和 EuroSys 2026

### 实践教训

- GPU 生命周期管理仍是碎片化环境的痛点（驱动冲突、云特定内核）
- Cilium + WireGuard 处理了跨站点安全连接，但分布式训练同步（torch.distributed）仍具挑战
- **核心结论**：跨地理、跨厂牌 GPU 异构训练在云原生基础设施上**现在已经可行**

### 运维意义

这是 CNCF 多集群管理栈（k0s→k0smotron→k0rdent）在 **AI 跨数据中心场景**下的完整实战验证。证明了将零散 GPU 资源池化为统一算力平台的云原生路径是可行的，"距离和硬件异构不再是分布式训练的绝对障碍"。

## 2️⃣ 🖥️ KubeVirt virtbench：VM 性能基准测试工具开源

**来源**: [CNCF Blog · 2026-06-08](https://www.cncf.io/blog/2026/06/08/benchmarking-kubevirt-performance-with-virtbench/)
**作者**: Bob Glithero (Portworx by Everpure)
**项目**: [virtbench](https://github.com/portworx/kubevirt-benchmark)（开源 CLI 框架）

### 为什么需要 virtbench

K8s 可观测性工具专为容器设计，无法衡量 VM 工作负载的关键性能指标：

| 指标 | 容器 | VM（KubeVirt） |
|:-----|:-----|:---------------|
| Time-to-Ready | 毫秒级（进程启动即 Ready） | **分钟级**（需等待 Guest OS 启动 + 网络可达） |
| 存储性能 | 单 PVC 简单 | **多 PVC**（boot/swap/data），CSI 驱动并发负载 |
| 在线迁移 | — | 需测量 **Stun Time**（SDN Overlay 中 VM 网络中断窗口） |

### 内置 6 大测试场景

| 场景 | 测试内容 |
|:-----|:---------|
| DataSource VM Provisioning | CSI 克隆效率和卷创建时间 |
| Single Node Boot Storm | 单节点并发 VM 启动能力 |
| Multi-Node Boot Storm | 集群级启动风暴（模拟故障恢复） |
| Live Migration | VM 迁移 stun time（顺序/并行） |
| Chaos Benchmark | 并发 chaos 操作 |
| Failure & Recovery | HA 验证 + 恢复时间 |

### 输出格式

- 结构化 JSON + CSV 数据
- 交互式 HTML 仪表盘
- 时间分解：clone_duration → running_time → ping_time，精确定位瓶颈在存储层/运行时/还是 Guest OS

### 对比其他工具

| 工具 | 聚焦 | virtbench 差异点 |
|:-----|:-----|:-----------------|
| kube-burner | API/控制平面吞吐 | virtbench 测**数据路径**：克隆速度、OS 启动、网络可达 |
| fio/iperf | 裸磁盘/网络微基准 | virtbench 测**组件交互**（迁移期间的网络、启动风暴期间的克隆） |
| KubeVirt E2E | 二值 pass/fail | **量化**："操作耗时多久？" |

### 运维意义

KubeVirt 是 CNCF 孵化项目（虚拟化管理），随着 AI 推理在边缘/工厂场景下沉，大量遗留 VM 工作负载需要迁移到 K8s。virtbench 填补了 KubeVirt **生产级性能基准测试的工具空白**——平台工程团队可以在 staging CI 中运行，升级前检测性能回归。

## 3️⃣ 🔄 Ingress NGINX → Envoy Gateway：零停机迁移实践

**来源**: [CNCF Blog · 2026-05-25](https://www.cncf.io/blog/2026/05/25/zero-downtime-migration-from-ingress-nginx-to-envoy-gateway/)
**作者**: Andrew Katsikas (Pelotech)

### 迁移驱动力

- Ingress NGINX 已停止安全补丁和新特性
- Gateway API 是 K8s 网络演进方向，比 Ingress 更表达力
- **迁移压力真实**，但不能以粗放切割为代价

### 控制器选型对比

| 控制器 | CNCF 状态 | 评估结论 |
|:-------|:----------|:---------|
| **Envoy Gateway** ✅ | CNCF 项目 | mTLS + request buffering 达标，选中 |
| Traefik | ❌ 非 CNCF | 与长期方向不符 |
| NGINX Gateway Fabric | ❌ 非 CNCF | 同上 |
| Istio | ✅ CNCF | 功能全面但较重 |
| Higress | ✅ CNCF | 未在官方 Gateway API 一致性测试列表 |

### 零停机方案：加权 DNS

**问题**：直接切换 DNS 记录时 TTL 期内请求会丢失。

**方案**（ExternalDNS + Route 53）：

```
Step 1: Ingress     weight=100  → 全部流量
         HTTPRoute  weight=0    → 就绪但不接流量
Step 2: 确认 HTTPRoute 正常后互换权重
Step 3: Ingress     weight=0
         HTTPRoute  weight=100  → 零失败请求切换
```

- **回滚同样简单**，权重互换即可，无需重建 DNS 记录
- 该模式**控制器无关**，任何 ExternalDNS 可管理的 Gateway API 实现都适用

### Gateway API 1.5 ListenerSet

生产中发现多 namespace 场景下 hostname 配置在 Gateway 级别（基础设施团队）与 HTTPRoute 级别（应用团队）之间的职责分离问题。

- **Gateway API 1.5** 引入 `ListenerSet` 新资源类型——允许应用团队定义自己的 listener，不触碰 Gateway 资源本身
- Envoy Gateway 已实现 ListenerSet（RC 阶段），正式版预计数日内发布

### 运维意义

**Ingress NGINX → Gateway API 迁移正在成为 K8s 网络的标准动作**。加权 DNS 为生产级迁移提供了可复用的零停机方案模板。ListenerSet 的引入解决了 Gateway API 在**多租户多 namespace 环境中**的一个关键职责分离痛点。
