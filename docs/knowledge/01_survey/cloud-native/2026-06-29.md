## 1️⃣ OTel + Service Mesh 指标融合：2026 年实践参考 🔴

**来源**: [CNCF Blog — OTel and mesh-derived metrics: A 2026 reference](https://www.cncf.io/blog/2026/06/29/otel-and-mesh-derived-metrics-a-2026-reference/) — 2026-06-29, Mesut Oezdil (Buoyant)

如果你已经在运行 OTel 管道，你能看到应用层的行为。但**东西向流量（east-west）** — 服务之间的流量 — 通常对 OTel 不可见，除非你引入 Service Mesh 层（Linkerd/Istio 等）。

### 核心洞察

| 维度 | 说明 |
|:-----|:------|
| **问题** | 应用级 OTel instrumentation 看不到网络层的重试、超时、延迟分布 — 这些由 Sidecar Proxy 处理 |
| **方案** | 将 Service Mesh 暴露的指标（TCP/HTTP 延迟、请求量、成功率）通过 OTel Collector 摄取、转换、关联到现有管道 |
| **关键挑战** | 指标标签对齐（Mesh 的 `dst_service` ↔ OTel 的 `service.name`），重复计数防止（双报告问题） |
| **推荐架构** | Mesh 指标 → Prometheus → OTel Collector (Prometheus Receiver) → 关联到 Trace 上下文 |

### 运维意义
- ✅ OTel 与 Service Mesh 的**指标融合**是 2026 年可观测性核心趋势
- ✅ 解决「应用层正常但用户感觉慢」的盲区问题
- ✅ 适合 AI 推理服务的端到端延迟诊断（Model Server ↔ Router ↔ Cache 之间的 mesh 流量）

## 2️⃣ Security Profiles Operator v1: 稳定 API + 上游 K8s 影响 🔴

**来源**: [CNCF Blog — Security Profiles Operator v1](https://www.cncf.io/blog/2026/06/26/security-profiles-operator-v1/) — 2026-06-26, Sascha Grunert (Red Hat)

Security Profiles Operator (SPO) 发布 **v1.0 稳定版**，经过 6 年演进，8 个 CRD 全部稳定。

### v1 关键能力

| 特性 | 说明 |
|:-----|:------|
| **稳定 API 集** | 8 个 CRD：`SeccompProfile`/`SelinuxProfile`/`AppArmorProfile`/`ProfileBinding`/`ProfileRecording`/`AllowedSeccompActions`/`AllowedSelinuxMount`/`AllowedSelinuxProfile` |
| **安全审计** | 0 个严重漏洞，所有高危修复完毕 |
| **上游贡献** | SPO 的 seccomp profile 管理逻辑正在被反向贡献到 Kubernetes 上游 sig-node |
| **Profile Recording** | 从运行中的 Pod 自动录制系统调用行为 → 生成最小权限 profile |

### 意义
- ✅ **K8s 安全进入标准化时代**：从手动写 seccomp/SELinux profile 到自动录制+分发
- ✅ SPO v1 的 Profile Recording 功能特别适合 AI 训练容器的安全基线生成（训练框架涉及大量系统调用，手动写不可行）
- ✅ 上游回馈 sig-node 意味着未来 K8s 原生支持 seccomp profile 管理

## 3️⃣ OTel Blog: "Don't Wrap OpenTelemetry" — 维护者深度剖析包装反模式 🟡

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/dont-wrap-opentelemetry/) — 2026-06-24, Cijo Thomas (Microsoft)

OTel 维护者发布**重磅文章**，指出许多团队为"简化"而包装 OTel API 的做法正在破坏性能。

### 两大反模式

| 反模式 | 症状 | 影响 |
|:-------|:-----|:-----|
| **签名强制堆分配** | 自定义 `IMetric` 接口或用 `List<KeyValuePair>` 包装 → 强制每次测量堆分配 | OTel .NET SDK 专为 1/2/3 属性设计的零分配重载被绕过 |
| **按名称查找 Instrument** | 每次调用用 `ConcurrentDictionary.GetOrAdd` / `Mutex<HashMap>` 查找 | OTel Instrument 设计为启动时一次性创建；查找破坏性能模型 |

### 核心教训
- ✅ **不要包装 OTel** — 直接使用 SDK 提供的 API，包装层往往弊大于利
- ✅ OTel SDK 已经考虑了性能，包装只会退化
- ✅ 如果必须抽象，用 facade 模式而不改变签名，且不要每次都查找 instrument

## 4️⃣ Jaeger v2.18 ClickHouse 后端：8.6× 压缩比 🟡

**来源**: [CNCF Blog — Building Jaeger's ClickHouse backend](https://www.cncf.io/blog/2026/06/23/jaeger-clickhouse-backend-8-6x-compression/) — 2026-06-23, Mahad Zaryab (Meta)

### 指标

| 指标 | 数值 |
|:-----|:-----|
| 压缩比 | **8.6×**（对 1000 万 spans） |
| 吞吐 | **>50k spans/s** |
| 版本 | Jaeger v2.18.0 |

### 意义
- ✅ ClickHouse 成为可观测性后端的新主流选择（和 Elasticsearch 竞争）

## 5️⃣ Dapr 1.18 Verifiable Execution: AI Agent 信任链 🟡

**来源**: [CNCF Blog — Dapr 1.18 Verifiable Execution](https://www.cncf.io/blog/2026/06/11/verifiable-execution-dapr-1-18/) — 2026-06-11

Dapr 1.18 引入**可验证执行**能力，将 SPIFFE + 签名 + 证明机制集成到工作流和 AI Agent 中。

### 关键特性
- **Attestation**: 每个执行步骤生成可验证的证明记录
- **Provenance**: 完整的执行历史溯源
- **Tamper-evident**: 执行历史防篡改
- **目标场景**: AI Agent 的信任链建立

### 意义
- ✅ 云原生 AI Agent **从"能跑"走向"可信"**
- ✅ 结合 Fedora AI Agent 账户入侵事件（see ops-system/2026-06-29），Agent 信任链成为刚需
- ✅ Dapr 的去中心化架构天然适合多 Agent 协作的可信审计
