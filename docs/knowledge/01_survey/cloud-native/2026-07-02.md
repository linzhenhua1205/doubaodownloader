## 1️⃣ 🔴 K8s Dynamic Resource Allocation (DRA) GA — GPU 调度迎来架构级变更

**来源**: CNCF Blog (2026-07-01)
**链接**: https://www.cncf.io/blog/2026/07/01/understanding-dynamic-resource-allocation-in-kubernetes/

**关键信息**:

- **DRA 在 K8s v1.35 正式 GA**，NVIDIA 的 `dra-driver-nvidia-gpu` 已移入 K8s SIGs
- DRA 彻底改造 GPU 分配模型：**不再需要 nodeSelector/Affinity + Device Plugin 的绑定模式**
- 核心 API 对象：`DeviceClass` / `ResourceSlice` / `ResourceClaim` / `ResourceClaimTemplate`
- 四大核心能力：
  - **精确设备选择**: 通过 CEL 表达式筛选（如 `device.attributes["gpu.nvidia.com"].productName == "NVIDIA RTX A5000"`）
  - **优先级排序**: `firstAvailable` 支持首选 + 回退（如优先 A5000 → 回退 T10）
  - **资源切片**: GPU Time Slicing 原生支持（配置 `sharing.strategy: TimeSlicing`）
  - **共享模式**: 单个 ResourceClaim 可被多个 Pod/容器共享
- K8s v1.36 将加入 **Device Health Reporting**，区分设备故障 vs 应用错误
- 未来 Cluster Autoscaler 可按 GPU 类型自动伸缩节点

**影响判断**: 这是从 Device Plugin 到 DRA 的**代际切换**。DRA 类比 Storage 领域从 in-tree 到 CSI 的变革。对 AI 基础设施来说，DRA 意味着 GPU 资源分配从"节点级粘性"变为"集群级灵活调度"，是万卡集群调度效率提升的关键基础设施。

## 2️⃣ 🔴 Kepler 重写：放弃 eBPF，回归 /proc + /sys

**来源**: CNCF Blog (2026-06-30)
**链接**: https://www.cncf.io/blog/2026/06/30/kepler-re-architected-improved-power-accuracy-and-a-community-call-to-action/

**关键信息**:

- **重构最大变化**: 删除 eBPF 依赖，改用标准 `/proc` 和 `/sys` 接口
- **动机**: eBPF 导致三个问题：
  1. 需要 `CAP_BPF` + `CAP_SYSADMIN` → 生产环境部署门槛高
  2. 进程级跟踪精度不足 → 短生命周期进程丢失 → 能耗低估
  3. 学习曲线陡峭 → 难以扩展和维护
- **硬件感知增强**: 动态发现主机功率计量结构（不再硬编码 RAPL core/DRAM/other）
- **精度验证**:
  - 实验 1（CERN）：新 Kepler `node_cpu_watts` 紧密跟随 IPMI 基准，**消除了旧版的 multi-kW 尖峰**
  - 实验 2（Red Hat）：Process Power Attribution Gap 仅**几毫瓦**，近乎零
- **测试覆盖**: 从零提升到 **90%**
- **待贡献领域**:
  - GPU 功率监控（实验性 flag，需要 AI/ML 用户测试）
  - VM 功率建模（ML 模型重新训练）
  - Idle Power Attribution 改进

**影响判断**: Kepler 的"eBPF 回退"是深刻的工程决策——不是所有问题都需要 eBPF，对于功率计量这类场景，`/proc` + `/sys` 的确定性读数可能比 eBPF 的"细粒度但高噪音"更可靠。**GPU 功率监控是 AI 基础设施能耗管理的关键试点方向。**

## 3️⃣ 🟡 OTel + Service Mesh 指标融合：2026 年参考架构

**来源**: CNCF Blog (2026-06-29)
**链接**: https://www.cncf.io/blog/2026/06/29/otel-and-mesh-derived-metrics-a-2026-reference/

**关键信息**:

- **核心主张**: OTel 覆盖应用层业务语义（traces/business metrics），Service Mesh 覆盖网络层东西向流量（mTLS 身份标识/成功率/延迟），两者互补而非替代
- **参考实现**: Linkerd 2.19+ → OTel Collector contrib 0.118.0 → VictoriaMetrics → Grafana
- **关键发现**:
  - Mesh proxy 暴露 `client_id`（mTLS 身份）——这是应用层无法获取的**加密级身份标识**
  - gRPC 场景：mesh 能检测到 `status_code=200` 但 `grpc_status=14 (UNAVAILABLE)` 的隐形故障
  - 9,280 个 series 来自 30 个 meshed pods，需用 OTTL filter 控制到 5 个 metric family
- **运维要点**: OTel Collector 的 `$` 环境变量展开与 Prometheus relabel 的 `$1` 替换冲突（contrib <= 0.112.0 有问题，0.118.0 修复）

**影响判断**: 对于运行 Istio/Linkerd + OTel 的集群，这篇是**实操级参考**。mesh-derived metrics + OTel traces 形成了"检测到问题 + 定位根因"的完整闭环。
