### 1️⃣ 🖥️ CNCF 项目 Arm64 支持里程碑：OCI 信用额度赋能多架构 CI

**来源**: [CNCF Blog · 2026-06-15](https://www.cncf.io/blog/2026/06/15/improving-arm64-support-in-cncf-projects-with-oci-credits/)
**作者**: Dave Neary（Ampere Computing 开发者关系总监）
**项目**: CNCF 基础设施

#### 背景：Arm64 已是云原生标配

| 指标 | 数据 |
|:-----|:------|
| AWS 新实例 Arm64 占比 | **> 50%**（截至 2025 年底） |
| Azure 新实例 Arm64 占比 | **> 33%** |
| Oracle 捐赠 | $3M Arm64 计算资源（OCI 信用额度，Ampere CPU） |
| 项目已受益 | OTel, Longhorn, Crossplane, Jaeger, Falco, containerd 等十几个 |

#### OCI Arm64 GitHub Actions Runner

CNCF Infra 团队将 OCI Arm64 实例配置为 **自托管 GitHub Actions Runner**，供 CNCF 项目使用：

| 用途 | 说明 |
|:-----|:------|
| 原生 Arm64 CI | 解决 GitHub 托管 Runner 在 CPU/内存上的容量限制 |
| 多架构容器构建 | 构建 Arm64 + x86 多架构 manifests |
| 性能验证 | 比 GitHub 托管 Runner 更大的实例规格 |
| 资源上限 | 默认 **$5,000/月**，可根据实际需求申请例外 |

#### 维护者反馈

> **OTel 维护者 Antoine Toulmé**：「Arm 性能远高于旧 x86 服务器。**Arm64 下载量大幅增长**。毫无疑问，我是 Arm 的皈依者。」
>
> **Falco 维护者 Federico Di Pierro**：「Falco 确实需要 Arm64 GitHub Runner 来提升对架构的支持并扩大用户群。」
>
> **containerd 维护者 Phil Estes**：「如果 Arm CI 现在出问题，我们会拦下直到修复。**我们对 Arm64 作为目标平台有完全信心。**」

#### 申请方式

CNCF 项目维护者通过 **CNCF Service Desk** 提交工单（说明用途 + 使用估算），即可接入已有 OCI Runner 池。CNCF Slack 上的 Infra 团队提供全程支持。

#### ⭐ 信号强度

⭐⭐⭐⭐ — 这不是技术突破新闻，而是**云原生基础设施的多架构里程碑**。Arm64 从"可选"变为"标配"已是既定事实。对运行多架构 K8s 集群的运维团队来说，**CNCF 项目对 Arm64 的 CI 覆盖质量直接决定了 Arm64 生产集群的可信度**。

### 2️⃣ 💾 K8s SIG Storage 深度访谈：AI 时代的存储演进路线图

**来源**: [K8s Blog · 2026-06-15](https://kubernetes.io/blog/2026/06/15/sig-storage-spotlight-2026/)
**受访者**: Xing Yang（VMware by Broadcom，SIG Storage 联合主席）
**项目**: K8s SIG Storage

#### SIG Storage 核心职责

SIG Storage 负责 Kubernetes 中**持久数据**、卷管理和存储接口的标准定义。核心产出：
- **CSI**（Container Storage Interface）— 存储供应商开发外部驱动器的标准
- **COSI**（Container Object Storage Interface）— 对象存储桶的标准接口
- **Data Protection WG** — 数据保护工作组的备份/快照特性

#### 近期发布的关键特性

| 特性 | 版本 | 状态 | 说明 |
|:-----|:-----|:-----|:------|
| **VolumeGroupSnapshot** | v1.36 | **GA** | 多卷原子级 crash-consistent 快照，数据库类应用的关键需求 |
| **CSI Changed Block Tracking** | v1.36 | **Beta** | 增量备份，仅传输自上次快照后变更的块 |
| **VolumeAttributesClass** | v1.34 | **GA** | 通过 K8s API 动态调优存储 IOPS/吞吐量，不停机 |
| **COSI** | 开发中 | v1alpha2 | 像 CSI 标准化块存储一样标准化对象存储桶的声明式管理 |
| **Volume Health** | 规划中 | Alpha | 提供 PV 健康状态的可观测性，支持自动化修复 |

#### AI 浪潮下的 K8s 存储四大趋势

| 趋势 | 说明 | 关联技术 |
|:-----|:------|:---------|
| 🧠 **智能数据管理** | CSI 驱动器和数据管理工具提供自动分层、快照、迁移和复制 | 高级 CSI 驱动器 |
| 📦 **对象存储一等公民** | AI 数据集已达 EB 级，对象存储是首选。COSI 标准化 S3 兼容存储的声明式管理 | COSI + BucketClaim |
| ⚡ **高性能低延迟** | 存储需跟上 GPU 处理速度。NVMe-oF 等高速技术原生接入 K8s | NVMe-over-Fabrics |
| 📍 **数据感知调度** | 调度器根据数据局部性决定 Pod 放置，权衡"移动数据 vs 移动计算"的成本 | 拓扑感知调度 |

#### 用户面临的三大存储挑战

| 挑战 | 描述 |
|:-----|:------|
| **数据引力（Data Gravity）** | Pod 秒级移动，数据有引力。节点宕机时使用本地存储的 Pod 会被卡住 |
| **Day 2 复杂度** | 数据库配置容易，长期运维是真正挑战：Schema 升级、引擎补丁、集群级升级 |
| **数据移动性** | 跨存储层迁移、跨可用区迁移、跨集群迁移仍是重大障碍 |

#### ⭐ 信号强度

⭐⭐⭐⭐ — 这不是技术公告而是**方向宣言**。SIG Storage 正从"让有状态应用跑起来"进入"为 AI 工作负载优化存储基础设施"的新阶段。特别值得关注的是 **COSI**（对象存储成为 K8s 一等公民）和 **Volume Health**（自动化存储健康管理）两条路线。
