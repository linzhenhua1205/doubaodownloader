### 1. Agent Auth: AI Agent 身份认证的法庭类比

**来源**: [CNCF Blog - Agent Auth](https://www.cncf.io/blog/2026/06/23/agent-auth-a-lawyers-day-in-court/) · 2026-06-23 · Lin Sun (CNCF Ambassador)

AI Agent = Microservices+，在传统微服务需求之上多了三层新要求：
- **身份认证**：Agent 可能代表多个不同用户行事
- **策略执行**：Agent 行为不可预测，需要护栏和策略
- **可观测性**：上下文、prompt、tool call、请求/响应内容

**核心架构洞见**：将 Agent 视为"律师"，用户视为"委托人"
- Agent Identity（律师是谁）→ Principal Identity（代表谁）→ Delegation Token（授权范围）
- **OBO Token（On-Behalf-Of）** 携带委托人身份 + Agent 身份 + 委托权限 + 范围
- 需要 **AI Native Gateway** 集中化身份验证、授权、审计

**技术栈组合**：SPIFFE（身份框架）+ cert-manager（证书管理）+ Istio（服务网格）+ AgentGateway（AI网关）

**关联知识库**：
- 上一轮 [2026-06-22](2026-06-22.md) 的 Agent on K8s（Orange 多Agent安全平台）

### 2. Jaeger v2.18: ClickHouse 后端正式推出（Alpha）

**来源**: [CNCF Blog - Jaeger ClickHouse Backend](https://www.cncf.io/blog/2026/06/23/building-jaegers-clickhouse-backend-8-6x-compression-on-10-million-spans/) · 2026-06-23 · Mahad Zaryab (Meta)

**基准测试结果**（10M spans / 1M traces / 单节点）：
| 指标 | 数值 |
|------|------|
| 写入吞吐 | **>50k spans/sec** |
| 压缩比（spans表） | **8.6×**（~6 GiB → ~722 MiB） |
| Trace查询 | **~100 ms** |
| 搜索查询 | **<50 ms**，复杂过滤 ~140 ms |

**架构决策要点**：
- **Primary Key** 选择 trade-off：按 `(service_name, name, start_time)` 排序（优化搜索）而非 `trace_id`（优化trace检索）。辅助用 bloom_filter + materialized view 补偿 trace 查询性能
- **5级属性存储**：resource / scope / span / event / link 五层，各层独立 `Nested` 列
- **Materialized Views** 预计算 service names / operations / trace ID 时间区间

**OTel 数据模型原生支持**：Jaeger v2 采用 OpenTelemetry 数据模型（v1 是 Jaeger 自有模型），支持 typed attributes（Bool/Int64/Float64/String/Bytes/Slice/Map）

## 🚀 K8s v1.36 关键特性（持续跟踪）

### Workload-Aware Scheduling GA 增强

**来源**: [Kubernetes Blog](https://kubernetes.io/blog/2026/05/13/kubernetes-v1-36-advancing-workload-aware-scheduling/) · 2026-05-13（收录更新）

**架构演进**：
- v1.35 → v1.36：**Workload → PodGroup 解耦重构**
  - Workload → 静态模板（template only）
  - PodGroup → 运行时对象（runtime state），晋升为一等公民 API（`scheduling.k8s.io/v1alpha2`）
- 新增 **PodGroup 调度周期（atomic scheduling cycle）**：
  1. 一次 cluster state snapshot → 2. 为全组 Pod 寻找合法 Node 放置 → 3. 原子性提交
  - 成功：全部绑到 Node；失败：全部回退并 backoff
- **Topology-aware scheduling**：支持按物理拓扑域（如 rack）约束 PodGroup 放置
- **Workload-aware preemption**（初版）：抢占时考虑工作负载语义

**对 AI/ML 调度的意义**：Gang scheduling + 拓扑感知 + DRA ResourceClaim 集成，为 AI 训练/推理工作负载提供原生调度能力
