## 1️⃣ 🔴 OTel: "Don't Wrap OpenTelemetry"（今日发文）

**来源**: OTel Blog (Jun 24, 2026) — Cijo Thomas (Microsoft)
**链接**: https://opentelemetry.io/blog/2026/dont-wrap-opentelemetry/

- **核心警示**: 很多团队为"简化"包装 OTel API（IMetric 接口、TelemetryHelper 等），但好心办坏事
- **反模式 #1 — 签名强制分配**: 包装器签名要求 `List<KeyValuePair<string, string>>` 或 `Vec<(String, String)>`，**每次调用都堆分配**；而 OTel 原生 API 针对 1-3 属性有零分配重载（.NET: `new("status", "ok")`；Rust: `&[KeyValue::new(...)]`），4-8 属性用 `TagList`（struct，栈上分配）
- **反模式 #2 — 查找包装器**: 按名称缓存仪表，每次 record 调用执行字典查找（ConcurrentDictionary GetOrAdd）甚至 Mutex 加锁，破坏 OTel 的「仪表创建一次、持有引用」性能模型
- **连锁问题**: 开发者学了包装器没学 OTel → 换团队/读文档要重学；维护负担随 OTel API 演进递增；调试时多一层间接性
- **正确做法**: 指向官方文档；测试用 InMemoryExporter（零包装）；治理需求用代码生成（OTel Weaver 是上游方向）
- **合理场景**: 遗留系统迁移期双写、治理约束（属性白名单/命名规则）→ 用编译时代码生成，而非运行时包装

**意义**: 这是 OTel 官方首次系统性批判 API 包装模式，对中大规模 OTel 接入团队有很强指导价值。

## 3️⃣ Agent Auth: AI Agent 身份与授权框架

**来源**: CNCF Blog (Jun 23, 2026) — Lin Sun, CNCF Ambassador
**链接**: https://www.cncf.io/blog/2026/06/23/agent-auth-a-lawyers-day-in-court/

- **核心论点**: AI Agent = Microservices+，它们需要传统微服务的一切，再加上：
  - 身份（Identity）— 谁在请求
  - 委托（Delegation）— Agent 代表谁行动
  - 策略（Policy）— Agent 被允许做什么
  - 审计（Audit）— 记录了哪些行动
- **技术栈建议**: OAuth 2.0 Token Exchange / OBO Token + SPIFFE 工作负载身份 + Istio 服务网格
- 提出 **AgentGateway** 概念：代理层统一处理身份验证、授权、审计和速率限制
- 与 Dapr 1.18 Verifiable Execution 形成互补——后者关注可验证执行历史


## 4️⃣ K8s v1.36: Pod-Level Resource Managers（Alpha）

**来源**: Kubernetes Blog
**链接**: https://kubernetes.io/blog/

- **Pod-Level Resource Managers** (Alpha): 在 Pod 级别而非容器级别管理资源分配
- **Tiered Memory Protection with Memory QoS**: 分层内存保护，热点数据优先保留
- **In-Place Vertical Scaling for Pod-Level Resources** (Beta): Pod 级资源的原地垂直扩缩容
- **PSI Metrics for Kubernetes Graduates to GA**
- **Workload-Aware Scheduling**: PodGroup 解耦 + atomic scheduling cycle
- **Volume Group Snapshots GA**: v1.27 Alpha → v1.32 Beta → v1.36 GA
