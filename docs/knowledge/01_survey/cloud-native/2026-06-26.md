## 1️⃣ OpenTelemetry 正式成为 CNCF 毕业项目 🔴

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/otel-graduates/) — 2026-05-21

CNCF 于 5 月 21 日宣布 **OpenTelemetry 正式毕业**（Graduated）。这标志着 OTel 成为 CNCF 最高成熟度级别的项目。

**关键意义**:
- OTel 由 OpenTracing 与 OpenCensus 合并而来，现已成为业界标准的 vendor-neutral 可观测性框架
- 数千贡献者、维护者、最终用户和组织参与了 OTel 的生态建设
- **不是终点**：社区将持续推动互操作性、高质量的可观测性标准与工具

**与之前信息的关联**: 此前跟踪过的 OTel 1.0、Profiles Public Alpha、GenAI 语义约定标准化系列里程碑，为毕业奠定了坚实基础。

## 2️⃣ Dapr 1.18: Verifiable Execution（可验证执行） 🔴

**来源**: [CNCF Blog - Dapr 1.18](https://www.cncf.io/blog/2026/06/11/introducing-verifiable-execution-in-dapr-1-18/) — 2026-06-11

Dapr 1.18 引入三大新能力，为工作流和 AI Agent 构建 **可验证执行（Verifiable Execution）** 基础：

**三大核心能力**:

| 能力 | 功能 |
|:-----|:------|
| **Workflow History Signing** | 对工作流历史记录生成**加密签名**，创建防篡改的执行历史，可独立验证 |
| **Workflow History Propagation** | 执行**溯源（lineage）** 随请求传递，使下游服务可追踪执行起源、参与方和事件序列 |
| **Workflow Attestation** | 基于 SPIFFE 工作负载身份，下游可接收**可加密验证的执行上下文** |

**架构意义**:
- SPIFFE 回答"你是谁？" → Verifiable Execution 回答"你怎么到这来的？"
- 对 AI Agent 尤为重要：Agent 调用工具、委派工作、协调多个 Agent 时，需要可验证的执行链
- 适用场景：银行转账需验证来自审批工作流；医疗索赔需验证经过资格核查和欺诈筛查

**对比传统方案**: 日志/审计记录需要信任假设，Dapr 1.18 提供**密码学保证**的执行完整性。

## 3️⃣ GenAI 可观测性: OTel Semantic Conventions 正式标准化 🟡

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/genai-observability/) — 2026-05-14

OTel 推出 **GenAI Semantic Conventions**，标准化 LLM 调用的监控数据模型：

**标准化的 GenAI 属性**:
- `gen_ai.request.model` — 使用的模型
- `gen_ai.usage.input_tokens` / `gen_ai.usage.output_tokens` — Token 计数
- `gen_ai.response.finish_reasons` — 停止原因（stop / tool_calls）
- `gen_ai.system_instructions`, `gen_ai.input.messages`, `gen_ai.output.messages` — 对话内容（可选捕获）

**生态落地**:
- **VS Code Copilot**: 已支持导出 traces/metrics/events
- **OpenAI Codex**: 导出结构化日志 + OTel metrics
- **Claude Code**: 导出 metrics + log events，trace 在 beta
- 提供 **GenAI Telemetry Visualizer** 开源工具，将 trace 渲染为对话视图

**可用指标**:
- `gen_ai.client.operation.duration` — LLM 调用延迟分布（可按模型过滤）
- `gen_ai.client.token.usage` — Token 消耗分布（按 input/output 区分）

## 4️⃣ OTel-Arrow Phase 2: 从高效传输到高效管道 🟡

**来源**: [OTel Blog](https://opentelemetry.io/blog/2026/otel-arrow-phase2/) — 2026-06-12

OTel-Arrow 进入 Phase 2，从高效传输协议扩展到**完整的列式遥测管道**：
- Phase 1 建立了 OTAP（OpenTelemetry Arrow Protocol），利用 Apache Arrow 列式格式实现高效传输
- Phase 2 扩展到管道各环节（收集、处理、存储）的列式效率
- 核心优势：列式格式在聚合、过滤和压缩方面比行式格式有数量级优势

## 5️⃣ Security Profiles Operator v1: 生产就绪的安全加固 🟡

**来源**: [CNCF Blog](https://www.cncf.io/blog/) — 2026-06-26, Sascha Grunert (Red Hat)

Red Hat 的 **Security Profiles Operator（SPO）** 达到 v1 里程碑：

**核心能力**:
- 管理 K8s 容器的 **seccomp**、**SELinux**、**AppArmor** 安全配置文件
- v1 目标：**稳定 API**、**安全加固**、推动功能合入上游 Kubernetes
- 解决了手动编写和维护安全配置文件的痛点

**与 K8s 生态关联**: 符合 CNCF "云原生 = AI-native" 趋势下对生产安全的要求。

## 7️⃣ Prometheus 3.0+: Native Storage for Composite Types 🟢

**来源**: [Prometheus Blog](https://prometheus.io/blog/) — 2026-02-14

Prometheus 社区正在推进一系列重大演进：
- **Native Storage for Composite Types**: 原生存储复合类型（如 histograms），解决累积的挑战
- 去弃用的 feature flags，稳定 OTel 支持
- **info() 函数**（PromQL）：一键为 metrics 添加标签元数据，解决 "churn problem"
- **UX Research Working Group** 成立，改善用户上手体验
