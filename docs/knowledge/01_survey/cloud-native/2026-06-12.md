## 🔥 重点信号

### 1️⃣ 🛡️ Dapr 1.18 Verifiable Execution：可验证执行框架

**来源**: [CNCF Blog · 2026-06-11](https://www.cncf.io/blog/2026/06/11/introducing-verifiable-execution-in-dapr-1-18/)
**项目**: Dapr（CNCF Graduated）| **版本**: v1.18

Dapr 1.18 引入了一套全新的 **可验证执行（Verifiable Execution）** 能力，将 Dapr 从"可靠执行"提升到"可信执行"层面。这是云原生领域少有的**密码学级执行完整性保证**。

#### 三大核心能力

| 能力 | 功能 | 类比 |
|:-----|:-----|:------|
| **Workflow History Signing** | 对工作流执行记录生成**加密签名**，可作为执行后篡改检测 | 相当于"执行日志的 HTTPS 证书" |
| **Workflow History Propagation** | 执行血统（lineage）随请求在系统中传播，下游组件知道请求来源 | 相当于"分布式版的 HTTP Referer + 签名" |
| **Workflow Attestation** | 工作流和活动可接收**加密可验证的执行上下文**，下游系统可基于验证血统做决策 | 相当于"工作流的数字护照" |

#### 为什么可观测性不够

日志、指标、追踪是"告诉我发生了什么"，但它们**需要被信任**。而 Verifiable Execution 提供的是**加密证明**——可独立验证，无法伪造。

#### 实际应用场景

| 场景 | 验证逻辑 |
|:-----|:---------|
| 🏦 银行转账 | 只接受经过批准支付工作流的请求，防止绕过欺诈检查的**直接 API 调用** |
| 🏥 医疗理赔 | 验证工作流依次通过了资格审核、欺诈筛查、医疗编码审查后，才允许理赔 |
| 💊 制药合规 | 质量控制和法规签核步骤完成之前，不得释放批次 |
| 🏨 医院 AI Agent | 验证用药建议来自授权的临床工作流，而非不可信 Agent |

#### SPIFFE 身份基础

Verifiable Execution 建立在 Dapr 已有的 **SPIFFE 工作负载身份** 之上：

```
SPIFFE 回答 "你是谁？"
Verifiable Execution 回答 "你是怎么到这来的？"
```

两者结合提供了比传统认证更完整的信任链。正如文章总结的：**"下一代分布式系统需要的不仅是弹性，而是信任。"**

#### ⭐ 信号强度评估

这是云原生领域在 **AI Agent 时代** 的关键基础设施演进。当 Agent 越来越多地自主做出业务决策时，能证明"这个决策是经过批准工作流产生的"将成为合规刚需。

### 2️⃣ 📈 KEDA v2.20.0 发布 + v2.20.1 补丁

**来源**: [KEDA Releases](https://github.com/kedacore/keda/releases)
**项目**: KEDA（CNCF Graduated）| **版本**: v2.20.0（June 1）+ v2.20.1（June 8）
**下一版本**: 预计 2026 年 9 月第二周

#### v2.20.0 主要新特性

| 类别 | 特性 | 说明 |
|:-----|:-----|:------|
| 🆕 新 Scaler | **OpenSearch Scaler** | 基于 OpenSearch 集群指标自动伸缩 |
| 🆕 新 Scaler | **Elastic Forecast Scaler** | **基于预测的弹性伸缩**——不再仅响应当前负载，而是预测未来负载 |
| 🔧 Fallback | **scalingModifiers fallback 行为** | 当 scaler 故障时定义回退策略 |
| 🔐 AWS | **AWS External ID** | 所有 AWS scaler 支持跨账号访问的 External ID |
| 📊 可观测性 | **Scaler HTTP 请求指标** | 新增 `keda_scaler_http_requests_total` 和 `keda_scaler_http_request_duration_seconds` 指标，追踪数据采集中的出站请求性能 |

#### 关键改进

- **CRD 级别验证** — ScaledObject/ScaledJob/TriggerAuthentication API 类型增加 `minimum`、`minLength`、`minItems`、`enum` 验证标记
- **Webhook OOM 修复** — 移除 admission webhook 验证热路径中的无条件 `json.MarshalIndent` 调用，改用 `reflect.DeepEqual`。**防止 60K+ ScaledObjects 规模下的 webhook OOM**
- **TLS 精细控制** — 新增 `KEDA_HTTP_TLS_CIPHER_LIST` / `KEDA_SERVICE_TLS_CIPHER_LIST` / `KEDA_SERVICE_MIN_TLS_VERSION` 环境变量
- **Leader Election** — 新增 `--leader-election-id` 标志，可配置 leader election Lease 名称
- **APIService 证书可选** — `APIService` 证书注入改为可选
- **Temporal Scaler** — 支持基于 **Worker Deployment Version** backlog 伸缩，废弃旧的 `buildId`/`selectAllActive`/`selectUnversioned` 参数

#### v2.20.1 补丁（June 8）

修复关键并发问题：多触发器同时缩放时的 **concurrent map read/write data race**（#7838）

#### ⚡ Elastic Forecast Scaler 意义

这是 KEDA 从**响应式伸缩**走向**预测式伸缩**的关键一步。基于预测的弹性伸缩意味着 K8s 工作负载可在负载到达前就调整副本数——对于 AI 推理等负载突发场景特别有价值。

### 3️⃣ 🏛️ CNCF IAM Whitepaper：身份即新安全边界

**来源**: [CNCF Blog · 2026-06-04](https://www.cncf.io/blog/2026/06/04/identity-and-access-management-whitepaper/)
**作者**: CNCF TAG Security and Compliance

CNCF TAG Security 发布了 **IAM 白皮书**，为云原生环境的身份和访问管理提供实践指南。这是云原生安全领域的重要参考文档。

#### 核心论点

> **随着云原生架构变得更加分布式、动态化、自动化，身份越来越成为新的安全边界。**

传统的认证和授权方法难以跟上短期工作负载、服务间通信和零信任需求。

#### 白皮书覆盖内容

| 主题 | 说明 |
|:-----|:------|
| 为什么 IAM 是云原生安全的基石 | 分布式架构中身份成为新的信任根 |
| 用户和工作负载的现代认证方法 | 短生命周期凭证、OIDC、SPIFFE |
| 基于边界 vs 零信任架构 | 何时选择哪种模式 |
| **PEP/PDP 架构** | 策略执行点（PEP）+ 策略决策点（PDP）分离的授权最佳实践 |
| **SPIFFE** | 如何使用 SPIFFE 实现安全的工作负载身份和服务间认证 |
| 有状态/无状态工作负载的安全模式 | 参考架构 |

#### 运维意义

对平台工程团队：PEP/PDP 架构是可复用的企业级授权模式。SPIFFE 不再是"可选增强"而正在成为云原生基础设施的基础组件——Dapr 1.18 的 Verifiable Execution、Istio 的服务间 mTLS 都建立在其上。

### 4️⃣ 🕵️ Jaeger v2 + AI Agent 追踪：MCP/ACP/AG-UI 三协议集成

**来源**: [CNCF Blog · 2026-05-26](https://www.cncf.io/blog/2026/05/26/how-jaeger-is-evolving-to-trace-ai-agents-with-opentelemetry/)
**作者**: Jonah Kowall（Jaeger Maintainer）
**项目**: Jaeger（CNCF Graduated）

#### 为什么 Jaeger 需要演进

AI Agent 的执行路径包括 **Prompt 组装 → 向量数据库检索 → 多个外部工具调用**，传统追踪工具难以映射这种复杂路径。Jaeger v2 + OTel + MCP/ACP/AG-UI 的三协议集成正是为此而生。

#### 架构设计

```
Jaeger UI ──AG-UI──→ Agent Gateway ──ACP──→ AI Agent
                         │
                         └──MCP──→ Jaeger MCP Server
```

| 协议 | 全称 | 作用 |
|:-----|:-----|:-----|
| **MCP** | Model Context Protocol | AI 模型安全访问外部数据源 |
| **ACP** | Agent Client Protocol | UI ↔ AI Agent 的标准化通信 |
| **AG-UI** | Agent-User Interaction Protocol | 工程师 ↔ AI Agent 的交互界面 |

#### 关键能力

- **自然语言查询 → 确定性追踪**：用户输入"找到支付服务中 500 错误且延迟超过 2s 的 trace"→ Agent Gateway 翻译为精确的 trace 查询
- **云端 LLM + 本地 SLM 双模式**：复杂推理用云端 LLM，隐私敏感场景用本地小模型
- **GenAI Semantic Conventions 可视化**：追踪 RAG pipeline 的 embedding 模型延迟、工具调用链、token 用量
- **统一二进制从开发到生产**：Jaeger v2 基于 OTel Collector，本地测试到生产部署使用相同二进制

#### 运维意义

这是 CNCF 可观测性生态对 AI 时代的直接回应。Jaeger 正在从"分布式追踪系统"演变为"**工程师 + AI Agent 协作调试平台**"。
