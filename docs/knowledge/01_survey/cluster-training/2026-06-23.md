## Ⅰ. KV Cache 优化

### ① Kamera: 位置无关多模态 KV Cache 训练-Free 复用 (arXiv:2606.23581, 6/22)

- **问题**: 多模态 Agent 反复查看相同视频帧/UI截图，每次滑动上下文窗口或迭代推理都要从零重编码——Prefix Cache 只服务于固定前导位置
- **核心洞察**: 盲重用 KV 块会丢失跨块条件依赖（cross-chunk conditioning），单跳检索不受影响但多跳准确率下降一半；此损失非对称——直接读出部分可通过标准状态合并精确恢复，但深层残留的低秩信号才是多跳推理的关键
- **方案**: 为每个位置无关的 KV 块附带一个**小规模训练-free 低秩条件补丁（low-rank conditioning patch, rank-m）**，复用退化为一个算子：精确 RoPE 再旋转到任意目标位置 + 补丁还原跨块绑定
- **三个窗口操作为零重编码**: reorder（同一补丁服务缓存集的所有排序）、sliding-window survival（幸存块仅通过旋转搬迁）、recall（驱逐块通过补丁重新水化）
- **效果**: 在 MM-NIAH 跨块绑定基准和两页文档 QA 上恢复完整任务准确率；在 SGLang 生产内核中跨6个 backbone 将重前置 KV 重建到 bf16 舍入精度
- **意义**: 多模态 Agent 的 KV 重用进入"补丁 + 旋转"时代，训练-free 低秩补丁思路可扩展到 MLA/GQA/MHA 全系注意力架构

### ② Recency/Frequency 自适应 KV Cache 调度 (arXiv:2606.21238, ICML 2026 Workshop AdaptFM)

- **问题**: vLLM 的 LRU 逐出策略在多租户多工作负载场景下，不相关工作互相冲刷对方缓存
- **方案**: 在近期（recency）和频繁（frequency）出现 KV 块之间动态分配缓存空间，替代单纯 LRU
- **效果**: 合成文档 QA 上 KV 命中率提升 **10.8%**，TTFT 降低 **12.6%**；真实对话工作负载上分别提升 **2.1%/2.0%**
- **亮点**: 方法对 batch inference 泛化良好，具备可解释性，能有效区分多样化工作负载
- **评述**: 从简单 LRU 走向 workload-aware 自适应策略是 KV 缓存调度的重要演进方向

## Ⅱ. PD 分离部署与异步推理

### ③ ASAP: MoE Prefill 的分离与异步推理系统 (arXiv:2606.22541, 6/21)

- **问题**: MoE 模型推理采用 DP(Attention) + EP(Expert) 混合并行，DP 组间频繁全局同步屏障，在线场景下请求到达率和序列长度差异导致 DP 不平衡，产生严重同步等待
- **方案**: 将 Attention 与 MoE 阶段**完全解耦（disaggregate）**，实现全异步执行流水线——通过一套专用异步通信原语 + 请求调度和模型执行四维协同优化，拆除全局同步屏障
- **测试平台**: CloudMatrix384 超节点
- **效果**: SLO 合规的 prefill 吞吐量相比最新同步方案提升 **90%**
- **意义**: 首次将 PD 分离理念引入 MoE Prefill 阶段，证明异步流水线可大幅降低 DP 同步瓶颈

## Ⅲ. Checkpoint 容错

### ④ Concordia: JIT 编译持续内核 Checkpoint 实现容错 LLM 推理 (arXiv:2606.23521, 6/22)

- **问题**: 长时间运行 LLM Agent 的 GPU 状态（KV cache、请求调度器、通信状态、在线 adapter）丢失后会导致数分钟到数小时的推理成果作废；现有机制要么重启整个 serving 栈，要么需在每个 attention 组件中嵌入应用级 checkpoint 逻辑
- **方案**: **Concordia** — 以设备驻留持续内核（device-resident persistent kernel）作为容错推理底层基板：
  - 拦截 GPU 模块加载，支持 PTX 级和 SASS 级指令插桩
  - 对每个注册的 LLM 状态区域 JIT 编译专用 delta-checkpoint handler（如 KV-block scanner、adapter-page scanner、recovery applier）
  - 持续内核消费无锁 ring buffer 的 compute/checkpoint/append-log/recovery 任务流
  - Delta 记录通过 CXL 内存或 host DRAM 的 CPU 可见日志追加
- **关键设计**: checkpoint hooks 在设备同步点执行，无需 host CPU 介入关键路径
- **意义**: 首次将"持续内核（persistent kernel）"范式引入 LLM 推理容错，checkpoint 粒度精细（delta）、开销隐藏、故障恢复路径不经过 host

## Ⅳ. 分布式训练并行策略与通信调度

### ⑤ Factored Gossip DiLoCo: 降低 DiLoCo 的阻塞通信 (arXiv:2606.22768, ICML 2026)

- **问题**: DiLoCo 虽降低了通信频率，但其外同步（outer sync）仍为带宽密集型和对 straggler/瞬态故障脆弱
- **方案**: 将精确同步松弛为**近似同步（mixing/gossip）**，将 DiLoCo 同步分解为：
  - **非阻塞 mixing 步骤** — 与计算重叠，零 staleness
  - **阻塞 mixing 步骤** — 收紧工作节点一致性
  - 两者组合实现计算利用率与优化稳定性之间的**可调权衡**
- **效果**: 低带宽设置下（如跨数据中心），相比 DiLoCo 显著提升计算利用率，训练进度可比较至接近匹配，且对故障更鲁棒
- **意义**: 通过 gossip 协议将同步通信异步化，为"弱连接"分布式训练（跨 DC、异构网络）提供了可行路径

### ⑥ Online Dynamic Batching (ODB): 带形式化保证的 LLM 训练动态批处理 (arXiv:2606.19989, 6/18)

- **问题**: 现代 LLM 训练中，样本的真实成本（padding/内存/GPU 饱和度）只有在预处理、增强、模板、tokenization 和多模态视觉 token 扩展后才可观测——离线 batch 采样器"盲"
- **方案**: DataLoader 侧的 drop-in 系统，在精确可观测点动态构建 batch，同时保持 DDP step 对齐——形式化为**分布式组对齐问题（Distributed Group Alignment Problem, DGAP）**，证明无死锁有界终止
- **效果**（Qwen3-VL 2B/8B）:
  - 单节点 Full FT/LoRA: 1.58-2.51× 吞吐提升
  - 两节点 Full FT: 1.71-3.78×
  - 生产多模态混合: **4.43×**
  - 相比离线 token-budget oracle 方案仅差距 15%，且在高 CV 数据集上反而更快 2.24-3.69×
- **无需任何**: 模型/优化器/attention kernel 修改，零侵入
- **意义**: 将动态 batching 从"工程 hack"提升为带形式化保证的系统方案

## Ⅴ. LLM Serving 系统演进

### ⑦ LiveServe: 交互感知的实时全模态 LLM 服务 (arXiv:2606.22983, 6/22)

- **问题**: 实时全模态 LM 支持语音交互（流式输入、音频生成、随时打断），现有 serving 系统仍采用吞吐优化调度和 LRU KV offloading，忽略音频回放进度和多轮复用
- **方案**: LiveServe 将回放进度、语音活动检测和打断事件暴露给 serving 流水线：
  - **调度器**: 优先首音频和接近欠载的会话，限制超出回放前沿的生成
  - **KV 管理器**: 基于下次使用感知的逐出策略，在用户语音期间预加载可能需要的 KV
- **效果**（在 vLLM-Omni 上）: P90 音频 TTFP 平均降低 **1.55×**（最高 2.21×），completed-request 吞吐提升平均 **1.15×**（最高 1.56×），KV 重载工作移至非关键路径
- **意义**: 实时全模态场景下，"交互感知"正取代"吞吐优化"成为 serving 调度的第一原则

### ⑧ Human-Less LLM Serving: 量化"人类税"对吞吐的影响 (arXiv:2606.20577, 5/3)

- **问题**: 所有主流 LLM serving 系统以 TTFT/TPOT SLO 为设计目标，但长时间运行的 AI 任务（程序化调用 LLM）无需满足人类感知延迟
- **发现**: "人类税"（为满足 TTFT/TPOT 而牺牲的吞吐）随上下文长度剧增：
  - **64K token 上下文** → 人类税高达 **60-93%**
  - 高并发下人类税更大
  - 在 SGLang 和 Sarathi-Serve 上定性结果一致
- **主张**: Serving 系统应暴露 workload-class-aware SLA 配置，而非统一施加人类税
- **意义**: 揭示了一个"房间里的大象"——相当比例的 LLM 调用（Agent 循环、批处理推理）根本不需要亚秒级 TTFT

## Ⅵ. 集群可观测性与性能诊断

### ⑨ ARGUS: 万卡级生产集群的 Tracing 与性能诊断 (arXiv:2606.20374, 6/18)

- **已在昨日跟踪中覆盖**，今日补充关键数据：
  - 在 10,000+ GPU 生产集群已部署运行 **6个月以上**
  - 压缩比: 每 rank 每 step 从 10 MB 原始 kernel 事件压缩至 **2.7 KB** (~3,700×)
  - 诊断覆盖: 计算 straggler、链路降级、流水线气泡放大、FlashAttention JIT stall、通信伪装计算 straggler 等 5 类典型异常
