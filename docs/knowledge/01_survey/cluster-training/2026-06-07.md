# 万卡集群/训推优化动态（2026-06-07）

> **来源**: arXiv / vLLM GitHub / JD Tech / SGLang  
> **更新方向**: Speculative Decoding新范式、长上下文Prefill加速、RL WAN训练、企业级推理框架、梯度无损压缩  
> **核心趋势**: SD从加速推理→加速RL训练→Agent系统全链路；Prefill加速走向架构无关+原生引擎集成；RL训练不再依赖RDMA集群


## 1️⃣ Speculative Decoding: 从加速推理到全链路效率革命

### Domino — 解耦因果建模与自回归草案执行 (arXiv:2605.29707)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | 自回归草案器建模因果依赖但引入顺序开销；并行草案器降低开销但削弱块内依赖建模 |
| **方法** | 并行草案骨干网络生成初步草案分布 + 轻量Domino Head用前缀相关因果信息精炼 → 因果建模与自回归执行的彻底解耦 |
| **训练** | Base-anchored training curriculum：先强化并行骨干 → 逐步偏移向因果修正最终分布 |
| **加速比** | **5.49x** end-to-end (Transformers后端) / **5.8x** throughput (SGLang serving) on Qwen3 |
| **意义** | 首次严格解耦因果依赖建模与自回归执行，打破了SD中长期存在的draft质量 vs 成本trade-off |

### SpecActor — Fastest-of-N多草案方法选择 (arXiv:2511.16193, 阿里)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | RL post-training中rollout占训练时间主导，但SD在大batch训练场景下效率下降 |
| **方法** | (1) Decoupled speculation：拆解SD与大batch的冲突；(2) Fastest-of-N：根据rollout进程动态选择和组合不同draft方法，即使最优方法未知 |
| **加速比** | **2.0-2.4x** 平均rollout加速，最高**2.7x**；相比vanilla SD额外提升**1.1-2.6x** |
| **覆盖模型** | 同时适用于Dense和MoE模型 |
| **意义** | 解决了"训练场景下哪种draft方法最优"这一不确定性问题 |

### ReSpec — SD适应RL训练的三重机制 (arXiv:2510.26475)

| 维度 | 内容 |
|:-----|:------|
| **核心挑战** | SD直接用于RL训练三个gap：大batch下加速递减、drafter随actor更新而过时、drafter导致的policy退化 |
| **方法** | (1) 动态调整SD配置；(2) 通过知识蒸馏持续演进drafter；(3) 按rollout reward加权更新 |
| **加速比** | 最高**4.5x**速度提升 (Qwen 3B-14B) |
| **意义** | 首次系统解决SD在RL训练环境中的适配问题 |

### DAS — 历史rollout分布感知加速 (arXiv:2511.13841)

| 维度 | 内容 |
|:-----|:------|
| **核心洞察** | RL rollout长度呈长尾分布：少量长序列主导响应时间；历史rollout揭示prompt级别的稳定模式 |
| **方法** | (1) 非参数自适应草案器：增量维护suffix tree利用近期rollout；(2) Length-aware speculation：对长轨迹分配更激进draft预算 |
| **加速比** | rollout时间减少最多**50%**，训练曲线完全一致 |
| **意义** | 利用RL训练本身产生的历史rollout作为免费draft资源 |

### EDA — 参数高效草案模型适配 (arXiv:2603.09527)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | Target模型finetune后，draft模型需全部重训，代价高 |
| **方法** | (1) 解耦架构：共享组件+轻量私有组件；(2) 数据再生：利用finetuned target重新生成训练数据；(3) 样本选择机制 |
| **加速比** | 显著降低训练成本的同时保持接近全重训的acceptance length |
| **意义** | 使SD在target模型持续更新场景下保持实用 |

### FLy — Training-Free松散SD，超EAGLE-3 (arXiv:2511.22972, ICLR 2026)

| 维度 | 内容 |
|:-----|:------|
| **核心洞察** | 严格exact-match验证丢弃大量语义等价的正确continuation |
| **方法** | 利用target模型自纠正行为判断draft-target不匹配是否语义有效；两阶机制：entropy gate + token-level deferred window |
| **加速比** | **2.81x** (Llama-3.1-70B), **5.07x** (405B)；OOD场景超EAGLE-3达**1.62x** |
| **意义** | Training-free + 任意draft-target组合通用，语义级验证范式突破 |


## 2️⃣ Prefill加速与PD分离生态演进

### UniPrefill — 通用长上下文Prefill加速 (arXiv:2605.06221)

| 维度 | 内容 |
|:-----|:------|
| **核心突破** | 现有prefill加速仅针对full-attention模型，对linear/full-attention hybrid或sliding window模型性能大幅下降 |
| **方法** | Block-wise动态稀疏化，在token级别直接加速模型计算；实现为continuous batching operator |
| **引擎集成** | 原生扩展vLLM调度策略：支持UniPrefill的PD co-processing + tensor parallel |
| **加速比** | 最高**2.1x** TTFT加速，随并发请求数增加效果更显著 |
| **意义** | 首个架构无关（任意attention pattern）的prefill加速框架，原生融入vLLM |

### ESS — DeepSeek-V3.2长上下文Latent-Cache卸载 (arXiv:2512.10576)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | PD分离下Decode stage中Latent-Cache随序列长度线性增长 vs GPU内存有限，限制batch size |
| **方法** | Offload Latent-Cache至CPU内存，关键数据保留GPU；batch-size扩展与GPU内存解耦 |
| **加速比** | 32K context +**69.4%** throughput，128K +**123%** |
| **意义** | 使DeepSeek稀疏注意力机制在长上下文场景真正释放性能 |

### PPD — Not All Prefills Are Equal (arXiv:2603.13358, ICML 2026)

| 维度 | 内容 |
|:-----|:------|
| **核心洞察** | 多轮对话中append-prefill（仅处理新token+复用KV cache）对decode的干扰远小于full prefill |
| **方法** | PPD disaggregation：动态路由Turn2+请求到decode节点本地处理（复用缓存的KV） |
| **加速比** | Turn2+ TTFT降低约**68%**，有效缓解KV传输拥塞 |
| **意义** | 基于prefill类型差异的精细化PD分离路由 |

### Trinity — 将向量搜索从PD分离中再分离 (arXiv:2512.02281)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | RAG检索+prompt cache等向量搜索任务仍与PD纠缠，放大tail latency |
| **方法** | 独立共享向量搜索GPU池 + continuous batching + stage-aware调度 |
| **意义** | 从PD双分离走向P/D/Vector Search三分立架构 |


## 3️⃣ 梯度压缩与通信优化

### ZipCCL — 无损压缩集合通信库 (arXiv:2604.27844)

| 维度 | 内容 |
|:-----|:------|
| **核心洞察** | 训练中通信数据（激活值/梯度/参数）呈近高斯分布 → 可利用指数编码高效无损压缩 |
| **方法** | (1) 理论指数编码（无需在线统计）；(2) GPU优化压缩/解压kernel（通信感知数据布局）；(3) 自适应策略动态切换collective操作 |
| **平台** | 64-GPU集群，MoE + Dense transformer模型 |
| **加速比** | 通信时间减少**1.35x**，端到端训练**1.18x**加速，无损模型质量 |
| **意义** | 首个针对LLM训练中高斯分布特性的无损压缩库，直接集成到NCCL兼容层 |

### SparrowRL — WAN上RL训练，逼近RDMA性能 (arXiv:2602.11456)

| 维度 | 内容 |
|:-----|:------|
| **核心洞察** | RL微调每步仅~1%参数变化 → 稀疏delta checkpoint代替全weight广播 |
| **方法** | (1) 每步表示为稀疏delta checkpoint；(2) 多流pipeline提取；(3) 与rollout生成重叠传输；(4) 吞吐/带宽感知调度+租约容错 |
| **加速比** | 8B模型payload减少**79x**；WAN吞吐提升**2.4-9.5x**；与理想RDMA单数据中心差距缩小至**8.91%** |
| **成本效益** | 1.21-1.59x tokens/美元 vs 预留RDMA集群 |
| **意义** | 使RL post-training脱离RDMA依赖，可在跨云普通以太网上运行 |


## 4️⃣ 检查点与容错

### ReCoVer — 512卡丢256卡仍保持轨迹 (arXiv:2605.11215)

| 维度 | 内容 |
|:-----|:------|
| **方法** | 三层解耦协议：(1) 故障容忍collectives隔离故障传播；(2) 步内细粒度恢复；(3) 动态微批次重分配 |
| **并行兼容** | 直接集成3D并行/HSDP |
| **加速比** | **2.23x** 有效吞吐 vs checkpoint-restart；多处理**74.9%** tokens |

### SPARe — 600K GPU容错 (ICML 2026, arXiv:2603.00357)

| 维度 | 内容 |

| **加速比** | 近常数2-3x计算开销；极端规模(600K GPU)减少训练时间**40-50%** |

### LLMTailor — 按层选择性检查点 (PDSW'25, arXiv:2602.22158)

## 6️⃣ 3D并行优化

### TrOPD — 信任区域On-Policy蒸馏 (arXiv:2606.01249, 6月3日提交)

| 维度 | 内容 |
|:-----|:------|
| **核心问题** | OPD训练中teacher-student分布差异大时，teacher监督在student生成token上的policy gradient不可靠 |
| **方法** | 仅在teacher提供可靠监督的区域执行OPD；outlier区域用gradient clipping/masking/forward-KL；off-policy guidance |
| **加速比** | 在数学推理、代码生成、通用领域持续超越OPD/EOPD/REOPOLD |
| **意义** | 解决distribution mismatch这一OPD核心难题 |

### MIST — 异构多阶段LLM推理协同设计 (arXiv:2504.09775)

| 维度 | 内容 |
|:-----|:------|
| **能力** | 建模RAG+KV检索+reasoning+prefill+decode多阶段pipeline；支持异构硬件（AMD GPU/TPU/ASIC）混合部署 |
| **方法** | 真实硬件trace + 分析建模捕捉memory bandwidth contention/跨集群通信/batching效率 |
| **意义** | 为跨vendor PD分离配置提供统一评估框架 |
