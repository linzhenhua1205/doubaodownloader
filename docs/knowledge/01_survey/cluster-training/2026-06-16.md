## 1️⃣ 🏆 CacheWise：面向Coding Agent工作负载的KVCache管理层（arXiv:2606.16824, 06/15新提交 ⭐）

**核心问题**: Coding Agent（如GitHub Copilot、Cursor、Claude Code）是增长最快的LLM应用场景，但它们的行为与传统Chat工作负载根本不同——长运行闭环session中LLM生成与外部工具调用交替进行，产生**持续的高KVCache压力**，传统LLM serving策略应对极差

**核心观察**: 对真实coding assistant trace的数据集分析发现——
- Coding agent session会**重复重用大规模前缀**（同一代码上下文多次请求）
- 持续的高KV cache压力下，传统vLLM eviction策略导致**频繁的重计算**
- 工具调用元数据（tool call metadata）可作为**轻量级前缀重用预测器**

**三部分构成**:

| 组件 | 机制 | 效果 |
|:-----|:-----|:------|
| **Prefix-Aware Scheduling** | 识别coding agent session中的重复前缀模式 | 提高cache命中率 |
| **Reuse-Aware Eviction** | 结合tool call metadata的轻量级预测，决定保留哪些KV以最大化复用 | 减少不必要的eviction |
| **vLLM原生集成** | 作为cache管理插件层，无需修改底层调度 | 低迁移成本 |

**性能数据**:
- KVCache evictions减少 **2-2.6×**（vs vLLM默认策略）
- 完成总session时间提升 **up to 3.5×**
- 基于真实coding assistant traces验证

**意义**:
- **首个针对Coding Agent工作负载的系统级KVCache研究工作** — 传统研究（chat/长上下文）与coding agent workload有根本差异
- 工具调用元数据作为cache预测器的思路极具实用价值——session中下一个工具调用是什么，决定了哪些KV将被重用
- 随着AI编程工具（Cursor/Claude Code/Copilot Codex）的爆发式增长，这个方向将成为推理基础设施的核心优化点

## 2️⃣ Gimbal：面向MoE LLM级联的端到端协同调度系统（arXiv:2606.15177, 06/13新提交 ⭐）

**核心问题**: MoE LLM服务面临双重挑战——动态请求负载与稀疏Expert路由交织，导致**数据并行（DP）引擎不平衡**和**Expert级热点**。现有系统将frontend调度和backend expert均衡分开做，无法对fine-grained engine压力、backend MoE压力和source-dependent expert流量做出协同响应

**两层协同调度架构**:

| 层级 | 机制 | 关键创新 |
|:-----|:-----|:---------|
| **Fine-Grained DP-Engine调度** | 在线backend压力信号：KV cache用量、剩余prefill工作、队列压力、MoE Expert压力 | **不是用粗粒度请求计数器，而是用微观系统信号**做路由决策 |
| **轻量级Prefill-Aware队列策略** | 带aging的优先队列排序 | 不需要输出长度预测就能减少HOL blocking |
| **Source-Aware Expert负载均衡** | 混合整数非线性规划(MINLP)启发式专家放置 | 同时考虑expert负载、source感知通信、迁移稳定性 |

**性能数据**:
| 指标 | 改善 | 对比基线 |
|:-----|:-----|:---------|
| **TTFT** | **↓42.9%** | over vLLM |
| **TPOT** | **↓33.3%** | over vLLM |
| 高负载请求吞吐 | **↑3.0%** | over vLLM |

**意义**:
- **MoE服务的"感知盲区"被打破** — DP-engine调度不应该只用请求计数器，而应该看backend实时压力信号（KV cache余量、prefill队列等）
- MINLP专家放置是**首个同时优化负载均衡+通信代价+迁移稳定性**的综合方案
- 对DeepSeek-V4等大规模稀疏MoE模型的在线服务架构有重要启示

## 3️⃣ NVIDIA CuTe DSL融合MoE Kernel：全栈训练优化（NVIDIA技术博客, 06/15发布 ⭐）

**核心问题**: MoE训练中三大系统瓶颈——激活函数内存密集（Tensor Cores空闲）、CPU-bound同步（MoE tok/exp动态计算）、量化额外内存开销

**三项融合Kernel创新**:

| Fusion Kernel | 解决的问题 | 技术方案 |
|:--------------|:-----------|:---------|
| **GroupGemm + Quantize** | 消除量化的额外内存读写 | 量化fusion到GEMM epilogue |
| **GroupGemm + Activation + Quantize/Transpose** | 消除激活函数内存bound + 量化开销 | GLU权重repack使同一thread block同时访问input和gate |
| **GroupGemm + dActivation + Quantize/Transpose** (反向) | 反向传播同样瓶颈 | CuTe DSL实现 |

**关键使能技术**:
1. **GLU权重Repack** — 将input和gate权重在checkpoint加载时预打包成列拼接，使同一thread block能从epilogue中直接计算SwiGLU，无需回global memory
2. **GPU内存内tok追踪** — CuTe DSL GroupGEMM在GPU内存中追踪tokens/group，**无需CPU依赖**，使完整iteration使用CUDA Graph消除CPU bottleneck
3. **MXFP8/NVFP4量化融合** — 量化直接进入GEMM epilogue，BF16→MXFP8的额外memory pass被消除

**性能数据**:
| 层次 | 加速比 |
|:-----|:-------|
| Kernel级Forward | **1.3×** |
| Kernel级Backward | **2.1×** |
| **DeepSeek-V3端到端训练** | **+8%** |
| **GPT-OSS端到端训练** | **+93%** |

**可用性**: cuDNN Frontend v1.23.0+ / Transformer Engine v2.15+ / Megatron Core 26.04-alpha.rc2+

**意义**:
- **MoE训练的全栈模版化** — CuTe DSL使得硬件感知kernel编写从手工优化走向**声明式DSL**，且实现"一次编写，多框架可用"
- 93%的GPT-OSS端到端提升说明：**传统unfused MoE实现的优化空间远大于一般人估计**
- GLU repack思路对后继SWiGLU/GeGLU变种具有通用借鉴意义

## 4️⃣ cuTile Rust：安全的GPU Kernel作者系统（arXiv:2606.15991, 06/14新提交 ⭐）

**核心问题**: Rust使CPU上的安全系统编程成为可能，但编写自定义GPU Kernel仍然迫使开发者离开Rust的所有权保证——Rust的安全优势无法延伸到GPU

**cuTile Rust的设计**:
- 将Rust的**所有权纪律扩展**到tile-based GPU Kernel：可变输出被分割为不相交的chunk，kernel launch保持host侧所有权契约
- 支持**可组合的host执行模型**：同步启动、异步pipeline、CUDA Graph重放
- 提供local opt-out机制，当需要底层控制时，可在局部绕过安全检查

**性能数据**:
| 指标 | 值 | 对比 |
|:-----|:---|:-----|
| 逐元素操作 | **7 TB/s** | B200 |
| GEMM | **2 PFlop/s** = **96% cuBLAS** | 与cuTile Python在测量噪声内一致 |
| Grout (Qwen3-4B, RTX 5090) | **171 tokens/s** (batch-1 decode) | 与vLLM/SGLang竞争 |
| Grout (Qwen3-32B, B200) | **82 tokens/s** (batch-1 decode) | 与vLLM/SGLang竞争 |
| HBM Roofline | 一致 | sanity check通过 |

**Grout Inference Engine** — 基于cuTile Rust构建的端到端推理引擎

**意义**:
- **"Fearless Concurrency on the GPU"** — 这是NVIDIA Research将Rust内存安全移植到GPU Kernel的里程碑式工作
- **性能无损** — 96% cuBLAS打破了"安全=性能折中"的惯性思维
- Grout引擎的初步结果（与vLLM/SGLang竞争）说明**从安全Kernel DSL构建生产级推理引擎是可行的**
- 对可靠性和安全性要求极高的AI基础设施（如金融、医疗、自动驾驶）场景有长远意义
