## 一、KV Cache 优化

### ① CompressKV — 语义检索引导的 GQA KV Cache 压缩
- **论文**: [CompressKV: Semantic-Retrieval-Guided KV-Cache Compression for Resource-Efficient Long-Context LLM Inference](https://arxiv.org/abs/2606.24467)
- **日期**: 2026-06-23 | **机构**: TU Darmstadt
- **核心思路**: 在 GQA 架构 LLM 中，注意力头功能不同（语义检索头 SRH vs 其他头），现有方法聚合所有头的注意力分数会导致关键 token 被误逐。CompressKV 识别 SRH（同时关注 prompt 首尾 token 和语义重要的中间证据 token），仅用 SRH 分数指导 token 保留决策。
- **关键数据**:
  - LongBench QA: **97%** 全缓存性能，仅用 **3%** KV 缓存
  - Needle-in-a-Haystack: **90%** 准确率，仅用 **0.7%** KV 存储
- **意义**: 首次将 GQA 头功能分化引入 KV 缓存压缩，3% 缓存保 97% 性能是当前该领域最优水平之一
- **代码**: https://github.com/TUDa-HWAI/CompressKV

### ② Block-GTQ — RoPE 感知的 KV Cache 量化位宽分配
- **论文**: [RoPE-Aware Bit Allocation for KV-Cache Quantization](https://arxiv.org/abs/2606.24033)
- **日期**: 2026-06-23 | **机构**: JIA Lab
- **核心思路**: 在 RoPE 下，key 对注意力 logit 的贡献分解为二维频率块的逐位置和。不同 RoPE 能量块的量化敏感性不同。Block-GTQ 对每个 RoPE 块计算无标签能量分数，按边际增益贪心分配整数位宽。
- **关键数据**:
  - K2V2 (Llama-3.1-8B-Instruct): NIAH 六任务平均 **70.6→97.4**，LongBench-EN **36.87→53.31**
  - K3V2 (DeepSeek-R1-Distill-Qwen-7B, AIME 2024/2025): **51.7/37.5** (vs fp16 **54.2/37.9**)，均匀 TQ-MSE 崩塌到 **0.0/0.0**
  - 单 H800 (Qwen2.5-3B-Instruct): K3V3 **3.24×** 压缩，**1.34×** 快于 fp16 FA2 at 128K，峰值内存 56.31 GB → **19.85 GB**
- **意义**: 首次将 RoPE 分块结构引入量化位宽分配，2 bit K-only 量化下在 AIME 上将均匀基线从 0% 拉回 ~52%
- **代码**: https://github.com/JIA-Lab-research/blockgtq

### ③ Nexus Sampling — 流式 KV Cache 逐出的概率采样框架
- **论文**: [Forget Without Compromise: Nexus Sampling for Streaming KV-Cache Eviction Under Fixed Budgets](https://arxiv.org/abs/2606.23961)
- **日期**: 2026-06-22
- **核心思路**: 现有确定性 Top-K 逐出策略的问题在于，单步的分数噪声可能永久删除关键 token。Nexus Sampling 用加权蓄水池采样替代确定性 Top-K，将包含概率而非硬边界作为保留依据。
- **关键数据**:
  - 80% KV 逐出下，与密集注意力偏差 <**1%** (LongBench)
  - 在检索密集型任务上优势显著，每序列缓存内存减少高达 **10×**
- **意义**: 提出"概率保留 + 桥接 token"新范式，挑战了已有认知中"确定性逐出最优"的默认假设

### ④ Kamera — 位置无关多模态 KV Cache 的低秩复用修补
- **论文**: [Kamera: Unified Position-Invariant Multimodal KV Cache for Training-Free Reuse](https://arxiv.org/abs/2606.23581)
- **日期**: 2026-06-22
- **核心思路**: 多模态 Agent 反复查看同帧/同截图。已有位置无关缓存恢复单跳检索准确，但跨块多跳准确率减半。Kamera 识别出丢失的是每个 chunk 吸收的邻居跨块条件化信号（残差集中在深层、低秩），用 rank-m 低秩补丁恢复。
- **关键数据**: rank-m 补丁在 MM-NIAH 和双页 doc-QA 上恢复完整准确率，在 SGLang 生产核上恢复到 bf16 精度
- **意义**: 首次从理论和实证上分离"直接读出恢复"和"跨块多跳绑定丢失"，用低秩补丁精确修复

### ⑤ SAC — 面向稀疏注意力的 CXL 解聚 KV 缓存
- **论文**: [SAC: Disaggregated KV Cache System for Sparse Attention LLMs with CXL](https://arxiv.org/abs/2606.19746)
- **日期**: 2026-06-18 | **机构**: 清华/Tsinghua
- **核心思路**: 现有 RDMA 解聚 KV 缓存对稀疏注意力极度低效——解码时只激活少量 KV 条目但仍需完整拉取。SAC 利用 CXL 的 cache-line 粒度 load/store 语义按需获取 Top-K KV 条目。
- **关键数据** (DeepSeek-V3.2 + SGLang):
  - 吞吐 **2.1×**
  - TTFT 降低 **9.7×**
  - TBT 降低 **1.8×**
- **意义**: CXL 取代 RDMA 成为稀疏注意力 LLM 的解聚 KV 基础设施

### ⑥ UltraQuant — 上下文重 Agent 的 4-bit KV 缓存
- **论文**: [UltraQuant: 4-bit KV Caching for Context-Heavy Agents](https://arxiv.org/abs/2606.20474v2)
- **日期**: 2026-06-18
- **核心思路**: 上下文重 Agent（长前缀+短回合复用）的特殊压力下，4-bit KV 压缩的性能提升。AMD CDNA4 原生 FP4 支持。
- **关键数据**: P50 TTFT 降低 **3.47×** (缓存压力后期)，**2.3×** (全部轮次)；输出吞吐提升 **1.63×** vs FP8 KV 基线

## 二、MoE 推理与服务

### ⑦ CrossPool — 冷 MoE 多模型 KV Cache 与权重解聚
- **论文**: [CrossPool: Efficient Multi-LLM Serving for Cold MoE Models through KV-Cache and Weight Disaggregation](https://arxiv.org/abs/2606.24506)
- **日期**: 2026-06-23 | **机构**: 北理工/BIT
- **核心思路**: 冷 MoE 模型极少同时达到 KV 缓存峰值需求。CrossPool 将 FFN 权重和 KV 缓存分离到两套 GPU 内存池：权重池合并冷模型 FFN 权重，KV 缓存池动态服务活跃请求。
- **关键数据**: P99 TBT 降低高达 **10.4×** vs 最先进的基于 KV 缓存的多模型服务系统
- **意义**: 首次为"冷 MoE"场景设计专业化服务系统，权重/KV 双池分离为 GPU 内存利用率带来数量级提升

### ⑧ ASAP — 解聚异步 MoE Prefill 推理系统
- **论文**: [ASAP: A Disaggregated and Asynchronous Inference System for MoE Prefill](https://arxiv.org/abs/2606.22541)
- **日期**: 2026-06-21 | **机构**: HUAWEI Cloud
- **核心思路**: MoE 的 DP(注意力)+EP(MoE) 混合并行需要频繁全局同步屏障。ASAP 用异步通信原语+4 项协同优化，实现 Attention↔MoE 全异步管道，拆除全局同步。
- **关键数据** (CloudMatrix384): SLO 合规吞吐提升 **90%** vs SOTA 同步方案
- **意义**: 实证证明 MoE 中 DP↔EP 全局屏障可完全拆除，为 CloudMatrix 超节点提供了系统级验证

### ⑨ UltraEP — 机架级 MoE 训练与推理的精确负载均衡
- **论文**: [UltraEP: Unleash MoE Training and Inference on Rack-Scale Nodes with Near-Optimal Load Balancing](https://arxiv.org/abs/2606.04101v3)
- **日期**: 2026-06-18
- **核心思路**: 机架级节点（RSN）内数十 GPU 扩展连接使每个微批次和层重新均衡成为可能。UltraEP 使用配额驱动的贪心规划器+持久 tile 流传输+relay 扇出缓解。
- **关键数据** (256 GPU, 106B-671B MoE):
  - 平均 **94.3%** 力平衡理想吞吐
  - 训练和服务平均 **1.49×** vs 无均衡
  - 最终跨 rank 不均衡从 1.30-4.01 降至 **1.01-1.04**
- **意义**: 首次在机架级节点上实现每微批次/层级的精确实时重均衡，收敛到 1.01 的近乎完美均衡

## 三、PD 分离部署

### ⑩ MOCAP — 面向晶圆级芯片的 Prefill-Only 内存编排流水线
- **论文**: [MOCAP: Wafer-Scale-Chip-Oriented Memory-Orchestrated Chunked Pipelining Framework for Prefill-Only LLM Inference](https://arxiv.org/abs/2606.22968)
- **日期**: 2026-06-22
- **核心思路**: 长上下文 prefill 的因果依赖导致 KV 跨流水线阶段不均积累——>内存不均衡（限制序列长度）和 chunk 级延迟不均衡。MOCAP 设计 MBKR（KV 重分配）+ LBCP（均衡 chunk 分割）。
- **关键数据**:
  - 端到端延迟降低 **76.4%** vs GPipe
  - 吞吐 **3.24×**
  - 最大支持序列长度扩展 **1.31×** vs Terapipe
- **意义**: 首次为晶圆级芯片优化 prefill-only 流水线，揭示了 chunked prefill 中因果依赖导致的双重不均衡

### ⑪ SpectrumKV — PD 解聚下逐 token 混合精度 KV 传输
- **论文**: [SpectrumKV: Per-Token Mixed-Precision KV Cache Transfer for Prefill-Decode Disaggregated LLM Serving](https://arxiv.org/abs/2606.08635)
- **日期**: 2026-06-07
- **核心思路**: 已有 PD 端 KV 减少是二元的（传或不传）。SpectrumKV 给每个 token 分配精度等级：attention sink + 高重要 token FP16，中等 INT8，低重要 INT4（模型可容忍时）。含部署时自动探测 INT4 容忍度。
- **关键数据**: 50% KV 预算下，WikiText-2 PPL 变化 +1.97%/-0.06%/-0.44% vs PDTrim 的 +25.85%/+22.07%/+35.63%；TTFT 降低 **50-62%**
- **意义**: 将 PD KV 传输重新定义为精度分配问题而非 token 剪枝问题

### ⑫ FlexNPU — 透明 NPU 虚拟化实现动态 PD 共位
- **论文**: [FlexNPU: Transparent NPU Virtualization for Dynamic LLM Prefill-Decode Co-location](https://arxiv.org/abs/2606.04415v2)
- **日期**: 2026-06-03 | **机构**: 华为
- **核心思路**: 在 Ascend NPU 上实现用户态透明虚拟化层，拦截 AscendCL API 实现动态 PD 共位调度——利用 prefill 计算密集和 decode 访存密集的互补资源特征。
- **关键数据**:
  - 384 卡 Ascend 910C 部署 DeepSeek-R1: 吞吐提升 **5.15%** 和 **26.33%** vs 静态 PD 解聚
  - Qwen2.5-7B: TTFT 降低 **>92%**，TPOT 几乎不变
- **意义**: 证明透明 NPU 虚拟化层可以零开销实现动态 PD 共位，且静态 PD 解聚并不总是最优

## 四、分布式训练与通信调度

### ⑬ HelixPipe — 长序列 Transformer 的注意力并行流水线
- **论文**: [HelixPipe: Efficient Distributed Training of Long Sequence Transformers with Attention Parallel Pipeline Parallelism](https://arxiv.org/abs/2507.00394)
- **日期**: 2026-07-01 (已接受)
- **核心思路**: 不同微批次的注意力计算可以跨流水线阶段并行调度，减少气泡。双层 FIFO 微批次调度平衡内存和通信重叠。
- **关键数据**: 7B/128K/64 H20: **26%** 加速 vs 基线方法
- **意义**: 打破现有 PP 在长序列下的效率瓶颈，序列越长加速比越大

### ⑭ TrainVerify — 分布式 LLM 训练的等价性验证
- **论文**: [TrainVerify: Equivalence-Based Verification for Distributed LLM Training](https://arxiv.org/abs/2506.15961v2)
- **日期**: 2026-06-24 (更新)
- **核心思路**: 给定模型逻辑规约（ground truth），TrainVerify 形式化验证分布式并行执行计划是否数学等价。引入形状缩减技术和分阶段并行验证算法。
- **关键数据**: 成功验证 Llama3 (405B) 和 DeepSeek-V3 (671B) 的训练计划
- **意义**: 首次在千亿参数 LLM 上实现分布式训练的形式化等价验证，可避免数百万 GPU 小时的静默错误损失

## 五、Checkpoint 容错与故障恢复

### ⑮ Concordia — JIT 编译持久核 Checkpoint 实现 LLM 推理容错
- **论文**: [Concordia: JIT-Compiled Persistent-Kernel Checkpointing for Fault-Tolerant LLM Inference](https://arxiv.org/abs/2606.23521)
- **日期**: 2026-06-22
- **核心思路**: 长运行 LLM Agent（KV 缓存、请求调度器、在线适配器等）的 GPU 状态丢失可丢弃数分钟到数小时的工作。Concordia 使用 GPU 驻留持久核作为容错基板，对 PTX/SASS 级插桩，JIT 编译 delta-checkpoint handler（KV 块扫描器、适配器页扫描器、恢复应用器）。
- **关键设计**:
  - 持久核消费 lock-free ring buffer 的计算/checkpoint/append-log/恢复任务
  - CXL 内存或 host DRAM 做 CPU 可见日志
  - 无需 host CPU 介入关键路径
- **意义**: 首次将 persistent kernel 范式引入 LLM 推理容错，GPU 自主管理 checkpoint 生命周期

### ⑯ LUMEN — 分布式 LLM Serving 的负载感知协调恢复
- **论文**: [LUMEN: Coordinated Failure Recovery for Distributed LLM Serving](https://arxiv.org/abs/2606.17787)
- **日期**: 2026-06-16 | **机构**: CUHK
- **核心思路**: 现有容错要么从零重试（丢弃 KV 缓存），要么固定邻居恢复（恢复中 GPU 闲置）。LUMEN 将恢复视为三决策点的负载感知协调问题：失败前的 checkpoint 放置 + 失败时的中断请求分配 + 模型重新加载期间的容量恢复。
- **意义**: 首次将失败恢复建模为动态协调问题而非固定策略

### ⑰ Spotlight — 利用 spot GPU 进行 DiT RL 后训练
- **论文**: [Spotlight: Synergizing Seed Exploration and Spot GPUs for DiT RL Post-Training](https://arxiv.org/abs/2606.19004)
- **日期**: 2026-06-17 | **机构**: HKUST
- **核心思路**: DiT RL 后训练成本极高，spot GPU 便宜 69-77% 但难以利用。两个洞察：(1) 探索可用上一轮过时权重（相对排序不变）; (2) SP 重配置可重用节点状态。
- **关键数据**: 达到相同验证分数 **4×** 更快，总成本降低 **1.4-6.4×**
- **意义**: 首次为 DiT RL 后训练设计 spot GPU 利用方案，弹性序列并行和 pull-based 调度器为关键实现

## 六、量化与推理优化

### ⑱ Quantization Inflates Reasoning — 量化增加推理链长度的隐藏成本
- **论文**: [Quantization Inflates Reasoning: Token Inflation as a Hidden Cost of Low-Bit Reasoning Models](https://arxiv.org/abs/2606.25519)
- **日期**: 2026-06-24 | **机构**: Microsoft
- **核心思路**: INT4/INT3 量化可能保持准确率，但会意外地增加推理链长度。提出了 CoT Token Inflation Ratio 指标。
- **关键数据**: 量化增加推理 token 使用量，抵消预期的每 token 加速；INT4/INT3 下准确率不变但推理链更长、中间步骤和语义重复更多
- **意义**: 指出量化推理模型评估不能只看准确率和每 token 延迟，必须报告 CoT token 膨胀率

## 七、本周趋势总结

1. **CXL 生态成型**：SAC (2.1×吞吐/9.7× TTFT) 和 Concordia (CXL 日志) 显示 CXL 从实验走向稀疏注意力 KV 解聚和 GPU 容错的双重基础设施角色
2. **GQA 头分化利用**：CompressKV 首次将 GQA 头功能分化设计到 KV 缓存压缩中，3% 缓存保 97% 性能
3. **KV 量化精细化**：Block-GTQ (RoPE感知位宽) + SpectrumKV (逐token精度分配) + UltraQuant (4-bit Agent 场景) 三条路线共同将 KV 量化从"统一位宽"推向"结构感知"
4. **MoE 全异步化**：ASAP 实证拆除 DP↔EP 全局屏障，UltraEP 实现每微批次精确重均衡 (1.01)，CrossPool 冷模型双池分离
5. **PD 部署新维度**：MOCAP 揭示 chunked prefill 双重不均衡，SpectrumKV 定义精度分配问题，FlexNPU 证明动态共位优于静态解聚
6. **容错范式转型**：Concordia (persistent kernel) + LUMEN (负载感知协调) + Spotlight (spot GPU 弹性) 共同将容错从"事后回滚"转向"始终在线"
7. **分布式训练验证可操作化**：TrainVerify 首次在 405B/671B 模型上实现形式化等价验证
8. **量化有隐藏成本**：Quantization Inflates Reasoning 揭示 INT4 下 CoT token 膨胀——量化评估多维化
