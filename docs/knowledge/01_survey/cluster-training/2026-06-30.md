## 1. 🏗️ 大规模分布式训练并行策略

### 1.1 HSAP: 分级序列感知并行——混合上下文 Packed 训练（重点 ⭐）

- **作者**: Songxin Zhang, Zejian Xie, Zhuoyang Song 等
- **来源**: arXiv:2606.30460 (cs.LG, cs.DC), 提交日期 2026-06-29
- **核心问题**: 序列打包(packing)是提高 LLM 预训练/微调训练吞吐的实用技术。但在序列并行(SP)框架下，packing 后同一序列段内包含多个独立上下文——直接计算因果注意力会造成**跨上下文污染**(cross-contamination)。现有 SP 方案要么忽略 hybrid-context 场景，要么以牺牲并行度为代价支持该场景。
- **核心创新**: **HSAP** 提出分级序列感知并行框架：
  1. **序列感知并行算法**(Sequence-Aware Parallelism): 解决多设备组间的密集张量传输和部分注意力计算的障碍
  2. **JIT 编译 NCCL 级通信优化**: 所有设备组的通信策略通过 JIT 编译优化，消除不必要的跨组通信
  3. **分级框架**: 集成现有序列并行范式的优点，通过序列感知算法正确处理 hybrid-context causal attention
- **关键量化**: 在多个指标上**超越现有序列并行方案**；支持 packing 训练中的混合上下文因果注意力正确计算
- **意义**: 序列 packing 结合 SP 是降低训练通信开销的核心手段之一。HSAP 首次系统性解决 packing + SP 下因果自注意力的正确性+通信效率两难问题，对**大规模高效预训练**有直接推动

## 2. 📡 PD 分离部署

### 2.1 工业级设计空间: 异构 PD 推论的边界决策（重点 ⭐⭐）

- **作者**: Zhixin Wang, Zhengbo Wang, Fangcheng Fu 等（阿里巴巴/业界）
- **来源**: arXiv:2606.29708 (cs.DC), 提交日期 2026-06-29
- **核心问题**: 异构 PD(prefill-decode)推断已投入生产——prefill 跑在成本优先的加速器上，decode 跑在带宽优先的加速器上，KV 状态跨混合互联穿越混合数值精度。但每个部署各做各的决策，缺一个跨配置的**统一设计空间分析**。
- **核心分析**: 沿**四轴**组织设计空间：(1) 加速器类型, (2) 精度政策, (3) 互联, (4) KV 驻留度——以及它们响应的**工作负载阶段压力**(stage pressure)。
  - 只有一部分交互会成为**绑定约束**——通过三个重复出现的**边界决策**呈现：
    - **计算放置**(compute placement): prefill 放哪里、decode 放哪里
    - **KV 表示**(KV representation): 两端精度格式是否兼容
    - **KV 所有权**(KV ownership): 生命周期——预约、释放、故障恢复
- **关键指南**:
  1. **精准政策属于运行时角色**：同一低比特格式在 prefill 端缓解计算瓶颈，在 decode 端缓解带宽瓶颈——不能全局统一设
  2. **KV 传输引擎搬运字节而非张量语义**：生产者和消费者不同时，表示兼容性是显式边界问题
  3. **KV 交接带生命周期**——预约/释放/故障恢复跨 prefill 和 decode，需显式所有权管理
- **意义**: **第一部工业级 PD 分离设计指南**，来自有真实生产部署经验的团队。如果之前 PD 分离多是单点技术改进，这里是第一次系统地回答"当你真正跨厂牌、跨互联、跨精度做 PD 分离时，哪些必须一起决策、哪些可以独立选择"

### 2.2 HMA-Serve: HBM 不是唯一——跨厂牌 MemHA PD 分离（重点 ⭐🔥）

- **作者**: Zhixiang Wei, Yun Wang, James Yen, Mingyuan Xia, Zhengwei Qi
- **来源**: arXiv:2606.29986 (cs.AR, cs.DC), 提交日期 2026-06-29
- **核心观察**: 数据中心 GPU 依赖昂贵的 HBM，其带宽在 prefill 阶段几乎完全闲置。**Memory-heterogeneous accelerators (MemHA)** 思路——GDDR 卡做 prefill、HBM 卡做 decode，有潜力大幅降低成本。但推到最经济的形态下，MemHA 是**天生跨厂牌**的——最优的 prefill 卡和 decode 卡可能来自不同厂商。
  - 这打破单厂牌分离的两个假设：(1) KV 格式两端原生兼容, (2) 共享软件栈。
- **核心创新**: **HMA-Serve** 系统：
  1. **阶段感知量化**(phase-wise quantization): prefill 端用厂商标配低精度做高吞吐，decode 端保持 BF16 高精度
  2. **计算-传输流水线**(compute-transfer pipeline): 每层的 KV Cache 传输与后续层 prefill 重叠，降低 TTFT
  3. **延迟反量化**(deferred dequantization): 传输原始量化 byte，在 decode GPU 端延迟重建——减少网络带宽和 HBM 使用
- **关键量化**: Qwen3 4B-32B × 3 条生产 trace: **3.2×** goodput (vs SOTA 内存同构方案), **4.8×** goodput-per-dollar；无损生成质量指标
- **意义**: 首次在大规模系统中证明跨厂牌 PD 分离不仅可行而且经济上更优。"HBM is not all you need"——这个标题本身就是对"GPU 只能上 HBM"行业共识的挑战，对 AI 基础设施 TCO 有直接影响

### 2.3 KernelFlume: Attention-FFN 解耦的弹性核心注意力扩展（重点 ⭐🔥）

- **作者**: Guangyu Xiang, Xueze Kang, Lin Zhang 等
- **来源**: arXiv:2606.29207 (cs.DC), 提交日期 2026-06-28
- **核心观察**: LLM 推理 workload 正被 Agent/推理模型/长对话产生的**长上下文动态解码负载**主导。突发长上下文需求超过部署容量时，现有系统只能启动完整模型副本——内存开销大、启动延迟高、batch fragmentation 严重。
- **核心创新**: **KernelFlume**——解码中心架构，将稳定的 projection/FFN 路径与核心注意力计算解耦：
  - **权重节点**(weight nodes): 执行密集 projection/FFN kernel，是**有状态有重量**的
  - **无权重注意力节点**(weightless attention nodes): 存储 token-range KV 分片，纯粹按请求状态需求**弹性伸缩**
  - **路由表**: token 范围到 attention 节点的端点映射，在 token 边界更新路由
  - **查询优先核心注意力分发**(query-first core-attention dispatch) + 层间 kernel 流水线——远程注意力和通信与本地 projection/FFN 重叠
- **关键量化**: Llama-3.1-8B, 动态长上下文 Agentic workload:
  - Intra-node A6000: p99 TPOT ~74ms; Cross-node H100: p99 TPOT ~34ms
  - 成本降低 **32-61%** vs ServerlessLLM (full-instance 弹性扩展)
  - 模拟更大模型规模: **56-66%** 成本降低，若用廉价异构 attention 节点可扩大到 **80-85%**
  - 百万 token 上下文范围持续有效
- **意义**: 如果说之前的 PD 分离是在模型级(整个 prefill vs 整个 decode)，KernelFlume 在 **kernel 级**分离了"计算"与"注意力"——attention 节点可以像无状态服务一样弹性扩缩。这是对注意力弹性扩展的一次方法级突破

## 3. 💾 KV Cache 优化

### 3.1 HARD-KV: 头自适应正则化——桥接动态驱逐与静态 CUDA Graph（重点 ⭐）

- **作者**: Yuxuan Yang, Feiyang Ren, Bowen Zeng 等
- **来源**: arXiv:2606.28831 (cs.LG), ICML 2026 Poster, 提交日期 2026-06-27
- **核心问题**: 长上下文 LLM 推理面临一个**根本矛盾**：头部自适应压缩算法（如 Top-p nucleus 采样)通过动态波动的内存预算获得更高精度，但现代推理引擎(如 vLLM)依赖 CUDA Graph 和 PagedAttention，要求**刚性静态内存模式**。这个"Static-Dynamic 不匹配"让精度高的压缩方案无法实际落地。
- **核心创新**: **HARD-KV** 框架：
  1. **Cascade Cache 层级架构**: 管理 token 在 dense/sparse/condensed 三层间的生命周期
  2. **Logits Calibration 机制**: 将不同头部各自的 importance 指标归一化到统一概率空间——使 Top-p 预算在各异构头部间一致分配
  3. **系统级重构**: 将碎片化动态索引重写为连续物理布局，兼容高性能推理引擎
- **关键量化**: AIME/U-Math 数学推理 benchmark: **2×** 吞吐提升 vs 静态基线，10k+ token 场景保持高保真生成
- **意义**: 很多系统(KV-Ring/Top-k/Heavy-Hitter/ScissorHands)想用动态驱逐但被 CUDA Graph 拦住——HARD-KV 是第一个**系统级解法**，在保持 CUDA Graph 加速的同时享受动态压缩的精度收益

## 4. 🔄 通信调度与推理优化

### 4.1 CAEE: MoE 成本感知专家执行（重点 ⭐）

- **作者**: Hui Zang, Pengfei Xia, Hong Liu 等
- **来源**: arXiv:2606.29982 (cs.DC), 提交日期 2026-06-29
- **核心观察**: MoE 推理的两个耦合挑战：
  - **重要性无关成本**: 低贡献度专家与高贡献度专家占用几乎相同的内存和传输成本——严重浪费带宽
  - **系统级不平衡**: 多设备部署的端到端延迟被最慢设备卡住——单设备的局部减负不转化为整体改善
- **核心创新**: **成本感知专家执行(CAEE)**——硬件引导的运行时框架：
  1. 轻量校准成本模型预估硬件开销
  2. 选择性剪枝低重要性高成本专家
  3. 通过低开销补偿机制重新分配剪枝专家的贡献——避免额外数据搬运
- **关键量化**: DeepSeek-R1 671B: 端到端推理延迟降低 **8-18%**（含专家卸载/设备上执行两种部署），精度损失 <1%
- **意义**: 打破"所有专家成本相同"的隐含假设——基于实际硬件开销做 token 级专家选择性跳过。对 MoE 推理的通信优化有直接参考价值

### 4.2 KernelSight-LM: 跨代 GPU 推理模拟器（重点 ⭐）

- **作者**: Xiteng Yao, Taeho Kim, Hengzhi Pei 等
- **来源**: arXiv:2606.28565 (cs.PF), 提交日期 2026-06-26
- **核心创新**: 细粒度推理模拟器，建模逐 token 执行，产出内核级延迟分解：
  - **Roofline kernel 模型 + 学习效率项 + 通信模型 + 主机开销模型** → 离散事件调度器
  - 支持 prefix caching 和 continuous batching
  - **两个预测层级**:
    - 跨代层级: **无靶卡数据**，仅用硬件规格+已测 GPU 的 kernel 微基准——**12.1%** per-kernel error (1.8× 优于 roofline 22.0%)
    - 有测量层级: 一次模型无关的 kernel 微基准扫描——**3.8%** per-kernel error (7.3× 优于对照基线 27.7%)
- **关键量化**: 6 模型家族端到端: 跨代层级 p50 error 15.4%(TTFT)/12.8%(TPOT)/3.0%(吞吐)；有测量层级 14.3%/6.2%/2.7%
- **意义**: 在 GPU 迭代加速的背景下，"在新一代 GPU 上我的推理性能会怎样"是高频问题。KernelSight-LM 证明了**无需在目标 GPU 上跑一次完整推理就能以 ~12% error 预测性能**——对集群规划和容量设计有实用价值

### 4.3 TraceLab: 编码 Agent 负载特征——Agent Serving 优化指南（重点 ⭐）

- **作者**: Kan Zhu, Mathew Jacob, Chenxi Ma 等
- **来源**: arXiv:2606.30560 (cs.LG, cs.PF, cs.AI), 提交日期 2026-06-29
- **核心贡献**: 发布~**4300 个编码 Agent 会话**的生产轨迹——**~35 万 LLM 步骤**、**43 万工具调用**，来自日常 Claude Code + Codex 使用
- **关键发现**:
  1. **长自主循环**: Agent 的任务执行往往不需要人类介入的长时间连续推理循环
  2. **长上下文 + 短输出**: 上下文很长但每步输出很短——prefill 长、decode 短的极端不均衡
  3. **多样化且长尾的工具调用**: 工具调用模式和分布高度异构
  4. **高但不完美的 prefix cache 命中**: 部分上下文可复用但并非完美命中
- **对 serving 优化的启示**:
  - 低开销工具调用支持
  - append-length-aware prefill（感知"仅追加少量内容"来优化 prefill）
  - 语义感知工具延迟预测
  - 人类操作间隙的 KV-Cache 管理优化
- **意义**: 编码 Agent 是 Agentic LLM 的主要使用形态。这是**首个公开的编码 Agent 生产轨迹**，揭示了与常规聊天推理截然不同的负载模式——集群 serving 系统需要为 Agent 重新优化
