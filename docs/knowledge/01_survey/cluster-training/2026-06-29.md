## 1. 🏗️ 大规模分布式训练并行策略

### 1.1 RolloutPipe: 解耦 RLVR 训练管线（重点 ⭐🔥）

- **作者**: Rongjian Chen, Jianmin Hu, Kejiang Ye, Minxian Xu
- **来源**: arXiv:2606.26997 (cs.DC, cs.LG), ICCC 2026
- **提交日期**: 2026-06-25
- **核心问题**: 现代 RLVR(RL with Verifiable Rewards) 采用 PD 分离架构——rollout GPU 池和 policy 训练 GPU 池独立部署。但同步 on-policy GRPO 系统必须等待 **整个 rollout 完成**后才开始训练，导致 trainer GPU 在 rollout 期间完全空闲。异步方案虽可重叠，但需接受数据过期(off-policy)的代价。
- **核心创新**: **RolloutPipe** 提出**完组流水线(Complete-Group Pipelining, CGP)** + **前沿组调度(Frontier-Group Dispatch, FGD)** 两项技术，将固定权重 rollout 转化为完整组流水线——只要任意完整组的生成完成，立即 dispatch 到 trainer FIFO 开始训练，同时保持 on-policy 正确性。
  - **CGP**: 每组(complete group)由同一固定权重的所有 rollout 轨迹组成；某组一旦素材就绪，立即 dispatch 到 trainer——不等待所有组完成
  - **FGD**: Rollout 节点上的准入策略——优先处理构成下一训练 batch 所必需的"前沿组"请求，使 trainer 就绪组更早更稳定到达
- **关键量化**: Qwen3-1.7B，4 个推理/科学基准，12 种 rollout 配置下：rollout-to-train-end 时间缩短 **30.7%-42.3%**，trainer 等待比降低 **37%-76%** vs SOTA baseline Slime
- **意义**: 首次在**不引入数据过期代价**的前提下实现 rollout 与训练的流水线重叠，将 RLVR 的可扩展性从训练/推理资源绑定中解放出来——大模型 RL 后训练的 rollout-train 瓶颈开始被系统性解决

### 1.2 Moebius: MoE 运行时 TP↔EP 在线切换（重点 ⭐🔥）

- **作者**: Shaoyu Wang, Yizhuo Liang, Jaeyong Song, Chong Li, Seo Jin Park
- **来源**: arXiv:2606.26607 (cs.DC)
- **提交日期**: 2026-06-25
- **核心观察**: MoE 模型服务中，TP 在低并发下更快，EP 在高并发下更优。生产负载不断跨越这个边界：在线服务有突发/安静的交替，RL rollout 以高并发突发开始然后衰减为长尾 straggler。固定任一并行策略都会在另一端丧失性能。
- **核心创新**: Moebius 首次实现**运行时 TP 与 EP 之间无缝切换**——不重启引擎、不丢弃 in-flight 请求。核心洞察：EP 和 TP 是一个模型的两种布局而非两个模型，它们在字节相同的 expert 权重和 KV Cache 上计算相同函数。切换仅改变各 rank 拥有哪些切片的归属，而现代 GPU 高带宽互联使这个迁移足够快到在两 decode 步之间完成。
  - 每种并行运行时保持驻留，以固定地址重分片单份 expert 权重和 KV Cache
  - 使用融合 GPU-to-GPU 传输内核加速
- **关键量化**: 8×H200 + Qwen3-235B-A22B：每切换 **215-434 ms**；RL rollout 场景 **1.16-1.25×** 优于最优静态并行；双布局驻留仅 **2.4%** 内存开销
- **意义**: 打破"模型部署必须选一种并行策略"的范式——首次在实践上证明了 TP 和 EP 可以动态切换，MoE 服务可以从"either-or"走向"both-and"

### 1.3 CommFuse: 消除通信重叠中的尾延迟

- **作者**: Rezaul Karim et al.
- **来源**: arXiv:2604.24013 (cs.LG, cs.DC)
- **核心创新**: 现有通信-计算重叠方案基于数据切片，存在尾延迟问题。CommFuse 将 reduce-scatter 和 all-gather 等集体操作**分解为 P2P 通信**，并调度分块计算以实现细粒度重叠。提供消除尾延迟的精确算法。
- **关键量化**: 兼容 DP/TP(含 TPSP/UP)；持续实现更低延迟、更高 MFU 和吞吐
- **意义**: 通信-计算重叠的一个瓶颈是否决——数据切片方案的尾延迟根源在于集体操作的同步屏障，CommFuse 用 P2P 分解从根本上消除它

### 1.4 Federation of Experts (FoE): KV-Head 粒度 MoE 分布式推理

- **作者**: Muhammad Shahir Abdurrahman, Chun Deng, Azalia Mirhoseini, Philip Levis
- **来源**: arXiv:2605.06206 (cs.LG)
- **核心创新**: 将 MoE block 重构为**多个 MoE 簇(cluster)**，每簇仅负责一个 KV-Head，簇内应用 expert 并行。跨簇的 sum 同步连接后注意力残差。单节点场景消除 all-to-all 通信；多节点场景将 all-to-all 限制到**节点内互联**。
- **关键量化**: LongBench：端到端前向延迟 **5.2×**，TTFT **3.62×**，TBT **1.95×** 加速
- **意义**: 不像通行的 EP/TP 之争，FoE 从**注意力机制层面**重新设计 MoE 的分布式推理布局——KV-Head 天然的可并行性被利用为通信优化约束

## 2. 🚀 Speculative Decoding 与推理加速

### 2.1 Speculation at a Distance: 边缘-云端 SD 的理论边界

- **作者**: Yuan Lyu, Bharath Irukulapati, Jaya Prakash Champati
- **来源**: arXiv:2606.25091 (cs.DC)
- **提交日期**: 2026-06-23
- **核心贡献**: 以闭式不等式推导边缘-云端分布式 SD(DSD)的收益边界，主要发现：
  1. 若云可共置两个模型，共置 SD 的延迟和通信始终优于同步 DSD
  2. 流水线化仅在 **低 RTT** 场景使 DSD 具有竞争力（RTT < 边缘 draft 时间窗）
  3. WAN RTT 下，云往返太大使流水线 DSD 无法击败共置 SD
  4. 对闭源 API（无 verifier-only 接口），DSD 不可行
  5. **DSD 的主要价值在于多租户容量**——卸载 draft 计算使饱和云服务器可支持 **(1+γ·t_d/t_v)** 倍并发客户端
- **意义**: 提供了 DSD 领域的首个系统性理论分析，明确了"边缘-云端加速推理"的适用边界——**延迟不是理由，并发才是**，这扭转了行业对 DSD 的价值认知

## 3. 🗜️ MoE 压缩与高效服务

### 3.1 FlexMoE: 嵌套内-专家通道剪枝（重点 ⭐）

- **作者**: Fan Mo, Yuxuan Han, Geng Zhang, Wangbo Zhao, Yang You
- **来源**: arXiv:2606.27866 (cs.LG)
- **提交日期**: 2026-06-26
- **核心创新**: 将预训练 MoE LLM 转换为**跨预算的嵌套式子网络族**：
  1. 对专家 FFN 通道按重要性排序
  2. 让每个专家学习离散动作来剪裁通道
  3. 逐步递增成本压力 → 单一 action 训练运行导出从高到低预算的动作掩码序列
  4. 在中间预算(40%)进行单次恢复微调，使恢复的模型转移到其他未见过的预算
- **关键量化**: Qwen2-57B-A14B：无需微调剪裁 50% routed expert 参数即保留 **~99.8%** 基座性能；支持在线预算切换 + kernel 级协同设计
- **意义**: 不同于以往的固定预算剪枝，FlexMoE 的**一次训练导出所有可部署子网络**——为 MoE 的"设备-用户-工作负载"三位一体差异化部署提供了系统化方法

### 3.2 CrossPool: 冷 MoE 的 KV-Cache+Weight 分离池化（重点 ⭐🔥）

- **作者**: Zhuoren Ye et al.
- **来源**: arXiv:2606.24506 (cs.DC, cs.AI, cs.LG, cs.PF)
- **提交日期**: 2026-06-23 (v2: 06-26)
- **核心洞察**: 新兴 LLM 服务托管大量稀疏 MoE 模型，大多数模型访问稀疏而处于"冷"状态。GPU 内存冲突在于：**权重是"稳定且 model-determined"**，**KV-Cache 是"瞬态且 demand-determined"**。冷模型的 KV-Cache 峰值不会同时到来，但权重与 KV-Cache 混在单 GPU 内存池中——静态权重与动态 KV-Cache 竞争，而冷低并发下 KV-head-limited attention 只暴露一小部分复制 KV 容量。
- **核心架构**: CrossPool 将 FFN 权重和 KV-Cache**分离为两个 GPU 内存池**：
  - **权重池**: 跨冷模型合并 FFN 权重
  - **KV-Cache 池**: 动态服务活跃请求，attention 保持 local 到 KV-Cache
  - KV-Cache 规划器 + 虚拟化器 + 层间流水线调度器(隐藏 hidden-state 传输) + 持久化内核(降低 CPU-GPU 控制开销)
- **关键量化**: 突发长上下文请求下 P99 TBT 降低 **10.4×** vs SOTA 多模型 KV-Cache 共享系统
- **意义**: 首次提出针对"冷 MoE 模型群"的系统级优化——权重和 KV-Cache 的解耦不再是微架构层面的想法，而是系统池化的可行方案

## 4. 🎯 级联成本感知 LLM 服务

### 4.1 Cluster, Route, Escalate: 两级级联

- **作者**: Yasmin Moslem et al.
- **来源**: arXiv:2606.27457 (cs.PF, cs.CL)
- **提交日期**: 2026-06-25
- **核心方法**: 两级级联：
  1. **第一级**: 聚类输入查询，为每个簇分配成本效益最高的模型（可调超参数离线 tuning 成本预算）
  2. **第二级**: 质量估计(QE)级联——当 Stage 1 输出被判断为低质量时，升级到更强模型
- **关键量化**: 保留最强模型 **97-99%** 精度，同时降低 TPOT；仅需任务正确性标签，适应模型池变化无需手动重配置
- **意义**: 生产环境 LLM 服务的成本-精度 trade-off 工程实践——"用便宜模型处理简单查询，把贵模型留给难查询"的系统化实现

**核心信号**: 本周各条线均指向**系统级弹性化**——训练管线解耦(RolloutPipe)、推理并行切换(Moebius)、冷模型内存池化(CrossPool)，都在解决同一问题：**大规模 GPU 集群的静态资源分配无法匹配动态负载波动**，而下一阶段的竞争焦点正在从"更高的每 GPU 利用率"转向"更灵活的跨 GPU/跨时间资源重组"。
