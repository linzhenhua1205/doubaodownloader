## Ⅰ. KV Cache 优化

### ① SAC: CXL 碎片化 KV Cache 系统 for 稀疏注意力模型 (arXiv:2606.19746, 6/18)
- **问题**: 传统 RDMA-based 分离内存池对稀疏注意力模型低效——每次解码需全量拉取整个 prefix KV cache
- **方案**: 利用 CXL 的 cache-line 粒度 load/store 语义，仅按需拉取 top-k KV entries
- **实现**: DeepSeek-V3.2 on SGLang
- **结果**: vs RDMA baseline → **2.1× 吞吐**, **9.7× TTFT 降低**, **1.8× TBT 降低**
- **意义**: 确立了 CXL-based 碎片化作为稀疏注意力模型推理的基础设施路径

### ② UltraQuant: 4-bit KV Cache for 上下文密集型 Agent (arXiv:2606.20474, 6/18)
- **问题**: 长上下文 Agent workload 中 KV cache 压力在后期 rounds 急剧恶化
- **方案**: 4-bit KV cache 压缩（TurboQuant 旋转 + codebook 量化），Walsh-Hadamard 旋转 + 非对称 K/V 处理 + block-scale 变体
- **AMD GPU 优化**: UltraQuant FP4 近似路径（FP8 query + FP4 KV tensor + UE8M0 group scale + CDNA4 native scaled-MFMA）
- **结果**: P50 TTFT 在 cache 压力后期 rounds 降低 **3.47×**（全 rounds 平均 2.3×），输出吞吐提升 **1.63×** vs FP8 KV baseline

### ③ Execution-State Capsules: 图绑定执行状态检查点/恢复 (arXiv:2606.20537, 6/18)
- **创新**: 从 token 寻址 KV fragment 转向 graph-bound 完整执行状态复用——不仅 KV cache，还包括 recurrent state、convolution state、MTP state、metadata
- **FlashRT 运行时**: CUDA backend, contiguous static buffers, 无 block-table indirection
- **结果**: RTX 5090 上 sub-millisecond snapshot/restore, TTFT speedup 从 2K tokens 的 **3.9×** 到 16K tokens 的 **27×**
- **定位**: 不是替代高吞吐 KV cache serving，而是定义了一个互补的 latency-first serving point（on-device physical-AI）

## Ⅱ. Speculative Decoding

### ④ EfficientRollout: 系统感知自推测解码 for RL Rollouts (arXiv:2606.18967, 6/17)
- **问题**: RL rollout 生成存在两个 SD 无法直接加速的难点：(i) 目标策略持续演化，固定 drafter 输出分布迅速不匹配；(ii) 活跃 batch size 随解码缩小，从 compute-bound 转向 memory-bound
- **方案**: 
  - **Self-SD**: 从目标模型诱导量化 drafter（无需额外预训练或在线适配）
  - **系统感知 SD 切换策略**: 仅在有益 regime 进行推测
  - **接受率感知 draft-length 自适应**: 匹配演化中的 drafter 质量
- **结果**: Rollout 延迟降低 **19.6%**，端到端延迟降低 **12.7%**
- **见代码**: github.com/furiosa-ai/EfficientRollout

### ⑤ SafeSpec: 安全感知推测推理 (arXiv:2606.19755, 6/17)
- **问题**: 推测推理加速但无安全保证，现有安全防御与 SD 不兼容
- **方案**: 在验证过程中集成轻量 latent safety head，同时评估语义有效性和安全性；发现不安全时执行 rollback + 安全引导反射多采样
- **结果**: Qwen3-32B 上攻击成功率降低 15%，同时保持 **2.06× 推理加速**（良性 workload）

## Ⅲ. 模型并行与动态重配置

### ⑥ ReMP: 运行时模型并行性重配置框架 (arXiv:2606.18741, 6/17)
- **问题**: 现有系统将 TP/PP 拓扑视为静态配置，服务重启导致数分钟中断 + KV cache 丢失 + 高昂重算开销
- **三大关键技术**:
  1. 模型并行拓扑与运行时状态解耦（避免全服务重建）
  2. **二维 KV cache 迁移机制**（TP/PP 变更后保留可复用 cache state）
  3. 端到端在线重配置
- **结果**: 7B-70B 模型上大部分拓扑切换在 **1-7 秒**内完成，较重启方法加速数十至上百倍
- **意义**: 首次实现 LLM serving 时 TP/PP 拓扑的动态调整

## Ⅳ. 分布式训练

### ⑦ ARGUS: 10,000+ GPU 集群生产级 Tracing 与性能诊断 (arXiv:2606.20374, 6/18)
- **需求**: 大规模 LLM 训练需要 always-on、细粒度可观测性
- **方案**: 沿训练调用层次分解为 CPU call stacks + 框架语义 + GPU kernel execution，always-on 采集总开销 **< 2%**
- **压缩**: 原始 kernel events 压缩约 **3,700×**（10 MB → 2.7 KB per rank per step）
- **渐进式诊断框架**: 自动隔离异常窗口 → straggler rank → 退化 kernel（迭代时间/phase-level/kernel-level 三级分析）
- **生产部署**: 10,000+ GPU 生产集群运行 6+ 个月，已支持持续 fail-slow 检测和性能优化
- **案例**: compute straggler、link degradation、pipeline-bubble amplification、FlashAttention JIT stalls

### ⑧ Online Dynamic Batching (ODB) (arXiv:2606.19989, 6/18)
- **问题**: LLM 训练的真实样本成本在预处理/增强/模板化/分词/多模态视觉 token 扩展后才可观察，传统离线 batch sampler 盲目构造 batch
- **方案**: DataLoader 侧 drop-in 系统，将 batch 构建移到准确可观察点，同时保持 DDP step 对齐
- **形式化**: 定义为 Distributed Group Alignment Problem，证明无死锁有界终止
- **结果**: Qwen3-VL 2B/8B 上，emitted-sample throughput 提升 **1.58-2.51×**（单节点 Full FT/LoRA），**1.71-3.78×**（两节点 Full FT）；生产 MM-Mix 达 **4.43×**
- **与 token-budget oracle 对比**: 15% 以内，高 CV 场景更快（2.24-3.69×）

### ⑨ FoMoE: 打破全副本壁垒的 MoE 联邦训练 (arXiv:2606.19025, 6/17)
- **问题**: DiLoCo/Photon 等跨数据中心方法要求每个节点全模型副本，内存和通信开销高昂
- **方案**: 跨 workers 分片 expert layers（打破全副本范式）
- **三项贡献**:
  1. 通信成本降低 **1.42×** vs 高效 baseline，**45.44×** vs DDP（通过部分 expert 复制）
  2. **Skip-token 机制**实现经验吞吐 **1.4× 加速**
  3. 训练代理 regime 中路由稳定，系统建模将通信/内存优势投影至 100B 级配置

## Ⅴ. Speculative Generation (Agentic Kernel Optimization)

### ⑩ SpecGen: 推测生成加速 Agentic Kernel 优化 (arXiv:2606.17518, 6/16)
- **问题**: Agentic kernel 优化中三种系统级低效：(i) LLM 推理产生长生成延迟；(ii) profiling 反馈不足；(iii) 验证/profiling 资源未充分利用
- **核心洞察**: 正在进行的推理生成暴露出一个窗口——在完成前即可产生额外候选 kernel
- **方案**: 
  - 在推理轨迹的 trigger points fork 非推理生成 → 增加候选 kernel 数
  - 候选 kernel 与推理并行验证/profiling
  - 满足终止条件时提前终止推理生成
  - 动态 reallocate 验证/profiling GPU pools
  - 利用验证/profiling GPU 的 spare memory 作为远程 KV cache 存储
- **结果**: H200 上，端到端时间显著降低，profiling 反馈增多，资源利用率提升

## Ⅵ. 容错恢复

### ⑪ LUMEN: 分布式 LLM Serving 协同故障恢复 (arXiv:2606.17787, 6/16)
- **问题**: 当 worker 故障时，集群同时丢失故障 worker 的 GPU-resident KV cache 和服务容量
- **现有方案不足**: 固定 neighbor worker 检查点恢复，不考虑当前集群负载；恢复期间恢复 worker 空闲
- **解决方案**: 将恢复视为负载感知协同问题的三阶段决策：
  1. **故障前**: checkpoint 放置策略
  2. **故障时**: 中断请求分发
  3. **模型重载期间**: 服务容量恢复
- **评估**: 原型实验 + 大规模模拟，serving 和 recovery 时间显著改善

## Ⅶ. 本周趋势总结

| 方向 | 关键进展 |
|:-----|:---------|
| **KV Cache 分离** | CXL-based 碎片化（SAC 2.1×, 9.7× TTFT）vs 传统 RDMA 全量拉取的范式迁移 |
| **KV Cache 量化** | UltraQuant 4-bit 在 AMD CDNA4 上 3.47× TTFT；完整执行状态封装（27× speedup） |
| **Speculative Decoding** | 扩展到 RL rollout（EfficientRollout, 12.7% 端到端），安全感知 SD（SafeSpec, 2.06× 加速+15% ASR↓） |
| **模型并行动态性** | ReMP 实现 TP/PP 运行时重配置（1-7 秒），打破静态拓扑约束 |
| **大规模训练可观测性** | ARGUS <2% 开销在 10K+ GPU 集群部署 6 个月，3,700× 事件压缩 |
| **训练数据加载** | ODB 动态 batching 提升 1.58-4.43× 吞吐，形式化保证无死锁 |
| **分布式 MoE 训练** | FoMoE 跨数据中心 expert 分片，打破全副本限制，1.4× 加速 |
| **Agentic Kernel 优化** | SpecGen 推测生成提前终止推理窗口，减少端到端时间 |
| **容错恢复** | LUMEN 负载感知三阶段恢复，替代固定 neighbor 检查点 |

**核心趋势**: ① **CXL 正在成为 KV Cache 分离的新基础设施底座**（SAC 是这一趋势的里程碑论文）；② **SD 从纯推理加速扩展到 RL rollout 和安全性**（EfficientRollout / SafeSpec 标志新边界）；③ **训练系统从静态配置走向动态可重配置**（ReMP 打破 TP/PP 拓扑僵化）；④ **大规模集群可观测性进入 always-on 时代**（ARGUS 展示 <2% 开销的精度）。
