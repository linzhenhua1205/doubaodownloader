## 2. 📡 PD 分离部署/分布式推理

### 2.1 Omni-Flow: 超越 PD 分离——跨模态分布式 KV Cache 共享框架（重点 ⭐⭐🔥）

- **作者**: Bin Xiao, Jingfu Dong, Changran Wang, Yitian Chen 等
- **来源**: arXiv:2606.31093 (cs.DC), 提交 2026-06-30（周三新刊）
- **核心问题**: 多模态 LLM 推理面临三大挑战——
  1. **灵活编排**: 异构计算单元(LLM+扩散模型)存在复杂依赖和并发控制
  2. **中间数据传输**: 跨进程/节点的大张量高速传输
  3. **KV Cache 与模型权重共享**: 消除冗余 GPU 内存
  现有方案将 LLM 和扩散模型独立部署，缺乏多模态管线的系统级抽象
- **三层抽象架构**:
  1. **Control Flow 层**: Python DSL 定义工作流，编排异构单元为统一数据流图，支持静态 DAG + 动态路由 + 服务发现 +
     多种负载均衡
  2. **Data Flow 层**: **超越 PD 分离的分布式 KV Cache 抽象**——统一分配，支持跨角色的零拷贝低延迟直接传输；
     三层分页存储层次(GPU/CPU/SSD)
  3. **Compute Flow 层**: 多模态前缀匹配实现跨轮对话 KV 复用；通过统一 SGLang 接口接管 KV Cache 和采样逻辑，
     让扩散模型直接复用 LLM 前向路径（统一并行语义）
- **演示场景**:
  - 全模态对话 (LongCat-Next)
  - 复杂图像生成管线 (HunyuanImage-3)
- **意义**: **首次将分布式 KV Cache 共享从"PD 分离"扩展到"跨模态管线"**。之前工作的核心关注点是 prefill 和 decode 之间的 KV 传递；Omni-Flow 将这一概念推广到 LLM↔扩散模型的 KV+权重复用。如果后续扩散模型在推理过程中的中间表征可以与 LLM 的 KV Cache 统一管理，这对**多模态 AI 推理集群的资源利用率**有深远影响

### 2.2 LASER: 负载感知早退推理服务（泛 PD / Edge 推理）

- **作者**: Zhiqing Tang, Size Li 等
- **来源**: arXiv:2606.31580 (cs.DC), WASA 2026
- **核心**: 推理 LRM(DeepSeek-R1 类)的边缘部署时，早退方法从单请求视角出发使用固定置信度阈值，忽略多请求并发和负载波动
- **创新**: 负载感知自适应早退阈值 + 难度感知推理预算预分配
- **关键数据**: 平均延迟降低 **17-38%**，SLO 满足率提升 3-6%，精度损失仅 ~1%
- **评价**: 针对边缘推理场景，对大集群训推场景直接参考价值有限，但早退与 PD 分离结合的思路值得关注

## 3. 🧠 Speculative Decoding

### 3.1 BlockPilot: 样本自适应扩散 SD 块大小（重点 ⭐⭐）

- **作者**: Hao Zhang, Yiming Hu, Yong Wang 等
- **来源**: arXiv:2606.31315 (cs.CL), 提交 2026-06-30（周三新刊）
- **核心问题**: 扩散式 SD(diffusion-based speculative decoding)通过 block-level diffusion 每前向步生成多 token，达到 SOTA。但现有方法采用**固定推理块大小**，假设所有输入的最优解码策略一致
- **关键发现**: 此假设不成立——最优块大小**跨样本变化显著**，且具有明确的局部结构（集中在训练块大小附近），将问题降为低维结构化决策空间
- **核心技术**: **BlockPilot**——从预填充表征学习轻量策略预测最优块大小
  - 块大小选择形式化为策略学习问题
  - 基于预填充阶段表征预测最优块大小
  - 预填充后执行一次预测，可无缝集成
- **关键数据**:
  - Qwen3-4B 下 acceptance length **5.92**，**4.20×** 加速 (T=1)
  - 即插即用，引入极低开销
- **意义**: **"自适应块大小"是扩散 SD 从静态到动态的关键一步**。之前 JetSpec(06-28 跟踪)和 Hydra 等扩散 SD 方法都假设块大小固定，BlockPilot 证明了差异化策略的价值。与 SMART(本节 3.2)的边际分析形成互补——SMART 在 token 级决定是否扩展树，BlockPilot 在块级自适应选择初始块大小

### 3.2 SMART v2: 系统感知边际分析——何时值得扩展推测树（重点 ⭐）

- **作者**: Lifu Wang, Pan Zhou
- **来源**: arXiv:2604.09731v2 (cs.DC), **已接收 ECCV 2026**（本次为更新版本，新增 ECCV 接收信息，核心内容不变）
- **核心**: **"效率悖论"**——大树的开销增长可能超线性，导致负加速。SMART 将树扩展重新定义为硬件感知优化问题，直接最大化端到端加速比
- **边际收益-成本规则**: 仅当边际收益/成本比 > 树级加速比时才扩展节点
- **关键数据**: MLLM 平均额外加速 **+20.0%**，LLM **+15.4%**，无质量损失
- **评价**: 已在 06-28 跟踪中详细记录。本次 v2 更新确认被 **ECCV 2026** 接收，实用性验证进一步加强。与 EntMTP(3.3) 的拓扑切换思路形成对比——SMART 在树内做边际决策，EntMTP 在不同拓扑间切换

### 3.3 EntMTP: 熵引导 MTP——按局部熵动态切换推测拓扑（重点 ⭐）

- **作者**: Carrie Chen
- **来源**: arXiv:2606.27550 (cs.CL, cs.LG), 提交 2026-06-25（周三新刊）
- **核心问题**: 现有 MTP(Multi-Token Prediction)模型在整个生成序列中使用**固定静态树注意力拓扑**，推测深度和验证计算量恒定不变。但自然语言熵模式不均匀——低熵区域支持可靠多步推测，高熵区域需要保守推测
- **技术创新**: **EntMTP**——训练无关调度器
  - 从任务特定帕累托最优树集合中，根据局部生成熵的运行估计切换注意力拓扑
  - 将推测深度与上下文可预测性匹配
  - 最大化期望接受 token 吞吐量
- **关键数据**:
  - 对 Hydra 一致 **1.15×** 加速
  - 对 Medusa 峰值 **1.36×** 加速
  - 在 Humaneval/ShareGPT/GSM8k/Litbench 上验证
- **意义**: **"熵感知推测"将 SD 从静态树提升到动态策略**。与 SMART 的边际分析(3.2)不同——SMART 决定"树中每个节点是否值得扩展"，EntMTP 决定"不同区域用哪个拓扑更高效"。两者正交互补，可组合使用

## 4. 💾 KV Cache 优化

### 4.1 SeKV: 分辨率自适应语义 KV Cache——按需重建不丢信息（重点 ⭐⭐🔥）

- **作者**: Amirhossein Abaskohi, Giuseppe Carenini, Peter West, Yuhang He
- **来源**: arXiv:2606.31145 (cs.CL), 提交 2026-06-30（周三新刊）
- **核心问题**: 长上下文下 KV Cache 是主要内存瓶颈——线性增长且全程保留。现有压缩方法：
  - Token 驱逐 → **丢失信息**
  - 语义分组 → 预填充时固定压缩决策，生成中一旦压缩跨度变得相关**无法恢复 token 级细节**
- **核心技术: SeKV**:
  1. **熵引导语义跨度**: 将上下文组织为语义一致的跨度
  2. **GPU-CPU 分层存储**:
     - GPU: 每个跨度仅保留轻量摘要向量（粗粒度路由）
     - CPU: 低秩 SVD 基（按需 token 级重建）
  3. **已训练的"放大"机制**: 解码时选择性展开查询相关的跨度，实现精确检索而无需在 GPU 上实例化完整 KV Cache
- **两个核心优势**:
  - **不丢弃信息**：全量数据留存于 CPU SVD，GPU 只保存路由所需摘要
  - **自适应分辨率**：粗→细按需切换
- **关键数据**:
  - 四个基准测试上比最强语义压缩基线平均好 **+5.9%**
  - GPU 内存减少 **53.3%** vs 全量 KV Cache (128K 上下文)
  - 基 LLM 完全冻结，可训参数 **<0.05%**
- **意义**: **"分辨率自适应"是 KV Cache 优化的关键新方向**。之前 HGA(06-29 跟踪)也用了层次路由思路，但 HGA 是 Dropping(丢弃)，SeKV 是 SVD 可重建——本质上不丢信息。与 HARD-KV(06-30 跟踪)的 Cascade Cache 三层+Logits Calibration 方案形成互补：HARD-KV 解决"动态驱逐 vs CUDA Graph 不兼容"，SeKV 解决"如何处理被压缩部分的信息恢复"

### 4.2 RaBitQCache: 旋转二值量化自注意力稀疏框架（重点 ⭐🔥）

- **作者**: Wenhao Li, Jinhao Dong, Hailin Zhang, Wenhang Shi, Wei Lu, Xiaoyong Du
- **来源**: arXiv:2606.31519 (cs.LG, cs.CL), **已接收 ICML 2026**（周三新刊）
- **核心问题**: 长上下文 LLM 推理被 KV Cache 严重瓶颈化，现有稀疏注意力方法要么使用固定预算 Top-k 检索，要么依赖计算昂贵且有偏的代理分数
- **核心技术**:
  1. **随机旋转二值量化**: 对 KV 进行旋转二值量化，通过高吞吐 binary-INT4 算术高效估计注意力权重
  2. **无偏代理分数**: 带**有界误差证明**的无偏估计器
  3. **自适应 Top-p 检索**: 根据实际注意力稀疏度动态调整 token 预算
  4. **硬件感知实现**: 异步流水线 + 惰性更新掩盖开销
- **关键数据**:
  - 显著加速推理 + 减少内存 I/O
  - 生成质量无损
  - 代码开源
- **意义**: **旋转二值量化作为稀疏注意力的代理分数，证明是一种有效且有理论保证的方案**。与 SeKV(4.1)的分辨率自适应不同——RaBitQCache 走的是"量化压缩→快速检索"路线。有趣的是两者都使用"无偏/无损"的保证，代表了**"不丢弃信息"和"不引入误差"两条技术路线**。ICML 2026 接收意味着其方法论被顶会认可

### 4.3 KV Cache 安全审计: 多 Agent 协作中的 KV Cache 完整性

- **来源**: arXiv:2606.28958 (cs.MA, 提交 2026-06-27，本次轮入)
- **核心**: 当 LLM Agent 之间共享完整 KV Cache 状态(不仅是文本)时，恶意 Agent 可以在视觉消息看起来无害的同时篡改隐藏状态
- **发现**: 文本验证器无法发现此问题；HMAC-SHA256 清单绑定身份/会话/模型/可见承诺/张量元数据/负载摘要——774 个诚实转发全通过，295 个篡改全拒绝
- **评价**: 分布式推理系统中 KV Cache 跨节点传输的安全性正在成为新问题。虽不属核心优化范畴，但在**万卡环境下跨节点 KV 共享**场景下需要考虑信任边界

## 6. 📦 Checkpoint 容错

### 6.1 LLM 驱动的自动 Checkpoint/Restart 代码生成

- **来源**: arXiv:2606.30921 (Towards Transparent Checkpointing with AI-driven Code Generation, cs.DC)
- **核心**: 前端 LLM (Claude Opus 4.7 + OpenCode CLI) 为 6 个 MPI 科学应用**自动生成端到端 checkp​oint/restart 代码**
- **关键数据**: 平均 **50 分钟**/应用，消耗 3.4M token；5/6 应用正常运行开销可忽略，故障恢复效率与人工实现相当
- **评价**: 虽非 LLM 训练场景，但证明了**LLM 可将 checkpoint 集成实现从"专家数周工作"压缩到"自动 50 分钟"**。对大规模分布式训练场景中 checkpoint 框架的自动部署和适配有潜在参考价值

### 6.2 StreamGuard: 实时 HPC 数据流的低开销弹性

- **来源**: arXiv:2606.30848 (StreamGuard, ICS'26)
- **核心**: 异步非阻塞 checkpoint + 进度感知负载再分配，针对 producer-consumer 流模式
- **关键数据**: 故障和性能异常影响降低 **6×**，无故障运行开销 <1%
- **评价**: 更偏 HPC 科学计算流场景，但其**非阻塞 checkpoint 思想**对大规模 LLM 训练中异步 checkpoint + 训练重叠的设计有借鉴
