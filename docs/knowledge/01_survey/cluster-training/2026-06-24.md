## Ⅰ. KV Cache 优化与量化

### ① HyperQuant: 统一权重+KV Cache PTQ 管线 (arXiv:2606.23406, 6/22)

- **核心思想**: 将四个已知技术组合成**统一的 PTQ 管线**——per-tile RHT + 最优格点量化（E8/D4/A2/Z）+ Rice 熵编码 + KV cache 偏差校正
- **效果对比**:
  - 权重 3-5 bps: **全面超越 HIGGS 方案**
  - KV cache 低至 **1.7 bps**: 超越 TurboQuant 和 OCTOPUS
  - 4 bps 下: 线性层压缩 ~**3.9×**，KV cache 压缩 ~**3.79×**，近无损质量
- **硬件适配**: 集成 fp8-e4m3 / int8 / nvfp4 / mxfp4 张量核路径，实验发现 RHT 后 int8 优于 fp8
- **验证**: 19B LTX-2 DiT 视频模型量化后无 perceptible artifacts；全部实验自洽（Table 1）
- **意义**: "RHT + 格点量化"已成 PTQ SOTA 标配，HyperQuant 首次统一了权重和 KV cache 的完整管线

### ② MAC-Attention: Match-Amend-Complete 方案 (arXiv:2604.00235, 3/31)

- **问题**: 长期上下文解码 IO-bound——每个 token 重读不断增长的 KV cache
- **方案**: 对近期语义相似 query，**复用先前 attention 计算结果**，避免完整重计算：
  - **Match**: 预-RoPE L2 匹配短局部窗口
  - **Amend**: 重算 match 边界附近的窄带校正注意力
  - **Complete**: 融合校正结果 + KV 尾部完整注意力（数值稳定合并）
- **命中时复杂度**: 常数（与上下文长度无关），无需精度折损
- **效果**（LongBench v2/RULER/LongGenBench 上 vs FlashInfer）:
  - KV 访问减少 **99%**
  - 128K 下 token 生成延迟降低 **60%**
  - Attention 阶段加速 **14.3×**，端到端加速 **2.6×**
- **兼容性**: MQA/GQA/paged-KV/IO-aware kernels 均可集成
- **意义**: 注意力计算重用的理念与 Kamera（位置无关补丁复用）形成互补——前者复用计算结果，后者复用 KV 块本身

### ③ SANTA: 随机稀疏注意力 (arXiv:2605.01910, ICML 2026)

- **方案**: 从 post-softmax 分布采样 $S \ll n_k$ 个索引，只聚合采样到的 value 行——无偏估计器，消除乘法累加，替换为 gather-and-add
- **分层/系统采样设计**: 方差缩减 GPU 友好变体
- **效果**（Llama-3.1-8B, 32K 上下文, RTX 6000 Ada）:
  - Decode-step attention kernel **1.5×** 加速（vs FlashInfer/FlashDecoding）
  - 批量长期上下文生成端到端加速 **1.25×**
- **互补性**: 与量化/低秩投影/KV cache 压缩/选择可叠加
- **开源**: ICML 2026 接收

## Ⅱ. PD 分离部署与跨实例 KV 通信

### ④ Move the Query, Not the Cache: 跨实例 MLA Attention 路由表征 (arXiv:2606.01502, 5/31)

- **问题**: 当 KV cache 超出单 GPU 容量需跨实例分区时，query 和其选择的 block 往往位于不同 GPU——传统直觉是"搬 cache"
- **核心发现**: **MLA 翻转了算术结构**——每个 token 的 K/V 被压缩为一条窄向量，routed query row 仅 ~1 KB，比被查询的 chunk 还小。**路由 query 经常比搬 cache 便宜**
- **实验**: 真实多节点 H100 集群 + IBGDA RDMA
  - **定量**: vs 搬 cache（~3 ms splice / scattered gather）→ 路由 query（tens-of-microseconds 往返）
  - **关键工件1**: 拓扑感知成本模型（probe/transfer/compute/return/merge 五段）
  - **关键工件2**: 闭式 route/fetch/local 判别谓词，在 IBGDA 上实测偏差仅 ~**7%**
- **适用范围**: DeepSeek-V3.2/DeepSeek-V4/GLM-5.1 等 MLA 架构；扩展到新架构仅需测量 routed payload 和 fetch cost 两个系数
- **意义**: 跨实例 KV 共享的"第一性原理"问题——当 query 足够小时（MLA ~1KB），网络拓扑应由 probe 延迟而非峰值带宽决定

### ⑤ KVServe: 服务感知自适应 KV 压缩 (arXiv:2605.13734, **SIGCOMM 2026**)

- **背景**: PD 分离后 KV 成为显式跨网络/存储 payload，构成端到端瓶颈
- **方案**:
  1. **统一模块化压缩策略空间**（含跨方法重新组合的新组件）
  2. **贝叶斯性能画像引擎**: 高效搜索策略空间，提炼 3D Pareto 候选集，搜索开销降低 **50×**
  3. **服务感知在线控制器**: 分析延迟模型 + 轻量级 bandit 算法实时选择
- **效果**（vLLM 集成，多模型多 GPU 评估）:
  - PD 分离场景: JCT 加速最高 **9.13×**
  - KV 分离场景: TTFT 降低最高 **32.8×**
- **意义**: 首个"服务感知"KV 压缩方案——同一压缩策略在不同 workload mix/带宽/SLO 下效果不同，需动态适应

### ⑥ SplitZip: GPU 友好的无损 KV 压缩 (arXiv:2605.01708, 5/3)

- **背景**: PD 分离后 prefill 产生的 KV cache 需传输到 decode worker，长输入和 Agent 工作负载加剧瓶颈
- **方案**: 利用 KV 激活值浮点指数的冗余性——top-16 离线指数 codebook（免在线直方图）+ 固定长度编码 + 稀有指数 escape 流
- **性能**（BF16 现实激活值）:
  - 压缩吞吐: **613.3 GB/s**
  - 解压吞吐: **2,181.8 GB/s**
  - BF16 KV 传输加速 **1.32×**，TTFT 加速 **1.30×**，请求吞吐提升 **1.23×**
  - FP8 KV 上额外压缩 **1.14×** over native E5M2
- **意义**: 将无损 KV 压缩推进到 GPU 原生生产级——解压比 cuBLAS GEMM 还快，码流仅数十微秒延迟

## Ⅲ. Speculative Decoding 前沿

### ⑦ RLM-Cascade: 响应级 SD 降低 API 成本 (arXiv:2606.22840, 6/22)

- **方案**: 无需架构访问/共享词典的 proxy-layer 系统——快速 draft 模型生成候选回答，大型 verify 模型接受/增强/跳过
- **生产部署**（Claude Code 上线，125 个生产请求）:
  - Draft 使用率 **88.8%**
  - API 成本降低 **45.8%**
  - P50 延迟 **2,026 ms vs 3,698 ms (Native Opus)** — 1.83× 加速
  - 质量: 20-task Code/Math/Instruct 基准 **100%** 通过率 vs Native Opus 95%
- **路由策略**: 规则复杂度路由器 + 混合工具调用策略（schema 关键 turn 跳过 SD）
- **开源**: Prometheus + 实时仪表盘
- **意义**: SD 从学术概念进入生产级 API 代理，88.8% 的 draft 使用率表明"低成本先试"模式已足够可靠

### ⑧ EfficientRollout: RL Rollouts 的自推测解码 (arXiv:2606.18967, 6/17)

- **问题**: RL post-training 中 rollout generation 是延迟瓶颈——长尾生成长度决定完成时间；传统 SD 的两个独特挑战：
  - (i) 策略持续演化 → 固定 drafter 与目标策略分布不匹配
  - (ii) 活跃 batch 缩小 → 从计算 bound 转向内存 bound 区域
- **方案**: 从目标模型诱导**量化自 drafter**（self-speculative decoding），无需独立 drafter 预训练
  - 系统感知 SD 开关策略（仅在有利 regime 推测）+ 接受感知 draft 长度自适应
- **效果**: Rollout 延迟降低 **19.6%**，端到端延迟降低 **12.7%**，最终模型质量无损
- **意义**: SD 首次系统性应用于 RL 训练阶段的 rollout，而非仅服务推理

### ⑨ SafeSpec: 安全感知推测解码 (arXiv:2606.19755, 6/17)

- **问题**: 现有安全防御与推测解码不兼容——额外计算破坏加速机制
- **方案**: 在验证过程中集成**轻量级潜在安全头**，与语义验证共享单次前向传播
  - 检测不安全生成时：**回滚 + 安全引导反射多采样**（非终止生成）
  - 将 jailbreak 攻击建模为生成轨迹上的分布偏移
- **效果**（Qwen3-32B）:
  - ASR 降低 **15%**
  - 同时保持 **2.06×** 推理加速（良性工作负载）
- **意义**: 首次证明推测加速与推理时安全可联合优化，非零和博弈

### ⑩ Draft-OPD: On-Policy Distillation 训练 Speculative Draft 模型 (arXiv:2605.29343, 5/28)

- **问题**: SFT 训练 draft 模型很快 plateau——offline-to-inference mismatch 导致测试时接受长度不再增长
- **原因**: SFT 学固定目标生成轨迹，推断时实际是 draft 自身策略产出的块来验证
- **方案**: 利用目标模型**辅助 rollout 保持续稳定** + 从验证暴露的错误位置 replay drafting → 让 drafter 在"自己出错的地方"接受目标反馈
- **效果**: 跨多样任务 **5× lossless 加速** thinking models；超 EAGLE-3 和 DFlash 分别 **23%/13%**
- **意义**: "online distillation for drafter" 成为 SD 训练新范式

### ⑪ Domino: 解耦因果建模与自回归 Draft (arXiv:2605.29707, 5/28)

- **方案**: 并行 draft backbone → 轻量 Domino head 精炼（含前缀依赖因果信息）
- **关键训练**: base-anchored curriculum（先加强并行 backbone，再逐步转向因果校正分布）
- **效果**: Qwen3: 端到端 **5.49×** (Transformers backend) / **5.8×** (SGLang)
- **意义**: 将 SGLang 后端 SD 加速推向 5×+ 级别

### ⑫ PPOW: 性能驱动策略优化 SD (arXiv:2605.14978, 5/14)

- **问题**: 大多数学习型 drafter 仍用 token 级监督目标，但 SD 的效用本质上是**窗口级和前缀敏感的**
- **方案**: RL 框架 PPOW 组合 Cost-Aware Speedup Reward + Distribution-Based Proximity Reward + Adaptive Divergence-Aware Windowing
- **效果**: 平均接受长度 **6.29-6.52**，加速比 **3.39-4.36×**（跨多模型家族）
- **意义**: 将 drafter 优化从 token 级模仿提升到窗口级效用优化

## Ⅳ. 分布式训练并行策略与动态重配置

### ⑬ DynaTrain: 亚秒级在线并行度切换 (arXiv:2605.18815, 5/12)

- **问题**: LLM 训练本质上是动态的——资源波动/RLHF 阶段切换/集群弹性持续改变最优并行布局，现有框架基于静态执行模型
- **方案**: 
  - **虚拟参数空间 (VPS) 抽象**: 将所有分布式训练状态统一到单一逻辑坐标空间，任意并行配置 = 确定性映射
  - **状态路由-过渡层**: 在 VPS 上执行 rank-local 传输，内存感知 + 无死锁调度
  - **弹性管理器**: 新世界构建与持续训练重叠，mask 拓扑变化开销
- **效果**: 70B 密集模型 < **2s** 重配置，235B MoE 模型 **4.36s**；超越基于 checkpoint 的弹性系统达 **三个数量级**
- **意义**: 当"次秒级重配置"成为现实，训练资源弹性调度将从规划问题退化为 runtime 问题

### ⑭ BACE-Pipe: 带宽感知的地域分布式 PP 训练调度 (arXiv:2605.25375, 5/24)

- **问题**: 跨地域 GPU 训练中 PP 调度面临异构带宽/区域电价/多租户头部阻塞三大挑战
- **方案**: 动态优先级机制（联合考虑作业特征+实时网络利用率）+ 带宽感知路径搜索 + 成本最小化分配
- **效果**: 平均 JCT 降低 **27.9%-64.7%**，总电费降低 **12.6%-30.6%**
- **意义**: 当 GPU 稀缺迫使训练跨越地理区域，调度器需同时优化延迟和电价

### ⑮ FoMoE: 打破完整副本范式的联邦 MoE 训练 (arXiv:2606.19025, 6/17)

- **方案**: 将专家层跨 worker 分区，本地训练跳过非本地专家——基于 Factored Gossip DiLoCo 实现跨地域 MoE 训练
- **效果**:
  - 通信成本较 DiLoCo 降低 **1.42×**，较 DDP 降低 **45.44×**
  - 兜售 token 机制实现 **1.4×** 吞吐加速
  - 系统建模将收益投射至 100B 级配置
- **意义**: 首次实现 MoE 参数不完整副本的跨地域训练

## Ⅴ. 通信调度与压缩

### ⑯ NEURON-Fabric: CXL 侧低比特梯度聚合 (arXiv:2606.15045, 6/12)

- **方案**: CXL 内存控制器在梯度 cache line 经过时执行原位低比特聚合（G-Binary sign-count, G-Ternary gated），同时保留 FP32 通道
- **关键**: **层感知准入**——将分类器 head 保持 FP32，backbone 使用低比特聚合
- **效果**:
  - 梯度流量降至 FP32 基线 **3.6-5.4%**
  - 聚合 datapath 5 周期，最大暴露开销 1.67%
  - FPGA place-and-route 验证：512-bit 聚合可视为 near-memory 扩展而非独立加速器
- **意义**: CXL 内存控制器的原位计算能力远超"内存扩展"——从存储设备变为在线聚合引擎

### ⑰ ZipCCL: 无损通信集合压缩 (arXiv:2604.27844, 4/30)

- **问题**: 无损压缩效果取决于数据分布——传统认知认为压缩/解压开销超过带宽节省
- **发现**: LLM 训练通信数据（激活/梯度/参数）**近似高斯分布** → 可利用指数编码高效无损压缩
- **方案**: 理论接地指数编码 + GPU 优化 kernel（通信感知数据布局的内存访问模式）+ 自适应 collective 切换
- **效果**（64 GPU: MoE + 密集 Transformer）:
  - 通信时间降低 **1.35×**
  - 端到端训练加速 **1.18×**（完全无损）
- **意义**: 证明无损压缩在 LLM 训练场景下可以实际加速——高斯先验分布是关键使能因素

### ⑱ CommFuse: 通信分解与融合消除尾延迟 (arXiv:2604.24013, 4/26)

- **方案**: 将传统 reduce-scatter/all-gather 集体操作**分解为 P2P 通信** + 细粒度调度分区计算 → 消除 tail latency
- **兼容性**: DP 训练、TPSP、UP 等各类张量级并行策略
- **效果**: 持续更低延迟、更高 MFU、更高吞吐
- **意义**: 当集体操作的 tail latency 成为瓶颈时，P2P 分解提供显式的尾延迟控制路径

## Ⅵ. Checkpoint 容错

### ⑲ ReCoVer: 弹性 LLM 预训练容错系统 (arXiv:2605.11215, 5/11)

- **核心设计**: 单一不变量——每次迭代保持 **恒定 microbatch 数**，保证梯度随机等价于无故障运行
- **三层协议**:
  1. **容错集体**: 隔离故障跨副本传播
  2. **步内细粒度恢复**: 保持 intra-iteration 进度，防止梯度损坏
  3. **灵活工作负载策略**: 动态将 microbatch 配额重新分配给幸存节点
- **并行无关**: 直接嵌入 3D 并行和 HSDP
- **效果**（最多 512 GPU 端到端预训练）:
  - 256 GPU 故障（分散在整个运行期间）→ 成功维持无故障参考轨迹
  - vs checkpoint-and-restart: **2.23×** 有效吞吐
  - 234 GPU-hours 内多处理 **74.9%** token
- **意义**: 容错设计从"发现故障→回滚 checkpoint"跃迁到"故障期间梯度仍等价的持续训练"

### ⑳ SPARe: 100k+ GPU 的堆叠并行自适应重排序 (arXiv:2603.00357, **ICML 2026**)

- **问题**: 100k+ GPU 规模故障是常态而非例外，重启时间主导 wall-clock 训练时间
- **方案**: 在同步期间通过跨并行组**堆叠冗余数据片** + 自适应重排序执行来掩盖节点故障
- **理论**: 闭式可承受故障数和计算开销表达式，SimGrid 离散事件仿真验证 + 联合优化冗余和 checkpoint 以最小化 time-to-train
- **效果**（600k GPU 规模）: 相比传统复制，time-to-train 降低 **40-50%**
- **规模验证**: 仿真扩展到 **600k GPU**，提供实际可操作的建议
- **意义**: 首次针对"重启主导阶段"（100k+ GPU）设计容错框架，从 checkpoint 频率优化转向完全避免重启

### ㉑ SparrowRL: 稀疏 Delta Checkpoint 实现跨 WAN RL 训练 (arXiv:2602.11456, 2/11)

- **关键观察**: RL 微调每步更新高度稀疏——仅 ~**1%** 参数发生变化
- **方案**: 
  - 将每步表示为稀疏 delta checkpoint
  - 多流传输流水线化 delta 提取
  - 传输与 rollout 生成重叠
  - 吞吐/带宽感知调度 + lease 容错
- **效果**（Qwen3 4B-14B，最多 4 地理区域跨 WAN）:
  - Qwen3-8B 传输负载降低 **79×**
  - 吞吐提升 **2.4-9.5×**
  - vs 理想 RDMA 单数据中心基线差距缩小至 **8.91%**
  - 每美元 token 数比 RDMA 集群高 **1.21-1.59×**
- **意义**: 跨广域网 RL 训练首次接近 RDMA 集群性能——"弱连接"训练实用化的重要里程碑

## Ⅶ. LLM Serving 与前向模拟

### ㉒ Frontier: 面向现代 LLM Serving 的全面精确模拟器 (arXiv:2605.21312, 5/20)

- **问题**: 现有模拟器的 monolithic-replica 抽象不适应 PD 分离 + 复杂并行 + 运行时优化的现代 serving
- **方案**: 离散事件模拟器，支持:
  - **PD 分离 (PDD)**, Attention-FFN 分离 (AFD), 共置
  - CUDA Graphs / Speculative Decoding 等运行时优化
  - 状态请求（推理/Agent/RL rollout）
- **精度**（16×H800 测试台）: 平均吞吐误差 < **4%**
  - vs SOTA 模拟器: 共置延迟误差从 **44.9%→6.4%**，分离误差从 **51.7%→2.6%**
  - 扩展至 1K+ GPU（商用 CPU）
- **新用例**: SLA 依赖 Pareto 前沿探索、异构分离分配、Agent 推理调度验证、RL post-training 重配置
- **开源**: https://github.com/NetX-lab/Frontier
- **意义**: 现代 serving 系统设计空间爆炸，模拟精度是决策前提——Frontier 将误差从 50% 下拉到 6% 以内

### ㉓ SPECTRE: 多租户远程 Draft 模型复用 (arXiv:2605.08151, 5/3)

- **问题**: 多数 LLM 服务平台请求呈长尾分布——少数热门模型占大部分请求，大量尾部模型闲置
- **方案**: 将闲置尾部模型作为热门模型的**远程 drafter**，通过 SD 加速
  - 混合普通-并行 SD 策略（阈值门控）
  - 推测优先级调度保持 draft-target 重叠
  - Draft 侧 prompt 压缩
- **效果**（SGLang 实现, Qwen3-235B-A22B, TP=8）: 最高 **2.28×** 加速，相对于最强 SD 基线额外 **66%** 提升
- **开源**: https://github.com/sgl-project/sglang/pull/22272
- **意义**: 将空闲 GPU 转化为加速资产——多租户 SD 是"算力利用率"的新维度
