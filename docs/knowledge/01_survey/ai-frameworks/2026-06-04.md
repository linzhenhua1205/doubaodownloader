## 1️⃣ PyTorch 2.12 发布（2026-05-13）

> **来源**: [PyTorch Blog](https://pytorch.org/blog/pytorch-2-12-release/) | 2,926 commits, 457 contributors

### 核心性能提升

| 特性 | 具体数据 | 关键PR |
|:-----|:---------|:-------|
| **linalg.eigh** 批处理 eigendecomposition | **最高100x加速**（cuSolver syevj_batched） | PR #174619, #175403 |
| **Adagrad fused=True** | 单CUDA kernel完成整个优化器步骤 | PR #159008, #177672 |
| **torch.accelerator.Graph** | 统一的 device-agnostic 图捕获与回放 API（CUDA/XPU/第三方后端） | PR #171269, #171285 |
| **torch.export** 支持 MX 量化 | 完整支持 float8_e8m0fnu → MXFP4/MXFP6/MXFP8 导出 | PR #176270 (ARM) |
| **torch.cond** CUDA Graph 内控制流 | 利用 CUDA 12.4 的 conditional IF nodes，GPU 内完整图捕获 | PR #168912 (NVIDIA) |

### 分布式训练

- **ProcessGroup 自定义算子**: 自定义算子可直接接受 ProcessGroup 对象作为参数
- **Profiler 增强**: 流ID、流类型、活动类型，NCCL seq_num 跨 rank 关联
- **FlightRecorder**: 扩展到 ncclx/gloo 后端 + torchcomms 操作追踪
- **torchcomms 即将成为默认**: PyTorch 2.13+ 将默认使用 torchcomms

### 平台更新

| 平台 | 关键更新 |
|:-----|:---------|
| **ROCm** | expandable segments (ROCm≥7.02) · rocSHMEM 对称内存集合操作 · hipSPARSELt 2:4 稀疏 FP8 (MI350X) · FlexAttention pipelining **5-26% 加速** |
| **CUDA** | Green Context workqueue limit · CUDA Graph 内核注解（kernel annotation）|
| **Apple MPS** | Metal-4 离线着色器预编译，消除首次运行开销 |
| **CUDA 12.8 wheel 弃用** | 默认 wheel 切换为 CUDA 13.0，CUDA 13.2 实验构建 |

> **意义**: PyTorch 2.12 延续了从研究框架向统一的硬件无关生产平台的演进路线。torch.accelerator.Graph 跨后端统一是重大架构变化；MX 量化导出填补了模型部署的关键空白；ROCm 功能密集补齐表明 AMD GPU 生态加速成熟。

## 2️⃣ DeepSpeed 支持 Muon Optimizer（2026-06-03）

> **来源**: [PyTorch Blog](https://pytorch.org/blog/using-muon-optimizer-with-deepspeed/) | DeepSpeed 团队

### 关键背景

Muon Optimizer 已成为大规模训练的主流选择：
- **DeepSeek-V4** (1.6T 参数) — 使用 Muon 加速收敛
- **Kimi-K2** (1T 参数) — MuonClip 变体
- **GLM-5** (744B 参数) — 使用 Muon + "Muon Split" 技术
- NanoGPT 速度提升 **35%**，1.5B 参数规模达到 GPT-2 XL 性能快 **25%**

### DeepSpeed Muon 支持状态

| 特性 | 状态 |
|:-----|:------|
| ZeRO Stage 2 | ✅ **已合并** |
| ZeRO Stage 3 | ✅ **已合并** |
| Gram-Schmidt Newton-Schulz (更快的正交化 kernel) | 🔄 审核中 |
| CPU Offloading | 🔄 进行中 |
| MuonClip（Kimi-K2 变体） | 📋 规划中 |

### 性能数据 (Moonlight-16B-A3B MoE, ZeRO-2, bf16, 4 GPU)

| Benchmark | AdamW | Muon | 差距 |
|:----------|:-----:|:----:|:----:|
| MBPP | 0.661 | 0.646 | -1.5pp |
| **MBPP+** | 0.534 | **0.548** | **+1.4pp** ✅ |
| **MMLU** | 0.660 | **0.678** | **+1.8pp** ✅ |
| **GSM8K** | 0.805 | **0.810** | **+0.5pp** ✅ |

### 内存节省

| 指标 | AdamW | Muon | 节省 |
|:----|:-----:|:----:|:----:|
| 优化器状态/参数 | 8 bytes (m+v) | 4 bytes (momentum) | ~50% |
| Qwen2.5-3B 微调峰值显存 | 34.5 GiB | 31.4 GiB | **9%** |

> **意义**: Muon 正在成为替代 Adam 的默认训练优化器。DeepSpeed 的完整支持将推动更多大模型训练采用 Muon。实测内存节省 9% 在显存紧张场景中可能决定是否可用。

## 3️⃣ TokenSpeed 推理引擎：Qwen3.5 达 580 tps（2026-05-27）

> **来源**: [PyTorch Blog](https://pytorch.org/blog/up-to-580tps-new-speed-record/) | TokenSpeed Team · Qwen Team  
> **项目**: [LightSeek Foundation](https://github.com/lightseek-foundation) | MIT 协议 | 开源

### 核心设计

TokenSpeed 是一个全新的**高性能开源 LLM 推理引擎**，定位为：
> "达到 TensorRT-LLM 级性能，保持 vLLM 级开发者友好性"

| 特性 | 说明 |
|:-----|:------|
| **架构** | 原生 SPMD 架构 + 静态编译 |
| **目标场景** | Agentic workload（多步工具调用、长上下文、高前缀命中） |
| **模型支持** | Qwen3.5 混合注意力（全注意力 + GDN 线性注意力） |
| **关键优化** | Mamba 前缀缓存（COW/checkpoint）· PD 分离·索引间接寻址消除状态拷贝·FA4 支持 |

### 性能数据 (Qwen3.5-397B-A17B, B200, NVFP4)

| 场景 | 配置 | 吞吐 |
|:-----|:------|:----:|
| **Agentic 单用户** | TP8, bs=1 | **~580 tok/s** 🏆 |
| Agentic 单用户 | TP4 | ~500+ tok/s |
| Agentic 16 并发 | TP4EP4 | ~2K tok/min/GPU |
| 长上下文 128K | TP8, bs=1 | ~530 tok/s |
| 长上下文 256K | TP8, bs=1 | ~495 tok/s |
| 长上下文 1M | TP8, bs=1 | ~445 tok/s |
| 128K→1M 退化 | TP8 | 仅 **~16%** |

### 关键技术亮点

1. **GDN/Mamba 前缀缓存**: radix-tree + MambaSlot，四阶段生命周期（working→checkpoint→COW→published）
2. **PD 分离状态传输**: 统一 RDMA 机制传输 KV Cache + Mamba 状态，三阶段握手保证一致性
3. **零拷贝状态更新**: 用指针移动替代中间缓存（O(1) vs O(L·D)）
4. **MTP 加速**: bs=1 时 +100%~+159%，长输出场景 (≥4096) 持续 +38%~+90%
5. **Agentic 前缀命中率 >90%**: PV 粒度前缀缓存极大减少 Prefill 开销

> **意义**: TokenSpeed 是推理引擎领域的新竞争者。其 Agentic workload 优化（580 tps）和 GDN 混合架构支持填补了 vLLM/TensorRT-LLM 的空白。但目前仅针对 Qwen3.5 优化，通用性待验证。

## 4️⃣ TensorRT-LLM AutoDeploy Beta（2026-02-09 发布）

> **来源**: [NVIDIA 技术博客](https://developer.nvidia.com/blog/automating-inference-optimizations-with-nvidia-tensorrt-llm-autodeploy/) | TensorRT-LLM v1.3.0rc1

### 核心创新

AutoDeploy 将推理优化从**手动重写**转变为**编译器驱动流程**：
- 接受原生 Hugging Face PyTorch 模型 → 自动生成推理优化图
- 支持 **100+ 文本模型**，早期支持 VLM 和 SSM
- 无需手动添加 KV Cache、分片、量化、kernel 融合

### 技术架构

```
PyTorch Model → torch.export → 模式匹配/规范化 → 分片/融合 → TensorRT-LLM Runtime
```

### 性能数据

| 模型 | 配置 | 在线吞吐 (tok/s/user) | 离线吞吐 (tok/s) |
|:-----|:-----|:----------------:|:---------------:|
| Nemotron 3 Nano FP8 | DGX B200, TP1 | **~350** | **~13,000** |
| Nemotron-Flash 3B vs Qwen2.5 3B | H100, ISL/OSL=8k/16k | AutoDeploy 不弱于手动调优 | — |

### 关键能力

- **模式匹配**: 自动识别 MoE、Attention、RoPE、SSM 层 → 规范化为 custom ops
- **缓存集成**: 自动将注意力/SSM/线性注意力的缓存机制接入 TensorRT-LLM cache manager
- **自定义算子注入**: 模型作者可通过 PyTorch custom op 装饰器注入自定义 kernel
- **编译工具链集成**: torch.compile + CUDA Graph + 多流优化

> **意义**: AutoDeploy 解决了推理框架的"长尾模型问题"——新架构需要数周手动适配，现在缩短至数天。Nemotron-Flash 这种混合SSM/注意力/线性注意力的复杂架构也能快速部署。

## 5️⃣ NVIDIA Dynamo Snapshot：21x 冷启动加速（2026-05-27）

> **来源**: [NVIDIA 技术博客](https://developer.nvidia.com/blog/nvidia-dynamo-snapshot-fast-startup-for-inference-workloads-on-kubernetes/) | Dynamo Team

### 核心技术

基于 **CRIU (Checkpoint/Restore In Userspace)** + **cuda-checkpoint** 实现 GPU 推理工作负载的近乎即时恢复。

### 三大优化

| 优化 | 效果 |
|:-----|:------|
| **KV Cache unmap/release** | Qwen3-0.6B 检查点从 ~190 GiB → ~6 GiB |
| **并行 memfd restore** | 多线程并行恢复共享内存 |
| **Linux native AIO** | 同时最多 128 个 I/O 并发（替代串行 preadv 循环） |
| **GPU Memory Service (GMS)** | 权重与进程状态解耦，支持 GPUDirect Storage 并行恢复 |

### 恢复时间对比

| 模型 | 冷启动 | CRIU (上游) | CRIU (优化后) | 加速比 | 速度极限 (SOL) |
|:-----|:-----:|:-----------:|:------------:|:-----:|:-------------:|
| Qwen3-0.6B | ～ | 6.8s | **2.4s** | 2.8x | 0.95s |
| Qwen3-8B | ～ | 24s | **4.7s** | 5.1x | 1.8s |
| **gpt-oss-120b** | ～ | 119s | **15s** | **7.9x** | 11s |
| **gpt-oss-120b + GMS + NVMe** | ～ | 119s | **<5s** | **21x** 🏆 | — |

> **意义**: Dynamo Snapshot 将推理工作负载的弹性伸缩从分钟级加速到秒级，对 SLA 敏感的生产部署至关重要。目前支持单GPU vLLM/SGLang，规划支持 TensorRT-LLM 和多GPU场景。

## 6️⃣ Kernel 编译与优化前沿动态

### 🏆 6.1 Kernel-Smith：统一进化式 Kernel 优化（arXiv 2026-04-23）

> **来源**: [arXiv:2603.28342](https://arxiv.org/abs/2603.28342) | KernelSmith-235B-RL

- **方法论**: 稳定评估驱动的进化式 Agent + 进化导向的后训练（RL）
- **性能**: KernelBench Triton backend **SOTA**，超越 Gemini-3.0-pro 和 Claude-4.6-opus
- **跨平台**: MetaX MACA backend 验证，Kernel-Smith-MACA-30B 超越 DeepSeek-V3.2-think
- **生产贡献**: 已贡献到 **SGLang** 和 **LMDeploy** 的生产系统

### 🏆 6.2 AgentKernelArena：GPU Kernel 优化 Agent 基准（arXiv 2026-05-16）

> **来源**: [arXiv:2605.16819](https://arxiv.org/abs/2605.16819) | 196 tasks

- **覆盖**: HIP-to-HIP / Triton-to-Triton / PyTorch-to-HIP
- **最佳 Agent**: 平均加速 PyTorch-to-HIP **6.89x** / HIP-to-HIP **6.69x** / Triton-to-Triton **2.13x**
- **泛化探测**: HIP/HIP 和 Triton/Triton 高度可迁移，但 PyTorch-to-HIP 正确率显著下降（形状硬编码问题）

### 🏆 6.3 Neptune 高级算子融合编译器（arXiv 2026-04-19，CGO'26）

> **来源**: [arXiv:2510.08726](https://arxiv.org/abs/2510.08726)

- **创新**: 主动打破循环依赖 + 代数校正表达式补偿
- **性能**: 跨 NVIDIA/AMD 平均 **1.35x**，最高 NVIDIA **2.65x** / AMD **3.32x**
- **输入**: 仅需 plain attention + 高级调度模板 → 生成 FlashAttention / FlashDecoding

### 🏆 6.4 Tawa 自动 Warp 特化编译器（CGO'26）

> **来源**: [arXiv:2510.14719](https://arxiv.org/abs/2510.14719) | NVIDIA H100

- **创新**: 自动将 tile 程序编译为 warp-specialized 代码 + 异步引用 (aref) IR 抽象
- **性能**: GEMM 达 cuBLAS 的 **1.1x**，Attention 达 Triton 的 **1.2x**，媲美手写 CUTLASS FlashAttention-3

### 🏆 6.5 微调 GPT-5 生成 Triton Kernel（arXiv 2026-02-11）

> **来源**: [arXiv:2602.11000](https://arxiv.org/abs/2602.11000) | Makora

- **方法**: RL 微调 GPT-5 生成 Triton 代码
- **单次正确率**: 43.7% → **77.0%** (+33.3pp)
- **超越 TorchInductor**: 14.8% → **21.8%** (+7pp)
- **Agent 模式**: 解决 **97.4%** 的扩展 KernelBench 问题，几何平均加速 **2.12x**
