## 一、 arXiv 论文

### 1️⃣ Block-GTQ: RoPE-Aware Bit Allocation for KV-Cache Quantization

> **arXiv**: 2606.24033 | **提交**: 2026-06-23 | **代码**: https://github.com/fengfengliang/Block-GTQ

**核心思想**: 现有低比特 KV Cache 量化器把每个 cached key 当作扁平向量处理。但在 RoPE 下，key 对 attention logit 的贡献分解为二维频率块的和。因此 key cache 量化本质上是块级比特分配问题——高能量 RoPE 块对量化误差更敏感，应分配更多比特。

**方法**: Block-GTQ 为每个 RoPE block 计算无标签能量分数，通过边际增益贪心分配 integer bit width。在匹配 K/V 比特预算下，Block-GTQ 在 10 模型诊断面板上更好地保留了 RoPE query-key logits。

**性能数据（重要）**:
- **单层 MAE 降低 32–80%** 在 2 和 3 b/dim K-only 量化下，全部 367/367 层比较胜出 uniform TQ-MSE
- **K2V2 on Llama-3.1-8B-Instruct**: NIAH 六任务均值从 70.6 → **97.4**，LongBench-EN 均值从 36.87 → **53.31**
- **DeepSeek-R1-Distill-Qwen-7B (AIME 2024/2025)**: K3V2 得分 **51.7/37.5**，fp16 基线 54.2/37.9（uniform TQ-MSE 崩溃到 0.0/0.0）
- **单 H800 GPU + Qwen2.5-3B-Instruct**: packed K3V3 实现 **3.24× KV cache 压缩**，fp16 可比质量
- **1.34× 快于 fp16 FlashAttention2** 在 128K context
- **峰值内存: 56.31 GB → 19.85 GB**（降低 36.46 GB）
- 在 256K 和 512K 下保持可行，fp16 OOM

**意义**: 标志着 KV Cache 量化从"uniform bit-width"走向"RoPE结构感知"的第二阶段。Block-GTQ 在极低比特（K2V2）下几乎消除了与 fp16 的差距，对大规模部署有直接价值。

### 2️⃣ Nexus Sampling: Forget Without Compromise

> **arXiv**: 2606.23961 | **提交**: 2026-06-22

**核心思想**: 长上下文和 Agentic LLM 工作负载迫使推理栈在每步永久驱逐部分 token。现有方法的共同模板是：每步 direct-attention score + deterministic top-K 选择——一个单步低于阈值就变成不可逆判决。Nexus Sampling 提出了 **Nexus scoring**（一种在 direct attention 上迭代行走以发现 bridge token 的方法）+ **weighted reservoir sampling**（用包含概率替代 deterministic top-K）。

**理论贡献**: Nexus Sampling 在长期生存 subtlely important tokens 方面 dominates deterministic top-K。

**性能数据**:
- **80% KV cache 驱逐率下**，Nexus Sampling 在 LongBench 上匹配密集注意力 **1% 以内**
- 在 retrieval-heavy 任务上全面超越 top-K baseline
- **每个序列 cache 内存降低多达 10×**

**意义**: 这是 KV Cache 驱逐领域的重要理论突破。证明了确定性 top-K 驱逐在长尾 token 保留上有系统性缺陷，而随机化采样策略可以理论保证 + 实证验证地弥补这一缺陷。

### 3️⃣ MOCAP: Wafer-Scale-Chip Prefill Pipeline

> **arXiv**: 2606.22968 | **提交**: 2026-06-22 | **录用**: APPT 2026

**核心思想**: 面向 Wafer-Scale-Chip（晶圆级芯片）的 prefill-only LLM 推理的 chunked pipelining 框架，解决因果依赖导致的 KV cache 不均和 chunk 延迟不均问题。

**方法**: Memory-Balanced KV Reallocation (MBKR) + Latency-Balanced Chunk Partitioning (LBCP)

**性能数据**:
- **76.4% 更低端到端延迟** vs GPipe
- **3.24× 更高吞吐** vs GPipe
- 最大支持序列长度扩展 **1.31×** vs Terapipe

### 4️⃣ The Price of Anarchy in Disaggregated Inference

> **arXiv**: 2606.17081 | **提交**: 2026-06-11 | **平台**: NVIDIA Dynamo v0.9.0 on B200 cluster

**核心思想**: 首次对 PD 分离推理架构进行正式博弈论分析。建模为三个耦合博弈：P/D 池资源博弈、分层 KV Cache 自私缓存博弈、请求路由拥塞博弈。

**关键发现**:
- **3 节点 NVIDIA B200 集群**，Nemotron-4-340B (TP=8) + Llama-3.1-70B (TP=4)
- 70B 1P/5D 拓扑：**PoA-hat 降低 3.1×**（66.4 → 21.5）在饱和阶段，**13% 吞吐代价**
- 70B 1P/2D：**PoA-hat 降低 2.2×**，**TTFT P99 降低 7.6×**
- 自适应路由控制器在饱和/非饱和间自适应切换策略（cache-affinity → load-balanced congestion avoidance）

## 四、 编译器优化

### 4.1 Triton 3.7.1

> **发布**: 2026-06-18 | 补丁版本，无 API 变更

**修复**:
1. **FenceAsync 缺少 async read 依赖**: st.shared 与 async copy_local_to_global 之间的 fence 缺失导致 async copy 可能在 shared memory store 完成前读取共享内存，产生错误结果
2. **InstCombine shrink added constant**: 使用 LHS known zeros 修复 LLVM InstCombine miscompilation — add 简化使用了已知零 bits，但未正确 keep zero bits from the other operand


## 五、 KV Cache 优化深度分析

### 本周三重奏：量化 + 驱逐 + 卸载

| 方法 | 类别 | 关键技术 | 性能亮点 | 适用场景 |
|:-----|:-----|:---------|:---------|:---------|
| Block-GTQ | 量化 | RoPE块级感知比特分配 | 3.24×压缩, 1.34×加速, 接近fp16质量 | 推理部署（单卡/多卡） |
| Nexus Sampling | 驱逐 | 加权水库采样替代top-K | 80%驱逐率匹配<1%质量损失 | 长上下文/Agent 场景 |
| MOCAP | 卸载+Pipeline | MBKR + LBCP | 76.4%延迟降低, 3.24×吞吐 | 晶圆级芯片 Prefill 场景 |
| vLLM KV Offload | 卸载 | 三级卸载+对象存储+HMA | 生产就绪 | 大规模分布式部署 |
| SGLang HiCache | 卸载 | UnifiedTree + 混合模型默认 | 默认启用 | Hybrid 模型 (SWA/Mamba) |

**趋势观察**: KV Cache 优化正从"单一技术路线"走向"三维协同"。量化（Block-GTQ）解决内存占用的根本问题，驱逐（Nexus Sampling）解决长序列溢出问题，卸载（vLLM/SGLang）提供弹性扩展。三个方向互补而非替代。


## 六、 关键趋势观察

1. **RoPE 结构感知标准化**: Block-GTQ 验证了 RoPE 的频域结构可以（也应当）用于指导量化/驱逐决策。未来 KV Cache 工具链可能将 RoPE block 划分作为标准 API

2. **随机化驱逐取代确定性驱逐**: Nexus Sampling 从理论上证明了加权随机化在保留 long-tail token 方面的优势，这是 KV Cache 驱逐五年来从 heuristic →学习型→随机化的演进第三阶段

3. **Blackwell 生态加速成熟**: TensorRT-LLM TrtllmGenAttention 默认 + SGLang FlashInfer GDN on Blackwell + PyTorch Triton 3.7.1 for B200，Blackwell 的软件生态配套正在加速赶上

4. **PD 分离走向博弈论分析**: Dynamo 的 Price of Anarchy 分析揭示了 PD 分离架构在饱和状态下的系统性效率损失，自适应路由控制器是实用的应对方案
