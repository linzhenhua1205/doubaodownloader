## 1️⃣ CacheWise: 首个 Coding Agent KV Cache 管理方案 — 直接集成 vLLM 🚀

- **来源**: [arXiv:2606.16824](https://arxiv.org/abs/2606.16824) (June 15, 2026)
- **作者**: Shubham Tiwari, Tapan Chugh, Nash Rickert, Simon Peter, Ratul Mahajan, Haiying Shen (Virginia Tech / UT Austin)
- **摘要**: 首个收集真实 Coding Agent 运行时 Trace 并针对性优化 vLLM KV Cache 管理系统的工作

### 背景 — Coding Agent vs Chat 完全不同

现有 LLM 推理系统（vLLM、SGLang）主要为 **Chat 场景** 设计。但 Coding Agent 有截然不同的特征：

| 特征 | Chat | Coding Agent |
|:-----|:-----|:-------------|
| 会话模式 | 简短多轮 | **长时间闭环会话** |
| Prefix 复用 | 有限 | **大量重复大前缀** |
| KV Cache 压力 | 一般 | **持续高压** |
| 已有策略 | 适用 | 传统策略**表现糟糕** 😱 |

### CacheWise 创新

1. **Prefix-Aware Scheduling** — 优先调度共享前缀的请求，最大化 KV cache 复用
2. **Reuse-Aware Eviction** — 通过工具调用元数据的**轻量预测**指导驱逐决策，保护即将复用的缓存块
3. **集成 vLLM** — 无需修改引擎架构，作为 KVCache 管理层插入

### 性能数据

| 指标 | 提升 | 对比基线 |
|:-----|:-----|:---------|
| KV Cache 驱逐量 | **减少 2-2.6×** 🔥 | vLLM 默认策略 |
| Agent 会话完成时间 | **提升 3.5×** 🔥 | vLLM 默认策略 |

### 影响判断
这是业界**首个公开的 Coding Agent LLM 推理工作负载分析**。随着 Coding Agent（Claude Code、Cursor、Copilot）的爆炸性增长，CacheWise 的发现对推理框架的 KV Cache 策略设计有直接指导意义。

## 2️⃣ Gimbal: 跨层协同调度 MoE 推理 — vs vLLM TTFT 降低 42.9% 🎯

- **来源**: [arXiv:2606.15177](https://arxiv.org/abs/2606.15177) (June 13, 2026)
- **作者**: Yifan Sun, Zhexiang Zhang, Jiantong Jiang, Gholamreza Haffari, Minxian Xu, Feng Liu, Rajkumar Buyya, Adel N. Toosi
- **摘要**: 首个跨级别的 MoE LLM 协同调度系统：前端 DP-Engine 调度器 + 后端 Expert 负载均衡

### 核心痛点

现有 MoE 推理系统（vLLM 等）将**请求调度**与**Expert 放置**分开决策：
- 前端 scheduler 用粗粒度的请求计数器路由
- 后端 expert balancer 依赖聚合的激活次数
- 两者都无法感知**细粒度引擎压力**、**MoE 背压**和**源相关的 Expert 流量**

### Gimbal 创新

**① 细粒度 DP-Engine Scheduler**
- 综合 KV cache 使用量、剩余 prefill 工作量、队列压力和 MoE Expert 压力
- 将请求调度避让过载引擎
- 轻量级 Prefill-Aware 队列排序（无输出长度预测）

**② 在线 Expert Load Balancing**
- 引入源-DP-to-Expert 统计信息
- MINLP 启发的启发式放置：联合考虑 Expert 负载、源感知通信和迁移稳定性

### 性能数据（vs vLLM）

| 指标 | 降低/提升 |
|:-----|:----------|
| **平均 TTFT** | **↓ 42.9%** 🔥 |
| **平均 TPOT** | **↓ 33.3%** 🔥 |
| 高负载吞吐 | **↑ 3.0%** |

### 影响判断
MoE 模型（DeepSeek-V4、Mixtral 等）的普及使得 Expert 调度变得关键。Gimbal 证明了**前后端协同调度**可以大幅改善 MoE 推理性能——TTFT 降低近一半是令人印象深刻的成果。代码尚未开源。

## 3️⃣ cuTile Rust: 在 Rust 中安全编写 GPU Kernel — Grout 推理引擎 vs vLLM 性能持平 🦀

- **来源**: [arXiv:2606.15991](https://arxiv.org/abs/2606.15991) (June 14, 2026)
- **作者**: Melih Elibol, Jared Roesch, Isaac Gelado, Eric Buehler, **Michael Garland** (NVIDIA)
- **摘要**: 将 Rust 的所有权模型扩展到 tile-based GPU Kernel，在保证内存安全的同时提供与 Python/cuBLAS 相媲美的性能

### 核心创新

1. **cuTile Rust** — 将 Rust 的所有权纪律扩展到 tile-based GPU Kernel
   - 可变输出被拆分为不相交部分
   - Kernel launch 保持 host 侧的所有权合约
   - 可选的局部 unsafe 逃逸
   
2. **Grout 推理引擎** — 基于 cuTile Rust 构建的端到端 Qwen3 推理引擎

### 性能数据

**Kernel 级别（B200）**

| 操作 | cuTile Rust | 对比 |
|:-----|:------------|:-----|
| Element-wise | **7 TB/s** | 匹配 cuTile Python |
| GEMM | **2 PFlop/s** | **96% of cuBLAS** 🔥 |

**推理引擎（Grout vs vLLM/SGLang）**

| 模型 | GPU | Grout 性能 | 对比 |
|:-----|:----|:-----------|:-----|
| Qwen3-4B | RTX 5090 | **171 tok/s** (batch-1 decode) | **与 vLLM/SGLang 持平** |
| Qwen3-32B | B200 | **82 tok/s** (batch-1 decode) | **与 vLLM/SGLang 持平** |

**Rust 安全保证**
- 所有权：GPU 内存的编译时借用检查
- 并发：消除 kernel 间的数据竞争
- 检查：无需 GC 或 runtime 开销

### 影响判断
Michael Garland（NVIDIA 首席架构师）的参与表明这是 NVIDIA 内部重视的方向。用 **Rust 的类型系统** 来保障 GPU Kernel 的内存安全且**零性能损失**，这对未来推理框架的安全性设计有重要参考价值。Grout 在 batch-1 decode 场景下性能与 vLLM/SGLang 持平，说明 Rust 路径已经达到生产级水平。
