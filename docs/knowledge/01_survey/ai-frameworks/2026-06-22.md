# 2026-06-22 AI 框架最新动态跟踪

> **跟踪范围**: 推理框架(vLLM/TensorRT-LLM/SGLang)、训练框架(PyTorch/DeepSpeed)、KV Cache 优化论文

## 一、推理框架

### 🔷 vLLM v0.23.0（6月15日发布，本周重点）

**概况**: 408 commits, 200 位贡献者（63 位新贡献者），距 v0.22.0 仅 17 天，迭代速度持续加速。

#### 核心更新

1. **DeepSeek-V4 深度成熟**
   - 稀疏 MLA metadata 与 DeepSeek-V3.2 解耦（#44699），避免两代模型共用同一套注意力元数据的兼容问题
   - 新增 TRTLLM-gen attention kernel（#43827），在 Hopper/Blackwell 上获得专用 MLA kernel 路径
   - EPLB（Expert Placement Load Balancing）支持 Mega-MoE（#43339），DeepSeek-V4 的 MoE 路由更加均衡
   - 选择性 prefix-cache retention（#43447）：滑动窗口的 KV cache 中保留前缀缓存命中部分
   - Index-share for DSA MTP（#44420），多 token 推测解码的索引共享优化
   - 从 torch.compile 解耦（#43746, #43891），避免 compile 带来的编译开销和稳定性问题
   - XPU attention decode path（#42953）

2. **Model Runner V2（MRv2）推进为默认**
   - 在 Qwen3 之后，MRv2 现在对 **Llama 和 Mistral 密集模型也默认启用**（#43458）
   - 新增 FlashInfer sampler（#42472）、breakable CUDA graphs（#44050）
   - Pipeline-parallel bubble elimination（#42187）：消除流水线并行中的气泡浪费
   - Kernel block-size support for hybrid models（#38831）
   - Gemma 4 MTP 支持（#43241）

3. **Rust 前端成熟**
   - 新增 streaming generate endpoint（#43779）、dynamic LoRA endpoints（#43778）
   - /version（#43854）、/server_info（#43942）等管理 API
   - Server-router extension hook（#43774）
   - 新 tool parsers：InternLM2、hy_v3、Phi-4-mini、Gemma4

4. **多级 KV Cache Offloading 架构**
   - Object-store secondary tier（#41968）：支持将 KV cache 卸载到对象存储（如 S3）作为第三级
   - HMA（Hybrid Memory Allocator）默认启用（#41847）
   - Tiering support for HMA models（#44287）
   - Per-request offloading policy 通过 on_new_request 生命周期钩子（#43205）

5. **Transformers v5 兼容**
   - vLLM 正式转向 Transformers v5 作为目标版本
   - Vendor MiniCPM-V/O processors（#44282）
   - Sarvam（#38804）和 Voxtral（#44559）兼容性修复

#### 性能数据

|:-------|:---------|:-----|

| Fp8BlockScaledMM new_empty() 优化 | 减少显存分配开销 | #43677 |

### 🔷 TensorRT-LLM v1.3.0rc18（6月10日发布）

**概况**: 持续迭代，68 位贡献者。

#### 核心更新

1. **NIXL 升级到 v1.0.1 + UCX 1.21**（#14436）：分布式 KV 传输基础设施成熟
2. **Per-expert LoRA 支持 + Cutlass 后端**（#14801）：MoE 模型可按专家单独加载 LoRA 适配器
3. **FlashInfer GDN decode kernel 支持 Qwen3.5**（#13645）
4. **NVFP4 检查点支持 Nemotron-H on Hopper**（#14775）
5. **KVCacheManagerV2 adjust() 支持 + 磁盘缓存配置**（#14578, #14845）
6. **DWDP 从 CUDA IPC 重构为 CUDA VMM + MNNVL**（#14453）：减少显存碎片和服务中断

#### 已知问题
- **DSV3.2 在 GB200/GB300 长时间 perf 测试中 IMA 崩溃**（使用 CuteDSL MoE 后端时），需切换其他 MoE 后端规避

### 🔷 SGLang v0.5.13（6月13日发布）

**概况**: 最新重要版本，68 位贡献者。

#### 核心更新

1. **Speculative Decoding V2 成为默认路径**
   - Tree drafting (topk > 1) 在 triton/FA3/MLA/aiter 后端已生产就绪（#26997, #26972, #27463）
   - 支持 page_size > 1 和 Mamba/hybrid-linear 模型
   - Spec V1 废弃，EAGLE/MTP 迁移到统一 V2 worker（#25464）
   - topk=1 draft 更快（#26397, #26424）：跳过 full-vocab softmax + 冗余 cat/topk/sort/gather 操作

2. **DeepSeek V4 持续强化**
   - Context Parallel + MTP（#24934）、+ fused MoE kernel（#24947）
   - Sparse FlashMLA via flash_mla_sparse_fwd（#25418）
   - FP4 indexer（#26209）、SM120 支持（#24692）
   - DeepEP waterfill 负载均衡（#25391）
   - Breakable CUDA Graph（#25195）

3. **新模型 Day-0 支持**: Nemotron 3 Ultra, Step-3.7-Flash, Cosmos3, Command A+

4. **HiCache 默认启用**（#27759）：HybridModel (SWA/Mamba) 通过 UnifiedTree 自动启用层级 KV cache offloading

5. **MoRI on AMD Instinct MI355X**
   - 与 AMD 联合优化，DeepSeek-R1 分解推理可达到 **$0.169/百万 token，129 tok/s/user**
   - 在 AMD MI355X 上实现成本竞争力的推理方案

6. **降低 per-step scheduler 开销**: FutureMap 统一异步值传递 + prefill 输入传输移动到 forward stream（#25945, #25879, #26380）

7. **Piecewise & Breakable CUDA Graph 扩展**: 覆盖 DSA 模型、Kimi-K2.5、DeepSeek V4

8. **Heterogeneous CPU+GPU EPD disaggregation**（与 Intel）：VLM 视觉编码卸载到 Intel Xeon CPU，**P99 TTFT 提升可达 ~1.3×**

## 二、训练框架

### 🔷 PyTorch 2.12.1（6月18日，bugfix）/ 2.12.0（5月13日）

**PyTorch 2.12.1 修复**:
- 修复 B200 GPU 上 FLASH_ATTN 的 batch_invariance 测试非确定性输出（#181248），通过更新 Triton 到 3.7.1
- 修复 B100/B200 上 Triton convolution2d_bwd_weight kernel 非法内存访问（#187081）
- 移除 CPython 3.13t 从二进制构建矩阵（#182951）

**PyTorch 2.12.0 Highlights**:
- **Batched linalg.eigh 在 CUDA 上提速最高 100×**：更新 cuSolver 后端选择
- **新 torch.accelerator.Graph API**：统一 CUDA/XPU/第三方后端的图捕获和重放
- **Microscaling (MX) 量化格式导出支持**：torch.export.save 现在支持 MX 量化格式
- **Adagrad fused=True**：加入 Adam/AdamW/SGD 的 single-kernel optimizer 实现
- **C++20 最低要求**：源码构建需要 C++20 编译器
- **CUDA 最低版本提升至 12.6**
- **ROCm**: expandable memory segments, rocSHMEM symmetric memory, FlexAttention pipelining
- **Deprecation**: CUDA 12.8 官方二进制已移除，推荐 CUDA 13.0
- **torch.compile(fullgraph=True)** 新增警告机制：当调用未执行编译代码时发出警告，2.13 将改为报错

## 三、KV Cache 优化最新论文（arXiv 6月第3周）

### 📄 SAC: CXL 分解式 KV Cache（arXiv:2606.19746）

> **SAC: Disaggregated KV Cache System for Sparse Attention LLMs with CXL**

- **核心思路**: 利用 CXL 的低延迟、cache-line 粒度 load/store 语义，替代 RDMA 的粗粒度全缓存预取
- **针对稀疏注意力模型**（如 DeepSeek-V3.2）：只需要按需获取 top-k KV entries
- **在 SGLang 上评估 DeepSeek-V3.2**:
  - **2.1× 更高吞吐量**
  - **9.7× 更低 TTFT（首 token 延迟）**
  - **1.8× 更低 TBT（块间延迟）**
- **结论**: CXL 分解式架构是新兴稀疏注意力模型的基础设施优选方案

### 📄 RKSC: Reasoning-Aware KV Cache Sharing（ICML 2026 Workshop）

> **RKSC: Reasoning-Aware KV Cache Sharing and Confident Early Exit for Multi-Step LLM Inference**

- 训练无关，即插即用
- **3.008× mean speedup over no-KV 基线（峰值 3.990×）**
- **1.66× mean improvement over vLLM 等价 prefix caching**
- CGEE（Confidence-Gated Early Exit）引入误差仅 0.37%（6/1,616 次 verify 调用错误）
- 适用于 multi-branch reasoning pipeline（如 self-consistency / tree-of-thoughts）

### 📄 IntentKV: Cross-Turn Intent-Aware KV Cache Pruning（arXiv:2606.09916）

- 面向多轮 Agent 推理的 token 裁剪
- 在 Qwen2.5-14B 上，最坏场景峰值请求 token 从 **92.3k 降至 20.5k（77.8% 减少）**
- KV reads 从 **411M 降至 31M（92.6% 减少）**
- 基础 LLM 冻结，无需微调

### 📄 Alignment Collapse Under KV Cache Quantization（arXiv:2606.09864）

- **关键发现**: 低比特 KV cache 量化可**静默破坏安全对齐**
  - Mistral-7B: 失去 15.2% 的 refusal 能力，perplexity 仅劣化 1.03×
  - **不存在通用的安全 bit-width**，模型之间存在急剧的相变
- **根因**: 安全特征占据低维激活子空间（10²-10³× 更易被量化噪声破坏）
- **修复**: Per-Channel Reduction (PCR) 诊断方法，使用 20 个校准 prompt 即可预测正确缓解方向
  - **恢复最高 97% 的安全对齐**，仅需约 35 GPU 分钟
  - 在 vLLM 生产环境 FP8 KV cache 上验证确认

### 📄 SwiftCache: Heterogeneous KV Cache Sharing（arXiv:2606.16135）

- 利用服务器内空闲 GPU 显存 + NVLink 做跨模型 KV cache 共享
- **P99 TTFT 降低最高 69%**
- **最大上下文长度扩展最高 3.98×**（对比 vLLM/SGLang）
- 共置模型最低干扰

## 四、交叉趋势总结

| 趋势 | 证据 | 影响判断 |
|:-----|:------|:---------|
| **Speculative Decoding V2 标准化** | SGLang 默认切换 Spec V2，vLLM MRv2+ spec decode 成熟 | 推理框架标配，topk>1 tree drafting 生产就绪 |
| **DeepSeek-V4 成为框架核心驱动** | 三框架均大量投入 DSV4 优化（vLLM 408 commits + SGLang 多 PR + TRTLLM） | DSV4 成为推理框架压力测试标准模型 |

| **CXL 替代 RDMA 做 KV Disagg** | SAC 论文 2.1×/9.7× 压倒性优势 | CXL 3.1 生态成熟后可能成为标准 |
| **安全对齐作为量化评估新维度** | Alignment Collapse 论文揭示现有量化评测盲区 | KV cache 量化方案需重新验证安全影响 |
