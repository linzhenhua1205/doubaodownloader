## 📄 KV Cache 优化论文 (Late June 2026)

本期 arXiv 涌现 **7 篇新 KV Cache 论文**，聚焦不同维度：

### 1️⃣ MemHA: 异构内存加速器 Prefill [arXiv:2606.29986](https://arxiv.org/abs/2606.29986)

**核心思路**: LLM Prefill 阶段是计算密集型, HBM 带宽几乎闲置。MemHA 将 GDDR 加速器(更便宜)用于 Prefill, HBM GPU 仅用于 Decode。
- Pushed to its most economical form: GDDR + PCIe 替代 HBM 做 Prefill
- **意义**: 大幅降低 Prefill 的硬件成本, 适合长上下文场景

### 2️⃣ Coverage-Driven KV Eviction [arXiv:2606.29563](https://arxiv.org/abs/2606.29563)

**核心思路**: 覆盖度驱动的 KV 缓存驱逐策略，替代传统 attention-score 驱逐。
- 提升长上下文推理准确率
- 减少关键 token 被误驱逐的概率

### 3️⃣ CompressKV: 语义检索引导压缩 [arXiv:2606.24467](https://arxiv.org/abs/2606.24467)

**核心思路**: GQA LLM 中区分不同类型 attention head 的功能差异, 针对性地保留关键 token。
- 现有方法对所有 head 使用相同 heuristic 打分 → 重要 token 被驱逐
- 头感知压缩 → 避免关键 token 丢失 → 提升资源受限设备上的推理质量

### 4️⃣ Nexus Sampling: 概率化 KV 驱逐 [arXiv:2606.23961](https://arxiv.org/abs/2606.23961)

**核心思路**: 现有方法全部使用每步 direct-attention score + deterministic top-K。单步低于截断值 = 永久擦除。
- Nexus Sampling 用概率采样替代确定性选择
- **单次低分不意味着永久删除** → 降低误驱逐惩罚

### 5️⃣ HERALD: CPU-GPU 协同 KV 检索 (dLLM) [arXiv:2606.21633](https://arxiv.org/abs/2606.21633)

**核心思路**: Diffusion LLM 的 KV cache 随上下文线性增长。CPU-GPU 协作: KV 卸载到 host DRAM, PCIe 带宽限制下只召回稀疏子集。
- Block dLLM 中: 同一 block 内的去噪步保持 KV entry 一致性 → 检索更高效
- **意义**: 突破 GPU 显存限制, 利用 CPU 内存扩展 KV 容量

### 6️⃣ Concordia: JIT 编译持久化内核检查点 [arXiv:2606.23521](https://arxiv.org/abs/2606.23521)

**核心思路**: 长时间运行的 LLM Agent (KV cache + scheduler + adapters) 在 GPU 故障时丢失状态。
- GPU 级持久化检查点: 无需重新启动整个 serving stack
- 适用于 MLA, 通信状态, online adapter 的联合检查点
- **意义**: 万卡集群故障容忍的关键基础设施

### 7️⃣ Dustin: 草稿增强稀疏验证 (投机解码) [arXiv:2606.24957](https://arxiv.org/abs/2606.24957)

**核心思路**: 长上下文投机解码中, KV cache 加载是瓶颈 (verification bottleneck)。
- 静态驱逐: 准确率损失 (saliency shift)
- 动态选择: 验证路径上计算开销过大
- **Dustin**: draft-augmented sparse verification → 兼顾效率和准确率

### 8️⃣ Cache-Resident LLM (补充) [arXiv:2606.25353](https://arxiv.org/abs/2606.25353)

**核心思路**: 3D-stacked GB-scale LLC 使 CPU 可驻留 LLM 权重。但 deeper pipelining 增加 in-flight requests + KV cache footprint。
- 权重驻留 LLC → 利用缓存带宽和延迟
- Operator 层面有特殊要求: cache-resident 算子需要在 LLC 访问模式下最优

## 📊 综合趋势判断

### 1. 推理框架进入「大版本密集期」
- vLLM v0.24.0 (Jun 29) + TRT-LLM rc20 (Jun 30) + SGLang v0.5.14 (Jun 26) = **三天三版本**
- 三个框架均在 DeepSeek-V4 / MiniMax-M3 / DiffusionGemma 上深度适配
- **NVFP4/MXFP4** 成为 Blackwell 量化事实标准

### 2. TRT-LLM 历史性转折
- **TensorRT 后端将在下个版本移除** — 全面 PyTorch 原生路线
- 这是 NVIDIA 推理框架的「去 TRT 化」战略转折
- 验证了 PyTorch 生态的不可逆胜利

### 3. MoE 负载均衡进入「算法博弈」阶段
- vLLM: DeepEP v2
- SGLang: Waterfill + LPLB (线性规划)
- TRT-LLM: GVR-TopK + EPLB
- 方法从 heuristic → 可优化/线性规划演化

### 4. KV Cache 创新持续加速
- 本周 8 篇新论文覆盖: 异构内存(GDDR+HBM) / 概率驱逐 / 头感知压缩 / CPU-GPU 协作 / 故障检查点 / 投机解码验证
- 关注焦点从「压缩率」转向「系统集成」— 如何与不同硬件/架构配合

### 5. Diffusion LLM 进入推理框架视野
- vLLM: DiffusionGemma
- SGLang: DiffusionGemma
- TRT-LLM: LTX-2, WAN, Cosmos3
- 扩散 + 自回归融合推理成为新趋势
