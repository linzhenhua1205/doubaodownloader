## 1️⃣ Helion 内核集成 vLLM — PyTorch 原生 Kernel DSL 进入生产推理 🚀

- **来源**: [PyTorch Blog](https://pytorch.org/blog/portable-vllm-model-inference-kernels-in-helion/) (June 10, 2026)
- **作者**: Sean Chen (Red Hat), Yanan Cao (PyTorch, Meta Platforms)
- **摘要**: Helion（PyTorch 原生的硬件无关 Kernel DSL）被正式集成到 vLLM 中，替代 FP8 推理前向通路的全部非 GEMM 内核

### 性能数据（H100 + B200）

| Kernel | vs torch.compile (H100) | vs torch.compile (B200) | vs torch.ops._C (H100) | vs torch.ops._C (B200) |
|:-------|:------------------------|:------------------------|:----------------------|:----------------------|
| rms_norm_dynamic_per_token_quant | **1.180×** | **1.240×** | **1.802×** | **1.969×** |
| silu_and_mul_dynamic_per_token_quant | **1.256×** | **1.420×** | N/A | N/A |
| fused_qk_norm_rope | **1.383×** | **1.133×** | **1.204×** | **1.155×** |
| rms_norm_per_block_quant | **1.674×** | **1.424×** | **2.055×** | **2.128×** |
| silu_and_mul_per_block_quant | **1.731×** | **1.483×** | **2.269×** | **2.325×** |
| dynamic_per_token_scaled_fp8_quant | **1.237×** | **1.311×** | **1.405×** | **1.495×** |

### 端到端吞吐（H100，per-token 量化 + EAGLE3 推测解码）

- Qwen3-8B (batch=16, spec=1): **1.09×** 总吞吐提升
- Qwen3-8B (batch=16, spec=3): **1.06×** 总吞吐，**1.33× TTFT** 加速 🔥
- Qwen3-32B (batch=8, spec=2): **1.08×** 总吞吐，**1.23× TTFT** 加速
- B200 仅启用量化核群组即有 **~1.05×** 吞吐提升（因 Blackwell Triton GEMM 尚未成熟）

### 关键限制与后续
- **GEMM 内核在 B200 上落后 CUTLASS**（scaled_mm: 0.739×, scaled_mm_blockwise: 0.782×）— Triton 对 Blackwell GEMM 代码生成是瓶颈
- Helion 团队正在推进 **CuteDSL 后端**以解决 Blackwell GEMM
- 自动调优耗时仍高（168 种形状全量调优需 ~1 天），LLM 引导调优已在探索中
- 每个内核启动增加数十 μs CPU 开销，**CUDA Graph 模式必须开启**

### 影响判断
Helion 作为 **PyTorch 原生 Kernel DSL** 已证明在非 GEMM 算子上有 1.2-2.3× 的显著优势。结合 AOT 自动调优，开发效率极高（多数内核 1 天内完成实现+验证）。未来 CuteDSL 后端补齐 GEMM 短板后，有望成为 vLLM 内核开发的默认选择。

## 2️⃣ Ada-MK: 自适应 MegaKernel 首次部署 TensorRT-LLM 商业化广告系统 🏭

- **来源**: [arXiv:2605.11581](https://arxiv.org/abs/2605.11581) (May 12, 2026)
- **作者**: Wenxin Dong 等（阿里巴巴）
- **摘要**: 提出 Ada-MK，基于 MLIR 的细粒度 DAG 编译时搜索，将 MegaKernel 的运行时动态分支完全提升到编译时确定，消除分支开销

### 核心创新
1. **三维共享内存约束模型 + K-维度分割**：峰值共享内存使用降低 **50%**
2. **MLIR DAG 离线搜索**：将最优执行路径固化到编译时，**完全消除运行时分支**
3. **异构推理引擎**：MegaKernel 作为 TensorRT-LLM 插件嵌入，High Throughput Prefill + Low Latency Decode

### 性能数据（NVIDIA L20）
| 对比对象 | 单批吞吐提升 |
|:---------|:------------|
| vs vanilla TensorRT-LLM | **+23.6%** 🔥 |
| vs vLLM | **+50.2%** 🔥 |
| 测试场景 | 所有场景均为正增益（首次工业化验证） |

### 背景
- Decode 阶段 kernel launch 开销占端到端推理 **14.6%**
- 现有 MegaKernel 在 portability 与 efficiency 之间存在根本张力
- 关键洞察：在固定部署配置下，MegaKernel 最优执行路径**唯一确定**，运行时决策可全部提升到编译时

### 影响判断
首次 MegaKernel 在**商业化在线广告系统**中工业化部署的验证。TensorRT-LLM 生态中重要的算子融合里程碑。+50.2% vs vLLM 的对比数据极具说服力。

## 3️⃣ Blink: 完全移除 CPU 的 LLM 推理架构 — SmartNIC + GPU 直接推理 🖥️❌

- **来源**: [arXiv:2604.07609](https://arxiv.org/abs/2604.07609) (April 8, 2026)
- **作者**: Mohammad Siavashi 等 (KTH Royal Institute of Technology)
- **摘要**: 提出 Blink 端到端推理架构，将 Host CPU 完全从稳态推理路径中移除，用 SmartNIC (RDMA 直写 GPU 显存) + GPU 持久 Kernel (自行完成调度和 KV cache 管理) 替代

### 核心架构
1. **SmartNIC** 处理请求接收 → 通过 **RDMA** 直接写入 GPU 显存
2. **GPU 持久 Kernel** 替代 CPU 调度器，自行完成 Batching + Scheduling + KV cache 管理
3. **零 CPU 参与稳态推理**

### 性能数据
| 指标 | 提升幅度 | 对比对象 |
|:-----|:---------|:---------|
| P99 TTFT | **8.47×** 🔥🔥 | TensorRT-LLM / vLLM / SGLang |
| P99 TPOT | **3.40×** 🔥 | 同上 |
| Decode 吞吐 | **2.1×** | 同上 |
| 单 token 能耗 | **-48.6%** | 同上 |
| CPU 干扰下性能 | **稳定维持** | 现存系统退化最多 **两个数量级** |

### 影响判断
虽为架构研究（非生产可用），但 **8.47× TTFT** 的数据极具震撼力。指出了当前推理栈的根本性瓶颈：CPU 在推理路径中不可被替代的角色。对未来推理框架架构设计有重大启示——特别是为 **CPU-Free Inference Server** 提供了可行性验证。
