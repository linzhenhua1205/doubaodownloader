## 1️⃣ RTP-LLM：阿里巴巴工业级推理引擎开源（2026-05-28）

> **来源**: [arXiv:2605.29639](https://arxiv.org/abs/2605.29639) | Alibaba Group | 28位作者 | Open Source

### 概述

RTP-LLM 是阿里巴巴集团内部大规模部署的推理引擎，服务**超过1亿用户**。论文首次公开了其全栈设计，并与 vLLM/SGLang 进行了系统对比。

### 核心架构

| 特性 | 说明 |
|:-----|:------|
| **模型加载** | 文件顺序驱动 I/O + I/O-通信流水线重叠 |
| **PD 分离** | Prefill-Decode Disaggregation，解耦计算密集型与内存瓶颈阶段 |
| **KV Cache 管理** | 分层多级缓存，生产流量调度中 **215% 缓存复用提升** |
| **推测解码** | 模块化设计，支持多种算法 |
| **量化推理** | 自适应 KV Cache 量化 |
| **多模态** | 解耦式多模态处理 |

### 性能数据（vs vLLM / SGLang）

| 指标 | 提升幅度 | 模型规模 |
|:-----|:--------:|:---------|
| **模型加载速度** | **4.7× - 6.3×** 🏆 | 8B-235B |
| **TTFT P95 延迟** | **-35% ~ -37%** | 生产流量 |
| **缓存复用率** | **+215%** | 生产流量调度 |
| **推测解码吞吐** | **1.12× - 2.48×** | 全模型范围 |
| **多模态推理吞吐** | **1.86× - 2.52×** | 全模型范围 |
| **批处理量化推理延迟** | **-35% ~ -40%** | batch 场景 |
| **量化 TTFT** | **1.9× - 3.0× 提升** | 量化推理 |

> **意义**: RTP-LLM 是继 vLLM、SGLang、TensorRT-LLM 之后的又一重量级工业推理引擎。论文提供了目前最全面的生产级推理对比数据，PD 分离和 KV Cache 管理的设计思路与业界趋势一致。开源可获取。

## 2️⃣ Albireo：突破 Amdahl 定律的并行推理系统（2026-06-01）

> **来源**: [arXiv:2606.01927](https://arxiv.org/abs/2606.01927) | Alan Zhao, Cyril Y. He, Wei Xu | University of Chicago

### 核心洞察

Tensor 并行（TP）是 LLM 推理的标配，但随着 TP 度 t 增大，跨 GPU 通信和不可扩展的运行时开销使扩展呈次线性——这正是 Amdahl 定律预测的现象。然而，增大 t 又能提升显存效率、减轻 KV Cache 争用和交换。Albireo 的核心贡献是**识别并验证了经验最优 TP 度 t_e**，并通过消除不可扩展部分来提升 t_e 的可达范围。

### 技术方案

- **调度与计算重叠**: 将不可扩展的调度开销与计算完全重叠
- **I/O 与计算重叠**: 类似地重叠 I/O 操作
- **序列并行采样**: 分布式采样阶段与计算并行执行
- **无需模型架构修改**: 纯运行时优化

### 性能数据（vs vLLM）

| 指标 | 最佳值 |
|:-----|:------:|
| **吞吐量提升** | **up to 1.9×** 🏆 |
| **延迟降低** | **up to 48%** |
| **GPU 利用率提升** | **+28%** |
| **能耗降低** | **-54%** |
| **生产环境吞吐提升** | **up to 2.0×** |

> **意义**: Albireo 通过纯运行时优化（无需修改模型）即可实现接近 2x 的吞吐提升。它将 Amdahl 定律这一"硬限制"转化为可优化的工程问题，对在线推理服务的 TP 度选择提供了理论指导和工程方案。

## 3️⃣ KForge：LLM 驱动的跨平台 Kernel 生成框架（2026-06-01，ISCA'26 Workshop）

> **来源**: [arXiv:2606.02963](https://arxiv.org/abs/2606.02963) | Taras Sereda 等 | ISCA 2026 Workshop MLArchSys

### 核心设计

KForge 是一个基于双 Agent 迭代优化循环的跨平台 Kernel 生成框架：

1. **生成 Agent**: 根据编译反馈和正确性验证，逐步改进 Kernel 实现
2. **性能分析 Agent**: 解读性能分析数据（包括 GUI 工具输出），生成优化建议指导下一轮生成
3. **双阶段循环**: 功能优化（Functional Passes → 正确性）→ 性能优化（Optimization Passes → 性能差距）

### 性能数据

| 平台 | 场景 | 性能提升 |
|:-----|:-----|:--------|
| **NVIDIA B200** | gpt-oss-20b 推理（vs TensorRT-LLM） | **+2.12%** 端到端吞吐 🏆 |
| **Intel Arc B580** | 37 个 GEMM + tail-ops（KernelBench L2） | **5.13× 几何平均加速** vs PyTorch eager/torch.compile 🏆 |
| Intel Arc B580 | 主要优化手段 | 算子融合 + 混合精度执行 |

> **意义**: KForge 展示了 LLM Agent 在 Kernel 生成领域的又一突破——在一个框架内同时支持 NVIDIA（高质量基准可参考）和 Intel（几乎无参考实现）两种完全不同的平台。其在 Intel Arc 上 5.13× 的加速比表明，LLM 驱动的方法在**缺乏手写 Kernel 的长尾硬件**上尤为有效。

## 4️⃣ RaMP：运行时感知的 MoE Megakernel 多态调度（2026-04-28）

> **来源**: [arXiv:2604.26039](https://arxiv.org/abs/2604.26039) | Vyom Sharma, Debajyoti Datta | Preprint

### 核心洞察

MoE 推理的最佳 Kernel 配置不仅取决于 batch size，还取决于 **Expert 路由分布**。但生产系统仅根据 batch size 分派，**10-70% 的 Kernel 吞吐未被利用**。

### 技术方案

| 组件 | 说明 |
|:-----|:------|
| **性能区域分析** | 仅从硬件常数推导出每种优化何时有效，正确预测全部 8 种架构（含 3 种未见过的） |
| **四参数 Wave 成本模型** | 从运行时 Expert 直方图选择最快配置，**0.93% 平均遗憾率** vs 穷举搜索 |
| **一次分析时间** | 每个模型仅需 **10-24 分钟**一次性 profiling |
| **CuTe DSL 协同设计** | 暴露 **134-268 个多态配置**的 Kernel |

### 性能数据（vLLM 端到端）

| 对比对象 | 加速比 |
|:---------|:------:|
| **vs Triton（vLLM 默认）** | **1.30×** 🏆 |
| **vs DeepGEMM** | **1.41×** 🏆 |
| vs FlashInfer CUTLASS | 1.13× |
| vs 静态分派（kernel 级） | 1.22× |
| Alpha-MoE（零代码修改） | 1.14× |

> **意义**: RaMP 切中了 MoE 推理的一个被忽视的关键瓶颈——路由分布对 Kernel 配置的影响。10-70% 的性能损失意味着大多数 MoE 推理系统都存在显著的优化空间。其 Kernel 无关的设计（仅依赖 CTA grid 几何）使其可广泛集成。

## 5️⃣ NVIDIA CompileIQ：CUDA 13.3 中的 AI 驱动编译器自动调优（2026-05-26）

> **来源**: [NVIDIA 技术博客](https://developer.nvidia.com/blog/extract-more-kernel-performance-with-nvidia-compileiq-auto-tuning/) | CUDA 13.3 | `pip install compileiq`

### 核心创新

CompileIQ 是 NVIDIA 在 CUDA 13.3 中首次发布的 AI 驱动编译器自动调优框架，使用**进化算法和遗传算法**优化 GPU 编译器参数。

| 特性 | 说明 |
|:-----|:------|
| **搜索空间** | PTXAS 和 NVCC 的**内部编译器参数**（寄存器分配、指令调度、循环展开等），之前未通过任何公共 flag 暴露 |
| **输出形式** | Advanced Controls File (ACF)，通过 `--apply-controls` 注入编译器 |
| **多目标优化** | 同时优化运行时 / 编译时间 / 功耗，输出 Pareto 前沿 |
| **安全性** | 用户代码不离开本地环境，ACF 可安全提交到版本控制 |

### 性能数据

| 场景 | 提升 |
|:-----|:----:|
| **Meta 生产验证** | TritonBench + **Helion kernels 最高 15%** 提升 🏆 |
| CUDA reduction kernel | ~1% 提升（已经优化过的基线） |
| 输出可复现性 | 相同 ACF + 相同编译器 → 相同二进制 |

> **意义**: CompileIQ 将编译器本身变成了一个"可调参数"，为已经手工优化到极致的 Kernel 提供了新的优化维度。与 LLM Agent 驱动 Kernel 优化（KForge/DRTriton）不同，CompileIQ 是在**编译器后端**层面调优，两者正交叠加可能产生更大收益。

## 6️⃣ SiDP：离线 LLM 推理的高效数据并行范式（2026-05-27）

> **来源**: [arXiv:2605.28095](https://arxiv.org/abs/2605.28095) | Alan Zhao, Cyril Y. He

### 核心创新

SiDP 提出了一种新的数据并行（DP）范型——将模型权重视为 DP 组内**带宽保障的共享资源**，而非每 GPU 冗余存储：

| 模式 | 适用场景 | 机制 |
|:-----|:---------|:-----|
| **Weight-as-a-Service (WaS)** | 大 batch 场景 | 通过 NVLink 流式拉取远程权重到小缓存 |
| **Compute-as-a-Service (CaS)** | 小 batch 场景 | 将激活值发送到权重所属 GPU |

### 性能数据（vs vLLM）

| 指标 | 提升 |
|:-----|:----:|
| **可用 KV 容量** | **up to 1.8×** 🏆（H20/H200/B200） |
| **端到端吞吐** | **up to 1.5×** 🏆（离线场景） |
| 测试模型 | Qwen3-32B / Qwen2.5-72B / Llama-3.1-70B |

> **意义**: SiDP 将"以存储换计算"的 DP 思路翻转成"以带宽换显存"，利用 NVLink 的高带宽（H200/B200 上 900 GB/s+）实现接近无损的性能。对离线批量推理场景（如 数据标注、批量生成）尤其适用。
