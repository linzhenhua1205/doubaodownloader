## 1️⃣ 🌐 bmcweb: HTTP/2 连接修复 Client IP 传递问题 — 审计与安全关键修复

**Vijaysankar Ravi (AMI) · `c91086487c` · authored 6/11, merged 6/15-6/16**

- **问题**: HTTP/2 连接未将解析到的客户端 IP 传入 `req->ipAddress`，导致 SessionService API 中 `ClientOriginIPAddress` 始终显示 `0.0.0.0`。同时，`authenticate()` 被调用时传入空 IP，使得 Basic Auth 创建的 session 中 `clientIp` 字段错误
- **修复路径**: 
  - HTTP2Connection 构造函数中传入 client IP
  - `onRequestRecv` 时赋值 `req->ipAddress`
  - `authenticate()` 调用传入真实 IP
- **技术深度**: Redfish Session 审计链路中的关键一环——无真实 IP 意味着无法追溯谁在何时通过 HTTP/2 执行了什么操作，**是安全审计与合规的基础设施级修复**
- **测试结果**: 
  - Before: `ClientOriginIPAddress: "0.0.0.0"`
  - After: `ClientOriginIPAddress: "10.0.136.165"`
  - Redfish Service Validator v3.1.4: PASS 44, WARN 0, FAIL 0
- **修复链接**: https://github.com/openbmc/bmcweb/commit/c91086487c

## 3️⃣ 🔌 meta-facebook: yosemite5 power-cmd 信号安全重构 — 防 GPIO 锁定

**Cosmo Chou (Quanta) · `e04d90` · authored 6/7, merged 6/15**

- **问题**: 原有的电源控制函数 `open-code set_gpio/sleep/set_gpio` —— 如果进程在 `sleep()` 期间被 SIGINT/SIGTERM 杀死，GPIO 会保持 assertion 状态，导致电源按键物理上被持续按住，直到 BMC 复位
- **修复**: 新增 `gpio_pulse()` 函数，内置 `SIGINT/SIGTERM` 信号陷阱，在中断时 **保证 deassert GPIO**
- **额外改进**: 重构回调函数，增加 `force_power_off` 验证，移除冗余检查
- **⚠️ 严重性**: 产线级可靠性 Bug —— 在 BMC 进程被异常终止的场景下，不加这个修复可能导致服务器非预期的持续硬关机
- **修复链接**: https://github.com/openbmc/openbmc/commit/e04d9031ddbf56a8c41d8b4ac07e8956a17bfc75

## 4️⃣ 🖥️ webui-vue: 密码修改确认弹窗 + 网络接口 Tab 选择保持

**Aravinth S (AMI) · 2 commits · merged 6/16**

### 4a. 密码修改确认弹窗与自动登出（`f18e4f29a2`）
- 用户在 WebUI 修改自己的密码时，弹窗提示「该用户将被登出，需要使用新密码重新登录」
- 确认后执行 PATCH → 成功后清除前端认证状态并跳转至登录页
- 管理员修改**他人**密码时不触发此弹窗（保持了权限管理的合理性）
- **意义**: 解决了密码变更后前端 session 与后端状态不一致的安全隐患

### 4b. 网络接口 Tab 选择在数据刷新时重置修复（`aa7b42222d`）
- 每次 save 操作（如添加 IPv4）后 `getEthernetData()` 无条件将 `selectedInterfaceId` 重置为 eth0
- 修复: 只在首次加载且 `selectedInterfaceId` 为空时才设置默认值
- 引用 Issue #125
- **意义**: 用户在多网口 BMC 上切换 Tab 配置网络时不会再被自动跳回 eth0
