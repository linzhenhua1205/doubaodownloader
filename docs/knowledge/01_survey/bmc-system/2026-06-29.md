## 1️⃣ 🐛 bmcweb: account_service PropertyValueNotInList 参数顺序错误修复

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/bmcweb` |
| **提交** | `a7f0d0a7` (merged 2026-06-25) |
| **作者** | Chandramohan Harkude |
| **标题** | `account_service: fix swapped args in PropertyValueNotInList errors` |

**问题描述**: `redfish-core/lib/account_service.hpp` 中有 **3 处调用点**将 `PropertyValueNotInList` 消息的参数名（property name）和期望值列表（expected values）传反了：

| 调用点 | 正确语义 | 实际传递（错误） |
|:-------|:---------|:----------------|
| `RoleId` 校验 | `RoleId` 值不在有效角色列表中 | 有效角色列表被当作属性名，`RoleId` 被当作值列表 |
| `Password` 校验 | 密码不满足复杂度规则 | 同理参数颠倒 |
| `UserName` 校验 | 用户名格式不合法 | 同理参数颠倒 |

**影响**: 当 Redfish 客户端发送非法 `RoleId` 时，本应返回：
```
The value "admin" is not in the list of 'Administrator, Operator, ReadOnly'
```
实际返回：
```
The value "Administrator, Operator, ReadOnly" is not in the list of 'admin'
```
— **属性名和期望值被互换**，错误消息完全错乱，误导客户端调试。

**修复**: 调整 Redfish 消息 API 调用的参数顺序，使 `arg1` = 属性名、`arg2` = 期望值列表。

**重要性**: 🟡 **中** — 纯错误消息正确性修复，不涉及功能行为变更（校验逻辑本身正确），但直接影响客户端和运维人员的调试体验，以及 RSV 合规预期的一致性。

## 2️⃣ 🧪 bmcweb: TelemetryService handleTelemetryServiceGet 单元测试新增

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/bmcweb` |
| **提交** | `3b9e3d1c` (merged 2026-06-25) |
| **作者** | Ramya Sivakumar |
| **标题** | `Add unit test for TelemetryService` |

**变更内容**: 新增针对 `handleTelemetryServiceGet` 函数的单元测试用例，验证 Redfish TelemetryService 端点的**静态属性**（包括 `@odata.type`、`Id`、`Name`、`MinCollectionInterval`、`SupportedFunctions` 等）。

**测试范围**: `redfish-core/lib/telemetry_service.hpp` 中的 GET 处理逻辑。

**意义**: Telemetry Service 是 Redfish 规范中用于**遥测数据流订阅**的核心功能（DSP0266 §14），此前的测试覆盖度报告显示该路径的测试用例不完整。新增用例填补了这一空白，提升了对 TelemetryService GET 响应的回归保护。

**重要性**: 🟢 **低** — 单元测试增强，无功能变更。但 TelemetryService 在 AI 集群和超大规模部署中用于实时功耗/温度/利用率的流式监控（非轮询），其正确性对集群能效管理（TDP/Redfish Telemetry 集成）有间接影响。
