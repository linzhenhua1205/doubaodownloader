## ① MCTP over USB (DSP0283) — Intel提出OpenBMC集成状态质询

**来源**: [OpenBMC GitHub Issue #4075](https://github.com/openbmc/openbmc/issues/4075) — 2026-06-05  Mailing List
**提出者**: Song, MS (Intel)

**核心问题**：
- MCTP over USB (DSP0283) 是 DMTF 标准定义的管理通信协议在 USB 上的传输绑定
- Intel 正式提出在 OpenBMC 中的集成状态质询，要求明确：
  - Linux 内核侧 MCTP over USB 驱动的支持状态
  - OpenBMC 用户态实现（phosphor-mctp）是否已支持 USB 传输绑定
  - 集成路线图和时间表

**工程意义**：
- Intel 的主动质询意味着 **新平台要求 MCTP over USB 作为标准带外管理通道之一**
- 相比 I2C/PCIe 物理层，USB 作为 MCTP 传输层更灵活、更通用
- 但对 OpenBMC 社区来说，这是**还需要推动的工作**，尚未集成完毕（与邮件列表中 6/5 的回复呼应）

## ② phosphor-logging JSON 序列化完整落地 — 日志子系统架构转型

**来源**: [phosphor-logging GitHub Commits](https://github.com/openbmc/phosphor-logging/commits/master)
**贡献者**: Archit Mehrotra, williamspatrick, Manish Tiwari

**关键提交链（4月-5月）**：

| 日期 | 提交 | 内容 |
|:----|:-----|:------|
| 2026-04-24 | `Add JSON serialization for log entries` | ⭐ **首次引入** JSON 序列化能力 |
| 2026-05-12 | `log-manager: use JSON-specific directory` | 日志文件按 JSON 格式独立存储目录 |
| 2026-05-13 | `log-manager: restore JSON files and paths` | 支持从 JSON 文件恢复日志状态 |
| 2026-05-13 | `log-manager: deserialize from JSON` | **JSON 反序列化功能完成** |
| 2026-05-29 | `CLI11: simplify CLI11.wrap` + `CLI11: disable tests` | CLI 工具简化，减少编译负担 |

**版本范围**：提交范围 `ad67076c` → `e7df38325f`（已通过 srcrev bump 合入主分支）

**工程意义**：
- **架构转型**：phosphor-logging 从原有的 PEL（Platform Event Log）专用二进制格式，**正式转向 JSON 作为主要持久化序列化格式**
- **与 Redfish 对齐**：JSON 格式原生对齐 Redfish Event Log 的 Schema（`LogEntry.v1_*`），减少了 BMC WebUI/Redfish 服务的转换开销
- **互操作性**：JSON 格式便于外部工具和 AIOps 平台直接解析 BMC 日志
- **配合 Redfish 2026.1**：bmcweb 已于 5月19日 (df2c4aa) 升级到 Redfish 2026.1 Schema，日志格式对齐后可实现端到端的 Redfish 日志标准化

## ③ bmcweb Redfish 合规修复 + 核心依赖库老化问题

**来源**: [bmcweb GitHub Commits](https://github.com/openbmc/bmcweb/commits/master)
**贡献者**: jnin-dev, gtmills, liuer-sys (NVIDIA), edtanous, edtanous (Google)

**近期修复亮点（5月19日-6月4日）**：

| 日期 | 提交 | 内容 | 类型 |
|:----|:-----|:-----|:-----|
| 2026-05-19 | `Move to Redfish 2026.1` | 🆕 **bmcweb 升级到 Redfish 2026.1 Schema** | 规范升级 |
| 2026-06-01 | `Fix: POST Chassis.Reset for invalid chassis now returns 404` | 修复无效 Chassis Reset 返回 **200 OK** 的合规 BUG | Redfish合规 |
| 2026-06-04 | `PCIE: add DeviceType to PCIeDevice` | 向 Redfish PCIeDevice 暴露设备类型（GPU/NIC/SSD） | 功能增强 |
| 2026-06-01 | `http: zstd: do not attempt compressing already opened files` | Zstd 压缩逻辑修复 | 稳定性 |

**⚠️ 核心库依赖老化 — 需关注的问题**：

根据近期问题讨论和提交分析，bmcweb 对 **openssl** 等核心库的依赖严重落后：
- bmcweb 当前依赖 openssl **3.0.8**（2023年发布）
- 当前最新版本为 **3.5.6**（2026年5月）
- **版本差 > 3 年**，意味着 TLS 安全特性（包括新密码套件、证书验证增强等）无法使用

**工程意义**：
- `POST Chassis.Reset` 返回 200 OK 的 BUG 修复周期超过 **20 个月**（从首次报告到 6/1 修复），暴露了 Redfish 合规验证流程的短板
- PCIeDevice DeviceType 映射由 NVIDIA 的 Ed Tanous 推动，是 GPU/NIC 设备管理的基础功能
- 核心库老化是 bmcweb 长期存在的技术债，可能影响新平台（如 Intel Xeon 6+ eSPI、NVIDIA NVL32）的 TLS 安全对接
