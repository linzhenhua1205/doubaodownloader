## 1️⃣ 🔧 phosphor-host-ipmid: corrupt user JSON 导致 IPMI crash loop 修复 🆕

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/phosphor-host-ipmid` |
| **提交** | `90210075` (merged 2026-06-22) |
| **作者** | Nagavarapu Sai Pavan |
| **标题** | `phosphor-host-ipmid: Fix ipmid crash loop on corrupt user JSON` |

**问题描述**: 当持久化存储 `/var/lib/ipmi/ipmi_user.json` 文件损坏（corrupt）时，`readUserData()` 会抛出 `std::runtime_error`，但 `cacheUserDataFile()` 的 catch 块仅捕获 `std::ios_base::failure`，导致异常向上传播未捕获，ipmid 进程以 SIGABRT 终止。systemd 自动重启后陷入**崩溃重启循环（crash loop）**，直至 retry 计数耗尽，整个 IPMI 栈完全不可用。

**次级问题**: `Json::parse()` 使用了 `allow_exceptions=false` 但未检查 `is_discarded()`，损坏的 JSON 会静默通过解析检查，在后续 `size check` 处产生误导性错误信息 `"User count issues"`，而非指向实际解析失败根因。

**修复内容**:
- `readUserData()` 的 catch 块增加 `std::runtime_error` + `std::exception` 兜底捕获
- 在 JSON 解析后增加 `is_discarded()` 检查，明确报告 `"Failed to parse the json file"`
- 日志输出文件名和行号，方便定位

**重要性**: 🔥 **高** — 直接影响 IPMI 带外管理链路的可靠性。corrupt JSON 在固件升级中断或文件系统异常后真实可能发生，修复后即使损坏也能正常回退而不导致整个 IPMI 栈崩溃。

## 2️⃣ 🐛 bmcweb: OEM schema Chassis 属性指向无效底盘链接修复 🆕

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/bmcweb` |
| **提交** | `1bda4f73` (committed 2026-06-26) |
| **作者** | Alexander Hansen |
| **标题** | `fix 'Chassis' property in oem schema` |

**问题描述**: `redfish-core/schema/oem/openbmc/` 下的 OEM schema 为 Fan Zones 定义了 `Chassis` 属性，但后端实现生成的 chassis 链接指向无效路径。典型配置（单节点 2U 服务器 + 3 风扇 + PID 控制器，如 Tyan S8030 主板）下，FanZone 返回的 `@odata.id` 指向 `.../Managers/bmc#/Oem/OpenBmc/Fan/FanZones` 而不是有效 chassis 资源。

**影响范围**: `redfish-oem-manager-fan-data=enabled`（默认启用）的配置。

**修复**: 修正 OEM schema 中 chassis 属性的链接生成逻辑，使其指向正确的 chassis 端点。

**重要性**: 🟡 **中** — Redfish OEM schema 合规性修复，影响使用 OEM Fan Zone 功能的设备的散热管理监控面。
