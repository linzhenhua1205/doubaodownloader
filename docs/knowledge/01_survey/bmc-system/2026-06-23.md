## 1️⃣ 🏗️ phosphor-state-manager: 服务守护进程迁移至 /usr/libexec/ — FHS 规范合规（重大变更）

**Xavier Thompson · `0a308219f310` · authored 5/28, **merged to master 2026-06-22** (Andrew Geissler)**

- **变更**: 将状态管理守护进程（host-state-manager、chassis-state-manager 等）从 `/usr/bin/` 迁移至 `/usr/libexec/`，遵循 FHS（Filesystem Hierarchy Standard）规范
- **设计原则**: 系统守护进程（非面向用户直接执行）应归属于 `/usr/libexec/`，而非 `/usr/bin/`。这是 FHS 长期推荐但 OpenBMC 此前未严格执行的准则
- **影响范围**: 涉及 service 文件中的 `ExecStart=` 路径、systemd unit 依赖路径、以及可能的自定义脚本引用
- **⚠️ 回滚兼容性**: 引用旧路径（`/usr/bin/`）的自定义脚本或集成代码需要同步更新
- **意义**: 这是 OpenBMC 向 FHS 合规迈出的重要一步，与 Linux 发行版的标准化实践对齐。可能预示着 OpenBMC 在长期方向上的「发行版化」趋势
- **🔗 链接**: https://github.com/openbmc/phosphor-state-manager/commit/0a308219f310

## 2️⃣ 🔒 phosphor-user-manager: 拒绝空权限创建用户 — 安全加固

**Vijaysankar Ravi (AMI) · 2 commits · merged to meta-layer 6/22 (Andrew Geissler)**

- **核心变更**: `user_mgr: reject empty privilege` — 当通过 D-Bus API 创建用户时，如果传入的 `Privilege` 属性为空字符串，系统将拒绝创建并返回错误，而非静默创建无权限用户
- **安全影响**: 空权限用户虽然表面上无法操作，但可能绕过某些基于「用户存在即可」的访问控制检查，属于「访问控制绕过」类安全加固
- **同步修复**: 同一批次还修复了 cppcheck 报告的 4 类代码质量问题（变量遮蔽/作用域优化等）
- **srcrev**: `8d84de262a..e5d0f30503`

## 3️⃣ 🐛 webui-vue: Sensors 页面空白 Bug 修复

**Grégoire Layet (9elements) · `bfad598fcb` · merged 6/18 (Nikhil Ashoka, IBM), pulled into meta-layer 6/22**

- **Bug 现象**: WebUI 的 Sensors 页面在点击后显示为**完全空白**，所有主机传感器列表不可见
- **根因定位**: 2026-06-09 的 Power Page 重构提交（`99099183438d8`，VueQuery + Composition API 迁移）在 `discoverParentsWithSubResource` 函数中给 parent collection 查询（`/redfish/v1/Chassis`）错误地附加了 `$select=Sensors` 参数。由于 Chassis Collection 本身不是 Sensor 资源，select 过滤后返回空结果集
- **修复**: 移除 `parentCollectionPath` 查询上的 `$select` 参数，保留子资源查询的 select 过滤
- **严重性**: 严重功能缺陷——Sensors 页面是 BMC WebUI 最高频访问页面之一，此 Bug 导致整个页面不可用约 9 天
- **影响范围**: `useAllSubResources` -> `discoverParentsWithSubResource` 路径唯一的使用方
- **🔗 链接**: https://github.com/openbmc/webui-vue/commit/bfad598fcb7b55178a9d1a0dbf60cdcfbcefd37f

## 4️⃣ 🔌 PLDM: fw_update target specific startUpdate 实现 — 目标特定固件更新路由

**Unive Tien (Wiwynn) · `f759a5a49d` · authored 2025-10-30, merged to pldm master 2026-06-15**

- **背景**: OpenBMC 引入了 `FirmwareInventory`/`FirmwareInventoryManager` 类管理固件更新 D-Bus 接口，但 `startUpdate` 方法尚未针对多目标（multi-target）场景实现
- **变更**: 实现 `ItemUpdateManager` + `AggregateUpdateManager` 组件：
  - **`ItemUpdateManager`**: 管理单个固件目标的更新会话
  - **`AggregateUpdateManager`**: 将 `StartUpdate` 命令路由到特定目标的更新会话
- **测试验证**（Yosemite4 平台, 全部通过）:
  ✅ FD item 固件更新（通过 D-Bus `StartUpdate` 手动触发）
  ✅ FD item 固件更新（通过 Redfish API 触发）
  ✅ 传统基于 inotify 的固件更新
  ✅ Multipart image 固件更新
  ✅ 失败重试场景的固件更新
- **设计参考**: OpenBMC docs `code-update.md` [1] 中定义的 target specific update flow
- **意义**: 这是 PLDM 在多目标固件更新场景下的关键基础设施建设，尤其适用于包含多个可更新固件组件的复杂平台（如 Yosemite4）
- **🔗 链接**: https://github.com/openbmc/pldm/commit/f759a5a49d3d1e7b2a64381188705ff605e5bf3a
