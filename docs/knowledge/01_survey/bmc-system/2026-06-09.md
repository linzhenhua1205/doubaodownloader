## 1️⃣ phosphor-state-manager multi-chassis SMP 完整投产 IBM huygens

**来源**: OpenBMC GitHub — phosphor-state-manager + meta-ibm

**技术详情**: geissonator 在 5月底至6月初密集提交了 multi-chassis SMP 完整系列，6月5日正式在 **IBM huygens** 平台使能：

- **multi-chassis-smp aggregation** — 多机箱 SMP 聚合器类实现，管理多个 chassis 的状态协调
- **SMP transitional state 保序** — 保留 SMP 过渡状态，避免状态跳变丢失
- **power off on failure** — 任一 chassis 目标启动失败时，关闭所有 chassis，保证一致性
- **absent chassis skip** — 跳过不存在的 SMP chassis，实现弹性部署
- **verbose chassis reporting to obmcutil** — obmcutil 工具输出更详细的 chassis 状态报告
- **direct start handling** — 处理 chassis 目标的直接启动路径
- **unit tests** — 完整单元测试覆盖
- **PACKAGECONFIG 条件编译** — `multi-chassis-smp` 作为可选 feature，`meson: Always define NUM_CHASSIS_SMP` 确保编译期常量一致性

## 2️⃣ bmcweb 合规加固：Redfish 2026.1 迁移 + Chassis Reset 404 修复 + PCIeDeviceType 扩展

**来源**: OpenBMC GitHub — bmcweb

**技术详情**: bmcweb 在 5月底至6月初完成多项重要更新：

### 2.1 Redfish 2026.1 Schema 迁移（May 19）
- **Commit**: `df2c4aa` (gtmills) — "Move to Redfish 2026.1"
- bmcweb 的 Redfish schema 绑定从之前版本升级到 **Redfish 2026.1**，同步 DMTF 最新的标准定义
- 配合 `update_schemas` 脚本迁移为 git-based 方案（`7038a05`）

### 2.2 无效 Chassis Reset → 404 Not Found（Jun 1）
- **Commit**: `9fa02b0` (jnin-dev + gtmills) — "Fix: POST Chassis.Reset for invalid chassis now returns 404 Not Found"
- 修复了一个 **长达约20个月的 Redfish 合规性 Bug**：向不存在的 chassis 发送 Reset 请求时，之前返回错误的 HTTP 响应码，现在合规地返回 **404 Not Found**
- 这是 Redfish 规范要求的标准行为，说明 bmcweb 持续在 Redfish 合规性上做打磨

### 2.3 PCIeDeviceType 支持（Jun 4）
- **Commit**: `af1b2b4` (liuer-sys) — "PCIE: add DeviceType to PCIeDevice"
- 在 PCIeDevice Redfish 资源中新增 **DeviceType** 属性
- 通过 phosphor-dbus-interfaces 的 `metric: add bandwidth usage of memory`（May 25, `3443c40`）和 `metric: add bandwidth usage of a processor`（Apr 15, `a12cc29`）配合，bmcweb 逐步完善 PCIe 子系统 Redfish 建模

### 其他安全修复：
- `9cdbe10` (May 29): Fix coredump in http_client during rapid connection destruction
- `5f15b5f` (Jun 1): zstd: do not attempt compressing already opened files
- `81dbaae` (May 12): Fix PayloadTooLarge to return HTTP 413
