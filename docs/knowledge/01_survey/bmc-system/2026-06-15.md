## 1️⃣ 📦 libpldm v0.16.0 正式发布 — 年度大版本，10 大新增领域

**来源**: [libpldm release v0.16.0](https://github.com/openbmc/libpldm/releases/tag/v0.16.0) · `03f6df07` (tag) · **发布: 2026-06-13**
**作者**: Andrew Jeffery (CodeConstruct)

### 技术详情

自 v0.15.0（2026-04-10）以来最大的功能性版本，覆盖 **10 大新增领域**：

#### 🆕 新增功能（重点）

1. **🔌 AF_MCTP FQE（Fully Qualified Endpoint）TID 映射** — 全新 `pldm_transport_af_mctp_map_tid_fqe()` / `unmap_tid_fqe()` API，允许将 PLDM TID 映射到 MCTP 网络和 EID 的完整限定端点标识。**意义**: 使 PLDM 能原生支持多网络拓扑中的跨网络 MCTP 通信（如同时管理多个 MCTP 网络段）。
2. **📋 DSP0249 v1.4.0 实体/状态集对齐** — 新增 entity types 和 state sets，包括 `predictive_condition`、`configuration_state`、`changed_configuration`、`version`、`communication_leash_status`、`acpi_power_state` 等 6 个新的枚举值集。
3. **🖥️ Redfish Resource PDR 和 Action PDR 解码**（DSP0248 v1.3.0）— 支持解码 Redfish 资源 PDR 和 Redfish 动作 PDR，使 PLDM 能够携带 Redfish 资源描述信息。
4. **📑 Redfish Device Enablement（RDE/DSP0218 v1.2.0）头文件新增** — PLDM 数据模型中加入 RDE 支持。
5. **💾 SMBIOS Data Transfer（DSP0246 v1.0.1）头文件新增** — 使 PLDM 可承载 SMBIOS 数据的带外传输。
6. **🔐 Firmware Update Security Revision（DSP0267 v1.3.0）** — 新增固件更新的安全版本号 API。
7. **🧩 PLDM Base MultipartSend 支持** — 大负载拆分传输的基础能力。
8. **🆕 libpldm++ C++ 绑定库** — 面向 C++ 调用者的高层绑定，首批稳定 API 包括 `pldm::fw_update::PackageParser::parse` 和 `pldm::fw_update::Package::~Package`。
9. **🏗️ Zephyr 支持** — 新增 Zephyr RTOS 构建所需的内置字节序定义和缺失宏。
10. **🧹 API 稳定化** — `{en,de}code_pldm_base_get_pldm_types_resp()`、Firmware Update Security Revision APIs、FQE TID mapping APIs 转为稳定（stable）状态。

#### 弃用/移除
- 旧 `{en,de}code_get_types_resp()` 标记为 deprecated，用户应切换至稳定版 `{en,de}code_pldm_base_get_pldm_types_resp()`

#### 影响
v0.16.0 是 libpldm 引入 **C++ binding**、**Zephyr 支持**、**多网络段 MCTP 通信** 的标志性版本，标志着 libpldm 从一个纯 C 库向跨平台、多语言的 PLDM 协议栈演进。特别是 FQE 支持使得 BMC 在大型多机箱超节点中管理多个 MCTP 网络段成为可能。

## 2️⃣ 🖥️ webui-vue: xterm v6 → v4 回退 — bmcweb CSP 策略兼容性修复

**来源**: [webui-vue commit `3e800a25`](https://github.com/openbmc/webui-vue/commit/3e800a25f85408ade252ac815bd461eceb7a6f8f) | **作者**: Jason Westover (NVIDIA) | **合并**: 2026-06-11

### 技术详情

- **问题背景**: `@xterm/xterm` v6 在渲染时通过 `DomRenderer._injectCss()` 动态注入 `<style>` 元素到 DOM 中，这直接违反了 bmcweb 的 `style-src 'self'` Content Security Policy（CSP）策略。**后果**: Serial Over LAN (SOL) 控制台在生产环境中完全不可用（终端显示空白）。
- **修复**: 将 `@xterm/xterm` 从 v6 回退到 **v4.19.0**，配套 `xterm-addon-attach` 和 `xterm-addon-fit` 也降级到对应 v4 版本，恢复 v4 的 import 路径。
- **额外调整**: `xterm.css` 在 unscoped style block 中导入，使 Vue scoped 属性选择器不会阻止 xterm 的动态 terminal DOM 的样式应用。
- **影响**: 维护了 bmcweb 严格的 CSP 安全策略同时，恢复了 SOL 控制台的核心功能。这是 **安全策略 vs 功能兼容性** 的典型案例——升级前端库时必须测试其在 CSP 环境中的行为。

## 5️⃣ 🔧 三平台 meta-facebook: 迁移至内核 DTS + SanMiguel VSER 波特率清理

**来源**: openbmc/meta-facebook (via openbmc) · 4 commits | **作者**: Patrick Williams, Potin Lai | **合并**: 2026-06-12

### 技术详情

**a) 三个 Facebook 平台统一切换到内核设备树 DTS** (`4bc565b8`, `22901702`, `4c73bef0` — Patrick Williams)
- **fbdarwin**、**sanmiguel**、**anacapa** 三个平台同步从 OpenBMC layer 自维护的 DTS 切换到**内核上游 kernel DTS**
- 意义：减少 OpenBMC 与 Linux 内核之间的设备树重复维护，DTS 的变更跟踪统一回到 Linux 内核社区

**b) SanMiguel: 移除 VSER 波特率设置** (`a32086d3` — Potin Lai)
- VSER（Virtual Serial）控制台通过 pts 设备节点访问，属**软件模拟终端**，配置波特率无实际意义
- 清理所有 VSER 配置文件中的波特率设置，避免混淆

**影响**: DTS 迁移是 OpenBMC 跟随 Linux 主线内核设备树规范的体现；VSER 波特率清理消除了运维中不必要的配置选项。

## 📊 趋势观察

1. **libpldm 体系能力大幅扩展** — v0.16.0 引入的 C++ binding、Zephyr 支持、FQE MCTP 映射和 Redfish PDR 解码，标志着 PLDM 从基础通信协议向覆盖全栈管理能力的平台协议演进。特别是 **Redfish PDR** 使 PLDM 协议层可以直接描述和传递 Redfish 资源信息，为「PLDM-over-MCTP → Redfish」的桥接提供了原生数据模型。

2. **WebUI 成为安全策略管控新界面** — Basic Auth 开关 UI、CSP 策略优先于 xterm 版本升级，表明 OpenBMC 正在将更多安全配置从底层的 Redfish API 提升到 WebUI 中，降低运维人员的安全配置门槛。

3. **代码基础设施质量持续打磨** — sdbusplus 的 use-after-free 修复和 invalid enum 处理，与持续多个版本的 CLI11/sdbusplus 代码清理，共同反映了当前 OpenBMC 核心库正处于「从功能交付到代码质量」的转型期。

4. **Facebook 平台紧跟内核 DTS 主线** — 三平台同步迁移至内核 DTS，意味着 OpenBMC 不再对硬件设备树做按需定制，而是复用上游内核的 DTS 描述。这对新平台 Bring-up 团队意味着需要确保 SoC DTS 先合入 Linux 内核主干。
