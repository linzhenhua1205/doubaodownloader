## 1️⃣ 🔌 meta-facebook santabarbara: 主机电源掉落状态同步增强

**来源**: OpenBMC Gerrit (meta-facebook) · commit `b5a1ed7` · 6/9 author, 6/12 merge  
**作者**: Fred Chen (Quanta) · **OpenBMC 仓库**: openbmc/openbmc

### 技术详情

当主机外部电源被切断时，BMC 必须在 **5 秒内**同步主机状态。若 5 秒后 `CurrentHostState` 仍为 `Running`，会触发功率传感器误报事件。

**关键改动**:
- `is_service_busy()` 替代 `systemctl is-active` — 解决 systemctl 检查的竞态脆弱性（systemctl 在服务状态转换期间可能返回过期结果）
- 在 assert/deassert 服务间添加 `Conflicts`/`After` 依赖 — 防止两个服务并行执行引发 Race Condition
- 在 3 秒去抖完成前就将 `CurrentHostState` 设为 `TransitioningToOff` — 尽早通知调用方"正在掉电中"
- 将 MCTP EID 移除逻辑抽取到独立 `mctp_remove` systemd service — 解耦、可单独调试

**影响**: 修复了 Santabarbara 产线实际遇到的**外部电源瞬断后 BMC 状态不同步**问题，属于生产环境级稳定性修复。

## 2️⃣ 📦 entity-manager: Anacapa 线缆库存 GPIO 存在检测

**来源**: OpenBMC entity-manager · commit `0dc442f` · 6/8  
**作者**: Rex Fu (AMD)

### 技术详情

为 AMD Anacapa 平台新增**基于 GPIO 的线缆存在检测**功能，通过 entity-manager 将线缆暴露为 Redfish Cable inventory 对象。

- 平台线缆的插拔状态通过 GPIO 信号读取，动态生成 `Cable` inventory 对象
- 线缆 DevicePresence 和 inventory 对象创建在 D-Bus 上可见
- Redfish Validator 验证通过

**影响**: 在 AMD Anacapa 平台上首次为 BMC 提供了**线缆级的 Redfish 资产清单管理**能力。运维人员可通过 Redfish API 直接查询各线缆的物理插接状态，无需人工巡检。

## 3️⃣ 🧩 entity-manager: Yosemite5 HSC Paddle 配置拆分

**来源**: OpenBMC entity-manager · commit `a822c7a` (Jackson Liu 2 commits) · 6/11  
**作者**: Jackson Liu (Quanta)

### 技术详情

Yosemite5 平台有多个不同型号的 Hot-Swap Controller (HSC)，此前共用同一套 entity-manager config 导致 `FirmwareInfo` 条目被暴露给不匹配的 HSC 类型，从而引发**固件更新操作误匹配**。

**关键改动**:
- 将不同 HSC 型号拆分到三个独立的配置文件（`RS31390_p12v_aux` 等）
- 每个 HSC 型号只暴露其支持的 FirmwareInfo 条目
- 同步新增 `TPS25990Firmware` 和 `RS31390Firmware` schema 支持

**影响**: 避免在 Yosemite5 产线/运维中出现「给 HSC-A 刷 HSC-B 固件」的误操作。每个 HSC 型号有独立的 entity-manager 配置，FirmwareInfo 按需暴露。

## 4️⃣ 📋 phosphor-dbus-interfaces: Redfish Update 消息注册表对齐

**来源**: OpenBMC phosphor-dbus-interfaces · commits `f597b1c` + `3f5a445` · 6/3-6/4  
**作者**: Jagpal Singh Gill

### 技术详情

OpenBMC 固件更新错误码体系同步对齐**DMTF Redfish Update v1.4.0 消息注册表**。

**两个提交**:
1. **`UpdateNotApplicable` 错误新增** (`3f5a445`) — 当固件更新镜像（如错误型号、版本不匹配等）不适用于目标设备时，返回 `UpdateNotApplicable` 而非通用错误。参考 Redfish Update registry v1.4.0 定义。
2. **`ApplyFailed` → `ActivateFailed` 更名** (`f597b1c`) — 废弃旧 `ApplyFailed` 错误码，替换为 `ActivateFailed`（Activation 比 Apply 更精确描述固件激活阶段）。Redfish Update registry v1.4.0 中 `ApplyFailed` 已标记为 deprecated。

**影响**: 这两个改动对齐了 OpenBMC 的 `Update`/`Activate` 错误语义与 DMTF Redfish 标准。对于依赖 Redfish Update 标准消息的自动化运维工具（Ansible、Satellite、OpenStack Ironic 等），错误消息可被标准解析器处理，无需额外定制映射。

## 📊 趋势观察

1. **Redfish 标准对齐持续深入** — 条目 4 表明 OpenBMC 在逐步消除与 DMTF 标准之间的语义差异，尤其是固件更新场景中的错误消息体系。

2. **AMD Anacapa 平台进入 DVT 批量配置阶段** — 条目 2（线缆库存）、条目 3（Yosemite5 HSC 拆分配置）以及上周的 Anacapa DVT configs 和 bridge board CPLD compatible strings 共同表明 Anacapa 已从 EVT 进入 **DVT 批量验证阶段**，entity-manager 配置持续丰富。

3. **产线级稳定性修复为主旋律** — 条目 1（电源掉落状态同步）和条目 3（HSC 配置拆分）都是直接受到产线运维反馈驱动的修复，而非提前预埋的功能开发。
