## 1️⃣ 🔐 bmcweb: 新增用户密码过期管理 REST API

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/bmcweb` |
| **提交** | `e80fca94a0cd` (merged 2026-06-29) |
| **作者** | Ivan Moiseev / Ivan Mikhaylov |
| **标题** | `Manage user password expiration via REST` |

**关键特性**: 通过 Redfish REST API 管理 BMC 用户密码过期时间，完整生命周期支持：

| 操作 | HTTP 方法 | Redfish 路径 | 说明 |
|:-----|:---------|:-------------|:-----|
| 创建带过期密码的用户 | `POST` | `/redfish/v1/AccountService/Accounts` | JSON 中设定 `PasswordExpiration: "<YYYY-MM-DDTHH:MM:SS>"` |
| 修改密码过期时间 | `PATCH` | `/redfish/v1/AccountService/Accounts/<user>` | 更新 `PasswordExpiration` 属性 |
| 查询密码过期时间 | `GET` | `/redfish/v1/AccountService/Accounts/<user>` | 返回 `PasswordExpiration` 字段 |
| 取消密码过期 | `PATCH` | `/redfish/v1/AccountService/Accounts/<user>` | 设置 `PasswordExpiration: null` |

**内部实现细节**:
- 密码过期时间以 **Epoch 时间** 内部存储，REST 接口层用 `dateStringToEpoch()`（`redfish::time_utils`）做格式转换
- 支持任意 `dateStringToEpoch` 可解析的时间格式，不限于 ISO 8601
- **完整输入校验**：无效类型/格式 → 返回 Redfish 标准错误，用户不创建/不修改
- 新增单元测试文件：`test/redfish-core/lib/account_service_test.cpp`（+66行）
- 配套依赖：`phosphor-dbus-interfaces`（#75236）+ `phosphor-user-manager`（#75237）
- 通过了 Redfish Service Validator 的 `/redfish/v1/AccountService/Accounts` 树验证

**改造范围**:
```
  modified | docs/Redfish.md                                      | +1/-0
  modified | redfish-core/lib/account_service.hpp                 | +169/-44
  modified | test/meson.build                                     | +1/-0
     added | test/redfish-core/lib/account_service_test.cpp       | +66/-0
```

**重要性**: 🔴 **高** — 密码过期管理是 BMC 安全合规的基础功能：

| 合规标准 | 要求 | 映射 |
|:---------|:-----|:-----|
| **NIST SP 800-207** (零信任) | 持续验证凭证有效性 | `PasswordExpiration` 强制定期轮换 |
| **ISO 27001:2022** A.9.4.3 | 密码过期机制 | Redfish API 提供配置入口 |
| **PCI DSS v4.0** 8.3.9 | 供应商默认密码轮换 | 新建用户即可设过期时间 |
| **OCP Security** | 带外管理账户生命周期 | 覆盖 create/modify/query/disable 全流程 |

此前 OpenBMC 缺少通过 Redfish 配置密码过期的能力，需登录 BMC shell 或用 ipmitool 间接管理，不满足自动化合规要求。此补丁填补了这一空白。

## 2️⃣ 🐛 phosphor-host-ipmid: 修复损坏用户 JSON 导致 ipmid 崩溃循环

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/phosphor-host-ipmid` |
| **提交** | `90210075c80f` (merged 2026-06-22) |
| **作者** | Nagavarapu Sai Pavan (AMI) |
| **标题** | `Fix ipmid crash loop on corrupt user JSON` |

**问题根因分析**:

当 `/var/lib/ipmi/ipmi_user.json` 文件因以下原因损坏时：

```
异常链路：
  读损坏JSON → readUserData() 抛 std::runtime_error
    → cacheUserDataFile() 只 catch std::ios_base::failure
      → runtime_error 未被捕获 → std::terminate() → SIGABRT
        → systemd 自动重启 ipmid → JSON 仍损坏 → 再次崩溃
          → 循环直到 systemd Restart= 上限耗尽
            → IPMI 栈完全不可用，需人工介入修复
```

此外，`Json::parse()` 虽设置 `allow_exceptions=false`（不抛异常），但未检查 `is_discarded()`，导致损坏 JSON 通过 parse 后进入 size 检查，产生 **误导性错误日志** `"User count issues"` 而非指明实际问题是 JSON 损坏。

**修复方案**:

| 问题 | 修复方式 |
|:-----|:---------|
| `is_discarded()` 未检查 | 在 `Json::parse()` 后添加 `is_discarded()` 检查，与 `transporthandler.cpp`、`dcmihandler.cpp`、`channel.cpp`、`entity_map_json.cpp` 一致 |
| `std::runtime_error` 未捕获 | 在 `cacheUserDataFile()` 中添加 `catch (std::runtime_error&)` 块 |
| 重复的安全初始化逻辑 | 提取 `initializeUserDataFile()` 复用，避免代码重复 |

**恢复行为**: 安全降级——使用默认值初始化用户数据文件，保持 BMC 继续运行，记录 `"Corrupted IPMI user data file - invalid JSON"` 日志。

**测试验证**:

| 测试场景 | 行为 |
|:---------|:-----|
| 损坏 JSON (`echo "CORRUPTED" > ipmi_user.json`) | 日志记录 + 初始化默认值 + 正常运行 |
| 错误结构 (`echo '[]' > ipmi_user.json`) | "User count issues" + 初始化 + 运行 |
| 缺失文件 | 原有恢复路径，正常工作 |
| 正常文件 | 正常加载，无回归 |

**重要性**: 🔴 **高** — 崩溃循环（crash loop）是生产环境最严重的故障模式之一：

| 维度 | 影响 |
|:-----|:-----|
| **故障面** | 整个 IPMI 栈不可用，带外管理通道中断 |
| **恢复方式** | 人工修复 JSON 或清理文件，不符合无人值守运维要求 |
| **触发概率** | 异常掉电时文件系统写半截、NAND flash 比特翻转、NFS 挂载网络闪断均可能触发 |
| **修复效果** | 从 crash loop → 自动降级恢复，容错能力质的提升 |

🔗 Issue参考: https://github.com/openbmc/phosphor-host-ipmid/issues/216

## 4️⃣ 🛠️ bmcweb: 修复 OEM Fan Zone Chassis 属性链接错误

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/bmcweb` |
| **提交** | `1bda4f7387d8` (merged 2026-06-26) |
| **作者** | Alexander Hansen (9elements) / Ed Tanous (NVIDIA) |
| **标题** | `fix 'Chassis' property in oem schema` |

**问题**: `redfish-core/schema/oem/openbmc/` 的 OEM schema 定义 Fan Zone 包含 `Chassis` 属性，但实现中使用 PID Zone 配置接口的 D-Bus 对象路径的 leaf 名作为 Chassis 链接。例如 Fan Zone 名为 `"Zone0"`，则返回的 Chassis 链接是 `/redfish/v1/Chassis/Zone0`——该 Chassis 根本不存在，导致 **Redfish Service Validator 验证失败**。

**修复**: 改用 `GetManagedObjects` 的结果遍历 D-Bus 对象，查找真实 Chassis 路径（如 `MBX_1_57_Chassis`），正确插入 Chassis 链接。

**验证结果**:
```
Redfish Validator Tree 测试：Pass: 766, Fail: 0, Warning: 0
```

**重要性**: 🟢 **低** — 配置依赖的验证失败修复，提升 OpenBMC 默认配置下对 RSV 的合规通过率。
