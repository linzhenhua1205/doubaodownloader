## ① ⚠️ 重大安全：phosphor-logging/openpower-pels 集中披露9个漏洞（2026-06-04）

**来源**: [OpenBMC Mailing List](https://lists.ozlabs.org/pipermail/openbmc/2026-June/thread.html)
**报告者**: Bhargava Naidu Gulla（安全审计团队）
**受影响组件**: `phosphor-logging` / `openpower-pels`（PEL — 平台事件日志子系统）
**影响范围**: 所有使用openpower-pels的OpenBMC平台（IBM POWER服务器、部分OpenPOWER系统）

| # | 漏洞类型 | 影响 | 严重程度 |
|:-:|:---------|:-----|:---------|
| 1 | **RAWPEL Path Injection** — 任意文件读取/删除 | 攻击者可通过RAWPEL路径注入读取或删除任意文件 | 🔴 **高危** |
| 2 | **peltool Untrusted Python Search Path** — 本地权限提升 | 不可信Python搜索路径导致LPE | 🔴 **高危** |
| 3 | **PHAL createDeconfigRecords Stack Buffer Overflow** — 栈溢出 | `std::copy` 导致的栈缓冲区溢出 | 🔴 **高危** |
| 4 | **dumpHex Signed Integer Overflow** — 堆溢出 | 有符号整数溢出导致的堆缓冲区溢出 | 🔴 **高危** |
| 5 | **FRUIdentity String Extraction OOB Read** — 越界堆读取 | FRU身份字符串提取时越界读取 | 🟡 **中危** |
| 6 | **getPEL Local File Descriptor Leak** — 文件描述符泄漏→DoS | 文件描述符不断泄漏最终导致拒绝服务 | 🟡 **中危** |
| 7 | **UserHeader Unhandled Exception** — 守护进程崩溃 | UserHeader解析时未处理异常导致BMC daemon崩溃 | 🟡 **中危** |
| 8 | **SbeFFDC::parse Integer Underflow + Heap OOB** — 整数下溢+堆越界 | SBE FFDC解析时的整数下溢和堆越界读取 | 🟡 **中危** |
| 9 | **HostNotifier Timer Overlaps** — 静默日志丢弃 | Timer重叠导致主机日志静默丢失（社区已回复） | 🟡 **中危** |

**社区响应**：Andrew Jeffery（核心维护者）已在第9项漏洞报告中进行回复讨论。

**工程意义**：
- 这是OpenBMC社区**首次大规模集中披露**安全漏洞报告
- 3个高危（文件注入/权限提升/栈溢出）+ 6个中危，涉及PEL日志子系统的多个层面
- phosphor-logging作为BMC核心日志组件，这些漏洞可能影响**生产环境的可靠性监控和安全隔离**

## ② ASUS Kommando IPMI Card — 新BMC硬件平台（2026-06-03/04）

**来源**: [OpenBMC Mailing List](https://lists.ozlabs.org/pipermail/openbmc/2026-June/thread.html) — [PATCH u-boot v3]
**贡献者**: Tan Siewert (ASUS), Anirudh Srinivasan, Ender Hsieh

ASUS 提交 **U-Boot v3 补丁系列** 为其 **Kommando IPMI 卡** 添加设备树支持：

| 补丁 | 内容 |
|:-----|:------|
| `[PATCH v3 1/2]` | 修复 ASPEED MAC 时钟延迟的默认值（`mac_clk_delay`） |
| `[PATCH v3 2/2]` | 添加 **ASUS Kommando IPMI 卡** 的设备树文件 |

**硬件特征**：
- 基于 **ASPEED** SoC 的独立 BMC/IPMI 卡
- U-Boot 层面专门针对ASPEED MAC时钟延迟做了修复，反映实际量产中的硬件时序调优

**工程意义**：ASUS加速进入BMC硬件市场，Kommando IPMI卡定位为独立BMC模块，可适配非OpenBMC原生支持的服务器主板，拓展OpenBMC生态覆盖范围。

## ③ AST2600 eSPI 控制器支持 — 新硅特性使能（2026-05-28）

**来源**: [OpenBMC Mailing List](https://lists.ozlabs.org/pipermail/openbmc/2026-June/thread.html) — [PATCH 0/7]
**贡献者**: YH Chung (ASPEED)

ASPEED 提交 **7个补丁** 在 Linux 内核中添加 **AST2600 eSPI 控制器支持**：

| 补丁 | 内容 |
|:-----|:------|
| 1-2 | eSPI 控制器 DT-bindings + 控制器驱动框架 |
| 3 | 外设通道支持 |
| 4 | 虚拟线缆 (VW) 通道支持 — **eSPI最重要的特性**（替代LPC的中断机制） |
| 5 | 调试消息通道 |
| 6 | 闪存（Flash）通道 — eSPI闪存共享支持 |
| 7 | 系统级联宽总线 — 多控制器互联 |

**工程意义**：
- **eSPI (Enhanced Serial Peripheral Interface)** 是 Intel 推广的 BMC-主机间新标准总线，替代传统 LPC
- AST2600 的 eSPI 支持意味着 OpenBMC 可以**原生对接 Intel/AMD 新一代处理器平台的 eSPI 接口**
- 特别是 VW（Virtual Wire）通道支持，使 BMC 能通过 eSPI 替代传统的 LPC IRQ/SMI 机制
- 这对 Intel Xeon 6+（Clearwater Forest）和 Diamond Rapids（2027）等新平台的 BMC 支持至关重要

## ④ MCTP over USB (DSP0283) 集成讨论（2026-06）

**来源**: [OpenBMC GitHub Issue #4075](https://github.com/openbmc/openbmc/issues/4075) / Mailing List
**提出者**: Song, MS (Microsoft?)

社区讨论 **MCTP over USB** 在 OpenBMC 中的集成状态：

- **MCTP（Management Component Transport Protocol）** 是 DMTF 标准的管理通信协议
- **DSP0283** 定义 MCTP 在 USB 上的传输绑定
- 讨论涉及：MCTP/USB 在内核和用户态的集成进展

**工程意义**：
- MCTP 是 Redfish 标准推荐的基础传输协议之一
- USB 作为 MCTP 物理层，比 I2C/PCIe 更通用（无需特定硬件线路）
- 适用于**开发和调试场景**，或作为无专用管理网络的轻量级带外管理方案

## ⑤ Container Registry 共享 BitBake Sstate-Cache 方案提案

**来源**: [OpenBMC Mailing List](https://lists.ozlabs.org/pipermail/openbmc/2026-June/thread.html)
**提出者**: Yash Patel

提出利用 **Container Registry**（如 Docker Hub / GitHub Container Registry）共享 BitBake 构建缓存（sstate-cache）：

- **问题**：OpenBMC 构建时间长（首次编译数小时），sstate-cache 分发依赖本地网络/NFS
- **方案**：sstate-cache 打包为 OCI 镜像，推送到容器注册表，CI/CD 和开发者拉取复用
- **优势**：全球加速、版本化、不依赖特定基础设施

**工程意义**：
- 如果实现，将**大幅降低 OpenBMC 新贡献者的编译门槛**
- 对于需要频繁构建多平台的开发团队（如 NVIDIA/Meta），可显著缩短迭代周期
