## 1️⃣ 🔧 phosphor-bmc-code-mgmt: 修复固件更新路径匹配 Bug — `starts_with` 导致设备误匹配

**来源**: [phosphor-bmc-code-mgmt commit `fca0bf42`](https://github.com/openbmc/phosphor-bmc-code-mgmt/commit/fca0bf421b188438a59ffd7e546e2dcd655fdae8) | **作者**: Alexander Hansen | **合并**: 2026-06-13

### 技术详情

- **问题**: 在固件更新设备兼容性检查中，`software_config.cpp` 使用了 `objectPath.str.starts_with(path)` 来匹配 inventory 对象路径，这存在**路径前缀误匹配**风险：

  ```cpp
  // 原代码 (错误):
  if (objectPath.str.starts_with(path))
  ```

  假设真实的 inventory 路径为 `/xyz/openbmc_project/inventory/system/chassis/motherboard/CPU1`，而另一个设备路径为 `/xyz/openbmc_project/inventory/system/chassis/motherboard/CPU`（不含尾数），原代码会将两者视为匹配，因为前者以 `CPU` 为前缀开头。

- **修复**:

  ```cpp
  // 修复后:
  if (objectPath.parent_path() == path)
  ```

  改用 `parent_path()` 获取对象的 D-Bus 父路径后做精确相等比较。

- **影响**: 这个 Bug 直接作用于**固件更新流程中的设备兼容性判定**。若 inventory 对象路径匹配不正确，可能导致 BMC 错误判断某一固件镜像适用于不兼容的设备，进而**刷入错误固件引发设备损坏**或「兼容性通过但实际失败」的异常行为。该修复确保了固件更新的目标设备路径匹配是严格的父子关系而非前缀字符串匹配。

## 2️⃣ 🌐 meta-google: NCSI 非主接口转发规则增强 — 多 NCSI 路径防火墙

**来源**: [openbmc commit `4a9e3831`](https://github.com/openbmc/openbmc/commit/4a9e3831c4bc83719b4fb73bad135c3aa41b822e) | **作者**: William A. Kennington III (Google) | **合并**: 2026-06-13

### 技术详情

- **背景**: 在 Google 的 BMC 网络架构中，gbmc-ncsi-ra 管理 NCSI（Network Controller Sideband Interface）路由通告。对于主 NCSI 接口，转发规则由 `gbmc-ncsi-br-deprecated-ips.sh` 自动处理，但**非主 NCSI 接口**缺少相应的防火墙规则，导致跨桥流量无法到达。

- **改动**: 在 NCSI 路由通告处理脚本 `gbmc-ncsi-ra.sh`（重命名为 `.in` 模板文件）中新增 nftables 规则生成逻辑：

  ```bash
  # 当非主 NCSI 接口收到路由通告时
  if [ "$RA_IF" != '@NCSI_IF@' ]; then
    cat >/run/nftables/40-gbmc-ncsi-"$RA_IF"-br.rules <<EOF
  table inet filter {
    chain ${RA_IF}_forward {
      ip6 saddr != $pfx/76 ip6 daddr $pfx/76 accept
    }
  }
  EOF
    systemctl reset-failed nftables && systemctl --no-block reload-or-restart nftables || true
  fi
  ```

  - **规则语义**: 对于非主 NCSI 接口分配的前缀 `pfx/76`，允许所有来自该前缀外的入站流量（`ip6 saddr != $pfx/76 ip6 daddr $pfx/76 accept`）进入该前缀范围
  - **动态加载**: 规则文件写入 `/run/nftables/`（tmpfs），通过 systemctl 触发热加载 nftables，无需重启
  - **模板化**: 脚本改为 `.sh.in` 模板，编译时通过 `sed` 注入 `@NCSI_IF@`（主接口名）

- **验证**: 已在 Google 的真实硬件上完成规则生效验证

- **影响**: 此修复针对多 NCSI 路径场景。在大型数据中心中，BMC 常通过多个 NCSI 接口接入不同管理网络段（带外管理、资产发现、固件下载等）。没有此规则，非主 NCSI 链路上的流量仅能出站但无法接收响应。这是管理面网络层面从「单 NCSI 路径可用」走向「多 NCSI 路径高可用」的基础设施补全。

## 3️⃣ ⚡ phosphor-modbus: BBU CBU 电容组健康状态深度监控 — 故障状态位 + 功率/硬件寄存器

**来源**: [phosphor-modbus commits `097746e7` + `81d307f1`](https://github.com/openbmc/phosphor-modbus/commits/master) | **作者**: Jagpal Singh Gill | **合并**: 2026-06-13 ~ 2026-06-14

### 技术详情

对 Delta ECD27010006 Capacitor Bank Unit（超级电容备电单元）的 Modbus RTU 监控配置文件做了两轮增强：

**a) 故障状态位全面扩展** (`097746e7`, 6/13)

新增 11 个可监控的状态位，覆盖超级电容电池的几个关键故障域：

| Bit 位置 | 状态名称 | 类别 | 意义 |
|:--------:|:---------|:----:|:-----|
| 13 | `TEMPERATURE_FAILURE` | 🔴 SensorReadingCritical | 模组温度越限 |
| 12 | `PACK_OVER_CURRENT` | 🔴 SensorReadingCritical | 电池包过流 |
| 11 | `CELL_UNDER_VOLTAGE` | 🔴 SensorReadingCritical | 单电芯欠压 |
| 10 | `PACK_UNDER_VOLTAGE` | 🔴 SensorReadingCritical | 电池包欠压 |
| 9 | `CELL_OVER_VOLTAGE` | 🔴 SensorReadingCritical | 单电芯过压 |
| 8 | `PACK_OVER_VOLTAGE` | 🔴 SensorReadingCritical | 电池包过压 |
| 5 | `FAN_3_FAULT` | 🌀 FanFailure | 风扇 3 故障 |
| 4 | `FAN_2_FAULT` | 🌀 FanFailure | 风扇 2 故障 |
| 3 | `FAN_1_FAULT` | 🌀 FanFailure | 风扇 1 故障 |
| 2 | `SUPERCAP_PACK_OVER_TEMP` | 🔴 SensorReadingCritical | 超级电容包过温 |
| 更多 | 温度/电压/电流状态字 | — | `STATUS_TEMP` 寄存器扩展 |

**b) 功率和硬件寄存器增强** (`81d307f1`, 6/12)
- 新增功率相关寄存器（当前功率、能量等）
- 新增硬件寄存器（固件版本、硬件版本、序列号等）

**影响**: 这两轮提交将 BBU 备电单元的监控精细度从「只知道是否在位」提升到「可逐位监控 11 种故障状态 + 功率趋势 + 硬件信息」。对于 AI 集群中 BBU Shelf 的高密度场景，BMC 可通过 Modbus RTU 轮询超级电容组的实时健康状态，提前发现电位不均衡、散热风扇退化等早期故障信号。

## 4️⃣ 🧪 phosphor-modbus: 测试改进 — 主动轮询替代固定睡眠等待

**来源**: [phosphor-modbus commit `da22ba9f`](https://github.com/openbmc/phosphor-modbus/commit/da22ba9f1d9ed50e3e3b8f6406d94fbfb9fd2d1b) | **作者**: Jagpal Singh Gill | **合并**: 2026-06-14

### 技术详情

- **原始代码**: 在 inventory 探测测试中使用固定 `co_await sdbusplus::async::sleep_for(ctx, 1s)` 等待 D-Bus 对象创建
- **修复**: 改为每 100ms 轮询检测 `checkInventoryObjectExists(objPath)`，直到对象出现为止：
  ```cpp
  while (!(co_await checkInventoryObjectExists(objPath)))
  {
      co_await sdbusplus::async::sleep_for(ctx, 100ms);
  }
  ```
- **影响**: 固定等待在测试执行环境不同时（CI 负载高/低）要么太慢（白白等 1s），要么不够用（1s 不够对象创建）。主动轮询大幅提升了测试的**时间鲁棒性**和**执行效率**。

## 📊 趋势观察

1. **路径匹配 Bug 仍是 BMC 固件更新的薄弱环节** — 条目 1 的 `starts_with` bug 与上周 6/14 的 HSC 配置拆分（条目 3）共同指向同一个问题：**固件更新流程中设备匹配的精准度**。一个是 Inventory 路径前缀匹配错误，一个是固件信息暴露给不匹配的设备。两者都可能导致**固件误刷**这一严重运维事故。

2. **BBU 备电监控从「在位检测」走向「健康诊断」** — 条目 3 的 11 个故障状态位扩展，标志着 BMC 对备电单元的监控不再满足于「是否在位/是否报警」，而是进入**诊断级健康管理**——能区分过压、欠压、过热、风扇故障等具体故障根因。这与 AI 集群对 BBU Shelf 的精细化管理需求一致。

3. **多 NCSI 路径高可用成为 Google 的管理面演进方向** — 条目 2 表明，Google 的 BMC 网络架构正在从单 NCSI 路径向多路径演进，NCSI 防火墙规则的补全只是第一步，后续可能持续跟进路径冗余和故障切换。

4. **测试基础设施持续改进** — 条目 4 的固定等待→主动轮询虽小，但反映了 OpenBMC 测试代码逐步从「硬编码超时」向「事件驱动」迁移的趋势，这对于跨平台 CI（时间行为不一致）尤为重要。
