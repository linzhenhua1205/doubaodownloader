## 1️⃣ 🔥 phosphor-dbus-interfaces: UserType 枚举 — 认证后端类型标准化

**来源**: [phosphor-dbus-interfaces commit `20f054a287`](https://github.com/openbmc/phosphor-dbus-interfaces/commit/20f054a287edde6815f045dae21a7003e5b343ea) + [phosphor-user-manager commit `4e6172685f`](https://github.com/openbmc/phosphor-user-manager/commit/4e6172685fe7b0e1505619d8ffe055d7c1e4239e) | 作者: Vijaysankar Ravi (AMI) | 日期: 2026-06-10

**技术详情**:
- **背景**: 此前 getUserInfo 只通过 `RemoteUser` boolean（true/false）区分本地/远程用户，但**无法区分远程用户的具体认证后端是 LDAP 还是 Active Directory**。这对多认证后端混合部署的大规模数据中心场景形成管理盲区。
- **dbus-interfaces 层** (`+25-0`): 在 `xyz/openbmc_project/User/Manager.interface.yaml` 中新增 `UserType` 枚举类型，定义三种认证后端：
  - `Internal` — 本地管理的用户（usersList）
  - `LDAP` — 通过 OpenLDAP 认证
  - `AD` — 通过 Active Directory 认证
- **user-manager 层** (`+14-0`): `getUserInfo()` D-Bus 方法返回数据中新增 `s UserType` 字段。类型通过已解析的 LDAP config D-Bus 对象路径推导，**不需要额外 D-Bus 调用**（直接从 privilege mapper 查找结果中获取）。
- **测试验证**: 在 AST2600 evb 上使用 `busctl` 调用 `GetUserInfo` 分别验证了 local、LDAP、AD 三种场景的正确返回。
- **影响**: 这是 OpenBMC 用户管理体系的一次重要标准化——将隐式的 `RemoteUser=true` 升级为显式的 `UserType` 枚举。Redfish + IPMI 的 Session 管理都可以基于此字段做更精细的认证后端感知策略。
- **同期代码清理**: 同一作者同日提交了 [cppcheck 告警修复](https://github.com/openbmc/phosphor-user-manager/commit/bb73805bcfa8)（`bb73805b`, `dd97aa6b` 等），修复了 virtualCallInConstructor、passedByValue、shadowVariable、redundantInitialization 等 8 类 cppcheck 告警，无功能变更。

## 2️⃣ phosphor-state-manager: 修复 JobRemoved 信号状态竞争（Wiwynn 生产发现）

**来源**: [phosphor-state-manager commit `4f51875dc3`](https://github.com/openbmc/phosphor-state-manager/commit/4f51875dc30a88e0fcc9cf6659f9b6616f95d86f) | 作者: Leo Wu (Wiwynn) | 日期: 2026-06-05

**技术详情**:
- **问题描述**: 系统通过监听 systemd `JobRemoved` 信号更新 D-Bus 状态（host/chassis/BMC state）。`JobRemoved` + result=`"done"` 只表示 systemd job 运行完成，**不代表目标 unit 已到达并处于 active 状态**。
- **竞争窗口**: 当 `Conflicts=` 依赖触发时（例如启动 obmc-host-start@N.target 时 systemd 自动停止 obmc-host-stop@N.target），停止 job 先完成并发出 `JobRemoved` + `"done"`。旧逻辑**误将此信号当作目标已到达**，可能导致状态被错误更新（例如：host 正在启动过程中，状态被错误置为 `Off`）。
- **修复方案**: 在 `JobRemoved` 处理流程中新增 `isActive()` 检查——只有目标 unit 确实处于 `active` 状态时，才将 job 完成视为合法状态转换。
- **测试验证**:
  - 确认 `Conflicts=` 触发的停止 job 不再将状态更新为 Off
  - 确认正常的 host/chassis/BMC 目标完成仍然正确更新对应状态
- **影响**: 此 bug 是**预存在的竞争条件**，`Conflicts=` 在 start/stop 目标之间的 ordering 只是让时间窗口更明显，并非引入该问题。该修复影响所有使用 `JobRemoved` 更新 D-Bus 状态的 OpenBMC 系统，尤其是 multi-chassis SMP 场景中多个 chassis 并发状态管理。

## 3️⃣ 🔧 OpenBMC: meta-facebook clemente MCTP 弹性增强（Quanta）

**来源**: [OpenBMC PR `28a07884ee`](https://github.com/openbmc/openbmc/commit/28a07884eeca435fe58736215462f6ec50e62bb2) + [`ecb53acde0`](https://github.com/openbmc/openbmc/commit/ecb53acde047d9aaa6a2982c86d3ecc78d36c2ad) | 作者: Potin Lai (Quanta) | 日期: 2026-06-10

**技术详情**:
- **Commit A — setup_eid 依赖控制** (`+4-2`):
  - **问题**: cx-ready 和 pldmd 在 MCTP EID（Endpoint ID）分配完成前启动，PLDM 服务在不完整的 MCTP 配置下运行。
  - **修复**: 添加 `setup_eid()` 返回值检查，**仅在所有 required MCTP 接口 EID 分配成功后**才启动 cx-ready 和 pldmd。防止不完整配置导致的异常 PLDM 行为。
  
- **Commit B — USB hub 重置机制** (`+24-0`):
  - **问题**: 实际产线中发现 **USB hub 偶发端口映射错乱**，导致 `mctpi2c0`/`mctpi2c1` 等 MCTP 接口名意外变化，init-sma 无法定位预期接口，SMA 初始化失败。
  - **修复**: 初始化时检查预期 MCTP 接口名是否存在；如果缺失，通过 GPIO 触发 USB hub 复位并**重试初始化流程**，恢复正确端口映射和接口枚举。
  - **硬件关联**: 该问题与特定 USB hub 型号的枚举行为相关，复位机制是产线级可靠性的关键保障。

- **测试**: Facebook clemente 平台产线验证。

## 4️⃣ bmcweb: JSON 解析日志增强（安全改进）

**来源**: [bmcweb commit `a857cf54c1`](https://github.com/openbmc/bmcweb/commit/a857cf54c14b1fdb61a20ecfc1b96bcd4dfcd703) | 作者: Akash Arunkumar | 日期: 2026-06-09

**技术详情**:
- **背景**: Redfish 客户端发送请求时，Content-Type header 为空或无效会导致 JSON 解析失败，但仅有泛化的日志信息，难以快速定位根因。
- **改进**:
  - 新增**显式警告日志**：当 `Content-Type` header 为空或缺失时，输出特定日志 `"Empty or missing Content-Type header"`
  - 保留已有的 `"Unsupported Content-Type"` 警告（针对 text/plain 等无效类型）
  - **安全考量**: 避免在日志中反射用户可控输入（URL、header 值），防止恶意请求日志洪泛
- **测试**: Ubuntu 24.04.1 LTS，确认空 Content-Type 的日志出现、无效类型的警告保留、无用户输入被反射到日志中。

## 🔟 phosphor-state-manager: Monitor JSON 文件打开验证（鲁棒性增强）

**来源**: [phosphor-state-manager commit `d6093cd519`](https://github.com/openbmc/phosphor-state-manager/commit/d6093cd51931b3e0a6e4e4e65f6e4a6afd9c278f) | 作者: Ramya Sivakumar (AMI) | 日期: 2026-06-10

**技术详情**:
- **背景**: `phosphor-state-manager` 的 monitor 组件在启动时解析 JSON 配置文件以确定 host/BMC 状态转换规则。当文件缺失、损坏或无权限时，解析过程会抛出泛化的 `std::exception`，日志中只有一个 `Parse error`，完全无法定位是哪个文件出了问题。
- **修复** (`+21-1`):\n  - 在 `parseMonitorJson()` 入口处新增 `is_open()` 检查，确认 `std::ifstream` 成功打开文件后再开始解析\n  - 如果文件打开失败，输出显式错误日志：`Unable to open <filename>`\n  - 失败文件被**跳过**而非阻塞整个启动流程（允许部分文件缺失/损坏情况下的 graceful degradation）\n- **影响**: 该修复对所有从 monitor JSON 文件启动的 OpenBMC 系统有效。在产线部署中，某些平台的 monitor JSON 文件可能在出厂烧录阶段未复制完整，此修复将隐式崩溃转化为可诊断的显式错误日志，大幅降低现场调试成本。
