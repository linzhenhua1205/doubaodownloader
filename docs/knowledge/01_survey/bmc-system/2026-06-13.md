## 1️⃣ 🔐 bmcweb: 生成 64 位证书序列号 — testssl.sh 合规加固

**来源**: [bmcweb commit `e5cd5009b2`](https://github.com/openbmc/bmcweb/commit/e5cd5009b23255637dd7f11701a65d8c8fa09b88) | 作者: Ed Tanous | 合并: 2026-06-12

**技术详情**:
- **问题**: bmcweb 自签名证书的序列号只有 4 字节（`4B32D4F0`），不满足安全扫描工具的安全基线要求。`testssl.sh` 在证书审计中报告：
  ```
  Serial 4B32D4F0 NOT ok: length should be >= 64 bits entropy (is: 4 bytes)
  ```
- **修复**: 将证书序列号生成逻辑从 32 位随机数改为 **64 位随机数**（8 字节），满足 testssl.sh ≥64 bits entropy 的安全要求。
- **技术实现**: 通过 OpenSSL API 生成 8 字节随机序列号，与原有自签名证书流程无缝集成。
- **影响**: 此修复直接影响 BMC 默认 HTTPS 证书的安全评级。部署在 SOC2/ISO 27001 等合规审计环境中的 BMC，默认证书不再触发 testssl.sh 告警。用户无需手动导入证书即可通过基本安全扫描。

## 2️⃣ 👤 bmcweb: Redfish 禁用用户时自动注销所有活跃会话（Session 安全）

**来源**: [bmcweb commit `430b30bd5f`](https://github.com/openbmc/bmcweb/commit/430b30bd5f800b5c7dc0b2bd856f71f72b93c785) | 作者: Chandramohan Harkude | 合并: 2026-06-08

**技术详情**:
- **问题**: 通过 `PATCH /redfish/v1/AccountService/Accounts/` 将 `{"Enabled":false}` 写入 Redfish 禁用某个用户后，D-Bus 上 `UserEnabled` 属性已切换为 false，但该用户的所有 **已签发 X-Auth-Token 会话仍然完全可用**。token 认证的请求继续返回 `200 OK`，直到 token 自然过期。而 Basic 认证已在禁用后正确返回 `401`，形成安全不一致。
  - `DELETE` 用户没有此问题 → 因为删除用户触发 `InterfacesRemoved`，`onUserRemoved()` 处理函数会调用 `removeSessionsByUsername()`。
  - `PATCH Enabled=false` 没有触发任何 session 清理。

- **修复方案**: 在 `include/user_monitor.hpp` 中新增 `onUserPropertiesChanged()` 处理函数，监听 `User.Attributes.UserEnabled` 属性变化：
  - 当 `UserEnabled` 从 `true` → `false` 时，调用 `SessionStore::removeSessionsByUsername()` 立即清除该用户的所有活跃 session。
  - 同时覆盖来自 **IPMI** 和 **Redfish** 两种途径的用户禁用场景。
  - 修复不依赖 `InterfacesRemoved` 信号，而是通过 D-Bus PropertiesChanged 信号驱动。

- **测试验证**（完整 curl 测试序列）:
  1. 创建 test_admin 用户 → 创建 session 获取 `x-auth-token: VcufgTshiDjEn8HbGh31`
  2. 用 token 请求 → `200 OK` ✅
  3. `PATCH {"Enabled":false}` → `204` ✅
  4. 再次用同一 token 请求 → `401 Unauthorized` ✅

- **影响**: 此修复对于多租户 BMC 环境至关重要——管理员禁用离职员工账号后，所有活跃 session 即时失效，消除 token 有效期窗口内的安全风险。

## 3️⃣ 🔧 fb-ipmi-oem: 修复 B-value 溢出检查 bug — IPMI SDR 传感器属性计算

**来源**: [fb-ipmi-oem commit `0d3d62544f`](https://github.com/openbmc/fb-ipmi-oem/commit/0d3d62544f6951d301fb8fcdbf5f644f3b2adcc3) | 作者: Potin Lai (Quanta) | 合并: 2026-06-12 | 受影响平台: Catalina, Clemente

**技术详情**:
- **原始代码 bug**:
  ```cpp
  // 原代码 (错误):
  if (bDouble / std::pow(10, rExp + minInt4 - 1) > bDouble)
  ```
  存在两个严重问题：
  1. **比较对象错误**: 使用了 `> bDouble`（与自身相比），而不是检查是否超过 10 位整数的容量 `maxInt10`（511）。
  2. **指数符号错误**: 使用了 `minInt4` (-8) 而不是 `maxInt4` (7) 来计算 B 值缩放最大范围。由于 `pow(10, rExp + minInt4 - 1)` 始终是一个极小的小数（10⁻⁹ 量级），任何正数 bDouble 除以该小数都会变得指数级大于自身，导致 `> bDouble` 在循环的第一次迭代就评估为 `true`。

- **修复**:
  ```cpp
  // 修复后:
  if (fabs(bDouble / std::pow(10, rExp + maxInt4)) > maxInt10)
  ```
  正确检查 B-value 偏移量的最小可能缩放幅度是否溢出 10 位整数表示。

- **受影响的数据集**（原代码失败 → 修复后正确计算）:

  | 数据集 {Min, Max}  | 单位 | 原代码结果 | 修复后结果 |
  |-------------------|:----:|:----------:|:----------:|
  | {-0.1, 100.0}     | Other | ❌ Fail | m=392, rExp=-3, b=500, bExp=2, signed=1 |
  | {1, 100.0}        | Other | ❌ Fail | m=388, rExp=-3, b=100, bExp=1, signed=0 |
  | {30000.0, 120000.0} | Pa | ✅ m=352, b=300, bExp=2 | ✅ 同上 |
  | {-128.0, 127.0}   | Cel | ✅ | ✅ |

- **验证**: `ipmitool sdr list` 命令在 Catalina & Clemente 平台上完成无错误。

## 4️⃣ 🔋 bmcweb: 新增 ChargeAh 电量传感器 Redfish 支持（BBU 电池备电）

**来源**: [bmcweb commit `dbc915a65b`](https://github.com/openbmc/bmcweb/commit/dbc915a65b3fb0ddd07f9bcac83a154179debbd5) | 作者: Jagpal Singh Gill | 合并: 2026-06-12

**技术详情**:
- **新增功能**: 在 bmcweb 的传感器解析层中新增 `ChargeAh` ReadingType 和 `Ah`（安时）单位映射，支持 D-Bus 路径 `/sensors/charge/` 下的电量传感器在 Redfish 中正确暴露。
- **应用场景**: BBU（Battery Backup Unit）备电单元的剩余电量（Remaining Charge Ah）和满充电量（Full Charge Ah）监控。
- **Redfish 输出示例**:
  ```json
  {
    "@odata.id": "/redfish/v1/Chassis/BBU_SHELF_1/Sensors/charge_BBU_SHELF_1_BBU5_REMAINING_CHARGE_AH",
    "Id": "charge_BBU_SHELF_1_BBU5_REMAINING_CHARGE_AH",
    "Reading": 10477.0,
    "ReadingType": "ChargeAh",
    "ReadingUnits": "Ah",
    "Status": { "Health": "OK", "State": "Enabled" }
  }
  ```
- **测试验证**:
  - Redfish Validator 通过（无合规告警）
  - `curl` 查询传感器成功返回读数
- **影响**: 对于 AI 集群/数据中心场景中的 **BBU Shelf 备电单元**，BMC 首次提供原生 Redfish 电量监控通道。运维人员可以通过 Redfish 统一 API 获取 BBU 电池的健康状态和剩余容量，无需依赖 IPMI OEM 命令或 SNMP OID。

## 📊 趋势观察

1. **安全合规驱动加固** — 条目 1（64位证书序列号）和条目 2（禁用用户即时注销 session）共同反映了 BMC 软件栈向更严格安全基线靠拢的趋势。testssl.sh、Redfish Validator 等外部工具正倒逼 BMC 修复长期遗留的安全细节。

2. **传感器类型持续扩展** — 条目 4（ChargeAh 电量传感器）表明 Redfish 传感器类型覆盖范围从传统的温度/电压/电流向电池备电、功率等更丰富的维度扩展，支撑数据中心基础设施的精细化监控。

3. **IPMI SDR 算法修复验证产线反馈** — 条目 3（B-value 溢出）这种底层的 IPMI SDR 传感器属性计算错误，在实验室测试中可能不会暴露（因为常用 0-min 数据集），但在产线实际数据（非零 min）下才会触发表格化的失败模式。这再次说明 BMC 软件测试不仅需要单元测试，更需要产线真实数据集的回归验证。
