## 1️⃣ 🏭 NVIDIA GPU PowerCap — Redfish 控制链路贯通 🆕🔥

| 元数据 | 值 |
|:------|:----|
| **仓库** | `openbmc/dbus-sensors` |
| **提交** | `7ea462e3` (committed 2026-06-24) |
| **作者** | Ender Hsieh (NVIDIA) |
| **标题** | `nvidia-gpu: support PATCH for PowerCap and PowerCapEnable` |

**核心改动**: `dbus-sensors` 的 NVIDIA GPU 传感器插件新增对 `PowerCap` 和 `PowerCapEnable` 属性的 **PATCH 写入支持**，打通了 Redfish → D-Bus → NSM → GPU 的完整功耗控制链路。

**架构设计要点**:

1. **写入链路**: Redfish `PATCH /redfish/v1/Chassis/{id}/Power` 中的 `PowerLimitWatts.SetPoint` → bmcweb 转发 → D-Bus `xyz.openbmc_project.Control.Power.Cap` → nvidia-gpu 插件 → NSM `SetPowerLimits (0x08)` 命令 → GPU 设备
2. **Debounce 防抖**: `PowerCap` 和 `PowerCapEnable` 两次写入合并为一次 `SetPowerLimits`。每个 setter 记录请求并启动短时 debounce timer；burst 平息后一次下发。飞行中请求被推迟到上次响应到达后再处理
3. **悲观读取**: 不乐观地更新 D-Bus 暴露值。`PowerCap`/`PowerCapEnable` 仅从 `GetPowerLimits` 响应更新（周期轮询 + Set 完成后的重新读取）。失败的写操作不留脏数据

**配套提交** `d41f8cb5` (`nvidia-gpu: align PowerCap and PowerCapEnable with PDI`) 修复了 PDI 映射不匹配问题：
- `PowerCap` 来自 one-shot requested limit（而非 firmware-enforced limit）
- `PowerCapEnable` 在 one-shot requested limit 匹配 enforced limit 时为 true
- 冷启动默认状态：`PowerCap=maxint` + `PowerCapEnable=false` → 映射为 Redfish `Disabled` 状态

**意义**: AI 服务器 BMC 首次实现 GPU 功耗上限的红鱼全链路控制，对**智算集群功耗封顶、动态功耗调度**场景至关重要。

## 2️⃣ 🔌 phosphor-power PCP — 系统 PGood 检查 + GPIO 监控

| 提交 | 日期 | 作者 | 描述 |
|:----|:----:|:----|:------|
| `a4264e27` | 06-24 | Brendan Singer | `PCP add system pgood check` — 电源控制平面查询 chassis 的 Power Good 状态 |
| `33741419` | 06-24 | Brendan Singer | `PCP Implement GPIO monitoring` — 1秒轮询 chassis GPIO，两次连续相同值后更新（去抖） |
| `482ccc3d` | 06-24 | — | `chassis-psu: Refactor chassis and chassis manager` — 支持单/多 chassis 识别、位置ID 增强 |

**要点**:
- **PGood 检查**：通过 chassis status monitor 的 `getPowerGood()` 函数查询整机是否已上电，对 PCP（Power Control Plane，电源控制平面）状态机完整性至关重要
- **GPIO 监控**：Chassis GPIO 以 1s 周期轮询，含两级去抖逻辑（2 consecutive reads），避免瞬态误报
- **Chassis 重构**：`ChassisManager` 增加多 chassis 场景识别（single/multi chassis），位置 ID 使用 `getParentEMPositionId()`，新增 `addChassisToList()` 方法提取创建逻辑

**配套提交** `79b43001` 为 IBM Huygens 平台增加 UCD 设备和全部电压轨配置。

| **提交** | `53deac89` (committed 2026-06-24) |
| **作者** | Joel Stanley |
| **标题** | `linux-aspeed: Inspur fp5280g2, ADM1266 driver` |

**内容**: 3 个 patch 的组合提交：
1. `ARM: dts: aspeed: inspur-fp5280g2: Add GPIO line names` — 浪潮 fp5280g2 平台 GPIO 引脚命名

## 4️⃣ 🧹 phosphor-logging — JSON 序列化迁移启动

| 提交 | 日期 | 作者 | 描述 |
|:----|:----:|:----|:------|
| `ad67076c` | 06-24 | Patrick Williams | `log-manager: deserialize from JSON` — 从 JSON 反序列化日志条目（Cereal 回退） |
| `0d5a895a` | 06-24 | Patrick Williams | `log-manager: restore JSON files and paths` — 恢复 JSON 文件与路径 |
| `c9f021ef` | 06-24 | Patrick Williams | `log-manager: use JSON-specific directory` — 使用 JSON 专用目录 |

**核心设计**:
- 新版本优先从 JSON 反序列化，如果 JSON 文件不存在则回退到 Cereal 格式
- BMC 升级时自动将 Cereal 文件升级为 JSON
- BMC 降级后 Cereal 格式仍可正常读取；再次升级时重新从 Cereal 生成 JSON
- 日志条目 D-Bus 的 `File.Path` 属性始终指向 JSON 路径
- 目录结构示例：`errors-json/4280095994.json`

**意义**: 从专有的 Cereal 二进制序列化向 JSON 文本格式迁移，提升**可读性、可调试性和跨版本兼容性**。
