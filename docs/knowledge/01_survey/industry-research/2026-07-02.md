### 8.1 🟢 NVIDIA官方博客 — CuTe DSL融合MoE内核（2026-06-15）

| 维度 | 内容 |
|:-----|:-----|
| **来源** | NVIDIA Technical Blog |
| **URL** | https://developer.nvidia.com/blog/boosting-moe-training-throughput-with-advanced-fusion-kernels/ |
| **作者** | Rachit Garg & Matthew Nicely |
| **发表** | 2026-06-15 |

**核心方法**:
用CuTe DSL重写MoE融合MLP内核，系统性拆除MoE block三大瓶颈：

1. **Activation Fusion**：将SwiGLU/GeGLU/sReLU激活函数融合进GEMM epilogue。关键技巧：将Input和Gate权重按列重排（repack），确保同一thread block同时访问二者，消除中间张量的global memory读写
2. **Sync-Free CUDA Graphs**：GroupGEMM内核在GPU内存内部自主追踪每个group的token数，消除CPU launch依赖。传统方法需CPU获取shape信息后launch动态GEMM，新方案使整轮迭代进入CUDA Graphs零同步执行
3. **Quantization Fusion**：MXFP8/NVFP4量化步骤融合进GEMM kernel，消除BF16→低精度整型的额外memory pass

**支持功能**: Dynamic Scheduling（预留SM margin供通信kernel并发）· 支持cuDNN Frontend v1.23.0+ / Transformer Engine v2.15+ / Megatron Core 26.04-alpha+

**性能数据**:
| 指标 | 数值 |
|:----|:-----|
| Kernel Forward speedup | **1.3×** (vs 未融合baseline) |
| Kernel Backward speedup | **2.1×** (vs 未融合baseline) |
| DeepSeek-V3 端到端 | **+8%** throughput |
| GPT-OSS 端到端 | **+93%** throughput |

**硬件影响 ⭐⭐⭐⭐⭐**:
- 最值得关注的不是1.3× kernel加速，而是 **sync-free CUDA Graphs的范式意义**："将判断逻辑下沉到GPU内存"可推广到其他动态workload
- GPT-OSS这种dense模型在MoE优化中意外+93%，说明融合kernel的通用价值远超MoE本身
- 表明NVIDIA从软件层面解决了MoE训练中CPU-GPU同步的经典瓶颈

### 8.2 🟡 arXiv:2606.26607 — Moebius：运行时无缝TP/EP切换（2026-06-25）

| 维度 | 内容 |
|:-----|:-----|
| **来源** | arXiv (cs.DC) |
| **论文编号** | 2606.26607 |
| **机构** | UIUC |
| **提交** | 2026-06-25 |

**核心方法**:

MoE推理服务中，**TP（Tensor Parallelism）和EP（Expert Parallelism）在不同并发度下互有优劣**：TP低并发快，EP高并发快。生产workload跨越这个边界是常态——在线服务突发→平息，RL rollout高并发开始→长尾散兵。

Moebius的关键洞察： **EP和TP是同一模型的两个布局，不是两个模型**——它们在字节相同的expert weights和KV cache上计算相同函数，切换只改变谁拥有哪个slice。因此**唯一不可约成本是迁移那些"所有者变更"的slice**，而现代高带宽GPU互联足以在decode步之间完成。

**技术实现**:
- 保持每种并行度的runtime resident
- 用fused GPU-to-GPU transfer kernel在固定地址对单份expert weights和KV cache做reshard
- 不需要drain in-flight requests

**性能数据**:
| 指标 | 数值 |
|:----|:-----|
| 测试平台 | 8× H200 GPUs |
| 模型 | Qwen3-235B-A22B |
| 切换开销 | **215–434 ms** |
| 内存开销 | 仅 **2.4%**（同时保有TP和EP两个布局） |
| RL rollout吞吐提升 | **1.16–1.25×**（全steps平均） |
| 静态最优对比 | 每个operating point匹配更优并行度 |

**硬件影响 ⭐⭐⭐⭐⭐**:
- 直接利用了NVLink的高带宽特性使"运行时切换"成为实用方案
- 2.4%内存开销持两个布局表明HBM充裕度的工程重要性
- 对RL post-training场景有显著收益——未来RL scaling law驱动下，Moebius类方案将成为MoE serving标配

### 8.3 🔴 arXiv:2603.08960 — qs不等式：量化MoE推理双重惩罚（2026-03-09）

| 维度 | 内容 |
|:-----|:-----|
| **来源** | arXiv (cs.LG/cs.AR/cs.DC/cs.PF) |
| **论文编号** | 2603.08960 |
| **机构** | AMD Research |
| **提交** | 2026-03-09 |
| **已收录** | —（预印本） |

**核心方法**:

AMD研究团队首次从体系结构角度正式量化MoE在推理阶段的**双重惩罚（double penalty）**：

1. **第一重惩罚：微批次碎片化**
   Expert routing将microbatch打散到不同expert，减少weight reuse，把原本计算密集的FFN推入**带宽受限**（bandwidth-bound）状态

2. **第二重惩罚：HBM竞争**
   巨大的resident expert池（仅部分激活）占满HBM，挤占KV cache头部空间。长上下文场景下，KV cache不足进一步放大延迟

**qs不等式**：一个统一判据，结合稀疏度 $s$（每token激活参数比例）和质量等价因子 $q$（dense模型需多大才能在质量上匹配MoE），判断MoE何时在推理中结构性劣于dense模型。

**性能数据**：
| 模型 | 上下文 | 结果 |
|:----|:-------|:----|
| **DeepSeek-V3** @ 128k context | 4.5× throughput优势 → **quality-matched dense baseline** |
| **Qwen3-235B** @ 128k context | 结构化验证通过，一致的趋势 |
| **Grok-1** | 双重惩罚存在于所有评估模型 |
| **Switch-C** | 大规模模型可能在某些集群规模下不可行，而quality-matched dense仍可部署 |

**硬件影响 ⭐⭐⭐⭐⭐**:
- 来自AMD的客观分析，打破了"MoE训练高效=推理高效"的业界共识
- 建议 **MoE应被视为训练时优化，蒸馏到dense模型可能是推理时高效部署的出路**
- 对数据中心硬件规划者：**不要因MoE训练FLOPs效率而低估其推理基础设施需求**
- qs不等式可作为硬件采购/集群配置时**判断MoE vs dense最优部署方案的决策工具**

### 9.2 ⭐ AMD Versal Premium Gen 2 — HBM 供应枯竭迫使架构大转弯 🆕（2026-06-30）

**来源**: ServeTheHome (Ryan Smith)
**URL**: https://www.servethehome.com/amd-pivots-from-hbm-to-lpddr5x-for-new-versal-premium-gen-2-memory-on-package-chips/
**硬件影响评级**: ⭐⭐⭐⭐（HBM 供应链脆弱性的清晰信号）

**剧情反转**:

AMD（原 Xilinx）早在 2018 年就推出了 HBM FPGA（Virtex UltraScale+ HBM），后升级为 Versal HBM（32GB HBM2e）。但在 2025 年 9 月，AMD 因 HBM2e 供应枯竭**不得不停产 Versal HBM 系列**。

**(插曲)** 2026年6月30日恰为 Versal HBM **最后下单日**。

**转折方案**: AMD 推出 **Versal Premium Gen 2 Memory on Package**，改用 **LPDDR5X 片上封装**。

**关键数据**: 三大 HBM 制造商（SK hynix/Samsung/Micron）都在快速终止 HBM2e 产线，转向 HBM3/HBM4。

**意义**:
- HBM 的供应周期远短于传统 DRAM——一款 HBM 产品只有 2-3 年供应窗口
- AMD 自家 FPGA 部门都拿不到 HBM——这对于计划完全依赖 HBM 的服务器架构规划是一个**严重警告**
- LPDDR5X 作为替代方案，尽管绝对带宽低于 HBM，但**供应充足 + 封装更具柔性**

**与 Qualcomm HBC 的关联**: 两家公司同日释放了同一个信号——**HBM 依赖在 AI 服务器领域可能是结构性的阿喀琉斯之踵**。
