### 🅰 运行时并行度切换：Moebius 打破 TP 与 EP 的非此即彼

**来源**: [Moebius: Serving MoE Models with Seamless Runtime Parallelism Switch](https://arxiv.org/abs/2606.26607)（一手，arXiv cs.DC, 2026-06-25）

**发现**: MoE serving 中 Tensor Parallelism (TP) 和 Expert Parallelism (EP) 各有最优域——TP 在低并发时更快，EP 在高并发时更好。生产负载持续跨越此边界（在线服务突发→安静期、RL rollout 高并发→tail straggler）。Moebius 在运行时无需重启或丢弃请求即可在 EP↔TP 间切换：
- 8×H200 Qwen3-235B-A22B：**每个切换点匹配更好的静态并行度**
- RL rollout 全周期 **1.16-1.25× 加速**
- **切换时间 215-434ms**（decode step 之间完成，不阻塞请求）
- 双布局常驻内存开销仅 **2.4%**
- 关键洞察：EP 和 TP 是同一模型的两套布局（byte-identical weights + KV cache），切换只改变 rank 拥有的 slice

> **硬件影响 ⭐⭐⭐⭐**: (1) 生产 MoE serving 不再需要在 TP/EP 间二选一构建时固定——运行时切换可行；(2) H200 的高带宽互联（NVLink）使切换足够快（200-400ms）；(3) **对下一代 GPU 互联的设计启示**：更高带宽、更低延迟的 scale-up fabric 将加速布局切换，使运行时自适应并行度成为标配能力；(4) RL 训练中 rollout 阶段作为受益场景，与 ForeMoE 的 micro-step 级波动发现互补。

### 🅱 235B MoE→双 A100 的 75% 内存压缩：硬感知压缩方法论

**来源**: [Agentic evolution of physically constrained foundation models](https://arxiv.org/abs/2606.25532)（一手，arXiv cs.AI/cs.AR, 2026-06-24）

**发现**: 基于进化知识图谱的多智能体自动发现引擎，演化出两种物理感知的压缩方法：
- **MoE-Salient-AQ**：利用 Sensitivity Profile（带宽高效的激活显著性映射）选择性压缩——sub-3-bit 低比特率下 **比 SOTA 手动设计的 MoE 量化方案好 3.7%**
- 235B 级 MoE 模型成功部署到 **受限的双 A100（160GB 合计）**：**75% 内存缩减，仅 0.64% 准确率下降**
- 方法论亮点：自动生成的 CoT 将盲搜索转变为有向结构演化

> **硬件影响 ⭐⭐⭐**: (1) 双 A100 运行 235B MoE 是硬件边界突破的实证——验证了 MoE「用稀疏性换内存密度」的推理部署可行性；(2) Sensitivity Profile 作为「带宽高效的硬件感知策略」可直接指导服务器端的 KV Cache/Expert 放置决策；(3) sub-3-bit MoE 量化在 3.7% 领先幅度下走向实用化，对硬件支持的低精度计算单元提出了更强需求。

### 🅲 35B MoE 在 2011 Fermi GPU 上运行的极端案例分析

**来源**: [A 35B Hybrid-Attention MoE Model on a 6GB 2011 GPU](https://arxiv.org/abs/2606.24031)（一手，arXiv cond-mat.other, 2026-06-23）

**发现**: 在 **2011 年 NVIDIA Tesla C2075**（Fermi, 6GB, 无 Tensor Core, 无 FP16）上完成 Qwen3.6-35B-A3B（~3B 活跃参数）的端到端推理：
- **混合执行策略**：GPU 做 prefill（逐层 streaming expert weights from host RAM）→ CPU 做 decode（手写 W4A8 SSSE3 pmaddubsw GEMV）
- 经过 expert pinning + single-pass prefill + NUMA interleaving：prefill 延迟 **57.2s → 37.5s**
- Decode 吞吐：**2.8 → 8.6 tps**（整数 SIMD kernel, ~3×）
- 负结果同样有价值：GPU offload head、hyper-threading、三版 GPU kernel 改写全部无效

> **硬件影响 ⭐⭐⭐**: (1) MoE 的稀疏性（35B 总参 / 3B 活跃）是此极端部署可行的根本原因——dense 35B 模型不可能装入 6GB；(2) **GPU prefill + CPU decode 的混合方案验证了异构执行的新范式**——与芯片架构中「prefill 加速器 + decode 加速器」分立的趋势一致；(3) 负结果（GPU head offload 失败）说明在带宽受限场景下，CPU decode 比 GPU 更有优势；(4) 对超节点设计的启示：MoE 在极端资源约束下的部署思路可用于边缘/嵌入式场景。

## ① GPU/AI 芯片竞争格局

### 📐 EmuGEMM: Hopper/Blackwell Tensor Core 精度仿真性能实测（arXiv:2606.25453, cs.DC, 6/24）

> **来源**: arXiv:2606.25453, Denghui Lu 等, ETH Zurich / CSCS
> **核心发现**: 使用融合整数Tensor Core内核消除冗余内存往返，首次给出Blackwell和Hopper的INT8 Tensor Core接近峰值利用率实测数据。

| 指标 | NVIDIA Hopper (H100) | NVIDIA Blackwell (B200) |
|:-----|:--------------------:|:-----------------------:|
| EmuGEMM Scheme I 持续吞吐 (INT8) | 1,639 Top/s | **3,654 Top/s** |
| 峰值利用率 (% of INT8 peak) | 83% | 81% |
| vs cuBLAS TF32 吞吐提升 (大矩阵) | up to 1.4× | **up to 1.7×** |
| vs cuBLAS ZGEMM (复数, Scheme II) | up to 2.3× | **up to 5.5×** |

- **技术要点**: Ozaki Scheme I/II 通过低精度操作重构高精度GEMM；EmuGEMM通过**消除全局内存中间结果往返**实现性能飞跃
- **意义**: 表明Blackwell INT8 Tensor Core峰值~4,511 Top/s (3,654/0.81)；Hopper ~1,974 Top/s (1,639/0.83)。在科学计算(需要高精度)场景下，EmuGEMM提供了一条无需FP64硬件即可接近FP64精度的路径
- **对竞争格局影响**: 验证了NVIDIA在低精度矩阵计算上的压倒性优势，但同时也说明Blackwell在高精度计算上依赖软件仿真来弥补FP64硬件的退化

### 🎯 B200 NV-HBI跨Die惩罚实测: 30-cycle / 15.5ns (arXiv:2606.24934, cs.CR/cs.AR, 6/22)

> **来源**: arXiv:2606.24934, Faruk Alpay 等
> **核心内容**: 基于CUDA探针的无特权云GPU认证方案，首次以软件方法**无侵入测量**了B200两Die NV-HBI封装的跨Die延迟

关键实测数据:
| 指标 | 数值 |
|:-----|:----:|
| B200 SM分区 | 74/74 (双Die对称) |
| 跨Die (Die0→Die1) 延迟惩罚 | **30 cycles, 15.5 ns** |
| 同一Die内SM延迟中位抖动 | 0.09 cycles (6小时满载RTX 5090) |
| Blackwell Die分类精度 (leave-one-out) | **100.0%** |
| H200 L2缓存拆分 | 双路L2拆分 |
| V100 统一内存域 | 单域统一 |

- **技术意义**: 跨Die 15.5ns延迟惩罚是Blackwell双芯片封装架构的关键性能代价，对于需要频繁跨Die通信的MoE模型路由、AllReduce集合操作有直接影响
- **安全意义**: 该技术可在无供应商密钥的情况下验证云GPU身份、类别和地理位置，对云GPU市场的信任机制有潜在价值

### 🅲 IEEE Spectrum: 功率缓冲器保护电网免受数据中心负载突变冲击

**来源**: [Power Buffer Protects Grid From Data Centers' Wild Load Swings](https://spectrum.ieee.org/tag/power-electronics) (IEEE Spectrum, 2026-04-29)

**核心要点**:
- 数据中心（尤其 AI 训练集群）存在剧烈的**负载波动**——训练任务启动/检查点/梯度同步间的功耗可以在秒级内变化数百 kW

- 双重价值：
  - **加速电网接入审批**——传统上数据中心的大幅波动被电网运营商视为风险，加装缓冲区可显著降低审批门槛
  - **降低对电网基础设施的冲击**——减少数据中心对周边电网的电压/频率扰动，降低与居民用电的冲突风险

> **供电架构影响 ⭐⭐⭐**: 功率缓冲器作为 **AI DC 供电的「第五层」**（Grid → HVDC → Rack → PSU → Buffer → Load）开始受到关注。对机架级 BBU（备用电池）的设计启示：BBU 不仅作为断电保护手段，也可作为 **UPS + 功率缓冲器**的双功能器件，在平时吸收负载波动、在断电时提供备电。

### 🅴 原材料涨价：Tungsten/WF₆ 短缺蔓延

**来源**: [Tungsten shortage spreads from raw material to chip manufacturing, Japan warns](https://www.digitimes.com) (DIGITIMES, 2026-06-26 最新头条)

**核心要点**:
- **钨（Tungsten/WF₆）短缺**已从原材料端蔓延至芯片制造环节，日本政府发出警告
- WF₆（六氟化钨）是 **半导体 CVD/WD 工艺中钨互连沉积**的关键气体，用于 **Via/W plug 填充**、**接触孔金属化**
- Tungsten 在 **先进制程（3nm/2nm）** 中的用量更大——线宽缩小后，Contact/Via 孔深径比增加，需要更厚的钨沉积
- 短缺根因：
  1. **中国钨矿供应收紧**（中国占全球钨产量的 80%+，出口管制升级）
  2. **AI 芯片需求暴增**推动晶圆开工率 → 耗材消耗加速
  3. **钨冶炼产能扩张滞后**（新建氟化/还原线周期 18-24 个月）
- 已观察到的影响：
  - 日本半导体材料商被迫寻找替代供应源
  - Tungsten 靶材/气体价格上行压力传导至晶圆厂

> **BOM 影响 ⭐⭐⭐**: 钨/WF₆ 短缺对 BOM 的直接影响（占芯片成本比例低）有限，但**二次影响显著**——可能延后先进制程产能爬坡，间接推高 TSMC 3nm/2nm 代工报价，最终提高 GPU/ASIC 的晶圆成本。
