## 📌 1. Atlassian Cursor in Jira 正式发布（2026-05-20）— Jira 成为 Agent 编排平台

**来源**: [Atlassian Blog - Introducing Cursor in Jira](https://www.atlassian.com/blog/announcements/cursor-in-jira)

Atlassian 正式发布 **Cursor in Jira**，将 Jira 定位为 **Agent 编排平台**（agent orchestration platform）：

**关键能力**：
- **直接分配工作给 Agent**：在 Jira 中将工作项分配给 `@Cursor`，或在评论中 @Cursor 启动 Agent 会话
- **自动化工作流**：创建规则自动将特定任务分配给 Cursor，处理重复性工作并提升代码质量
- **全员参与**：任何团队成员都可使用 Cursor 启动任务并提交 PR，无需搭建本地开发环境
- **Spec-driven 开发**：将 Agent 可读的规格说明自动同步到 Confluence 协作文档、工作项、计划和目标
- **完整的审计追踪**：Cursor 的每个 Action 都在 Jira 中记录，管理员控制哪些 Agent 在哪些项目运行、何时运行

**关键数据**：
- **Teamwork Graph CLI 实测**：利用 Teamwork Graph 上下文的 Agent，**答案质量提升 44%**，**Token 消耗减少 48%**（Atlassian 内部测试）
- **Cisco 工程运营 VP Jason Andrews** 评价："我们不是在问如何让工程师使用更多 AI，而是如何构建一个人类和 Agent 共享同一上下文、朝着同一目标的系统"
- **Atlassian CPO Tamar Yehoshua**："我们看到当团队给 Agent 尽可能多的上下文时，他们交付最快。Jira 是定义工作的地方，工程师在 Cursor 中构建。现在这些上下文可以轻松共享，Jira 成为连接这一切的编排层"

**意义**：标志着项目管理工具从"人用的工作跟踪系统"向"人+Agent 协同工作编排平台"的根本转型。Jira 不再是单纯的状态跟踪工具，而是 Agent 任务的分配器、上下文的提供者和审计的记录层。

## 📌 2. Atlassian Teamwork Collection 更新：Agents in Jira GA + Rovo 多模态 Agent 协作（2026-05-06）

**来源**: [Atlassian Blog - Built for the Next Era of Teamwork: What's New in Teamwork Collection](https://www.atlassian.com/blog/announcements/built-for-the-next-era-of-teamwork-whats-new-in-teamwork-collection)

**Agents in Jira（GA）**：
- Agent 可以直接**承接任务所有权**：分配 issue、更新代码（如 bug 修复）、在状态变更时自动被触发
- 支持 **Studio 第一方 Agent + 第三方 Agent**（Amplitude/Canva/Cursor/Figma/Gamma/GitHub Copilot 等）
- 每个 Agent 动作都有**完整的审计追踪**，管理员控制哪些 Agent 运行在哪里

**Confluence 第三方 Agent（Open Beta）**：
- 在 Confluence 页面中 `@mention` Agent，就像 @同事一样
- 首批支持：Lovable、Replit、Databricks、Gamma
- Agent 读取页面上下文后，可跨连接工具执行操作

**Create with Rovo in Jira（Beta）**：
- 自动将 Confluence 文档、会议摘要、邮件线程转为**结构化 Jira 工作项**
- 团队启动速度**提升最多 30%**，减少协调开销

**Loom Agent Briefings（Coming Soon）**：
- 多模态输入：录屏演示需求/设计/反馈 → 自动翻译为**结构化 Agent Prompt**
- 生成行动方案 → 一键转为 Jira 工作项
- Bug Report with Loom + Jira GA：自动捕获设备信息、console 日志、网络数据，打包为 Jira ticket 可分配给 Rovo Dev 自动修复

**Remix with Rovo（Beta）**：
- 选中 Confluence 页面上任意内容 → 一键转为**图表、时间线、信息图、思维导图**
- 包含可视化元素的 Confluence 页面，阅读率几乎是纯文本页面的 **2 倍**

## 📌 3. Linear Team Documents + Code Intelligence + Diffs — 团队知识空间与 Agent 上下文引擎（2026-05~06）

**来源**: [Linear Blog](https://linear.app/blog)

**Team Documents（2026-06-03）**：
- 每个团队拥有**专属知识空间页面**，集中存放 notes、docs、共享参考资料
- 解决"重要团队上下文不一定属于某个具体 issue/project/initiative"的问题

**Code Intelligence for Linear Agent（2026-05-14）**：
- Linear Agent 可以**读取代码库**，直接从源码层面回答问题
- 将仓库转化为**共享产品上下文**，整个团队可用

**Linear Diffs（2026-05-27）**：
- 代码审查直接嵌入 Linear，Agent 驱动代码审查范式变革
- **核心判断**："Code review should be fast. Agents are forcing a change in how we review code." — Tuomas Artman
- Agent 生成大量代码，但个人仍对合入负责，传统审查工具能力跟不上

**Issue Tracking is Dead（2026-03-24）— 范式宣言**：
- **核心论点**：下一代产品开发建立在**上下文（Context）和自主性（Agency）** 之上
- Linear 从 issue tracker 进化为产品上下文+Agent 协作平台
- Ramp 案例：内部 Coding Agent 完成 **60%+** 的合并 PR，以 Linear 作为底层产品上下文层

## 📌 4. Product School：AI Agent 决策框架 + AI 实验方法论（2026-03）

**来源**: [Product School Blog](https://productschool.com/blog)

### AI Agent 决策框架（2026-03-03）
- **Gartner 预测**：到 2027 年，超过 **40%** 的 agentic AI 项目将被取消
- **MIT NANDA 报告**：95% 的组织从 GenAI 项目中获得**零 ROI**
- **四大自主性阶段**：Reactive（无自主）→ Tool-using（辅助自主）→ Semi-autonomous（人机协作）→ Fully autonomous（目标驱动）
- **推荐**：大多数团队应长期停留在 B（辅助）和 C（人机协作）阶段
- **Agent 适用性评分矩阵**：17-20 分=适合带审批的 Agent，21-24 分=适合带护栏的 Agent，0-8 分别用 Agent
- **核心警告**：Prompt Injection 是 OWASP #1 LLM 风险；不要让能采取行动的 Agent 自由读取不受信任的输入

### AI 实验方法论（2026-03-18）
- **五类实验方法**：A/B 测试、多变量测试、假门测试（fake-door）、AI 原型用户测试、金丝雀发布
- **关键原则**：实验期间**冻结模型版本**，对非确定性输出设种子
- **指标四象限**：模型性能（accuracy/latency）× 用户体验（使用率/满意度）× 业务影响（转化率/成本节约）× 信任与公平（偏差率）

## 📌 5. HBR：AI Augmentation 六阶段增长路径 vs Automation 六阶段衰退路径（2026-04-15）

**来源**: [HBR - Why Companies That Choose AI Augmentation Over Automation May Win in the Long Run](https://hbr.org/2026/04/why-companies-that-choose-ai-augmentation-over-automation-may-win-in-the-long-run)

**关键数据**：
- 仅有 **44%** 的员工所在组织正式宣布了 AI 计划
- **62%** 的员工相信组织用 AI 增强（Augmentation），**34%** 怀疑是用来自动化（Automation）裁员
- **60%** 的员工对 AI 导致失业表示担忧
- 高管层认为组织"全力支持增强"的比例为 **81%**，但基层员工仅 **53%** 认为如此（**28pp 认知差距**）
- 感知到增强意图的员工离职意愿比感知到自动化意图的低 **32%**

**六阶段 J-curve 模型**：
| 阶段 | Automation 路径（衰退循环） | Augmentation 路径（增长循环） |
|:-----|:----------|:--------|
| 1 | 早期抵制 AI 采纳 | 信任加速 AI 采纳（培养"驾驶员"而非"乘客"） |
| 2 | 裁员破坏幸福感→生产力下降 | 持续幸福感提升生产力（快乐员工生产力高 13%） |
| 3 | 缩编团队过载→Workslop 泛滥 | 团队发展新能力→Workslop 最小化 |
| 4 | 流失上升→高绩效者先走 | 留任加强→机构知识积累 |
| 5 | 雇主品牌下降→人才吸引困难 | 雇主品牌成为人才磁铁→良性循环 |
| 6 | 领导力断层+文化侵蚀 | 领导力管道深化+文化强化 |

**实践案例**：
- **Aon（60,000 员工）**：CEO Greg Case 承诺 AI 素养提升而非裁员，疫情期间曾公开承诺不裁员，高层自愿减薪——用历史行为建立信任
- **Microsoft Nadella（2014）**：从 "know-it-all" 到 "learn-it-all" 文化转型，大量投资再培训而非自动化裁员

## 📋 综合趋势判断

1. **Jira→Agent 编排平台**：Atlassian 与 Cursor 的合作标志着项目管理工具从"工作跟踪"到"Agent 编排"的范式转变。Jira 成为人+Agent 的共享工作空间，Teamwork Graph 作为上下文引擎，44% 质量提升 + 48% Token 节省的数据证明了上下文的价值
2. **Linear 的"Context + Agency"路线**：Linear 从 Issue Tracking 转型为产品上下文平台，强调"issue tracking is dead"，Agent 需要的是上下文而非状态追踪
3. **AI 实验方法论成熟化**：Product School 提供了可操作的实验框架（五类实验 + 四象限指标 + Agent 适用性评分），强调"人类仍然拥有判断权和责任"
4. **增强 vs 自动化路线选择**：HBR 的六阶段 J-curve 模型提供了清晰的决策框架——自动化路径短期省成本但长期侵蚀竞争力；增强路径短期投入大但长期回报更高。Augmentation 感知带来 32% 更低的离职意愿
