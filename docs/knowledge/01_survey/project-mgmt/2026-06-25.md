### 1.1 教 AI 学会你的决策方式 — 知性输出的最大瓶颈

**来源**: [Teach Your AI How You Make Decisions — HBR](https://hbr.org/2026/06/teach-your-ai-how-you-make-decisions) (2026-06-25)
**作者**: Jen Stave (HBS AI Institute 创始主任), Ryan Kurt (The AI Lab CEO), John Winsor (HBS AI Institute)

**核心论点**：

> 随着 AI Agent 承担越来越复杂的工作，关键约束已不再是技术接入，而是组织将自己的决策过程显式化的能力。

许多公司用 AI Agent 做了真正的流程转型，但更多公司陷在小规模实验中，无法让 Agent 在规模化场景中稳定表现。

**「AI-readiness」的核心指标发生变化**：

| 传统视角 | 新视角 |
|:---------|:-------|
| 你有多少数据？ | 你的决策逻辑是否已显式化？ |
| 你有多少算力？ | 你的隐性知识能否被 AI 理解？ |
| 你有多少 AI 工具？ | 你的团队能否清晰地描述决策流程？ |

> **核心框架**：公司需要将隐性决策原则（tacit decision principles）转化为结构化指南（structured guidance）供 Agent 使用。这不是技术问题，而是组织设计问题——**「AI-readiness 取决于你能多好地描述自己如何做决策」**。

**对 PM 的含义**：
- Agent 编排的成功取决于工作流上下文的结构化程度
- 引入 Agent 前，先回答「这件事的决策逻辑是什么？输入是什么？输出标准是什么？」——如果人自己都说不清，Agent 更不可能做对
- 这与 Product School 的 Autonomy Budget 框架、Atlassian Teamwork Graph 的「Context 质量决定 Agent ROI」一脉相承

### 1.2 员工不愿质疑 AI 的建议 — 「认知舒适区」陷阱

**来源**: [Employees Aren't Questioning AI Advice Enough — HBR](https://hbr.org/2026/06/employees-arent-questioning-ai-advice-enough) (2026-06-24)
**作者**: Ben Rand, 基于哈佛商学院 Alex Chan 的研究

**核心发现**：

> 新研究发现，人们在面对 AI 的建议时，**常常主动避免寻求额外信息——因为额外信息可能让决策变得更复杂或引发道德不适。**

这意味着：
- **AI 建议的「权威感」在扼杀批判性思考** — 人类默认信任 AI 输出，尤其是当 AI 的推理看似严谨时
- **更糟糕**：即使 AI 的决策过程变得更透明（"show your work"），人们也不会自动变得更挑剔
- **归因**：质疑需要认知努力，而人类大脑有「最小认知努力原则」——如果 AI 的答案看起来合理，大脑就不愿费劲去验证

**对 AI 产品运营模型的含义**：
- **AI 产品运营模型的设计不能只关注「AI 多准确」**，还必须设计「人类如何正确地挑战 AI」
- **具体实操**：
  - 在产品中嵌入「适度摩擦」— 高影响决策要求人类必须主动确认（不仅仅是点「同意」）
  - 团队 AI Working Agreements 中加入「质疑门槛」— 什么情况下必须人工验证 Agent 输出
  - 避免出现「AI 自动决策」的默认闭环——**Autonomy Budget 不仅仅是 Agent 的，也是人类的**

> 与 Atlassian 的「AI 使用披露被评价为 10× 懒散」共同揭示：**AI 产品运营最棘手的问题不是技术，而是人类的心理惯性和组织规范。**

### 1.3 AI 也有「性格」— 支持型 vs 敌对型 AI 对绩效的影响

**来源**: [Does Your AI Have a Personality Problem? — HBR](https://hbr.org/2026/06/does-your-ai-have-a-personality-problem) (2026-06-24)
**作者**: 六位学者（Harvard/Kozminski/Quinnipiac/Yale）

**核心发现**：

> 大多数领导者在评估 AI 时仍像当年 IT 部门评估数据库一样——只关注能力、速度和成本。但研究显示，**AI 的个性和交互风格可能同等重要**。

**研究设计**：
- 对比两种 AI 交互风格对人绩效的影响：
  - **支持型 AI persona**（鼓励性、引导式、承认错误时安全）
  - **敌对型 AI persona**（批评性、强调「你错了」）
- **实验结果**：AI 的行为风格直接影响人的任务表现和信心水平

**对团队协作工具演进的启示**：
- **当你引入 AI Agent 作为「团队队友」时，Agent 的「性格」是一个产品设计决策**
- Agent 成为 Jira 中的一等公民（可 Assign、@Mention、自动化规则驱动）后，**Agent 如何与人类互动——语气、反馈方式、错误处理——成为团队动态的一部分**
- 这超出了传统 PM 的能力范围——需要 **AI 产品运营模型包含「交互风格设计」维度**

## 2️⃣ Atlassian: AI-Native 开发者的真实工作流 —「语境组装 > 代码编写」

**来源**: [How AI is Changing Developer Workflows: Lessons From Engineers — Inside Atlassian](https://www.atlassian.com/blog/ai-at-work/how-ai-is-changing-developer-workflows) (2026-05-27)
**作者**: Abby Stiris

### 2.1 核心发现：开发者的一天已被 AI 重塑

Atlassian 访谈了大量以 AI Agent 为主要实现工具的工程师，发现以下模式：

**① 大部分时间花在「上下文组装」，而非编码**
- 下载 PDF → 粘贴到聊天框 → 打开仓库复制路径 → 找人问问题等回复 → 重读架构文档 → 拼接足够的上下文后才能让 Agent 干有用的活
- **Atlassian State of Developer Experience 2025**：开发者仅 16% 的时间在写代码，其余 84% 是上下文组装类工作
- **结论**：2026 年代码生成很快，但上下文组装很慢——**这个差距是 AI 原生开发中最大的摩擦力来源**

**② 容量上去了，野心也上去了**
- 工程师一致表示：同样的时间内做更 ambitious 的工作
- 更快的循环周期没有缩短工作日，而是转化为更多并行的项目、更多的并发上下文
- **68% 的开发者报告每周节省 >10 小时**——但被重新投入到代码质量提升、新功能开发和更好的文档中

**③ 规划成为人类做的最有价值的事**
- "规划甚至比之前更重要了，因为 AI 不管你说什么都会给出一个自信的答案——这可不总是好事。"
- 没有精心规划的 prompt → AI 产生自信但方向偏移的输出 → 可能直到生产环境才暴露
- **将大需求拆解为定义良好的、AI 可 prompt 的子任务，带回清晰验收标准——这是 PM 的核心价值所在**

### 2.2 信任构建的序列

工程师对 Agent 信任的演进路径：

| 阶段 | 行为 | 当前采纳状态 |
|:-----|:-----|:------------|
| **① Reactive Assistant** | 人类提问，Agent 回答/生成 | 所有人都已到达 |
| **② Proactive Observer** | Agent 主动观察并发出提醒 — "这个 PR 已停滞，那个依赖被阻塞" | **下一阶段目标** |
| **③ Autonomous Coordinator** | Agent 在窄范围内自主协调，强护栏 | 多数工程师表示"将来可能信任，但非现在" |

**数据支撑**：Stack Overflow 2025 调查 — 76% 的开发者不打算用 Agent 做部署和监控，69% 不打算用 Agent 做项目规划。**高责任任务最晚获得信任。**

> **核心洞察**：Agent 工具设计必须尊重信任序列——跳过 Reactive → Proactive 直接跳到 Autonomous 的组织，往往会失去工程师的买账。

### 2.3 对项目管理工具的启示

- **文档在做新工作**：原本给人写的 Wiki 现在被 Agent 读取。自服务信息能力强的团队，效率高出 4.9×，生产力高出 4.4×
- **规划重新成为团队运动**：验收标准、边缘情况、所有权——以前觉得是额外工作，现在是让 AI 有用的输入
- **工作在哪里发生正在迁移**：工程师活在终端和 IDE 里。**项目管理工具如果在 CLI 和 IDE 中不可见，就容易被忽略**

## 3️⃣ Atlassian: ADS 成为 AI-native 设计系统 — 结构化的上下文才是 Agent 的燃料

**来源**: [Atlassian Design System: Building the context engine for the AI era — Inside Atlassian](https://www.atlassian.com/blog/announcements/atlassian-design-system-building-the-context-engine-for-the-ai-era) (2026-05-28)

### 3.1 核心指标：结构化上下文带来的量化提升

Atlassian Design System (ADS) 团队将其设计系统（设计原则、Token、组件文档）转化为 AI 可读的上下文文件后，获得以下效果：

| 指标 | 提升 |
|:-----|:----:|
| AI 调用准确率 | **+52%** |
| ADS 特定任务速度 | **+34%** |
| AI 工具调用次数 | **-26%** |
| Token 消耗 | **-16%** |

> **深层次原因**：原来 AI 需要「猜测」设计系统的规则（从大量非结构化文档中推测），现在上下文文件是明确的结构化编码——AI 不再需要推理「应该怎么做」，而是直接读取「就是这么做的」。

### 3.2 «Atlassian Design System» → «团队知识库» 的类比

AMS 的经验对 **项目管理工具中的知识管理** 有直接借鉴意义：

| ADS 的做法 | 团队知识管理的对应 |
|:-----------|:-------------------|
| 结构化内容文件 → AI 可读 | 项目规范、决策记录、架构文档结构化 → Agent 可理解 |
| MCP 服务器 → 通过协议接入 | Teamwork Graph → 统一的上下文 API |
| Token/Component → 设计意图显式编码 | 决策原则/边界条件/约束 → 工作流上下文显式化 |
| Eval → 量化知性提升 | 定期检查 Agent 输出的质量和 Token 消耗效率 |

**结论**：**Agent 不是第二大脑，而是结构化上下文的高效执行器。** 当上下文不结构化时，Agent 只能做「猜谜」，猜对了靠运气，猜错了用户买单。

## 4️⃣ Atlassian: @Jira 在 Slack 中 — 聊天生成 Agent-ready 工作项

**来源**: [Introducing @Jira — create work items from any Slack conversation — Inside Atlassian](https://www.atlassian.com/blog/jira/introducing-at-jira-slack-create-work-items) (2026-06月)

**核心更新**：Jira Cloud for Slack 引入 `@Jira` 能力

- **@Jira 在任何 Slack 频道中** → `@Jira create a task to update the API rate limit doc` → 自动生成 Issue
- **Agent-ready**：生成的 Issue 直接带有合适的上下文（来源 Slack thread 链接、@提及人、优先级建议），可供 Agent 直接捡起执行
- **规模数据**：Jira Cloud for Slack 是 Slack 市场 Top 5 应用之一，**每月 350 万用户**在使用
- **战略意义**：**工作项的创建成本趋近于零**——聊天即创建。Agent 从 Issue 自动执行的链条变得更短：「聊天中提到问题 → @Jira 创建 Issue → Agent 自动捡起 → 修复 → 自动更新状态」

### 📈 三大趋势判断（更新）

1. **「显式化」是 AI 时代项目管理的核心竞争力**：HBR 的「教 AI 你的决策方式」+ Atlassian ADS 结构化上下文的 52% 准确率提升 + Atlassian 开发者上下文组装困境——三条独立证据指向同一结论：**Agent 时代的竞争力 = 你能多好地把隐性知识显式化**。这不仅是技术问题，是组织设计问题。

2. **Agent 信任阶梯不可跃迁**：Atlassian 访谈明确显示 Reactive Assistant → Proactive Observer → Autonomous Coordinator 的信任序列。试图跳步骤直接上 Autonomy 的组织会失去工程师信任。这与 Product School Carlos 的 AI 运营模型 L1-L4 层级完全对应——**组织成熟度和信任度是渐进式的**。

3. **「AI 的运营」正在从技术问题变为「组织心理学问题」**：本周 HBR 三篇文章从三个独立角度指向人类的心理惯性和认知偏见（不愿质疑 AI、AI persona 影响表现、决策显式化受阻于隐性知识），加上 Atlassian 的 10× 懒散偏见——**最难的 AI 产品运营问题不是模型多好，而是人是否按照正确的方式与 AI 交互**。
