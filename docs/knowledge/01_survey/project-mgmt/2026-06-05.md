## 一、🏗️ Jira 成为 Agent 编排平台：工程维护自动化实践

> **来源**：[Atlassian Blog - How We Cut up to 80% of Engineering "Chores" Using AI Agents in Jira](https://www.atlassian.com/blog/development/ai-agents-jira-engineering-maintenance) | **日期**：2026-06-01 | **作者**：Arnaud Moret & WaiYee Loo（Jira 团队 Principal/Senior Engineer）

### 🚀 核心成果数据

| 指标 | 数值 |
|:-----|:-----|
| Flaky test 修复人力节省 | **最高 80%**（从每月+1 工程周 → 约 1/5） |
| 废弃 Feature Flag 自动清理 | **500+ 合并 PR in 70 天** |
| 单次 Flaky test 处理时间 | 从 2h 人工 → 分钟级审查 |

### 🧩 方法论：工作项即 Agent Prompt

Jira 团队的核心洞察——**每个 Jira 工作项本身就是 AI Agent 的 prompt**：

1. **Jira = Agent Orchestration Platform**：工作项包含所有上下文（任务描述、代码引用、期望结果），Agent 直接读取执行
2. **模式识别→Agent Skill**：团队积累多年的模式识别（异步时序问题、竞态条件、mock 不稳定等）编码为可复用的 Agent Skill，按类型分：单元测试专家、集成测试专家、视觉回归专家
3. **Human-in-the-loop 三重保障**：Agent 做第一遍调查→评论+打开 Draft PR→工程师审查后才合并
4. **Feature Flag 清理系统**：每日 Cron 检测老旧 Flag→创建 Jira 工作项→工程师审核后 Assign→Agent 自动执行→Draft PR
5. **仓库级 Agent Skill**：不搞"一刀切"，为每个仓库定制清理程序，无定制时 fallback 到通用清理 Skill

### 💡 关键洞察

> 「如果你的团队在花时间做重复模式的工程任务，那就是你实施 Agent 自动化机会的信号。」

这个模式对任何 PM/工程管理团队都可迁移：**把团队已有的知识编码为工作项定义 + Agent 指令 + Agent Skill**。

## 二、🔄 Teamwork Collection：从工作跟踪到执行赋能

> **来源**：[Atlassian Blog - Beyond the Jira board: how autonomous workflows unlock engineering velocity](https://www.atlassian.com/blog/ai-at-work/beyond-the-jira-board-how-autonomous-workflows-unlock-engineering-velocity) | **日期**：2026-04-27 | **作者**：Hannah Fischbein

### 📊 效率基线与产出数据

**关键基准数据**：
- 工程师仅 **16% 时间写代码**，**84% 花在执行支撑**（搜索文档、明确需求、工具切换、等待审查、跨团队对齐）—— IDC Survey Spotlight
- 这 84% 不是浪费——它是工作得以协调、审查、交付的方式

**Teamwork Collection 用户实测数据**：

| 指标 | 数据 |
|:-----|:-----|
| Sprout Social | 新员工 80% 的问题由 Rovo Agent 回答 |
| Procore 工程 VP | 季度路线图编写时间减少 **75%** |
| Atlassian 内部 | Rovo 为每个工程师每年节省 **100+ 小时** |
| Loom | 会议减少 **15%**，新员工上手速度提升 **30%**，三年 ROI **232%**（Forrester TEI） |
| 项目启动速度 | 提升 **53%** |

### 🧩 Teamwork Graph：AI 的上下文引擎

Teamwork Collection 的"魔法"是 **Teamwork Graph**——跨 Jira/Confluence/Loom/Rovo 共享数据模型，映射人员、工作项、知识、关系：

- 更快的搜索与发现
- Rovo 的上下文感知推荐
- 基于权限的 AI 上下文（不越权）
- 第三方工具集成

**四步激活路径**：
1. 连接知识到执行（Jira + Confluence 通过 Teamwork Graph 打通）
2. 用 Loom 改变上下文捕获方式（会议→视频→摘要→工作项）
3. 部署 Agent（Loom 录 Bug → Jira Issue → Agent 分析 → 开发者审查 → Copilot 实现）
4. 连接工作到成果（Jira 工作项关联业务目标）

### 🤖 Agentic 工作流：从「if X then Y」到「我需要发布功能」

> 标准自动化：if X then Y
> Agent 体验：我需要发布功能 → 找文档空白 → 通知干系人 → 起草发布说明

三级 Agent 协作模型：
- Assign Agent 一个工作项（Triage/Analyze Bug）
- @Mention Agent 在评论中（总结讨论/回答问题）
- 嵌入 Workflow Transition（分析、分类、文档自动生成）

## 三、🧠 AI-native PM：产品管理的新范式

> **来源**：[Atlassian Blog - The future of product craft: Why AI-native PMs build better products, not just work faster](https://www.atlassian.com/blog/how-we-build/the-future-of-product-craft) | **日期**：2026-05-28 | **作者**：Tim Lelek

### 📊 核心矛盾数据

| 指标 | 数值 |
|:-----|:-----|
| 89% 高管 | 认为 AI 提升了工作速度 |
| 仅 6% | 能指出具体的组织级 AI ROI |
| $1610 亿/年 | Fortune 500 AI 碎片化税（协调无法跟上AI加速） |
| 2x ROI | 将 AI 视为共享伙伴 vs 个人捷径的团队（AI Collaboration Index） |

### 🧩 AI-native PM 的三重模型

Atlassian 团队观察到三种 PM 演化原型：

| 原型 | 特征 | 判断 |
|:-----|:------|:-----|
| PM→工程师 | 核心价值转移到写代码 | ❌ 真实 PM 工作被搁置 |
| AI 生产力层 | 用 AI 提速但不改变决策方式 | ⚠️ 有用但不具变革性 |
| **AI-native PM** | **团队决策更快，因为 PM 改变了上下文流动方式** | ✅ 方向所在 |

### 📋 AI-native PM 的三大实践转变

**1. 客户理解：从定期到持续（Learn）**
- 传统：两周调研周期 → AI Agent 综合 support ticket/NPS/论坛/产品行为 → 一下午获得同级洞察
- **交付效率**：分钟级回答客户问题，无需预约研究环节

**2. 原型替代文档成为决策起点（Decide）**
- 原型回答「应不应该建这个」比 PRD 更快
- **交付效率**：从 idea 到工作原型到客户反馈到构建决策 = **一周**

**3. 质量成为反馈循环，而非发布关卡（Act）**
- AI 特性的质量是光谱，需要定义"好"的标准
- 每个 AI 特性在构建前就定义 eval 标准，上线后持续监控

### 📐 AI Fluency 六维能力框架

| 能力维度 | L1: Curious → L5: Pioneering |
|:----------|:------------------------------|
| 工具熟练度 | 知道用什么、什么时候用、什么时候覆盖 |
| 发现能力 | AI 加速客户信号→洞察 |
| 数据洞察 | 证据流自动化 |
| Eval & 质量 | 定义/测试/监控「好」的标准 |
| 原型能力 | 快速转化为可学习的工作产物 |
| 技术杠杆 | 接近代码/系统/权衡以做出更犀利的产品决策 |

## 四、🎯 AI 超级用户度量：从 WAU 到功能级百分位

> **来源**：[Atlassian Blog - Stop counting who uses AI. Start finding who's transforming with it.](https://www.atlassian.com/blog/ai-at-work/how-to-identify-ai-superusers) | **日期**：2026-06-01 | **作者**：Alicia Lenart（Atlassian VP of HR）

### 📊 三个阶段演进

| 阶段 | 指标 | 时间 | 结果 | 问题 |
|:-----|:------|:-----|:------|:-----|
| **Phase 1** | WAU（周活跃用户） | 早期 | 95%+ WAU | 跟 spell-check 一样浅层使用 |
| **Phase 2** | 全公司统一超级用户阈值（~40次/周） | 2025 H2 | 14%→34% | 工程 vs 财务/HR 数量级差异，偏向量而非质 |
| **Phase 3** | **按功能定义超级用户**（各岗位本身上10%） | 2026 Q1 起 | 每功能独立基准 | 持续优化中 |

### 🔑 关键度量方法

- 每个岗位（Engineering/Design/PM/Marketing/Legal/Sales/Finance/People）有独立的**90 百分位**周交互基线
- CTO 组织超级用户工程师：**270+ AI 交互/周**
- Finance/Legal 超级用户完全不同——**不同工具、不同节奏、不同模式**
- 公司级得分 = 各功能得分的**平均**（防止大组织淹没小组织）
- 2026 H2 目标：每个功能翻倍其 1 月基线

### 🧩 超级用户的 5 种行为与 3 种特质

**5 种行为**：
1. **重新设计工作而非多用 AI**：拆分为 AI 友好+人类独有部分，构建可复用工具
2. **为他人构建轻量系统**：分享 Agent/Prompt/Workflow，成为团队的 AI Champion
3. **舒适地展示思考中状态**：展示错误和死路，让他人更安全地实验
4. **不混淆"初稿"与"完成"**：扫描 gap/错误/语气问题，对最终输出负责
5. **在非预期的角色和层级出现**：不限于技术/AI 岗位，许多在"胶水"角色中

**3 种可招聘的特质**（比工具技能重要得多）：
1. **模式驱动的实用主义**：发现反复摩擦→"AI 能不能接过来？"
2. **分享优先而非守门**：想到好方法就教给别人
3. **与不完美共存**：不期待 AI 正确，与 AI 交换不同角度

> 「如果你认为 AI 采纳是合规 → 衡量 WAU；
> 如果你认为是参与度 → 衡量使用量 + 培训；
> 如果你认为是转型 → 找到正在重新设计工作的那一小群人。」
