## 1️⃣ Atlassian Rovo Dev: Agent Context Pruning — 长会话上下文效率关键技术

**来源**：Atlassian Blog（Development 分类，2026-05/06 发布）
**链接**：https://www.atlassian.com/blog/development/

**背景**：AI Agent 在长任务执行中会积累大量中间产物（工具输出、代码 diff、日志等），导致上下文膨胀失控。

**核心方案**：

| 维度 | 详情 |
|:-----|:------|
| **剪枝策略** | 使用 `Semantic Pruning` 识别上下文中的冗余/过时内容并移除，而非简单截断 |
| **优先级保留** | 保留：用户原始指令 + 当前任务相关代码 + 最近的工具调用链；剪枝：中间调试输出、已确认的错误信息、过时的搜索结果 |
| **上下文评分** | 为上下文中的每个 segment 分配相关性分数，低分内容被主动归档或移除 |
| **实施结果** | Rovo Dev 长会话（>50 轮交互）的上下文利用率提升 **显著**（具体数据未公开），会话质量未退化 |

**PM 意义**：这是 Agent 生产效率的核心瓶颈之一。长会话中 Agent 的记忆管理直接决定了 Agent 能否完成**多步骤产品工作流**（如持续调研合成 → 决策支持 → 执行跟踪）。没有上下文剪枝，Agent 在 30+ 轮后就开始"遗忘"上下文。

**交叉参考**：MCP 工具压缩（见 06-09 跟踪）解决的是**横向工具链**上下文膨胀，Context Pruning 解决的是**纵向会话深度**上下文膨胀，两者互补。

## 2️⃣ Atlassian: AI Agents in Jira 削减 80% 工程杂务

**来源**：Atlassian Blog（Development 分类，How We Cut up to 80% of Engineering "Chores" Using AI Agents in Jira，2026-06）
**链接**：https://www.atlassian.com/blog/

**技术细节**：

| 维度 | 详情 |
|:-----|:------|
| **目标问题** | Flaky test 和 stale feature flag 无声消耗工程时间，属于"看不见的债务" |
| **Agent 方案** | Jira 中配置 AI Agent，自动检测 flaky test 失败 → 创建/更新票据 → 分配 owner → 跟踪修复状态；自动扫描 stale feature flag → 创建清理票据 → 关联代码库 |
| **量化结果** | 工程杂务工作量削减 **80%**（Atlassian 内部数据） |
| **实现机制** | Agent 利用 Rovo Dev + Jira 集成，通过 Teamwork Graph 获取项目上下文、代码仓库状态、Pipeline 结果，自动编排工作流 |
| **自动编排** | Agent 自主判断优先级（如 flaky test 失败率 > 阈值自动升级），人类仅在需要决策时介入 |

**PM 意义**：这代表了 **Agent 编排平台**从"辅助编码"向"自动治理"的进化。Flaky test 和 stale flag 是工程团队长期的痛点，AI Agent 在此类**规则明确 + 多步操作 + 低判断成本**的场景中表现最优。这符合 Product School 的 Agent 决策框架中的"强 Agent 候选"条件（9-16 分推荐模式）。

## 3️⃣ Product School: AI Builder — 重塑产品交付方式的新角色

**来源**：Product School Blog（Skills 分类，2026-06 最新）
**链接**：https://productschool.com/blog/skills/what-is-an-ai-builder-new-role

**定义**：AI Builder = 利用 AI 进行**原型设计、分析、评估和交付**的产品专业人员，将产品思考转化为 artifact 的速度远超传统构建周期。

**与传统 PM 对比**：

| 维度 | 传统 PM | AI Builder |
|:-----|:--------|:-----------|
| 核心能力 | 文档撰写、需求评审、沟通协调 | AI 原型快速迭代、Prompt 工程、AI 评估与质量验证 |
| 交付方式 | 写 PRD → 设计评审 → 开发 → 测试 → 发布（周/月） | 用 AI 工具直接生成 UI、逻辑、数据 → 快速验证 → 迭代（小时/天） |
| 关键技能 | 流程图、线框图、Excel 分析 | Bolt/Lovable/Cursor 等 AI 工具、AI Evals、系统提示设计 |
| 产出物 | PRD 文档、需求清单 | 可交互原型、AI 评估套件、Prompt Sets |
| 工作模式 | 分工明确（PM 写需求，工程师实现） | PM 直接交付 → 工程师负责架构/安全/规模化 |

**附带工具实测**：同篇文章系列《I Tested 11 AI Tools to Find the Best Picks for PMs》覆盖 Bolt / Lovable / Cursor / Claude Code / ChatGPT / Perplexity / Gamma / Notion AI / Otter.ai / Grammarly / Miro AI 等工具，每个附有 **copy-paste prompt** 和 **screenshot checklists**。

**PM 意义**：AI Builder 不是"PM 用 AI 工具"那么简单，它代表**产品岗位的 identity 重构**——不再是"写需求的人"，而是"用 AI 直接构建验证原型的人"。这与 Cursor / Lovable 等 AI 编码工具的爆发直接相关。

## 4️⃣ Product School: AI 运营模型 — 80% 产品团队卡在 L1-L2

**来源**：Product School Blog（Digital Transformation 分类，2026-06 最新）
**链接**：https://productschool.com/blog/digital-transformation/

**核心判断**：**80% 的产品团队的 AI 运营模型卡在 L1-L2**，只有前 1% 触及 L4。

**AI 运营模型四个层级**（Product Partners 框架）：

| 层级 | 名称 | 特征 | 占比（估算） |
|:----|:-----|:------|:-----------|
| L1 | **虚假顶峰**（False Summit） | 购买 ChatGPT/Copilot 企业版，零散使用 AI 助手，**没有结构化运营模型** | ~40% |
| L2 | **实验阶段**（Experimentation） | 团队开始用 AI 工具做低风险工作（总结、写作、搜索），但缺少**度量、治理和反馈闭环** | ~40% |
| L3 | **运营化**（Operationalized） | 将 AI 嵌入核心工作流，有系统性度量、AI 治理策略、Agent 工作流编排 | ~15% |
| L4 | **AI-native 闭环**（Closed Loop） | AI 不仅是工具而是**产品运营的底层协议**——数据飞轮驱动自动决策、自动实验、自动治理 | ~5%（甚至 1%） |

**为什么 80% 卡住**：
- 买了工具 but 没有设计**运营模型**（谁用、怎么用、怎么度量）
- AI 工具降低了产出成本，但团队**缺乏判断力**来区分"有用输出" vs "看似好但无效输出"
- **没有反馈闭环**：AI 输出 → 人类判断 → 数据捕获 → 模型/流程改进，这个循环没有建立

**关键命题**：**"运营模型即产品"**（The Operating Model is the Product）——AI 时代，团队的运营方式本身就是需要被设计的核心产品。

## 5️⃣ Product School: AI 实验方法论 — 5 类实验 + 3 层指标

**来源**：Product School Blog（AI Experimentation: How AI PMs Test and Learn Faster，2026-03-18）
**链接**：https://productschool.com/blog/artificial-intelligence/ai-experimentation-how-ai-pms-test-and-learn-faster

**五种实验类型及其最佳场景**：

| 实验类型 | 最佳场景 | 成本 | 适用阶段 |
|:--------|:---------|:-----|:---------|
| **A/B 测试** | 需要隔离单一变量，获取确定性结论 | 中等 | 验证 Impact |
| **原型测试** | 需要验证用户是否理解和信任 AI 功能 | 低 | 验证可用性/信任 |
| **Fake-door 测试** | 验证用户是否需要该功能（需求验证） | 极低 | 验证 Demand |
| **灰度发布（Canary）** | 功能已构建，需要安全上线 | 高 | 安全上线 |
| **多变量/自适应实验** | 需要优化已确认有价值的特性 | 中高 | 优化阶段 |

**三层指标架构**：

| 层级 | 关注点 | 典型指标 |
|:-----|:-------|:---------|
| **模型层** | AI 是否正确、够快 | 准确率、精确率、召回率、F1、延迟 |
| **产品层** | 用户是否用、是否觉得有用 | 采用率、满意度、任务成功率、留存 |
| **业务层** | 对商业目标的影响 | 转化率、收入提升、成本节省、风险降低 |

**AI 实验特有挑战**：
- **非确定性输出**：同输入可能得到不同输出 → 需**冻结模型版本**、**固定随机种子**、**延长实验周期**
- **模型漂移**：实验结束后数据变化可能导致结果失效 → 需**持续监控 + 定期重测**
- **偏见与公平**：AI 可能对某类用户表现更好/更差 → 需**按人群分段分析 + 对抗性测试**

## 6️⃣ Product School: AI Agent 决策框架 — 11 维评分矩阵（精炼版）

**来源**：Product School Blog（When Should Product Teams Use AI Agents?，2026-03-03 更新至 2026-04-27）
**链接**：https://productschool.com/blog/artificial-intelligence/when-should-product-teams-use-ai-agents

**核心数据**（与 06-09 跟踪交叉引用，本次补充 4 个关键用例）：

| 用例 | Agent 适用性 | 判断理由 |
|:----|:------------|:---------|
| 持续调研合成 ✅ | **强 Agent 候选**（17-20 分） | 多步骤跨工具、弱规则、成功可度量、错误可逆 |
| 决策支持 ✅ | **推荐 Agent 模式**（9-16 分） | 人类判断不可替代，Agent 负责"拉数据 + 选项 + 预判" |
| 跨系统自动化 ✅ | **强 Agent 候选**（17-20 分） | 多工具协调是 Agent 优势，但需"Agent 规划 + 确定性执行" |
| 版本发布就绪 / QA ✅ | **推荐 Agent 模式**（9-16 分） | Agent 做可观测性 + 验证循环，人类做最终审核 |

**新的关键判断**：
- **AI Builder 范式**：Agent 可完成 50-70% 的工作，人类拥有最后的 30%（判断 + 质量把关）
- **"人类在环"的陷阱**：如果 Agent 的每步都需要 Human Review，就变成**代理式审查流水线**而不是效率提升
  - 设计原则：Agent 在**模糊/高影响**时才 Escalate，在**低风险/可逆**时全自主运行
- **"过度信任"（Over-trust）** 是 Agent 部署最昂贵的失败模式——Agent 自信地犯错，而团队没有机制发现
