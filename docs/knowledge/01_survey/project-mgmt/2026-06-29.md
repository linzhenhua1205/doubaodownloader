## 1️⃣ 多 Agent 编排：Jira-anchored 解决 Agent 漂移问题

**来源**: [Why AI Agents Drift Mid-Task and How a Multi-Agent System Fixes It — Atlassian Blog / Development](https://www.atlassian.com/blog/development) (2026年6月)
**作者**: Atlassian Engineering（具体作者见原文）

### 1.1 Agent 漂移问题

> 文章开篇描述的现象：你给 Agent 一个真正的工作任务（含验收标准和边界条件的完整 spec），前几分钟它很精准，然后开始**滑脱** — 忘记了自己做过的决策、偏离了 spec、产生与预期不一致的输出。

这不是 LLM 能力不足的问题，而是**单 Agent 长任务中的「上下文衰减」**。

### 1.2 Atlassian 的方案：Multi-Agent 编排 anchored to Jira

| 组件 | 角色 |
|:-----|:-----|
| **Jira Issue** | 作为 Spec 的「锚点」，包含验收标准和边界条件 |
| **Specialist Agents** | 每个 Agent 专注一个子任务（代码生成、测试、审查、文档） |
| **Coordinator** | 监控各 Agent 输出与 Jira Spec 的偏差，漂移时重新校准 |
| **Jira 状态** | 问题状态实时回写，全流程可追踪 |

**核心设计原则**: Agent 不靠自身长上下文保持一致性，而是**每次决策都回溯到 Jira Issue 中定义的 Spec**，形成「锚定机制」。

### 1.3 工程落地数据

| 指标 | 数值 | 说明 |
|:-----|:----:|:-----|
| **团队规模** | 5 名工程师 | 迁移 100+ GraphQL fetchers |
| **原工期估算** | ~2 个季度 | 传统方式完成该任务 |
| **实际完成时间** | 1 周 | 使用 Jira-anchored 多 Agent 编排 |
| **方法论** | Jira ticket → Behavioral Spec（行为规格说明）→ Agent 按规格执行 |

> **这一条数据质量极高**。不是简单的「Agent 加速开发」泛泛述，而是给出了**具体任务类型（GraphQL fetcher 迁移）**、**团队规模（5人）**、**原工期（~2季）**、**实际工期（1周）** 四个维度可交叉验证的数据。

| 此前跟踪 | 连接点 |
|:---------|:-------|
| Claude Agent for Jira / Cursor in Jira (06-26, 06-28) | Agent 编排层定位已确立，本文给出了解决 Agent 实际工程问题的方案 — 多 Agent + Spec anchoring |

## 2️⃣ PM 角色迁移：从「需求翻译」到「Context Wrangler」

**来源**: [The bottleneck keeps shifting: What AI is changing about how we build — Atlassian Blog / How We Build](https://www.atlassian.com/blog/how-we-build/the-bottleneck-keeps-shifting-what-ai-is-changing-about-how-we-build) (2026-05-06)
**作者**: Peter Morris, Senior Principal PM — Future of Teamwork
**访谈对象**: Jon Noronha, Co-founder of Gamma（AI-native 演示平台）

### 2.1 核心论点：瓶颈从「工程产能」转向「决策质量」

> 「Tech teams are experiencing a revolution unlike anything they've seen before. The barriers to entry for building have all but disappeared. The constraint no longer comes from producing enough output, but from deciding what to build, and the restraint required to build something good.」

| 旧时代瓶颈 | 新时代瓶颈 |
|:-----------|:-----------|
| 没有足够的工程师 | 没有足够的**好决策** |
| 没有足够的时间 | 没有足够的**辨别力（discernment）** |
| 没有足够的算力 | 没有足够的**克制（restraint）** |

Atlassian CPO&AIO Tamar Yehoshua + Gamma CEO Jon Noronha 对谈。

### 2.2 PM:Engineer 配比数据

| 指标 | 传统经验值 | Gamma 实际值 |
|:-----|:---------:|:-----------:|
| PM : Engineer 配比 | **1:10** | **1:4** |
| PM 写代码比例 | 极少 | **~50%**（一半的 PM 和设计师在 commit 代码） |
| 4 工程师团队产出 | 约等于传统 ~8-10 人团队 | Gamma 实际观察 |

> **🔴 1:10 → 1:4 的配比变化是 AI 时代组织设计的第一个量化信号。** 当工程师生产力提升 2-3× 时，PM 并未等比提升（因为 PM 工作涉及大量人类约束），导致瓶颈转移到了**需求端**。

### 2.3 PM 的新职责：Context Wrangler

Jon Noronha 对自身角色的描述：

> 「I'm increasingly viewing my job as context wrangler. I feel like it's my job to take all the information that is only in my head and get it into writing where my team has access to it, but also all of their agents have access to it.」

**Context Wrangler ≠ 文档撰写者**，而是：
1. 将**隐性知识（团队大脑中的信息）** 转化为**显性文档（人和 Agent 都可消费）**
2. 保证文档的可 Agent 消费性（结构化、规格化、有边界条件）
3. 充当 AI 和人类之间的**信息翻译层**

### 2.4 PM 的 bug triage 新工具

> Half of Gamma's PMs and Designers are now committing code.

PM 使用 AI 进行 bug triage：
- 自己用 AI 调查 root cause
- 决定是否值得叫工程师修
- 或自己直接创建 ticket、甚至自己修

> **「Where previously PMs would have had to consult engineers about how the code works, they can now ask AI and start reporting and escalating bugs and typos at speed.」**

### 2.5 Framing: 用户说和用户做的不一致

Gamma A/B 测试案例：用户口头上说「保留我的精确文本」，但实际行为显示「请在小地方帮我润色一下」。

> 「People were saying, preserve my exact text, but they didn't mean preserve my exact text. They meant 'mostly preserve my text, but you gotta zhuzh it a little bit in the right places'.」

→ **人类判断力和主观 taste 在 AI 时代反而更关键** — 因为 AI 倾向于 literal interpretation，而真实需求往往是 nuanced 的。

### 2.6 与本知识库的关联

| 此前跟踪 | 连接点 |
|:---------|:-------|
| Knowledge Architect (06-27) | Context Wrangler = Knowledge Architect 的 PM 侧镜像角色，一个在组织设计层，一个在实操层 |
| AI 坦诚悖论 (06-28) | Context Wrangler 要解决的「隐性知识显性化」问题与文化层面「坦白 AI 使用被惩罚」形成张力 |
| Cursor in Jira / Claude Agent (06-26, 06-28) | PM 直接 assign Agent 干活 = Context Wrangler 的日常工具 |

## 3️⃣ Atlassian AI Agent 削减 80% 工程杂务

**来源**: [How We Cut up to 80% of Engineering "Chores" Using AI Agents in Jira — Atlassian Blog / Development](https://www.atlassian.com/blog/development) (2026年6月)

### 3.1 问题：Flaky tests + stale feature flags

工程团队中常见但高消耗的低价值工作：
- **Flaky tests**：不稳定测试用例反复触发重跑、排查、修复
- **Stale feature flags**：已上线的 feature flag 无人清理，累积成技术债

### 3.2 AI Agent 处理方法

| 杂务类型 | Agent 工作流 | 效果 |
|:---------|:-------------|:----:|
| **Flaky tests** | Agent 自动检测 flaky test 模式 → 创建 Jira Issue → 关联最近提交 → 建议修复方案 | 削减「大量」手动排查时间 |
| **Stale feature flags** | Agent 扫描代码库 → 识别已稳定发布的 flag → 自动创建清理 PR | 削减 ~**80%** 的清理工作 |

### 3.3 方法论

> Jira 作为编排层：Agent 发现的每个问题 → 自动创建 Issue → Assign → 追踪 → 闭环。

这与 06-26/28 跟踪的「Jira 作为 Agent 编排平台」完全一致—Agent 不只是被动的执行者，也是**主动的问题发现者**。

### 3.4 与本知识库的关联

| 此前跟踪 | 连接点 |
|:---------|:-------|
| Linear 30% auto-fix bugs (06-27) | 两个独立来源（Linear + Atlassian）都把 Agent 用在「主动发现→自动修复」的窄范围任务上，且都给出了量化数据 |
| Agent 作为 Jira 一等公民 (06-26) | Flaky test Agent 自动创建 Issue = Agent 作为编排平台中的一等公民的典型用例 |
| AI efficiency paradox (06-27) | 杂务自动化的价值在于释放工程精力到「好决策」— 这正是「瓶颈从工程产能转向决策质量」的直接佐证 |

### 📈 三大趋势判断

1. **Multi-Agent + Jira-anchored Spec 成为 Agent 一致性的工业级方案。** Agent 漂移是 LLM 长上下文的天生缺陷，Atlassian 的方案（Coordinator 回溯 Jira Spec 校准各 Specialist Agent）从根本上改变了「让 Agent 稳定工作」这件事的实现路径。**不是靠更大的上下文窗口，而是靠外部锚定点。**

2. **PM 角色正在经历「30 年来最大的定义变迁」**。从 06-27 的 Knowledge Architect（组织设计的 AI 上下文结构岗），到今天 Gamma CEO 自述的 Context Wrangler（实操层面的知识显性化执行者），两条线索汇合为一个清晰结论：**AI 时代，PM 的核心交付物从「PRD」变成「可被 Agent 消费的结构化上下文」。**

3. **Agent 从「辅助人类工作」走向「主动发现问题」。** Flaky test Agent 自动建 Issue、Stale flag Agent 自动清 PR、Linear Agent 自动修 bug — Agent 不再只是「接到任务执行」，而是**在人类没察觉时就发现了问题**。这进一步加剧了 Code Review 的稀缺性，也解释了为什么 Diffs / Guided Reviews 成为 Linear 和 Atlassian 的共同发力点。
