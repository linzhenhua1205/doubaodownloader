## 1️⃣ 🏗️ Perseus: 消除多节点 Megakernel 通信的隐藏序列化 (arXiv:2605.00686)

- **来源**: arXiv:2605.00686, "Eliminating Hidden Serialization in Multi-Node Megakernel Communication" (2026-05-01)
- **机构**: 作者 Byungsoo Oh, Rachee Singh
- **标签**: `MoE` · `Megakernel` · `RDMA` · `GPU-Initiated` · `Serialization`

### 问题

近期 megakernel 设计将 MoE 推理的专家计算与细粒度 GPU-initiated 通信融合到单个持久 GPU kernel 中，在单节点上通过 tile 级重叠超过基于集合通信的 MoE。但**多节点场景完全逆转**：

- 通信受限的 MoE 模型在 8 节点上退化高达 **10×**
- 退化随着节点数增加而加剧

### 根因：代理 RDMA 的隐藏序列化

Perseus 将退化追踪到 **proxy-based RDMA 传输中的隐藏序列化**：

```
问题本质:
每个tile传输与其完成信号之间的排序要求
→ 强制 fence 排空 NIC 流水线
→ 成本随并发传输数增长
→ 计算量小的 expert 无法吸收这种膨胀的网络延迟
→ 通信暴露在关键路径上
```

### 两大方案

| 方案 | 技术 | 效果 |
|:-----|:-----|:-----|
| **Decoupled Signaling** | 批量 fence 到每个 destination 粒度 | fence 计数减少 **8×** |
| **NIC-side Ordering** | 用硬件 fence flag 替代 proxy stall | proxy 永不被阻塞 |

### 性能数据

| 配置 | 加速比 | 对比基准 |
|:-----|:-------|:---------|
| **Proxy-based 传输** | 最高 **10.3×** 端到端 | vs 原始 megakernel 基线 |
| **IBRC（IB RC）** | 匹配或超过 IBGDA | GPU-direct 方案 |
| **Perseus on IBRC vs IBGDA** | 最高 **1.2×** | 说明序列化才是边界，而非 proxy vs GPU-direct 的选择 |

### 意义

> ⚠️ **关键洞察**：此前行业共识暗示 proxy-based RDMA 天生不如 GPU-direct RDMA。Perseus 证明**序列化才是真正的瓶颈**，而非传输路径选择。这对于设计 NVLink 域外 RDMA 架构（如超节点场景中跨 tray/rack 的通信）有直接影响——消除序列化的 proxy 可以接近甚至超过 GPU-direct 方案，且实现成本更低。

## 2️⃣ ⚡ Celeris: ML 容忍的 RDMA 传输 — 消除重传 (arXiv:2510.16606, IEEE CAL 2025)

- **来源**: arXiv:2510.16606, "Reimagining RDMA Through the Lens of ML" (IEEE Computer Architecture Letters, 2025)
- **机构**: 作者 Ertza Warraich, Ali Imran 等 (Purdue/MIT/VMware)
- **标签**: `RDMA` · `RoCE` · `ML Tolerance` · `Loss Recovery` · `Domain-Specific Transport`

### 核心洞察

**现有 RDMA 的问题**: RoCE、IRN、SRNIC 等 RDMA 设计强制严格可靠性和按序交付，依赖重传和包排序保证正确性。这些机制在 ML 场景中引入不必要的复杂性和延迟：
- 即使罕见的丢包或延迟也会一致性地降级系统性能
- 硬件复杂度高（BRAM/逻辑消耗大）
- ML 工作负载本质上对部分数据丢失有容忍度

### Celeris 设计

**Celeris** 是一个领域特定的 RDMA 传输方案：

| 保留的机制 | 移除的机制 |
|:-----------|:-----------|
| DGQCN 拥塞控制 | 硬件重传 (**完全移除**) |
| 软件级自适应超时 | 硬件按序交付 (**完全移除**) |
| 数据优先级管理 | — |

**丢失恢复转移至 ML 流水线**：利用 Hadamard Transform 等方案在 ML 层面处理丢失数据，而非在 NIC 层面重传。

### 性能数据

| 指标 | 数值 |
|:-----|:------|
| **P99 尾延迟** | 降低 **2.3×** |
| **BRAM 使用** | 减少 **67%** |
| **NIC 故障容错** | 几乎翻倍（nearly doubles resilience） |

### Celeris vs CSA-UD（2026-06-29）

两者都挑战了 RDMA RC 模式，但路径不同：

| 维度 | CSA-UD | Celeris |
|:------|:-------|:---------|
| **核心方法** | 利用通信语义（同步barrier）+ UD 协议解耦 | 利用 ML 对丢失的容忍度 |
| **硬件改动** | 已有 RNIC 可用（软件层实现） | 需定制 NIC（移除重传逻辑） |
| **可靠性** | 软件RDMA层保障（解耦恢复） | 不保障，数据级别容错（Hadamard） |
| **效果** | P99 FCT 降 30%+ | P99 尾延迟降 2.3×，BRAM 减 67% |
| **成熟度** | 论文+仿真 | IEEE CAL 短文，早期结果 |

> 两者互补：CSA-UD 在**保持可靠性**前提下突破 QP 可扩展性；Celeris 在**放弃可靠性**前提下追求极致硬件效率。对于超节点架构设计，Celeris 的思路更激进但也更具长期价值——如果 ML 训练本身能容忍部分丢失，为什么还要花大量硬件资源做无损保障？

## 3️⃣ 🧩 FlexLink: NVLink+PCIe+RDMA 异构链路聚合 (arXiv:2510.15882)

- **来源**: arXiv:2510.15882, "FlexLink: Boosting your NVLink Bandwidth by 27% without accuracy concern" (2025-08-29)
- **机构**: 作者 Ao Shen, Rui Zhang, Junping Zhao
- **标签**: `NCCL` · `NVLink` · `PCIe` · `RDMA` · `Heterogeneous` · `Load Balancing`

### 问题

当前 NCCL 在 H800 等 GPU 上仅使用单一互联（NVLink），导致：
- **H800 NVLink 带宽成为瓶颈**（H800: 400GB/s NVLink vs H100: 900GB/s）
- **PCIe 和 RDMA NIC 在全速工作时基本空闲**
- 每个 H800 节点有 **8× 400Gb/s NIC 带宽（合计 3.2Tb/s）闲置**

### FlexLink 设计

FlexLink 将 NVLink、PCIe 和 RDMA NIC 聚合为**单一高性能通信结构**：

```
传统方案:
NVLink ─→ 100% 流量 → 瓶颈
PCIe   ─→ 闲置
NIC    ─→ 闲置

FlexLink 方案:
NVLink ─→ 78-98% 流量 (主要路径)
PCIe   ─→ 2-22% 流量 (offload)
NIC    ─→ 2-22% 流量 (offload)
         ↓
两阶段自适应负载均衡 → 所有链路满载
```

| 组件 | 技术详情 |
|:-----|:---------|
| **Stage 1** | 静态链路容量评估（NVLink/PCIe/NIC 的带宽+延迟特征建模） |
| **Stage 2** | 运行时动态流量分配，保证**快速互联不被慢速互联节流** |
| **兼容性** | 无损 drop-in 替换，兼容 NCCL API |

### 性能数据

| 集合操作 | 提升幅度 | 说明 |
|:---------|:---------|:-----|
| **AllReduce** | 最高 **26%** | 8-GPU H800 |
| **AllGather** | 最高 **27%** | 8-GPU H800 |
| **offload 比例** | 2-22% | 从 NVLink 转移到 PCIe+NIC |

### 意义

> FlexLink 的理念与 NIMBLE（2026-06-27 已覆盖）相似但更实际：**不动拓扑只做负载分配**。对于 H800 用户来说，非对称 NVLink 带宽是痛点，而 FlexLink 以几乎零成本（纯软件）释放了闲置的 PCIe 和 NIC 带宽。对超节点架构的启示：**如果节点内互联不够用，是否可以通过策略性利用 NIC 做补充？**

## 4️⃣ 🖥️ SCENIC: 开源 200G SmartNIC 的 RDMA/集合通信 Offload (arXiv:2604.15128)

- **来源**: arXiv:2604.15128, "SCENIC: Stream Computation-Enhanced SmartNIC" (2026-04-16)
- **机构**: ETH Zurich
- **标签**: `SmartNIC` · `RDMA` · `TCP/IP` · `Collective Offload` · `Open Source`

### 核心设计

SCENIC 是**全开源**的 200G 数据中心 SmartNIC，关键特性：

| 模块 | 技术详情 |
|:-----|:---------|
| **网络数据面** | 200G 卸载 TCP/IP + RDMA 栈 |
| **Stream Compute Units (SCUs)** | 数据面流计算单元，支持在 datapath 上做数据处理 |
| **ARM 控制核** | 灵活控制路径，直接访问 GPU 和 SSD |
| **OS 集成** | 暴露原生 Linux 网络和 RDMA verb 接口 |

### 应用场景

| 场景 | 详细 |
|:-----|:------|
| **集合通信 Offload** | SCU 直接处理 AllReduce/AllGather 的部分计算 |
| **网络到 GPU 哈希数据分区** | SmartNIC 直写 GPU 内存 |
| **可编程拥塞控制** | ARM 控制核实现自定义 CC 算法 |

### 对比现有 SmartNIC

| 维度 | 商业 SmartNIC (BlueField) | 研究 SmartNIC | SCENIC |
|:-----|:--------------------------|:-------------|:--------|
| 带宽 | 高 (200-400G) | 通常低 (~100G) | **200G** |
| 定制性 | 有限 | 高 | **高** |
| 软件兼容 | 好 | 差（非真正 NIC） | **OS 原生 RDMA verbs** |
| 开源 | 否 | 部分 | **完全开源** |

### 意义

> SCENIC 作为**首个真正开源的 200G RDMA SmartNIC**，为集合通信硬件加速提供了学术级平台的参考设计。其 Stream Compute Unit (SCU) 架构——在 NIC datapath 上做计算而不走 ARM 慢路径——是集合通信 offload 的未来方向。对于超节点中的 smart switch/NIC 选型有参考价值。

## 5️⃣ 🔄 TawPipe: 拓扑感知权重流水线并行 (arXiv:2511.09741, AAAI 2026)

- **来源**: arXiv:2511.09741, "TawPipe: Topology-Aware Weight Pipeline Parallelism for Accelerating Long-Context Large Models Training" (2025-11-12, **AAAI 2026 已接收**)
- **机构**: 作者 Houming Wu, Ling Chen
- **标签**: `Topology-Aware` · `Pipeline Parallelism` · `Weight Passing` · `LLM Training`

### 问题

训练 LLM 受限于有限的设备内存和昂贵的设备间通信。现有方案：
- **传统流水线并行**：传输激活值，通信量随序列长度线性增长 → 长上下文训练效率低
- **WeiPipe (weight-passing)**：传权重而非激活值 → 但冗余 P2P 传输 + 节点内带宽利用不足

### TawPipe 设计

TawPipe 利用分布式集群的层级带宽层次优化通信效率：

| 策略 | 技术 | 效果 |
|:-----|:-----|:------|
| **拓扑分组** | 基于拓扑对设备分组 | 优化节点内 collective + 节点间 P2P 通信 |
| **固定分片赋值** | 每个设备分配固定权重/梯度分片 | 消除冗余传输 |
| **通信计算重叠** | 深度重叠隐藏延迟 | 大部分通信限制在节点内 |

### 与现有方案对比

| 维度 | FSDP (全分片) | WeiPipe | TawPipe |
|:-----|:-------------|:--------|:---------|
| **通信模式** | 全局 collective | P2P 权重传递 | **节点内 collective + 节点间 P2P** |
| **跨节点流量** | O(n) all-to-all | O(n) P2P | **显著减少（节点内聚合后跨节点）** |
| **拓扑感知** | 否 | 否 | **是** |
| **24 GPU 扩展性** | — | — | **优于 SOTA** |

### 性能

- 在 24 GPU + LLaMA 规模模型上，**吞吐量和可扩展性均优于 SOTA**
- 关键指标：**跨节点流量显著减少**（利用节点内 NVLink 带宽聚合后跨节点传输）

### 意义

> TawPipe 是拓扑感知调度在**流水线并行**方向的新应用——核心思路"在节点内做完能做的 collective，再跨节点传输最小化的数据"。与 NCCL EP（2026-06-27 覆盖）中 HT 模式的层级通信思路一致：**先聚合再传输**。对超节点架构的启示：权重流水线并行可能比传统激活流水线更适合带宽受限的跨 tray 场景。

> OpenUCX v1.21.0 的亮点：
> 1. **`UCX_MAX_HCA_PER_GPU`** — 这是对 NUMA 拓扑感知的直接加强：限制 GPU 只使用最近的 HCA 做 GDR，避免跨 socket 的 PCIe 延迟惩罚
> 2. **GDA 修复密集** — 6 个 GDA 相关修复覆盖了 DMA-BUF 偏移、WQE 排序、CUDA context 处理、GPU 映射等，表明 GDA 正在走向生产级成熟
> 3. **CUDA 13 完整支持** — v1.21.0 是首个完整支持 CUDA 13 的 UCX 版本

## 7️⃣ 📡 Nezha: 协议无关的多轨网络 — 为遗留集群释放 2.36× 潜力 (arXiv:2405.17870)

- **来源**: arXiv:2405.17870, "Nezha: Breaking Multi-Rail Network Barriers for Distributed DNN Training" (v3: 2026-02-09)
- **机构**: 国防科技大学
- **标签**: `Multi-Rail` · `Protocol Agnostic` · `SHARP` · `RDMA` · `TCP`

### 问题

> 超过 **60% 的生产 HPC 系统**仍依赖遗留基础设施（V100 GPU、多平面以太网/InfiniBand），现有方案三方面受限：
> 1. **静态单轨绑定** → 多轨带宽利用不足
> 2. **协议异构**（TCP-RDMA共存）→ 同步延迟
> 3. **主流库**（NCCL/MPI）缺乏跨协议协调

### Nezha 设计

Nezha 是首个**协议无关的多轨网络系统**：

| 核心贡献 | 技术详情 |
|:---------|:---------|
| **硬件无关的跨协议协调** | 统一抽象使 SHARP、GLEX（自适应RDMA）、TCP 无缝协作 |
| **协议感知动态负载均衡** | 冷/热启动状态机：小载荷降低启动延迟，大载荷提高吞吐 |
| **容错多轨协作** | 自恢复：单轨故障时 **200ms** 内重路由数据流 |

### 性能数据

| 场景 | 结果 |
|:-----|:------|
| **Homogeneous TCP-TCP** | 比 MPTCP 吞吐高 **74%** |
| **Heterogeneous TCP-SHARP** | 比 MPTCP 吞吐高 **80%** |
| **128 节点超算** | 比 Gloo 训练效率高 **2.36×** |
| **单轨故障恢复** | **200ms** 内自恢复，训练不中断 |

### 意义

> Nezha 对**大规模生产集群**有直接价值：多数集群跑着混合协议（部分链路用 RDMA、部分被迫用 TCP），Nezha 提供了不换硬件的优化路径。对于企业级数据中心场景，Nezha 的"协议无关"思路意味着可以在现有网络资产上获得 2× 的训练效率提升。

## 8️⃣ 📐 RAMC: HPE Slingshot 上的显式单边通信库 (arXiv:2606.05094)

- **来源**: arXiv:2606.05094, "RAMC: Remote Access Memory Channels over HPE Slingshot" (2026-06-03)
- **机构**: Sandia National Laboratories
- **标签**: `HPE Slingshot` · `One-sided Communication` · `MPI` · `RDMA`

### 设计

RAMC (Remote Access Memory Channels) 是面向 HPE Cray Slingshot 网络硬件的**显式单边通信库**：

| 维度 | 详细 |
|:-----|:------|
| **核心概念** | 持久化单向通信通道 |
| **利用 Slingshot 特性** | 内存区域计数器（memory region counters）实现高效完成通知 |
| **与传统方法对比** | MPI RMA / OpenSHMEM 依赖**共享内存模型**，RAMC 采用**显式通道模型** |

### 性能

| 消息大小 | 相比 Cray MPI 的带宽提升 (libfabric 1.15.2) | (libfabric 2.3.1) |
|:---------|:-------------------------------------------|:------------------|
| **1B-4KiB** 小消息 | **100-130%** 提升 | **30-45%** 提升 |
| **扩展性** | **19,600 进程 / 250 节点** 无扩展性问题 |

### 意义

> RAMC 证明 Slingshot 的硬件特性（内存区域计数器）可以比传统 MPI RMA 更高效地实现单边通信。对于**超节点中使用 Slingshot 交换机**的架构设计有参考价值——Slingshot 的显式 RDMA 语义通过 RAMC 可以释放比标准 MPI 高 2× 的小消息带宽。
