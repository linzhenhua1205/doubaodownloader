## 1️⃣ MLA 跨实例注意力路由：Move the Query, Not the Cache (arXiv:2606.01502, May 31)

**来源**: arXiv:2606.01502 | 作者: Bole Ma, Jan Eitzinger, Harald Köstler, Gerhard Wellein (FAU Erlangen-Nürnberg)

**核心洞察**: 前沿 LLM（DeepSeek-V3.2/V4、GLM-5.1）使用 **Multi-head Latent Attention (MLA)**，将每个 token 的 key/value 压缩为窄向量。当一个查询是 ~1 KB 时，**跨实例移动查询比移动 KV cache 更便宜**。

**问题背景**:
- Agentic 工作负载中，多个子 Agent 查询同一个大型代码库，复用相同 KV cache 块
- 当数据集超出单 GPU 容量，需跨实例分区存储 → 查询和它选择的 KV 块在不同 GPU 上
- 此前跨实例 KV 系统的直觉是：移动 cache（将选中块拉到请求方 GPU）
- MLA 反转了算术：压缩后的路由查询只有 ~1 KB，小于它要关注的 KV 块

**方法 — 三个可复用产物**:

1. **拓扑感知代价模型**: probe / transfer / compute / return / merge 五阶段
2. **闭式 route/fetch/local 预测器**: 决定应该路由查询、移动 cache、还是本地计算
3. **IBGDA 实测校准**: 模型追踪批量往返延迟误差 **~7%**

**在 H100 集群 + 真实 IBGDA 上的关键发现**:

| 策略 | 操作 | 代价 |
|:-----|:-----|:-----|
| **路由查询 (Move Query)** | tens-of-microseconds 往返 | **数十微秒** |
| **移动 cache (Move Cache)** | 连续块重组拼接 | **~3 ms** |
| **分散收集** | 收集选中 KV 块 | 高于路由查询 |

**关键结论**: decode 阶段路由查询最优——将 ~3 ms 的 cache 移动代价变为数十微秒的往返，且 fabric 选择依据 probe 延迟而非峰值带宽。

**📌 深度分析**: 这篇论文很好地衔接了跨实例推理（PD 分离场景）与集合通信（IBGDA 底层传输）。它对 **cross-instance KV 系统的架构选择有直接影响**：在 MLA 架构下，"移动查询"策略理论上可以将跨实例 KV 访问延迟降低 1-2 个数量级。成本模型的两个系数（路由负载和 cache 移动代价）只需对每种新架构测量即可扩展，不限于 MLA。

## 2️⃣ TensorHub: 生产级 RDMA 权重传输系统 (arXiv:2604.09107, April 2026)

**来源**: arXiv:2604.09107 | 作者: 阿里云/蚂蚁集团研究团队

**核心洞察**: LLM RL 训练需要频繁在 trainer 和 rollout actors 之间同步大模型权重。传统权重传输方案要么无法弹性扩缩集群，要么产生巨大的数据搬运开销。

**创新方案 — Reference-Oriented Storage (ROS)**:
- **不物理存储任何副本**：ROS 追踪哪些 worker 在 GPU 上持有模型权重
- 请求时直接让持有 GPU 提供读取服务（类似 pointer-based 的存储抽象）
- 结合拓扑优化传输、强一致性保证和容错机制

**在 TensorHub 上的实现和实测**:

| 场景 | 提升 |
|:-----|:-----|
| **RDMA 带宽完全饱和** | 生产级验证 |
| 单机 rollout GPU stall 时间 | **↓ 6.7×** |
| 弹性 rollout 权重更新 | **↑ 4.8×** |
| 跨数据中心 rollout stall 时间 | **↓ 19×** |

**部署**: 已在生产环境中支撑前沿 RL 训练。

**📌 深度分析**: TensorHub 展示了 RL 训练中权重传输的**存储即通信**范式——不再是"存好再传"，而是"追踪谁持有，直接读取"。这与 RC 面之前的 UCCL-EP（UCCL-EP 用 CPU proxy 替代 GPU-init RDMA）的理念互补。Topology-optimized transfer 暗示了与 NCCL 拓扑感知调度（如 `NCCL_TOPO_DUMP`）的集成可能。

## 3️⃣ UCCL-Zip: 无损压缩加速 GPU 通信 (arXiv:2604.17172, April 2026, 更新至 May)

**来源**: arXiv:2604.17172 | 作者: 伯克利/微软/阿里研究团队 (含 Ion Stoica)

**核心洞察**: 现有压缩集合通信主流采用量化和有损压缩引入精度误差。UCCL-Zip 将**无损压缩**直接集成到 GPU 通信原语中，无需修改用户 API 且不牺牲数值精度。

**双栈设计**:

**A. Uzip-P2P — 点对点无损压缩**:
- **Split-Send 流水线**: 提前暴露可传输数据，压缩与通信重叠
- **大数据块操作**: 保持高 GPU 效率

**B. Uzip-NCCL — 集合通信无损压缩**:
- 集成到 NCCL 的 **persistent kernel 模型**中
- **融合执行**: 消除冗余内存流量和 kernel 启动

**实测 (真实工作负载)**:

| 场景 | 加速 |
|:-----|:-----|
| RL 权重同步 | **↑ 47.5%** |
| vLLM 端到端推理延迟 | **↓ 10%** |
| 无需应用代码修改 | — |

**📌 深度分析**: UCCL-Zip 与 NCCLZ（05-12, arXiv:2605.12396）思路互补但实现不同：
- **NCCLZ**: 量化+熵编码解耦，有损压缩，最高 9.65×
- **UCCL-Zip**: 纯无损压缩，Split-Send 流水线 + persistent kernel 融合
- 两者分别解决不同场景：NCCLZ 追求极致压缩比（有损），UCCL-Zip 保证精度无损（金融/科学计算场景关键优势）

**适用场景对比**:

| 场景 | 推荐方案 |
|:-----|:---------|
| 训练梯度同步（可容忍小误差） | NCCLZ（量化+熵编码，9.65×） |
| RL 权重同步（需精度无损） | UCCL-Zip（无损，47.5% 加速） |
| 推理（延迟敏感） | UCCL-Zip（端到端 10% 延迟降低） |

## 4️⃣ GenModel: 重访 AllReduce 时间代价模型 (arXiv:2409.04202, 2024年9月, 2026年3月更新)

**来源**: arXiv:2409.04202 | 作者: 清华/阿里研究团队

**核心洞察**: 经典的 $(\alpha, \beta, \gamma)$ AllReduce 代价模型在现代集群上已失效，最少缺少两项关键贡献：**incast 项** 和 **内存访问项**。

**GenModel — 扩展后的五参数模型**:
- $\alpha$: 启动延迟
- $\beta$: 带宽倒换时间
- $\gamma$: 计算时间
- **新 +  $\delta$ (incast 项)**: 多节点同时发送到同一接收节点时的拥塞开销
- **新 +  $\epsilon$ (内存访问项)**: GPU 显存带宽约束导致的额外时间

**GenTree — 基于新模型的 AllReduce 算法生成**:

**实测 (64 GPU 真实集群)**:

| 对比 | 加速比 |
|:-----|:-------|
| GenTree vs NCCL | **1.22× ∼ 1.65×** |
| GenTree vs SOTA（大规模仿真） | **1.2× ∼ 7.4×**（incast/内存项主导时） |

**两个新最优性**: GenModel 发现两个新的算法最优性，并证明它们无法同时达到——需要在二者间权衡。

**📌 深度分析**: 这篇论文虽然提交时间较早（2024），但 2026 年 3 月的更新版本增加了更多实测。**关键价值**在于：现有集合通信库（包括 NCCL）的算法选择基于过时的代价模型。随着 GPU 数量和拓扑复杂度增加（incast 加剧），以及 HBM 带宽成为瓶颈（内存访问项），老模型给出的"最优算法"可能并非最优。GenTree 为 NCCL 未来的算法选择启发式提供了理论基础。

## 5️⃣ Laminar: Probe-First 调度范式 (arXiv:2602.13789, Feb 2026, 更新至 April)

**来源**: arXiv:2602.13789

**核心思想**: 在 exascale 级别 GPU 集群中，刚体拓扑任务留下碎片化的剩余容量。Laminar 采用**去中心化 probe-first, execute-later** 调度范式。

**关键设计**:
- **Zone 级概率流拆分**: 热路径控制平面工作保持 ~$\mathcal{O}(1)$
- **持久轻量级 Agent 的 bounded in-Zone 探测**
- **节点级仲裁**

**Airlock — 运行时生存层**:
- 将严重内存压力转化为有序策略：挂起 → 原地恢复 → 二级重寻址 → 回收
- 按优先级排序保留高价值长驻任务
- 逼近物理饱和但没有协议级超额承诺

**📌 深度分析**: 拓扑感知调度从"选最优拓扑"进入了"碎片生态下的生存调度"。Laminar 的 probe-first 思路与 MonkeyTree（06-04 已追踪）的迁移型拥塞缓解形成互补：MonkeyTree 解决集群拥塞，Laminar 解决碎片化调度。

## 6️⃣ 趋势判断更新 (2026-06-06)

| 方向 | 热度变化 | 关键新信号 |
|:-----|:---------|:-----------|
| MLA 跨实例注意力路由 | 🔥🔥🔥🔥🔥 **新增🔥** | Move Query > Move Cache，数十 μs vs ~3 ms，IBGDA 实测 |
| RDMA 权重传输 → 存储即通信 | 🔥🔥🔥🔥 **新增🔥** | TensorHub 6.7x stall ↓，ROS 抽象，生产中验证 |
| 无损压缩集合通信 | 🔥🔥🔥🔥 **新增🔥** | UCCL-Zip 47.5% RL 加速，与 NCCLZ 互补 |
| AllReduce 代价模型突破 | 🔥🔥🔥 **新增🔥** | GenModel 五参数模型，incast+内存项，1.22-1.65× vs NCCL |
| 去中心化调度 + 运行时生存 | 🔥🔥 **新增🔥** | Laminar probe-first + Airlock 生存层 |

### 📊 核心判断

1. **MLA 架构改变跨实例通信的基本策略**: Move the Query, Not the Cache 是继 PD 分离后的又一架构级洞察——注意力压缩做得越极致（MLA），跨实例通信越应该路由查询而非移动 KV cache。对推理集群的拓扑设计和网络选型（尤其 IBGDA）有直接指导意义。

2. **RDMA 权重传输正走向"存储即通信"范式**: TensorHub 的 ROS 抽象将 GPU 显存直接作为可寻址存储，这是 RL 训练弹性扩缩场景下的关键架构创新。与 UCCL-EP 的 CPU proxy 方案形成两条路径。

3. **无损 vs 有损压缩的双轨并行**: NCCLZ（有损，9.65×）和 UCCL-Zip（无损，47.5% RL 加速）各有适用场景。未来 NCCL 可能同时集成两种路径，由用户根据精度要求选择。

4. **AllReduce 的理论基础正在被重写**: GenModel 揭示了经典 $(\alpha, \beta, \gamma)$ 模型在现代 GPU 集群上的不足。incast 和内存访问项在大规模场景下不可忽略，这解释了为什么 NCCL 在某些拓扑上表现不如预期。
