## 1️⃣ 🏆 SAC: CXL 解聚 KV Cache 系统 for 稀疏注意力 LLM (arXiv:2606.19746, Jun 18) 🆕

**链接**: https://arxiv.org/abs/2606.19746

**核心问题**: 长上下文 LLM 推理的瓶颈从计算转向**内存容量**。现有 RDMA 解聚方案在解码前**粗粒度拉取整个 prefix KV 缓存**到本地，这对稀疏注意力模型**根本性低效**——解码时仅少量 KV 条目活跃，却传输全部缓存。

**SAC 方案**: 首个针对稀疏注意力模型优化的高效解聚 KV Cache 系统。

| 技术要点 | 机制 |
|:---------|:------|
| **CXL 细粒度访问** | 利用 CXL 的 cache-line 粒度 load/store 语义 |
| **按需拉取** | 推理时只按需拉取 top-k KV 条目 |
| **集成** | 基于 SGLang，在 DeepSeek-V3.2 上验证 |

**性能对比 (vs RDMA 基线)**:

| 指标 | 加速比 |
|:-----|:-------|
| **吞吐 (Throughput)** | **2.1×** |
| **TTFT (首 Token 延迟)** | **9.7× 降低** |
| **TBT (Token 间延迟)** | **1.8× 降低** |

**🎯 意义**:
- 证明了稀疏注意力模型（如 DeepSeek-V3.2）的解聚基础设施应当使用 **CXL 而非 RDMA**
- CXL 的 cache-line 粒度访问与稀疏注意力的按需访问模式**天然匹配**
- 对分布式推理系统的 KV Cache 架构设计提供根本性方向转变：CXL 内存池化 → 替代 RDMA 解聚 → 成为主流

## 2️⃣ ARGUS: 万卡集群生产级追踪与性能诊断 (arXiv:2606.20374, Jun 18) 🆕

**链接**: https://arxiv.org/abs/2606.20374

**核心问题**: 万卡级 LLM 训练需要始终在线、细粒度的可观测性。粗粒度监控无法定位根因，而细粒度 profiler 引入 5%-30% 的开销和海量追踪数据。

**ARGUS 方案**:

| 技术 | 机制 |
|:-----|:------|
| **三路分解** | CPU 调用栈 + 框架语义 + GPU Kernel 执行 |
| **始终在线** | **总开销 <2%** |
| **数据压缩** | 原始 kernel events 从 10 MB 压缩至 **2.7 KB / rank / step** (**~3,700× 压缩**) |
| **渐进诊断框架** | 迭代时间→Phase 级→Kernel 级，自动隔离异常窗口、straggler rank、退化 kernel |

**生产部署**: 10,000+ GPU 集群运行 **6 个月以上**，持续支撑 fail-slow 检测和性能优化。

**案例覆盖异常类型**:
- 计算 straggler ✅
- 链路退化 (Link degradation) ✅
- 流水线气泡放大 (Pipeline-bubble amplification) ✅
- FlashAttention JIT stalls ✅
- 通信症状掩盖的计算 straggler ✅

**🎯 意义**:
- **<2% 开销 + 3,700× 压缩** = 始终在线可观测性成为万卡集群标配
- 渐进诊断框架(AIG: Anomaly Isolation → straggler → degraded kernel)是生产级诊断标准套路
- **五个案例覆盖了分布式训练最典型的异常模式**，特别是"通信症状掩盖计算 straggler"——这是 NCCL 监控盲区

## 3️⃣ Execution-State Capsules: 完整执行状态检查点/恢复 (arXiv:2606.20537, Jun 18) 🆕

**链接**: https://arxiv.org/abs/2606.20537

**核心思想**: 主流 LLM 服务系统仅管理 KV Cache（执行状态的一个位置片段）。本文研究**反极**场景：低延迟、小批量、边缘设备物理 AI 服务（交互式 LLM Agent、语音系统、机器人策略），需要频繁分支、重置、中断和重新进入。

**方法**: Execution-State Capsules — 有状态计算边界上的**完整可恢复状态检查点/恢复机制**。

| 组件 | 机制 |
|:-----|:------|
| **FlashRT 运行时** | 白盒后端 CUDA runtime，连续静态缓冲区运行捕获的计算图 |
| **完整状态** | KV + 循环状态 + 卷积状态 + MTP 状态 + 元数据 |
| **操作** | snapshot / restore / fork / rollback |

| 场景 | 加速比 |
|:-----|:-------|
| **GPU 驻留快照/恢复** | **亚毫秒级** (sub-millisecond) |
| **TTFT vs 冷 prefll (2K tokens)** | **3.9×** |
| **TTFT vs 冷 prefill (16K tokens)** | **27×** |
| KV-only ablation → 发散 | 证明 **循环状态是负载支撑的关键** |

**验证平台**: RTX 5090 ✅ / Jetson AGX Thor ✅ / DGX Spark ✅

**🎯 意义**:
- 与主流 KV cache 方案**互补而非替代**：高吞吐服务继续 KV cache，延迟敏感场景用 Capsules
- **完整执行状态复用**（不只是 KV）—— 对交互式 Agent / 机器人 / 语音等频繁打断场景是质的飞跃
- 亚毫秒级 GPU 驻留快照 = 状态化推理基础设施的**新基元**

## 4️⃣ 量子 Ring All-Reduce: 通信与隐私双重优势 (arXiv:2606.20344, Jun 18) 🆕

**链接**: https://arxiv.org/abs/2606.20344

**核心贡献**: 首次探索量子通信如何让分布式训练**既通信高效又信息论安全**。

| 优势 | 机制 | 量化 |
|:-----|:------|:------|
| **带宽节省** | 预共享纠缠 + 超密编码 | **2× 通信减少** (每链路在线通信减半) |
| **隐私保证** | 可验证纠缠 → 可组合 ε-安全聚合 | 经典协议**信息论不可能**实现 |
| **梯度冲突检测 GapIP** | 二次优势 | $\widetilde{O}(\tau^{-1}\log P)$ qubits vs $\widetilde{O}(\min(\tau^{-2},P))$ bits |
| **符号一致性审计 TieAudit** | **指数级分离** | $\Omega(\sqrt{P})$ bits vs $O(\epsilon^{-2}\log P)$ qubits |

**亮点**: 无需修改学习模型或梯度计算方式，对经典和量子学习模型均适用。

**🎯 意义**:
- **混合量子-经典通信架构**首次被系统提出，在 ring all-reduce 这个最基础的原语上量子化
- 信息论隐私是经典协议**永远无法实现**的特性
- 通信分离（TieAudit 的指数级优势）具有理论深刻性

## 5️⃣ UltraEP: 超大规模 MoE 精确负载实时均衡 (arXiv:2606.04101, 更新 Jun 18) 🔄

**链接**: https://arxiv.org/abs/2606.04101

**核心问题**: 大规模 Expert Parallelism (EP) 加剧专家负载不均衡，导致计算 straggler、所有连接通信瓶颈、激活内存尖峰。

**UltraEP 方案**: 首个**精确负载**、**实时**均衡器，针对 MoE 训练和 serving prefill。

| 技术 | 机制 |
|:-----|:------|
| **实时重均衡** | 每个 microbatch 和 layer 在关键路径重均衡 |
| **配额驱动规划器** | 事后 gating 负载感知，高效求解 |
| **RSN 原生 Tile Streaming** | 持久 tile 传输 + 中继 fan-out 缓解 |
| **跨节点互联** | 利用机架级节点 (RSN) 的 scale-up 互联 |

**性能 (256 GPU, MoE 106B-671B)**:

| 指标 | 数值 |
|:-----|:------|
| **接近理想吞吐** | **94.3%** of force-balanced ideal |
| **vs 无均衡** | **1.49×** improvement |
| **最终 rank 间不均衡** | **1.01-1.04** (基线 1.30-4.01) |

**🎯 意义**:
- UltraEP 与 MoE-Hub (ISCA'26) 和 DisagMoE 形成 **MoE 通信三件套**：硬件范式(UltraEP×MoE-Hub) + 软件均衡(UltraEP) + 分离训练(DisagMoE)
- 94.3% of force-balanced ideal 表明**软件均衡已接近理论极限**
- RSN 原生技术路线表明：**MoE 通信优化正从纯软件转向 scale-up 互联协同设计**

## 6️⃣ ODB: 在线动态 Batching 带形式化保证 (arXiv:2606.19989, Jun 18) 🆕

**链接**: https://arxiv.org/abs/2606.19989

**核心问题**: LLM 训练中样本的真实成本只有在预处理、增强、模板化、tokenization 和多模态视觉 token 扩展之后才可观测。离线批处理采样器在 batch 构造时**盲于真正的计算成本**。

**ODB 方案**: DataLoader 端的 drop-in 动态 batching 系统，在准确可观测时构造 batch。

| 技术 | 机制 |
|:-----|:------|
| **在线动态 Batching** | 在预处理后、GPU 前构造 batch |
| **DGAP 形式化保证** | Distributed Group Alignment Problem → 无死锁有界终止证明 |
| **零侵入** | 无需修改模型、优化器、attention kernel |

**性能 (Qwen3-VL 2B/8B)**:

| 场景 | 加速比 vs 固定 Batch |
|:-----|:---------------------|
| 单节点 Full FT/LoRA | **1.58-2.51×** |
| 两节点 Full FT | **1.71-3.78×** |
| 生产 MM-Mix | **4.43×** |
| 高 CV ShareGPT4o | **2.24-3.69×** |

**🎯 意义**:
- 对高样本异构性（多模态、不同长度）的 LLM 微调场景有巨大价值
- **数学上证明了无死锁有界终止** — 这在动态系统的形式化保证中非常罕见
- 实操层面：无需任何模型/优化器/kernel 改动，**drop-in 部署**
