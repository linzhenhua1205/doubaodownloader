## 1️⃣ OpenURMA：Huawei UB 传输协议的纯室开源实现 (arXiv:2605.28717, May 27)

**来源**: arXiv:2605.28717 | 作者: Bojie Li (独立研究者/华为前员工?)  
**链接**: [arxiv.org/abs/2605.28717](https://arxiv.org/abs/2605.28717)

### 背景：RDMA 的 PCIe 瓶颈

现代数据中心 RDMA 的瓶颈**不在网络线缆上，而在网络接口上**。运行 RoCE 或 InfiniBand 的 NIC 持有每个 (应用, 远端端点) 对的连接状态 —— 1024 应用扇出时达到**数百 MB**，且在 64-byte 操作上经历**四次 PCIe 往返**，将延迟放大一个数量级。根源在于 RDMA 从 InfiniBand 继承的 **Queue Pair over PCIe 抽象**。

### Huawei UB 协议的核心变革

Huawei **Unified Bus (UB)**，2025 年发布的公开规范，改变了这个抽象：

| 维度 | 传统 RDMA (RoCE/IB) | UB 协议 |
|:-----|:--------------------|:--------|
| **连接状态** | 每 (应用, 端) 一对 → 线性增长 | 每应用端点状态与每主机传输状态**解耦** → **加性增长** |
| **排序** | 强制严格保序 | **排序可选 (opt-in)** |
| **内存访问方式** | QP over PCIe, 四次往返 | **原生 CPU load/store** 到片上总线控制器 |
| **端到端 64B 延迟** | ~2186 ns | **~500 ns** (4.37× 更低) |
| **吞吐** | 基线 | **2.80× 更高** |

### OpenURMA 实现

OpenURMA 是首个 UB 传输层和事务层的纯室开源实现，包含三层：

| 层级 | 实现方式 |
|:-----|:---------|
| **RTL** | Alveo U50 FPGA，可综合，带配对 OpenRoCE (RoCEv2 RC) 基线 |
| **SystemC** | 双节点周期级模拟器 |
| **gem5** | 全系统完整架构模拟 |

**关键性能数据**:
- 64-byte remote fetch (UB Sec.8.3 LOAD vs RoCEv2 RC READ)：**500 ns vs 2186 ns → 4.37× 延迟降低**
- 吞吐：**2.80× 高于 RoCE**
- 资源：仅消耗 **~14% 的 U50 LUTs**
- UB 芯片只在华为闭源 Ascend 950 硅片中可用，**OpenURMA 首次使社区能在标准 FPGA 上评估 UB 架构**

**行业意义**: Huawei UB 对 RDMA 的根本性重构（解耦连接状态 + CPU load/store 语义）首次通过开源实现得到验证。如果 UB 成为 DC 互联的新方向，将对现有 RoCE/IB 生态产生颠覆性影响。⚠️ 此为独立实现，未经验证 UB 规范完全兼容性。

## 2️⃣ DMA Streaming Framework / dmaplane：内核级 RDMA 缓冲区编排 (arXiv:2603.10030, Mar 25)

**来源**: arXiv:2603.10030 | 作者: Marco Graziano  
**链接**: [arxiv.org/abs/2603.10030](https://arxiv.org/abs/2603.10030)

### 问题

AI 传输库（NCCL / RCCL / UCX）高效地搬运字节，但它们假设缓冲区已经被正确地分配、放置、共享、注册，且在完成/拆除压力下安全。**这个"中间层"从未被显式抽象**。

### dmaplane 设计

dmaplane 是一个 Linux 内核模块，将缓冲编排作为显式服务暴露：

| 组件 | 功能 |
|:-----|:------|
| **/dev/dmaplane UAPI** | 稳定内核用户态接口 |
| **基于 ring 的命令通道** | 低开销命令提交 |
| **DMA 缓冲区生命周期管理** | 分配→注册→共享→回收全周期 |
| **dma-buf export** | 跨设备共享 (GPU ↔ NIC ↔ CPU) |
| **内核态 RDMA 引擎** | 在模块内部直接执行 RDMA 操作 |
| **NUMA 感知分配 + 校验** | 避免跨 NUMA 惩罚 |
| **基于 credit 的流控** | 防止缓冲区溢出 |
| **PCIe BAR pinning** | GPU 显存直接映射到 PCIe BAR，绕开 cudaMemcpy |

### 端到端验证

在 Soft-RoCE 上实现了跨机 KV-cache chunk 传输：使用 **RDMA WRITE WITH IMMEDIATE** 传输 KV-cache，在接收端重建 tensor 视图。

### 意义

dmaplane 代表了一种**系统软件层级的新方向**：将当前被各库（NCCL/UCX/MPI）重复实现的缓冲区编排逻辑，抽象为通用的内核基础设施。这与 RDMA、CXL、NVLink 等互联技术的系统软件堆栈演进趋势一致。

## 3️⃣ HetCCL (新)：Border-Communicator 机制的跨厂商集合通信 (arXiv:2605.31000, May 29)

**来源**: arXiv:2605.31000 | 作者: Yuejie Wang et al.  
**链接**: [arxiv.org/abs/2605.31000](https://arxiv.org/abs/2605.31000)

> ⚠️ 与早期 HetCCL (arXiv:2601.22585, Kim et al., Jan 2026) 是**独立的两项工作**，本项为不同团队提出的不同方案。

### 核心创新

| 技术 | 原理 | 效果 |
|:-----|:------|:------|
| **Border-Communicator 机制** | 在聚合集合通信中利用**厂商集体通信库内在的 reduction 操作**实现厂商无关性 → 消除 host-device 内存拷贝 | 跨厂商功能正确，性能匹配单厂商原生库 |
| **高效 P2P 异构传输** | 跨厂商 GPU 间直接 RDMA 传输，控制面卸载到 CPU | 无驱动修改 |
| **层次化拓扑抽象** | 将集合通信分解为集群级原语，保证最优跨集群数据传输量和带宽利用 | — |

### 性能数据

| 场景 | 对比对象 | 加速比 |
|:-----|:---------|:-------|
| 异构 P2P 带宽 | Gloo (传统跨厂商方案) | **17-19× 更高** |
| 异构集群端到端训练 (per-step-time) | — | **↑ 16.9%** |
| 同构场景 | NCCL / RCCL | **基本持平** |

**实现规模**: 支持 4 种厂商后端，在 4 种异构配置下完成评估。

**意义**: 随着 GPU 供应链多元化（NVIDIA + AMD + 国产），跨厂商集合通信成为必须。本方案同构无损 + 异构显著的特性使其具备生产级部署潜力。

## 4️⃣ OpenUCX v1.21.0-rc1 (May 24, 2026) 关键更新

**来源**: [github.com/openucx/ucx/releases/tag/v1.21.0-rc1](https://github.com/openucx/ucx/releases/tag/v1.21.0-rc1)

### GPU 通信重大增强

| 功能 | 详情 |
|:-----|:------|
| **UCX_MAX_HCA_PER_GPU 策略** | 限制 GPU 内存注册到最近 HCA → 避免远端 NIC 导致的延迟惩罚 |
| **Device lanes 支持 host 内存** | GPU 传输 fallback 路径，设备 lane 可访问 host 内存 |
| **gdr_copy 启用 memtype endpoint** | GDR copy 现在可通过 memtype endpoint 传输使用 |
| **UCP 协议模拟选项** | `UCX_PROTO_EMULATION_ENABLE` 强制零拷贝 RMA 协议选择 |

### RDMA CORE 修复

| 修复 | 详情 |
|:-----|:------|
| **GDA DMA-BUF 偏移修复** | 修复 GPU DMA-BUF 在 interior addresses 的偏移处理 |
| **GDA WQE 排序修复** | 使用 **CAS-based readiness marking** 保证 WQE 正确排序 |
| **GDA 保留无效 CUDA 上下文** | `UCX_IB_GDA_RETAIN_INACTIVE_CTX` 选项控制 |

### Bugfix 影响

- 修复 CUDA IPC 缓存分离（local vs remote rkeys）
- 修复 CUDA IPC cache/LRU 不变性，修复 DMA-BUF 内部地址偏移
- 修复 RoCE GDA 相关问题
- 修复 HIP MPI 和 OMB 测试的 ROCm 挂起问题

**相比 v1.20.1（May 6）亮点**: v1.21.0-rc1 主要增加了 GPU 亲和性策略 + 协议模拟选项 + 关键 RDMA 排序修复。v1.20.1 的关键特性是 **GDAK (GPU Direct Async Kernel-Initiated) 的 GDAKI 命名修正** 和与 v1.20.0 (Feb 5) 一致的完整 GDA/GDAKI 功能基线。

## 5️⃣ RoCE BALBOA：开源 100G RoCE v2 FPGA 实现 (arXiv:2507.20412, Jul 2025)

**来源**: arXiv:2507.20412 | ETH Zurich  
**链接**: [arxiv.org/abs/2507.20412](https://arxiv.org/abs/2507.20412)

### 设计

RoCE BALBOA 是首个**完全开源**、**RoCE v2 兼容**、**100G 级 FPGA 实现**的 RDMA 协议栈：

| 特性 | 数据 |
|:-----|:------|
| 速度 | 100Gbps |
| QP 规模 | 数百队列对 |
| 兼容性 | RoCE v2 RC，与商业 NIC 相当 |
| 可定制性 | 完全开放 → 可嵌入加密、深度包检测、在线计算 |

### 两个用例

1. **协议增强**: 加密、基于 ML 的深度包检测 → 在可编程网关上直接实现
2. **在线计算卸载**: 实现推荐系统的**数据预处理管线**，数据流经网络时即完成预处理，直接传输到 GPU，**不需要 CPU 参与**

### 意义

BALBOA 代表了 SmartNIC/可编程网络的新方向：**不再依赖闭源商业 NIC**，而是通过开放 FPGA 实现自定义 RDMA 协议栈。对于 GPU 集合通信场景，它使"在数据到达 GPU 前完成聚合/过滤/压缩"成为可能。

## 6️⃣ OptiNIC：ML 专用的弹性 RDMA NIC (arXiv:2512.22743, Dec 2025)

**来源**: arXiv:2512.22743 | Hyperstack / UC Davis / VMware / Purdue  
**链接**: [arxiv.org/abs/2512.22743](https://arxiv.org/abs/2512.22743)

### 核心理念

现有 RDMA 传输 (RoCE / IRN / SRNIC) 强制执行**严格可靠性和保序交付** → 依赖重传和包排序。这在 ML 训练中存在问题：即使是**罕见的包延迟也会停滞整个模型流水线**。

OptiNIC 基于 ML 对**部分/缺失数据的容忍性**，重新审视可靠性保证：

| 特性 | 传统 RDMA | OptiNIC |
|:-----|:----------|:--------|
| 重传 | 强制 | **全部消除** |
| 保序交付 | 强制 | **out-of-order** |
| 完成信号 | 全数据交付完成才触发 | **自适应超时触发向前推进** |
| 丢包恢复 | NIC 内部 | **ML 管线自身** (Hadamard Transform + Erasure Coding) |
| 拥塞控制 | 保留 DCQCN/EQDS/Swift | 保留 |

### 性能数据

| 指标 | 提升 |
|:-----|:------|
| 时间到精度 (TTA) - 训练 | **↑ 2.0×** |
| 吞吐 - 推理 | **↑ 1.6×** |
| 99 分位延迟 | **↓ 3.5×** |
| BRAM 使用 | **↓ 2.7×** |
| 故障弹性 | **几乎翻倍** |

### 与 Celeris 的关系

Celeris (IEEE CAL 2025, arXiv:2510.16606, Oct 2025) 是 OptiNIC 团队的前期工作。Celeris 展示 2.3× tail latency 降低、67% BRAM 减少。**OptiNIC 是 Celeris 的完全实现版本**，在 Hyperstack 和 CloudLab 两个公云上完成端到端验证。

**意义**: ML 专用 RDMA NIC 正成为一个明确的方向 —— 传统 RDMA 的严格可靠性假设在 ML 场景下是**过度约束**，放弃这些约束可以同时获得更好的性能和弹性。
