## 1. NCCL v2.30.7-1 — Zero-SM Collectives 正式到来 🚀

- **来源**: [NVIDIA NCCL Release v2.30.7-1](https://github.com/NVIDIA/nccl/releases/tag/v2.30.7-1) (2026-06-04)
- **版本**: 2.30.7-1

### 核心特性

#### 🔥 Zero-SM Collectives（分层零SM通信）
NCCL 新增**层次化零SM集合通信**（AllGather 和 All2All），核心机制：
- **节点间**：使用 RMA CPU Proxy 进行通信，不占用 SM
- **节点内**：使用 GPU Copy Engine 进行数据搬运
- 通过 `NCCL_CTA_POLICY_ZERO` 环境变量启用
- **效果**：SM 完全释放给计算，通信与计算可真正重叠

#### 🔥 GIN GPI 后端（GPU Push Interface，实验性）
- 新增 **GPI（GPU Push Interface）** 传输后端作为实验特性
- 引入 Strong/Weak 信号语义，提供显式信号化通信模型
- 新增 `ncclGinFenceLevel` 屏障语义
- 独立的 `NCCL_GIN_IB_TC` 控制 GIN 流量类别
- `NCCL_GIN_RESOURCE_SHARING_THREAD` 选项启用更多优化
- QP 开销优化：GDAKI 模式下跳过计数器管理
- GIN 与 NIC Fusion 兼容

#### 🔥 Symmetric Memory（对称内存）重大重构
- RMA 插件架构重组，支持**非对称 buffer 大小**的窗口注册
- ReduceScatter 对称 kernel 性能优化
- 批量化 CE（Copy Engine）操作优化 RMA CE put/wait 路径性能
- 支持 CUDA graph 捕获时的窗口注册

#### 🔥 MPS + MLOPart（实验性）
- NCCL 首次利用 CUDA MLOPart 特性
- 单物理 GPU 支持最多 **2 个 rank** 通过 MPS+mlopart 共享

#### 🔥 QP 负载均衡 RoCE LAG 修复
- 修复了 RoCE LAG 配置下 QP 负载均衡问题，采用 round-robin queue affinity 策略
- 解决了多链路场景下 QP 分布不均导致的性能瓶颈

#### 其他改进
- IB 端口 GRH（Global Route Header）支持
- NCCL Device API 新增自定义 AllReduce/AllToAll kernel（`contrib/custom_algos/`）
- 新增 RMSNorm 示例，演示 Device API 融合计算与通信
- 编码风格统一为 clang-format
- 废弃 v11/v12 GIN 插件 API

## 2. NCCL EP v0.1.0 — 首个 MoE 集合通信扩展库

- **来源**: [NCCL EP v0.1.0 Release](https://github.com/NVIDIA/nccl/releases/tag/nccl-ep-v0.1.0) (2026-06-08)
- **定位**: 基于 NCCL Device API 的专家并行（EP）通信专用扩展库

### 核心特性
- **双模式**：HT（High-Throughput）模式和 LL（Low-Latency）模式
- HT 模式支持 JIT 编译、零拷贝、CUDA Graph 兼容
- 灵活 Dispatch 输出布局：Flat、Expert-major、Rank-major
- **活跃 rank 掩码**：识别失败 rank 并排除，避免整任务 abort
- 显式 Forward/Backward pass 选择器
- 变量 token 数支持（MoE prefill 场景）
- 8+ 节点支持修复（原 HT 模式限制 8 节点，v0.1 完全解除）

### 性能优化
- `ncclTeamRail` 替代 split communicator，优化 HT 模式性能
- LL 模式 staging buffer 按 rank 去重，减少专家数倍内存
- Rank-major 布局将内存占用缩减为 expert-per-rank 分之一

### 已知限制
- HT 模式下 RDMA 域（LSA Team）最多 32 个
- LL 模式 top-K ≤ 9，hidden dim 限 8 种特定值
- 当前版本不支持量化

## 4. OptCC — 首个 AllReduce 故障容错信息论下界 📐

- **来源**: arXiv:2606.01680, "Don't Let a Few Network Failures Slow the Entire AllReduce" (2026-06-01)
- **机构**: 华为 + 约翰霍普金斯大学

### 核心贡献

1. **信息论下界**：首次给出非对称带宽下 AllReduce 完成时间的信息论下界
   - 当 straggler（故障节点）保留 ≥50% 原始带宽时，与无故障最优的差距仅为 **O(1/p)**（p 为 GPU 数）
   - 即：故障的惩罚随集群规模增大而衰减

2. **OptCC 算法**：四阶段流水线 AllReduce 设计
   - 基于下界的理论最优逼近
   - 在 SimAI 仿真上验证

### 性能数据
- **实际故障（50% 带宽损失）**：OptCC 仅退化 **2-6%**（vs NCCL 无故障基准）
- **SOTA 对比**：现有故障容错方案退化高达 **57%**
- 差距从 57% 缩至 2-6%，**10× 改善**

### 意义
从"工程 hack"走向"理论指导"的故障容错——NCCL 当前的容错策略（故障 reroute 到幸存 NIC）远未达到理论最优。这是对分布式训练可靠性的根本性突破。

## 5. VCCL — 可观测·可靠·高效的集合通信库（已开源）🔬

- **来源**: arXiv:2510.00991, "An Efficient, Reliable and Observable Collective Communication Library" (2025-10, 更新于 2026-05-31)
- **机构**: 月之暗面（Moonshot AI）+ 清华大学
- **开源**: 已在生产环境部署数月

### 三大核心设计

#### 🧹 SM-Free P2P
- **问题**：NCCL 的 P2P kernel 消耗 SM，导致计算与通信争抢
- **方案**：将节点内数据搬运和流依赖管理卸载到 CPU 线程 + GPU Copy Engine
- **效果**：训练吞吐提升 **5.28%**

#### 🛡️ 主备 QP 容错
- **问题**：NIC 端口故障导致昂贵的任务重启
- **方案**：主备 QP 机制，链路故障时无感切换
- **效果**：大幅减少 GPU 资源浪费

#### 📊 O(μs) 级网络异常监控
- **问题**：NCCL 缺乏细粒度可观测性，短暂异常难以捕获
- **方案**：基于滑动窗口的监控器，微秒级检测网络异常
- **效果**：首次实现训练过程中的实时异常感知

## 6. NCCLbpf — eBPF 安全策略验证

- **来源**: arXiv:2603.11438, "NCCLbpf: Verified, Composable Policy Execution for GPU Collective Communication" (2026-03-11, 更新于 2026-05-04)
- **核心理念**：将 eBPF 安全沙箱引入 NCCL 插件系统

### 性能数据
- **额外开销**：每次 tuner 决策仅 **80-130ns**（不到集合通信延迟的 0.03%）
- **安全验证**：100% 拦截所有测试的不安全插件行为
- **性能提升**：消息尺寸感知的 eBPF 策略在 4-128 MiB 范围内提升 AllReduce 吞吐 **27%**

## 7. DynaTrain — 亚秒级在线并行度切换

- **来源**: arXiv:2605.18815, "DynaTrain: Fast Online Parallelism Switching for Elastic LLM Training" (2026-05-12)
- **机构**: 中科院计算所 + Infinite (AI 公司)
- **代码**: [github.com/infinigence/ElasticMegatron](https://github.com/infinigence/ElasticMegatron)

### 核心机制

- **VPS（Virtual Parameter Space）**：统一所有分布式训练状态至单一逻辑坐标空间
- **状态路由-转换层**：内存感知、无死锁的 rank-local 传输调度
- **Elastic Device Manager**：新 world 构建与正在进行的训练重叠，掩盖拓扑变更开销

### 性能数据
- **70B 稠密模型**：**<2s** 完成重新配置
- **235B MoE 模型**：**4.36s** 完成重新配置
- 相比 SOTA checkpoint 和弹性系统加速 **3 个数量级**

## 10. SCIN — Switch-Centric In-Network 架构

- **来源**: arXiv:2603.28239, "A Switch-Centric In-Network Architecture for Accelerating LLM Inference" (2026-03-30)
- **对比**：针对 NVLink SHARP（NVLS）的两大局限——GPU 主动加载导致冗余、无法卸载不可分解算子

### 核心创新
- **ISA（In-Switch Accelerator）**：交换机内加速器，直接访问 GPU 内存
- **INQ（In-Network Quantization）**：All-Reduce 精度降至 8-bit，带宽近乎翻倍

### 性能数据
- **小消息 All-Reduce**：加速 **8.7×**
- **大消息 All-Reduce**：加速 **3.8×**
- **TTFT 加速**：**1.74×**
- **TPOT 加速**：**1.34×**

## 13. FlexLink — NVLink+PCIe+RDMA 多链路聚合

- **来源**: arXiv:2510.15882, "FlexLink: Boosting your NVLink Bandwidth by 27%" (2025-08)
- **方案**：将 NVLink、PCIe、RDMA NIC 聚合为统一通信平面，两阶段自适应负载均衡
- **性能数据**：AllReduce 提升 **26%**，AllGather 提升 **27%**，仅卸载 2-22% 流量到原本空闲的 PCIe/RDMA

## 14. GenTree — 新的 AllReduce 时间模型

- **来源**: arXiv:2409.04202, "Revisiting the Time Cost Model of AllReduce" (2024-09)
- **发现**：经典的 (α,β,γ) 模型在现代集群上已不适用，新增 incast 项和内存访问项

## 15. OpenUCX v1.21.0 — GPU 内存注册优化

- **来源**: [UCX v1.21.0](https://github.com/openucx/ucx/releases/tag/v1.21.0) (2026-06-24)
- **核心新增**：
  - `UCX_MAX_HCA_PER_GPU` 策略：限制每 GPU 关联的 HCA 数，优化 GPU 内存注册
  - `UCX_IB_GDA_RETAIN_INACTIVE_CTX`：控制 GDAKI 中非活跃 CUDA context 的保留策略
  - CUDA 13 全面支持（aarch64 + x86_64）
  - GDA（GPU Direct Async）多项修复：DMA-BUF offset、WQE ordering（CAS-based）、CUDA_VISIBLE_DEVICES 隐藏物理 GPU 时的正确映射
