## 1️⃣ 🏆 CrossPool: 冷 MoE 多模型 KV-Cache 与权重解聚服务 (arXiv:2606.24506) 🆕

**链接**: https://arxiv.org/abs/2606.24506

**核心问题**: 新兴 LLM 服务承载大量稀疏 MoE 模型，多数模型请求稀疏处于"冷"状态。模型权重是稳定的、模型决定的；KV-cache 是瞬态的、需求决定的。冷模型很少同时达到 KV-cache 峰值需求，为每个模型预留最坏情况 KV 容量浪费内存。但单纯 KV-cache 共享不足，因权重和 KV-cache 在同一 GPU 内存池中竞争。

**CrossPool 方案**: 将 FFN 权重和 KV-cache 分离到两个独立的 GPU 内存池。

| 组件 | 职责 |
|:-----|:------|
| **Weights Pool** | 跨冷模型整合 FFN 权重 |
| **KV-cache Pool** | 动态服务活跃请求，保持 attention 本地化 |
| **KV-cache Planner + Virtualizer** | 规划与虚拟化 KV-cache 分配 |
| **Layer-wise Pipeline Scheduler** | 隐藏隐状态传输开销 |
| **Persistent Kernels + Control Lowering** | 降低 CPU-GPU 控制开销 |

**核心数据**:
- **P99 TBT 降低最多 10.4×** over state-of-the-art kvcache-based 多 LLM 服务系统
- 支持爆发性长上下文请求

**相关性**: ★★★★★ — 分布式推理 OS 层优化，KV-Cache 与权重分离的内存管理范式，层间流水线调度

## 2️⃣ 🔥 Blackwell GPU 机密计算下的 LLM 性能剖析 (arXiv:2606.23969) 🆕

**链接**: https://arxiv.org/abs/2606.23969

**核心发现**: 在 NVIDIA B300 上，BF16 matmul 在机密计算模式下达到 **0.998×** 非机密性能。然而，在 Intel TDX + GPU-CC 下 LLM 服务仍损失 **13-27%** 吞吐，KV-cache restore 延迟可能翻倍以上。根因是**机密 VM-GPU bridge（序列化通道）**，而非 GPU 计算本身。

**核心机制与数据**:

| 发现 | 数据 |
|:-----|:-----|
| 根因定位 | 机密 VM-GPU bridge 成为带高固定开销的序列化通道 |
| 小型 alloc-and-copy 开销 | **44× 更慢**（在 vLLM dense decode 场景） |
| Scheduling flag 恢复 | 恢复 **57%** 差距 |
| Worker-thread drain 恢复 | 恢复最高 **92%**（高并发场景） |
| KV-restore 惩罚 | +131% |
| Model-load 惩罚 | 34× 变慢 |
| B300 NVSwitch 多 GPU 租户 | 510 GB/s NVLink P2P 已验证 |

**结论**: Blackwell 改变了机密租户单元。B300 支持 CVM 内 510 GB/s NVLink P2P 和并发隔离租户，但仍有 fabric-attestation 差距待解决。

**相关性**: ★★★★★ — GPU Direct / 机密计算 / LLM 推理 OS 层优化

## 3️⃣ Aquifer: CXL+RDMA 层次化内存池 (arXiv:2606.24079) 🆕

**链接**: https://arxiv.org/abs/2606.24079

**核心问题**: 云集群中 **25-35%** 的已安装 DRAM 因内存"搁浅"被浪费。CXL 提供低延迟 load/store 透明访问，但限于 pod 内；RDMA 提供集群范围覆盖但延迟更高。两层级结合是实际路径，但在 MicroVM serverless 中尚未探索。

**Aquifer 方案**:

| 技术 | 细节 |
|:-----|:------|
| **热度感知快照格式** | 消除零页，热页放 CXL 池，冷页放 RDMA 池 |
| **Ownership-based Coherence Protocol** | CXL 2.0 multi-headed 设备缺乏硬件缓存一致性 |
| **Copy-based Page Serving** | CXL 热页预安装 + RDMA 冷页异步按需加载 |

**核心数据**:
- **2.2×** 端到端几何平均加速 over Firecracker
- **1.1×** over 次优替代方案

**作者**: Junliang Hu, Huaicheng Li, Ming-Chang Yang (emory)

**相关性**: ★★★★ — RDMA 内存池化、CXL+RDMA 层次化架构

## 4️⃣ RDMA 哈希表设计：挑战与机遇 (arXiv:2606.24073) 🆕

**链接**: https://arxiv.org/abs/2606.24073

**内容**: 系统性综述 RDMA 远程内存哈希表设计的三大挑战：

| 挑战 | 描述 |
|:-----|:------|
| **网络访问次数** | 单次逻辑哈希表访问可能转化为多次远程网络访问，冲突处理和探测进一步增加 RDMA 请求数 |
| **并发控制** | 远程 CPU 被 bypass，依赖远程线程的传统并发控制不再适用 |
| **NIC 硬件约束** | RDMA NIC 资源限制（队列、缓存、内存注册、原子操作）对哈希表结构施加新约束 |

**相关性**: ★★★ — RDMA 系统设计、远程内存数据结构的核心参考

## 5️⃣ DigenRL: 扩散生成式 LLM 解聚 RL 框架 (arXiv:2606.24369) 🆕

**链接**: https://arxiv.org/abs/2606.24369

**核心**: 首个专为扩散生成式 LLM（HunyuanVideo-13B, Wan2.1-14B, FLUX.1-12B, QwenImage-20B）设计的解聚 RL 框架。

**关键创新**:
- **Generation-Axis Pipeline (GAP)** — 扩散架构中的生成轴流水线，支持 rollout 和 training 之间更细粒度的流水线
- **Time-Step Parallelism (TSP)** — 扩散时间步并行
- **Elastic Trainer-Assisted Generation (TAG)** — 训练 GPU 弹性辅助 rollout 生成
- **Tightly One-Step Constrained Async Strategy** — 利用流水线尾部气泡

**核心数据**: **1.56-2.10×** 吞吐提升 over veRL-Omni 和 GenRL（16-32 GPU 实验）

**相关性**: ★★★★ — 分布式训练系统 OS 层优化、解聚架构、GPU 资源弹性调度

## 6️⃣ Kops: eBPF 编译管道的安全扩展 (arXiv:2606.24213, cs.OS) 🆕

**链接**: https://arxiv.org/abs/2606.24213

**内容**: 允许用户空间编译器和内核模块引入新的 eBPF 操作，而不修改内核核心。每个操作有 proof sequence（现有 verifier 检查）和 native emit（JIT 编译）。

**核心数据**:
- **7 条 EInsn 操作**（rotate、conditional select 等 CPU 单指令）
- **最高 24%** eBPF 微基准加速（x86-64 和 ARM64）
- **最高 2.358×** 全程序原生替换加速
- **Lean 4 形式化证明**：每个 native emit 计算结果与 proof sequence 相同

**相关性**: ★★★ — OS 内核优化、eBPF 虚拟化基础设施、可扩展编译管道
