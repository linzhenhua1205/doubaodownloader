## 1️⃣ 🎯 Moebius: MoE Serving 运行时 TP↔EP 无缝切换 (arXiv:2606.26607, Jun 25) 🔥[本周最强]

**链接**: https://arxiv.org/abs/2606.26607
**标签**: `cs.DC` · `MoE Serving` · `Runtime Parallelism` · `Distributed Inference`

### 核心洞察

| 维度 | 内容 |
|:-----|:------|
| **问题** | MoE 模型服务时 TP（Tensor Parallelism）和 EP（Expert Parallelism）各有利弊：低并发 TP 快，高并发 EP 快。产线负载在两者间持续切换（在线服务流量突发 + RL rollout 高→低并发衰减），固定任一并行度都会在另一侧损失性能 |
| **核心洞察** | **TP 和 EP 不是一个模型的两个副本，而是同一个模型的两套布局**——两者计算相同函数，共享字节相同的 expert weights 和 KV cache。切换只改变"谁拥有哪个切片" |
| **方案** | 运行时熔合 GPU-to-GPU 转移核，在 decode step 之间（无需排空）完成 owner-changed 切片的重新分片。两个并行度保持常驻内存 |

### 关键指标

| 指标 | 数值 |
|:-----|:-----|
| **切换时间** | 215-434 ms（8× H200，Qwen3-235B-A22B） |
| **内存开销** | 仅 2.4%（两套布局常驻） |
| **RL rollout 加速** | 1.16-1.25×（vs 最优静态并行度） |
| **无掉线** | switch 在 decode step 间完成，不需排空 in-flight 请求 |

### 相关性分析

| 维度 | 评价 |
|:-----|:------|
| **分布式推理 OS 层优化** | ★★★★★ — 运行时重新分片涉及 GPU 间权值/KV 的 RDMA 或 NVLink 传输，切换时机在 decode step 间体现了 OS 调度思维 |
| **GPU 集合通信** | ★★★★ — fused GPU-to-GPU transfer kernels 实现高性能重分片 |
| **拓扑感知** | ★★★ — 切换代价取决于 NVLink 域内带宽，即拓扑相关性 |

## 2️⃣ 🔬 Xsim: 异构 AI 训练仿真器 (arXiv:2606.26633, Jun 25) 💻

**链接**: https://arxiv.org/abs/2606.26633
**标签**: `cs.DC` · `Heterogeneous Training` · `Simulator` · `Topology-Aware`

### 背景：为何需要异构仿真？

| 异构来源 | 驱动因素 |
|:---------|:---------|
| 多模态/MoE 模型利用异构提升利用率 | 架构本身有异构需求 |
| 公有云同构硬件可用性受限 | 硬件快速迭代导致供应短缺 |
| 跨地域分布式部署 | 基础设施多样且异构 |

### Xsim 关键特性

| 特性 | 技术方案 | 意义 |
|:-----|:---------|:-----|
| **负载均衡** | 跨异构设备组的非均匀工作负载分区 | 让每个 GPU 按能力分配工作 |
| **异构集合通信** | 自定义 ring 构建 + chunk 分区 | 打破传统 NCCL 同构 ring 假设 |
| **流水并行** | 可复用的异构感知抽象 | 支持非均匀 tensor resharding |
| **仿真集成** | NS-3 和 htsim 可插拔集成 | 精度/性能可权衡 |
| **部署描述** | 灵活输入：自定义设备组 + 设备到并行度映射 | 覆盖各种真实部署 |

### 仿真精度

| 配置 | 误差 |
|:-----|:-----|
| 异构 DP/TP | < 5% |
| PP 通信建模 | ~2% |

### 相关性分析

| 维度 | 评价 |
|:-----|:------|
| **集合通信** | ★★★★★ — 自定义 ring 构建 + chunk 分区直接影响 NCCL ring 算法，拓扑感知 |
| **拓扑感知调度** | ★★★★★ — 直接涉及异构设备的并行策略和通信拓扑构建 |
| **分布式推理训练 OS** | ★★★★ — 仿真器层面需要支撑的 NS-3 集成涉及网络建模 |

## 3️⃣ ⚡ NEURON-Fabric: 受控低比特梯度通信 (arXiv:2606.25759, Jun 24) 📡

**链接**: https://arxiv.org/abs/2606.25759
**标签**: `cs.DC` · `Gradient Compression` · `Distributed Training` · `Communication Optimization`

### 核心问题

低比特梯度聚合可以减少通信开销，但**静态替换**全精度通信会破坏训练稳定性——安全精度取决于训练阶段、模型结构、运行时 bucketization 和通信底层。

### 方案设计

| 组件 | 作用 |
|:-----|:------|
| **Profile-Guided Runtime** | 根据离线校准的 profile 在线决策 |
| **模型感知运行时绑定** | 不同模型区域（不同层/不同参数类型）采用不同策略 |
| **在线训练健康监控** | 实时监控训练指标，检测精度退化信号 |
| **Reducer 容量检查** | 检查 reducer 是否有能力处理低比特压缩 |
| **动态 FP32 回退** | 检测到不安全时自动回退到全精度 |

### 关键结论

| 发现 | 意义 |
|:-----|:------|
| 静态低比特会崩训练精度 | 精度需动态调节，不能一刀切 |
| Profile-guided 控制能恢复精度 | 接近全精度 + 显著降低通信量 |
| Reducer admission 是共设计问题 | 不是孤立压缩算子，需架构-运行时协同 |

### 相关性分析

| 维度 | 评价 |
|:-----|:------|
| **集合通信** | ★★★★ — 梯度通信是训练中最大的通信负载 |
| **分布式训练 OS** | ★★★★ — Profile-guided runtime + 动态 FP32 fallback 体现了 OS 层调度思维 |
| **拓扑感知** | ★★ — 与 reducer 容量相关，但主要关注通信精度而非拓扑 |

## 4️⃣ 📋 UCX v1.21.0 正式版回顾 & NCCL 状态 (续上周)

### UCX v1.21.0 (2026-06-24)

> 详情见上周 [2026-06-25 tracking](../bmc-system/2026-06-25.md)，本周补充关键修复深度分析：

| 修复 | 产线影响 |
|:-----|:---------|
| **GDAKI NIC/GPU 映射（CUDA_VISIBLE_DEVICES）** | 🔴 容器环境：K8s 动态挂载 GPU 时，CUDA_VISIBLE_DEVICES 隐藏了物理 GPU ID，导致 GDAKI 查询 NIC→GPU 亲和性时映射到错误的（被隐藏的）GPU，从而选择非最优 HCA。此修复让 GDAKI 在 mapping 时使用**逻辑 GPU 索引**而非物理索引 |
| **GDA WQE CAS 排序** | 🟡 GDA transport 中 WQE 的顺序保证依赖于 CAS（Compare-And-Swap）操作的读写序，此前缺少 CAS 后的内存屏障导致重排序，此次修复增加了基于 CAS 的 readiness marking 确保正确序 |
| **UCX_MAX_HCA_PER_GPU** | 🟡 新策略控制每个 GPU 内存注册可以到达的最近 HCA 数量。在高密度多 HCA 节点（如 8 GPU × 8 NIC）中限制注册传播拓扑，减少不必要的 HCA 资源占用 |
