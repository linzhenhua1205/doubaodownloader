## 1️⃣ NCCL EP v0.1.0 正式版发布 (2026-06-08) 🏆

**来源**: [GitHub NVIDIA/nccl Releases - nccl-ep-v0.1.0](https://github.com/NVIDIA/nccl/releases/tag/nccl-ep-v0.1.0) | 作者: bhramesh-nvidia

### 背景

NCCL EP (Expert Parallelism) 是 NCCL 团队为 MoE 架构量身打造的通信扩展——与近期 arXiv:2603.13606 (NCCL EP 论文) 对应，但这是**正式的软件发布版本**。这是 NVIDIA 首次将 MoE 专用通信原语原生集成到 NCCL 生态体系中。

### 核心 API

```c
// 统一 MoE 通信原语
ncclEpDispatch(/* token -> expert 路由 */) 
ncclEpCombine(/* expert -> token 汇聚 */)

// 双模式
LL (Low-Latency)  → 推理 decode (1-128 tokens), 全量 all-to-all RDMA+NVLink
HT (High-Throughput) → 训练 prefill (4096+ tokens), NVLink 域内聚合后跨节点 RDMA
```

### v0.1.0 版本关键特性

| 方向 | 特性 | 细节 |
|:-----|:-----|:------|
| **编译** | **JIT Compilation for HT mode** | HT 模式全面迁移到 JIT 编译，解决性能与限制问题；LL 模式下一版本迁移 |
| **NVLink** | **Full Multi-Node NVLink 支持** | 完整 MNNVL 支持 + 单边 buffer staging 消除 |
| **CUDA Graph** | **全面支持** | EP Handle 管理 API 重设计以兼容 CUDA Graph capture/replay |
| **容错** | **Active rank mask** | 识别故障 rank 并排除，操作继续而非 abort |
| **零拷贝** | HT mode zero-copy flows | NCCL Window 关联实现零拷贝 |
| **API** | **重构** | 拆分 ncclEpInitHandle (控制路径) / ncclEpUpdateHandle (数据路径)，支持静态栈分配 |
| **绑定** | **nccl4py 0.3** | Pythonic API，但当前仅 CUDA 13 wheel（CUDA 12 需源码编译） |
| **拓扑** | **ncclTeamRail HT mode** | 取代 split communicator，优化跨节点 RDMA 通道利用率 |
| **内存** | **Staging buffer 缩减** | LL 模式 per-rank token dedup + rank-major layout，缩减量 = experts_per_rank 倍 |
| **前向/后向** | **显式 Forward/Backward 选择** | Dispatch/Combine 操作显式区分正反向传播 |
| **SM 控制** | 允许用户设置 EP 使用的 SM 数量 | 可调资源分配 |

### 性能评测 (ep_bench)

- 通过 **CUPTI** 报告 kernel-only 性能指标（可用时）
- 支持多变量：SM 数量、expert 数量、layout 选择
- 支持 Dispatch/Combine 两阶段的**内存损坏与路由正确性验证**

### 已知限制

| 限制 | 说明 |
|:-----|:------|
| HT mode 最大 RDMA domains | **32**（由于算法限制） |
| 共享内存资源溢出 | 隐藏维度 + LSA team 大小影响；需在构建时调整宏值 |
| LL mode 最大 top-K | **9** |
| LL mode 支持的隐藏维度 | 2048, 2560, 4096, 5120, 6144, 7168, 8192 |
| Quantization | v0.1 **不**支持量化（下一版本计划恢复） |
| nccl4py wheel | 仅 CUDA 13，CUDA 12 需源码编译 |

### 意义

> NCCL EP 标志着 NVIDIA 正式将 **MoE 通信原语** 从碎片化的第三方库（DeepEP、Hybrid-EP 等）纳入官方 NCCL 体系。这带来了：1️⃣ 官方的长期支持路径 2️⃣ 与 NCCL 版本同步更新 3️⃣ 统一的拓扑感知和 Device API 集成。对于在生产环境中部署 MoE 模型的团队，这是**决定性的事件**。

## 2️⃣ NCCL v2.30.7-1 稳定版发布 (2026-06-04)

**来源**: [GitHub NVIDIA/nccl Releases - v2.30.7-1](https://github.com/NVIDIA/nccl/releases/tag/v2.30.7-1) | 作者: bhramesh-nvidia

### Zero-SM Collectives ⭐ **重大新特性**

首次引入 **零 SM 集合通信**——不占用 GPU SM 的集合通信操作。利用 CPU proxy 做 RMA 跨节点通信 + Copy Engine (CE) 做节点内通信。

| 操作 | 实现方式 |
|:-----|:---------|
| **AllGather** (zero-SM) | CE 做 NVLink intra-node + RMA CPU proxy 做 inter-node |
| **All2all** (zero-SM) | 同上实现层次化 |
| **启用** | `NCCL_CTA_POLICY_ZERO` 标志 |

> **意义**: 零 SM collectives 让集合通信完全**不占用 GPU 计算资源**，与计算 kernel 可以真正并行。这对于训练中计算-通信重叠有根本性改善——通信不再消耗 GPU SM，计算 kernel 独占全部 SM 资源。

### GIN (GPU-Initiated Networking) 增强

| 特性 | 细节 |
|:-----|:------|
| **GPI 后端** (Experimental) | 新增 GPU Push Interface 作为 GIN 新后端 |
| **显式信号语义** | Strong/Weak 信号区分 |
| **ncclGinFenceLevel** | barrier 语义正式定义 |
| **流量控制** | `NCCL_GIN_IB_TC` 控制 GIN 流量等级 |
| **资源共享线程** | `NCCL_GIN_RESOURCE_SHARING_THREAD` 优化 |
| **QP 开销** | GDAKI 模式下 counter 未使用时优化 QP 创建 |
| **NIC fusion** | 确保 GIN 在 NIC fusion 启用时可用 |
| **插件示例** | `plugins/gin/example` 添加 GIN 插件示例 |

### MPS + MLOPart (Experimental)

- 利用 CUDA **Memory Locality Optimized Partition (MLOPart)** 特性
- 支持每物理 GPU **最多 2 个 rank**
- 关键场景：显存密集型推理、小 batch 多 worker 场景

### Symmetric Memory 持续增强

| 改进 | 细节 |
|:-----|:------|
| RMA 插件架构重构 | 更清晰的插件接口 |
| 非对称 buffer 大小 | Window 注册支持不等大小 |
| ReduceScatter 对称 kernel 优化 | 性能提升 |
| RMA CE 操作优化 | 批量化 CE 操作 + put/wait 路径改善 |
| CUDA Graph capture 中 Window 注册 | 图捕获期间的注册支持 |

### 其他关键更新

| 更新 | 影响 |
|:-----|:------|
| **支持 GRH (Global Route Header)** | 支持需全局路由头的 IB 端口 |
| **权重传输 API** (`contrib/nccl_xfer`) | 优化权重传输路径 |
| **自定义 kernel** (`contrib/custom_algos`) | Alltoall 和 AllReduce 的 Device API 示例 |
| **RMSNorm 示例** | 计算与通信融合的 Device API 演示 |
| **RoCE LAG 轮询负载均衡** | 修复 `#2150`——多端口 RoCE 绑定下 QP 负载分配 |
| **PXN 死锁修复** | cuda stream allocation 在 PXN 下 memset buffer 时的死锁 |

### Bug 修复精选

| Issue | 症状 | 修复 |
|:------|:------|:------|
| `#2150` | RoCE LAG 下 QP 不均衡 | 轮询队列亲和策略 |
| `#1834` | proxy 线程 hang | 外部 TCP 请求导致 ncclProxyService 挂起 |
| `#2179` | GIN init 上下文泄漏 | 修复初始化时 context 泄漏 |
| `#2187` | RMA proxy MR 注册类型错误 | host-NUMA cpuAccessSignals 内存类型修正 |
| `#2119` | 高消息速率下请求丢弃 | 单侧 host API 修复 |

### 已知问题

- `ncclPutSignal` 等单侧 host RMA API 需要每个 rank 调用一次作为初始化热身
- 同一 backing memory 分配中刻出多个对称窗口时存在潜在的 corruption（已在 dev 分支修复）

## 3️⃣ ASTRA-sim 3.0: 分布式 ML 基础设施高保真模拟框架 (arXiv:2606.10440, June 9)

**来源**: arXiv:2606.10440 | 作者: William Won 等 19 人 (AMD + Georgia Tech)  
**链接**: [arxiv.org/abs/2606.10440](https://arxiv.org/abs/2606.10440)

### 核心创新

| 维度 | ASTRA-sim 2.x | ASTRA-sim 3.0 |
|:-----|:--------------|:---------------|
| **模拟粒度** | 粗粒度 | **Cache-line-sized load-store** |
| **GPU 模型** | 简化 | **详细 GPU 执行模型** |
| **网络表示** | 手动配置 | **InfraGraph 标准化表示** |
| **设计空间** | 有限 | 最优集合算法 + 网络需求 + GPU 架构 |

### InfraGraph 标准化

> "InfraGraph, a standardized representation to capture distributed ML network infrastructure in detail."

为分布式 ML 网络基础设施提供**标准化图表示**，涵盖：
- 拓扑结构（节点、链路、带宽）
- 路由策略
- 交换机/网卡配置

### 模拟场景

作者展示了利用 ASTRA-sim 3.0 进行的三类设计空间探索：
1. **最优集合通信算法** — 在给定的拓扑和 GPU 模型下搜索最优算法
2. **网络需求分析** — 确定训练特定模型所需的网络带宽阈值
3. **GPU 架构协同设计** — GPU 执行模型与集合算法联合优化

### 意义

> 与 NCCL 的进展形成**模拟-执行闭环**：NCCL 提供底层通信原语，ASTRA-sim 3.0 提供高层设计空间探索。对于超大规模集群设计者和集合算法研究者，这是**关键的模拟基础设施**。

## 4️⃣ 504-GPU B200 集群生产故障运维分析 (arXiv:2605.09370 v4, June 9 更新)

**来源**: arXiv:2605.09370 v4 | 作者: Daemyung Kang 等 14 人 (SKT, Upstage, Lablup, NVIDIA Korea, VAST Data)  
**链接**: [arxiv.org/abs/2605.09370](https://arxiv.org/abs/2605.09370)

> 5 月 10 日提交，6 月 9 日 v4 更新，42 页技术报告

### 环境

| 参数 | 值 |
|:-----|:---|
| GPU | 63 nodes × 8 = **504 NVIDIA B200** |
| 监控 | 55 天 Prometheus (751 指标) + 73 天操作日志 |
| 训练会话 | **224** 个多节点会话 |
| 组织 | 5 方协作 (SKT/Upstage/Lablup/NVIDIA Korea/VAST Data) |

### 四项关键发现

**① 无单一主导指标**

> "Over 751 Prometheus metrics and 10 XID-identified GPU failures, no single metric is consistently dominant across failure types."

- 多信号检测是必要的——**没有万能指标**
- 印证了 CCL-D (06-09) 的多层分布式追踪方案的必要性

**② 检查点 I/O 瓶颈**

| 操作 | 带宽 | 瓶颈 |
|:-----|:-----|:------|
| 重启加载 | 21.5% of 最大读带宽 (700 GB/s) | NFS/RPC 排队 |
| 保存突发 | 16.0% of 最大写带宽 (250 GB/s) | Transport-layer backlog |
| **60 节点级 I/O 瓶颈** | 2-4 节点测试中不可见 | 仅生产规模可复现 |

> 生产规模的 I/O 瓶颈在 2-4 节点测试中完全不可见——跨组织协作是唯一可隔离根因的方式。

**③ 节点故障高度集中**

> "Top 3 of 63 nodes account for over 50% of node exclusions."

- 故障分布高度偏斜——少数节点承担了大部分问题
- 提示：更换比修复性价比更高

**④ 自动重试 vs 手动重试**

| 方式 | 成功率 | 中位间隔 |
|:-----|:-------|:---------|
| **自动重试** | **33.3%** (12 chains, 73 attempts) | **11 min** (IQR 10-11) |
| 手动重试 | 12.5% | N/A |
| 自动 vs 手动 | **2.7× 更优** | — |

### 与分布式 OS 的关联

这份报告虽然不是新的集合通信算法，但它提供了**大规模 GPU 集群故障模式的真实数据**，对分布式 OS 设计直接影响：
- **检查点策略**: 发现 NFS 检查点在 60 节点规模会饱和，验证了分布式检查点 (如 TierCheck, ReCoVer) 的必要性
- **故障分布**: top-3 节点占 50% 的发现支持 **选择性容错**（针对少数高故障节点做差异化保护）
- **自动重试**: 33.3% vs 12.5% 的对比验证了自动故障恢复的巨大价值
