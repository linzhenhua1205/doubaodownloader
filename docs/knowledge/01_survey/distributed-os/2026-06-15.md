## 1️⃣ Tiara: 可编程Line-Rate ISA解决RDMA间接性墙 (arXiv:2606.13708, Jun 10) 🏆🔥

**来源**: [arXiv:2606.13708](https://arxiv.org/abs/2606.13708) — Bojie Li
**提交**: 2026-06-10（5天前）| 标签: cs.AR, cs.NI

### 核心问题：RDMA间接性墙 (Indirection Wall)

RDMA one-sided verbs 是内存解耦的自然原语，但要求客户端提供**精确的远程地址**。当目标地址依赖于必须先从远程内存读取的数据时，1-RTT 性能完全失效——这就是 **Indirection Wall**。

| 场景 | 间接层级数 | 传统RDMA代价 |
|:------|:-----------|:--------------|
| 图遍历指针追踪 | 10+ 跳 | 10× RTT |
| 多级页表遍历 | 4-5 级 | 4-5× RTT |
| 分布式锁条件逻辑 | 2-3 级 | 2-3× RTT |
| 解耦LLM推理 PagedAttention | 2+ 级 (block table → KV) | 2+× RTT |
| MoE expert-gather 路由表 | 1-2 级 | 1-2× RTT |

### Tiara 方案

Tiara 是一个紧凑的、**静态可验证**的指令集，运行在**内存端 NIC** 上：

> Tiara operators 是预注册的程序，类似于内核中的 eBPF 程序，**本地解析间接性**，将多RTT依赖链折叠为**单次往返**。

| 特性 | 详情 |
|:-----|:------|
| 架构定位 | NIC 端（内存侧）可编程加速器 |
| 设计类比 | "eBPF for RDMA NICs" |
| 验证性 | **静态可验证** — 类似 eBPF verifier |
| 实现 | FPGA 原型 |
| 操作语义 | 预注册程序 + 本地间接解析 |

### 性能数据 (FPGA 原型)

| 场景 | 指标 | 性能提升 |
|:-----|:------|:---------|
| **10-hop 图遍历** | 延迟 | **2.85× 降低** |
| | 吞吐 | **3.4× 提升** |
| **页表遍历** | 延迟 | **62% 降低** |
| **分布式锁 (无争用)** | 延迟 | **2.9× 降低** |
| **解耦 PagedAttention** (8 KB blocks) | 吞吐 | **2.8×** |
| **MoE expert-gather** (32 experts) | 延迟 | **1.88×** |

### 🎯 深度分析

**Tiara 的意义远不止于 RDMA 性能提升**——它本质上是对 **NIC 计算范式**的一次重新定义：

1. **与现有方案的关系**: 现有的 SmartNIC/DPU 方案（BlueField, IPU）虽然也提供可编程能力，但通常需要消耗 CPU 周期或依赖复杂的 Arm 核心。Tiara 的 ISA 是 **cache-line 级别轻量原语**，更像 eBPF 而非通用 CPU。

2. **与先前的 GDA 工作对比**: 
   - GDA (GPUDirect Async) / GDAKI 关注 GPU 端直接触发 NIC 操作（GPU-initiated RDMA）
   - Tiara 关注 **NIC 侧智能**——程序在 NIC 上运行，本地解析间接性
   - 两者互补：GPU 端发起 + NIC 端智能解析

3. **对分布式系统架构的颠覆性**:
   - **内存解耦**：Tiara 解决了内存解耦最关键的性能障碍——间接访问
   - **MoE 推理**：1.88× expert-gather 加速直接利好所有 MoE 架构 LLM
   - **分布式 KV Cache**：2.8× PagedAttention 加速对 PD 分离推理中的 KV 传输有直接影响

4. **技术成熟度**: FPGA 原型验证，离 ASIC 量产还有距离，但概念验证充分

## 2️⃣ ScaleAcross: EVPN-VXLAN 跨数据中心 AI 训练基础设施 (arXiv:2606.12963, Jun 11)

**来源**: [arXiv:2606.12963](https://arxiv.org/abs/2606.12963) — Naved Inam, Aryan Bhavsar, Masabattula Nikhil, Sidharth Sharma
**提交**: 2026-06-11（4天前）| 标签: cs.NI, cs.DC, cs.ET

> ⚠️ 与 Meta 的 **ScaleAcross Explorer** (arXiv:2605.24326, Li et al., 64.62%↑, 五月已跟踪) 是**完全不同的两组工作**。Meta 版关注并行策略调度优化，本项关注**网络基础设施层**（EVPN/VXLAN/ECMP/BFD）。

### 背景

AI 模型的快速增长和数据主权要求推动跨数据中心地理分布式训练。这引入了以下系统级挑战：
- **同步密集型通信** — AllReduce 对延迟高度敏感
- **跨站数据交换** — 需要可靠的 WAN 连接
- **广域延迟约束** — 跨 DC 延迟比单 DC 高 10-100×

### 框架设计

基于 **ContainerLab + FRRouting (FRR)** 构建的可扩展仿真框架：

| 组件 | 用途 |
|:------|:------|
| **VXLAN 覆盖网络** | 跨 DC L2 扩展 |
| **EVPN 控制平面** | MAC/IP 路由分发 |
| **ECMP 多路径** | 负载均衡 |
| **BFD 双向检测** | 快速故障切换 |
| **Queue-pair-aware 流量分发** 🆕 | 改进同步密集型工作负载通信 |

### 评估与方法

- 使用真实 WAN 仿真环境
- 评估 AllReduce 和 Parameter Server 两种通信模式
- 关注：流量分布、弹性、基础设施行为

### 🎯 分析意义

| 维度 | 评价 |
|:-----|:------|
| **独特价值** | 首次系统研究 EVPN-VXLAN 在跨 DC AI 训练中的适用性 |
| **与 Meta ScaleAcross 互补** | Meta 关注上层调度，本项关注下层网络基础设施 |
| **局限** | 仿真环境验证（非生产），且聚焦标准以太网而非 InfiniBand/RoCE |
| **配套价值** | 为"以太网替代 InfiniBand 做 AI 网络"提供了跨 DC 视角 |

### 核心判断

Tiara 是本期**真正的重磅发现**。它提出了一个全新的视角：与其让 GPU 端做更复杂的通信卸载（GDAKI、GPUDirect Async），不如让 **NIC 端具备智能间接解析能力**。这个思路与 eBPF 在 Linux 内核中的成功革命如出一辙——"把计算放到数据经过的地方"。

Tiara 对分布式系统的影响可能是深远的：
- **LLM 推理系统** → 解耦 PD 分离中 KV Cache 远程访问性能瓶颈部分消除
- **MoE 系统** → expert 路由 + gather 的单 RTT 化
- **内存解耦系统** → 间接指针访问不再需要多次 RTT
