## 1️⃣ 🏆 ASAP: 解聚异步 MoE Prefill 推理系统 (arXiv:2606.22541, Jun 22) 🆕

**链接**: https://arxiv.org/abs/2606.22541

**核心问题**: MoE 服务系统采用 DP (Attention) + EP (MoE) 混合并行，需要在 attention DP groups 与 experts 之间频繁全局同步。在线服务中请求到达率和序列长度的大幅波动导致 DP 不平衡，引发严重同步停顿，劣化 TTFT 和系统吞吐。

**ASAP 方案**: 专为 MoE 模型 prefill 阶段设计的异步推理系统。

| 技术 | 机制 |
|:-----|:------|
| **解耦 Attention 与 MoE stages** | 完全分离两个阶段，消除全局同步屏障 |
| **异步通信原语套件** | 支持完全异步执行流水线 |
| **四种协同优化** | 覆盖请求调度和模型执行两个层面的协调优化 |

**验证平台**: CloudMatrix384 超级节点

**性能**: SLO 合规预填吞吐提升 **90%** vs 最先进同步服务方案

**🎯 意义**:
- CloudMatrix384 是华为的超级节点架构，ASAP 在其上验证——**MoE 异步解聚已进入国产超节点生态**
- DP 不平衡是 MoE 服务的根本性难题，ASAP 的**彻底异步化思路**（而非优化同步开销）是更激进的解法
- 与上周的 UltraEP (软件均衡) 形成互补：UltraEP 解决 EP 内专家均衡，ASAP 解决 DP 同步瓶颈

## 2️⃣ RDMACell: Token 驱动的 Flowcell 级 RDMA 负载均衡 (arXiv:2606.20581, Jun 19) 🆕

**链接**: https://arxiv.org/abs/2606.20581

**核心问题**: RDMA 默认的 ECMP (Equal-Cost Multi-Path) 负载均衡严重受哈希冲突和大象流影响。Flowlet 方案缺少触发切换的时间间隙，packet 级方案遭乱序重排，可编程交换机方案难以商用部署。

**RDMACell 方案**: **纯主机侧** flowcell 级负载均衡系统。

| 技术 | 机制 |
|:-----|:------|
| **Flowcell 切割** | 主动将流切分为多个 flowcells，跨多路径分发 |
| **Token 接收端控制** | 基于 token 的接收端控制，实现**微秒级路径切换** |
| **原子双 WQE** | 通过 atomic dual Work Queue Elements 实时反馈 |
| **零硬件修改** | 不修改 NIC 固件或交换机硬件 |

**性能 (fat-tree, 80% 负载, all-to-all 流量)**:

| 对比 | P99 FCT 改善 |
|:-----|:------------|
| vs ECMP | **44% 降低** |
| vs LetFlow | **42.2% 降低** |
| vs 可编程交换机方案 | **可比的尾延迟性能** |

**🎯 意义**:
- **纯主机侧方案达到可编程交换机级别的性能**——这是 RDMA 负载均衡领域的突破性成果
- All-to-all 流量下 44% P99 FCT 降低对 AI 训练（大量 all-to-all 通信）直接利好
- 微秒级路径切换 + 零硬件修改 = **大规模部署可行性强**
- 与 CSA-UD (同组另一篇) 形成 RDMA 通信优化的姐妹篇

## 3️⃣ CSA-UD: 通信语义感知的 RDMA 丢包恢复 (arXiv:2606.20582, Jun 19) 🆕

**链接**: https://arxiv.org/abs/2606.20582

**核心问题**: 万亿参数模型训练的集合通信产生密集 RDMA 流量。当前 RC (Reliable Connection) 模式每进程对需专用 QP，RNIC 只能缓存数千 QP，超限触发 PCIe 往返惩罚。同时全局同步使训练对尾延迟极度敏感。

**CSA-UD 方案**: 结合可扩展性和可靠性的**通信语义感知 Unreliable Datagram**。

| 技术 | 机制 |
|:-----|:------|
| **数据/恢复解耦** | 数据传输与丢包恢复分离 |
| **自适应丢包检测** | 动态调整检测间隔，加速尾恢复 |
| **语义感知** | 利用分布式训练的同步语义优化恢复策略 |
| **多路径 + Bitmap 重组** | 支持多路径传输，无需无损网络 |

**性能**:

| 指标 | 改善 |
|:-----|:-----|
| **大规模集合通信尾延迟** | **显著降低** |
| **vs RC 可扩展性** | **更优** (无 QP 缓存瓶颈) |
| **高负载 P99 FCT** | **比现有方案低 30%+** |

**🎯 意义**:
- **UD 替代 RC** 在 AI 训练领域一直是"理想但不敢用"的方向——CSA-UD 首次系统性地使 UD 兼具可靠性和语义感知
- QP 可扩展性问题是万卡集群 RDMA 通信的根本瓶颈，**CSA-UD 从根本上消除了这个瓶颈**
- 与 RDMACell 同组（清华团队）联手解决 RDMA 两大难题：负载均衡 + QP 扩展性

## 4️⃣ SAGA: 工作流原子调度 — AI Agent 推理调度新范式 (arXiv:2605.00528, Jun 23 更新) 🔄

**链接**: https://arxiv.org/abs/2605.00528

**核心问题**: AI Agent 每个任务执行数十到数百次链式 LLM 调用，但 GPU 调度器将每次调用视为独立请求，丢弃步骤间大量中间状态，端到端延迟放大 **3-8×**。

**SAGA 方案**: 将**完整 Agent 工作流**（而非单个推理调用）作为**调度基本单元**的分布式调度器。

| 技术 | 机制 |
|:-----|:------|
| **Agent Execution Graphs** | 捕获工作流结构，预测工具调用边界的 KV Cache 复用 |
| **Session-Affinity Batching + Work Stealing** | 关联请求共置 + 全局负载均衡 |
| **Agent Fair Share** | 任务完成时间公平性度量，带**有界偏差保证** |

**性能 (64-GPU 集群, SWE-bench / WebArena 任务)**:

| 指标 | vs vLLM v0.15.1 + prefix caching + affinity routing |
|:-----|:------------------------------------------------------|
| **Task Completion Time** | **1.64× 降低** (几何均值, p < 0.001) |
| **GPU 内存利用率** | **1.22× 提升** |
| **SLO 达成率** | **99.2%** (多租户干扰下) |
| **峰值吞吐** | ~30% 低于吞吐最优批调度 (可接受的权衡) |

**🎯 意义**:
- **Compound AI 推理"程序级调度"范式变革** —— 这是从 request-level 到 program-level 的深刻转变
- Agent Execution Graphs 的 KV cache 预测达到 Belády 最优离线策略的 1.31× 以内——**缓存命中率极优**
- 与上周的 Execution-State Capsules 理念互补：Capsules 解决执行状态化，SAGA 解决跨步状态复用调度

## 5️⃣ SwarmX: 神经预测驱动的 Agent 推理调度 (arXiv:2606.21401, Jun 19) 🆕

**链接**: https://arxiv.org/abs/2606.21401

**核心问题**: Agentic AI 应用组合多次模型调用和工具执行。推理时间和模型调用结构通常依赖于提示语义，传统调度方法无法满足低延迟服务需求。

**SwarmX 方案**: 基于调度专用**神经预测器**的低延迟 Agent 推理调度系统。

| 技术 | 机制 |
|:-----|:------|
| **Neural Predictors** | 捕捉 prompt、设备、运行时、目标模型特征 |
| **分布预测暴露** | 向路由器和伸缩器暴露分布预测，支持尾感知决策 |
| **在线自适应** | 预测器训练和在线适应机制 |
| **Scheduler-Agent 框架** | 通用基板，集成现有调度和模型服务基础设施 |

**验证**: 生产部署 (~1000 GPUs + 1M CPU Cores) + 128-GPU 受控实验

| 场景 | 改善 |
|:-----|:------|
| **多 Agent 代码生成尾延迟** | **最多降低 61.5%** vs 最先进调度器 |
| **深度研究 Workloads** | 同等 SLO 下**吞吐提升 2×** vs 生产调度器 |
| **多模态 Agent Workflows** | 持续改善 |

**🎯 意义**:
- **Agent 推理延迟的不确定性由 prompt 语义决定**——这个洞察改变了调度器设计思路
- 生产级 1000 GPU + 1M CPU cores 验证——**不是实验室玩具**
- 与 SAGA 互补：SAGA 做工作流级调度，SwarmX 做调用级神经预测调度

## 6️⃣ Concordia: JIT 编译持久内核检查点 — 容错 LLM 推理 (arXiv:2606.23521, Jun 22) 🆕

**链接**: https://arxiv.org/abs/2606.23521

**核心问题**: 长时间运行的 LLM Agent 在 GPU 上保有大量状态（KV cache、请求调度器、通信状态、在线适配器）。GPU 或通信器故障后丢失这些状态会浪费数分钟到数小时的工作，现有恢复机制要么重启整个服务栈，要么需要应用级检查点逻辑。

**Concordia 方案**: 基于**设备常驻持久内核**的容错运行时。

| 技术 | 机制 |
|:-----|:------|
| **PTX/SASS 级插桩** | 在框架代码和库边界以下插入检查点/pause 钩子 |
| **JIT 编译增量检查点** | 为每个注册的 LLM 状态区域生成专用 delta-checkpoint handler（KV-block scanner, 适配器 page scanner, recovery applier） |
| **Lock-Free Ring Buffer** | 持久内核消费计算/检查点/追加日志/恢复任务的 ring buffer |
| **CXL Memory / Host DRAM** | 检查点落盘到 CPU 可见的 CXL 内存或主机 DRAM |

**🎯 意义**:
- **GPU 故障后不重启服务栈**——持久内核+增量检查点让 GPU 级别的容错成为原生能力
- 与 Execution-State Capsules (上周) 相呼应：Capsules 做主动状态快照，Concordia 做故障恢复
- CXL 内存作为检查点目标——**CXL 在推理基础设施中的角色进一步扩展**

## 7️⃣ BatchGen: Sequence Coroutine 计算模型 — 批推理架构革命 (arXiv:2606.21712, Jun 19) 🆕

**链接**: https://arxiv.org/abs/2606.21712

**核心问题**: 现有推理引擎沿用交互式服务的执行模型。扩展到百万级序列时，运行时才出现的极端负载变化和大规模 GPU 集群的利用率维持成为根本性挑战。

**BatchGen 方案**: 全新的批推理架构基础——**Sequence Coroutine 计算模型**。

| 技术 | 机制 |
|:-----|:------|
| **序列协程模型** | 每个序列表示为细粒度、事件驱动的协程 |
| **动态工作重组** | 运行时暴露表达性原语，实现更大的 expert-level batch、缓减 straggler、设备间工作重分配 |
| **低成本 GPU 适配** | 在内存受限加速器上通过 offloading 保持利用率 |

**性能**:

| 场景 | 改善 |
|:-----|:------|
| **128-GPU 集群批完成时间** | **2.3× 降低** |
| **vs 强 offloading 基线 (内存受限加速器)** | **9.6×** |
| **开源** | 已计划 |

**🎯 意义**:
- **协程模型取代传统批处理**——这是批推理架构层面的范式改变
- 在低成本/内存受限 GPU 上保持高利用率——**经济上极具价值**
- 大的 expert-level batch 直接利好 MoE 推理
