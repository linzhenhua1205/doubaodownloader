## 1️⃣ 🏆 NIMBLE: 节点互联多路径运行时流量均衡 (arXiv:2604.00317, Mar 31)

**链接**: https://arxiv.org/abs/2604.00317

**核心问题**: 现代 GPU 集群虽有高聚合带宽（NVLink + 多 Rail NIC），但**流量倾斜 (Traffic Skew)** 导致少数链路饱和而其他链路闲置。NCCL/MPI+UCX 依赖静态最快路径路由或哈希多 Rail 条带化，在运行时流量偏离预期分布时浪费大量带宽。

**NIMBLE 方案**: 运行时通信编排系统，动态重分配流量以均衡所有可用路径的链路利用率。

| 技术组件 | 机制 |
|:---------|:------|
| **容量归一化最小拥塞优化** | 使用 **multiplicative-weights 算法** 高效求解 |
| **CUDA-aware GPU kernel RDMA 流水线** | 通过中间 GPU 和 rail-matched NIC 路由流量 |
| **端点驱动** | 透明集成现有通信库，无需修改应用 |
| **保序确定性** | 保持排序、确定性和低开销 |

**测试环境**: H100-SXM4 节点 + 全互联 NVLink + 4× NDR400 rails

| 场景 | 指标 | 加速比 | 对比基线 |
|:-----|:------|:-------|:---------|
| 单节点内带宽 | 吞吐↑ | **2.3×** | 单路径基线 |
| 节点间吞吐 | 吞吐↑ | **3.8×** | 单路径基线 |
| **Skewed All-to-Allv** | 吞吐↑ | **最高 5.2×** | **NCCL / MPI** |
| **LLM MoE 端到端** | 吞吐↑ | **1.35×** | NCCL / MPI |
| 均衡流量 | 性能 | == 基线 | NCCL / MPI |

**🎯 意义**: 
- MoE 的 all-to-all 通信本身就是高度偏斜的（不同 GPU 的 token 数不均匀），NIMBLE 的偏斜感知均衡正好切中要害
- 不需修改 NCCL 源代码，端点驱动意味着可以作为 NCCL 之上的编排层部署
- 是 **NCCL EP** (调底层) 和 **Lagom** (参数调优) 之外的**第三条路线** — 运行时流量重分配

## 2️⃣ MoE-Hub: 硬件加速 MoE 通信 (arXiv:2605.05888, ISCA 2026, May 7)

**链接**: https://arxiv.org/abs/2605.05888

**核心洞察**: MoE 的**根本瓶颈**不是通信带宽本身，而是**抽象不匹配** — MoE 的动态、不规则 token-to-expert 映射 vs. 现代 GPU 的静态、address-centric 通信模型。每次通信前必须先经过复杂的**软件地址解析阶段**，这个阶段限制了重叠效率和编程灵活性。

**MoE-Hub 方案**: 引入 **destination-agnostic 通信范式** — 硬件在 GPU Hub 中透明处理地址分配和数据流编排。

| 要素 | 机制 |
|:-----|:------|
| 逻辑目标通信 | 生产者路由后直接用逻辑目标发送数据，无需等待地址 |
| 硬件地址分配 | 轻量级硬件在 GPU Hub 中处理 |
| 数据流编排 | 硬件透明编排，软件介入移除 |

| 场景 | 加速比 |
|:-----|:-------|
| **per-layer** (单MoE层) | **1.40× - 3.08×** |
| **端到端** | **1.21× - 1.98×** |

**🎯 意义**:
- 被 **ISCA 2026** 接收，硬件-软件协同设计的标杆工作
- 明确指出 NCCL EP / DisagMoE 等纯软件方案的天花板：**抽象不匹配只能通过硬件解决**
- 如果 GPU Hub 硬件集成该方案，MoE all-to-all 的软件开销将大降

## 3️⃣ DisagMoE: 分离 Attention/FFN 的 MoE 训练 (arXiv:2605.11005, May 10)

**链接**: https://arxiv.org/abs/2605.11005

**核心问题**: Expert Parallelism (EP) 遭受严重的 all-to-all 通信瓶颈。现有计算-通信重叠方案在 attention 和 FFN 层的计算-通信比固有失衡下，仍留有**残留网络瓶颈 (residual network-bound stall)**。

**DisagMoE 方案**: 将 **Attention 和 FFN 层分离到不同 GPU 组**，引入多阶段流水线。

| 技术 | 机制 |
|:-----|:------|
| **分离式放置** | Attention GPU 组 + FFN GPU 组，单向多对多通信 |
| **多级流水线** | 单向多对多通信消歧延迟 |
| **roofline 模型** | 均衡 attention 组和 FFN 组的 GPU/网络带宽分配 |

**实现**: Megatron-LM 上实现

| 场景 | 加速比 |
|:-----|:-------|
| **16节点 8×H800** (128 GPU) | **最高 1.8×** |

**🎯 意义**:
- 与 MoE-Hub 形成互补：软件侧通过**放置优化**解决，硬件侧通过**通信范式**解决
- 训练场景比推理更激进：可以直接把注意力从 all-to-all 路径上完全拿掉

## 4️⃣ NCCL v2.30.7-1 (Jun 4, 2026) — Zero-SM Collectives GA 🆕

**链接**: https://github.com/NVIDIA/nccl/releases/tag/v2.30.7-1

**核心亮点**: **Zero-SM Collectives** (零 SM 集合通信) 正式可用。

| 新特性 | 详情 |
|:-------|:------|
| **Zero-SM AllGather / All2all** | 分层方案：节点间使用 **RMA CPU proxy**，节点内使用 **Copy Engine** |
| **NCCL_CTA_POLICY_ZERO** | 启用 Zero-SM 模式的flag |
| **GPI 后端 (GPU Push Interface)** | 新的 GIN 后端，通过 GPU 直接推数据 |
| **Explicit Signal Semantics** | Strong/Weak 信号区分，更精细的同步控制 |
| **NCCL_GIN_IB_TC** | 独立控制 GIN 的 IB 流量等级 |
| **MPS + MLOPart** | 每个物理 GPU 最多 2 个 rank 的 MPS 支持（实验性） |
| **GRH 支持** | 支持需 global route headers 的 IB 端口 |

**Bug 修复**:
- RoCE LAG 环境下的 **QP 负载均衡** (round-robin 队列亲和策略) ✅
- PXN 下的动态内存死锁修复 ✅
- RMA CE 批量操作性能优化 ✅
- GIN proxy thread NUMA 本地 CPU pinning ✅

**🎯 意义**:
- **Zero-SM = 计算 kernel 不被通信占用任何 SM**，对 PD 分离推理中的 prefill 密集计算至关重要
- RMA CPU proxy 策略对 RDMA 网络延迟不敏感的场景（跨节点）更友好

## 5️⃣ NCCL EP v0.1.0 (Jun 8, 2026) + NCCL4py v0.3.1 (Jun 11, 2026) 🆕

**NCCL EP**: 专用 MoE 通信库，基于 NCCL Device API (LSA + GIN) 实现高效的 Dispatch/Combine 原语。

**v0.1.0 关键增强**:

| 维度 | 改进 |
|:-----|:------|
| **性能** | Dispatch HT 模式 JIT 编译 + 零拷贝；HT ncclTeamRail 替代 split communicator |
| **可编程性** | ncclEpInitHandle/UpdateHandle 分离控制路径和数据路径 |
| **容错** | Active rank mask 支持故障 rank 排除 |
| **CUDA Graph** | EP Handle 管理支持 CUDA Graph capture/replay |
| **多节点** | 完全支持 Multi-Node-NVLink (MNNVL)；8 节点以上修复 |
| **内存** | LL 模式每 rank token 去重 + rank-major 布局，staging buffer 缩小 expert_per_rank 倍 |

**NCCL4py v0.3.1**:
- **`nccl.ep`** — Pythonic NCCL EP API: `Group.create()`, `Handle.dispatch()/combine()`
- **`nccl.core.device.cute`** — CuTeDSL kernel 直接调用 NCCL Device API
- **PyTorch interop** — `get_nccl_comm_from_group()` 从 PyTorch 进程组直接创建 NCCL 通信器
