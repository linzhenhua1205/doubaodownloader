## 1️⃣ 🚀 OpenUCX v1.21.0 正式发布 (2026-06-24) 🏆[本周最重磅]

**链接**: https://github.com/openucx/ucx/releases/tag/v1.21.0

**概述**: OpenUCX v1.21.0 从 v1.21.0-rc2 (Jun 8) 正式转正。这是继 v1.20.0 (Feb 2026) 后时隔 4.5 个月的首个正式版，包含 GDAKI CUDA 修复、GGA transport 配置选项等关键更新。

### 核心特性

| 分类 | 特性 | 意义 |
|:-----|:-----|:-----|
| **UCP 协议** | `UCX_PROTO_EMULATION_ENABLE` 强制零拷贝 RMA 协议选择 | 用户可强制绕过非最优协议路径 |
| **UCP GPU 管理** | `UCX_MAX_HCA_PER_GPU` 限制 GPU 内存注册到最近 HCA | 控制多 HCA 环境下 GPU 绑定策略 |
| **UCP 回退** | 设备 lanes 可访问 host 内存用于 GPU 传输回退 | GPU 直接 path 不可用时的降级路径 |
| **UCP gdr_copy** | 为 memtype endpoint 传输启用 gdr_copy | 提升 GPU 直接拷贝场景性能 |
| **UCT 设备池** | 新增设备 channel pool 支持 | 更好的资源池化 |
| **UCT AMO** | CPU 内存作为 AMO 的本地 buffer | 设备原子操作的本地缓冲优化 |
| **IB GDAKI CUDA 上下文** | `UCX_IB_GDA_RETAIN_INACTIVE_CTX` 控制非活跃 CUDA context 保留 | 减少 GDAKI context 切换开销 |
| **GGA Transport** | 新增 `--with/without-gda` 构建选项控制 GGA transport | GGA 可作为可选功能而非强制依赖 |

### 关键 Bug 修复

| 修复 | 场景 | 严重度 |
|:-----|:------|:-------|
| **GDAKI CUDA 上下文处理** | endpoint 创建期间 CUDA context 错误 | 🔴 产线级 — 多 GPU + CUDA_VISIBLE_DEVICES 场景 |
| **GDAKI NIC/GPU 映射** | CUDA_VISIBLE_DEVICES 隐藏物理 GPU 后映射错误 | 🔴 产线级 — 容器环境常见配置 |
| **GDA WQE 排序** | CAS-based readiness marking 修复 | 🟡 影响 GDA transport 的数据一致性 |
| **CUDA IPC cache 分离** | 本地和远程 rkey 的 CUDA IPC 可达性缓存分离 | 🟡 影响多节点 CUDA IPC |
| **CUDA IPC cache/LRU** | 已引用区域的 LRU 不变量修复 | 🟡 内存泄漏场景 |
| **DMA-BUF 偏移** | 内部 CUDA 地址的偏移计算修正 | 🟡 影响 Grace Hopper 等 DMA-BUF 场景 |
| **IPv4 link-local 跳过** | TCP 接口选择跳过 link-local 地址 | 🟢 多网络接口场景 |
| **ROCm HIP MPI 挂起** | ROCm 端 HIP MPI/OMB 测试挂起修复 | 🟢 AMD GPU 场景 |
| **Debian 打包** | GDA RPM/devel 子包布局修复 | 🟢 包管理场景 |

**相关性**: ★★★★★ — 分布式通信的基础栈 OpenUCX 正式版更新，GDAKI CUDA 上下文修复对 Kubernetes GPU 容器环境至关重要

## 2️⃣ SplitZip: GPU 友好无损 KV Cache 压缩用于解聚 LLM 推理 (arXiv:2605.01708, 替换更新) 🔄

**链接**: https://arxiv.org/abs/2605.01708

**核心问题**: 解聚 LLM 推理中，Prefill Worker 生成的 KV Cache 必须传输给 Decode Worker 才能开始生成。随着长输入和 Agentic 工作负载的兴起，KV Cache 传输成为**显著的端到端瓶颈**。

**SplitZip 方案**:

| 创新点 | 技术细节 |
|:-------|:---------|
| **GPU 友好的无损压缩器** | 保留 KV 张量的比特级完整性，无需修改模型执行 |
| **浮点指数冗余利用** | 利用 KV 激活中浮点指数的统计冗余，用固定长度编码表示频繁指数值，稀疏逃逸流覆盖罕见指数 |
| **Top-16 指数码本** | 校准的 top-16 指数码本消除了在线直方图计算 |
| **规则密集路径 + 稀疏逃逸纠正** | 编解码均在 GPU 上高效运行 |

**核心数据**:

| 指标 | 值 |
|:-----|:----|
| 压缩吞吐 | **613.3 GB/s**（比之前最先进的无损压缩器快一个数量级） |
| 解压吞吐 | **2181.8 GB/s** |
| BF16 KV Cache 传输加速 | 最高 **1.32×** |
| TTFT 加速 | **1.30×** |
| Request Throughput 提升 | **1.23×** |
| FP8 E5M2 KV Cache 兼容 | 可额外压缩 **1.14×** 于原生 E5M2 |
| BF16 平均压缩比 | ~**1.17×**（报告 chart 估计） |

**相关性**: ★★★★ — 分布式推理系统 KV Cache 传输优化，解聚推理的核心 IO 瓶颈缓解

## 3️⃣ ActPlane: 面向 AI Agent Harness 的可编程 OS 级策略执行 (arXiv:2606.25189, cs.OS 新投稿) 🆕

**链接**: https://arxiv.org/abs/2606.25189

**核心问题**: AI Agent 在生产环境中通过 harness 运行，需要执行安全和有效性策略（如"提交前先运行测试"）。但策略意图用非精确自然语言表达，执行必须作用于具体系统动作。现有 tool-call guardrails 无法覆盖绕过工具层的系统动作，OS sandbox 仅控制资源而非动作。

**ActPlane 方案**:

| 组件 | 描述 |
|:-----|:------|
| **策略声明 DSL** | 基于信息流控制 (IFC) 的简单 DSL，支持跨事件策略 |
| **eBPF 内核执行** | 策略在 OS 内核层执行，覆盖所有执行路径 |
| **语义反馈** | 向 Agent 返回语义错误（而非 opaque 拒绝） |
| **隔离** | 失败的策略执行不会影响其他 Agent |

**核心数据**:
- 策略合规性提升（包含 tool-call 拦截无法观察的间接执行路径）
- **1.9%-8.4%** 性能开销
- 支持 test-before-commit、数据流限制等常见 Agent 策略

**相关性**: ★★★ — AI Agent OS 层优化，eBPF 在内核层执行 Agent 策略的首个系统。对分布式 Agent 操作系统（容器/K8s 环境下的 Agent 策略管理）有参考价值

## 4️⃣ Birkhoff 分解与光子互连：别忘记计算！(arXiv:2605.26845, 替换更新) 🔄

**链接**: https://arxiv.org/abs/2605.26845

**核心问题**: MoE 模型的 all-to-all 通信模式高度倾斜，成为光电路交换互联的自然用武之地。但**现有方法把通信孤立考虑，忽略了通信与后续 Expert 计算的交互**。

**关键发现**:

| 问题 | 描述 |
|:-----|:------|
| **BvN 非双随机性** | MoE 通信矩阵极少是双随机的，BvN 调度产生大量气泡 |
| **BvN 匹配过多** | BvN 分解产生过多 matchings，将执行碎片化为小批量，固定执行开销导致严重的计算效率损失 |
| **方案** | Greedy max-weight 分解策略，在限制 matching 数量的同时保留大批量，接近理想无拥塞 all-to-all |

**机制**: Dispatch → Compute → Combine 结构从根本上挑战了经典 Birkhoff-von Neumann 调度技术。通信-计算重叠因为 BvN 的 excessive matchings 反而被破坏。

**相关性**: ★★★★ — 分布式训练/推理网络拓扑调度，MoE all-to-all 的电路交换调度优化，直接关联到光互联方案在 AI 集群中的应用
