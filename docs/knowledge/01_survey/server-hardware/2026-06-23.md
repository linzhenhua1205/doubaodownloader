## 1️⃣ 🏗️ MiTAC Computex 2026：52U 液冷整机柜 96×MI355X + Diamond Cooling + MiOBMC 开源固件

- **来源**: [ServeTheHome — Ryan Smith (Jun 21)](https://www.servethehome.com/mitac-computex-2026-booth-tour/)
- **关键词**: MiTAC, 52U, MI355X, Diamond Cooling, MiOBMC, OpenBMC, ORv3

### 52U 液冷整机柜 — 96×AMD Instinct MI355X

MiTAC 在 Computex 2026 展示了一款旗舰级液冷整机柜：

| 规格 | 数值 |
|:-----|:-----|
| 机柜高度 | **52U** (高于标准的 42/48U) |
| 服务器 | 12× **G4826Z5** 服务器 (4U, 双 EPYC 9xx5 + 8×MI355X) |
| GPU 总数 | **96×AMD Instinct MI355X** |
| 网络交换 | **Broadcom Tomahawk 5** 800Gbps Ethernet |
| CDU | **Nidec** 200kW 液冷能力 |
| 密度宣称 | 比标准 AI 配置(风冷) **密度高 50%** |
| 定位 | **Turnkey 整机柜**—到货即用 |

**关键架构设计**:
- 单台 G4826Z5 4U 服务器分上下两区：**上部 GPU、下部 CPU+其余组件**
- 正面 8× U.2 SSD 本地存储托架
- 后部高速网络，**Broadcom Tomahawk 5** 800GbE 做无阻塞交换

### Diamond Cooling — 金刚石散热技术 (MI350X 8U 机型)

MiTAC G8825Z5 (8U, 双路 AMD, 8×MI350X) 集成了 **Akash Systems 的 Diamond Cooling 技术**：
- 使用**合成金刚石**作为导热介质 (结构化碳)
- 金刚石导热系数远高于铜 → **更小温差即可导出同等热量**
- 允许数据中心**运行更高温度环境** → 减少冷却需求 → 更高能效
- 该机为 **风冷** 而非液冷，金刚石降低对进风温度要求

### MiOBMC / MiOPF — 开源固件栈

MiTAC 同时发布了完整开源固件栈：
- **MiOBMC**: 基于 OpenBMC 的开源 BMC 软件
- **MiOPF**: 基于 EDK2 的系统固件
- 优势展示: **快速启动时间**、标准监控和管理能力
- 定位: 符合 OCP 开源固件趋势，**开放标准**为基础而非从闭源出发

> 💡 **解读**: MiTAC 沿着 "62U? 48U → 52U" 加高趋势，配合液冷提升密度 50%。Diamond Cooling 是风冷路径的另类创新（用材料科学解决散热瓶颈）。MiOBMC 表明**整机柜厂商正在将 BMC 软件差异化作为核心竞争力**。

## 2️⃣ 🧠 Building a Dense Agentic AI CPU Rack Today — Agentic AI 驱动 CPU 需求范式转变

- **来源**: [ServeTheHome — Patrick Kennedy (Jun 19)](https://www.servethehome.com/building-a-dense-agentic-ai-cpu-rack/)
- **关键词**: Agentic AI, CPU, OpenClaw, Hermes, 确定性工作流

### 核心论点：Agentic AI 是 CPU 故事，不是 GPU 故事

Patrick Kennedy 的深度分析文章提出几个关键判断：

**事实数据**:
- **Cloudflare CEO (Jun 3)**: AI bot 流量已**超过人类互联网流量**
- 实验数据: GPT-5.5/Qwen3.5-397B/Fable-5/Opus-4.8 通过 SSH 执行命令时，仍有 **~25% 的工作流因格式错误陷入循环**（哪怕模型自我修复）
- CPU 上的**确定性执行**是解决 token 浪费的核心手段

**运行 Agent 需要 CPU 的五大场景**:
1. **Agent 框架自身** (OpenClaw/Hermes) — 运行在 CPU 上
2. **短生命周期 Sandbox** — 创建、执行命令、销毁
3. **确定性工具执行** — LLM 格式化请求 → CPU 端工具确定性执行
4. **后端数据库/应用** — 承载 Agent 产生的请求 (Oracle/SQL Server 等 License 成本需重新评估)
5. **Web 服务器/边缘** — 处理 bot 请求

**关键洞察**:
- 从"人工运营商"到"机器运营商"的转换是**净新增负载**
- 基础设施从服务 80 亿人口 → 服务 80 亿人 + 数量级更多的 Agent
- 应用需要为 **M2M (Machine-to-Machine)** 流量优化，而非仅 H2M
- Agent 有超时窗口，不会等人 → 基础设施必须演化

> 💡 **解读**: 这是本周最重要的 CPU 叙事文章。在 GPU 算力被 AI 训练/推理占据的同时，**Agentic AI 为 CPU 市场带来了全新的增量需求**。服务器 CPU 选型从 "需要多少核跑传统应用" → "需要多少核跑 Agent 工作负载"，这是一个根本性的需求变化。

## 3️⃣ 🔗 Intel E835 NIC 家族 — 200GbE/100GbE/25GbE, 半功耗 vs ConnectX-6 Dx

- **来源**: [ServeTheHome — Patrick Kennedy (Jun 18)](https://www.servethehome.com/this-is-the-new-intel-e835-nic-lineup-from-hpe-discover-2026/)
- **首发**: HPE Discover 2026 展台

### E835 产品线全貌

| 型号 | 端口 | 速率 | 形态 | 建议价 |
|:-----|:-----|:-----|:-----|:------|
| E835-CQDA2 | 2× QSFP28 | 2×100GbE / 1×200GbE | LP | <$600 |
| E835-CQDA1 | 1× QSFP28 | 1×200GbE | LP | 低于双口 |
| E835-XXVDA2 | 2× SFP28 | 2×25GbE | LP | $216 或更低 |
| E835-XXVDA4 | 4× SFP28 | 4×25GbE | LP | 中~$500 |
| E835-XXVDA4F | 4× SFP28 | 4×25GbE | FH | 待公布 |
| E835-XXVDA4 OCP | 4× SFP28 | 4×25GbE | OCP NIC 3.0 | 低~中$500 |

### 关键卖点
- **功耗**: Intel 宣称 E835 功耗约为 **NVIDIA ConnectX-6 Dx (2019) 的一半**
- **技术代差**: 7 年制程/工艺进步带来功耗大幅降低
- **PCIe**: PCIe 5.0 接口（支持 x4 即可跑满 100G）
- **RoCEv2 + iWARP + DCB** 全部支持
- **低矮卡(LP)** 适配几乎所有服务器

### 市场定位
- **AI 数据中心**: 25GbE 作为管理接口，400/800GbE 用于 AI 网络
- **传统 DC + 边缘**: 100/200GbE/25GbE 仍有大量需求，功耗敏感场景 (10W 节省即有显著意义)
- Intel 策略: **NVIDIA 在高端 AI 网络聚焦 → Intel 在主流/低功耗市场填空白**

> 💡 **解读**: Intel E835 的核心价值不是"更快"而是"更省电"——在 AI 数据中心中最容易被忽视的非 AI 网络层面，每卡节省 10W × 数千台服务器 = 可观的运营成本降低。同时 OCP NIC 3.0 规格的 25GbE 卡，适合大批量管理网络部署。

## 4️⃣ 🧵 HPE Cray GX5000: 8× AMD Venice 单 blade, 81,920 核/rack

- **来源**: [ServeTheHome — Patrick Kennedy (Jun 17)](https://www.servethehome.com/81920-cores-per-rack-with-amd-epyc-venice-at-hpe-discover-2026/)

### 核心规格

| 指标 | 数值 |
|:-----|:-----|
| 单 blade 形态 | **HPE Cray GX250a 计算 blade** |
| 每 blade CPU | **8× AMD EPYC Venice (SP7)** |
| 每 blade 内存 | 每 CPU 16 通道 DDR5 (MRDIMM), 全液冷 |
| 每 rack blade 数 | ~36 blades (40U 有效) |
| **每 rack 核数** | **81,920 核** (≈ 284 核/socket) |
| 本地存储 | Samsung E1.S EDSSF SSD, 安装在 CPU 冷板上方 |
| 散热 | **全液冷** (CPU + DIMM + 互联) |
| CDU | HPE 1.6MW CDU |
| 网络 | **Slingshot 400** (当前) / **Slingshot 800** (未来), 双 NIC blade 两侧 |
| 互联形态 | 类似 OCP NIC 3.0 连接器, 线缆到每颗 CPU |

### 架构看点
- **8× CPU blade 不是 4× 双路，而是 8× 单路**—每颗 Venice ~284 核
- 存储在冷板上方—极致空间利用率的物理表现
- 标注 **Vanover VP1-01** —证明是**可工作的系统**而非静态模型 (相比 SC25 时的原型机已成熟)
- 机柜前部网络部件以 **90° 角突出**—特别布线方案

### 超算目的地
- 该系统将用于 **Los Alamos "Mission" 超算** (与 NVIDIA 合作)
- 当前密度 ~8,000 核/rack (风冷) → **GX5000 ~10× 密度提升**

> 💡 **解读**: HPE Cray GX5000 代表了**超算级 CPU 密度**的巅峰。8× Venice per blade + 全液冷 + Slingshot 800 可升级架构，为下一代超大规模模型训练和科学计算做好了准备。注意这是**纯 CPU 超节点**而非 GPU 节点，面向的是高吞吐量计算而非 AI 训推。
