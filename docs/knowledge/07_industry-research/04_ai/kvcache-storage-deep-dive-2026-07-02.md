## 1. 专题概览

### 1.1 核心主题

KV Cache（Key-Value Cache）是 LLM 推理过程中，每一层注意力计算产生的历史 token K/V 向量的缓存。随着上下文从 4K 向 1M+ 扩展，KV Cache 的存储需求呈线性增长——单请求 128K 上下文需 448 MB（70B 模型），1M 上下文需 3.5 GB。**超节点（512 GPU）场景下，KV Cache 存储量可达 TB 级，远超单 GPU HBM 容量。**

G1-G4 分层存储体系正是为了以最低 $/Token 成本管理 KV Cache 而设计：

```
G1: GPU HBM    (~200GB/GPU)  → 活跃 KV, 纳秒级  → ~$80/GB
G2: CPU DRAM   (~2TB/节点)   → 温 KV, 百纳秒级  → ~$5/GB
G3: 本地 NVMe  (~TB/节点)    → 温 KV, 微秒级    → ~$0.15/GB
G3.5: 池化NVMe (PB级/超节点) → 冷 KV, 1-5μs     → ~$0.10/GB  ★ 新增层级
G4: 分布式存储  (PB级)       → 归档, 毫秒级     → ~$0.02/GB
```

**核心驱动力**：将 KV Cache 从昂贵的 HBM 卸载到低成本 NVMe 存储层，以存储成本换 GPU 利用率，实现最优的每 Token 总成本。

### 1.2 为什么跟踪

| 驱动因素 | 量化影响 | 时间窗口 |
|:---------|:---------|:---------|
| 长上下文模型爆发（1M+ tokens） | KV Cache 需求增长 32x（4K→128K） | 2026-2027 |
| MoE 模型普及 → KV Cache 占比升至 60-80% | 推理显存瓶颈 | 2026-2028 |
| Agentic AI 多轮推理 → KV 无法一次性释放 | 需持久化存储 | 2026-2027 |
| 国产 GPU 集群 HBM 受限更严重 | HBM<200GB/卡 → 卸载需求更强 | 当前 |
| PCIe 6.0 / CXL 3.0 生态成熟 | 为 G3.5 提供低延迟互联管道 | 2027-2028 |

### 1.3 关键判断

1. **G3.5 是超节点推理的"必争之地"** — 没有 KV Cache 存储层的超节点无法经济地运行 Agentic AI 工作负载
2. **NVIDIA CMX 领先 12-18 个月** — 软硬件深度耦合（BF4 + DOCA Memos + Dynamo），但封闭生态给国产方案留下窗口
3. **CXL 内存池是 Intel 差异化的王牌** — 延迟比 NVMe 低 1 个数量级（300ns vs 5μs），但容量和供应链成熟度仍是问题
4. **国产化挑战大于机会** — 除国产 GPU 的 HBM 受限外，DPU（BF4 级）、NVMe-oF KV 协议栈、集群调度软件均为短板
5. **存储层将决定超节点 TCO** — 推理场景中 KV Cache 存储成本占推理 TCO 的 15-30%，合理的 G3.5 方案可降低 40-60%

## 2. 技术深潜：KV Cache 存储设施全景（P1）

### 2.1 问题定义与第一性原理

#### 2.1.1 KV Cache 的存储特性（为什么需要专用存储）

KV Cache 是一种**独特的 AI 原生数据类型**，与通用企业数据有本质区别：

| 特性 | KV Cache | 企业数据 | 对存储架构的影响 |
|:-----|:---------|:---------|:----------------|
| **生命周期** | 瞬态（秒-分钟级） | 持久化（年） | 无需快照/备份/复制 |
| **可重建性** | 可重新计算 | 不可丢失 | CRUD 语义简化，无需 RAID6 |
| **访问模式** | 顺序写入 + 随机读取 | 随机读写混合 | 存储应针对顺序写优化 |
| **一致性要求** | 最终一致性 | 强一致性 | 无需分布式事务 |
| **粒度** | 4KB-64KB 块 | 4KB-1MB | 小 I/O 优化 |
| **带宽敏感度** | 极高 | 中 | 需要高带宽而非低延迟 |

**第一性原理推导**：

```
最小 KV Cache 存储成本 = f(介质成本, 带宽利用率, 协议开销, 冗余度)

按推理场景：
  - 介质成本: NVMe ≈ $0.15/GB, CXL DRAM ≈ $3/GB, HBM ≈ $80/GB
  - 带宽利用率: 顺序读 NVMe ~7GB/s/盘（PCIe 5.0 x4）
  - 协议开销: NVMe-oF ~3-5%, RDMA ~1-2%
  - 冗余度: KV Cache 可重建 → 无需副本 → 冗余度=1

结论: 最佳 TCO 点在 NVMe 介质 + RDMA 协议 + 无冗余模式
```

#### 2.1.2 超节点场景的特殊性

| 超节点特征 | KV Cache 存储要求 |
|:-----------|:-----------------|
| 512 GPU 共享推理 | 跨节点 KV 共享 → 存储必须池化（非本地） |
| Agentic AI 多轮对话 | KV 须持久化数分钟至数小时 |
| PD 分离架构 | Prefill 产生 KV → Decode 消费 KV → 跨阶段传输 |
| MoE 模型稀疏激活 | KV Cache 占比 60-80% → 存储压力更大 |
| 国产 GPU HBM 受限 | 国产 HBM2e (32-80GB) → 卸载需求更迫切 |

#### 2.2.1 四层存储层次

```
                   超节点 KV Cache 存储层级
                          
 层级     容量/超节点     延迟       带宽/GPU      成本($/GB)   适合策略
 ┌─────────────────────────────────────────────────────────────────────┐
 │ G1: GPU HBM    1.6 TB   ~80 ns   3-8 TB/s     ~$80     活跃 KV   │
 │     (8×200GB)                                                     │
 ├─────────────────────────────────────────────────────────────────────┤
 │ G2: CPU DRAM   16 TB   ~150 ns  200-500 GB/s  ~$5      暂态 KV   │
 │     (8×2TB)                                                       │
 ├─────────────────────────────────────────────────────────────────────┤
 │ G3: 本地 NVMe  64 TB   ~5 μs    5-10 GB/s    ~$0.15   本地复用   │
 │     (8×8TB/节点)                                                   │
 ├─────────────────────────────────────────────────────────────────────┤
 │G3.5: 池化存储  1-5 PB  ~1-5 μs  200-800 GB/s  ~$0.08-0.15 共享冷KV│

```

#### 2.2.2 各层级卸载决策逻辑

```python
def kv_storage_decision(kv_block):
    """KV Cache 块放置决策引擎"""
    if kv_block.is_active() and gpu_hbm_free > kv_block.size:
        return "G1: GPU HBM"         # 活跃块，直接放在 HBM
    elif kv_block.expected_reuse_within_ms(10):
        return "G2: CPU DRAM"        # 预计 10ms 内复用
    elif kv_block.expected_reuse_within_s(60):
        return "G3: 本地 NVMe"       # 60s 内复用
    elif kv_block.is_shared_across_nodes():
        if has_cmx_or_equivalent():
            return "G3.5: 池化存储"  # 跨节点共享 → 池化层
        else:
            return "G3: 本地 NVMe + RDMA传"
    else:
        return "G4: 对象存储"        # 超长间隔/归档
```

|:-----|:---------|
| **控制器** | BlueField-4 DPU（Vera CPU + CX9 SN + 硬件加速器） |

| 维度 | 国产等价方案 | 差距分析 |
|:-----|:------------|:---------|

**推荐的国产化改造路径**：

```
Step 1: 硬件层兼容
  - 使用标准 NVMe SSD + 通用 JBOF
  - GPU 通过 PCIe EP 直连 400G NIC
  - 实现基础 NVMe-oF RDMA 读写

Step 2: KV 协议增强
  - SPDK 上构建 KV 块管理引擎
  - 实现关键（Block ID → KV Block）映射
  - Prefix Cache 共享机制

Step 3: 与推理框架集成
  - vLLM / SGLang 插件化 KV offload
  - GPU Direct RDMA 零拷贝路径
  - KV 预取与缓存替换策略

Step 4: 集群级调度
  - 推理框架 + 存储拓扑感知调度
  - KV 热迁移（GPU 故障/负载均衡）
  - 多超节点 KV 共享
```

#### 2.3.4 方案四：开源/混合方案（CXL+NVMe 异构）

| 子方案 | 架构 | 优势 | 劣势 |
|:-------|:-----|:-----|:-----|
| **CXL 作为热缓存** | CXL DRAM 做 1-2TB 热缓存 + NVMe 做冷存储 | 延迟接近 DRAM，容量接近 NVMe | 两层管理复杂 |
| **多级 SSD 池** | SLC/TLC/QLC 分层（Optane 级→TLC→QLC） | 延迟/容量/成本梯度平滑 | SSD 层数多，管理成本高 |
| **计算-存储融合** | 存储节点集成推理加速器做近存处理 | 减少数据搬移 | 架构复杂，标准化难度高 |

### 4.3 北极星指标体系

#### 4.3.1 北极星指标

```
KV 存储效率指数 (KSEI) = 0.35 × 有效吞吐 + 0.25 × 延迟SLA达成率 + 
                          0.20 × 存储成本效率 + 0.20 × 能效比

其中:
  有效吞吐 = G3.5 层实际 KV 读写带宽 / 理论最大带宽
  延迟SLA达成率 = P99 读延迟 < 5μs 的请求占比
  存储成本效率 = 每 GB 存储处理的 Token 数量 / 存储成本
  能效比 = 每瓦特存储处理的 Token 数量
```

#### 4.3.2 四层指标体系

| 层级 | 指标名称 | 计算方式 | 目标值 | 数据源 | 采集频率 | 告警阈值 |
